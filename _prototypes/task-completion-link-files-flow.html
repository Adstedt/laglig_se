Story 6.7c · UX prototype

Task completion → link files to law list items

The law list item modal already aggregates files via the Länkade filer & dokument section — task attachments show up there transitively (filter chip Via uppgifter). Today those files only stay visible while the task→list-item link exists; if the task is later deleted or unlinked, the file disappears from the law list item. This feature promotes them to first-class direct attachments at task completion — the file gains a 🔗 icon (Direktlänkad) while keeping its 📋 task-provenance, so it survives the task lifecycle. Pattern is pre-checked confirmation (Option C) — friction is one button click; consent is explicit.

Strategic context

Why we're building this — preserving compliance history for agent reasoning

This feature is one mechanism in a broader pattern: convert ephemeral compliance work into durable, queryable history owned by the law list item. The end goal is that our AI agent can answer "what does this obligation actually mean for this specific customer?" by reasoning over a complete timeline — not just the current state.

The structural problem

Today, much of a list item's "story" is borrowed from related entities (tasks, audit cycles, kravpunkter, change events). When those entities are deleted, archived, or unlinked, the list item's history thins out. The agent loses signal — and the customer loses their audit trail.

The pattern

At natural commit moments — task completion, finding closure, kravpunkt review — promote borrowed history to owned history. The list item accumulates first-class evidence over its lifetime, surviving the lifecycle of any particular task or cycle.

Other instances of this pattern (not in this story — listed for context)

  • • Task completion → file promotion (this prototype)
  • • Audit cycle close → durable assessment record on each assessed list item
  • • Legal change event → permanent "reviewed on date X, outcome Y" pin per list item
  • • Workspace document version snapshot at link time → "v3 linked when task T was completed on D"
  • • Task comments referencing list items → snapshot to list item's activity log on task closure

Each is independently small. Together they make the law list item the canonical timeline for everything that has ever happened to this obligation in this workspace — exactly the foundation the agent needs.

State 5 · Edge cases (no dialog)

When the prompt is silently skipped

The dialog should never get in the user's way when there's nothing actionable to confirm. Three skip conditions:

A · No files

Task has 0 bilagor. Nothing to link → existing behavior. Status changes silently.

B · No linked dokument

Task is not linked to any law list item. Nowhere to link to → existing behavior. Status changes silently.

C · Re-opening

Transition is is_done → not-is_done (un-completing). Don't prompt — files stay where they are. Existing FileListItemLinks are not removed.

Open question

Should we also prompt on the inverse — when files are added to a task that's already completed and linked? Probably no (the completion moment is the natural commit point), but worth flagging for the spec.

Implementation

What ships if you approve

Schema

No changes. Uses existing FileListItemLink join with its (file_id, list_item_id) unique constraint.

Backend

One new server action linkTaskFilesToListItems(pairs) in app/actions/tasks.ts. Workspace-scopes file & list-item IDs, then createMany({ skipDuplicates: true }). ~40 LOC.

Frontend (new)

completion-confirm-dialog.tsx in components/features/tasks/task-modal/. Pre-checked checkbox grid. ~120 LOC.

Frontend (modified)

details-box.tsx intercept inside handleStatusChange. ~20 LOC delta.

Provenance UI

Already exists. The LinkedArtifactsPanel in components/features/document-list/legal-document-modal/ already renders both 🔗 (Direktlänkad) and 📋 (Via uppgifter) icons on each artifact row. After promotion, the file naturally shows both. No new chip needed. Optional: a brief green-row highlight on freshly-promoted files.

Effort

~3-4 hours for the core flow. No migration. No test refactor. No new visual primitives — re-uses the existing LinkedArtifactsPanel rendering.
