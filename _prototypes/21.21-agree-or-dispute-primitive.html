Story 21.21 — Agree-or-dispute primitive

Surfaces the audit's actual primitive: ratify the team's claim or challenge it. Replaces the current "pick a bedömning from a dropdown and write a motivering" pattern.

Four states shown below. The auditor moves through 1 → either 2a or 2b based on their judgment. State 4 is the edge case where there's nothing meaningful to "agree" with.

What changed
The bedömning + motivering inputs are replaced by the agree-or-dispute question. Two big buttons present the two meaningful choices the auditor has to make.
Why
The actual audit primitive is "ratify the team's claim or challenge it" — not "pick from a generic dropdown." Explicit question, two actionable buttons, source-of-truth state visible right above them.
Manual escape hatch
Advanced auditors who want to skip the agree/dispute UI can use the collapsed <details> at the bottom to manual-pick. Power user respect without confusing first-timers.

Two clicks to ship
State 1 → click "Bekräfta" → State 2a. Bedömning is auto-set to match källstatus. Motivering is pre-filled with a sensible default the auditor can edit. Click Signera = done. Most audits should hit this path.
Pre-fill copy
"Bekräftar nuvarande status från laglistan. Inga ändringar sedan föregående granskning." — neutral, audit-grade. Auditor can edit (e.g., add specifics). Empty motivering is still blocked by Story 21.12's server-side gate, so the pre-fill ensures the happy path doesn't trip the required-motivering check.
Ångra escape hatch
"↺ Ångra" reverts to State 1. Auditor can re-decide if they clicked Bekräfta by accident.

Forces a real explanation
When auditor disagrees with the team, the audit needs WHY in writing. Motivering is empty + required, button disabled until filled in. No accidental "click Signera with placeholder text."
Inline pill bedömning
Replaces the dropdown — 4 colored buttons show the choices upfront (one less click + the colors are educational about what each value means in compliance terms).
Subtle policy enforcement
The motivering placeholder explicitly nudges "explain why you disagree", not generic. Auditors writing "—" or "no comment" become obvious.

Edge case handling
When källstatus is EJ_PABORJAD or PAGAENDE, "Bekräfta" doesn't make sense — there's nothing meaningful to agree with. UI collapses to direct manual entry, with a copy hint explaining why.
Trigger conditions
källstatus IN (EJ_PABORJAD, PAGAENDE) → skip-the-prompt mode. Auditor still has to do all the work, but we don't pretend a meaningless "Bekräfta" button exists.
Why the inline pills
No "agree target" → manual entry IS the path. Same inline pills as State 2b (no dropdown). Consistency.

Decisions needed before dev (PO sign-off)

  1. Q1. Pre-filled motivering on Bekräfta — what exactly?

    Options: "Bekräftar nuvarande status från laglistan" (neutral) · "Bekräftar nuvarande status. Inga ändringar sedan föregående granskning." (assumes prior cycle existed) · "Bekräftar källstatus utan ytterligare anmärkningar" (auditor's confirmation, simpler).

    Recommendation: option 2, edited to "Inga ändringar sedan senaste översyn" so it works for first-time and recurring cycles.

  2. Q2. What does "Avvik" do for bedömning?

    Two interpretations: (a) reset bedömning to null (force pick) vs. (b) prefill with a "guess" (e.g. one step worse than källstatus). Option (a) is honest; (b) is opinionated.

    Recommendation: option (a) — null + force pick. Don't pre-bias the auditor's choice when they've explicitly said they disagree.

  3. Q3. Should manual escape hatch always be visible?

    State 1 has a collapsed "▾ Eller välj bedömning manuellt" <details>. Should it be (a) collapsed by default (current prototype) or (b) expanded for power users?

    Recommendation: collapsed. Default is the agree/dispute UI; power users learn the escape hatch on second use.

  4. Q4. Inline pill bedömning vs current dropdown?

    Item #5 in the rework list (small polish). The Avvik path uses pills; should the manual-mode escape hatch (in State 1) and the EJ_PABORJAD edge case (State 3) also use pills? Or keep dropdown there?

    Recommendation: pills everywhere. They're more discoverable and the colors carry meaning. Dropdown is a worse pattern at 4 options anyway.

  5. Q5. How do prereq pills (✓ Bedömning / ✓ Motivering) fit now?

    Currently the Signera card header shows two prereq pills. With the agree/dispute UI, the bedömning pill is essentially "have you clicked Bekräfta or Avvik+pick?" The motivering pill stays meaningful.

    Recommendation: keep both pills, drop them when in States 2a/2b (the state itself is now the indicator — pills become redundant).

  6. Q6. EJ_TILLAMPLIG handling on Bekräfta?

    If källstatus is "Ej tillämplig" and auditor clicks Bekräfta, that's still a valid audit verdict. Should pre-filled motivering differ for this case? ("Bekräftar att kravet inte är tillämpligt på denna verksamhet").

    Recommendation: yes, EJ_TILLAMPLIG gets a tailored pre-fill since the rationale category differs from compliance.

Migration impact / scope

No data model

Bedömning + motivering still use existing efterlevnadsbedomning + motivering fields on ComplianceAuditItem. No Prisma migration.

No server actions

Same updateItemBedomning, updateItemMotivering, signOffItem actions. The agree/dispute pattern is purely UI orchestration on top.

UI scope

Single file: cycle-item-modal/right-panel.tsx. Replaces the current Bedömning + Motivering + prereq-pills layout with the state machine above.

Estimated effort: 1.5–2 dev-days from approved AC. Story-worthy because of the UX decision set above (Q1–Q6) and because this is the primary action surface every audit goes through — worth getting AC + PO validation before building.
