Decision tree — does this finding get a task?

✓ Task spawned (1 atomic tx)

  • Type = AVVIKELSE + checkbox left at default (ON)
  • Type = OBSERVATION, user explicitly checks the box
  • Type = FÖRBÄTTRING, user explicitly checks the box
  • Any existing finding, "+ Skapa åtgärdsuppgift" clicked

– Finding only, no task

  • Type = AVVIKELSE, user unchecks (spot-fix, accepted, etc.)
  • Type = OBSERVATION, default OFF
  • Type = FÖRBÄTTRING, default OFF
  • Late addition still available via row button

What's intentionally NOT in 21.8

  • · No task-list-card chip or filter. Epic AC 2 "optional" deferred to post-MVP.
  • · No email. FINDING_READY_TO_CLOSE is in-app only for MVP.
  • · No deleteFinding action. FK onDelete: SetNull covers data integrity.
  • · No retroactive spawn on type change. If a finding type is edited from OBSERVATION → AVVIKELSE later, no auto-spawn. User can click "+ Skapa åtgärdsuppgift" on the row.
  • · No task-type enum (deferred). Current design uses compliance_finding_id as the de-facto type discriminator. A dedicated "Task Types" story lands when a second origin appears (see prior discussion).
  • · No task consolidation (many findings → one task). Current design is 1:1 finding:task. Consolidation is handled manually today by unchecking siblings and spawning only on the root-cause finding.
  • · No manual kontroll ↔ task linking UI (deferred to Story 21.15). The ComplianceCycleTaskLink M:N join table lands in 21.8 so the data model is future-proof, but the "+ Lägg till länk" affordance on the new card + cycle picker are deferred. In 21.8 the card only renders when the spawner populated it (via AVVIKELSE or late-addition).
  • · No cycle-detail "Uppgifter" tab (deferred to Story 21.15). The reciprocal surface — showing all tasks linked to a kontroll on the cycle detail page — is the natural pair to manual linking. Both ship together as the follow-up.
