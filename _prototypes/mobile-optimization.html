Design fidelity rules — read these before lifting markup into the app

  • · Status pills come from getStatusBadgeProps(domain, value) in lib/ui/badge-tones.ts — never hand-roll. "Pågår" for compliance/cycle is info · soft (blue), not amber. Amber is reserved for OBSERVATION, MINOR severity, and "Saknar bevis" warnings. The badge mockups below sometimes use amber for "Pågår" — that's prototype shorthand, not the spec.
  • · Safiro is registered at weight 500 only. Always pair with font-medium; never font-bold/font-semibold (browser fakes the weight and frequently falls back to system-ui).
  • · The mobile detection hook lives at @/lib/hooks/use-mobile (NOT use-is-mobile). MOBILE_BREAKPOINT = 768 matches Tailwind's md:.
  • · <Button size="icon"> is h-10 w-10 (40px) — clears WCAG AA but misses AAA's 44×44. Use the new icon-mobile variant on phones.
  • · Citation pills should reuse components/ui/inline-citation.tsx — don't invent a new badge style.
  • · Sticky-bottom bars and full-screen modal headers must respect both env(safe-area-inset-bottom) and env(safe-area-inset-top). 100dvh > 100vh for any full-screen surface.

Implementation sketch 
// components/ui/responsive-dialog.tsx
'use client'
import { useIsMobile } from '@/lib/hooks/use-mobile'  // NOT use-is-mobile
import * as Dialog from '@/components/ui/dialog'
import * as Sheet from '@/components/ui/sheet'

export function ResponsiveDialog({ open, onOpenChange, children }) {
  const isMobile = useIsMobile()                        // breakpoint = 768px (md:)
  if (isMobile) {
    return (
      <Sheet.Sheet open={open} onOpenChange={onOpenChange}>
        <Sheet.SheetContent
          side="bottom"
          className="rounded-t-3xl max-h-[85dvh] pb-[calc(env(safe-area-inset-bottom)+12px)]"
        >
          {children}
        </Sheet.SheetContent>
      </Sheet.Sheet>
    )
  }
  return (
    <Dialog.Dialog open={open} onOpenChange={onOpenChange}>
      <Dialog.DialogContent>{children}</Dialog.DialogContent>
    </Dialog.Dialog>
  )
}

A4 — iOS keyboard

Keep sticky bars above the keyboard

Every sticky-bottom action bar (modals + edit pages) covers up under the iOS keyboard by default. One viewport meta change fixes most of it; 100dvh instead of 100vh handles the rest.

Pattern 16 · highest product leverage

📷 Photo evidence capture (camera-first)

Unlike sections 1–15 (making the desktop product workable on a phone), this is a capability the desktop can't have: the user's camera + their physical presence at a worksite. For AFS protective-equipment compliance, ergonomics audits, machine guards, fire safety — a 30-second site walk with a phone produces all the evidence needed. Photos attach directly to the kravpunkt or law list item via the existing RequirementEvidenceLink model (Story 17.18).

Technical sketch

Native camera trigger

<input type="file"
  accept="image/*"
  capture="environment"
  multiple />

Client-side compression

6 MB iPhone photo → ~400 KB after canvas resize at max 1920px / JPEG q=0.85. 15× storage saving. Canvas re-encode handles HEIC → JPEG as a side-effect on iOS Safari (which decodes HEIC natively). Detect HEIC on non-Safari browsers via file.type, surface a fallback toast — only ship a polyfill (heic2any, ~80 KB) in v2 if real users hit it.

Storage-quota pre-flight

Check workspaceUsage.usedBytes against the cap (1 GB Solo / 5 GB Team) before the camera fires. A worker mid-walk can't recover from "kvot full" after taking 8 photos. Block + show "Aktivera Team för mer lagring" if <5 MB remains.

EXIF GPS strip

Privacy + GDPR. Use exifr to strip GPS tags before upload. Keep timestamp + camera model for audit trail.

Auto-association

Triggered from kravpunkt → RequirementEvidenceLink per Story 17.18. From LinkedArtifactsPanel → FileListItemLink.

Effort

~3–4 days for v1 (no offline queue, no HEIC polyfill). High product leverage per LOC.
