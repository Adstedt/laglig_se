UX prototype · Option A

Task modal: reuse LinkedArtifactsPanel instead of standalone EvidenceAccordion

Goal: single canonical "things linked to this entity" component shared between the law list item modal and the task modal. Same visual vocabulary, same affordances (upload, link, filter, toggle), same filter chip language — minus the chips that don't apply to tasks. The data model already supports it (FileTaskLink + WorkspaceDocumentTaskLink both exist), the task modal just doesn't surface the document side today.

Same Header, dropzone, file rows, link buttons Hidden on task Bevis, Via uppgifter chips New on task Styrdokument-toggle, "Länka styrdokument" button
Why

Tasks accumulate compliance evidence too

Same thesis as the file-promotion feature: a task with attached bilagor and linked styrdokument is itself a piece of audit history. Right now the task modal artificially limits itself to "Bilagor" — no surface for linking workspace documents (e.g. "this task implements Policy för mobiltelefoner") even though WorkspaceDocumentTaskLink already exists in the schema. Reusing the law list item's panel (a) closes that surface gap, (b) gives users one mental model for "linked artifacts on entity X", and (c) shrinks the maintenance surface from two components to one.

Filter chips

Which filter chips apply per entity

The component is parameterized by entity type. Filter chips that don't apply to the given entity are hidden entirely (not greyed-out disabled — that's noise). The strikethrough versions in the comparison above are illustrative only; in production those chips simply wouldn't render on a task.

Implementation

What ships if you approve

Schema

No changes. FileTaskLink + WorkspaceDocumentTaskLink already exist (schema lines 1276 + 1857).

Action layer

Refactor getLinkedArtifactsForListItem → getLinkedArtifactsForEntity({ type, id }) with branched data fetch. Single function, two pathways. ~80 LOC of refactor + new branch.

Component refactor

LinkedArtifactsPanel takes a new entity: { type, id } prop + chips: FilterChip[] config (which chips to show). All callers updated to pass these. ~100 LOC delta.

Task modal swap

Replace <EvidenceAccordion> with <LinkedArtifactsPanel entity={{ type: 'task', id: task.id }} ...> in left-panel.tsx. Delete evidence-accordion.tsx + evidence-box.tsx once nothing references them.

Server actions touched

Existing actions uploadFileAndLinkToTask, linkFileToTask, unlinkFile, getFileDownloadUrl all stay. New: a linkDocumentToTask + matching unlink (mirrors the existing list-item versions).

Effort

~6-8 hours. No migration. No new visual primitives. Existing file picker / document picker dialogs reused as-is.

Sequencing

How this lands relative to the file-promotion feature

Why this order is fine

The promotion feature operates at the data layer (server action + dialog modal). The component swap operates at the UI layer (which presentation primitive renders the file list). They're orthogonal — promotion will continue to work after the LinkedArtifactsPanel swap because the data shape FileTaskLink hasn't changed.
