§1

Filter-chip density & inactive state — two systems live in production

Two chip implementations coexist. The active color is actually identical in this theme — both bg-foreground and bg-primary resolve to the same near-white. The real divergence is size (py-1 text-xs vs. py-1.5 text-sm) and how the inactive state behaves (muted-and-subtle vs. full-strength-and-prominent).

decision

Pick the canonical density & inactive treatment — every chip follows

option A Keep canonical (tight + muted inactive)

Migrate Laglistor / Mallar / Krav onto the canonical FilterChip. Inactive chips recede; active chip pops by contrast. Reads more like a filter strip in a dense table.

  • Migration: replace 3 bespoke chip implementations with the canonical primitive.
  • Wins: smaller chips leave more room on the toolbar; inactive state visibly defers to the active one.
  • Risk: existing Laglistor users may perceive the chips as "shrunk."
option B Re-theme canonical to match Laglistor (loose + full-strength)

Update FilterChip primitive: py-1 → py-1.5, text-xs → text-sm, text-muted-foreground → text-foreground on inactive. Then migrate the 3 bespoke implementations to the (re-themed) primitive too.

  • Migration: ~5 line className change in the primitive, then consolidate the 3 bespoke implementations.
  • Wins: matches the gold-standard surface; chips stay legible even when inactive.
  • Risk: takes more horizontal room — narrow toolbars wrap earlier.
My recommendation: Option B. You've named Laglistor the gold standard, and three surfaces (the most-trafficked) already match it. Re-theming the canonical primitive is a five-line className change; consolidating the bespoke implementations comes for free after that. Either way the active color stays the same — the visible difference is density and how the inactive chip behaves.
