Pain points: three different font-weights (medium/semibold), two padding scales (px-2 vs px-2.5), inconsistent borders, two opacity scales (bg-50 vs bg-100), and no shared "what does this colour mean" semantics.

proposed

Single <Badge> with semantic colour scales

Extend components/ui/badge.tsx with a tone prop (semantic colour) that maps to the four states the app actually uses. variant is reserved for visual treatment (soft / solid / outline). One file, one source of truth.

tone × variant matrix — <Badge tone="warning" variant="soft">

Shadcn ships with 4 generic Badge variants. We could just rename them to map onto the domain, but they don't cover warning (amber) which we need for partial compliance and observations, and they collapse "draft / not started" and "secondary information" into the same secondary colour.

Verdict: the semantic tone approach above is closer to how the team already thinks about these states.

§3

Filter chip primitive

Three different chip patterns are in use, all rendering as role="tab". Two reinvent the same idea (filter the current view) with different active styles and sizes.

current

Three patterns in production

A · cycle-list-table.tsx · dark-fill chips with count badges

text-sm · active = high-contrast inverted · count badge inline

B · cycle-findings-tab.tsx · tinted-primary, no counts, smaller

text-xs · active = tinted overlay · no counts · no aria semantics

C · shadcn TabsTrigger (used by Laglistor / Tasks / Styrdokument / Cycle Detail)

shadcn segmented control · text-sm · active = filled tile · used for primary view-switch tabs (not subset filters)

proposed

Split semantics: Tabs vs FilterChip

Tabs are for switching views (Laglistor / Ändringar, Items / Findings / Rapport). Filter chips are for narrowing data within a view. Today both are rendered as role="tab" and that conflates them. The proposal keeps shadcn Tabs for view-switching and introduces a single <FilterChip> for everything else.

View switcher — keep shadcn Tabs

semantic <Tabs> · radio group of mutually exclusive views

Filter — new <FilterChip>

<FilterChip pressed count icon> · semantic toggle group · count badge optional · text-xs by default

hovering an inactive chip lifts to text-100 — no jarring colour change

Why the split: a screen-reader user landing on Findings tab today hears "tab Avvikelse", then activating it doesn't navigate — confusing. With the split, view tabs stay as tabs, filter chips announce as toggle buttons.

proposed

<PageHeader> + <TableToolbar>

Two thin compositional primitives, both with named slots. PageHeader owns title + subtitle + meta + primary action. TableToolbar owns view tabs (when present), search, filter chips/dropdowns, and a right-side affordance like "stats" or a secondary action.

API
<PageHeader
  breadcrumbs={[...]}
  title="Test S11"
  badge={<Badge tone="info">Pågående</Badge>}
  meta={<PageHeader.Meta items={[
    "Er laglista",
    "Intern revision",
    "22 apr — 25 apr 2026",
    {icon: <Avatar/>, label: "Alexander Adstedt"},
  ]}/>}
  stats={[
    {label: "Bedömda", value: "1 av 20"},
    {label: "Signerade", value: "0 av 20"},
  ]}
  primaryAction={<Button>Åtgärder ▾</Button>}
/>

<TableToolbar
  views={<Tabs value={view} onChange={setView}>...</Tabs>}
  search={<SearchInput placeholder="Sök items..."/>}
  filters={<>
    <FilterChip pressed={status==='oppen'} count={3}>Öppna</FilterChip>
    <FilterChip pressed={status==='stangd'} count={1}>Stängda</FilterChip>
  </>}
  rightSlot={<Button size="sm">+ Lägg till finding</Button>}
/>

decisions needed

Decision summary

  1. 1

    Adopt the tone-based Badge primitive?

    Adds neutral / info / success / warning / danger as semantic tones with soft / solid / outline variants. Migrates 4 hand-rolled pill implementations to the shared components/ui/badge.tsx.

  2. 2

    Priority labels & colours: option A (Hög·Medel·Låg in rose / amber / slate)?

    Recommended. Decide between A and B (B uses emerald for Låg, which conflicts with success states). Verify whether priority-editor.tsx is actually shared between Laglistor and Uppgifter — the divergence suggests it isn't.

  3. 3

    Split semantics: keep shadcn Tabs for views, introduce FilterChip for in-view filters?

    Replaces the two reinvented chip implementations with one. Tabs stay as role="tab"; filter chips become toggle buttons (aria-pressed).

  4. 4

    Adopt <PageHeader> + <TableToolbar> as the canonical pair?

    Forces a single slot order across Laglistor / Cycle detail / Tasks / Styrdokument. Cycle detail keeps its meta + stats via dedicated PageHeader props rather than ad-hoc ·-separated text.

  5. 5

    Defer: migrate Cycle Items + Styrdokument to shadcn <Table>?

    Larger refactor — would unlock sort, drag, selection, accessibility uniformly. Plan as separate stories once 1–4 land.

  6. 6

    Defer: per-surface empty-state pass.

    Out of scope for this audit; flagged as a follow-up after primitives land.

Suggested rollout order

  1. Land §1 (Badge primitive) and §2 (Priority alignment) together — shared colour tokens, low blast radius, immediate visual payoff.
  2. Land §3 (FilterChip) — touches two files (cycle-list-table, cycle-findings-tab) plus any future filter chips.
  3. Land §4 (PageHeader + TableToolbar) — biggest refactor; touches all six surfaces. Do incrementally, surface-by-surface.
  4. Revisit §5 once §4 settles — easier once the toolbar is no longer competing for layout decisions.
