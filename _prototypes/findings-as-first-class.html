Epic 21 · Lagefterlevnadskontroll

Phase 3 · Findings as first-class objects

Cycles identify · Registry resolves

Strict separation: kontroller surface and create findings (the audit act of identifying gaps); the new /anmärkningar workspace surface owns all lifecycle actions (the operational act of resolving them). One FindingModal invoked from both entry points — same UX everywhere.

1 — Architecture

Two surfaces, one modal — single source of truth for actions

Cycle (kontroll)

Identification surface

  • ✓ List findings (read-only badges)
  • ✓ Create "Lägg till anmärkning"
  • ✓ Click row → open FindingModal
  • ✗ Inline state actions
  • ✗ Verify / dismiss / reopen

The cycle's job: capture what was found at audit time. Locks cleanly at AVSLUTAD without unfinished-business friction — because there's no "business" to finish here.

→

FindingModal

Single shared component

  • Markera som åtgärdat
  • Avskriv (manual)
  • Återöppna
  • Skapa åtgärdsuppgift
  • Tilldela ansvarig
  • Kommentera
  • Redigera fält
←

/anmärkningar (registry)

Resolution surface

  • ✓ Cross-cycle table view
  • ✓ Filter / sort / segment
  • ✓ Click row → open FindingModal
  • ✓ All lifecycle actions
  • ✓ Bulk operations

The registry's job: ongoing remediation. Independent of any cycle's lifecycle. Open findings from cycle Q1 can resolve in Q3 without touching either cycle's snapshot.

Why this separation matters

  • One FindingModal means one place to test, one place to evolve action UX, no surface-drift.
  • AVSLUTAD/SEAL no longer asks "are findings editable?" — they were never edited from the cycle in the first place. Cycle locks normally; findings continue on their own clock.
  • Board-level questions ("how many findings open?", "recurring across cycles?") are native registry queries, not cross-cycle aggregation work.

6 — How the modal opens

One modal · two entry points · in-place overlay

Path A · From cycle row

/laglistor/kontroller/[id]

User on cycle's Anmärkningar tab

↓ click row

FindingModal opens (in-place overlay)

URL becomes ?finding=<id> for shareability

Same modal component as Path B

↓ close modal

/laglistor/kontroller/[id]

Back on cycle, row reflects updated badge

Path B · From registry row

/anmärkningar

User on workspace registry

↓ click row

FindingModal opens (in-place overlay)

URL becomes ?finding=<id>

Same modal component as Path A

↓ close modal

/anmärkningar

Back on registry, row reflects updated badge

Implementation note

FindingModal lives in components/features/anmärkningar/finding-modal/ — imported by both the cycle's findings-tab and the registry page. State (open finding id) lives at the page level via URL query param so deep-links work. Modal-on-modal behavior (clicking "Öppna källkontroll" from inside the modal) follows the existing law-list-item modal precedent.

7 — Phase split (revised)

What ships when

Phase 1 · DONE

Reframing copy

  • ✓ "Markera som åtgärdat" rename
  • ✓ AVSLUTAD advisory banner
  • ✓ Forward-looking dialog copy
  • ✓ Verify-dialog task link

Phase 2 · narrowed

Foundation refactor (~3 days)

  • + Schema columns (verification_note, close_reason)
  • + Server: drop AVSLUTAD from finding guard
  • + State derivation: 5-state in finding-copy.ts
  • + Cycle findings-tab: strip inline actions
  • + Cycle row click → opens FindingModal stub
  • – Inline action handlers DELETED

Phase 3 · the win (~1.5 weeks)

Findings as first-class

  • + /anmärkningar registry page
  • + FindingModal (full lifecycle UX)
  • + Recurring-finding signal (↻ pill)
  • + Quick segments / saved filters
  • + Law-list-item history accordion
  • + Cycle-item "Tidigare anmärkningar"

Phase 2 narrows because the MarkResolvedDialog work moves into Phase 3 (it's part of FindingModal, not a cycle-tab dialog). Phase 2 ships the schema + server unlock + cycle-tab-strip; Phase 3 builds the actual operational surface. After Phase 3, Laglig has a categorically different product than Notisum/Lextre.
