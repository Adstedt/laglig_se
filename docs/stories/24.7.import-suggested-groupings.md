# Story 24.7: Import — LLM-suggested groupings on the granska page

## Status

Ready for Done

## Story

**As a** workspace member who just reviewed a matched import on the granska page (Story 24.4),
**I want** Laglig to propose a sensible grouping of the accepted rows — derived first from the source spreadsheet's `Område` column verbatim, with an LLM clustering pass over anything still uncategorised — and let me edit the proposal before I commit,
**so that** my new laglista lands already organised by topic instead of as one long flat list, without forcing me to re-categorise rows by hand after the fact.

This is a **review-surface enhancement + commit-flow extension** — no new routes, no new schema. It mounts a "Föreslagna grupper" panel in `<ImportReviewPage>` (above the commit dialog), runs a 2-tier rules-first / LLM-fallback proposer when the user opens the panel (size-gated at ≥15 committable rows), and threads an optional `groupAssignments` payload through `commitImport` so the existing `LawListGroup` model picks up the suggested groups at commit time.

## Acceptance Criteria

### Size gate + entry point

1. **Size gate** — the suggestion panel is offered only when the import has **≥ 15 rows in `[ACCEPTED_BY_USER, REPLACED_BY_USER]`** at the moment the user opens the commit dialog (i.e. the rows that will actually become `LawListItem`s). Below that threshold, the panel is suppressed entirely and the commit flow runs unchanged from Story 24.4 — no "Föreslagna grupper" CTA, no LLM call, list lands flat.

2. **Entry point** — when the gate is met, the existing commit dialog grows a single `<Button variant="outline">` labelled **"Föreslå grupper"** placed above the "Bekräfta" CTA. Click → opens the suggestion panel (an inline expansion of the commit dialog — preferred over a second dialog so the user stays in one decision surface). Until clicked, no LLM call is made and `groupAssignments` stays `undefined` on commit (= flat list, current behaviour).

3. **Skip-and-commit is always allowed** — the suggestion panel never blocks the commit flow. A "Hoppa över" affordance returns the user to the plain commit dialog with `groupAssignments = undefined`. The entry-point button itself is dismissable. **Row decisions (Acceptera / Avvisa / Byt matchning) cannot be changed while the commit dialog is open** — the dialog overlays the row table; this avoids panel-state desync against the row set.

### Hybrid proposer — `proposeGroupings(importId)` server action

4. **New server action `proposeGroupings(importId: string)`** in `app/actions/law-list-import.ts`. Read-only — does NOT mutate any DB state. Returns:
   ```ts
   {
     success: true,
     data: {
       groups: Array<{
         name: string
         rowIds: string[]    // LawListImportRow.id values
         source: 'omrade' | 'llm'
       }>
       unassigned: string[]  // rowIds that landed in none of the proposed groups
       generatedAt: string   // ISO timestamp
       llmUsed: boolean      // true if Tier 2 ran successfully (telemetry / cost transparency)
     }
   }
   ```
   Authorisation: `withWorkspace(cb, 'tasks:edit')` — same scope as `commitImport` (proposing groups for a list you can't commit makes no sense). Status guard: import must be in `AWAITING_REVIEW` and have ≥ 15 committable rows; otherwise return `{success: false, error: 'GATE_NOT_MET'}` so the client can render a fallback message.

5. **Tier 1 — `source_omrade` direct grouping (deterministic, free).** For every committable row where `source_omrade` is non-null and non-empty after `.trim()`, group by the trimmed value. Group name = the trimmed `source_omrade` string verbatim (case-preserving — "Arbetsmiljö" ≠ "arbetsmiljö" ≠ "ARBETSMILJÖ" stays distinct so the user sees what they actually wrote). Rationale: most curated import lists already have an `Område` column populated by a consultant; respect that signal as authoritative.

6. **Tier 1 collapse rule** — if Tier 1 would produce a group with **only 1 row**, demote that row to "unmapped" so it falls through to Tier 2. Reason: a singleton group named after a one-off Område ("Krishanteringsplan 2024") is noise, not signal. Threshold is hardcoded as `MIN_TIER1_GROUP_SIZE = 2`.

7. **Tier 2 — LLM clustering on the residual rows.** Runs whenever Tier 1 leaves at least one unassigned row (no further size gate — the parent `≥15 committable rows` gate at AC 1 already bounds the cost). Rows with no matched document (shouldn't exist in committable rows, but defensively) are dropped from the proposal entirely.

   - **Model:** `claude-haiku-4-5` via `@ai-sdk/anthropic` + `generateObject` from the `ai` SDK. Pattern matches `lib/company-preview/company-analyzer.ts` for SDK integration.
   - **Temperature:** `0.1` — clustering is a low-creativity task; we want consistency across runs of the same input.
   - **Input payload to the model:** `{ rowId, title, document_number, content_type }` for each unassigned row — strictly the metadata the model needs to cluster, no full-text or HTML.
   - **Output schema (Zod):** `{ clusters: Array<{ name: string, rowIds: string[] }> }` where each cluster name is a Swedish noun phrase (≤ 30 chars) and rowIds is a non-empty subset of the input set.
   - **System prompt (Swedish), with embedded priors as natural-language examples:**
     ```
     Du grupperar svenska lagar och författningar efter ämnesområde. Returnera 2–8 grupper med beskrivande svenska namn. Varje rad ska tillhöra exakt en grupp. Slå inte ihop tydligt åtskilda områden bara för att jämna ut storlekar.

     Exempel på rimliga gruppnamn för vanliga författningsserier:
     - AFS-serien (Arbetsmiljöverket) → "Arbetsmiljö"
     - MSBFS (Myndigheten för samhällsskydd och beredskap) → "Brandskydd & krisberedskap"
     - NFS (Naturvårdsverket) → "Miljö"
     - BFS (Boverket) → "Bygg & boende"
     - SOSFS / HSLF-FS (hälso- och sjukvårdsmyndigheter) → "Hälso- och sjukvård"

     Använd dessa som vägledning men bestäm gruppnamn baserat på det faktiska innehållet — exempel kan ha undantag.
     ```
     The examples are **prompt content, not a code lookup table** — they are easy to extend (add a line to the prompt) without code changes. New agency series ship with no Laglig change.
   - **Hard cap: max 8 LLM-proposed clusters,** validated post-response. If the model returns >8, take the 8 largest by row count and append the rest to `unassigned[]`.
   - **Hard min: each LLM cluster must have ≥ 2 rows.** Singletons are rejected and pushed to `unassigned[]` (mirrors AC 6).
   - **All-singletons edge case:** if the LLM returns clusters that, after the singleton filter, leave zero non-empty groups, the proposer returns `groups = [Tier 1 results only]` and `unassigned = [all Tier 2 input rows]`. The panel still renders cleanly — Tier 1 groups visible, Övriga bucket holds the residuals. Do NOT render a "no clusters" empty state.

8. **LLM-call guards — degraded fallback on any failure mode.** The Tier 2 call has a 15-second `AbortController` timeout. **On any of the following, swallow the error, log via `console.warn` with prefix `[GROUPINGS_LLM_FAIL]`, and return Tier 1 results only with `llmUsed: false`:**
   - Timeout (15s elapsed)
   - Rate-limit (HTTP 429 from Anthropic)
   - Schema-violation (Zod rejects the response)
   - Network / 5xx errors
   - Any other unhandled exception from the SDK

   The proposer never throws to the caller. Degraded grouping (Tier 1 only) is always better than no grouping. Additionally:
   - **Abort on dialog close.** Client-side: when the commit dialog unmounts mid-call, the `AbortController` fires from the React `useEffect` cleanup. Server-side request keeps running but the client discards the result.
   - **Telemetry:** write a `ChatUsageEvent` row (per Story 14.27 conventions) with `context_type='import_grouping'`, the prompt + completion + cache token counts, and `cost_usd_estimate` from `lib/usage/cost-estimator.ts`. Failures-to-write follow the `[CHAT_USAGE_EVENT_WRITE_FAIL]` pattern from 14.27 — caught + logged, never re-thrown.
   - **Activity log:** on every successful return (including degraded Tier-1-only returns), emit `law_list_import.groupings_proposed` with payload `{importId, tier1Count, tier2Count, unassignedCount, llmUsed, llmDurationMs?, llmFailureReason?}`. `llmFailureReason` is one of `'timeout' | 'rate_limit' | 'schema_violation' | 'network' | 'unknown'` when `llmUsed: false`, else omitted.
   - **No streaming.** Single `generateObject` call; the panel UI renders a `Loader2` spinner (≤ 15s).

9. **Idempotency.** The user can click "Föreslå grupper" multiple times. Each call re-runs from scratch — there is no cache. Rationale: row decisions can change between import sessions, and the LLM step is cheap enough at Haiku rates (sub-cent per call) that caching adds invalidation complexity without meaningful savings. If proven hot in production telemetry, a follow-up story can add SWR-keyed deduping. **Anthropic-side prompt caching** on the system prompt (Story 14.26 pattern) IS still in scope — the system prompt is stable across all calls and benefits from cross-import cache hits at the provider level; this is a transport-layer optimisation, not a Laglig-side cache.

### UI — suggestion panel

10. **Panel mount point.** Inside the existing commit dialog in `components/features/law-list-import/import-review-page.tsx` (search for `commitImport` + `listName` state — the dialog is the JSX block around line 451). When the entry-point button (AC 2) is clicked, the dialog body switches from the plain "summary + name input" view to a "summary + name input + grouping panel" view. The "Bekräfta" CTA stays at the bottom; "Hoppa över grupper" appears next to it as a secondary action.

11. **Editable proposal.** The panel renders one section per proposed group:
    ```
    ┌─ Arbetsmiljö (12 rader) ────────────  [✎ namn]  [✕ ta bort gruppen] ─┐
    │  • AFS 2014:43 Kemiska arbetsmiljörisker                              │
    │  • AFS 2018:1 Hygieniska gränsvärden                                  │
    │  • ...                                                                │
    └────────────────────────────────────────────────────────────────────────┘
    ```
    - Group name is inline-editable (`<input>` with click-to-edit affordance — same pattern as `<EditableTitle>` if such a primitive exists; otherwise a plain `<input>` with `onBlur` save).
    - Removing a group sends its rows to the "Övriga (utan grupp)" bucket below.
    - Each row has a small "Flytta till →" dropdown listing the other proposed groups + "Övriga". No drag-and-drop in v1 (defer to a follow-up story if customer feedback asks for it).
    - Below the proposed groups, an "Övriga (utan grupp)" section lists `unassigned[]` rows. These commit without a group_id (= ungrouped, render under the existing "Övrigt" virtual bucket per Story 4.13).
    - "Lägg till grupp" inline button below the list lets users create a fresh empty group and move rows into it.

12. **Visual signal of source.** Each group section shows a small muted badge next to its name indicating provenance: `Område` (Tier 1) or `AI-förslag` (Tier 2). This is informational only — once the user edits, the badge can stay (showing the original source) or be hidden (implementor's call; preference is to keep it as a non-blocking trust signal). User-created groups (via "Lägg till grupp") get no badge.

13. **State management.** The panel's edited state is held in client React state (`useState`) inside the dialog — not URL-synced (modals don't deep-link). State shape mirrors the server response: `Array<{name, rowIds, source}>` + `unassigned: string[]`. On dialog close without commit, state is discarded (consistent with the existing "edit list name" affordance, which also discards on cancel).

14. **Re-propose affordance.** A small "Föreslå om" link at the top of the panel re-runs `proposeGroupings` and overwrites the local edited state. Confirms via inline confirmation if the user has already made edits ("Detta ersätter dina ändringar. Fortsätt?"). Avoids accidental discards.

### Commit-flow extension

15. **`commitImport` extended payload.** The existing server action signature in `app/actions/law-list-import.ts` (Story 24.4 AC 8) gains an optional `groupAssignments` param:
    ```ts
    commitImport({
      importId: string,
      listName: string,
      groupAssignments?: {
        groups: Array<{ name: string, rowIds: string[] }>
        asSuggested?: boolean   // telemetry hint — true if user committed without edits
      }
    })
    ```
    When `groupAssignments` is `undefined` (the default — preserves Story 24.4 behaviour), the list lands flat. When present, the action additionally creates one `LawListGroup` per entry inside the same transaction and assigns each `LawListItem.group_id` accordingly.

    **There is no `unassigned[]` field in the request payload.** Rows not listed in any `groups[].rowIds` commit with `group_id = null` (ungrouped — falls under the existing "Övrigt" virtual bucket per Story 4.13). The client may track unassigned rows in its local state for UI rendering, but does not send them to the server.

    **`groups: []` (empty array) is treated identically to `groupAssignments = undefined`** — list lands flat, no `LawListGroup` rows created. Defensive coalesce inside `commitImport`.

16. **Group creation rules inside the transaction.**
    - Group `position` is the index in the `groups[]` array (0-based, multiplied by 1 — Float column allows insertion later if needed). The order the user saw in the panel is the order they get on the new list.
    - Group `name` is taken verbatim from the payload — no normalisation, no de-dupe. If the user manually creates two groups with the same name, the existing `@@unique([law_list_id, name])` constraint will reject the transaction; the server action surfaces this as `{success: false, error: 'DUPLICATE_GROUP_NAME', conflict: 'Arbetsmiljö'}` so the client can highlight which group to rename.
    - **Validation rules** (rejecting the entire commit on any violation):
      - `INVALID_ROW_REFERENCE` — every `rowId` in `groups[].rowIds` must reference a row in `[ACCEPTED_BY_USER, REPLACED_BY_USER]` for **this** import. Foreign rowIds (rejected rows, catalog-request rows, rows from a different import) → reject.
      - `DUPLICATE_ROW_ASSIGNMENT` — the same `rowId` appearing in two different `groups[].rowIds` arrays → reject (the validation message includes the offending rowId).
      - `EMPTY_GROUP_NAME` — group `name` after `.trim()` must be non-empty → reject.
    - There is **no coverage check** — committable rows that aren't in any `groups[].rowIds` simply commit ungrouped. That's the intended ungrouping path.

17. **Activity log on commit with groups.** Existing `law_list_import.committed` payload extended (back-compat — additive fields, all optional):
    ```ts
    { rowsAdded, rowsRequested, rowsRejected, lawListId,
      groupsCreated?: number,           // new
      groupingSource?: 'user-edited' | 'as-suggested' | 'flat'  // new — flat = no groupAssignments
    }
    ```
    `groupingSource` is best-effort. It's `'flat'` when `groupAssignments === undefined` or `groupAssignments.groups.length === 0`. It's `'as-suggested'` when the client sets `asSuggested: true` (signalling no edits were made). Otherwise `'user-edited'`. The server trusts the flag for telemetry only.

18. **Idempotency preserved.** `commitImport` already returns the existing `lawListId` on second call (Story 24.4 AC 17). With grouping: a second call with `groupAssignments` after a first commit is a no-op for the grouping payload — the existing list and its groups are returned unchanged. (Implementation: the existing status-guard `import.status !== AWAITING_REVIEW` short-circuit in `commitImport` already handles this; integration test asserts groups are NOT recreated.)

### Compatibility & out-of-scope guards

25. **No schema changes.** `LawListGroup` already has everything needed (id, law_list_id, name, position). `LawListItem.group_id` already nullable per Story 4.13.

26. **No changes to Stories 24.1–24.6 server actions** beyond the additive `groupAssignments` param on `commitImport`. All existing call sites pass `undefined` and behave identically. Existing 24.4 integration tests must pass unchanged.

27. **Onboarding-modal import path (Epic 25 B.1) inherits this for free.** Both surfaces commit through the same `commitImport` action and both render `<ImportReviewPage>`. No coordination work required.

28. **Does NOT touch `/laglistor/{id}` post-commit.** Once the list is committed, it's a normal `LawList` and groups behave per existing Story 4.13 logic. Group editing post-commit uses the existing `<GroupManager>` UI.

29. **No drag-and-drop in v1.** "Flytta till →" dropdowns only. DnD is a UX nice-to-have that doubles the test surface; defer to a follow-up if customer feedback asks for it.

30. **No multi-list grouping.** Each import → one new list → its own groups. Reusing groups across lists is not in scope (and not supported by the schema — `LawListGroup.law_list_id` is required).

**Key decisions baked in:**
- **Pure LLM clustering (no hardcoded agency-prefix table) over a hybrid rules+LLM design.** v0.1 of this story had a Tier 2 with a hardcoded prefix → group-name lookup table (AFS → Arbetsmiljö, etc.). v0.2 dropped it for two reasons: (1) the table was a maintenance smell — Sweden has many regulatory series and the count grows over time; every new series would require code + tests + PR for a zero-judgement mapping; (2) Haiku already knows Swedish regulatory taxonomy and clusters AFS rows correctly without any hint. The 3–5 example mappings in the system prompt (AC 7) carry the priors as natural-language examples — easy to extend by editing the prompt, no code change required. Tier 1 (`source_omrade` verbatim) is preserved because it represents the user's own intent, which is always more authoritative than any LLM judgement.
- **Size gate at 15 rows.** Below that, grouping adds friction without payoff (eyeballing 10 rows is faster than reviewing a proposal). The threshold is a soft choice — if customer feedback says "I wish it grouped my 12-row list too", drop to 10. Hardcoded as a constant for easy adjustment.
- **Editable proposal, not auto-applied.** User is always in control. The proposal is a starting point, not a decision. This is the same UX pattern as `<ImportRowCard>` — Laglig proposes, user ratifies.
- **No client-side caching of LLM results across calls.** Cheap enough at Haiku rates; row decisions can change between proposal clicks; caching adds invalidation complexity for negligible savings. Anthropic-side prompt caching on the stable system prompt is a separate transport-layer optimisation that is in scope.
- **`source_omrade` group names preserved verbatim (case + whitespace).** If the user wrote "ARBETSMILJÖ" we don't normalise to "Arbetsmiljö" — they'll see what they wrote, which is the right trust signal. Casing inconsistencies are theirs to fix in the editing UI.
- **No drag-and-drop in v1.** Doubles test surface for marginal UX gain. Dropdown reassignment is enough for the editing affordances we expose.
- **Row decisions are frozen while the commit dialog is open.** The dialog overlays the row table; users physically cannot change row state mid-grouping. This avoids a panel-state-vs-row-set desync class of bugs without explicit reconciliation logic.

**Out of scope explicitly:**
- Drag-and-drop row reassignment (defer to follow-up if asked)
- Group reuse across multiple lists (schema doesn't support it)
- Client-side caching / deduping `proposeGroupings` calls (premature optimisation)
- Changes to post-commit grouping UI (`<GroupManager>` already handles edits)
- Onboarding-modal-specific styling (B.1 inherits the same dialog)
- Any code-level agency-prefix mapping table (deliberate v0.2 design — priors live in the LLM system prompt instead)

- **File**: `app/actions/law-list-import.ts`
