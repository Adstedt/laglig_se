## Story

**As a** user reviewing a proposed section edit,
**I want** to tweak the proposed text directly in the approval card,
**so that** small corrections ("1 → 2 gånger per år", a typo, a date) don't require a chat round-trip that re-reads the whole document and risks a duplicate proposal.

## Context

When the user wants a minor change to a *pending* `update_document` proposal, the only path today is to ask the agent in chat. The agent re-runs `list_styrdokument` + `get_workspace_document` and proposes again — slow, and it produces a second card for the same section (confusing; the consolidation fold then applies both). For trivial edits this is overkill: the proposed content already lives in `PendingAgentAction.params.newSectionContentJson`, and editing it in place needs no model call.

The CREATE_TASK / obligation renderers already prove the pattern (editable fields → debounced `updatePendingActionParams`). This story brings the same affordance to the proposed **section body** (and the optional `newTitle`).

**Scope decision — plaintext, not rich.** The proposed body is Tiptap JSON (`TiptapNode[]`). Full rich inline editing is out of scope. v1 offers a **plaintext** textarea seeded from the proposal's plain text; on save it re-wraps into paragraph nodes (one node per blank-line-separated block). This is **lossy for lists / tables / nested structure** — so the edit affordance is gated to "simple" proposals (body is only paragraphs) and otherwise falls back to "ask the agent / open in editor". `newTitle` is a plain string and always editable.

## Acceptance Criteria

1. In the `update_document` proposal card (PENDING only), the "Justera" disclosure exposes an **editable plaintext textarea** seeded from the proposal's proposed section body (`newSectionContentJson` → plain text), plus an editable `new_title` input when the proposal carries a rename.
2. Editing debounces 500ms → `updatePendingActionParams(id, params)` (reuse `useDebouncedParamsChange`), writing back `newSectionContentJson` (re-wrapped to paragraph `TiptapNode[]`) and/or `newTitle`. The colored diff re-renders against the edited proposed text.
3. **Lossy-edit guard:** the textarea is only offered when the proposed body is paragraphs-only. When the body contains lists/tables/headings/other block types, show a read-only diff with a hint ("Öppna i editor för att finjustera" / ask the agent) instead of a textarea — never silently flatten rich content.
4. The edited proposal flows through the existing approve path unchanged: dispatch reads the live document and applies `updateSection` with the edited `newSectionContentJson` (single-approve AND consolidated batch). No dispatch changes needed.
5. **No-op / empty guard:** an empty textarea (cleared body) disables approve for that row with a clear message; it must not persist an empty section.
6. **Non-regression:** rich proposals, rename-only proposals, ADD_DOCUMENT_SECTION proposals, and the consolidated batch approval all behave exactly as before for users who don't edit inline.
