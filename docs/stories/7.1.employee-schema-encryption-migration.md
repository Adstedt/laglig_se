## Dev Notes

> **Self-contained context — do not read architecture docs unless a task explicitly says so.** Everything needed to implement 7.1 is below, with sources.

### ⚠️ Key decision for this story — encryption pattern is NET-NEW

There is **no existing application-level field-encryption pattern** in the codebase. The only sensitive-data precedent is HMAC-SHA256 *signing* (not encryption) in `lib/email/unsubscribe-token.ts` (key from `NEXTAUTH_SECRET`, uses `crypto.timingSafeEqual`). So AC2's "consistent with existing sensitive-field handling" must be read as *"establish the pattern"*, not reuse one. Decision taken for this story (flagged for human validation):

- **Algorithm:** AES-256-GCM (authenticated encryption) via Node `crypto`.
- **Key:** new `ENCRYPTION_KEY` env var, 32 bytes base64. **Introduces a new production secret** — must be set in Vercel before the register writes any personnummer.
- **Storage format:** `base64(iv).base64(authTag).base64(ciphertext)` in the `personnummer` column (a `String?`).
- **Fail-closed:** if the key is missing or malformed, `encryptField` throws — never persist plaintext.
- Model after the key-management + constant-time practices in `lib/email/unsubscribe-token.ts`, but this is new code in `lib/crypto/field-encryption.ts`. [Source: Explore sweep of `lib/email/unsubscribe-token.ts`, `lib/utils/crypto.ts`]

### Data model (authoritative)

The Prisma block for `Employee`, `EmployeeGroup`, and the four enums is specified verbatim in the epic. Copy field names, types, decimal precisions, indexes, and `@@unique` exactly. [Source: `docs/prd/epic-7-hr-module-employee-management-detailed.md#Data Model — Employee`]

- Enum members ARE the Fortnox codes (identity-mappable; UI supplies Swedish labels). Do not rename or add codes. [Source: same, lines 111–117]

### Prisma conventions (this repo)

- Snake_case Prisma field names, table names via model-level `@@map("...")` (e.g. `@@map("law_list_items")`). Enums have no `@@map`. [Source: `prisma/schema.prisma` — `LawListItem` L372–458, `EmployeeCountRange` L748–754]
- Self-relations: see `LawListItem.responsible_user_id → User` (`@relation("LawListItemResponsible")`). Apply the same shape for `Employee.manager_id → Employee` with a named relation. [Source: `prisma/schema.prisma` L372–458]
- `CompanyProfile.has_collective_agreement` / `collective_agreement_name` / `workforce_composition` / `employee_count_range` already exist — do **not** re-add. [Source: `prisma/schema.prisma` L281–328]
- `WorkspaceFile` model already exists (target of `CollectiveAgreement.workspace_file_id`). Keep the FK nullable. The **only** edit to `WorkspaceFile` is the required virtual back-relation field `collective_agreements CollectiveAgreement[]` (Prisma relation completeness) — this adds **no DB column** and no migration change to the `workspace_files` table. [Source: Explore sweep]
- `ContentChunk` / `SourceType` are **not touched** in 7.1 (`COLLECTIVE_AGREEMENT` source type is added in Story 7.5). [Source: `prisma/schema.prisma` L1913–1950]
- **Prisma requires both sides of every relation.** For each FK added here, add the reverse array field on the referenced model: `Employee.manager_id`→`Employee.reports[]`, `Employee.group_id`→`EmployeeGroup.employees[]`, `Employee.collective_agreement_id`→`CollectiveAgreement.employees[]`, `CollectiveAgreement.workspace_file_id`→`WorkspaceFile.collective_agreements[]`. Missing reverse fields fail `prisma validate` before any migration is generated.
- **Audit fields** `created_by` (Employee) and `uploaded_by` (CollectiveAgreement) are plain `String` user ids with **no `User` FK relation** — matches the repo's lightweight audit-field convention and avoids widening the `User` model. Do not add relations for them.
