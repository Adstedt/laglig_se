# Story 21.6: Cycle lifecycle — Complete (PAGAENDE → AVSLUTAD) + revert

## Status

Approved

## Story

**As a** lead auditor,
**I want** to transition a cycle from `PAGAENDE` to `AVSLUTAD` once every item is signed off (and to revert back to `PAGAENDE` if I discover the completion was premature),
**so that** I can declare the audit work done, freeze the "completed" report template, and still retain an escape hatch before the irreversible seal in Story 21.9.

## Acceptance Criteria

### Functional

1. **"Slutför kontroll" action — visibility + affordance.** The `CycleDetailHeader`'s "Åtgärder" button (placeholder today per Story 21.5 `cycle-detail-header.tsx:63`) is replaced with a **functional dropdown menu** containing one live entry in this story: **"Slutför kontroll"**. Entry behaviour:
   - **Visible only** when `cycle.status === 'PAGAENDE'`. Hidden for `PLANERAD | AVSLUTAD | SEALED | ARKIVERAD`. In `PLANERAD` the cycle hasn't been materialised; in `AVSLUTAD` the revert entry applies instead (see AC 6); in `SEALED | ARKIVERAD` the header is already read-only per Story 21.5 AC 8.
   - **Enabled** iff every item in the cycle has `signedOffAt !== null` AND `items.length > 0`. Otherwise **visible-but-blocked** with a tooltip listing the outstanding blockers. Two tooltip variants:
     - **Unsigned items:** `"Slutför kontroll: {n} av {m} poster behöver signeras."` where `n = totalCount - signeradeCount`.
     - **Zero items** (`totalCount === 0`): `"Kontrollen innehåller inga poster att slutföra."` — mirrors the server-side zero-items error string so the UI and server tell the same story.
   - **Radix disabled-item tooltip gotcha (SF-1).** A Radix `<DropdownMenuItem disabled>` suppresses pointer events, so a wrapping `<Tooltip>` never fires. Two acceptable workarounds — pick one and apply it consistently:
     - **Option A (preferred):** render the item **enabled but non-actionable** via `onSelect={(e) => e.preventDefault()}` + apply `data-[disabled]`-style visual muting via Tailwind classes (`aria-disabled`, `cursor-not-allowed`, `opacity-60`). The `<Tooltip>` wrapping then fires correctly on hover. Radix itself permits this pattern.
     - **Option B:** render a native `<DropdownMenuItem disabled>` *wrapped* in a `<span>` that carries the tooltip trigger — the span captures pointer events the disabled item swallows.
     - Pattern at `cycle-detail-header.tsx:189-205` (the progress cluster's tooltip around an `<Tooltip><TooltipTrigger asChild><span>{button}</span></TooltipTrigger></Tooltip>`) is Option B. Follow it for consistency.
   - Counts come from the existing `useCycleItems()` context (`signeradeCount`, `totalCount`) per Story 21.5 `cycle-items-context.tsx:13-19`. **No new fetch** — the context value is already populated by `CycleDetailPage` before the header renders.

2. **Confirmation dialog.** Clicking "Slutför kontroll" opens a confirmation dialog (shadcn `<AlertDialog>`, same primitive as elsewhere in the codebase) with:
   - **Title:** `"Slutför kontrollen?"`.
   - **Body** (Swedish, two base paragraphs + an optional open-work advisory):
     1. `"Kontrollen går från Pågående till Avslutad. Rapportmallen fryses och du kan generera en första revisionsrapport."`
     2. `"Poster och findings förblir redigerbara — du kan fortfarande justera motiveringar eller lägga till findings. Försegling är ett separat, senare steg som inte kan ångras."`
   - **Open-work advisory block (conditional).** If the cycle has any **open findings** OR any **pending corrective-action tasks**, a third muted-styled paragraph (`<p className="text-sm text-muted-foreground">` or similar) acknowledges them explicitly so the user is not surprised:
     - **Both counts > 0:** `"Du har {openFindings} öppna findings med {pendingTasks} pågående åtgärdsuppgifter. Dessa kvarstår oförändrade och kan stängas när åtgärderna är klara — kontrollen behöver inte förseglas förrän allt är avslutat."`
     - **Only openFindings > 0, pendingTasks === 0** (e.g., observations / förbättringsförslag without auto-spawned tasks): `"Du har {openFindings} öppna findings utan aktiva åtgärder. Dessa kvarstår öppna och kan stängas när du är klar."`
     - **openFindings === 0 && pendingTasks === 0:** advisory paragraph is **omitted entirely**; the base two-paragraph body stands alone.
   - **Data sources for the counts:**
     - `openFindings` — reuse the existing `findingCounts.open` value already computed in `CycleDetailPage` at `cycle-detail-page.tsx:278-284` and threaded into `CycleDetailHeader`. Pass the count into `<CompleteCycleDialog>` as a prop. **No new fetch.**
     - `pendingTasks` — derive inline from the existing `findings` array: `findings.filter(f => f.correctiveActionTaskId !== null && f.closedAt === null).length`. **Approximation trade-off:** this counts findings with a linked task that is still open-at-the-finding-level, not the task's actual `status !== 'completed'`. For the dialog-advisory use-case this is good enough — the finding's `closedAt` proxies task state tightly because Story 21.7 AC 5 gates `closeFinding` on the linked task being `completed` (with manual override). A proper count would require an extra DB query; not worth the round-trip for an advisory.
     - Document both data sources inline in the dialog component's JSDoc so a future dev doesn't "fix" the approximation without realising why it's approximate.
   - **Primary button:** `"Slutför kontroll"` (default variant, not destructive — this is a reversible transition pre-seal).
   - **Cancel button:** `"Avbryt"`.
   - Primary button shows a spinner during submission; both buttons disabled mid-request. On success, the dialog closes and a success toast fires (`"Kontrollen är avslutad"`).
   - **Rationale — why surface, not block.** The epic's Complete gate is procedural: all items signed off. Åtgärdsuppföljning is the remediation phase which by design spans weeks; blocking Complete on åtgärd-closure would make Complete unusable for the real workflow. The protective gate for "evidence is correct and immutable" is **Seal** (Story 21.9), not Complete. This advisory makes the handoff visible without making it a blocker.

3. **Server action — `completeCycle(cycleId)`.** New export in **`app/actions/compliance-audit-cycle.ts`** (NOT `compliance-audit-item.ts` — cycle lifecycle belongs with the cycle module; the architecture §5.1 signature `completeCycle` is listed there). Contract:
   - **Zod input:** `{ cycleId: string().uuid() }` (reuses `CompleteCycleSchema` defined inline at file top).
   - **Auth:** `withWorkspace(cb, 'tasks:edit')` — same bar as `updateCycleMetadata` / `updateItemBedomning`. Story 21.14 AC 3 established that cycle mutations sit on `tasks:edit`; Complete is a cycle mutation.
   - **Load step:** `loadCycleScopedToWorkspace(cycleId, ctx.workspaceId)` (existing helper at `compliance-audit-cycle.ts:178-185`). Returns `'Kontrollen hittades inte'` if not found / cross-workspace.
   - **Status guard:** requires `existing.status === 'PAGAENDE'`; any other state → structured error `"Kontrollen kan bara slutföras från status Pågående"`. `// TODO(21.10): assertCycleEditable(tx, cycleId)` marker at callback entry (matches the convention at `compliance-audit-cycle.ts:458, 578`).
   - **Items-signed guard:** single Prisma `count` query: `prisma.complianceAuditItem.count({ where: { cycle_id, signed_off_at: null } })`. If `unsignedCount > 0` OR `totalCount === 0`, return structured error. On zero items: `"Kontrollen innehåller inga poster att slutföra"`. On unsigned: `"{n} av {m} poster är inte signerade"`. Paired with an explicit second count query for `totalCount` so the error message is informative (both queries run in parallel via `Promise.all`).
   - **Transition:** `prisma.complianceAuditCycle.update({ where: { id }, data: { status: 'AVSLUTAD' } })`. No other fields touched; `sealed_at` / `sealed_by_user_id` / `seal_hash` remain null. No transaction needed (single mutation).
   - **ActivityLog:** one row via `logActivity(ctx.workspaceId, ctx.userId, 'compliance_audit_cycle', cycleId, 'cycle_completed', { status: 'PAGAENDE' }, { status: 'AVSLUTAD', completedAt: <ISO> })`. The `completedAt` in `newValue` is the update's timestamp (use `new Date().toISOString()` captured immediately before the update).
   - **Revalidate:** `revalidatePath('/laglistor/kontroller')` (list route) + `revalidatePath('/laglistor/kontroller/' + cycleId)` (detail route — matches Story 21.5's cycle-specific revalidation convention).
   - **Returns:** `{ success: true, data: { cycle: CycleDetail } }` — the refreshed cycle (via a final `getCycleById` inside the same workspace context), so the client can replace the cached cycle in-place without a refetch. On any failure: `{ success: false, error: <string> }`.
   - **Outer try/catch:** mirrors the convention at `compliance-audit-cycle.ts:265-268` — `console.error('completeCycle error:', error); return { success: false, error: 'Kunde inte slutföra kontrollen' }`.

     - `PAGAENDE` → unchanged placeholder text.
     - `AVSLUTAD | SEALED | ARKIVERAD` → a neutral "first report" placeholder card with the copy `"Rapport redo för generering. HTML-renderer hanteras i Story 21.11, PDF i Story 21.12."` + a disabled `"Ladda ner"` button (tooltip: `"Aktiveras när Story 21.11 + 21.12 är levererade."`).
   - **NOT in-scope for 21.6:** actually calling any renderer. No `generateCycleReport` invocation, no `lib/compliance-audit/revisionsrapport-renderer.ts` wiring. That work belongs to 21.11 / 21.12. The acceptance bar for this AC is: the user visually observes that the Rapport tab acknowledges the cycle's new state.
   - `CycleDetailPage` already receives `cycle.status` — the Rapport tab's content becomes a thin status-aware branch, no new data-fetch.

5. **Progress cluster + header — post-completion state.** After the cycle moves to `AVSLUTAD`:
   - The **status badge** in the header flips from `Pågående` (blue) to `Avslutad` (amber) per Story 21.5 AC 3's `CycleStatusBadge` palette (already implemented).
   - The **progress cluster** (`"N av M bedömda"` / `"N av M signerade"`) remains visible. Since all items are signed off at this point, both metrics render as **"M av M"** with the "all signed"/"all bedömd" tooltips already in place (`cycle-detail-header.tsx:188-207`). No new UI work on the cluster.
   - The **"Åtgärder" dropdown** re-renders with the revert entry visible (AC 6) and the "Slutför kontroll" entry hidden.
   - Items tab remains **editable** — AC 8 of Story 21.5's read-only gate only fires for `SEALED | ARKIVERAD`. `AVSLUTAD` is NOT read-only. This is the intentional "completed-but-still-editable" window per the epic's AC 2 body.

6. **Revert transition — `revertCycleToPagaende(cycleId)`.** New export in **`app/actions/compliance-audit-cycle.ts`**. Contract:
   - **Zod input:** `{ cycleId: string().uuid() }`.
   - **Auth:** `withWorkspace(cb, 'tasks:edit')` **PLUS** a runtime check — `canCompleteOrRevertCycle(role, userId, cycleId, workspaceId)`. The epic's AC 6 says "by the lead auditor if not yet sealed". The runtime helper resolves true if `canSealAuditCycle(role)` (OWNER/ADMIN via the existing helper at `lib/auth/permissions.ts:191-192`) OR the user is the cycle's `lead_auditor_user_id` (runtime lookup mirroring `lib/compliance-audit/authorization.ts#isLeadAuditor`). This matches the seal-action pattern from Story 21.14 — cheapest source of truth + consistent with the authorization module.
   - **New helper** lives at `lib/compliance-audit/authorization.ts` as `canCompleteOrRevertCycle({ role, userId, cycleId, workspaceId }, prismaClient?): Promise<boolean>`. Short-circuits on role; falls through to `isLeadAuditor` on the workspace's active cycle (re-uses the existing helper pattern). Keep the same `PrismaLike` parameter shape — accept both top-level prisma + transaction client.
   - **Status guard:** requires `existing.status === 'AVSLUTAD'`. Any other state → structured error `"Endast avslutade kontroller kan återställas till Pågående"`. `sealed_at !== null` is redundant (SEALED would have failed the status guard already), but add a defensive `existing.sealed_at === null` check anyway so future logic can't slip past.
   - **Transition:** `prisma.complianceAuditCycle.update({ where: { id }, data: { status: 'PAGAENDE' } })` — status-only update; items' `signed_off_at`/`signed_off_by_user_id` are **NOT touched** (epic AC 6: "soft revert: status update only, keeps item state").
   - **ActivityLog:** one row via `logActivity(workspace, user, 'compliance_audit_cycle', cycleId, 'cycle_reverted_to_pagaende', { status: 'AVSLUTAD' }, { status: 'PAGAENDE' })`.
   - **Revalidate:** same two paths as completeCycle.
   - **Returns:** `{ success: true, data: { cycle: CycleDetail } }` via the same post-update `getCycleById`.
   - **Permission-denied error:** `"Endast revisionsledaren eller administratörer kan återställa kontrollen"` (Swedish) — returned when `tasks:edit` passes but `canCompleteOrRevertCycle` is false (e.g., a MEMBER who is not the lead auditor). **Wording must match the UI tooltip in AC 7 verbatim** — the authorized set is OWNER/ADMIN (via `canSealAuditCycle`) + lead auditor (via `isLeadAuditor`). Earlier draft said "Endast revisionsledaren" which was technically inaccurate; the corrected copy is truthful about OWNER/ADMIN privilege.

### Authorization shape — why "tasks:edit + runtime helper" rather than a new scope

The epic says Complete is for "a lead auditor" (permissive-sounding) and revert is "by the lead auditor" (tighter). Story 21.14 resolved the analogous seal question by keeping the permission scope (`audit:seal`) role-based and layering a runtime `isLeadAuditor` override so the cycle's lead — even if a MEMBER — can seal.

For 21.6, the same shape applies:

- **`completeCycle`:** role check `tasks:edit` (anyone who can edit bedömning can complete — same bar as the preceding sign-off work). No runtime override needed; if you can sign off the final item, you should be able to complete the cycle. Revert is the "protective" direction, not Complete.
- **`revertCycleToPagaende`:** `tasks:edit` + runtime `canCompleteOrRevertCycle` which short-circuits on `canSealAuditCycle(role)` (OWNER + ADMIN) and falls through to `isLeadAuditor`. This makes the revert a "lead auditor OR privileged role" gate, consistent with the epic's explicit wording and the seal pattern.

**Not introducing `audit:manage-lifecycle` as a new permission scope** is a deliberate simplicity choice — the permission matrix stays as close to 21.14's final shape as possible; runtime helpers express the narrower per-cycle authorization.

### State reconciliation — why `localCycle` state in `CycleDetailPage`

`CycleDetailPage` receives `cycle: CycleDetail` as a prop from the RSC. Today it passes `cycle` directly to child components. After Complete or Revert, the status must update without a full page navigation:

- **Option A** — call `router.refresh()` after the mutation. This triggers a full RSC re-fetch. Works; adds a round-trip that's unnecessary because the mutation already returned the updated cycle.
- **Option B** — mirror the existing items pattern: maintain local state seeded from the prop, update it in-place from the mutation's return value. Zero extra round-trips.

Option B matches the existing `replaceRow` pattern for items (`cycle-detail-page.tsx:136-151`) — the mutation returns the authoritative row, the orchestrator writes it into its local cache. Same discipline for cycles: `setLocalCycle(result.data.cycle)` on success.

One complication: the prop's `cycle.scopeDefinition` can be large (item ids array). On revert the server re-reads this. If this becomes a perf hotspot, shrink the returned cycle to a `CycleLifecyclePartial` (id + status + updated_at + sealHash) — but for MVP the full `CycleDetail` return keeps things simple.

### Carry-forward notes for future Epic-21 stories (NOT this story's concern)

- **Story 21.9 (Seal) — open design question flagged for PO.** Seal requires `cycle.status === 'AVSLUTAD'`. This story is the gate that produces that state. The seal action's confirmation dialog should link back to this story's revert affordance ("if you discover an error before sealing, use 'Återställ till Pågående'").

  **Unresolved design question for 21.9 — needs PO decision before drafting:** *Should `sealCycle` be blocked when the cycle has open findings and/or pending corrective-action tasks?* Complete (this story) is deliberately permissive — it treats åtgärdsuppföljning as a multi-week remediation phase that runs in parallel with a "completed" cycle. Seal is the opposite (irreversible tamper-evident freeze), and sealing with open AVVIKELSE-driven åtgärder outstanding is at minimum unusual and potentially misleading to an external certifieringsorgan.

  Three candidate policies for 21.9:

  1. **Permissive (symmetric with Complete):** allow Seal regardless of finding state. The rapport renders open findings as "pågående ärenden" in the sealed PDF. Most flexible, least protective.
  2. **Gate with manual override:** block Seal if any `type: 'AVVIKELSE'` finding has `closedAt === null`, with an optional `seal_override_reason: string` escape hatch that's logged to ActivityLog and renders in the sealed PDF's cover page. Mirrors Story 21.7's `close_reason` pattern. **My recommendation.**
  3. **Hard gate:** block Seal if any open finding of any type. Maximalist; may force teams to "close with notes" findings that should genuinely remain open.

  Leaving the decision to Story 21.9 drafting. No action in 21.6 — but 21.6's Complete-dialog advisory (AC 2) already prepares users for this: "Kontrollen behöver inte förseglas förrän allt är avslutat" tells them Seal is the point where open åtgärder matter, not Complete.
