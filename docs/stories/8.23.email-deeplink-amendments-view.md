# Story 8.23: Amendment Notification Deep-Links Open Amendments List

## Status

Ready for Review

## Story

**As a** user who clicks an amendment notification — whether from the email digest or the in-app notification bell,
**I want** to land on Hem with the full list of unacknowledged amendments visible,
**so that** I can see all pending changes at once and triage them — not just the single amendment I clicked.

## Context & Dependencies

**Problem with current flow:**

Two issues with amendment notification click targets:

1. **Email**: Links to `/dashboard?changeId=<id>`, which immediately enters `ChangeAssessmentView` for that one specific change. If the user has 5 unacknowledged amendments that day, the email shows all of them as separate cards, but clicking any CTA takes them straight into a single assessment with no visibility into the others.

2. **In-app notifications**: Click navigates to `/dokument/${slug}` — the document page. This is a dead end from a workflow perspective: the user sees the law text but has no path to the assessment flow. The notification effectively does nothing useful.

**Desired flow (both channels):**

Notification click → Hem with `ChangePicker` dialog auto-opened → user sees full list of unacknowledged amendments → picks one to assess. The clicked amendment could be highlighted or sorted first, but the user has full context.

**Builds on:**

- Story 14.11: Hem — chat-first dashboard with context cards
- Story 8.4: `notify-amendment-changes` cron — generates email with `assessUrl`
- `ChangePicker` component (`components/features/dashboard/change-picker.tsx`) — already exists, fetches `getUnacknowledgedChanges()`, renders scrollable list
- `ChangeAssessmentView` — full-screen assessment entered after picking a change
- Story 8.5: In-App Notification Bell — introduced `NotificationItem` and `GET /api/notifications`
- `GET /api/notifications` — enriches notifications with `link_url` based on `entity_type`
- `NotificationItem` component — navigates to `link_url` on click

**Current deep-link flow:**
```
Email CTA → /dashboard?changeId=abc123
  → server: getUnacknowledgedChangeById(changeId)
  → HemPage: initialChange → immediately renders ChangeAssessmentView
```

**Proposed deep-link flow:**
```
Email CTA → /dashboard?view=amendments (optionally &highlight=abc123)
  → HemPage: auto-opens ChangePicker dialog on mount
  → User sees all pending amendments, picks one → ChangeAssessmentView
```

2. Email template CTA label remains "Granska ändringen" (singular) per card, but all CTAs link to the same amendments list view

### Dashboard Deep-Link Handling

3. When `/dashboard?view=amendments` is loaded, `HemPage` auto-opens the `ChangePicker` dialog on mount
4. URL is cleaned after consuming the param (existing `replaceState` pattern)
5. If there are no unacknowledged amendments when the user arrives, show the existing empty state in the picker ("Inga ändringar att granska")

### In-App Notification Click

6. Clicking an amendment notification in the bell menu navigates to `/dashboard?view=amendments` instead of `/dokument/${slug}`
7. The `link_url` enrichment in `GET /api/notifications` returns `/dashboard?view=amendments` for `change_event` entity types (AMENDMENT_DETECTED, LAW_REPEALED)

### Backwards Compatibility

8. The existing `?changeId=` deep-link continues to work (direct-to-assessment) for any bookmarked or cached links
9. No breaking change to `ChangePicker`, `ChangeAssessmentView`, or `getUnacknowledgedChanges()`

- **State variable:** `showChangePicker` / `setShowChangePicker` lives in `hem-chat.tsx` (line 78), not `hem-page.tsx`

- **`?view=` param is extensible:** If future stories need other deep-links to Hem views (e.g., `?view=tasks`), the same pattern applies

### Why Not Highlight the Specific Amendment?

We could pass `&highlight=abc123` to scroll-to or visually highlight the clicked amendment in the picker. But since the picker is already sorted by priority + recency, and the list is typically short (< 10 items), the added complexity isn't justified for v1. Can add later if users request it.

### Email UX Consideration

Each amendment card in the digest email still has its own "Granska ändringen" button — they just all link to the same `/dashboard?view=amendments` URL. This is intentional: the email gives the overview, the app gives the interaction. An alternative would be a single CTA at the bottom of the email ("Granska alla ändringar") but that's a larger email template change for a separate story.

### Refactoring Performed

- **File**: `app/api/notifications/route.ts`
  - **Change**: Removed `async` from `.map()` callback and `await Promise.all()` wrapper
  - **Why**: After changing `link_url` for change_events from a DB lookup to a static string, no enrichment logic is async anymore. The `Promise.all` + `async` was dead complexity.
  - **How**: Replaced `await Promise.all(notifications.map(async (n) => ...))` with synchronous `notifications.map((n) => ...)`. Reduces runtime overhead and makes the code simpler to reason about.
