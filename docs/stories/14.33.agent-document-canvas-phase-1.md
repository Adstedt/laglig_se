## Story

**As a** user collaborating with the compliance partner on a document it has drafted,
**I want** the full draft to open in a dedicated panel beside the chat instead of a cramped card or a one-off modal,
**so that** I can read the whole document comfortably while keeping the conversation — and the approve/reject controls — in view.

## Context

Story 14.24 renders an agent document draft as an inline approval **card** (the anchor: title, ~200-char preview, links, Godkänn / Öppna i editor / Avvisa). A card cannot show a full multi-page policy, so 14.24 routes "Visa mer" to a read-only `document-draft` detail-panel item (the canvas-ready seam, AC 11a).

This story turns that detail-panel item into a **proper canvas**: the proven "card = anchor, canvas = viewport" pattern that ChatGPT Canvas and Claude Artifacts converged on. Both are built on ProseMirror; **our Tiptap is a ProseMirror wrapper**, so we adopt the proven engine exactly rather than reinventing it.

**Phase 1 is read-only on purpose.** It delivers the highest-value, lowest-risk slice: comfortable full-document viewing beside the chat, responsive across surfaces, with edits still happening conversationally in the chat ("Be om ändringar", 14.24's revision loop if present). In-canvas editing, selection-based "highlight-to-edit", and the **diff / "show changes" layer** are Phase 2; editing *existing* styrdokument + versioning + the law-change trigger are Phase 3. Those are scoped here only as explicit non-goals + a costing flag, so the architecture doesn't paint us into a corner.

> **Roadmap (for context, not scope):**
> - **Phase 1 (this story):** read-only canvas panel + card↔canvas wiring + responsive shell.
> - **Phase 2:** editable canvas (full Tiptap) + selection-based edits + the ③ diff/"show changes" layer.
> - **Phase 3:** open & edit *existing* `WorkspaceDocument`s in the canvas → agent proposes diffs → approve → new version; wired to law-change detection (SFS/AFS sync) and Epic 21 findings.

## Acceptance Criteria

### Canvas panel surface

1. A **document canvas** renders the full read-only content of a `document-draft` `ChatDetailItem` in the chat detail panel (`chat-detail-sidebar.tsx`), beside the conversation. It reuses Epic 17's `DocumentEditor` in read-only mode (`editor.setEditable(false)`) **or** a lighter `generateHTML(contentJson, extensions)` render — implementer's choice by weight, but the render MUST support headings, lists, bold/italic, and **tables** (compliance docs use them; this is our edge over ChatGPT Canvas, which renders tables poorly).
2. The canvas header shows the document title + a type badge (`DOCUMENT_TYPE_LABELS`) + a close affordance. The title is **read-only** in Phase 1 (the card still owns title editing per 14.24 AC 8).
3. The canvas is scrollable independently of the chat; long documents do not push the conversation.
4. Approve / reject controls are **mirrored** in the canvas header (delegating to the same `approvePendingAction` / `rejectPendingAction` the card uses) so the user can act from the canvas without scrolling back to the card. Single source of truth: both surfaces read the same `pending-action:${id}` SWR key; approving from either updates both.

### `ChatDetailItem` variant

5. Extend the `ChatDetailItem` union in `lib/ai/chat-detail-context.tsx` with `{ type: 'document-draft'; id: string; data: DocumentDraftDetailData }` (if 14.24 has not already added it). `DocumentDraftDetailData = { pendingActionId: string; title: string; docType: WorkspaceDocumentType; contentJson: unknown }`.
5a. **Forward-compatibility for Phase 3:** the variant's `data` is shaped so an existing-document mode can be added later without a redesign — i.e. it can carry **either** an inline `contentJson` draft (Phase 1) **or** a `documentId` referencing a saved `WorkspaceDocument` (Phase 3). Phase 1 only implements the `contentJson` path; the `documentId` field is reserved/optional and unused now. Document this in a code comment.

### Card ↔ canvas wiring

6. The 14.24 `DraftDocumentApprovalRenderer`'s "Visa mer" opens the canvas via `openDetail({ type: 'document-draft', ... })` (this is 14.24 AC 11a; this story makes the panel render richly).
7. **Auto-open on wide surfaces only.** When a `draft_styrdokument` proposal arrives on a wide layout (dashboard chat), the canvas auto-opens via the existing `sidebarHint: 'open'` path (reuse the `autoOpenedToolCallIds` dedupe so it fires once). On narrow layouts, do NOT auto-open — surface a prominent "Visa utkast" on the card and let the user open it. (Auto-covering a narrow chat is intrusive.)
8. `openDetail()`'s existing toggle behavior is preserved (opening the same draft again closes it). Closing the canvas returns focus to the trigger (already handled by `closeDetail()` / `triggerRef`).

### Responsive behavior

9. **Wide (dashboard chat, ≥ the desktop breakpoint):** canvas renders **side-by-side** with the chat (chat left, canvas right), both visible and independently scrollable.
10. **Narrow (contextual / sidebar chat, tablet):** the canvas **takes over** the chat width with a "‹ Tillbaka" back affordance (no room to split a sidebar). Opening the canvas does not destroy chat state; "Tillbaka" returns to the conversation.
11. **Mobile (future-proofing, not a Phase-1 deliverable):** the variant + controls must not assume side-by-side; the same component must be mountable in a full-screen sheet. Phase 1 ships the wide + narrow behaviors; the mobile sheet is wired in the mobile-app effort. Confirm the exact breakpoints + the chat container components (dashboard `ChatPanel` vs the contextual sidebar chat) during implementation.

### Design language (per 14.22/14.23)

12. The canvas chrome adopts the **"spine & whisper"** design system from 14.23 for visual continuity with the approval cards: the warm→sage `.agent-spine` accent (tokens `--spine-top` / `--spine-bot` in `app/globals.css`), Safiro `font-medium` for the header (never `font-bold`/`font-semibold` — only weight 500 is registered), `rounded-md` controls, and the breathing-dot motif for the "awaiting your decision" state on the mirrored approve control. No new card chrome is invented — the canvas reads as the same family as the cards. [Source: `app/globals.css` `.agent-spine`; `_prototypes/14.23-agent-action-card-v2.html`]

### Non-regression

13. All existing `ChatDetailItem` types (`citation`, `tool-result`, `task`, `assessment`, `document`, `law-list-item`) continue to render and toggle correctly; the new variant is purely additive.
14. The 14.24 approval card, batch card, and all 14.22/14.23 pending-action types render unchanged. Approving a draft from the canvas produces the identical result to approving from the card.
15. Reasoning blocks (14.20), tool-call collapsing (14.18), citation pills (14.16), and chat streaming continue to render correctly when the canvas is open.

### Out of scope (Phase 2/3 — explicit non-goals)

16. **NOT in this story** (each is a follow-up): in-canvas editing (editable Tiptap); selection-based "highlight-to-edit"; the diff / "show changes" layer; the quick-shortcuts menu (Korta ner / Förenkla språket / …); editing existing `WorkspaceDocument`s; version history UI. See Technical Constraints for the **③ diff-layer costing decision** that Phase 2 must resolve first.

## Dev Notes

### Design research (proven pattern — not reinventing)

ChatGPT Canvas and Claude Artifacts independently converged on **card/anchor + side-panel viewport**, both built on **ProseMirror**. Our Tiptap is a ProseMirror wrapper, so Phase 1 adopts the same engine. Phase 1 mirrors their *read* surface; their *edit* primitives (highlight-to-edit, targeted edits, "show changes") are Phase 2. ChatGPT Canvas keeps formatting deliberately basic (headings/bold/italic/lists) and is weak on tables — our `DocumentEditor` already does tables, so we ship a small edge for free.

### Previous Story Insights

- **Story 14.24** owns the draft card + the `document-draft` seam (AC 11a). This story only enriches the panel render + responsive shell + card wiring; it does not touch the tool, dispatch, or approval semantics.
- **Story 14.23** established the inline-card design language (`.agent-spine`, Safiro `font-medium`, `rounded-md`, breathing dot) and removed sidebar *write-preview* cards. This `document-draft` variant is a **read-only preview**, not a write-preview — consistent with 14.23 keeping read-tool detail items. Approval controls stay inline/mirrored, never a sidebar form.
- **Story 14.15b** built the detail-sidebar; `openDetail()` already toggles + returns focus.

### Data Models

- **No schema changes.** Phase 1 reads `PendingAgentAction.params` (the draft `contentJson`) already created by 14.24. The canvas is a pure view + control-mirror over existing data. [Source: 14.24 AC 2; `lib/ai/chat-detail-context.tsx`]

### Component Specifications

- **Canvas = composition, not one widget:** ① panel/responsive shell (`chat-detail-sidebar.tsx`) + ② document surface (`DocumentEditor` read-only / `generateHTML`) + ④ chrome/controls (header, mirrored approve/reject). The ③ diff/suggestion layer is **Phase 2** (not built here).
- **Read-only render:** `DocumentEditor` (`components/features/documents/editor/document-editor.tsx`) computes `editable` from status and supports `editor.setEditable(false)`; it renders headings + tables (`TableContextMenu` present). Confirm whether mounting it read-only is heavier than a `generateHTML` render and pick the lighter path for a pure preview.
- **Detail context:** `lib/ai/chat-detail-context.tsx` — `openDetail(item, triggerElement?)`, toggle on same `type`+`id`, focus return via `triggerRef`. Reuse as-is.

### Technical Constraints

- **③ DIFF-LAYER DECISION (Phase 2 blocker — flagged here for costing, NOT built in Phase 1).** Rendering "show changes" in-place requires editor-level decorations. Three options, decreasing fidelity / decreasing cost:
  1. **Tiptap Pro track-changes / Comments** — commercial; **verify licensing + cost** before committing. (NB: the npm package literally named `@tiptap/suggestion` is for mentions/slash-commands, **not** track-changes — do not confuse them.)
  2. **Custom ProseMirror decoration diff** — compute a diff between two doc versions, render inserted/deleted ranges as decorations + per-change accept/reject. More build, no license.
  3. **Cheap before/after** — render the proposed version as a side-by-side or a list of changes, document stays read-only. Lowest fidelity; viable v1 of Phase 2.
  Architect (Winston) to evaluate before Phase 2 is drafted. This decision determines Phase 2's size.
- **Design language is mandatory, not optional** — the canvas must read as the same family as the 14.22/14.23 cards (AC 12). Safiro **must** pair with `font-medium` (only weight 500 is registered; bolder weights fake-bold / fall back to system-ui). [Source: MEMORY — Safiro weight pairing]
- **Don't reintroduce a write-preview** — this variant is read-only; approval is inline/mirrored, never a sidebar form (respect 14.23's decommission).
- **Additive only** — no schema, no changes to existing `ChatDetailItem` types, no changes to approval semantics.
