**I want** the Cmd+K search to find tasks, comments, and files — not just laws,

## Acceptance Criteria

### Multi-Entity Autocomplete (Cmd+K Dialog)

1. Typing in the Cmd+K dialog searches across **four entity types**: Legal Documents, Tasks, Comments, and Files
2. Results are grouped by type with Swedish group headers: "Lagar", "Uppgifter", "Kommentarer", "Filer"
3. Each group shows max 3 results in autocomplete. Legal documents group includes "Visa alla X resultat" link to full search page (`/sok`). Task/Comment/File groups show "Visa alla" only if their respective workspace pages support a `?search=` query param; otherwise omit the link
4. Each result shows: **title/name**, **snippet** (content preview), **breadcrumb** (parent context)
5. Breadcrumb examples:
   - Task: "Min lagsamling > Arbetsmiljölagen > Uppdatera brandskyddsplan"
   - Comment: "Min lagsamling > Arbetsmiljölagen > Anna Andersson"
   - File: "Min lagsamling > Arbetsmiljölagen > brandskyddsprotokoll.pdf"
   - Legal document: "Lag > Arbetsmiljölagen (1977:1160)"

### SFS Number Instant Lookup

6. Typing an SFS-pattern string (e.g., `2024:317`, `1977:1160`) triggers a direct `document_number` prefix match — instant, no FTS needed. Use regex `/^(19|20)\d{2}:\d{1,5}$/` for precise matching
7. SFS matches appear in a separate "Direktträff" group above other results

### Recently Viewed Entities

8. When the search input is empty, show "Senast besökta" — the user's most recently visited entities (max 5)
9. Recently viewed entities include: legal documents, tasks, law list items (any entity the user navigated to)
10. Recently viewed is stored per-user in localStorage (key: `laglig_recent_entities`)
11. Each entry stores: `{ id, title, type, url, breadcrumb, visitedAt }`
12. Existing "Senaste sökningar" (recent search terms) remains below "Senast besökta"

### Workspace Scoping

13. Tasks, Comments, and Files are scoped to the user's current workspace — never leak cross-workspace data
14. Legal Documents remain globally searchable (same as today)
15. When searching from a workspace context (`isWorkspace=true`), workspace-scoped results appear first

### Existing Functionality Preserved

16. All current legal document search behavior (autocomplete, full search page, Elasticsearch fallback, caching, analytics) continues to work unchanged
17. Cmd+K / Ctrl+K shortcut, ESC to close, arrow key navigation, Enter to open — all preserved
18. Recent search terms in localStorage preserved alongside new recently viewed entities

### Existing Autocomplete Pattern (follow this)

The current `searchAutocompleteAction` in `app/actions/search.ts` (lines 392-439) uses simple Prisma `contains` mode with `mode: 'insensitive'` — NOT full-text search. This is fast enough for autocomplete (small result sets, ILIKE queries). Follow the same pattern for Tasks, Comments, and Files. Do NOT add tsvector columns — that's an optimization for full search pages, not needed here.

### Breadcrumb Resolution

To build breadcrumbs, join through these paths:

```
Task breadcrumb:
  Task.list_item_links → TaskListItemLink.law_list_item → LawListItem
    → LawListItem.law_list → LawList.name
    → LawListItem.document → LegalDocument.title
  Format: "{LawList.name} > {LegalDocument.title} > {Task.title}"

Comment breadcrumb (on task):
  Comment.task → Task (use Task breadcrumb above, replace last segment with author name)
  Format: "{LawList.name} > {LegalDocument.title} > {Author.name}"

Comment breadcrumb (on law list item):
  Comment.law_list_item → LawListItem
    → LawListItem.law_list → LawList.name
    → LawListItem.document → LegalDocument.title
  Format: "{LawList.name} > {LegalDocument.title} > {Author.name}"

File breadcrumb:
  WorkspaceFile.task_links → FileTaskLink.task → Task (use Task breadcrumb)
  OR WorkspaceFile.list_item_links → FileListItemLink → LawListItem
  Format: "{LawList.name} > {LegalDocument.title} > {filename}"

Fallbacks for unlinked entities:
  Task with no list_item_links:    "{Workspace.name} > Uppgifter > {Task.title}"
  Comment with no task/list item:  "{Workspace.name} > {Author.name}"  (unlikely but defensive)
  File with no links:              "{Workspace.name} > Filer > {filename}"
```

### Navigation Resolution

Both task and law list item modals already support URL-synced deep-linking via query params. No new implementation needed — just build the correct URLs.

```
Legal document:           {browsePrefix}/lagar/{slug} (or /eu/forordningar/{slug}, etc.)
Task:                     /tasks?task={task.id}           ← existing ?task= param
Comment on task:          /tasks?task={comment.task_id}
Comment on law list item: /laglistor?document={law_list_item_id}  ← existing ?document= param
File linked to task:      /tasks?task={task_id}           (first linked task)
File linked to list item: /laglistor?document={law_list_item_id}
Unlinked file:            /filer?file={file_id}
```

**URL param sources:**
- Task modal: `components/features/tasks/task-workspace/index.tsx` line 132 — `handleTaskClick` sets `?task=` param
- List item modal: `components/features/document-list/document-list-page-content.tsx` line 129 — `handleOpenModal` sets `?document=` param

**Priority for files/comments with multiple parents:** Use the first `task_link` if any, else first `list_item_link`, else fall back to unlinked URL.

### Performance Budget

- Autocomplete query: <300ms total for all 4 entity types (run in parallel)
- Each individual query: <100ms (ILIKE on indexed workspace_id + small result sets)
- SFS direct lookup: <50ms (prefix match on document_number, indexed)
- Recently viewed: <1ms (localStorage read)

### Recently Viewed — Future Migration Note

This story uses localStorage for recently viewed entities (fast, no migration, MVP-appropriate). A future story could migrate to server-side tracking (per-user `UserEntityVisit` model) for cross-device sync. Explicitly out of scope here.

### Important: "/" Shortcut Conflict

Do NOT implement the `/` keyboard shortcut for search. It conflicts with `lib/hooks/use-chat-keyboard-shortcuts.ts` which captures `/` for AI chat focus. Keep only Cmd+K / Ctrl+K.

## Risk Assessment

### Implementation Risks

- **Primary Risk**: Autocomplete performance degradation from 4 parallel queries
- **Mitigation**: Each query is workspace-scoped (small dataset), uses indexed columns, limited to 3 results. Monitor with `performance.now()` timing in action.
- **Verification**: Add timing logs; if any query >100ms, add database index or reduce scope

### Rollback Plan

- All changes are additive — existing `searchAutocompleteAction` is not modified, only new functions added alongside it
- `searchAllAutocomplete` wraps the existing function; if issues arise, revert the dialog to call `searchAutocompleteAction` directly
- Recently viewed is localStorage-only — no schema changes, no migration needed
