# Story 14.20: Anthropic Extended Thinking for High-Stakes Chat Contexts — Brownfield Addition

**Epic:** 14 (AI Chat)
**Status:** Backlog
**Size:** Single session (~2–3 hours)

## User Story

**As a** Laglig.se user asking complex compliance questions (especially change assessments and task-scoped chat),
**I want** the AI to reason through multi-step legal analysis visibly before answering,
**so that** I can trust the conclusion, verify the logic, and catch any flawed assumption before acting on it.

## Story Context

**Existing System Integration:**

- **Integrates with:** `app/api/chat/route.ts` (`streamText` config), `ReasoningBlock` in `components/features/ai-chat/chat-message.tsx`
- **Technology:** AI SDK `@ai-sdk/anthropic`, Vercel AI Gateway, Claude Sonnet 4.6
- **Follows pattern:** Existing conditional `providerOptions` / `experimental_transform` spread in the chat route; existing `sendReasoning: true` in `toUIMessageStreamResponse()`
- **Touch points:** `providerOptions.anthropic.thinking`, per-context budget resolution, existing reasoning-part rendering pipeline

The UI plumbing for reasoning already exists (`sendReasoning: true` is on, `ReasoningBlock` is wired to `message.parts` with state `streaming | done`). This story turns on the model-side feature and tiers the budget by stakes.

## Acceptance Criteria

**Functional Requirements:**

1. Extended thinking is enabled on the Anthropic model path (`AI_CHAT_MODEL === 'anthropic'`) with a per-context budget map:
   - `change` → `budgetTokens: 8000`
   - `task` → `budgetTokens: 3000`
   - `law` → `budgetTokens: 2000`
   - `global` → disabled (no `thinking` option passed)
2. When `budgetTokens === 0` / disabled, the `thinking` option is **omitted entirely** from `providerOptions.anthropic` (not `budgetTokens: 0`) — avoids sending an explicit off-config when OpenAI path is also a possibility.
3. `ReasoningBlock` continues to render reasoning parts correctly (already works via `sendReasoning: true`). For `change` context, the block opens **expanded by default** to emphasize the model's working-out; other contexts remain collapsed by default.
4. **System prompt includes explicit "when to reason" guidance** so the model self-selects between short answers and deep reasoning within the budget it's been given. The budget is a ceiling; this guidance shapes the model's choice of how much of it to use. Added to `lib/agent/system-prompt.ts` (inside `buildSystemPrompt`), only appended when thinking is enabled for the resolved context:

   ```text
   You have extended reasoning available. Use it when:
   - Evaluating whether a law applies to a specific company situation
     (size, industry/SNI, employee count, jurisdiction)
   - Weighing exceptions, thresholds, or conditions in the law
   - Synthesizing multiple sources (law + amendment + agency guidance)
   - Making a compliance assessment or risk-weighted recommendation
   - The user is about to act on your answer (save, delegate, publish)

   Skip lengthy reasoning when:
   - The user asks a factual lookup ("vad är SFS 2022:123?")
   - The answer is already in a document you fetched via search_laws
   - The question is procedural ("hur skapar jag en uppgift?")
   - A direct citation from a primary source fully answers the question

   When you do reason, focus on the decision at hand. Avoid restating
   facts; work through the factors that affect the conclusion.
   ```

   Translate key terms to Swedish if the rest of the system prompt is Swedish.

**Integration Requirements:**

5. Existing chat flow — tool calls, citation sources, `smoothStream` pacing, follow-mode scroll — continues to work unchanged.
6. New config follows the existing pattern used for `experimental_transform: smoothStream(...)` — conditional spread into `streamText()` args.
7. `stopWhen: stepCountIs(10)` unchanged. `maxDuration` increased from `60` to `90` to accommodate thinking tokens on tool-heavy change assessments.

**Quality Requirements:**

8. Manual QA covers all four contexts with both simple and complex prompts:
   - `global` — simple query, no reasoning block should appear (thinking disabled).
   - `law` — explain a law, small reasoning block.
   - `task` — scoped compliance task, visible reasoning before answer.
   - `change` — amendment assessment, reasoning block **expanded by default**, clear multi-step chain.
9. **Prompt-steering QA:** within an enabled context (e.g., `law`), compare two queries — a factual lookup ("vad är SFS 2022:123?") vs. a multi-factor question ("gäller arbetsmiljölagen för en visstidsanställd konsult?"). The factual lookup should show minimal reasoning; the multi-factor question should show substantive reasoning. This verifies the prompt guidance is actually shaping behavior, not just the budget.
10. Memory note in `MEMORY.md` ("AI Chat Streaming & Reasoning" section) updated with the new thinking config **and the prompt-steering pattern**.
11. No regression in existing chat behaviors (tool orchestration, citation rendering, stick-to-bottom scroll).

## Technical Notes

- **Two levers, not one:** The budget (`budgetTokens`) is a *ceiling*; the model decides per-response how much of it to use. Budget alone gives coarse control. The **system-prompt "when to reason" guidance (AC 4) is the complementary steering lever** — it tells the model which situations warrant deep reasoning vs. a direct answer. Ship both together; budget without prompt guidance leads the model to over- or under-use thinking inconsistently.
- **Integration approach:** Resolve `thinkingBudget` from a small map keyed on `contextType`. Spread `providerOptions.anthropic.thinking` into `streamText()` **only** when `thinkingBudget > 0`. Append the reasoning-guidance block to the system prompt only when `thinkingBudget > 0`.
- **AI SDK shape:**
  ```ts
  providerOptions: {
    anthropic: {
      thinking: { type: 'enabled', budgetTokens: thinkingBudget },
    },
  }
  ```
  Note: `budgetTokens` (camelCase) in AI SDK, NOT `budget_tokens` like the raw Anthropic SDK.
- **Existing pattern reference:** `app/api/chat/route.ts` — the `experimental_transform: smoothStream({...})` block is a good template for conditional provider config. `buildSystemPrompt` in `lib/agent/system-prompt.ts` already composes a multi-section prompt — extend it with an optional "reasoning guidance" section.
- **Interleaved thinking (automatic):** Claude 4.6 Sonnet reasons *between* tool calls when thinking + tools are both active. No explicit configuration needed; it's a natural consequence of enabling thinking on paths that already use `createAgentTools`. Expect reasoning blocks interleaved with tool rows in the UI on `change`/`task` contexts.
- **Key constraints:**
  - Only applies when `AI_CHAT_MODEL === 'anthropic'`; OpenAI path is unaffected.
  - Budget caps prevent runaway cost; thinking tokens are billed at the same rate as output tokens.
  - `ReasoningBlock` default-open state for `change` context: pass a new prop (e.g., `defaultOpen`) through `ChatMessage` → `ReasoningBlock`, sourced from `contextType`.
  - Prompt guidance must match the language of the rest of the system prompt (Swedish project — translate the sample copy accordingly).

## Risk and Compatibility Check

**Minimal Risk Assessment:**

- **Primary Risk:** Thinking tokens push response past `maxDuration` for long chains on tool-heavy change assessments.
- **Mitigation:** Raised `maxDuration` to 90s; budget capped per context to keep latency predictable (highest context `change` at 8000 tokens ≈ ~4–6s thinking overhead). System prompt guidance (AC 4) reduces the chance the model uses full budget on simple questions, keeping average latency down.
- **Secondary Risk:** Model reasons on *every* query within an enabled context, including simple lookups — wasting budget and adding latency.
- **Mitigation:** AC 4 (system prompt "when to reason" guidance) is the direct countermeasure. AC 9 (prompt-steering QA) validates the guidance is actually taking effect.
- **Rollback:** Remove the `thinkingBudget` map, the conditional `providerOptions` spread, and the prompt-guidance block. Revert `maxDuration` to 60. Single-PR revertable.

**Compatibility Verification:**

- [ ] No breaking changes to APIs
- [ ] No database changes
- [ ] UI change limited to `ReasoningBlock`'s default-open prop (component already renders)
- [ ] Performance impact bounded: no change on `global` context; +2–5s latency on `change` context only
