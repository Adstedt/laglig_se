# Story 24.3: Import — matching engine (fuzzy + LLM)

## Status

Done

## Story

**As a** workspace member who has just uploaded a 50-row Excel of laws,
**I want** Laglig to automatically resolve each row against the SFS / AFS / EU catalog using a two-stage fuzzy + LLM match — and tier the result as `MATCHED_HIGH`, `MATCHED_MEDIUM`, or `UNMATCHED`,
**so that** by the time I open the review surface (Story 24.4), the high-confidence subset is one bulk click away and only the ambiguous rows need my attention.

This is the **matching layer only** — rows transition from `LawListImportRow.match_status='PENDING'` to `MATCHED_HIGH | MATCHED_MEDIUM | UNMATCHED` with `confidence_score`, `matched_document_id`, and `match_candidates` populated. **No review UI** (Story 24.4) and **no commit logic** (Story 24.4). The story validates that on a hand-curated benchmark of real switcher exports, the matcher achieves **≥70% high-confidence on first pass** (per Epic PRD Story 24.3 acceptance).

### Fuzzy retrieval — new file `lib/search/match-candidates.ts`

1. **Create new file `lib/search/match-candidates.ts`** exporting `findMatchCandidates(input, limit?: number = 5): MatchCandidate[]`. **NOTE:** there is no `lib/legal-document/search.ts` in this repo — search infra lives at `lib/search/index.ts` + `lib/search/rerank.ts`. Place the new candidate retrieval next to those, scoped to import-matching use only (do not pollute the global search namespace). Signature:
   ```ts
   export type MatchCandidate = {
     document_id: string
     title: string
     document_number: string | null   // canonical id from LegalDocument.document_number, e.g. "SFS 1977:1160", "AFS 2020:5", "(EU) 2016/679"
     content_type: string             // ContentType enum value: 'SFS_LAW' | 'SFS_AMENDMENT' | 'AGENCY_REGULATION' | 'EU_REGULATION' | 'EU_DIRECTIVE' | 'COURT_CASE_*'
     fuzzy_score: number              // 0.0–1.0
     match_signals: {
       document_number_exact: boolean      // input.document_number canonical equals candidate.document_number
       document_number_suffix_match: boolean // input year:number (or year/number) suffix matches candidate, prefix differs (tier-2 fallback)
       title_trigram_score: number          // 0.0–1.0
       has_amendment_match: boolean         // optional — fuzzy match against amendment titles
     }
   }

   export type FindCandidatesInput = {
     titel: string | null
     /**
      * Free-form regulation id from the source row's mapped column.
      * Field name kept as `sfs_nummer` to align with `LawListImportRow.source_sfs_nummer`
      * (24.1 schema, in turn aligned with how Swedish exports label the column),
      * but treated semantically as a generic agency-or-EU document id and handed
      * to `parseDocumentNumber` for normalisation.
      */
     sfs_nummer: string | null
   }

   export function findMatchCandidates(input: FindCandidatesInput, limit?: number): Promise<MatchCandidate[]>
   ```

2. **Generalised document-number parser.** **The previously-named `normalizeSfsId` is renamed to `parseDocumentNumber` and lives in NEW file `lib/import/document-number.ts`** (NOT in `lib/sfs/ensure-prefix.ts` — the helper is now generic across agency series and EU forms, so an SFS-named file is the wrong home). Existing `ensureSfsPrefix` in `lib/sfs/ensure-prefix.ts` stays untouched.

   Signature + return shape:
   ```ts
   export type DocumentNumberKind = 'SFS_LIKE' | 'EU_REG' | 'EU_DIR' | 'OTHER'

   export interface ParsedDocumentNumber {
     /** Captured prefix as-it-appears in the input — e.g. "SFS", "AFS", "MSBFS",
      *  "HSLF-FS", "NFS", "EU", "EG". Open-ended set: do NOT close this into an enum.
      *  When the input contains no recognisable prefix, `series` is null. */
     series: string | null
     /** Full canonical id matching the format stored in `LegalDocument.document_number`,
      *  e.g. "SFS 1977:1160", "AFS 2020:5", "(EU) 2016/679", "2010/75/EU".
      *  See Task 0 — dev MUST verify against a sample of real rows before locking
      *  the canonical form for EU references. */
     canonical: string
     /** Year component, e.g. "1977", "2016". Used for tier-2 suffix-fallback matching. */
     year: string
     /** Number component, e.g. "1160", "5", "679". Used for tier-2 suffix-fallback matching. */
     number: string
     /** Coarse classification used by the matcher to pick scoring branches and
      *  by the LLM prompt to phrase the candidate display. */
     kind: DocumentNumberKind
   }

   export function parseDocumentNumber(raw: string): ParsedDocumentNumber | null
   ```

   Patterns (apply in order; first match wins; case-insensitive matching but case-preserving canonical output):
   - **Agency / SFS-like** (covers SFS, AFS, MSBFS, NFS, BFS, SOSFS, HSLF-FS, ELSÄK-FS, SKVFS, TSFS, and **any future agency code** without code change): `/^([A-ZÅÄÖ][A-ZÅÄÖ\-]+)\s+(\d{4}):(\d{1,5})$/i` — group 1 = `series` (uppercased on output), group 2 = `year`, group 3 = `number`. Canonical = `"${SERIES} ${year}:${number}"` (single space separator, leading zeros preserved on number).
   - **Agency / SFS-like, prefix omitted** (just `YYYY:NNN`): canonical = `"${year}:${number}"`, `series=null`, `kind='SFS_LIKE'`. The matcher's tier-2 fallback (AC 4) handles disambiguation.
   - **EU regulation, parenthesised form**: `/^\((EU|EG|EEG|Euratom|EU,\s*Euratom)\)\s+(\d{4})\/(\d+)$/i` — `series` = group 1 normalised (e.g., "EU", "EG"), canonical form per **Task 0 verification** (likely `"(EU) ${year}/${number}"`), `kind='EU_REG'`.
   - **EU directive trailing-form**: `/^(\d{4})\/(\d+)\/(EU|EG|EEG)$/i` — `series` = group 3 normalised, canonical `"${year}/${number}/${SERIES}"`, `kind='EU_DIR'`.
   - **EU bare year/number** (no parens, no trailing /EU): `/^(\d{4})\/(\d+)$/` — canonical `"${year}/${number}"`, `series=null`, `kind='EU_REG'`.

   Inputs are pre-trimmed and have any leading/trailing whitespace + duplicate internal spaces collapsed before regex match. Returns `null` when no pattern matches.

   **Tests** (place at `tests/unit/lib/import/document-number.test.ts`): minimum coverage matrix:
   - SFS: `"1977:1160"` → `{series: null, canonical: "1977:1160", year: "1977", number: "1160", kind: "SFS_LIKE"}`
   - SFS prefixed: `"SFS 1977:1160"`, `"sfs 1977:1160"`, `"  SFS  1977:1160 "` all → canonical `"SFS 1977:1160"`
   - Leading zeros preserved: `"SFS 1962:0381"` → number `"0381"`
   - AFS: `"AFS 2020:5"` → `{series: "AFS", canonical: "AFS 2020:5", ...}`
   - HSLF-FS hyphen: `"HSLF-FS 2022:30"` → `{series: "HSLF-FS", canonical: "HSLF-FS 2022:30", ...}`
   - MSBFS, NFS, BFS, SOSFS, ELSÄK-FS, SKVFS, TSFS: at minimum 1 case each (any-agency proof)
   - Future-unknown agency: `"ZZFS 2030:1"` → `{series: "ZZFS", ...}` (proves the open-set principle)
   - EU paren: `"(EU) 2016/679"` → `{series: "EU", canonical per Task 0, kind: "EU_REG"}`
   - EU directive: `"2010/75/EU"` → `{series: "EU", kind: "EU_DIR"}`
   - EU bare: `"2016/679"` → `{series: null, kind: "EU_REG"}`
   - Invalid: `"foobar"`, `""`, `"NJA 2023 s. 45"` (court case, deliberately rejected — not in scope) → `null`

3. **Title trigram similarity.** Use Postgres `pg_trgm` extension. **PRECONDITION (hard gate before Task 2 begins):** verify `pg_trgm` is enabled on the staging database via `SELECT * FROM pg_extension WHERE extname='pg_trgm';`. If the row is absent, the dev MUST add a small migration `<TIMESTAMP>_enable_pg_trgm/migration.sql` containing `CREATE EXTENSION IF NOT EXISTS pg_trgm;` and apply it before any code in this story is written. There is **no** prior use of `pg_trgm` in this codebase to inherit from — confirmed via `grep -r pg_trgm`. Once enabled:
   ```sql
   SELECT id, title, document_number, content_type,
          similarity(title, $1) AS trgm_score
   FROM legal_documents
   WHERE similarity(title, $1) > 0.3
   ORDER BY trgm_score DESC
   LIMIT $2;
   ```
   Implement via `prisma.$queryRaw` (parameterised — never interpolated). The `0.3` threshold is empirical; tune in benchmarking. Add a GIN index `CREATE INDEX legal_documents_title_trgm_idx ON legal_documents USING gin (title gin_trgm_ops);` in the same migration to keep the query under the 500ms budget. **Schema note:** the canonical id column is `document_number` (`String @unique` on `legal_documents` — verified at `prisma/schema.prisma:482`). The classification column is `content_type` (`ContentType` enum). There is no `sfs_number` column on this table; an earlier draft of this story referenced one in error.

4. **Combined scoring.** For each candidate, compute `fuzzy_score` as the maximum of three branches:

   **Branch A — Document-number exact match (highest signal).** When the input parses cleanly via `parseDocumentNumber` AND its `canonical` string matches the candidate's `document_number` exactly (case-sensitive after both have been normalised through `parseDocumentNumber`), set `match_signals.document_number_exact = true` and `fuzzy_score = 1.0`. Stop scoring; this candidate is the unambiguous answer.

   **Branch B — Document-number suffix match (tier-2 fallback).** When the input parses cleanly but the `canonical` does NOT exact-match any candidate, AND the candidate's `document_number` (after `parseDocumentNumber`) shares the same `year:number` (or `year/number` for EU) tail as the input — but with a different `series` prefix — set `match_signals.document_number_suffix_match = true`. This catches cases where the user dropped the prefix ("2016/679" without "(EU)") or wrote a different prefix variant. Score contribution: `+0.4`.

   **Branch C — Title trigram + signal boosts (general fallback).**
   - `0.6 * title_trigram_score` (from AC 3's pg_trgm query)
   - `+ 0.2` if `match_signals.document_number_suffix_match` (Branch B may have fired alongside Branch C; combine cumulatively, capped at 1.0)
   - `+ 0.2 * series_inferred_match` where `series_inferred_match` is 1.0 when (a) input parsed with a known `series` AND candidate's parsed `series` equals it, OR (b) input parsed with `series=null` but `input.titel` contains a tokenised hint matching the candidate's `content_type`: "AFS"/"föreskrift" → `AGENCY_REGULATION`, "EU"/"förordning"/"direktiv" → `EU_REGULATION` or `EU_DIRECTIVE`. Otherwise 0.
   - Final `fuzzy_score = min(1.0, sum_of_above)`.

   Branch A's exact-match short-circuit means Branches B and C are only evaluated when the exact-match path didn't fire. The matcher's Stage-1 short-circuit (AC 7) reads `match_signals.document_number_exact` to decide whether to skip the LLM.

5. **Performance budget.** `findMatchCandidates` must return in <500ms for the p95 case (small workspaces, ~10k LegalDocument rows). Add an index hint or full-text-search migration if needed. **DO NOT** load all documents into memory; use the `pg_trgm` index.

### LLM disambiguation — `lib/import/matcher.ts` (NEW)

6. **New file `lib/import/matcher.ts`** — exports two functions:
   ```ts
   export type MatchResult = {
     matched_document_id: string | null
     confidence_score: number    // 0.0–1.0
     confidence_tier: 'high' | 'medium' | 'unmatched'
     candidates: MatchCandidate[]
     reasoning: string | null     // LLM-generated; null when fuzzy alone resolved
     llm_used: boolean            // true if Stage 2 fired
   }

   export async function matchRow(input: {
     titel: string | null
     sfs_nummer: string | null
     omrade: string | null
     kommentar: string | null
   }): Promise<MatchResult>

   export async function matchRowsBatch(inputs: Array<...>): Promise<MatchResult[]>  // parallelised, batches of 10
   ```

7. **`matchRow` two-stage logic.**
   - **Stage 1 — Fuzzy retrieval:** call `findMatchCandidates({titel, sfs_nummer}, 5)` → top 5 candidates.
   - **Stage 1 short-circuit:** if top candidate's `fuzzy_score >= 0.95` AND `match_signals.document_number_exact === true`, accept directly without LLM:
     - Return `{ matched_document_id: top.id, confidence_score: 1.0, confidence_tier: 'high', candidates, reasoning: null, llm_used: false }`
   - **Stage 1 short-circuit (no candidates):** if `findMatchCandidates` returned 0 results, return `{ matched_document_id: null, confidence_score: 0, confidence_tier: 'unmatched', candidates: [], reasoning: 'Inga kandidater i katalogen', llm_used: false }`. **Skip LLM** — no point asking it to disambiguate from nothing.
   - **Stage 2 — LLM disambiguation:** call Anthropic Sonnet via `lib/anthropic-client.ts` with the prompt from AC 8. Parse structured output. Returned confidence is the LLM's own `confidence` field (0.0–1.0).

8. **LLM prompt at `lib/import/matcher-prompt.ts`** — Swedish-language instructions wrapping an English-keyed JSON schema (English keys for code stability; Swedish prose in instructions for model alignment with the Swedish corpus):
   ```
   System prompt (cached via Anthropic prompt caching):
     Du hjälper Laglig.se att matcha en kunds importerade laglista mot vår katalog.
     Givet en källrad och upp till 5 kandidatdokument från vår katalog, välj det mest sannolika.
     Källraden kan referera till SFS-lagar, agency-föreskrifter (AFS, MSBFS, NFS, BFS,
     SOSFS, HSLF-FS, ELSÄK-FS, SKVFS, TSFS, m.fl.), eller EU-förordningar/direktiv.
     Returnera JSON med exakta nycklar: {chosen_document_id: string|null, confidence: number 0..1, reasoning: string}.
     - reasoning: kort förklaring på svenska, max 2 meningar.
     - Om dokumentnummer matchar exakt på en kandidat: confidence ≥ 0.9.
     - Om dokumentnummer endast matchar i sifferdelen (samma år och nummer men olika prefix): 0.6–0.85, och förklara osäkerheten.
     - Om bara titeln matchar (men nära): 0.6–0.85.
     - Om ingen kandidat verkar passa eller källraden är för otydlig: chosen_document_id: null, confidence: < 0.5.
     - Var skeptisk till "låg" matchning — false positives är värre än "saknas i katalogen".

   User prompt (per call):
     Källrad:
       Titel: ${titel}
       Dokumentnummer: ${sfs_nummer ?? '(inget angivet)'}    # field name kept for symmetry with the source-row column label; covers SFS/AFS/EU/etc.
       Område: ${omrade ?? '(inget angivet)'}
       Kommentar: ${kommentar ?? '(inget angivet)'}

     Kandidater:
       ${candidates.map(c => `- [${c.document_id}] ${c.title} (${c.document_number ?? c.content_type}) — fuzzy_score: ${c.fuzzy_score.toFixed(2)}`).join('\n')}

     Returnera JSON enligt schemat ovan.
   ```
   **JSON keys are English** (`chosen_document_id`, `confidence`, `reasoning`) for code stability. Instructional prose is Swedish for model alignment with the Swedish-corpus terminology. Validate with Zod against `LlmMatchResponseSchema` (defined in matcher-prompt.ts) which accepts those exact English keys. Retry once on parse failure (mirrors `lib/external/llm-amendment-parser.ts` retry pattern at `parseAmendmentPdf`). Use the Anthropic SDK's standard `messages.create` call — no `response_format` parameter; rely on Zod validation + 1 retry.

   **Source-row label note:** the user-prompt label is "Dokumentnummer" (not "SFS-nummer") to avoid steering the model toward an SFS-only assumption. The source-row column on `LawListImportRow` is still named `source_sfs_nummer` (24.1 schema, aligned with how Swedish exports label the column) but its semantic content is any agency-or-EU document id.

9. **Confidence tier mapping** (deterministic, applied to LLM output):
   - `>= 0.85` → `MATCHED_HIGH`
   - `0.5 ≤ score < 0.85` → `MATCHED_MEDIUM`
   - `< 0.5` → `UNMATCHED`

10. **Cost & cache.** Use Anthropic prompt caching (Story 14.26 pattern) on the system prompt. Per-row cost target: < $0.005 with cache hits. Total per 100-row import: < $0.50. Verify in 24.3 acceptance benchmarking.

    **Worked cost calculation** (validate against telemetry post-implementation):
    - System prompt: ~600 tokens (cached after first call)
    - User prompt per row: ~250 tokens (uncached — varies per row)
    - Output per row: ~100 tokens
    - Sonnet-4-6 pricing (per `lib/usage/cost-estimator.ts` `PRICING` map): input $3/MTok, output $15/MTok, cache-write $3.75/MTok, cache-read $0.30/MTok
    - Per row: cache-read 600 × $0.30/MTok + uncached 250 × $3/MTok + output 100 × $15/MTok ≈ $0.0018 + $0.00075 + $0.0015 ≈ **$0.0040 per row**
    - 100-row import: **~$0.40**, well within target. First-row cache-write: extra ~$0.0023 one-time.
    - If observed cost runs >2× this estimate, investigate cache-hit rate via `ChatUsageEvent.cache_read_input_tokens` aggregation.

### Server action — `runMatching`

11. **Implement `runMatching(importId)`** (skeletoned in 24.1):
    ```ts
    runMatching(importId: string)
    → returns { success: true, data: { matchedHighCount, matchedMediumCount, unmatchedCount, durationMs } }
    ```
    Steps:
    - Verify import exists, belongs to ctx.workspaceId, status=UPLOADED
    - Update status to `MATCHING`, persist
    - Load all `LawListImportRow` rows with `match_status='PENDING'`
    - Map each row to `MatcherInput` (pluck `source_titel`, `source_sfs_nummer`, `source_omrade`, `source_kommentar`)
    - Call `matchRowsBatch(inputs)` — internally batches of 10 concurrent calls. Use a small concurrency-limit pattern (e.g., `p-limit` if already in deps; otherwise inline a 5-line sliding-window helper) to cap at 10 in-flight to respect Anthropic rate limits (~50 RPM on Sonnet at our tier; 10 concurrent × ~3s avg = ~200 RPM peak — needs throttling or chunked-then-sequential dispatch). On 429 response: exponential backoff retry up to 3 times, then mark that row UNMATCHED with `reasoning='Rate-limit hit, försök importera om'`.
    - For each result, persist back to the row:
      - `matched_document_id`
      - `confidence_score`
      - `match_status` (mapped from confidence_tier)
      - `match_candidates` (top 5 stored as JSON: `[{document_id, title, document_number, content_type, fuzzy_score}, ...]`). **Field name `document_number`** — earlier drafts used `sfs_number`, which doesn't exist on `LegalDocument`. Story 24.4's review UI reads this JSON to render candidate-picker rows; keep the shape stable.
      - `match_reasoning`
    - Update import status to `AWAITING_REVIEW`
    - Send activity-log: `law_list_import.matching_completed` (new action constant)
    - Return aggregate counts

12. **Failure modes:**
    - Anthropic API down → set import status back to `UPLOADED`, return error; user can re-trigger via 24.4 retry button (deferred — for now, surface error and require manual support)
    - Single-row LLM parse failure → mark that row as `UNMATCHED` with `reasoning='Kunde inte avgöra matchning automatiskt'`, continue with other rows
    - Total failures > 50% → set import status to `FAILED` with `error_message='Matchningen misslyckades — kontakta support'`

13. **Idempotency.** Calling `runMatching` twice on the same import is a no-op on the second call (status check rejects unless status=UPLOADED). If the user wants to re-match, that's a future enhancement (delete + re-create import for now).

3. **Primary keys use `@default(uuid())` not `@default(cuid())`.** Not directly applicable to this story (no new models), but holds for any future addition.

These three points were spec-text vs project-pattern divergences in the original Epic 24 stories; PO ratified the project-pattern interpretation. They are NOT to be re-litigated in this story's QA review.

**Existing patterns to follow:**
- **Anthropic SDK usage** — there is **no central `lib/anthropic-client.ts` wrapper** in this repo. The canonical precedent for direct SDK use with structured output + retry is `lib/external/llm-amendment-parser.ts` (Story 2.13 / 8.20). It imports `Anthropic from '@anthropic-ai/sdk'`, instantiates `new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY })`, calls `client.messages.create(...)`, and retries up to 3 times on parse failure. Mirror that exact pattern for the matcher: instantiate locally, call `messages.create`, parse + Zod-validate response, retry once on parse failure (24.3 uses 1 retry not 3 because per-row cost matters).
- **Prompt caching** — Memory `MEMORY.md` line 105 documents the exact pattern: "the system prompt is wrapped as a `SystemModelMessage` with `providerOptions.anthropic.cacheControl = { type: 'ephemeral' }`. Default TTL 5 min. Cache marker goes on the MESSAGE, not top-level providerOptions." NOTE: that pattern is for the `ai` SDK (vercel/ai). When using the raw `@anthropic-ai/sdk` (this story's path, mirroring `llm-amendment-parser.ts`), prompt caching uses the SDK's native `cache_control: { type: 'ephemeral' }` block on the system message content array. Reference: `https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching`. Confirm exact SDK syntax by reading the latest `@anthropic-ai/sdk` types.
- **Fail-safe telemetry** — try/catch + `console.error` + never re-throw. See `app/api/chat/route.ts` `onFinish` callback.
- **Structured output via Zod** — see existing structured-output examples. `lib/sfs/llm-output-validator.ts` validates the amendment-PDF parser's output via custom HTML validation; for JSON output Zod parsing is the right primitive. No project-wide canonical "Zod-validate-JSON-from-Anthropic" helper exists yet — write one inline in `lib/import/matcher-prompt.ts` (the scope is just this story).
- **`prisma.$queryRaw` parameterisation** — see `lib/admin/queries.ts` for the input-clamping + `INTERVAL '1 day' * ${n}::int` pattern. Never string-interpolate user input.
- **Batch operations with concurrency cap** — no project-wide pattern exists. Either add `p-limit` (~1KB dep) and use `pLimit(10)` wrapper, or hand-roll a sliding-window helper. The hand-roll fits in ~10 lines and avoids a dep.

**Key decisions baked in:**
- **Two-stage matching** (fuzzy → LLM) is intentional. Pure-fuzzy is too rigid for paraphrased consultant lists; pure-LLM is expensive and slow. Fuzzy narrows to top-5, LLM disambiguates with cheap structured output.
- **0.3 trigram threshold** is the cutoff. Lower captures noise; higher misses paraphrases. Tune in benchmarking if needed.
- **0.85 high-confidence cutoff** is intentionally strict — false positives at "high" cost user trust ("how is this paragraph in my list, I never asked for it?"). When in doubt, push to medium.
- **Empty-candidates short-circuit skips LLM** — saves money + time. The LLM can't disambiguate from nothing.
- **No re-matching today** — if user wants to re-match, delete the import and re-upload. Re-match feature deferrable; flag as future enhancement.
- **Telemetry context_type='import_matching'** distinguishes matcher LLM cost from chat cost in the admin dashboard. Don't reuse `'global'` or `'change'`.
- **Open-set agency-prefix capture** — `parseDocumentNumber` does NOT enumerate agency prefixes (SFS, AFS, MSBFS, NFS, BFS, SOSFS, HSLF-FS, ELSÄK-FS, SKVFS, TSFS, …). The set is too large and grows over time as new agencies issue föreskrifter. Instead, the helper captures the prefix generically via `[A-ZÅÄÖ][A-ZÅÄÖ\-]+` so any future agency code works without code change. EU forms get dedicated patterns because their structure differs (slash separator, parenthesised prefix, optional trailing /EU).
- **Canonical match against `LegalDocument.document_number`** — the catalog stores the full canonical id ("SFS 1977:1160", "AFS 2020:5", "(EU) 2016/679") in a single `String @unique` column. The matcher compares input.canonical to candidate.document_number directly; no prefix-stripping or join-table-on-series needed. This is why `MatchCandidate.document_number` (not `sfs_number`) is the correct field name — there is no `sfs_number` column on `legal_documents`.

- Review UI (Story 24.4)
- Commit logic (Story 24.4)
- Re-match support
- Manual column-mapping override
- Cross-language matching (we support Swedish + English titles only because the corpus is Swedish; non-Latin scripts excluded)
