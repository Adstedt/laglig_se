# Story 17.8: Text Extraction Pipeline for Uploaded Files

## Status

Draft

## Story

**As a** system component,
**I want** uploaded workspace files (PDFs, .docx, .xlsx) to have their text content automatically extracted,
**so that** the agent can read and search across all workspace documentation, not just Tiptap-authored documents.

## Story Context

**Existing System Integration:**

- Integrates with: `WorkspaceFile` model (add `extracted_text` field), existing `uploadFile()` server action in `app/actions/files.ts`
- Technology: `pdf-parse` (new install — listed in tech stack but not yet in package.json), `mammoth` (new install — Story 17.5 adds this; install it here if 17.5 is not yet merged), `exceljs` (new install)
- Follows pattern: Existing upload flow — extraction is a post-upload enhancement, not a new upload path
- Touch points: `app/actions/files.ts` (trigger extraction after upload), `prisma/schema.prisma` (add field to WorkspaceFile)

**Note:** This story enhances the EXISTING `WorkspaceFile` system. It does not change the upload UX — extraction runs transparently after upload.

**Critical integration detail:** The existing `uploadFile(formData)` function in `app/actions/files.ts` constructs a `Buffer` internally (line ~283: `const buffer = Buffer.from(arrayBuffer)`) and uploads it to Supabase Storage, but does not return the buffer. The two wrapper functions `uploadFileAndLinkToTask(formData, taskId)` and `uploadFileAndLinkToListItem(formData, listItemId)` both delegate to `uploadFile()` — they do not have direct buffer access either. Therefore, extraction must either (a) re-download the file from Supabase Storage using the `storage_path`, or (b) be wired directly inside `uploadFile()` where the buffer is already in memory. Option (b) is preferred to avoid a redundant network round-trip.

## Acceptance Criteria

**Extraction:**

1. After a file is uploaded via existing `uploadFile()` action, text extraction runs automatically within the same server action
2. Extraction supports: PDF (via `pdf-parse`), .docx (via `mammoth`), .xlsx (via `exceljs` — cell values as text)
3. Extracted text stored in new `extracted_text` field on `WorkspaceFile` model (`String? @db.Text`)
4. Extraction is best-effort — if extraction fails (encrypted PDF, corrupt file), the file is still stored, `extracted_text` remains null, no error shown to user
5. Error details logged to console for debugging (not surfaced to user)

**Limits:**

6. Extracted text truncated at 100,000 characters (safety cap for very large files)
7. Images (PNG, JPG, GIF), PowerPoint (.pptx), and other unsupported types skip extraction silently
8. Files larger than 25MB skip extraction (they shouldn't exist given the upload limit, but defensive check)

**Schema:**

9. New nullable `extracted_text` field on `WorkspaceFile` model (additive migration, no impact on existing data)

**Re-extraction:**

10. `extractFileText(fileId)` server action available for manual re-extraction trigger (admin use or future re-processing)

**Integration Requirements:**

11. Existing upload flow, UI, and UX completely unchanged — user sees no difference
12. Extraction does not block the upload response — runs within the same action but after the file is stored and the DB record is created
13. `WorkspaceFile` records created before this story have `extracted_text = null` (no backfill required in this story)

**Quality Requirements:**

14. Extraction of a 50-page PDF completes within 5 seconds
15. Swedish characters extracted correctly (UTF-8)

## Dev Notes

**Actual uploadFile() structure (app/actions/files.ts):**

The existing `uploadFile(formData: FormData)` function (line 218) constructs the buffer in-memory:

```typescript
const arrayBuffer = await file.arrayBuffer()
const buffer = Buffer.from(arrayBuffer)  // ~line 283 — buffer available here
```

It then uploads to Supabase Storage and calls `prisma.workspaceFile.create(...)`. Extraction should be wired AFTER the `create` call succeeds, using the `buffer` variable already in scope. This avoids a re-download round-trip.

`uploadFileAndLinkToTask(formData, taskId)` and `uploadFileAndLinkToListItem(formData, listItemId)` are wrapper functions that call `uploadFile(formData)` internally — they will automatically benefit from the extraction wired into `uploadFile` without further modification.

**PDF extraction (`pdf-parse`):**

```typescript
import pdfParse from 'pdf-parse'

async function extractFromPdf(buffer: Buffer): Promise<string> {
  const result = await pdfParse(buffer)
  return result.text
}
```

**DOCX extraction (`mammoth`):**

```typescript
import mammoth from 'mammoth'

async function extractFromDocx(buffer: Buffer): Promise<string> {
  const result = await mammoth.extractRawText({ buffer })
  return result.value
}
```

**XLSX extraction (`exceljs`):**

```typescript
import ExcelJS from 'exceljs'

async function extractFromXlsx(buffer: Buffer): Promise<string> {
  const workbook = new ExcelJS.Workbook()
  await workbook.xlsx.load(buffer)
  const texts: string[] = []
  workbook.eachSheet((sheet) => {
    sheet.eachRow((row) => {
      const rowTexts = row.values
        ?.slice(1) // exceljs row.values has null at index 0
        ?.map((v) => String(v ?? ''))
        ?.filter(Boolean)
      if (rowTexts?.length) texts.push(rowTexts.join('\t'))
    })
  })
  return texts.join('\n')
}
```

**MIME type routing:**

```typescript
const EXTRACTORS: Record<string, (buf: Buffer) => Promise<string>> = {
  'application/pdf': extractFromPdf,
  'application/vnd.openxmlformats-officedocument.wordprocessingml.document': extractFromDocx,
  'application/vnd.openxmlformats-officedocument.spreadsheetml.sheet': extractFromXlsx,
  'application/msword': extractFromDocx, // .doc — mammoth may handle some
  'application/vnd.ms-excel': extractFromXlsx, // .xls — exceljs may handle some
}
```
