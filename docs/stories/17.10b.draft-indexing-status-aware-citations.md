## Story

**As an** owner editing my workspace's styrdokument,
**I want** the agent to search across drafts and in-review documents (not just approved ones), **but to clearly label non-approved content as a draft**,
**so that** I can ask questions while authoring, discover work-in-progress content, and trust that the agent will never quote a draft as canonical policy.

**Design decisions locked (2026-06-02):**

- **DEC-1: Single-current-version index.** Chunks stay keyed on `(source_type, source_id=document_id)`. Only the current version's content is chunked. Historical versions are audit data (`WorkspaceDocumentVersion` table), not RAG data. Rejected: multi-version chunk keying — quadratic growth, polluted results, weak user benefit. Compromise: chunk metadata carries `version_number` so citations can read `[Källa: X v3]` if rendered to surface it.
- **DEC-2: Content-hash, not status, is the reindex trigger.** Status transitions become metadata-only `UPDATE`s when content is unchanged. Content changes (DRAFT autosave OR explicit `saveDocumentVersion`) are the events that fire embedding work. This decouples lifecycle from index cost.
- **DEC-3: Citation label split — `[Källa:]` vs `[Utkast:]`.** Visible-by-design hedge. Display title stays clean per the 17.10 UX-call (only key/prefix changes). Agent system prompt teaches it to phrase non-APPROVED hits as *"enligt utkast till X..."*, not *"enligt X..."*.
- **DEC-4: 60s idle debounce on DRAFT autosave reindex.** Author UX accepts ~1min staleness on the agent's view of an actively-edited DRAFT. Embedding budget trade-off accepted by owner.
- **DEC-5: Cron sweep for the debounce mechanism (PO call 2026-06-02).** Server-side throttle (per-doc `last_reindex_at`) considered but rejected — cron sweep smooths embedding load across multi-author DRAFT editing (realistic for HR + compliance manager + safety officer all editing arbetsmiljöpolicy simultaneously) and reuses the established cron pattern (`/api/cron/process-chunks`, `/api/cron/extract-files`). New `/api/cron/sweep-draft-reindex` runs every 1 minute, picks docs where `needs_reindex = true AND last_marked_dirty_at <= NOW() - 60s`, calls `indexWorkspaceDocument`, clears the flag. Tradeoff accepted: +60s worst-case staleness vs Option A, in exchange for cost smoothing.
- **DEC-6: One-time backfill on rollout.** Idempotent script chunks every workspace's non-APPROVED-with-content doc. Cleaner UX than lazy-on-read (which would leave new agents context-empty until first query per doc). Owner-triggered run, not auto-fired by deploy.

## Acceptance Criteria

### Indexing rule changes

1. `decideReindexOnStatusChange` is **REPLACED** by content-aware triggers. Status transitions no longer drive INDEX/DEINDEX as the primary signal. The new helper surface in `lib/chunks/workspace-document-reindex.ts`:
   - `decideReindexOnStatusChange(oldStatus, newStatus): 'DELETE' | 'METADATA_UPDATE' | 'NONE'` — returns `'DELETE'` for transitions INTO `ARCHIVED` or `SUPERSEDED`; returns `'METADATA_UPDATE'` for any other transition (cheap UPDATE on `metadata.status`, no re-embed); returns `'NONE'` when status didn't actually change.
   - `decideReindexOnContentChange(oldHash, newHash): 'REINDEX' | 'NONE'` — returns `'REINDEX'` when hashes differ; `'NONE'` when identical (defensive — the inner `syncWorkspaceChunks` hash gate is the authoritative skip).
2. The "indexable set" expands from `{APPROVED}` to `{DRAFT, IN_REVIEW, APPROVED}`. `SUPERSEDED` and `ARCHIVED` remain non-indexable (chunks deleted on entry).
3. Chunk `metadata` gains two new keys:
   - `status` — the live status (`DRAFT` / `IN_REVIEW` / `APPROVED`)
   - `version_number` — the current version number at index time
   No schema migration (the field is `Json?`).
4. A new helper `updateChunkMetadata(sourceId, sourceType, partialMetadata)` lives in `lib/chunks/sync-workspace-chunks.ts` (sibling to `syncWorkspaceChunks`). Issues a single `UPDATE content_chunks SET metadata = jsonb_set(...) WHERE source_id = ? AND source_type = ?`. No embedding work. Returns `{ chunksUpdated, duration }`.

### New trigger hooks (in `app/actions/documents.ts`)

5. **`updateDocumentStatus`** hook (currently line 421) — replace the `INDEX | DEINDEX | NONE` branch with the new 3-way:
   - `DELETE` → `deindexWorkspaceDocument(documentId, workspaceId)` (existing helper)
   - `METADATA_UPDATE` → `updateChunkMetadata(documentId, 'WORKSPACE_DOCUMENT', { status: newStatus })` (new helper). Cheap UPDATE, no embedding cost.
   - `NONE` → no-op
   Continue to run inside `after()` for response-non-blocking.
6. **`autosaveDocument`** (line 509) — after the in-place `content_html` update, flip `needs_reindex = true` and `last_marked_dirty_at = NOW()` on the `workspace_document` row (the cron sweep mark — see Task 3). The 1-minute cron sweep is the consumer; this hook is fire-and-forget. The inner `syncWorkspaceChunks` hash gate (called by the sweep) skips no-op re-embeds for free. Skip the mark entirely when the doc's status ∈ `{SUPERSEDED, ARCHIVED}` (non-indexable — no point chunking). **Null `current_version` edge case:** if `current_version_id` is null (doc with no versions yet — pre-first-save state), the mark is a no-op. `indexWorkspaceDocument` already short-circuits on this case; documenting here so Dev doesn't add a redundant guard.
7. **`saveDocumentVersion`** (line 568+) — immediate (non-debounced) `indexWorkspaceDocument` call inside `after()`. Explicit checkpoint = real content change = embed now. Hash-gated.
8. ARCHIVED / SUPERSEDED transitions continue to DELETE chunks (covered by AC-1's `'DELETE'` branch). Document hard-delete continues to cascade via Prisma.

### Citation contract — status-aware labels

9. `lib/ai/citations.ts:208` `search_workspace_documents` branch — modify to read `item.status` (already present in tool result) and choose label prefix:
   - `status === 'APPROVED'` → `citationKey = ` `[Källa: <title>]` (current behavior preserved)
   - `status === 'DRAFT' || status === 'IN_REVIEW'` → `citationKey = ` `[Utkast: <title>]`
   - **The CITE-002 collision disambiguator applies to both labels.** A collision pair where both are DRAFT: `[Utkast: X (abcd1234)]` + `[Utkast: X (efgh5678)]`. Mixed APPROVED + DRAFT with the same title: `[Källa: X (abcd1234)]` + `[Utkast: X (efgh5678)]`. The `titleCounts` pre-scan groups by base title regardless of status (per DEC-3 — collision is per-title, not per-status-tier).
10. Display title stays clean — only the label prefix (`Källa:` vs `Utkast:`) changes. The chip rendering layer (`components/features/ai-chat/...`) reads the `citationKey` string and renders the prefix as part of the chip text — same component, same UX call as 17.10.
11. **Backwards-compatibility for chunks indexed before this story:** if `metadata.status` is missing (old chunk from 17.9b), default to `APPROVED` (the only status that could have been indexed under 17.9b). The backfill script (Task 6) rewrites these.

### System prompt — teach the agent to hedge

12. `lib/agent/system-prompt.md` line 95 — extend the `search_workspace_documents` bullet with a status-aware citation directive. Required Swedish text (substring, exact match in test):
    > Resultat har `status`: APPROVED (godkänt — gällande policy) eller DRAFT/IN_REVIEW (utkast — under arbete). Citera APPROVED-träffar som `[Källa: <titel>]`. Citera DRAFT/IN_REVIEW-träffar som `[Utkast: <titel>]` och hedra dem i texten: skriv "enligt utkast till …" eller "i utkastet till …" — ALDRIG "enligt …" eller "vår policy säger …" som om utkastet vore gällande policy.
13. Add a worked example in the prompt's "Korrekt exempel" section showing a mixed APPROVED + DRAFT response (one `[Källa: ...]` claim, one `[Utkast: ...]` claim with hedging).
14. `tests/unit/agent/system-prompt.test.ts` — new test `it('teaches status-aware hedging for workspace documents', ...)` asserts the substring from AC-12 is present.

### Debounce mechanism

15. Debounce implementation is **cron sweep** (DEC-5, locked by PO 2026-06-02). Migration adds two nullable columns to `workspace_documents`: `needs_reindex BOOLEAN NOT NULL DEFAULT FALSE` and `last_marked_dirty_at TIMESTAMP(3) NULL`. New cron route at `app/api/cron/sweep-draft-reindex/route.ts`, registered in `vercel.json` at `0 * * * *`-style 1-minute cadence. Sweep query: `SELECT id, workspace_id FROM workspace_documents WHERE needs_reindex = true AND last_marked_dirty_at <= NOW() - INTERVAL '60 seconds' LIMIT N`. For each picked row: call `indexWorkspaceDocument(id, workspace_id)`, then `UPDATE ... SET needs_reindex = false, last_marked_dirty_at = NULL`. Implemented in Task 3.
16. Rapid autosaves within a 60s window MUST NOT fire multiple reindexes. Test (in Task 3 / Task 7): 5 autosaves within 5 seconds (each calls the mark helper) produce at most one reindex per the next cron tick. Verifies the debounce, not just the hash gate.
17. Debounce window value (60s) is a named constant `DRAFT_REINDEX_DEBOUNCE_MS = 60_000` in `lib/chunks/workspace-document-reindex.ts` — single source of truth, easy to tune. The cron sweep reads this constant (or its derived seconds value) when computing the `NOW() - INTERVAL` predicate.

### Backfill script

18. `scripts/backfill-workspace-document-drafts.ts` — one-time, idempotent. Walks every workspace's `WorkspaceDocument` where `status ∈ {DRAFT, IN_REVIEW}` AND `current_version.content_html.length >= 100` (the same threshold the 17.10 seed script used). Calls `indexWorkspaceDocument` for each. Hash-gated re-runs return `skipped: true` for already-current chunks.
19. Script also walks existing APPROVED chunks and **backfills missing `status` + `version_number` metadata** via `updateChunkMetadata` — handles the AC-11 backwards-compat path so old 17.9b chunks get explicit `status: 'APPROVED'` rather than relying on the citation-layer default.
20. Script logs per-workspace and per-doc outcomes (chunks created / embedded / skipped / metadata-only-updated). Final summary: total chunks created, total embedding tokens spent, wall-clock duration.

### Adversarial tests — the load-bearing safety check

21. `tests/unit/lib/ai/citations.test.ts` — new tests covering the status-aware label branch:
    - status APPROVED → `citationKey` starts with `[Källa:`
    - status DRAFT → `citationKey` starts with `[Utkast:`
    - status IN_REVIEW → `citationKey` starts with `[Utkast:`
    - missing status (legacy chunk) → defaults to `[Källa:` (AC-11)
    - Mixed APPROVED + DRAFT same title → both prefixes + collision suffix on both
22. `tests/integration/agent/draft-citation-safety.test.ts` (new file) — behavioral tests via the chat path with seeded chunks:
    - **Seed 1 (DRAFT-only):** workspace has one DRAFT styrdokument. Agent receives a query that matches it. Assertion: response includes `[Utkast: ...]` (never `[Källa: ...]`) AND includes one of the hedge phrases (`enligt utkast`, `i utkast`, `ännu inte godkänd`).
    - **Seed 2 (Mixed canonical + draft):** workspace has APPROVED policy A on topic T, and DRAFT revision A' on the same topic. Agent receives a query about T. Assertion: response cites A as `[Källa: A]` canonically AND, if it surfaces A', cites it as `[Utkast: A']` with a hedge ("ett utkast föreslår", "ett pågående utkast nämner", etc.).
    - **Seed 3 (Adversarial prompt):** user asks "Vad är vår officiella policy?" against a DRAFT-only workspace. Assertion: agent does NOT pretend the draft is the official policy. Acceptable responses: "Ni har bara ett utkast — ingen godkänd policy ännu", or quotes the draft hedged as `[Utkast: ...]`. Unacceptable: any sentence citing the draft as `[Källa: ...]` or phrased as "Er policy säger …".
23. The integration tests use the real chat pipeline (mock the embedding API but exercise the real citation extractor + system prompt). Pattern mirrors existing `tests/integration/` if any; otherwise propose a new test harness in Task 7 with PO sign-off.

### Cross-tenant isolation (defence-in-depth)

28. Every new code path that reads or writes `content_chunks` MUST include `workspace_id` in its filter/scope — alongside the existing closure-captured `workspaceId` and the 17.9b write-side invariant. Specifically:
    - **`updateChunkMetadata` (AC-4):** the `UPDATE content_chunks SET metadata = jsonb_set(...)` MUST include `WHERE source_id = ? AND source_type = ? AND workspace_id = ?` (all three predicates). Test: insert chunks for the same `source_id` in workspace A and workspace B; call `updateChunkMetadata(sourceId, sourceType, {...}, workspaceA)`; assert workspace A's chunks were updated, workspace B's were NOT.
    - **Cron sweep route (AC-15):** the per-doc index call passes the doc's `workspace_id` through to `indexWorkspaceDocument`, which already enforces it. Test: queue a dirty doc; verify the sweep's `indexWorkspaceDocument` call receives the correct workspace_id and the resulting chunk INSERTs all carry it.
    - **Backfill script (AC-18):** walks ordered by `workspace_id` and passes it through to every `indexWorkspaceDocument` / `updateChunkMetadata` call. Test: dry-run mode prints the planned (workspace_id, document_id) pairs; verifies no chunk operation is workspace-scope-less.

    This is defence-in-depth — the existing layers (closure capture + write-side invariant) already enforce isolation, but a "cheap UPDATE" path is the easiest place for a future refactor to drop the `workspace_id` predicate by accident. The AC + tests lock it in.

## Dev Notes

**Critical implementation pin — don't break DEC-2 by accident:**

The whole point of this story is to extend indexing while preserving the citation-safety contract. The agent must NEVER quote DRAFT content with `[Källa: ...]` framing. The defence layers:

1. **Format layer** (citation extractor) — `citationKey` is auto-prefixed with `[Källa:]` or `[Utkast:]` based on chunk metadata. The agent picks the citationKey from the tool result; it can't construct a `[Källa:]` for a DRAFT chunk because the citationKey it sees IS `[Utkast: ...]`.
2. **Prompt layer** (system prompt) — explicit Swedish directive: APPROVED → Källa, DRAFT/IN_REVIEW → Utkast + hedge.
3. **Behavioral layer** (integration tests) — adversarial probes (AC-22) verify the agent actually hedges in practice, not just that the format layer works in isolation.

**Hash gate composition — don't double-fire:**

The existing `syncWorkspaceChunks` hash gate (`sync-workspace-chunks.ts:98-108`) already prevents wasted re-embeds when content didn't change. The new `decideReindexOnContentChange` helper is **defensive** — it's mostly informational, helping callers skip even queueing the work. The authoritative gate is still inside `syncWorkspaceChunks`. Don't introduce a code path that bypasses it.

**Why ARCHIVED is a hard-delete, not just "hide from search":**

Could we keep chunks for ARCHIVED docs and just filter at query time? Yes — but every retrieval query would pay a filtering cost forever, and there's a real risk of a filter bug surfacing archived content to users. ARCHIVED = chunks gone, no ambiguity. Same for SUPERSEDED. If we ever need "show me what was archived in March 2025," that's a job for the version table + a forensic query, not the search index.

**The in-place version mutation hidden state (VIM-001):**

Today's autosave at `documents.ts:511` does `prisma.workspaceDocumentVersion.update()` — mutates `content_html` on the existing version row. A v1 row after 50 autosaves contains the 50th-save content, not the original v1 content. This story DOES NOT FIX THAT — owner deferred. Implication for our hash gate: we hash `current_version.content_html` at index time, which is whatever was last saved. The semantic is "the chunk reflects what the doc currently says," which is correct for RAG. The audit semantic ("what did v1 say") is broken, but that's `VIM-001`'s problem.

**Why no version-aware retrieval (DEC-1):**

Three reasons we explicitly chose single-current:
1. Quadratic chunk growth: 10 docs × 5 avg versions = 50× chunk volume. Embedding cost spirals.
2. Stale-version pollution: search would surface v2's content years after v3 is canonical. User would have to mentally filter "is this current?"
3. Misaligned user model: users want "what does our policy say NOW." Audit needs are forensic, not exploratory — they belong in a dedicated audit UI over the version table, not in the RAG layer.

**Compromise:** chunk `metadata.version_number` is recorded at index time so a future story (or this one's stretch) can render `[Källa: Diskrimineringspolicy v3]` if precision is wanted. Not required for AC.

**Null `current_version_id` edge case (NTH-1):**

A `WorkspaceDocument` row can technically exist with `current_version_id = NULL` (pre-first-save state, or a transient transactional moment). The existing `indexWorkspaceDocument` short-circuits returning `void` when `doc.current_version` is null (`workspace-document-reindex.ts:62`). The new `markWorkspaceDocumentDirty` helper (Task 3) and the cron sweep both inherit this safety — calling them on a null-version doc is a no-op rather than an error. **Dev: do NOT add a redundant pre-check** in the autosave trigger; let the existing helper short-circuit. One layer of guard is sufficient and easier to maintain than two.

**Concurrent autosave + status change race (NTH-2):**

A user clicks "Submit for review" (`updateDocumentStatus`: DRAFT → IN_REVIEW) while an autosave is mid-flight (or pending in the 60s debounce window). Possible orderings:

1. Autosave's `markWorkspaceDocumentDirty` runs first, then status change's `updateChunkMetadata` runs. Result: chunks have `status: IN_REVIEW` (correct, latest); cron sweep later re-embeds with the same `IN_REVIEW` status (idempotent). ✅
2. Status change's `updateChunkMetadata` runs first (chunks get `status: IN_REVIEW`), then autosave's `markWorkspaceDocumentDirty` runs (queues a reindex). Cron sweep re-embeds with current content + current status (`IN_REVIEW`). ✅
3. Both metadata writes interleave on the same chunk rows (Postgres serializes at row level — last write wins). Whichever ran second leaves its `status` value. Since they both set the same target value (`IN_REVIEW`), either order is correct. ✅

All three orderings converge on the correct final state. **No lock or transaction needed.** This is a benefit of the metadata-UPDATE + content-REINDEX split — the two events touch different "axes" of the chunk state.

Edge case worth noting: if the status change is to a non-indexable state (ARCHIVED / SUPERSEDED) while a reindex is queued in the cron sweep, the reindex will run, then the DELETE will run, leaving the chunks correctly deleted. The intermediate state (~60s of stale chunks pointing at a now-archived doc) is acceptable for v1; the search tool filters out archived/superseded at the agent layer regardless. If this becomes an observed problem, add a pre-reindex status check in the sweep (cheap).
