# Story 7.10: Employee salary (Lön) + AI löneavtal-compliance

## Status

Done

## Story

**As an** HR manager,
**I want** to record what each employee is paid and ask the AI whether it aligns with our kollektivavtal,
**so that** I can check minimilön/löneavtal compliance without consulting outside help.

## Context

7.1 deliberately excluded salary amounts as "accounting-only / Fortnox's system-of-record." This story reverses that for one reason: **kollektivavtal set minimilöner (by personaltyp/role/experience — those tables are already ingested as RAG chunks), and "betalar vi enligt avtalet?" is unanswerable without the actual salary.** Salary is therefore load-bearing for löne-compliance, in scope. `SalaryForm` (MAN/TIM) already exists on the model; this adds the amounts.

## Acceptance Criteria

1. **Schema:** `monthly_salary` + `hourly_pay` added to `Employee`, **encrypted at rest** (TEXT ciphertext columns, mirroring `personnummer` — AES-256-GCM via `encryptField`; NOT plaintext Decimal). Migration authored, **applied manually by the user**. Documented consequence: encrypted ⇒ no numeric DB aggregation (fine — the AI does comparisons; salary-analytics is a separate future concern).
2. **Role-gated everywhere:** salary is decrypted only for `employees:manage`; `employees:view`-only roles get a mask/null and never receive the ciphertext or plaintext. This is stricter than personnummer's read gate is applied — salary is manage-level for both UI display AND the AI path.
3. **Personalkort:** Månadslön / Timlön input(s) on the Anställning tab, shown conditionally on the Löneform (`salary_form`) select (MAN → Månadslön, TIM → Timlön; both hidden until a form is picked). Encrypt-on-write **three-state contract** (undefined = keep stored ciphertext / '' = clear → null / value = validate + encrypt + set), identical to personnummer. Read-only + masked for view roles.
4. **Register:** a **Lön column, hideable and HIDDEN BY DEFAULT** (`defaultVisible: false` — a stronger screen-share-privacy default than personnummer). The repository (`toView`) strips/masks salary for non-manage roles **before** the row serializes to the client — the column must not ship salary to a view-only browser regardless of the visibility toggle.
5. **AI löne-compliance:** a new `get_employee_salary` agent tool, **registered ONLY when the session role holds `employees:manage`** (conditional registration — a `view`-only agent has no tool at all). Returns the decrypted salary + `salary_form` + `personel_type` + assigned agreement `{id,name}` for one employee (defaulting to the pill-selected employee via a closure, or an explicit `employeeId` param from a `lookup_employee` result). It composes with `search_collective_agreements` (agent fetches salary → searches the assigned avtal's minimilön clause → compares). **Salary NEVER enters `formatEmployeeContext` or `lookup_employee`** — those stay salary-free; their PII-absence tests are extended to assert salary absence too.
6. **Fortnox mapper (7.8):** `MonthlySalary` / `HourlyPay` mapped (Fortnox **strings**, via `toDecimalString`). The mapper stays a pure, crypto-free function operating on plaintext numbers; the future sync job owns encrypt/decrypt at its boundary (same contract as personnummer, doc-commented).
7. **Not a completeness criterion:** `assessEmployeeCompleteness` is unchanged — LAS-completeness does not require a salary; löne-compliance is its own layer.

### Traps (verified — do not ignore)

1. **[SECURITY] Permission asymmetry — the AI tool must gate on `employees:manage`, NOT `:view`.** The 7.7 AI paths gate on `employees:view` (chat route L272, lookup registration). Salary is manage-level in the UI. If `get_employee_salary` were registered on `:view`, a view-only role's agent could extract salary the UI masks. Gate the tool on `employees:manage`. (Sweep sharp-edge #5.2.)
2. **[SECURITY] Register column leaks salary to view browsers unless the repo gates first.** The register serializes rows to the client for all `employees:view` users. `toView` MUST strip/mask salary for non-manage roles BEFORE serialization — column-visibility is display-only, not an access control. (Sweep sharp-edge #8.)
3. **Encrypt ⇒ TEXT columns, not Decimal.** Salary is stored as an encrypted string (decimal-string plaintext → ciphertext). No `@db.Decimal`, no numeric SQL. The Fortnox mapper stays pure (operates on decrypted numbers; crypto lives in the future sync job — same as personnummer).
4. **Default-hidden needs seeding.** A `defaultVisible: false` salary column won't hide itself — the sanitizer defaults unknown ids to visible. Seed the hidden default from the options layer and test first-load-hidden.
5. **Salary in the model transcript.** `get_employee_salary` results persist in conversation history. Acceptable (it's an explicit, manage-gated, auditable tool call) — but do NOT also route salary through the always-on context block (which would put it in every prompt/transcript for that chat).
6. **Masked-prefill three-state** (like personnummer): a masked salary field submitting empty must KEEP the stored ciphertext (undefined → omit), never null it.

### Design decision (encrypt vs plaintext-role-gate) — flagged for the checkpoint

Chosen: **encrypt at rest**, for consistency with `personnummer` in the same sensitive-PII module and defense-in-depth on comp data; the numeric-analytics loss doesn't hurt the compliance use case. If a future salary-analytics/register-filter feature is planned, plaintext-Decimal + role-gate is the alternative — revisit then. (PO/user may flip at the checkpoint before dev.)

### Out of scope

Salary-analytics/aggregation, a "löne-compliance status" badge on the register (future — this story gives the AI answer, not a computed per-row verdict), bulk salary import, tier-gating (Story 7.9), the register-wide turordning tool.

### Testing
