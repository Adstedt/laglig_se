# Story 16.5: Request to Join Existing Workspace

## Status

Draft

## Dependencies

- None (standalone, but complements Epic 16 onboarding improvements)

## Story

**As a** new user signing up with an org number that already belongs to an existing workspace,
**I want** to request access to that workspace instead of hitting a dead-end error,
**so that** I can join my company's existing workspace without requiring the admin to pre-invite me.

## Context

Currently, when a user creates a workspace with an org number already registered, the system returns "Detta organisationsnummer ar redan registrerat" with no path forward. The user must somehow contact an admin they may not know about, or abandon onboarding. This story replaces the dead-end with a "request to join" flow.

## Acceptance Criteria

### Detection & Join Request UI

1. When `createWorkspace()` detects a duplicate org number (Prisma P2002 on `org_number`), instead of only returning an error string, also return the existing workspace's `id`, `name`, and `company_legal_name`
2. The onboarding confirm step renders a dedicated panel when a duplicate is detected, showing: company name, org number, and a "Skicka forfragan" (Send request) button
3. The panel also shows a secondary action: "Skapa workspace utan organisationsnummer" (Create workspace without org number) — allowing the user to proceed with a personal/unnamed workspace if desired
4. The "Skicka forfragan" button calls a new server action `requestWorkspaceAccess(workspaceId)` that creates a `WorkspaceJoinRequest` record

### Data Model

5. New Prisma model `WorkspaceJoinRequest` with fields: `id`, `user_id`, `workspace_id`, `status` (enum: `PENDING`, `APPROVED`, `DENIED`), `message` (optional text from requester), `decided_by` (nullable user id), `decided_at` (nullable datetime), `created_at`
6. Unique constraint on `[user_id, workspace_id]` with status `PENDING` — prevent duplicate pending requests from the same user

### Admin Notification

7. On join request creation, send an email notification to all workspace members with role `OWNER` or `ADMIN` containing: requester name, requester email, and a link to the workspace settings/members page
8. On join request creation, create an in-app `Notification` record for each `OWNER`/`ADMIN` member (follow existing notification patterns)

### Admin Approval/Denial

9. New section in workspace settings members page: "Forfragan om atkomst" (Access requests) — visible only to `OWNER`/`ADMIN` roles
10. Each pending request shows: requester name, email, request date, and "Godkann" / "Neka" buttons
11. "Godkann" calls `approveJoinRequest(requestId)` server action which: updates request status to `APPROVED`, creates a `WorkspaceMember` with role `MEMBER`, sends approval email notification to the requester, creates in-app notification for requester
12. "Neka" calls `denyJoinRequest(requestId)` server action which: updates request status to `DENIED`, sends denial email notification to the requester

### Requester Experience

13. After sending a join request, the onboarding page shows a waiting state: "Din forfragan har skickats till administratoren for [company name]. Du far ett meddelande nar den har behandlats."
14. The waiting state includes the secondary action to create a workspace without org number (same as AC 3)
15. If the request is approved while the user is on the waiting page, redirect to the workspace dashboard (poll `GET /api/workspace/join-request-status` every 5 seconds, or use existing notification patterns)
16. If the user returns to the app after approval (different session), the onboarding page detects existing workspace membership and redirects to dashboard

### Edge Cases

17. If the user already has a pending request for this workspace, show "Forfragan redan skickad" instead of the send button
18. If the workspace is not `ACTIVE` (paused/deleted), do not show the join request option — fall back to the existing error behavior
19. Rate limit: max 3 join requests per user per hour across all workspaces
