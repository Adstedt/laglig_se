# Story 17.16: Dual-Version Document Model + Dispatch Refactor (Foundation)

## Status

Approved

> 🆕 **Brownfield foundation (PO drafted 2026-06-03, Sarah; PO-approved 2026-06-03 after v1.1 revision + re-validation GO 9/10).** First of three coordinated stories under `docs/prd/epic-17-addendum-dual-version-visibility.md`. Establishes the dual-pointer schema (`current_approved_version_id` + `current_draft_version_id` + `draft_status`) on `WorkspaceDocument` and per-version audit timestamps on `WorkspaceDocumentVersion`. Refactors `createDraftFromApproved` to populate the draft pointer without touching approved metadata or deindexing. Introduces `promoteDraftToApproved` and `discardDraft` server actions. Reframes the agent write-tool guards in Stories 17.11 / 17.11b against the new pointer model (observationally equivalent for all cases the existing tests cover). Stories 17.17 (UX) and 17.18 (agent reads + citations) build on this foundation. Story 17.11c (agent auto-branch on APPROVED) is a separate follow-up that becomes trivial once this lands.

## Story

**As a** compliance officer maintaining a styrdokument,
**I want** the system to encode the currently-effective approved version AND the currently-in-progress draft as concurrent first-class states with stable audit metadata,
**so that** the live policy remains visible, searchable, and citable throughout any revision window — and so the agent's write guards gate on draft-presence rather than the misleading top-level status flip.

## Story Context

**Existing System Integration (load-bearing — story 17.16 is brownfield-foundation):**

- **`WorkspaceDocument` schema (current state):** single `current_version_id` (`String?`, `@unique`) pointer + single `status` enum (`DRAFT` / `IN_REVIEW` / `APPROVED` / `SUPERSEDED` / `ARCHIVED`) + `current_version_number` int + lifecycle metadata (`approved_by`, `approved_at`, `review_date`, `retention_until`). See `prisma/schema.prisma:2027–2080`. `WorkspaceDocumentStatus.SUPERSEDED` exists in the enum but is unused by any code path today.
- **`WorkspaceDocumentVersion` schema (current state):** immutable + append-only. Carries `version_number`, `source` (`TIPTAP` / `AGENT` — `AGENT` currently unused), `content_json`, `content_html`, `extracted_text`, `change_summary`, `created_by`, `created_at`. NO per-version approval metadata today — approvals are tracked solely via the parent doc's `approved_by`/`approved_at` (which `createDraftFromApproved` currently NULLs on branch — bug).
- **`createDraftFromApproved` (`app/actions/documents.ts:823`):** today's behavior — on a `status=APPROVED` doc: creates new version (number = current + 1, content copied), flips `status: APPROVED → DRAFT`, NULLs `approved_by`/`approved_at`, runs `after(deindexWorkspaceDocument(...))`, writes `document_status_changed` ActivityLog row. This story refactors that function to leave approved metadata + index intact and populate the new draft pointer instead.
- **`saveDocumentVersion` (`app/actions/documents.ts:590`):** today's behavior — appends a new version, bumps `current_version_*`, writes `document_version_saved` ActivityLog row, triggers `indexWorkspaceDocument` via `after()`. Used by editor autosave + agent dispatch paths (Stories 17.11 / 17.11b). This story extends it: when `current_draft_version_id` is set on the parent doc, the new version is written as a draft update (draft pointer advances, top-level status untouched).
- **Agent write-tool guards (Stories 17.11 / 17.11b — Done / Ready for Review):**
  - `lib/agent/tools/update-document.ts:170` — tool-time refuses when `document.status !== 'DRAFT' && document.status !== 'IN_REVIEW'`.
  - `lib/agent/tools/add-document-section.ts` (17.11b) — same shape.
  - `app/actions/pending-agent-actions.ts:496` (UPDATE_DOCUMENT case) — dispatch-time re-asserts the same guard.
  - `app/actions/pending-agent-actions.ts` (ADD_DOCUMENT_SECTION case, added by 17.11b) — same shape.
  - Story 17.16 reframes these to gate on draft-presence (`current_draft_version_id IS NOT NULL OR top-level status is "never-approved DRAFT"`). The reframe is **observationally equivalent for the cases the existing tests cover** (DRAFT/IN_REVIEW → write allowed, APPROVED/SUPERSEDED/ARCHIVED → write refused). The new model additionally allows the "APPROVED with current_draft_version_id" case, but that case is not yet reachable via any UI/agent path in this story — Story 17.11c will introduce it. Tests in this story extend rather than replace.
- **Approval workflow (current state):** approval is implicit — the doc's `status` field flips `IN_REVIEW → APPROVED` via `updateDocumentStatus` (`app/actions/documents.ts`), which stamps `approved_by`/`approved_at` on the doc. There is no atomic "promote this version to approved" action today. This story introduces `promoteDraftToApproved` as the explicit atomic transition; existing direct `updateDocumentStatus({newStatus: 'APPROVED'})` callers continue to work for never-had-a-draft promotions.
- **17.10b auto-reindex contract:** `saveDocumentVersion` wraps `indexWorkspaceDocument` in `after()` — fire-and-forget, hash-gated. Today this indexes whatever `current_version_id` points at. **Story 17.18 refactors the indexed-content source to track `current_approved_version_id` as the canonical entry**; this story keeps the index source unchanged (still tracks `current_version_id`) to keep the foundation cleanly separable. The deprecated alias makes that work transparently.

**Reuse from existing infrastructure (no re-implementation needed):**
- Prisma migration template from `prisma/migrations/20260603100000_add_update_document_pending_type/migration.sql` and `20260604100000_add_add_document_section_pending_type/migration.sql` — additive enum + nullable column pattern.
- `withWorkspace` wrapper from `lib/auth/workspace-context.ts` — workspace-scoped server actions.
- Manual-migration workflow per project convention — generate `--create-only`, hand the SQL to the user, never auto-apply.

**Out of scope (deferred to sibling stories):**
- All UI surfaces (table badges, doc page header, editor banner, discard-draft button) — Story 17.17.
- Agent read tool refactor (`search_workspace_documents`, `get_workspace_document`, `list_workspace_documents`) + citation grammar + auto-reindex source change — Story 17.18.
- Agent auto-branch on APPROVED (creating a draft transparently on `add_document_section`/`update_document` against an APPROVED doc) — Story 17.11c (separate follow-up).
- Dropping the deprecated `current_version_id` column — follow-up cleanup once telemetry confirms no callers remain.
- True supersession (one policy fully retired and replaced by a structurally different one) — separate concept; out of scope.

## Acceptance Criteria

### Schema

1. **`WorkspaceDocument` extended with the dual-pointer columns + draft sub-status:**
   ```prisma
   model WorkspaceDocument {
     // … existing fields …
     current_approved_version_id String?      @unique
     current_draft_version_id    String?      @unique
     draft_status                DraftStatus?

     current_approved_version    WorkspaceDocumentVersion?
       @relation("WorkspaceDocumentCurrentApprovedVersion",
                 fields: [current_approved_version_id], references: [id])
     current_draft_version       WorkspaceDocumentVersion?
       @relation("WorkspaceDocumentCurrentDraftVersion",
                 fields: [current_draft_version_id],    references: [id])
     // existing `current_version_id` / `current_version` relation stays in place
     // throughout this epic (deprecated alias). Drop in follow-up.

     @@index([workspace_id, current_draft_version_id])  // for "docs with draft pending" queries
   }

   enum DraftStatus {
     DRAFT
     IN_REVIEW
   }
   ```
   `@unique` constraints on both new pointer columns mirror the existing `current_version_id @unique` shape (a version row can only be "the current X" for at most one doc). `DraftStatus` is a new enum, NOT a reuse of `WorkspaceDocumentStatus` (the doc-level status retains its existing meaning for now; deprecation of overlapping enum values is out of scope here).

2. **`WorkspaceDocumentVersion` extended with per-version audit timestamps:**
   ```prisma
   model WorkspaceDocumentVersion {
     // … existing fields …
     approved_at   DateTime?  // populated iff this version was ever promoted to approved
     approved_by   String?
     superseded_at DateTime?  // populated iff a later version replaced this one as approved
     approver      User?       @relation("WsDocVersionApprover", fields: [approved_by], references: [id])

     @@index([document_id, approved_at])  // for "what was approved on date X" queries
   }
   ```
   Nullable on all three. Populated on the `promoteDraftToApproved` path (AC 6). Backfill (AC 3) stamps historical approvals from `ActivityLog` where possible.

3. **Migration (`prisma migrate --create-only`):** Single migration file at `prisma/migrations/20260605100000_add_dual_version_pointers/migration.sql` (timestamp after 17.11b's `20260604100000_*`). Migration body:
   - `CREATE TYPE "DraftStatus" AS ENUM ('DRAFT', 'IN_REVIEW');`
   - `ALTER TABLE "workspace_documents" ADD COLUMN "current_approved_version_id" TEXT;`
   - `ALTER TABLE "workspace_documents" ADD COLUMN "current_draft_version_id" TEXT;`
   - `ALTER TABLE "workspace_documents" ADD COLUMN "draft_status" "DraftStatus";`
   - `ALTER TABLE "workspace_document_versions" ADD COLUMN "approved_at" TIMESTAMP(3);`
   - `ALTER TABLE "workspace_document_versions" ADD COLUMN "approved_by" TEXT;`
   - `ALTER TABLE "workspace_document_versions" ADD COLUMN "superseded_at" TIMESTAMP(3);`
   - Unique indexes on the two pointer columns; FK constraints; per-AC-1 secondary index.
   - **Backfill SQL** (idempotent, in a single transaction with assertion at the end) — **two-pass operation** to handle multi-save draft histories correctly:

     **Pass 1 — Per-version `approved_at` / `approved_by` stamping from ActivityLog (independent of doc state):**
     - For each `WorkspaceDocumentVersion` row `v`, find the `ActivityLog` row matching `entity_type = 'workspace_document' AND entity_id = v.document_id AND action = 'document_status_changed' AND new_value->>'status' = 'APPROVED'` whose `created_at` falls in the interval `[v.created_at, next_version.created_at)` (or `[v.created_at, NOW())` for the latest version of the doc). The "next version" is the version row for the same doc with `version_number = v.version_number + 1` — or NULL if `v` is the latest. If a matching ActivityLog row is found, stamp `v.approved_at = ActivityLog.created_at` and `v.approved_by = ActivityLog.user_id`.
     - For each version `v` that has `approved_at IS NOT NULL` AND a later version `v2` for the same doc also has `approved_at IS NOT NULL`: stamp `v.superseded_at = v2.approved_at`.
     - This produces an accurate per-version audit trail of approval history independent of which version is "current" at backfill time.

     **Pass 2 — Pointer population from stamped timestamps:**
     - **Case (a)** — `status = 'APPROVED'`: `UPDATE workspace_documents SET current_approved_version_id = current_version_id WHERE status = 'APPROVED';`. Additionally, if the current version's `approved_at` is null (the ActivityLog correlation didn't catch this approval — e.g., legacy data), stamp `current_version.approved_at = doc.approved_at` and `current_version.approved_by = doc.approved_by` so Pass 1 + Pass 2 produce a complete per-version audit history.
     - **Case (b)** — `status ∈ ('DRAFT', 'IN_REVIEW')` AND some version row for the doc has `approved_at IS NOT NULL`: `current_approved_version_id = the version row with the most-recent approved_at`; `current_draft_version_id = current_version_id`; `draft_status = (status = 'DRAFT' ? 'DRAFT' : 'IN_REVIEW')`. **Note:** the "most-recent approved" version may be MANY version_numbers behind `current_version_number` because editor autosaves between branch and backfill have created additional draft version rows; that's the case `version_number = current_version_number - 1` arithmetic would silently mis-handle, which the timestamp-driven approach correctly addresses.
     - **Case (c)** — `status ∈ ('DRAFT', 'IN_REVIEW')` AND NO version row has `approved_at IS NOT NULL` (never-approved doc): `current_draft_version_id = current_version_id, draft_status = ('DRAFT'/'IN_REVIEW' per status), current_approved_version_id = null`. Top-level `status` stays.
     - **Case (d) — archived / superseded** — `status ∈ ('ARCHIVED', 'SUPERSEDED')`: set `current_approved_version_id = current_version_id` (the version that was in effect at archive time is what the historical record points at); leave `current_draft_version_id` and `draft_status` null. (Archived/superseded docs are out of active editing scope; this just preserves the "what was effective when archived" anchor.)

     **Assertion** (refuses commit if violated):
     ```sql
     -- (1) No doc with a populated alias has both new pointers null:
     SELECT COUNT(*) FROM workspace_documents
       WHERE current_version_id IS NOT NULL
         AND current_approved_version_id IS NULL
         AND current_draft_version_id IS NULL;
       -- Must be 0.

     -- (2) Case-(b)-eligible docs have BOTH pointers populated:
     SELECT COUNT(*) FROM workspace_documents
       WHERE status IN ('DRAFT', 'IN_REVIEW')
         AND EXISTS (
           SELECT 1 FROM workspace_document_versions v
           WHERE v.document_id = workspace_documents.id
             AND v.approved_at IS NOT NULL
         )
         AND (current_approved_version_id IS NULL
              OR current_draft_version_id IS NULL);
       -- Must be 0.
     ```
     If either count is non-zero, `RAISE EXCEPTION 'Backfill incomplete: <details>'` aborts the transaction.
   - Backfill is **idempotent** — running it twice is safe (each UPDATE uses `WHERE` conditions that exclude already-populated rows; per-version stamping is safe because it only fills nulls).
   - Migration is **manual-apply** per project convention. NOT auto-applied; SQL handed to user / ops.
   - **Pre-backfill verification step (Task 1):** before writing the backfill SQL, `grep -A 20 "document_status_changed" app/actions/documents.ts` to confirm the actual `old_value` / `new_value` JSON shape that `updateDocumentStatus` writes. If the shape differs from `{status: 'X'}` (e.g., it's wrapped in another object), update the JSON-path expressions in the Pass-1 backfill SQL accordingly. The backfill is a one-shot data migration; getting the JSON path wrong silently mis-stamps production data, so the grep-then-write order is mandatory.
   - **Warning enumeration:** Pass-1 rows where ActivityLog correlation fails (no clean match in the version's time interval) should be enumerated post-migration via a one-line SQL: `SELECT id, document_id, version_number FROM workspace_document_versions WHERE approved_at IS NULL AND document_id IN (SELECT id FROM workspace_documents WHERE status = 'APPROVED');`. Engineer hand-inspects these (typically a tiny set on a brownfield codebase) and either stamps them manually from the doc's `approved_at`/`approved_by` or files a follow-up ticket.

### Server-action refactors

4. **`createDraftFromApproved` refactor (`app/actions/documents.ts:823`):** new behavior on a `status=APPROVED` doc:
   - Create the new `WorkspaceDocumentVersion` row exactly as today (number, content, change_summary).
   - `prisma.workspaceDocument.update({ data: { current_draft_version_id: new.id, draft_status: 'DRAFT', current_version_number: new.number } })` — populate the draft pointer, set `draft_status: DRAFT`, advance the `current_version_number` denormalized "highest version" counter. **Do NOT touch `current_version_id` (the deprecated alias) — it must continue to point at the approved version throughout the draft window so the 17.10b auto-reindex contract keeps indexing approved content.** **Do NOT touch `status`. Do NOT NULL `approved_by`/`approved_at`. Do NOT call `deindexWorkspaceDocument`.**
   - **The deprecated alias semantics under Model B:** `current_version_id` continues to point at the **last approved version** during a revision window (or at the lone draft version for never-approved docs, where no approved exists). This is the load-bearing change: the alias's job in the foundation story is to preserve compliance reads (auto-reindex source = approved content) WITHOUT requiring 17.18's full read refactor to ship first.
   - ActivityLog: write a `document_draft_created` row (NEW action name) with `new_value: { draft_version_id: new.id, draft_status: 'DRAFT', source_approved_version_id: doc.current_approved_version_id }`. The legacy `document_status_changed` row is NOT written by this path anymore — drafts no longer flip the top-level status.
   - Refuses if doc has `current_draft_version_id IS NOT NULL` (a draft is already in progress) — Swedish error: `"Ett utkast pågår redan för det här dokumentet. Slutför eller förkasta det innan ni skapar ett nytt."`. Refuses if doc is not `status=APPROVED` (today's check) — unchanged Swedish error.

5. **`saveDocumentVersion` refactor (`app/actions/documents.ts:590`):** routing logic depends on whether a draft is in progress:
   - **Path A — draft in progress** (`current_draft_version_id IS NOT NULL`): create the new version row (today's behavior), then update parent: `current_draft_version_id = new.id`, `current_version_number = new.number`. **`current_version_id` (the deprecated alias) is NOT touched** — it stays pointing at the approved version so the 17.10b auto-reindex continues to index approved content. `current_approved_version_id` is NOT touched. `status` is NOT touched.
   - **Path B — never-approved doc** (`current_draft_version_id IS NULL` AND `status IN ('DRAFT', 'IN_REVIEW')` AND `current_approved_version_id IS NULL`): today's flow unchanged — create the version row, advance `current_version_id` (alias) AND `current_draft_version_id` together (they point at the same row for never-approved docs), advance `current_version_number`. The alias remains useful for legacy reads of never-approved DRAFTs.
   - **Path C — defensive guard** (`current_draft_version_id IS NULL` AND `status = 'APPROVED'`): refuse with Swedish error `"Det godkända dokumentet kan inte ändras direkt. Skapa ett utkast först."`. (Defends against a misuse where editor autosave somehow targets an APPROVED-no-draft doc directly.)
   - ActivityLog: `document_version_saved` row continues to be written exactly as today. The agent-authorship stamp from Stories 17.11/17.11b (`by: 'agent', pendingActionId, operation`) continues to work — it patches the version row's `new_value` after creation.
   - **Why the asymmetric alias treatment (advance for Path B, freeze for Path A):** during a revision window (Path A), the alias's freeze keeps `[Källa:]` citations grounded in approved content — the 17.10b DEC-2 contract is preserved at the foundation layer without requiring 17.18's full read refactor. For never-approved docs (Path B), there's no approved version to protect, so the alias gracefully degrades to the draft for legacy callers. Story 17.18 removes this asymmetry by migrating every read site to the explicit pointers and dropping the alias entirely.

6. **New `promoteDraftToApproved(documentId)` server action** (`app/actions/documents.ts`, new export):
   - Workspace-scoped via `withWorkspace`. **Gated on `tasks:edit`** — the established convention for agent-driven write paths and editor authoring across Stories 14.22 – 14.30 + 17.11 / 17.11b. A future story may introduce a dedicated `documents:approve` permission (e.g., mapped to OWNER + ADMIN only, for explicit separation-of-duties between authoring and approving); that change is out of scope for the foundation story. The current permission enum in `lib/auth/permissions.ts` carries `documents:add`, `documents:remove`, `tasks:edit`, so `tasks:edit` is the only valid choice today.
   - Reads the doc; refuses if `current_draft_version_id IS NULL` (Swedish: `"Det finns inget pågående utkast att godkänna."`); refuses if `draft_status !== 'IN_REVIEW'` (Swedish: `"Utkastet måste vara skickat för granskning innan det kan godkännas."`).
   - In a single `$transaction`:
     - Stamp the OLD `current_approved_version` (if any) with `superseded_at = now()`.
     - Stamp the draft version with `approved_at = now()`, `approved_by = userId`.
     - Update the doc:
       - `current_approved_version_id = current_draft_version_id` (the draft pointer's value moves to the approved slot).
       - **`current_version_id = current_draft_version_id`** — **this is the moment the deprecated alias finally advances** (it was frozen on the prior approved version throughout the draft window per AC 4 + AC 5; promoting the draft is when the alias safely catches up to the newly-approved version, so the next auto-reindex picks up the new approved content).
       - `current_draft_version_id = null`, `draft_status = null`.
       - `status = 'APPROVED'` (idempotent — already APPROVED in the dual-state model, but explicitly stamped).
       - `approved_by = userId`, `approved_at = now()` (doc-level denormalized convenience fields, mirror the just-promoted version's per-version metadata).
     - ActivityLog: `document_draft_promoted` (NEW action) with `new_value: { promoted_version_id, prior_approved_version_id, version_number }`.
   - `after()` callback: re-trigger `indexWorkspaceDocument` so the search index picks up the new approved content. (17.10b contract continues to apply — Story 17.18 may further refine the indexed-content source.)
   - Returns `{ success: true, data: { newApprovedVersionId, versionNumber } }` or `{ success: false, error }`.

7. **New `discardDraft(documentId)` server action** (`app/actions/documents.ts`, new export):
   - Workspace-scoped via `withWorkspace`. **Gated on `tasks:edit`** (same rationale as AC 6 — the established agent-driven write-path convention; the speculative `documents:edit` scope does not exist in `lib/auth/permissions.ts` and adding it is out of scope here).
   - Reads the doc; refuses if `current_draft_version_id IS NULL` (Swedish: `"Det finns inget pågående utkast att förkasta."`); refuses if `current_approved_version_id IS NULL` (never-approved doc — Swedish: `"Det finns ingen godkänd version att återgå till. Använd 'Arkivera dokument' istället om utkastet inte ska användas."`).
   - In a single `$transaction`:
     - Update the doc: `current_draft_version_id = null`, `draft_status = null`. **The deprecated `current_version_id` alias requires no change** — it was already frozen on the approved version throughout the draft window (per AC 4 + AC 5), so discarding the draft is a no-op for the alias. This is by design: the alias correctly returns to "approved-only" semantics the instant the draft pointer is cleared.
     - ActivityLog: `document_draft_discarded` (NEW action) with `old_value: { discarded_version_id, draft_status }`.
   - **The discarded draft version row is NOT deleted** — version history stays append-only. It just stops being pointed at. (Optionally a follow-up story could surface "discarded drafts" in the version history UI; out of scope here.)
   - `after()` callback: re-trigger `indexWorkspaceDocument`. **Note this reindex is defensive — not load-bearing under Model B** (the index already tracks approved content throughout the draft window via the frozen alias). Triggering it on discard guarantees a clean post-condition even if the indexing pipeline was mid-flight on a stale signal; cheap insurance, no behavioral dependency.
   - Returns `{ success: true }` or `{ success: false, error }`.

### Agent write-tool guard refactor

8. **`update_document` tool (`lib/agent/tools/update-document.ts`) guard refactor:**
   - Tool-time `select` extends to read both new pointer columns + `draft_status` + the linked `current_draft_version.content_json` + `current_approved_version.content_json`.
   - Refusal condition changes from `document.status !== 'DRAFT' && document.status !== 'IN_REVIEW'` to:
     ```ts
     const writeable =
       document.current_draft_version_id != null ||  // doc has a draft in progress
       (document.status === 'DRAFT' && document.current_approved_version_id == null)  // never-approved
     if (!writeable) { ... refuse ... }
     ```
   - The refusal Swedish message stays the same for the case the agent will encounter today (APPROVED with no draft pending) — the user-facing copy doesn't reveal the internal pointer model. The guidance to `createDraftFromApproved` stays unchanged.
   - **Content source for the edit:** the tool computes the agent's proposed edit against `current_draft_version.content_json` when the draft pointer is set, else against `current_approved_version.content_json` for the never-approved-DRAFT case. **Do NOT read `current_version.content_json` (the deprecated alias)** — under the corrected alias semantics, the alias points at the approved version during a revision window, which is NOT the content the agent's edit should target. Reading the alias here would cause the agent to compute its diff against approved content while writing to the draft pointer — silent corruption of the draft.
   - The `entity_version` snapshot continues to be `document.updated_at.toISOString()` (14.31 staleness guard contract — unchanged).

9. **`add_document_section` tool (`lib/agent/tools/add-document-section.ts`) guard refactor:** identical reframe to AC 8.

10. **Dispatch-time guard refactor (`app/actions/pending-agent-actions.ts`):**
    - `UPDATE_DOCUMENT` branch (line ~496, verified): re-read the doc with the new selects (both pointers + linked `content_json`s); apply the same writeable predicate as AC 8; refuse with the same Swedish error if the predicate fails.
    - `ADD_DOCUMENT_SECTION` branch (added by 17.11b): identical reframe.
    - When the writeable predicate holds via the draft-pointer path, the dispatch calls `updateSection`/`addSection` on the **draft version's `content_json`** (NOT the approved version's, NOT the deprecated alias's), then calls `saveDocumentVersion(documentId, fullContentJson, changeSummary, ...)`. `saveDocumentVersion` per AC 5 routes to Path A and advances the draft pointer correctly.

### Backwards compatibility

11. **`current_version_id` stays as a deprecated alias throughout this epic — with corrected Model B semantics.** The alias points at the **last approved version** during a revision window (the load-bearing change from the v1.0 draft of this story: under v1.0 the alias advanced to track the draft, which silently broke the 17.10b DEC-2 contract by causing the auto-reindex to chunk draft content with APPROVED-tier metadata). Per the corrected AC 4 + AC 5 + AC 6 + AC 7:
    - On `createDraftFromApproved` (AC 4): alias **frozen** on the prior approved version (or null for the never-approved-DRAFT edge case).
    - On `saveDocumentVersion` Path A (AC 5, draft-active): alias **frozen** on the approved version. Draft pointer advances.
    - On `saveDocumentVersion` Path B (AC 5, never-approved doc): alias and draft pointer advance together (same value — there's no approved version to protect).
    - On `promoteDraftToApproved` (AC 6): alias **advances** to the newly-approved version (catches up to the just-promoted draft).
    - On `discardDraft` (AC 7): alias **no-op** (already on the approved version throughout the draft window).
    **Net effect:** the alias always equals the doc's current "effective for compliance reads" version (last approved if any, else the lone draft). Legacy reads via `document.current_version.content_json` continue to return correct content for the auto-reindex / search / agent baseline paths.
    **One legacy read site MUST be migrated in this story** — the editor route (per Task 8 / new AC 13 below) — because it specifically needs the draft content during a revision window, not the approved content. Other read sites (table, doc page header, agent reads) continue to read through the alias and get the right compliance answer; Story 17.17 + 17.18 migrate them explicitly to the named pointers. The alias is NOT dropped in this story.

12. **Stories 17.11 + 17.11b test suites must remain green** post-refactor. The reframe is observationally equivalent for the cases existing tests cover (DRAFT/IN_REVIEW → write allowed, APPROVED/SUPERSEDED/ARCHIVED → write refused). New tests in this story extend the suite to cover the new pointer-driven case.

### Editor read-path migration (CRIT-1 partner)

13. **Editor route read-path migration (`app/(workspace)/workspace/styrdokument/[documentId]/edit/page.tsx`, plus any client-side hooks that load the doc):**

## Dev Notes

- **Pointer model semantics** — at any moment, a doc has one of these states (queryable from the two pointers alone):
  - Both pointers null → doc just created, no versions yet (edge case; `createDocument` populates `current_draft_version_id` immediately).
  - Only `current_draft_version_id` set → never-approved DRAFT (in-progress doc that hasn't reached first approval). Top-level `status: DRAFT`. Alias = draft pointer.
  - Only `current_approved_version_id` set → stable APPROVED policy, no revision in progress. Top-level `status: APPROVED`. Alias = approved pointer.
  - Both pointers set → APPROVED policy with revision in progress. Top-level `status: APPROVED`. **Alias = approved pointer (FROZEN throughout the draft window).** This is the new state Model A couldn't represent, and the alias-freeze is the load-bearing behavior that keeps 17.10b's DEC-2 contract intact at the foundation layer.
- **Why the alias freezes on draft-active docs** — under the v1.0 draft of this story, the alias was specced to advance on every draft save (mirroring Model A's behavior). That would have caused `indexWorkspaceDocument` (the 17.10b reindex source) to chunk draft content but stamp it with the doc's APPROVED status → agent `[Källa:]` citations would silently leak draft content. The corrected v1.1 spec freezes the alias on the approved version during the draft window, so the auto-reindex naturally keeps grounding `[Källa:]` in approved content. The trade-off: the editor route must read the draft pointer explicitly (Task 8 / AC 13), because the alias no longer points at "the content the editor wants to load." That's a single-site migration, not a system-wide refactor.
- **Why we don't change top-level `status` when branching** — the top-level `status` field semantically describes the **document's overall lifecycle** (created → first approved → maybe archived). It's NOT the same thing as the draft sub-state. Separating them is the core of Model B.
- **`promoteDraftToApproved` vs existing `updateDocumentStatus`** — the existing function continues to handle simple IN_REVIEW → APPROVED transitions for **never-approved drafts** (where there's no prior approved version to supersede; `current_approved_version_id IS NULL` at the time of approval). `promoteDraftToApproved` is the new explicit transition for the dual-state case (where there's a prior approved version that gets superseded by the promotion). Tests must cover both. **Per NTH-2:** add a JSDoc on `promoteDraftToApproved` and an inline comment on `updateDocumentStatus` explaining the boundary, so future devs don't re-litigate the question.
- **Why `current_version_id` stays as alias** — every read site in the existing codebase reads `document.current_version` for "the current content of this doc". Migrating ALL those reads in one story is risky and bloats scope. The corrected v1.1 alias semantics (frozen on approved during draft windows) keep the legacy reads returning the right content for the **compliance-baseline** path (auto-reindex, search, agent reads, doc page header before 17.17 ships) WITHOUT requiring 17.18's full read refactor. The editor is the one exception (Task 8) because it needs draft content. The drop is a follow-up cleanup after Story 17.18.
- **17.11c (agent auto-branch) is deliberately separate** — under Model B, when the agent calls `add_document_section` against an APPROVED doc with no draft pending, the most natural UX is for the tool to transparently create the draft AND write against it. That's a 50-line extension on top of this story's foundation. Keeping it separate keeps story boundaries clean and lets 17.16 ship without UX scope creep. PO will draft 17.11c after 17.17 + 17.18 land.
- **`source: 'AGENT'` enum value on `WorkspaceDocumentVersion`** — still unused by this story. The agent-authorship stamp lives on the ActivityLog row, not the version row, per 17.11/17.11b's choice. Consolidation to `source: 'AGENT'` is a future cleanup.
- **Backfill case (b) timestamp-driven correctness** — the v1.0 draft of this story used a fragile `version_number = current_version_number - 1` arithmetic to locate the prior-approved version, which silently broke for any doc edited >1× since branching. The corrected v1.1 backfill (AC 3 Pass 1 → Pass 2) is timestamp-driven: per-version `approved_at` is stamped from ActivityLog correlation, then the pointers are populated from the stamped timestamps. This handles multi-save draft histories correctly. For ambiguous cases (e.g., ActivityLog correlation fails for a specific version's interval), Pass 1 leaves that version's `approved_at` null and the migration's warning enumeration SQL surfaces those rows for hand inspection (per NTH-3 — engineer runs the enumeration on Supabase preview-env before applying to production).
- **Permission gate posture (SF-1 resolution)** — both new actions (`promoteDraftToApproved`, `discardDraft`) gate on `tasks:edit`, matching the existing `lib/auth/permissions.ts` enum (`documents:add`, `documents:remove`, `tasks:edit`) and the established 14.22–14.30 + 17.11/17.11b convention. Introducing a dedicated `documents:approve` scope (e.g., for OWNER + ADMIN-only separation-of-duties between authoring and approving) is a future small story — out of scope here so we don't bloat the foundation with role-mapping decisions.
- **17.10b auto-reindex source** — intentionally NOT changed in this story. `indexWorkspaceDocument` continues to index the content `current_version_id` points at (which the corrected v1.1 alias semantics keep pinned to the **approved** version during draft windows — so the indexed content stays compliance-correct). Story 17.18 refactors this to track `current_approved_version_id` as the canonical embedding source AND to optionally maintain a secondary DRAFT-tier index — both improvements built on top of an already-correct baseline.

## Risk and Compatibility Check

**Primary Risk:** Backfill migration on production data could leave rows in an inconsistent state — specifically, a case-(b) row (status=DRAFT/IN_REVIEW with a prior approved version) where ActivityLog correlation in Pass 1 fails to stamp the prior approved version's `approved_at`, leaving Pass 2 unable to populate `current_approved_version_id` correctly.

**Mitigation:**
- Backfill SQL is wrapped in a single transaction with TWO explicit row-count assertions: (1) no alias-with-null-both-pointers rows; (2) every case-(b)-eligible doc has both pointers populated. Either non-zero count triggers `RAISE EXCEPTION 'Backfill incomplete: ...'` and aborts the transaction.
- Migration applied to a Supabase preview-env snapshot first; smoke verifies all four cases (a/b/c/d) backfill correctly AND the multi-save regression scenario passes (a doc with `current_version_number = approved.version_number + 5` correctly identifies the prior approved version via timestamp, not arithmetic).
- Pre-write JSON-shape verification (Task 1 Step 1a): engineer runs `grep -A 20 "document_status_changed" app/actions/documents.ts` to confirm the ActivityLog payload shape BEFORE writing the backfill SQL.
- Warning enumeration query (per NTH-3): engineer runs the surfacing SQL on Supabase preview-env to catch Pass-1 stamp gaps and hand-resolve them before applying to production.
- The deprecated `current_version_id` alias retains its read-fallback property — even if a backfill row is missed, the alias still points at the doc's `current_version` and existing reads continue to function (returning whichever version was most-recent in the legacy model).

**Rollback:**
- Schema additions are nullable; rollback is "stop reading the new columns" + revert the code. Backfilled data stays in place harmlessly; the deprecated `current_version_id` continues to drive reads.
- Code revert: each refactored function has a clear pre-refactor shape (today's behavior is preserved in git history) — straight revert restores the prior behavior.
- Data corruption recovery: version-row immutability means we can reconstruct correct pointer state per doc from `WorkspaceDocumentVersion` + `ActivityLog` history at any future point.

- [ ] No breaking changes to existing tool APIs (Stories 17.11 + 17.11b tool surfaces unchanged; guard refactor is internal)
- [ ] Database changes are additive only (3 new columns + 1 new enum on each of two tables; 0 drops; 0 renames)
