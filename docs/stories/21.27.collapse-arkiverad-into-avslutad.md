# Story 21.27: Collapse ARKIVERAD into AVSLUTAD (3-state lifecycle)

## Status

In Progress (2026-04-28)

## Story

**As a** product owner,
**I want** the cycle lifecycle simplified from `PLANERAD → PAGAENDE → AVSLUTAD → ARKIVERAD` to `PLANERAD → PAGAENDE → AVSLUTAD`,
**so that** SMB self-audit users have one terminal active state instead of a phantom "archived" state that nothing in the UI ever transitions a cycle into.

## Context

Story 21.26 collapsed SEAL into AVSLUTAD on the grounds that the cryptographic ceremony was bureaucracy without payoff for SMB self-audit. The same logic applies one step further:

- **ARKIVERAD has no UI transition**. Nothing in the codebase moves a cycle from AVSLUTAD to ARKIVERAD — only direct DB writes can land it there. The state was scoped in the Epic 21 brief but a transition action was never implemented.
- **The only behavioral difference vs AVSLUTAD** is that ARKIVERAD locks findings (items already lock at AVSLUTAD per Phase 2). After 21.26 there were two ways findings could be locked: ARKIVERAD on the cycle, or… nothing else. So the lock semantic depended entirely on this phantom state.
- **Activity log + DB write history give defensible change history** if anyone disputes a late edit. No Nordic regulator (AML, AFS 2001:1, ISO certifying bodies, GDPR Art 30, Bokföringslagen) requires findings to be cryptographically frozen at a specific state transition.
- **List hygiene for old cycles** is solvable later via a "show only last 12 months" filter on `/laglistor/kontroller` — not via a state.

## Dev Notes

**Migration safety**: Same pattern as 21.26's migration — wrapped in `BEGIN;…COMMIT;`, with `DROP DEFAULT` / `SET DEFAULT` around the enum surgery to avoid Postgres error 42804. Pre-flight query: `SELECT count(*) FROM compliance_audit_cycles WHERE status = 'ARKIVERAD';`. If non-zero, Step 1 of the migration folds them into AVSLUTAD with all metadata preserved; if zero, Step 1 is a no-op.

**Activity log**: no `cycle_archived` action constant existed in the codebase, so no legacy formatter to retain. The migration silently absorbs ARKIVERAD rows into AVSLUTAD; their existing `cycle_completed` activity entries (if any) continue to render unchanged.

**`linked-cycles-box.tsx` defensive map**: kept the `ARKIVERAD: 'Arkiverad'` and `SEALED: 'Fastställd'` entries as legacy fallbacks. The join table `compliance_cycle_task_links` doesn't store a denormalised status, so any task linked to a pre-collapse cycle reads the current status — but if a task is linked to a cycle that was archived before this migration, the current status will already be AVSLUTAD post-migration. The fallback is purely defensive.

**Test surface**: ~25 test references narrowed to 0 (excluding comments). Two test files lost entire `describe` blocks (the ARKIVERAD read-only banner test in items-tab; the ARKIVERAD-blocks finding tests). One test file (`cycle-detail-page-lifecycle.test.tsx`) had stale `SEALED` references from before 21.26 — those got swept too.
