# Story 16.1: Auto-Login After Signup (Delayed Email Verification)

## Status

Approved

## Dependencies

- **Prerequisite (infrastructure):** Supabase Auth must be configured to allow sign-in before email confirmation. In Supabase Dashboard → Authentication → Settings → Email Auth: disable "Confirm email" enforcement for sign-in (emails still sent, just not blocking). This MUST be verified before development begins.

## Story

**As a** new user signing up for Laglig.se,
**I want** to land directly in the onboarding wizard after creating my account,
**so that** I don't lose momentum waiting for a verification email before experiencing the product.

## Acceptance Criteria

### Auth Flow Changes

1. After `signupAction()` returns success, the client (`_signup-form.tsx`) immediately calls NextAuth `signIn('credentials', { email, password, redirect: false })` to create a JWT session — the NextAuth `authorize` callback handles `signInWithPassword()` against Supabase. User is logged in without clicking email link.
2. Redirect to `/onboarding` (not "check your email" message) after successful NextAuth sign-in
3. Verification email is still sent by Supabase (non-blocking, sent in background by `signUp()`)
4. `email_verified` field on User model continues to track verification status from `email_confirmed_at`
5. `email_verified` exposed on the client-side NextAuth session object via the JWT and session callbacks, and `next-auth.d.ts` type declarations updated

### Soft Verification Gate

6. Unverified users can: browse dashboard, view law lists, complete onboarding, use settings
7. Unverified users cannot: use AI chat, invite team members, export data (CSV/PDF), access admin features
8. Persistent verification banner rendered conditionally in the server layout (`app/(workspace)/layout.tsx`) — layout reads `email_verified` from Prisma User (already available via `getCurrentUser()`). Banner includes "Skicka igen" client-side resend button.
9. Banner dismissed after verification — layout re-checks on each page load; banner not rendered when `email_verified === true`
10. Banner style: subtle info banner (not blocking), dismissable but returns on next page load if still unverified

### Abuse Mitigation

11. IP-based rate limiting on signup endpoint: max 5 signups per hour per IP
12. Cloudflare Turnstile CAPTCHA on signup form (invisible mode by default, managed challenge as fallback)
13. Cron job: auto-delete User records where `email_verified = false` AND `created_at < NOW() - 48 hours` AND user has zero WorkspaceMember records (never completed onboarding). Do NOT delete users who have any workspace membership — they may have been invited.
14. Cron job runs daily at 03:00 UTC. Note: existing crons do not use Sentry `withMonitor()` — this story introduces it as a new practice. If Sentry cron monitoring is not yet configured, fall back to console.log + error reporting only.

### Verification Completion

15. When user clicks verification link in email, `/auth/callback` exchanges code, updates Supabase `email_confirmed_at`, AND syncs `email_verified = true` on the Prisma `User` model
16. If user is already logged in when clicking verification link, banner disappears on next navigation
17. If user clicks verification link on a different device/browser, they get a session on that device too

## Dev Notes

### Current Auth Flow (What Changes)

**Current (`app/actions/auth.ts` → `signupAction()` + `_signup-form.tsx`):**
1. User fills form, submits
2. `signupAction()` validates with `SignupSchema`, calls `supabase.auth.signUp()`
3. Returns success → `_signup-form.tsx` shows "Kontrollera din e-post" with CheckCircle icon (lines 89-111)
4. User must click email link → `/auth/callback?code=XXX` → `exchangeCodeForSession` → Supabase session created → redirect to `/dashboard`
5. Dashboard layout detects no workspace → redirects to `/onboarding`

**New flow:**
1. Same form, same validation
2. Same `signUp()` call → returns success
3. **NEW:** `_signup-form.tsx` receives success, calls `signIn('credentials', { email, password, redirect: false })` from `next-auth/react`
4. **NEW:** NextAuth `authorize` callback (`[...nextauth]/route.ts:19`) calls `signInWithPassword()` → Supabase validates → upserts Prisma User → returns user → JWT session created
5. **NEW:** `_signup-form.tsx` receives successful signIn result, calls `router.push('/onboarding')`
6. Email verification link still sent by Supabase in step 2 (non-blocking)

**Why client-side `signIn()` instead of server-side?** `signupAction()` is a `'use server'` action that cannot set browser cookies for NextAuth JWT sessions. The NextAuth `signIn()` function is designed to be called from the client — it makes a POST to `/api/auth/callback/credentials` which sets the session cookie via the NextAuth route handler.

### Session Exposure for email_verified

Currently `email_verified` is set during the NextAuth `authorize` callback (`app/api/auth/[...nextauth]/route.ts` line 50: `email_verified: data.user.email_confirmed_at !== null`) but is NOT propagated to the session object. The JWT callback (line 72) only sets `id`, `email`, `name`. The `authorize` return object (line 54) also doesn't include `email_verified`. Must fix both:

```typescript
// authorize — include email_verified in returned user
return {
  id: data.user.id,
  email: data.user.email!,
  name: data.user.user_metadata?.name || null,
  email_verified: data.user.email_confirmed_at !== null, // ADD THIS
}

// JWT callback — propagate to token
async jwt({ token, user }) {
  if (user) {
    token.id = user.id
    token.email = user.email!
    token.name = user.name
    token.email_verified = (user as { email_verified?: boolean }).email_verified ?? false // ADD
  }
  return token
}

// Session callback — expose to client
async session({ session, token }) {
  if (session.user) {
    session.user.id = token.id as string
    session.user.email = token.email as string
    session.user.name = token.name as string | null
    session.user.email_verified = token.email_verified as boolean // ADD
  }
  return session
}
```

And extend `types/next-auth.d.ts`:
```typescript
declare module 'next-auth' {
  interface User {
    id: string
    email: string
    name: string | null
    email_verified: boolean  // ADD
  }
  interface Session extends DefaultSession {
    user: {
      id: string
      email: string
      name: string | null
      email_verified: boolean  // ADD
    }
  }
}
declare module 'next-auth/jwt' {
  interface JWT {
    id: string
    email: string
    name: string | null
    email_verified: boolean  // ADD
  }
}
```

### Banner Architecture

The verification banner is a **server-rendered** component in the workspace layout, not a client component reading the NextAuth session. This is simpler and avoids hydration issues:

```typescript
// app/(workspace)/layout.tsx — after getCurrentUser() (line 47)
const user = await getCurrentUser()
if (!user) redirect('/login')

// Check verification status from Prisma
const dbUser = await prisma.user.findUnique({
  where: { email: user.email },
  select: { email_verified: true },
})
const isVerified = dbUser?.email_verified ?? false

// In the return JSX:
{!isVerified && <VerificationBanner email={user.email} />}
<WorkspaceShell user={user}>{children}</WorkspaceShell>
```

The "Skicka igen" button inside the banner is a small `'use client'` island component that calls the Supabase resend API.

### Supabase Configuration (PREREQUISITE)

The core premise of this story requires Supabase to allow `signInWithPassword()` before email confirmation. This is configured in Supabase Dashboard → Authentication → Settings → Email Auth. By default, Supabase may enforce email confirmation before sign-in. This setting MUST be adjusted before development begins. The setting is a Supabase dashboard toggle, not a codebase change.

### Existing Patterns

- **Rate limiting:** Follow `app/api/chat/route.ts` pattern with `Ratelimit.slidingWindow()` + `isRedisConfigured()` guard
- **Cron jobs:** Follow `app/api/cron/sync-sfs-updates/route.ts` pattern with `maxDuration = 300`. Note: existing crons do NOT use Sentry `withMonitor()` — this story introduces that practice. If Sentry cron monitoring isn't configured, use standard error reporting.
- **Permission checks:** Follow `hasPermission()` pattern in `lib/auth/permissions.ts`
- **Turnstile:** Invisible mode renders no visible widget. If Cloudflare detects suspicious behavior, it escalates to a managed challenge. Server-side verification via `POST https://challenges.cloudflare.com/turnstile/v0/siteverify` with `{ secret, response }`.
