## Story

**As a** user proposing a task via the compliance partner,
**I want** the proposal to render inline in chat with editable fields and persist if I close the chat,
**so that** I can review and approve it at my own pace without losing context.

## Context

Today when the user asks the agent to create a task, the agent calls `create_task(execute: false)` which returns a `WriteToolResponse` envelope. That envelope opens `write-preview-task.tsx` in the **right sidebar** (`components/features/ai-chat/details/write-preview-task.tsx`). If the user closes the chat modal without deciding, the proposal is lost — the envelope is ephemeral, tied to the tool result's in-memory SDK state.

Phase 5 replaces that ephemeral sidebar flow with a persistent, inline card. This first story ships the foundation (DB model, server actions, card primitive) and migrates the single `create_task` pathway end-to-end behind a feature flag so the existing sidebar flow remains available as fallback. Stories 14.23 and 14.24 extend to additional approval types, batching, and full migration.

**Key constraint:** Existing `write-preview-task.tsx` must keep working. Rollout is gated behind `FEATURE_INLINE_AGENT_APPROVALS`. Story 14.23 removes the flag and deletes the sidebar component.

### Lifecycle flow (reference)

```

  │
  ▼

  • Invokes streamText() with tool loop
  │
  ▼  Agent decides to propose a task

  │
  ▼  Stream finishes

  │
  ▼  Client re-renders

```

The context assembly must inject this exact shape (caps: 5 entries per bucket, newest first):


## Acceptance Criteria

### Persistence Model

1. A new Prisma model `PendingAgentAction` is added with fields: `id` (uuid), `workspace_id`, `user_id`, `conversation_id` (nullable — null = active conversation), `chat_message_id` (FK to `ChatMessage`), `context_type` (`ChatContextType`, default `GLOBAL`), `context_id` (nullable), `action_type` (`PendingAgentActionType` enum, initial value `CREATE_TASK`), `status` (`PendingAgentActionStatus` enum with values `PENDING | APPROVED | REJECTED | EXPIRED`, default `PENDING`), `params` (JSON, editable snapshot of proposal parameters), `result_ref` (JSON nullable, e.g., `{ taskId }` after approve), `created_at`, `decided_at` (nullable), `expires_at` (default now + 7 days).
2. Indexes: `(workspace_id, user_id, conversation_id, status)`, `(chat_message_id)`, `(expires_at)`. Table mapped to `@@map("pending_agent_actions")`.
3. Relations: `Workspace` (onDelete Cascade), `User` (onDelete Cascade), `ChatMessage` (onDelete Cascade). The matching inverse relations are added on `Workspace`, `User`, `ChatMessage`.
4. Migration is **additive** (new table, new enums; no existing model modified).

### Server Actions

5. `approvePendingAction(id)` — loads the row, verifies ownership (`workspace_id` matches active workspace context, `user_id` matches session user), dispatches to the concrete server action for the `action_type` (for `CREATE_TASK`: calls `createTask` with `params`), sets `status = APPROVED`, writes `result_ref = { taskId }`, sets `decided_at`, and `revalidatePath` on relevant routes. Returns `{ success, data?: { resultRef }, error? }`.
6. `rejectPendingAction(id)` — verifies ownership, sets `status = REJECTED`, sets `decided_at`. No side effects beyond the status update.
7. `updatePendingActionParams(id, params)` — verifies ownership and `status === 'PENDING'`, merges/replaces `params`. Rejected/approved rows cannot be updated.
8. `expirePendingActions()` — batch helper: finds all rows where `status = PENDING AND expires_at < NOW()` and sets them to `EXPIRED`. Invoked from the new cron endpoint (see AC 8a).
8a. Cron endpoint: `app/api/cron/expire-pending-actions/route.ts`. Schedule: **`0 3 * * *` (daily 03:00 UTC)** — added to `vercel.json` `crons[]` array. Rationale: off-peak window, doesn't collide with existing `cleanup-invitations` (02:00), `cleanup-workspaces` (06:00), or notification crons (07:00+). Follows the same auth-header + `maxDuration` pattern as existing cron handlers (see `app/api/cron/cleanup-invitations/route.ts` for the minimal template).
9. Read helper: `getPendingAgentAction(id)` — loads a single row, verifies ownership, returns `{ success, data?: PendingAgentAction, error? }`. Used as the SWR fetcher from `AgentActionCard`.
9a. All actions live in `app/actions/pending-agent-actions.ts`. All actions are workspace-scoped — cross-workspace access returns `{ success: false, error: 'Forbidden' }`.

### `AgentActionCard` Primitive

10. New component `components/features/ai-chat/agent-action-card.tsx` — single-action layout.
11. Props: `{ pendingActionId: string }`. The component is SWR-backed; the SWR key is `pending-action:${pendingActionId}`. Fetcher calls a new `getPendingAgentAction(id)` server action (or route — implementer chooses the consistent pattern per the codebase).
12. The card **routes to a per-type renderer** via `actionType`. For 14.22, only `CREATE_TASK` is supported; other types render a non-intrusive fallback: "Den här typen av förslag stöds inte ännu".
13. Card states: `PENDING` (editable + approve/reject), `APPROVED` (read-only summary + link to created entity), `REJECTED` (read-only summary with muted styling), `EXPIRED` (read-only, "Förslaget har gått ut" message).
14. Approve/reject/edit optimistic updates via SWR mutate. On server error, the optimistic state is rolled back and an error toast is shown (reuse the existing toast pattern — `sonner` is already wired).

### `TaskApprovalRenderer`

15. Component `components/features/ai-chat/agent-action-renderers/task-approval-renderer.tsx` — identical field surface to today's `write-preview-task.tsx` (title, description, priority, related document) so UX is unchanged from the user's perspective.
16. In `PENDING` state: editable fields + "Godkänn" (primary) and "Avvisa" (secondary ghost) buttons. Edits debounce 500ms and call `updatePendingActionParams`.
17. In `APPROVED` state: read-only summary with a link to the created task (uses `result_ref.taskId`). Styling matches the existing "done" state in `write-preview-task.tsx` for visual continuity.
18. Copy is Swedish throughout, following the existing `write-preview-task.tsx` vocabulary.

### Chat Message Integration

19. `components/features/ai-chat/chat-message.tsx` gets a new render branch: when a `tool-*` part has `output-available` state AND the output shape indicates a proposal with a `pendingActionId` (see AC 22), render an `AgentActionCard` **inline** after the tool call row (or in place of the `ToolCallRow`'s detail area — implementer chooses based on visual fit).
20. The card is rendered at the same nesting level as existing `ToolCallRow` / `CollapsedToolGroup` output — it participates in collapsing logic from 14.18 (running tool parts still flush a group; the inline card renders beneath a collapsed group when the group is expanded, or after the group if the proposal tool is the only one in the contiguous block).
21. Feature flag `FEATURE_INLINE_AGENT_APPROVALS` (read from `process.env.NEXT_PUBLIC_FEATURE_INLINE_AGENT_APPROVALS === 'true'`) gates the new branch. When `false`, `chat-message.tsx` falls back to the existing `sidebarHint === 'open'` → open `write-preview-task` sidebar flow. Both paths must coexist cleanly during rollout.

### Tool Rewiring

22. `lib/agent/tools/create-task.ts` updated: on `execute: false`, in addition to returning the `WriteToolResponse`, the tool's `execute` handler creates a `PendingAgentAction` row with `action_type: CREATE_TASK` and `params: { title, description, relatedDocumentId, priority }`. The row's `chat_message_id` **MUST** reference an assistant-message-id that has been preallocated in `app/api/chat/route.ts` before the tool loop begins — see AC 22a. The returned envelope includes `data: { pendingActionId }` so the chat renderer can resolve it.
22a. `app/api/chat/route.ts` preallocates a UUID for the next assistant `ChatMessage` before invoking `streamText()`, passes it into `createAgentTools(...)` via a new `assistantMessageId` parameter, and writes the assistant message row with that exact id in `onFinish` (content filled from stream state). Tools read the id synchronously via closure — no async resolver needed. This guarantees every `PendingAgentAction` carries a valid `chat_message_id` the moment it's created.
23. On `execute: true`, the tool continues to behave as today (direct `createTask` call, no pending row created). Rationale: `execute: true` is only called by the agent when re-confirming after an out-of-band sidebar approval; the inline flow never calls `execute: true` directly — approval goes through `approvePendingAction`. This parallel path is retained during rollout and removed in 14.23.

### Agent Feedback Loop

24. `app/api/chat/route.ts` injects a new system-message block on each turn listing pending / completed-since-last-turn / rejected-since-last-turn actions for the current `workspace_id + user_id + (conversation_id | context_type + context_id)` scope.
25. Context shape (Swedish prompt content, stable JSON structure):
    ```
    <pending_agent_actions>
    Väntar på beslut: [ { id, action_type, created_at, summary } ]
    Godkända sedan senaste svar: [ { id, action_type, result_ref, decided_at, summary } ]
    Avvisade sedan senaste svar: [ { id, action_type, decided_at } ]
    </pending_agent_actions>
    ```
26. "Since-last-turn" is computed as `decided_at > (previous assistant message created_at)` OR `decided_at > session_start_time` when no prior assistant message exists. Caps: last 5 entries per bucket, newest first.
27. The block is injected **after** the company-context section and **before** any context-specific context (law/task/change), so the agent treats it as workflow state rather than subject content.

### Swedish Copy

28. All user-facing strings use Swedish (`"Godkänn"`, `"Avvisa"`, `"Godkänt — uppgift skapad"`, `"Förslaget har gått ut"`, etc.). Maintain tone consistency with existing write-preview components.

### Non-Regression

29. Existing chat streaming, reasoning blocks (Story 14.20), tool-call collapse (14.18), and citation pills continue to render correctly.
30. When `FEATURE_INLINE_AGENT_APPROVALS === 'false'`, every existing sidebar write-preview flow works unchanged (visual + behavioral parity verified).
31. `expirePendingActions` has no effect on rows that are not in `PENDING` state.

## Dev Notes

### Previous Story Insights

- **Story 14.7c** (Write Action Tools, Done) established the `execute: false/true` two-phase tool pattern. `execute: false` returns a `WriteToolResponse` envelope via `wrapWriteToolResponse`; `execute: true` performs the real side-effect. This story extends (not replaces) the `execute: false` path to additionally persist a `PendingAgentAction` row. [Source: `lib/agent/tools/create-task.ts:30-74`, `lib/agent/tools/utils.ts:49-63`]
- **Rejected alternative for assistant-message-id timing**: an earlier draft considered returning `pendingActionId: null` from the tool and reconciling `chat_message_id` in `onFinish`. Rejected because it leaves rows temporarily invalid and complicates the batch query in 14.23. Preallocation (AC 22a) is the invariant-preserving choice.
- **Story 14.11** (Chat-First Dashboard "Hem", Done) introduced `conversation_id` grouping on `ChatMessage`. `null` = active conversation; archived conversations get a cuid via `archiveConversation()`. Pending-action scoping follows the same key. [Source: `prisma/schema.prisma:1290-1310`, `app/actions/ai-chat.ts:290-310`]
- **Story 14.15b** (Interactive Agent Surfaces, Done) set the visual language (Card header + field form + primary/ghost button pair) that the new inline renderer should preserve for continuity. Re-implement, don't copy. [Source: `components/features/ai-chat/details/write-preview-task.tsx`]
- **Story 14.18** (Collapsed Tool Calls, Done) added the `groupCompletedToolParts` render pass. Proposal-carrying tool parts must NOT be grouped with non-proposal peers — document the choice inline in the grouper. [Source: `components/features/ai-chat/chat-message.tsx`]
