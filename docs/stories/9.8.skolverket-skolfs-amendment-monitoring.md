# Story 9.8: Skolverket (SKOLFS) Amendment Monitoring (Phase 2)

## Status

Draft

## Story

**As a** kommun (municipal) compliance owner,
**I want** Skolverket's SKOLFS författningar monitored for amendments, repeals, new in-force laws, **and announced-but-not-yet-effective changes** — the same way SFS laws are,
**so that** when a SKOLFS document on my laglista changes (or is *about* to change on a known date) I get an AI-bedömd notification and can assess its impact — closing the loop that Story 9.7 (ingestion) opened.

## Context

Story 9.7 ingested the **~1,400 gällande** SKOLFS documents as `AGENCY_REGULATION` rows (API enumeration → PDF intake → attribution → chunks/embeddings → linkify) **and captured a per-doc baseline snapshot** in `metadata` (`contentHash`, `validity`, `latestChangeBySkolfsNo`, `amendmentChain[]`, `upcoming[]`, dates). This story adds **forward monitoring**. Per the 9.6 investigation, the downstream half of the pipeline (`ChangeEvent` → notify → `ChangeAssessment` → email/in-app) is **source-agnostic**; the work here is a SKOLFS-specific **detector** (a state-diff over the API) that emits `ChangeEvent`s, plus the small enrich-decoupling 9.6 identified. **SKOLFS detection is structurally easier than SFS or Socialstyrelsen**: the API *hands us* the amendment graph + `validity` flips, so the detector is a JSON state-diff, not a scraper. Per-version historik remains out of scope (deferred). [Source: Story 9.7 Dev Notes → "Monitoring fit"; Story 9.6 source-agnosticism map]

## Acceptance Criteria

1. **Scheduled detector cron.** `app/api/cron/discover-skolfs-changes` runs daily (e.g. `0 6 * * *` UTC) via `vercel.json`, makes **one** `GET /api/statute?size=20000` call for the full current state, loads the per-doc baseline from `metadata`, and classifies each diff. Daily cadence is justified (SKOLFS changes are low-frequency and batch-published; the poll is a single cheap request). The existing `notify-amendment-changes` cron (07:00 UTC) is **reused unchanged**. **The endpoint MUST mirror the existing cron-auth guard** — `CRON_SECRET` env + `authorization: Bearer ${CRON_SECRET}` → 401 on mismatch (do not ship an unauthenticated cron). [Source: vercel.json SFS crons; app/api/cron/discover-sfs-amendments:55,92-94 (CRON_SECRET/Bearer guard)]
2. **Four detection signals**, each producing the right outcome:
   - **NEW_LAW** — a base appears `VALID` that we don't have (brand-new, or `UPCOMING → VALID`): ingest its PDF (Story 9.7 ingester, `--force`/`--filter`) and emit `ChangeEvent(NEW_LAW)`. **PO decision (MVP): NEW_LAW is silent-ingest, NOT notified** — `NEW_LAW` is deliberately absent from `notify-amendment-changes`'s `NOTIFIABLE_CHANGE_TYPES = [AMENDMENT, REPEAL, NEW_RULING]` (route lines 216-220), so emitting it requires **no notify change** and produces no email (matches current SFS behavior). [Source: app/api/cron/notify-amendment-changes route:216-220]
   - **AMENDMENT** — an in-force change: `latestChangeBySkolfsNo` advanced, OR a chain amendment flipped `UPCOMING → VALID`, OR a new `VALID SENASTE_LYDELSE` appeared for the base. Re-ingest the (preferred consolidated) PDF, diff prior `full_text` vs new, emit `ChangeEvent(AMENDMENT)` with `amendment_sfs = <amending SKOLFS no.>`, `changed_sections` (from the API `change` string, e.g. `"ändr. 5, 6 §§"`), and `ai_summary` from the diff. **AMENDMENT and REPEAL are already in `NOTIFIABLE_CHANGE_TYPES`** → they notify with no notify change.
   - **REPEAL** — a base flips `VALID → EXPIRED`: set `status = REPEALED`, emit `ChangeEvent(REPEAL)`.
   - **UPCOMING (kommande ändring)** — a `VALID` base has an `UPCOMING` amendment in its chain that we have **not yet emitted an event for**: emit a "coming change" event carrying the amendment's effective date (in `ai_summary`, per AC 2a); **no re-ingest yet**. When that date passes and the amendment flips `VALID`, the next run picks it up as an AMENDMENT signal. **First-run backfill (important):** Story 9.7 captures `skolfs.upcoming[]` into the baseline but emits **no** `ChangeEvent`s (9.7 AC 13), so **9.8's first detector run must emit UPCOMING events for the pending amendments already in the baseline** (e.g. the ~18 July-2026 ämnesplan amendments) — otherwise the initial cohort gets no proactive notice and jumps straight to AMENDMENT when effective. Dedup (AC 5) makes this one-time, not per-run. **⚠️ This 4th signal needs schema + notify work (see AC 2a)** — there is no UPCOMING `ChangeType` today and the effective date has no column to flow through. [Source: 9.7 AC 9 (baseline `upcoming[]`) + AC 13 (no events emitted); verified API `validity`/`relatedDocumentMetadata` semantics; ChangeType enum below]
2a. **UPCOMING representation (PO decision — CONFIRMED by Alexander 2026-06-25: keep UPCOMING in 9.8, accept the migration — requires a migration):** the verified `ChangeType` enum is `{ NEW_LAW, AMENDMENT, REPEAL, METADATA_UPDATE, NEW_RULING }` (schema lines 695-701) — there is **no UPCOMING value**, so the "ChangeType already supported"/"no migration" framing of the earlier draft was wrong. **Decision:** add a new `ChangeType.UPCOMING_AMENDMENT` value via a **hand-off migration** (migrations are run manually — hand the user the command, never apply it), and add it to `NOTIFIABLE_CHANGE_TYPES` so kommande-ändringar notify. The **effective date is carried inside `ai_summary` text** (e.g. `"Ändring (SKOLFS 2026:23) träder i kraft 2026-07-01"`) because `ChangeEvent` has **no `effective_date` column** — this avoids a second column migration. [Source: prisma/schema.prisma:695-701 (ChangeType); user memory — migrations manual]
3. **Pending changes are detected via the chain, not `latestChangeBySkolfsNo`.** The detector scans `relatedDocumentMetadata` for `validity == "UPCOMING"` entries (the base's `latestChangeBySkolfsNo` reflects only the last *in-force* change and ignores pending ones — verified on the 20 amended-but-unconsolidated bases). [Source: Story 9.7 AC9; verified 2026-06-25]
4. **Content-hash guard before any re-ingest.** Even when the API signals an AMENDMENT, re-fetch the PDF and `computeContentHash`; only re-ingest (pay for Claude) if the hash changed vs the stored baseline. No hash change → metadata-only snapshot update. [Source: lib/agency/agency-pdf-registry.ts `computeContentHash`; Story 9.7 AC9 baseline]
5. **ChangeEvent dedup / idempotency.** Events are keyed on `(document_id, amendment_sfs)` — for UPCOMING, `amendment_sfs` = the **pending amendment's** SKOLFS number (there is no `effective_date` column to key on) — so a doc that sits in a changed state across runs (including the first-run UPCOMING backfill) emits at most one event per amendment; the snapshot in `metadata` advances to the new "last seen" state each run. [Source: Story 8.21 change-event-deduplication; app/api/cron/discover-sfs-amendments idempotency]
6. **Notification + assessment reuse — core path unchanged, with one known limitation.** `notify-amendment-changes` picks up the new `ChangeEvent`s (no content_type filter; recipients via `document_id → LawListItem → workspace`; title/`ai_summary`) and per-workspace `ChangeAssessment` works as-is — that path is genuinely source-agnostic (route lines 223-237, 334). **Limitation (verified):** the email's structured `effectiveDate` / `pdfUrl` / section-table fields are **enriched from `AmendmentDocument` via `sfs_number = ce.amendment_sfs`** (route lines 282-308); a SKOLFS event has no `AmendmentDocument` row, so `findUnique` returns null and those fields stay empty. **Mitigation:** carry the human-readable effective date + changed sections inside `ai_summary` (per AC 2a), which DOES render in both the email and the Ändringar tab. A structured SKOLFS enrichment branch is a future enhancement, not MVP. **Task 4 verifies** e2e: seed a workspace with a SKOLFS doc on its laglista, fire a synthetic ChangeEvent, confirm notification + assessment. [Source: app/api/cron/notify-amendment-changes route:216-237,282-308,334; lib/notifications/recipient-resolution.ts]
7. **Enrich decoupling (small).** `enrich-amendment-summaries` selects via `amendment_documents.sfs_number = ce.amendment_sfs`; a SKOLFS ChangeEvent has no such row. Set `ChangeEvent.ai_summary` **directly in the detector** (the diff is in hand) — preferred over generalizing the SFS-coupled join. [Source: app/api/cron/enrich-amendment-summaries — SFS-coupled join; Story 9.6 AC5]
8. **Re-ingest preserves attribution.** Re-ingestion on `--force` must not clobber `regulatory_body` / `agency_prefix` / `metadata.issuedBy` (esp. the Regeringen-issued läroplaner). Carry issuer through or re-derive from the API. [Source: Story 9.7 AC8; Story 9.6 AC6]
9. **Vercel runtime resilience.** `maxDuration = 300`; the diff is fast but a large same-day batch (e.g. the verified 2026-07-01 cluster of ~18 ämnesplan amendments) may exceed one slot for re-ingest. Process until a timeout buffer, then stop — the snapshot diff is idempotent so the next run drains the rest (same pattern as `discover-sfs-amendments`). Log each run to `CronJobRun` (`items_processed`/`items_failed`). [Source: app/api/cron/discover-sfs-amendments timeout handling; prisma/schema.prisma#CronJobRun]
10. **No regression:** lint/typecheck/build green; full unit suite green — including the new `classifySkolfsDiff` unit tests (AC 2 / Task 1).

## Dev Notes

### Source-agnosticism map (from 9.6 — the key reuse fact)

| Stage | Verdict |
|---|---|
| `ChangeEvent` model | **Source-agnostic** — `ContentType.AGENCY_REGULATION` exists; global, keyed on `document_id` |
| `notify-amendment-changes` | **Source-agnostic** — no content_type filter; `document_id → LawListItem → workspace` |
| `ChangeAssessment` | **Source-agnostic** — hangs off any ChangeEvent |
| Discovery | **Source-specific** — build the SKOLFS detector (this story) |
| `enrich-amendment-summaries` | **SFS-coupled join** — bypass by setting `ai_summary` in detector |
| version-history / historik | **SFS-coupled** — out of scope (deferred) |

[Source: docs/stories/9.6.socialstyrelsen-amendment-monitoring.md]

### Why SKOLFS detection is the robust kind (vs SFS / Socialstyrelsen)

SFS/Socialstyrelsen detection *reconstructs* "what changed" by crawling an index/HTML and classifying. SKOLFS exposes the answer: `validity` flips and `latestChangeBySkolfsNo` advancing **are** the signal, and `relatedDocumentMetadata` carries the full amendment graph with per-amendment `validity`/`effectiveDate`/`change`. The detector is a **state-diff over one JSON document** the cron already fetches — no scraping, no LLM classification of document type. [Source: verified API shape 2026-06-25; Story 9.7 Dev Notes]

### What the detector must produce to reuse the rest

A global `ChangeEvent` against the base `AGENCY_REGULATION` document, `change_type ∈ {AMENDMENT, REPEAL, NEW_LAW}` (+ an UPCOMING variant per PO), `notification_sent: false`, with `ai_summary` populated (the old-vs-new consolidated-text diff, or a templated "ändring träder i kraft <date>" for UPCOMING). That alone makes notify + assess work unchanged.

### The 2026-07-01 worked example (verified data)

18 ämnesplaner are `VALID` now, each with one `UPCOMING` amendment effective `2026-07-01`. Expected behavior: 9.7 ingest captures these into each base's `skolfs.upcoming[]` baseline but emits **no** events; **9.8's first detector run backfills the kommande-ändring (`UPCOMING_AMENDMENT`) events** (deduped per AC 5, so one-time). On 2026-07-01 the cron sees `UPCOMING → VALID`, Skolverket publishes each `SENASTE_LYDELSE`, the detector re-ingests (hash changed) + emits AMENDMENT events; `notify-amendment-changes` mails affected workspaces. The batch may span >1 cron slot → drains over successive runs (AC 9). [Source: verified amendment-chain inspection 2026-06-25; 9.7 AC 9/13]

### Constraints

- Reuse the 9.7 enumerator/ingester; **don't fork the pipeline.**
- Cron on Vercel (`maxDuration` 300s) — batch/limit per run; large läroplaner re-ingests are slow (carry the 9.5/9.7 large-doc + pool considerations). [Source: Story 9.6 AC7 pool note; Story 9.7 AC6a]
- **One migration IS required** (corrected from draft 0.1): the `ChangeType.UPCOMING_AMENDMENT` enum value for the 4th signal (AC 2a). `ChangeEvent`/`ContentType.AGENCY_REGULATION`/`CronJobRun` all already exist — only the enum value is new. Migrations run **manually** → hand the user the command, never apply it. UPCOMING is confirmed in scope (AC 2a), so the migration is on the critical path. [Source: prisma/schema.prisma:695-701; user memory — migrations manual]

### Testing requirements

- **`classifySkolfsDiff` unit tests** (Task 1) — the pure signal-classifier is the core logic and must be tested in isolation, not buried in the route handler. Fixtures: new `VALID` base → NEW_LAW; `latestChangeBySkolfsNo` advance / new `VALID SENASTE_LYDELSE` → AMENDMENT; `VALID → EXPIRED` → REPEAL; `VALID` base gains a chain `UPCOMING` not in baseline → UPCOMING; **baseline `upcoming[]` present on first run → UPCOMING backfill**; identical baseline vs. current → **no event**. Plus a dedup test: same diff across two runs → exactly one event.
- **Notify/assess integration** (Task 4) — seed a workspace with a SKOLFS doc on a laglista, fire a synthetic `ChangeEvent`, assert in-app notification + email + `ChangeAssessment`; assert the `ai_summary`-carried effective date renders and that the `AmendmentDocument`-enriched email fields are (expectedly) empty for SKOLFS.
- No prose snapshot tests. Full regression per AC 10. [Source: 9.5/9.7 testing approach]
