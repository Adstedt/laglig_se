## Story

**As a** product owner,
**I want** automated measurement of how well agent answers are grounded in the chunks actually retrieved for them,
**so that** we can detect ungrounded/hallucinated claims, track a grounding rate over time, and catch regressions before users lose trust.

## Story Context

**Existing System Integration:**

- **Retrieval:** `retrieveContext()` (`lib/agent/retrieval.ts`) already returns the ranked chunks for a query — the "ground truth" set to score answers against.
- **Chat route hook:** `app/api/chat/route.ts` already has an `onFinish` callback that writes `ChatUsageEvent` (Story 14.27, tokens/cost only). The grounding eval attaches at the same seam — but must also capture the **retrieved chunks + the final answer text**, which `ChatUsageEvent` does not store today.
- **Citations:** `lib/ai/citations.ts` parses `[Källa: …]` markers + resolves them to `SourceInfo`. A first-cut grounding signal = "did the answer cite, and do its citations resolve to chunks that were actually retrieved this turn?"
- **Admin dashboard pattern:** Story 14.27 shipped `/admin/(dashboard)/usage` (server component, range params, `lib/admin/queries.ts` raw-SQL with input clamping). A grounding panel follows the same pattern.

**Out of scope:** real-time blocking of ungrounded answers (this is observability/eval, not a runtime gate); user-facing thumbs feedback (19.12); per-tool-call audit (19.5 `AgentDecisionLog`).

## Acceptance Criteria

### Capture
1. New `AgentAnswerEval` model (or extend the 14.27 telemetry path) capturing, per assistant turn that used retrieval: `query`, `retrieved_chunk_ids[]` (+ minimal snippet/source refs), `answer_text`, `citations_emitted[]`, FKs to workspace + user + conversation, `created_at`. **Fail-safe** write (mirror 14.27's `[CHAT_USAGE_EVENT_WRITE_FAIL]` try/catch — never break the turn). Migration `--create-only` + manual handoff.
2. Capture is sampled-configurable (env-gated rate, default 100% in non-prod, sample in prod) to bound storage/cost.

### Detection
3. Grounding scorer `lib/agent/eval/grounding.ts`: given `{ answer, retrievedChunks, citations }`, returns a grounding result `{ grounded: boolean, score: number, ungroundedClaims?: string[] }`.
4. **v1 (cheap, deterministic):** citation-coverage + lexical-overlap heuristic — every `[Källa: …]` resolves to a chunk retrieved *this turn*, and factual sentences have sufficient token overlap with some retrieved chunk. No LLM call.
5. **v2 (deferred, flagged):** LLM-as-judge verification of claims vs chunks. Spec the interface so v2 slots in behind the same `grounding.ts` API; do not build v2 in this story.
6. The "I cannot answer from sources" case (the 14.9 refusal) counts as **grounded** (correct behaviour), not a miss.

### Dashboard
7. Admin grounding panel (alongside `/admin/(dashboard)/usage`): grounding rate over a selectable range, trend over time, and a drill-down list of low-score turns (query + answer + which claims were unsupported). Workspace filter, per the 14.27 pattern.
8. Surfaced metric: **grounding rate** with an explicit target (e.g. ≥95% grounded / <5% ungrounded, per 3.9 AC 5). Target is a displayed threshold, not an enforcement gate.

### Eval harness
9. A fixture set of ~50 Swedish compliance edge-case questions (`lib/agent/eval/__fixtures__/`) runnable as a scripted regression (`scripts/run-grounding-eval.ts`) that reports the aggregate grounding rate — so prompt/retrieval changes can be checked against a baseline before merge.

### Quality
10. Unit tests for the v1 scorer (grounded answer, ungrounded claim, refusal-counts-as-grounded, citation-not-in-retrieved-set). Dashboard query tests follow the 14.27 clamping/raw-SQL safety pattern. `pnpm typecheck` + `pnpm eslint` clean.

## Dev Notes

- **Attach at the existing telemetry seam.** 14.27 already writes per-turn telemetry in `app/api/chat/route.ts` `onFinish`. Reuse that callback; the net-new is persisting `retrieved_chunk_ids` + `answer_text` + `citations` (not in `ChatUsageEvent`). Decide: extend `ChatUsageEvent` vs a sibling `AgentAnswerEval` table — prefer a **separate table** (different cardinality + retention; grounding rows may be sampled, usage rows are not).
- **Reuse citation resolution.** `lib/ai/citations.ts` (`extractSourcesFromToolResult`, `resolveSource`) already maps `[Källa: …]` → source; v1 grounding leans on "do emitted citations resolve to *this turn's* retrieved chunks?".
- **Refusal = grounded.** The 14.9 "Jag har inte tillräcklig information…" path is the *correct* low-evidence behaviour; the scorer must not penalise it (AC 6).
- **Dashboard safety.** Follow 14.27's `lib/admin/queries.ts` conventions: `INTERVAL '1 day' * ${n}::int`, `clampRangeDays`/`clampLimit`/`clampOffset`; never interpolate into an `INTERVAL '${n} days'` literal.
- **This is observability, not a runtime gate.** Do not block or rewrite answers based on the score in this story — measure first.

### Testing

Vitest + mocked Prisma. Scorer unit tests are pure. Dashboard query tests mirror 14.27. The 50-question harness is a script (not a CI gate in v1) — it produces a baseline grounding rate for manual review.
