## Story

**As** the agent platform, as the skill library grows,
**I want** `createAgentTools` to narrow the active tool registry to `ALWAYS_AVAILABLE ∪ (active skills' whitelists)` — behind a no-regression harness that proves no context loses a tool its flows use,
**so that** future skills (19.8 `draft_policy` and beyond) can ship their own specialised tools without pushing every chat past the ~30-tool model-confusion threshold (PRD Risk 4), and so the "which tools are live per context" contract is locked down by a test rather than by hope.

This is the third and final leg of the 19.7 split (a→b→**c**), un-deferred now that the PO's precondition (≥2 skills exist) is met. **Be honest about the payoff:** at today's 27-tool registry the narrowing gates exactly **one** tool (`save_assessment`), because every other tool is either a universal read or an ad-hoc write. **The lasting value is the mechanism + the harness**, not today's one-tool reduction — the harness is the regression guard that makes every future skill safe to add.

## Acceptance Criteria

### The narrowing mechanism

1. **`ALWAYS_AVAILABLE` baseline (explicit + documented)** in `lib/agent/tools/index.ts`: define `SKILL_GATED_TOOLS: ReadonlySet<ToolName> = { save_assessment }` and derive `ALWAYS_AVAILABLE = (all factory tool names) \ SKILL_GATED_TOOLS`. Rationale comment: a tool is in `ALWAYS_AVAILABLE` unless it is *meaningless outside a specific skill*. Today only `save_assessment` qualifies (it persists a `ChangeAssessment` — inherently a change-assessment action). All 18 reads + the 9 ad-hoc write tools are universal.
2. **`createAgentTools` gains `activeSkills?: string[]`** (5th param). The skill filter composes with the existing role filter as a **single `allowed(name)` predicate** applied once (do NOT keep the role filter's early `return`):
   - `roleAllows(name)` = `!role || hasPermission(role, 'tasks:edit') || TOOL_REGISTRY_POLICY[name] !== 'write'` (unchanged AUDITOR semantics).
   - `skillAllows(name)` = `activeSkills === undefined` *(legacy callers → no narrowing)* `|| ALWAYS_AVAILABLE.has(name) || activeSkills.some(s => getSkillToolWhitelist(s).includes(name))`.
   - Keep a tool iff `roleAllows(name) && skillAllows(name)`. Return the same `as typeof tools` cast already used for the role filter.
3. **`activeSkills === undefined` ⇒ no narrowing** (full set) — backward-compat for every existing caller (tests, `generate-law-list`, any non-route caller). **`activeSkills === []` ⇒ narrow to `ALWAYS_AVAILABLE`** (the global/task/law case). The distinction is load-bearing: only the chat route opts into narrowing.
4. **Route wiring** (`app/api/chat/route.ts`, the `createAgentTools` call ~`:287`): compute `const primarySkill = getPrimarySkillForContext(contextType)` and pass `activeSkills: primarySkill ? [primarySkill] : []`. (Reuse the lowercase `contextType` already passed to `buildSystemPrompt`.)

### Behaviour (precise + behaviour-preserving except the one intended gate)

5. **Net effect = `save_assessment` is present only when `assess_change` is active** (the change context). It is removed from global/task/law chats. **Every other tool is unchanged in every context.** Concrete counts (non-AUDITOR): change = **27** (full — `assess_change`'s whitelist re-adds `save_assessment`); global/task/law = **26** (full minus `save_assessment`); `activeSkills` omitted = **27** (legacy).
6. **No assessment-flow regression:** the tools `assess_change`'s PROCEDURE actually calls but does *not* declare in its `tools` frontmatter — `get_company_context`, `search_laws`, `get_change_details`, `suggest_followups` — are all in `ALWAYS_AVAILABLE`, so they survive narrowing in the change context (and everywhere). This is the specific trap the harness guards.
7. **AUDITOR unaffected by narrowing:** AUDITOR already gets reads-only, and all reads are in `ALWAYS_AVAILABLE`, so the narrowed AUDITOR set == the un-narrowed AUDITOR set in every context.

### The no-regression harness (the durable deliverable)

8. **A harness test** (`tests/unit/agent/tools/registry-narrowing.test.ts`) that, for each context × the relevant role, asserts the narrowed registry **⊇ the context's flow-required tool set** (defined as a checked-in constant per context, derived from the skill PROCEDUREs + `system-prompt.md` tool-guidance). At minimum:
   - **change** (`activeSkills:['assess_change']`) ⊇ `{get_company_context, search_laws, get_change_details, get_law_list_item, list_linked_artifacts, suggest_followups, save_assessment, create_task, add_obligation, update_compliance_status, add_context_note}` (assessment flow + proposals).
   - **global** (`activeSkills:[]`) ⊇ the gap_analysis flow `{list_bevis_gaps, list_unassessed_changes, list_overdue, list_stale_documents, create_task, add_obligation, update_compliance_status, draft_styrdokument, activate_skill}` **and** asserts `save_assessment` is the *only* tool absent vs the full set.
   - **task / law** (`activeSkills:[]`) keep their ad-hoc action tools; assert `save_assessment` absent, everything else present.
   - **AUDITOR** in change context: narrowed set === un-narrowed AUDITOR set (narrowing is a no-op for read-only roles).
   - A guard asserting `ALWAYS_AVAILABLE ∪ assess_change-whitelist === full factory set` (i.e. the change context loses nothing).
   - **Limitation (PO-noted, document in the test):** the per-context flow-required sets are **hand-maintained constants** sourced from the skill PROCEDUREs + `system-prompt.md`. The harness catches `ALWAYS_AVAILABLE`/whitelist *misconfiguration*, but a maintainer who adds a new tool call to a PROCEDURE must also add it to the relevant required-set constant. Add a header comment pointing at the PROCEDUREs as the source of truth.

### Scope guards

9. **Out of scope:**
   - **Mid-conversation `activate_skill` → `activeSkills` re-injection** (deferred since 19.7a — the live activation path is the tool result, not a registry change). `activeSkills` is derived from `contextType` only.
   - **Completing the skills' `tools` frontmatter** to exhaustive lists. The current declared lists suffice (the only load-bearing whitelist entry is `save_assessment`; the read tools they omit are covered by `ALWAYS_AVAILABLE`). Note as future hygiene for when `ALWAYS_AVAILABLE` shrinks.
   - No new tool, no schema, no migration, no prompt change.
   - **SA-001** (`save_assessment` auto-saves without an approval card) is untouched — note the cross-link: removing `save_assessment` from non-change contexts *incidentally neutralises SA-001's secondary symptom* (the global-chat save path) but does **not** fix its core (the change-context auto-save bug → still 14.32).

### Pinned model

| Concept | Definition |
|---|---|
| `ALWAYS_AVAILABLE` | Every factory tool *except* `SKILL_GATED_TOOLS`. The universal baseline (all reads + all ad-hoc writes). |
| `SKILL_GATED_TOOLS` | Tools meaningless outside a specific skill. **Today = `{save_assessment}`** only. |
| Skill whitelist | The skill's declared `tools`. Functionally only matters for its `SKILL_GATED` entries (the rest are redundant with `ALWAYS_AVAILABLE` — harmless union). |
| `activeSkills` at route | `[primarySkill]` for the context, else `[]`. (Activation-only skills like `gap_analysis` never appear here — their tools are all `ALWAYS_AVAILABLE` anyway.) |
