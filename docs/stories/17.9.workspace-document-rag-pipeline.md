# Story 17.9: Chunk & Embed Uploaded Files into the RAG Pipeline

## Status

Ready for Review

> 🔄 **Re-scoped 2026-05-22 (v2.1)** to **uploaded files only** (`source_type = USER_FILE`), consuming Story 17.8's markdown. The authored-styrdokument (`WorkspaceDocument` → `WORKSPACE_DOCUMENT`) path is **split out to Story 17.9b** (deferred) — per PO validation, that path carried all the deep risk (flat-plaintext content, no delete action, no content_hash, multi-trigger coverage, draft-status gating). This story is the clean, unblocked file path. PO fixes applied: reuse the *merge algorithm* (not the private legal-hardcoded `chunkFromMarkdown`), cron-triggered (not fire-and-forget), lifecycle cleanup on `deleteFile` + `deleteFilesBulk`, defined content-hash dedupe, write-side isolation invariant.

## Story

**As a** system component,
**I want** uploaded files' extracted markdown (Story 17.8) chunked + embedded into the vector pipeline, **strictly workspace-scoped**,
**so that** the agent can semantically search a workspace's *own* uploaded files through dialog — never another tenant's and never the global legal corpus.

## Story Context

**Existing System Integration:**

- Reuses the legal-document chunk pipeline: `embed-chunks.ts` (`generateEmbeddingsBatch`, `:119`; throws if a batch >100, `:126`), the chunk-emit shape in `chunk-document.ts`, and `retrieveContext()` (`lib/agent/retrieval.ts:96`).
- Consumes **`WorkspaceFile.extracted_text`** (markdown, from Story 17.8) — populated when 17.8's `extract-files` cron sets a file `DONE`.
- `ContentChunk` (`prisma/schema.prisma:1838`): `source_type SourceType`, `source_id`, `workspace_id String?`, `content_role ContentRole`, `embedding vector(1536)`, `metadata Json?`, indexes `[source_type, source_id]` + `[workspace_id]`.
- Source type: **`USER_FILE`** (already in `SourceType`, `schema:1822`). `formatSourceLabel` already renders it ("Källa: <filename>", `context-assembly.ts:287`) — but it reads `chunk.metadata.filename`, so chunks MUST write that (AC 3).

**Out of scope:** styrdokument RAG (Story 17.9b); the global legal corpus (`LEGAL_DOCUMENT`, unchanged).

## Acceptance Criteria

### Source-type + isolation (load-bearing)

1. File chunks: `source_type = USER_FILE`, `source_id = WorkspaceFile.id`, `workspace_id = <ws>` (**never null**). **No enum migration** — `USER_FILE` exists (`schema:1822`).
2. **Write-side isolation invariant:** every emitted file chunk has a non-null `workspace_id`. The column is nullable (used by the global corpus), so there's no DB constraint — assert it in code + test. A `USER_FILE` chunk written with `workspace_id = null` would leak cross-tenant.
3. `contextual_header` = filename + `FileCategory` — this is what `formatSourceLabel` uses first (`context-assembly.ts:283`). Also write `metadata.{ filename, category }` defensively (the `USER_FILE` metadata branch at `:287` is only the fallback when `contextual_header` is empty).

### Chunking

4. Chunk `WorkspaceFile.extracted_text` (markdown) via a **paragraph-merge** strategy (target 400 tokens, cap 1000) — markdown structure (headings/tables from 17.8) yields clean boundaries. **Reuse the merge *algorithm*, not the function:** `chunkFromMarkdown` is **private** (`chunk-document.ts:582`, not exported) and **hardcodes `source_type:'LEGAL_DOCUMENT'`, `workspace_id:null`** (`:623-624,:637-639`); `ChunkInput` likewise hardcodes those (`:34,:36`). So define a parallel `WorkspaceChunkInput` + a `chunkUserFile()` that parameterises `source_type`/`workspace_id` (extract/fork the merge loop — do not call the private fn as-is).
5. `content_role` set appropriately (PARAGRAPH/HEADING/TABLE).

### Sync + trigger

6. New `lib/chunks/sync-workspace-chunks.ts` → `syncWorkspaceChunks(sourceId, sourceType, workspaceId, markdown, meta)`: delete existing chunks for `(source_id, source_type)`, chunk via `chunkUserFile`, embed via `generateEmbeddingsBatch` (≤100/batch), `createMany` into `ContentChunk`. Content is a **param** (caller provides the file's markdown) — unlike `syncDocumentChunks` which fetches its own. **`meta` MUST carry `content_hash`** (alongside `filename`/`category`) — the AC 9 dedupe compares against it, so the caller threads `WorkspaceFile.content_hash` through here.
7. **Trigger = 17.8's cron, not a server action.** When 17.8's `extract-files` cron sets a file `DONE`, it calls `syncWorkspaceChunks(file.id, 'USER_FILE', file.workspace_id, file.extracted_text, { filename, category, content_hash })` at the marked seam (`route.ts:189-194`). ⚠️ The cron's `findMany` select (`route.ts:84-91`) currently fetches **neither `category` nor `content_hash`** — Task 3 must add `category: true` + `content_hash: true` to that select. (Consistent with 17.8's "cron, not fire-and-forget" rule — no `void sync(...)`.)

### Lifecycle (prevents ghost retrieval)

8. **Deletion de-indexes:** `deleteFile()` (`app/actions/files.ts:954`) **and** `deleteFilesBulk()` (`:1324`) delete the file's `ContentChunk`s (`deleteMany` where `source_type='USER_FILE'`, `source_id IN ids`). **`ids` = the permission-filtered set actually deleted** — `deleteFilesBulk` deletes `files.map(f => f.id)` (`:1366`), not the raw `fileIds` input, so the chunk cleanup must mirror that set (never de-index files the caller couldn't delete). `ContentChunk` has **no FK to source** (polymorphic `source_id` across 5 source types) so `onDelete: Cascade` is infeasible — manual cleanup in the delete actions is the correct mechanism. The agent must never retrieve a deleted file's content.
9. **Re-index + dedupe:** a replaced file re-runs sync (delete-then-insert). **Dedupe:** before re-syncing, compare the file's current `content_hash` (17.8, threaded via the `meta` param per AC 6/7) against the hash recorded on its chunks' `metadata.content_hash`; if unchanged AND chunks exist → skip re-embed (no wasted cost). (`ContentChunk` has no `content_hash` column → store it in chunk `metadata`.)

### Retrieval

10. `retrieveContext()` `RetrievalOptions` gains `sourceTypes?: string[]` (keep the existing single `sourceType?` for back-compat). Current SQL: workspace clause `cc.workspace_id IS NULL OR cc.workspace_id = ${workspaceId}` (`retrieval.ts:166`); source clause `${sourceType}::text IS NULL OR cc.source_type = ${sourceType}::"SourceType"` (`:167`). Extend the source clause to `cc.source_type = ANY(${sourceTypes}::"SourceType"[])` when the array is provided (the enum-array cast is required — mirror the existing `::"SourceType"` cast at `:167`; there's no `ANY()` precedent in `lib/` to copy).
11. **Isolation:** a `sourceTypes: ['USER_FILE']` query must return ONLY the caller's workspace files. Note: the workspace clause *also* matches `workspace_id IS NULL` (global corpus), so it's the `source_type = USER_FILE` filter that excludes `LEGAL_DOCUMENT`; both filters together = strict isolation. Test BOTH sides: (a) read — a foreign-workspace `USER_FILE` chunk is never returned; (b) write — `chunkUserFile` never emits `workspace_id = null` (AC 2).
12. Embedding `text-embedding-3-small` (1536d); Cohere Rerank applies unchanged.

### Non-regression / cost

13. Legal-document chunking + retrieval unchanged (additive). Existing `[source_type, source_id]` + `[workspace_id]` indexes cover the new queries.
14. Embedding batched (≤100) + content-hash-gated (AC 9) so bulk file ingestion doesn't re-embed unchanged content.

### Quality

15. Chunk + embed runs inside 17.8's cron batch — this is **cron-batch work, not a request SLA**; budget it within the cron's `maxDuration` (≈20-page file well under the per-file slice).

### Completion Notes List
- **All 6 tasks implemented.** 31 new unit tests pass (chunk-workspace-document 11, sync-workspace-chunks 8, files-chunk-cleanup 4, retrieval +2 for `sourceTypes`); `pnpm typecheck` clean; `pnpm eslint` on all changed files clean.
- **AC 5 deviation (documented, no migration):** the Prisma `ContentRole` enum has **no `PARAGRAPH`** value (`schema.prisma:1828`), and the story forbids a schema migration (AC 1, File Locations). Paragraph content therefore maps to **`MARKDOWN_CHUNK`** — the established markdown-derived role used by the legal fallback (`chunk-document.ts`) — with `HEADING` (standalone single-line heading) and `TABLE` (majority pipe-delimited lines, e.g. 17.8's xlsx/csv tables) detected via `detectContentRole`. This honours AC 5's intent (distinguish heading/table/paragraph) within the existing enum. If a dedicated `PARAGRAPH` role is later wanted, that's a one-value enum migration.
- **Merge algorithm forked, not imported (AC 4):** `chunkFromMarkdown` is private and hardcodes `LEGAL_DOCUMENT`/`workspace_id:null`, so `mergeParagraphs` / `splitOversized` / `remergeToTarget` were copied verbatim into `chunk-workspace-document.ts` with the same tuning (target 400 / cap 1000 tokens, min 20 chars). `estimateTokenCount` is imported (shared, parameterless).
- **No context-prefix LLM step for files:** unlike `syncDocumentChunks` (which calls Anthropic for per-chunk prefixes), `syncWorkspaceChunks` embeds with an empty `contextPrefix` — the `contextual_header` (filename + category) carries the retrieval context. Cheaper and no Anthropic dependency on the upload hot path.
- **Trigger wired at the 17.8 seam:** `extract-files/route.ts` now calls `syncWorkspaceChunks(...)` on `DONE` (fail-safe try/catch — a chunking error never reverts the file's DONE status nor breaks the batch; it's retryable via re-sync / `extractWorkspaceFile`). Added `category` + `content_hash` to the cron's `findMany` select.
- **Dedupe (AC 9):** `syncWorkspaceChunks` compares the incoming `content_hash` against the `metadata.content_hash` of an existing chunk; on match it returns `{ skipped: true }` with no delete/embed (no wasted OpenAI cost).
- **Write-side isolation invariant (AC 2):** both `chunkUserFile` and `syncWorkspaceChunks` throw on an empty `workspace_id` rather than emit a null-workspace chunk (covered by tests). Read-side isolation (AC 11) is asserted at the SQL-parameter level in `retrieval.test.ts` — `$queryRaw` is mocked, so a true cross-tenant row test belongs in an integration suite (noted in the story's Testing section).
- **No schema migration, no new deps, no `vercel.json` change** — `USER_FILE` + `ContentChunk` pre-exist; `content_hash` lives in chunk `metadata`; the cron was registered by 17.8.
