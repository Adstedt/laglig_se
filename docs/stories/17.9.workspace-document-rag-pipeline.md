# Story 17.9: Chunk & Embed Workspace Documents into RAG Pipeline

## Status

Draft

## Story

**As a** system component,
**I want** workspace documents (Tiptap documents and extracted file text) embedded into the vector search pipeline,
**so that** the AI agent can semantically search across all workspace documentation.

## Story Context

**Existing System Integration:**

- Integrates with: Existing `ContentChunk` model and embedding pipeline (`lib/chunks/`), `DocumentVersion.extracted_text`, `WorkspaceFile.extracted_text` (from Story 17.8), `lib/agent/retrieval.ts`
- Technology: pgvector (HNSW), OpenAI `text-embedding-3-small`, Cohere Rerank v4
- Follows pattern: Existing `chunk-document.ts` / `sync-document-chunks.ts` / `embed-chunks.ts` pipeline for legal documents
- Touch points: `ContentChunk` model (new `WORKSPACE_DOCUMENT` source_type added to `SourceType` enum by this story, dependent on Story 17.1 schema migration), `retrieveContext()` function, chunk sync triggers

## Acceptance Criteria

**Chunking Tiptap Documents:**

1. When a `DocumentVersion` is saved (new version created), its `extracted_text` is chunked using the existing chunking pipeline
2. Chunking strategy for Tiptap documents: each top-level node (heading + its following paragraphs until next heading) becomes one chunk
3. If a section exceeds 1000 tokens, split at paragraph boundaries (same as existing markdown fallback)

**Chunking Uploaded Files:**

4. When a `WorkspaceFile` has `extracted_text` populated (after Story 17.8 extraction), it is chunked similarly
5. Chunking strategy for files: paragraph-merge approach (target 400 tokens, cap 1000) — same as existing legal document fallback

**Chunk Metadata:**

6. All chunks use `source_type: 'WORKSPACE_DOCUMENT'`
7. `workspace_id` is always set (workspace-scoped, never public)
8. `source_id` references `Document.id` for Tiptap documents, `WorkspaceFile.id` for uploaded files
9. `contextual_header` includes: document title, document type (for Documents) or filename and category (for WorkspaceFiles)
10. `content_role` set to appropriate value (PARAGRAPH, HEADING, etc.)

**Re-indexing:**

11. When a new `DocumentVersion` is created, old chunks for that `source_id` are deleted and replaced (full re-index of that document)
12. Re-indexing does not block the user — runs after the save completes

**Retrieval Integration:**

13. Existing `retrieveContext()` function in `lib/agent/retrieval.ts` extended to support `WORKSPACE_DOCUMENT` source type as a filter option
14. Retrieval can filter by `source_type: 'WORKSPACE_DOCUMENT'` only, `source_type: 'LEGAL_DOCUMENT'` only, or both
15. Embedding uses same `text-embedding-3-small` model (1536 dimensions) as legal documents
16. Cohere Rerank applied to workspace document results same as legal document results

**Integration Requirements:**

17. Existing legal document chunking and retrieval unchanged — new source type is additive
18. ContentChunk table indexes work for the new source type (existing `[source_type, source_id]` and `[workspace_id]` indexes)

**Quality Requirements:**

19. Chunking + embedding of a 20-page document completes within 30 seconds
20. Semantic search returns relevant workspace document chunks in top-5 results for targeted queries

  - [ ] Create and run Prisma migration (additive — no existing data affected)
  - [ ] Verify `ContentChunk` table indexes still cover new source type (they are on `[source_type, source_id]` and `[workspace_id]` — no changes needed)

### Chunk Generation

- [ ] **Task 1: Create workspace document chunking function** (AC: 1, 2, 3, 4, 5, 9, 10)
  - [ ] Create `lib/chunks/chunk-workspace-document.ts`
  - [ ] Export type `WorkspaceChunkInput` (mirrors `ChunkInput` from `chunk-document.ts` but with `source_type: 'WORKSPACE_DOCUMENT'`)
  - [ ] Function: `chunkWorkspaceDocument(doc: { id: string, title: string, type: string, extractedText: string, workspaceId: string }): WorkspaceChunkInput[]`
  - [ ] For Tiptap documents: split by heading boundaries (H1/H2/H3 create section breaks)
  - [ ] For uploaded files: use paragraph-merge approach (reuse logic from `chunkFromMarkdown` in `chunk-document.ts`)
  - [ ] Set `contextual_header` with document title and type
  - [ ] Respect 1000-token cap per chunk, target 400 tokens
  - [ ] Set `source_type: 'WORKSPACE_DOCUMENT'`, `workspace_id`, `source_id`

### Sync Pipeline

- [ ] **Task 2: Create workspace document sync function** (AC: 6, 7, 8, 11, 12)
  - [ ] Create `lib/chunks/sync-workspace-document-chunks.ts`
  - [ ] Function: `syncWorkspaceDocumentChunks(sourceId: string, workspaceId: string, extractedText: string, metadata: { title: string, type: string }): Promise<void>`
  - [ ] Delete existing chunks for this `source_id` and `source_type: 'WORKSPACE_DOCUMENT'`
  - [ ] Generate new chunks via `chunkWorkspaceDocument()`
  - [ ] Generate embeddings via existing `generateEmbeddingsBatch()` from `lib/chunks/embed-chunks.ts`
  - [ ] Insert chunks into `ContentChunk` table via `prisma.contentChunk.createMany()`
  - [ ] Note: unlike `syncDocumentChunks()` (which fetches its own document from DB), this function accepts content as a parameter — caller is responsible for providing `extractedText`

### Trigger Points

- [ ] **Task 3: Wire chunking into document save flow** (AC: 1, 12)
  - [ ] In `saveDocumentVersion()` server action (`app/actions/documents.ts`): after creating new version, call `syncWorkspaceDocumentChunks()`
  - [ ] Run async (don't block the save response) — use fire-and-forget pattern (see Dev Notes)
  - [ ] Log errors but don't fail the save

- [ ] **Task 4: Wire chunking into file upload flow** (AC: 4)
  - [ ] In `uploadFile()` server action (`app/actions/files.ts`) after text extraction (Story 17.8): if `extracted_text` is non-null, call `syncWorkspaceDocumentChunks()`
  - [ ] Same fire-and-forget pattern as Task 3

### Retrieval Integration

- [ ] **Task 5: Extend retrieveContext() with workspace document support** (AC: 13, 14, 15, 16, 17)
  - [ ] Modify `lib/agent/retrieval.ts` `RetrievalOptions` interface
  - [ ] Add optional `sourceTypes?: string[]` to `RetrievalOptions` (array for multi-source filtering, alongside existing `sourceType?: string` — keep backward compat)
  - [ ] In the SQL query: when `sourceTypes` is provided, use `cc.source_type = ANY(...)` filter instead of single `sourceType`
  - [ ] When `WORKSPACE_DOCUMENT` is included: the existing `workspace_id` filter already scopes results correctly (`AND (cc.workspace_id IS NULL OR cc.workspace_id = ${workspaceId})`)
  - [ ] Default behavior unchanged — callers that don't pass `sourceTypes` continue to work as before
  - [ ] Cohere Rerank applies to all results regardless of source type (no changes needed)

**Fire-and-forget pattern for non-blocking sync:**

```typescript
// In saveDocumentVersion() — app/actions/documents.ts:
const version = await prisma.documentVersion.create({ ... })

// Non-blocking: sync chunks in background
void syncWorkspaceDocumentChunks(document.id, workspaceId, version.extracted_text ?? '', {
  title: document.title,
  type: document.document_type,

return version // Return immediately — don't await the sync
```
