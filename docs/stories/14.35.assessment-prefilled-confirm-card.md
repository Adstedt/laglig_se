## Story

**As a** user assessing a flagged law change,
**I want** the agent's recommendation pre-filled into the assessment form so I confirm or adjust it rather than re-keying it,
**so that** the assessment is faster and not surfaced twice — while I still own the final, recorded decision.

## Context

In the change-assessment workflow the agent produces a **"Rekommendation"** block in its chat response — prose analysis plus a table (`Påverkansnivå`, `Status`, `Ikraftträdande`). The user then **separately** fills the **"Din bedömning"** form (`AssessmentResolution`): status select, impact select, notes. So the impact + status are **surfaced twice and manually re-keyed**, which is friction and invites transcription drift.

This story collapses recommendation + form into **one pre-filled, confirmable card**: the agent emits a **structured** recommendation (suggested impact + suggested status + a draft note), which pre-fills the form. The user **reviews, adjusts, and confirms.** Accountability is preserved — nothing auto-saves, every field is one-click overridable, and the agent's reasoning stays visible (mitigating automation bias). The saved `ChangeAssessment` remains the human's legal record (`assessed_by` = the user).

> **Design crux (for Architect):** today the recommendation is the agent's *prose + a formatted table* — it is **not** structured data the form can consume. The core of this story is making the agent emit a **structured recommendation** that drives the card. `save_assessment` currently takes `impactLevel` + `analysis` + `recommendations` but **no suggested status** (it hardcodes `status: 'REVIEWED'` on save). That needs to change.

## Acceptance Criteria

### Structured recommendation

1. The agent's assessment recommendation is produced as **structured data**: a suggested `impactLevel` (`HIGH|MEDIUM|LOW|NONE`), a suggested `status` (`REVIEWED|ACTION_REQUIRED|NOT_APPLICABLE|DEFERRED`), and an `analysis` (used to seed the draft note). Extend `save_assessment` (or the assessment-workflow proposal path) to carry the suggested `status` — it does not today.
2. The agent's analysis prose stays available as **visible reasoning** in/above the card so the user sees *why* the recommendation was made.

### Pre-fill + single surface

3. The recommendation pre-fills `AssessmentResolution`'s initial values — status select, impact select, and a draft note seeded from the analysis — via `useAssessmentForm`'s `initial` parameter.
4. **No double-surfacing:** the structured values (impact, status) appear **once** — as the pre-filled form controls — not also as a separate read-only "Rekommendation" table. (The analysis prose remains; the *structured* duplication is removed.)
5. **`Ikraftträdande` (effective date) is read-only display** — a fact, never a form field.

### Confirm + override (accountability)

6. The card is pre-filled but **NOT auto-saved** — the user must explicitly confirm ("Spara bedömning"). Every field is **one-click overridable** (change either select, edit the note).
7. The saved `ChangeAssessment` records `assessed_by` = the current user and stores **whatever the user confirmed** (not necessarily the agent's suggestion). The pre-fill is a starting point; the human owns the recorded values.

### Styling / non-regression

8. The card keeps the 14.23 spine & whisper treatment (already applied 2026-05-22).
9. Existing entry points unchanged: the chat-sidebar `AssessmentDetail` (already pre-fills from `existingAssessment`), re-editing a completed assessment ("Ändra"), and the `contextType='change'` workflow (flagging, navigation).

### Technical Constraints

- **Compliance record** — `ChangeAssessment` is the auditable legal artifact; `assessed_by` must remain the human. Pre-fill is a convenience, not the source of truth — the saved values are whatever the user confirmed. Do NOT auto-save the agent's suggestion.
- **Automation bias** — keep the agent's reasoning visible + require an explicit confirm; never silently default-and-close. Override must be one click.
- **Architectural risk** — the "structured recommendation → pre-fill" flow is the crux. Needs Architect (Winston) input on whether to extend `save_assessment` in place vs. a dedicated structured proposal, and how the recommendation reaches the inline form (it renders in the chat scroll area).
