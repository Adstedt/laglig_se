## Story

**As a** user iterating on a proposed change with the agent,
**I want** "adjust the last suggestion" to revise the existing proposal in place,
**so that** I don't get a duplicate card for the same section and the agent doesn't pointlessly re-read the whole document each time.

## Context

Observed 2026-06-11: after the agent proposed several section edits, the user said "ändra det sista förslaget till 2 gånger per år." The agent responded "Låt mig läsa dokumentet…" and re-ran `list_styrdokument` + `get_workspace_document`. Two problems:

1. **Redundant re-read.** The proposal isn't applied yet, so the document's live content is unchanged from when it was first read — and the proposed new content is already in the pending row's `params`. The re-read costs latency + tokens for nothing.
2. **Duplicate cards.** Re-proposing the same section creates a *second* `UPDATE_DOCUMENT` row for that heading. The batch consolidation fold (`applyConsolidatedDocumentEdits`) then folds both in `created_at` order (last-wins on the in-memory tree → same final content, but **two cards and two APPROVED rows** for one section). The system prompt's "never re-propose a pending action" rule makes the revise case ambiguous rather than clean.

Root cause for (1): `buildPendingActionsContext` summarizes each pending row via `summarizePendingAction`, whose `switch` has **no `UPDATE_DOCUMENT` / `ADD_DOCUMENT_SECTION` case** — it falls through to `default` and emits only the bare action-type string (`"UPDATE_DOCUMENT"`). So today the agent's pending context tells it a document edit is pending but carries *none of its content* — it cannot revise without re-reading.

The fix is a **supersede** semantic: when `update_document` proposes a (document, section) that already has a PENDING row in the same chat context, **replace that row's params** instead of inserting a new one — and give the agent the proposal content + explicit permission to revise pending proposals so it skips the re-read.

## Acceptance Criteria

1. **Supersede on same (document, section):** when `update_document` would create a proposal for a `documentId` + `sectionHeading` (case-insensitive) that already has a **PENDING** `UPDATE_DOCUMENT` row in the same chat context (same `workspace_id` + `user_id` + `conversation_id` + `context_type` + `context_id`), it **updates that existing row's `params`** (freshly recomputed `newSectionContentJson` / `oldSectionContentJson` / `changeSummary` / `newTitle` / `entity_version` / `creates_draft` / `newVersionNumber`) rather than inserting a second row. The card count does not grow; the existing card re-renders with the revised diff. Rename-only proposals (no `sectionHeading`) supersede on `documentId` alone.
2. **Pending content in agent context:** `buildPendingActionsContext` → `summarizePendingAction` gains an `UPDATE_DOCUMENT` (and `ADD_DOCUMENT_SECTION`) case that includes enough for the agent to revise without re-reading — at minimum the `sectionHeading` + a compact plain-text rendering of the proposed new text (and a current-text excerpt). Size-bounded (e.g. ~200 chars per side) to protect the token budget; the pending rows already carry full `params`, so no extra query.
3. **Prompt guidance:** the agent is told that when the user asks to *adjust* an existing pending proposal it should (a) NOT re-run `list_styrdokument` / `get_workspace_document` when the pending context already carries the needed content, and (b) re-issue `update_document` for that section — which now supersedes rather than duplicates. The existing "Återföreslå ALDRIG en åtgärd som redan listas i `<pending_agent_actions>`" rule ([`lib/agent/system-prompt.md`](../../lib/agent/system-prompt.md):131) is reworded to allow *revising* a pending proposal on explicit user request, while still forbidding redundant re-proposal of an unchanged one.
4. **Decision-log integrity:** superseding **reuses the same `PendingAgentAction` row id** (it is returned as `pendingActionId` in the tool envelope). The `AgentDecisionLog` row is NOT written by `update_document` — it is written by the central `wrapWithDecisionLog` wrapper ([`lib/agent/tools/decision-log.ts`](../../lib/agent/tools/decision-log.ts):51, applied to every tool at [`lib/agent/tools/index.ts`](../../lib/agent/tools/index.ts):60), which reads `output.data.pendingActionId` into `pending_action_id`. So a revision auto-writes a **second** decision-log row pointing at the **same** pending id — an intended audit trail (original proposal + revision), never an orphan. The dev requirement is therefore: (a) reuse the same pending id (Task 1.1/1.3 already do this), and (b) verify the approve/reject `updateMany` keyed by pending id (`app/actions/pending-agent-actions.ts`:1047/1258/1490) stamps both rows sensibly. Additionally stamp `revised_at` (+ keep a prior-params snapshot) on the pending row so the superseded content is recoverable. Do **not** manually mutate the prior decision-log row.
5. **Approve/consolidation unchanged:** a superseded-then-approved proposal flows through the single-approve path (`approvePendingAction`) and the consolidated-batch path (`applyConsolidatedDocumentEdits`) exactly as a first-time proposal — one section edit → one fold contribution → one version.
6. **Non-regression:** proposing a *different* section, or the same section in a *different* chat context, still creates a distinct card. Rename-only and `ADD_DOCUMENT_SECTION` behaviour for new proposals is unchanged. Non-document action types are unaffected (supersede is scoped to the `update_document` path only).

### Key Constraints

- **Terminology — "supersede" here ≠ document `SUPERSEDED` status.** Epic 17 uses `SUPERSEDED` as a *document lifecycle status* (`DRAFT → IN_REVIEW → APPROVED → SUPERSEDED → ARCHIVED`, `docs/prd/epic-17-document-management-system.md:95`). In THIS story "supersede" means **replacing a PENDING `PendingAgentAction` row's params in place** — a proposal-row operation, unrelated to document status. Don't touch the document status enum.
- **Recompute params on supersede, don't merge.** The execute path already recomputes `entity_version` (`document.updated_at.toISOString()`), `creates_draft`, `newVersionNumber`, `oldSectionContentJson` from the **live** document on every call. On supersede, overwrite the existing row's `params` with this fresh computation (plus the new `newSectionContentJson` / `newTitle`) — don't shallow-merge, or stale snapshot fields will diverge from the new diff.
- **Consolidation invariant (AC 5).** With supersede there is at most one PENDING row per (documentId, sectionHeading), so `applyConsolidatedDocumentEdits` cannot fold the same section twice. Keep an explicit assertion/test for this (Task 4.4) — it is the user-visible "two cards / two APPROVED rows" bug this story removes.
- **Scope match must mirror the context query.** AC 6 non-regression (different chat context → distinct card) only holds if the supersede lookup uses the *same* scope tuple as `buildPendingActionsContext` (`conversation_id` + `context_type` + `context_id`, not just `workspace_id`).
- No DB migration expected — supersede writes to the existing `PendingAgentAction.params` JSON (`revised_at` / prior snapshot live inside `params`, not new columns). Confirm before assuming; if a column is desired, hand the migration command to the user (do not apply it).
- **Don't add an index for the supersede lookup.** The existing `pending_agent_actions_scope_idx` (`[workspace_id, user_id, conversation_id, status]`, `prisma/schema.prisma:1592`) doesn't cover `context_type` / `context_id`, but the pending set per conversation is tiny (≤5 surfaced; query caps at 50) — the extra filter runs over a handful of rows. No new index warranted; don't "optimize" this prematurely.
