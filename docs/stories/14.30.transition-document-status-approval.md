## Story

**As a** user managing a styrdokument's lifecycle,
**I want** the agent to propose a status transition (submit for review, supersede, archive) as a reviewable card — but never to mark a document approved itself,
**so that** the agent can move documents along the workflow while the approval decision stays a deliberate, accountable human act.

## Context

Styrdokument move through a lifecycle ladder: `DRAFT → IN_REVIEW → APPROVED → SUPERSEDED → ARCHIVED` (Story 17.4). The agent will often be the one that knows a document is ready to advance ("det här utkastet är klart för granskning") or that an approved policy is now obsolete ("den här policyn har ersatts — arkivera den"). Letting it *propose* the transition closes the loop after a draft (14.24) or an assessment.

The hard rule: **the agent cannot move a document to `APPROVED`.** Approval is the point where a human takes responsibility for the document's content — separation of duties. The guard is enforced **server-side in the dispatch** (authoritative), with the tool refusing it up front as a first line of defence. Every other transition on the ladder is proposable, subject to the existing `VALID_STATUS_TRANSITIONS` validity check.

## Acceptance Criteria

### Schema

1. `PendingAgentActionType` enum is extended with `TRANSITION_DOCUMENT_STATUS`. Additive migration.
2. No new tables. `params` JSON shape:
   ```
   {
     documentId: string,
     newStatus: WorkspaceDocumentStatus,   // never APPROVED — see AC 4 / AC 13
     oldStatus: WorkspaceDocumentStatus,    // denormalised at propose time for the badge
     comment?: string,                      // optional status-change note
     entity_version: string                 // document.updated_at ISO at propose time (14.31)
   }
   ```

### Tool

3. New tool `lib/agent/tools/transition-document-status.ts`. Zod schema: `{ documentId: string, newStatus: WorkspaceDocumentStatus, comment?: string, execute: boolean }`.
4. **Separation-of-duties guard (tool level — first line):** if `newStatus === 'APPROVED'`, return a `ToolError`: "Agenten kan inte godkänna styrdokument — godkännande måste göras manuellt av en behörig användare." No pending row is created.
5. **Ladder validity (tool level):** read the document's current status; if `newStatus` is not in `VALID_STATUS_TRANSITIONS[currentStatus]` (or `getValidNextStatuses(currentStatus)`) from [`lib/validation/documents.ts:58`](../../lib/validation/documents.ts), return a `ToolError` naming the invalid move ("Ogiltig statusövergång: ${current} → ${proposed}."). No pending row is created. **Note:** the ladder permits `IN_REVIEW → APPROVED` (`lib/validation/documents.ts:63`), so this check does **not** by itself block self-approval — the dedicated APPROVED guard (AC 4 + AC 13) is the load-bearing protection.
6. On `execute: false` (valid, non-APPROVED transition): capture `oldStatus` + the document's `updated_at` (stored as `params.entity_version` per AC 2), create the `PendingAgentAction` row, return the envelope with `data: { pendingActionId }`.
7. Tool description (Swedish): use to submit a draft for review (`DRAFT → IN_REVIEW`), send a document back to draft, supersede or archive a document. Explicitly: **never** propose `APPROVED` — approval is a manual human step in the styrdokument UI.
8. `execute: true` is **not** supported (returns a `ToolError`).

### Renderer

9. New component `components/features/ai-chat/agent-action-renderers/transition-document-status-renderer.tsx`, registered in the `AgentActionCard` dispatch map and the batch card compact path.
10. `PENDING` state: header ("Ändra dokumentstatus") + the resolved document title + a status transition row showing `oldStatus → newStatus` as two `Badge`s. Reuse the Swedish status labels from `STATUS_CONFIG` (`DRAFT→Utkast, IN_REVIEW→Under granskning, APPROVED→Godkänd, SUPERSEDED→Ersatt, ARCHIVED→Arkiverad`) at [`components/features/documents/document-status-badge.tsx:18`](../../components/features/documents/document-status-badge.tsx), or render the existing `<DocumentStatusBadge>` (`:32`) directly — do not duplicate the label literals. The target status is **fixed** (read-only — the user approves the agent's proposal or rejects it; changing the target to APPROVED is not offered here). The `comment` field is editable (debounced 500ms → `updatePendingActionParams`). Controls: "Godkänn" / "Avvisa".
11. `APPROVED` state (proposal accepted): read-only "Status ändrad: ${oldStatus} → ${newStatus}" + link to the document at `/workspace/styrdokument/${documentId}/edit` (the `/edit` form, matching Story 14.24 AC 9; both the detail and edit routes are confirmed to exist). `REJECTED` / `EXPIRED` muted variants per 14.22.
12. `compact?: boolean` variant: collapsed summary "Status: ${oldStatus} → ${newStatus}".

### Approval Dispatch

13. `approvePendingAction` dispatch gains a `TRANSITION_DOCUMENT_STATUS` branch:
    - **Authoritative separation-of-duties guard:** if `params.newStatus === 'APPROVED'`, return `{ success: false, error: 'Agenten kan inte godkänna styrdokument.' }` **without** calling `updateDocumentStatus` and **without** marking the row APPROVED (the row stays `PENDING`, visibly blocked). This guard is independent of the tool-level guard (defence in depth — the dispatch is the trusted gate). (The shorter dispatch string vs. the fuller tool-level string in AC 4 is **deliberate** — the dispatch is a last-line backstop the agent should rarely hit; not an inconsistency for QA to flag.)
    - Otherwise → `updateDocumentStatus({ documentId, newStatus, comment })` at [`app/actions/documents.ts:345`](../../app/actions/documents.ts). That action re-validates the ladder against the document's *current* status and writes the `document_status_changed` `ActivityLog` row itself.
14. `result_ref = { documentId, status: newStatus }`. On dispatch failure (including a now-invalid transition because the live status drifted) the row stays `PENDING` with the Swedish error surfaced on the card.
15. `revalidatePath('/workspace/styrdokument')` and the document detail/edit path.

### Staleness Coordination (14.31)

16. `params.entity_version` (document `updated_at` at propose time) is captured. Independent of 14.31, `updateDocumentStatus`'s own ladder re-validation provides partial staleness protection (a transition proposed against a stale status fails with "Ogiltig statusövergång").

### Non-Regression

17. The existing styrdokument status UI, the `VALID_STATUS_TRANSITIONS` map, and `updateDocumentStatus` (including the manual human APPROVED path that sets `approved_by`/`approved_at`) are unchanged.
18. The batch card (14.23) includes `TRANSITION_DOCUMENT_STATUS` rows; the "Godkänn alla" handler dispatches them like any other type (the APPROVED guard fires per-row, leaving a blocked row `PENDING` with its error).
19. Reasoning blocks (14.20), tool-call collapsing (14.18), citation pills (14.16), chat streaming render correctly.
