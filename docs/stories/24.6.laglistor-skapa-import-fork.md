# Story 24.6: `+ Skapa ny lista` modal — add Importera path

## Status

Ready for Review

## Story

**As an** existing workspace member who wants to create a second (or third) law list — perhaps for a sibling entity or to migrate mid-contract from a competitor tool,
**I want** the in-app **"+ Skapa ny lista"** modal in the laglistor dropdown to offer an Importera path alongside the existing Börja-från-mall and Tom-lista paths,
**so that** I can bring an existing curated list into Laglig from the same modal I already use to create new lists — no extra menu, no separate page, no friction.

This is the **wiring story** — no new logic, no new routes. It extends the existing path-choice modal (`<CreateListChooser>` + `<ManageListModal>`, shipped in Story 12.10b) with a third entry-point card that mounts the existing `<ImportUploadStep>` from Story 24.2. The destination on success — `/laglistor/skapa/[importId]/granska` — is unchanged from Story 24.4.

## Acceptance Criteria

### Modal extension — third path-choice card

1. **Extend `components/features/document-list/create-list-chooser.tsx`** to add a third option card alongside the existing two ("Börja från mall" / "Tom lista"):
   - Card label: **"Importera"**
   - Card description: "Ladda upp en befintlig laglista från Excel, CSV eller klistra in raderna"
   - Icon: `Upload` from lucide-react (or `FileUp` — pick whichever pairs best visually with the existing `Library` + `Plus` icons)
   - Same visual treatment as the existing **"Tom lista"** card (no recommended-chip / primary-ring; the chooser already gives "Börja från mall" the recommended ring and we don't reorder that hierarchy)
   - Three cards on viewport ≥ `sm`: change the existing `grid-cols-1 sm:grid-cols-2` to `grid-cols-1 sm:grid-cols-3` so all three fit at desktop width
   - On mobile (single column) the cards stack vertically — Importera renders below Tom lista
   - New prop: `onSelectImport: () => void` callback added to `CreateListChooserProps` (matches the existing `onSelectBlank` and `onSelectTemplate` shape)

### Modal step — new `'import'` step in `<ManageListModal>`

2. **Extend `components/features/document-list/manage-list-modal.tsx`** with a new `'import'` step:
   - `Step` union becomes: `'choose' | 'from-template' | 'blank' | 'import'`
   - When `step === 'import'`: render `<ImportUploadStep>` (the component built in Story 24.2) inside the modal's `<DialogContent>`
   - The chooser's new `onSelectImport` callback simply does `setStep('import')`
   - "← Tillbaka" header affordance returns to `step = 'choose'` (mirroring how the existing template + blank steps allow backing up)
   - Modal stays open during the upload + parse phase; on successful navigation the modal closes

3. **`<ImportUploadStep>` mounts unchanged.** Same component, same submit behaviour as Story 24.2's reusable export. The component owns its dropzone + paste-tab UI; the modal owns the `<DialogContent>` chrome (header, close button, "Tillbaka" link if added).

   **Override the default routing via the `onSuccess` callback** that already exists on `ImportUploadStepProps`:
   ```ts
   <ImportUploadStep
     onSuccess={(importId) => {
       onOpenChange(false) // close the modal
       router.push(`/laglistor/skapa/${importId}/granska`)
     }}
   />
   ```
   This avoids the upload component pushing a route while the modal is still open (which causes a brief flicker of the granska page rendering on top of the dialog).

4. **No URL state for the modal step.** Modal-internal step is React state (matches the existing template / blank pattern). Bookmarkability of the entry point isn't a goal; the granska URL is what users would share, and that's still bookmarkable on its own.

### Reuse vs duplicate

5. **`<ImportUploadStep>` is reused from Story 24.2 unchanged.** Same component, same Zod-validated server-action calls. No styling fork. The modal is a thinner container than a page would have been — visual differences are limited to the surrounding dialog chrome.

6. **`commitImport`'s redirect target** is unchanged from Story 24.4: `/laglistor/{newLawListId}`. Users land in the standard workspace law-list view; no "imported from" badge or banner.

### Empty-state hint on `/laglistor`

7. **Update `/laglistor` empty state copy** (when user has zero law lists) to mention import:
   - Was: "Du har inga laglistor än. Skapa din första lista." with a single CTA
   - Now: "Du har inga laglistor än. Skapa en ny eller importera en befintlig." with the existing primary CTA + a small text-link "eller importera en lista" that **opens `<ManageListModal>` directly to the `'import'` step** (skips the chooser since the user has clearly signalled intent)

   **Implementation note on opening directly to `'import'`:** the existing `ManageListModal` initialises its step from a `useEffect` that depends on `mode` + `hasTemplates`. Add an optional `initialStep?: Step` prop on `ManageListModalProps` that overrides the default; when caller passes `initialStep='import'`, the modal opens with `<ImportUploadStep>` rendered. Default behaviour preserved (existing call sites continue to land on `'choose'` or `'blank'` exactly as today).

   **Empty state lives inside** `components/features/document-list/document-list-page-content.tsx` (loaded from `app/(workspace)/laglistor/page.tsx`). Read that file first to confirm where the empty-state branch is. Search for "inga" / "skapa din första" string fragments. The empty state may also be inside `LawListPrimaryAction` — both candidates worth checking.

### Coordination with Epic 25

8. **Epic 25's first-run onboarding modal** has its own dedicated path-choice surface (B.0 / B.1 prototype frames). Story B.1's import card routes the user away from the onboarding modal and into the same destination 24.6 lands at: `/laglistor/skapa/[importId]/granska` (Story 24.4's surface). Both entry surfaces — onboarding modal AND `<ManageListModal>` — converge on this URL after a successful upload.

    **Smoke verification step (manual):** once Epic 25 B.1 is live, open the first-run modal as a fresh user, click the Importera card, confirm the upload flow lands on `/laglistor/skapa/[importId]/granska` with the matched rows visible. Document result in story Completion Notes. If B.1 hasn't shipped when 24.6 is QA'd, this verification is deferred — 24.6's modal-card path is independently testable.

9. **`<ImportUploadStep>`** is the same component used by both surfaces (Epic 25 B.1's onboarding-modal import step AND 24.6's `<ManageListModal>` import step). The component does not own its container chrome — the parent does. Identical submit behaviour, identical destination.

**Source artefacts:**
- Epic PRD: `docs/prd/epic-24-import-existing-law-list.md` — see "Story 24.6" — minimal wiring story (PRD references the page-route shape; story body has been revised to modal-card per v0.4 Change Log entry)
- Story 24.2 — `<ImportUploadStep>` component (already designed for this — `onSuccess` callback exists exactly for the modal-host case)
- Story 24.4 — `/laglistor/skapa/[importId]/granska` route (post-success destination)
- Story 12.10b — `<ManageListModal>` + `<CreateListChooser>` (the existing path-choice surface this story extends)
- Epic 25 B.1 — sibling onboarding-modal entry surface that converges on the same granska destination

**Existing patterns to follow:**
- **Modal step state** — `<ManageListModal>`'s existing `Step = 'choose' | 'from-template' | 'blank'` pattern uses React state with `useState<Step>(initialStep)`. Add `'import'` as the fourth value. No URL-param state — the modal is short-lived and the back-button doesn't need to step through internal modal states.
- **Path-choice card layout** — copy the existing "Tom lista" card's visual treatment (no recommended-chip) for the new Importera card. Don't promote it above Tom lista; equal visual weight.
- **`<ImportUploadStep>`'s `onSuccess` override** — already wired in Story 24.2 for exactly this use case. No changes to the component needed.
- **Modal-with-router pattern** — see `app/(workspace)/laglistor/page.tsx`'s use of `<ManageListModal>` paired with the dropdown trigger. The router push on success is owned by the caller, not the modal.

**Key decisions baked in:**
- **Modal extension, not new page.** The `+ Skapa ny lista` modal already does path-choice for templates vs blank. Adding Importera as a third card keeps the same UX surface and avoids inventing a new entry route. (Original spec — `/laglistor/skapa` page route — was based on a misread of the existing surface; the page route does not exist and would be redundant.)
- **No URL state for the modal step.** Unlike a multi-step page (where bookmarkability matters), modal-internal step is React state. Users who want to share progress share the granska URL once an import exists. Browser back from the import upload step inside the modal closes the modal — that's the expected modal UX.
- **No special handling for "imported from" lists.** Once committed, an imported list is indistinguishable from a generated list (same `LawList` schema). The fact that it came from import is in the `committed_law_list.import_committed_from` back-reference + activity log.
- **Empty-state opens modal directly on `'import'` step.** When a user has zero lists and clicks "eller importera en lista", they've signalled intent — skip the chooser and land on the upload step. The default chooser entry is preserved for the dropdown trigger.

**Out of scope explicitly:**
- Onboarding modal changes (Epic 25's territory)
- Re-import support (delete + re-upload is the only "redo")
- Differentiation of imported lists in the `/laglistor` index view
- New page routes — the import path enters and exits through existing surfaces
