# Story 7.3: Personalkort — create & edit employee (tabbed modal)

## Status

Done

## Story

**As an** HR manager,
**I want** the employee card to open as a tabbed modal like the law-list-item detail,
**so that** it feels consistent with the rest of the app and I never leave the register.

## Acceptance Criteria

1. Personalkort is a **modal, not a page route** — reuse the canonical `SplitPanelModal` shell (`components/shared/split-panel-modal/index.tsx`), the same primitive behind `LegalDocumentModal`. **Do not build a new dialog.**
2. Opened via the `?anstalld=<id>` URL param (History API — the param wiring and `'ny'` sentinel were built in Story 7.2; this story mounts the modal on that state), from a row click or "Lägg till anställd"; closed by removing the param.
3. **Tabs on top** using the shared `Tabs` primitive (`components/ui/tabs.tsx`) — compliance subset only: **Personalinformation**, **Anställning**, **Semester** (no Skatt/ATK/Ingående Saldo). Rendered in the modal's `leftPanel`.
4. The modal's **`rightPanel` = compliance summary sidebar**: current Status (Aktiv/Inaktiv) and assigned Kollektivavtal, using the `DetailRow`/details-box conventions. The "Ej komplett" reasons slot is a **placeholder** (Story 7.4 owns the completeness rule and fills it). AI chat (`renderChat`) is **omitted** in this story (epic marks it optional; the employee-aware chat is Story 7.7).
5. Gated by `employees:manage` for edit/create; `employees:view` may open read-only (fields disabled, no save).
6. Only **förnamn + efternamn** block save (matching the model's lenient-required design). All other fields optional — missing LAS-critical data becomes "Ej komplett" in 7.4, not a save blocker.
7. Personnummer validated when present: Swedish format (`YYYYMMDD-XXXX` or `YYMMDD-XXXX`) **+ Luhn checksum** (net-new validator — none exists in the codebase); inline validation errors.
8. Save via server actions in `app/actions/employees.ts` under `withWorkspace(..., 'employees:manage')`; personnummer **encrypted on write** via `encryptPersonnummer` (the 7.1 write-side contract); `created_by` set from ctx on create, `updated_at` via Prisma `@updatedAt`. Optimistic list update via a lifted `onEmployeeChange` callback (same pattern as the law-list modal).
9. Manager field is a dropdown of existing **active** employees, excluding the employee being edited (no self-management).
10. **Kollektivavtal assignment available in both create and edit mode** — a select of the workspace's uploaded agreements (new read action over the 7.1 `CollectiveAgreement` model). If none exist, show an inline **"Ladda upp kollektivavtal" placeholder affordance** (disabled with "kommer snart" hint) — the real shared upload component is Story 7.5 and gets wired here in 7.5. Writes `Employee.collective_agreement_id`.
11. Create mode = the same modal opened with `?anstalld=ny` (empty form); Save creates the record and optimistically inserts the new row into the register.

### Permissions

`employees:manage` gates create/edit (server-enforced in actions + UI-disabled); `employees:view` opens read-only. Both currently held by OWNER + HR_MANAGER only. The server action is the real boundary — UI disabling is UX, not security.

### Traps

1. **Never send plaintext personnummer back in action responses or logs** — return sanitized views only; the round-trip is: form plaintext → action → encrypt → store → sanitized view back.
2. `'ny'` is the create sentinel — never look it up as an id (7.2 reserved it).
3. `updateEmployee` must not allow cross-workspace writes: filter on `workspace_id` in the write itself, and validate `manager_id`/`group_id`/`collective_agreement_id` belong to ctx workspace (otherwise a crafted request links a foreign record).
4. `full_time_equivalent` is `Decimal(4,3)` (0–1), shown as % in UI — convert at the edge, store the 0–1 decimal; `average_weekly_hours`/`vacation_days_paid` are `Decimal(5,2)`. **Client-side these arrive already serialized** (7.2's `toEmployeeRow` — Decimals don't cross the RSC boundary); the form works with plain numbers/strings, the server action converts to Prisma-acceptable values on write.
5. Keep panels' ScrollArea wrappers or long forms will clip (shell doesn't scroll).
6. Story 7.2's dev may have shipped names that differ from assumptions here — **Task 0 binds first**.

**On the dev-flagged personnummer-wipe trade-off specifically:** the ordinary path is safe — a manage user with a healthy ciphertext gets the decrypted value prefilled, so editing any other field re-submits and re-encrypts it (no wipe). But the flagged edge is worse than "corrupt ciphertext gets cleared": `decryptPersonnummer` also throws when `ENCRYPTION_KEY` is missing/rotated, and `toView` then degrades EVERY row to the mask. During such a transient ops misconfiguration, any routine edit by an HR manager silently nulls a personnummer that was fully recoverable (fix the key and it decrypts again). The write-side fail-closed guard cannot catch this, because the prefill already emptied the field — empty skips encryption entirely and writes null. Silent, workspace-wide-on-touch, irreversible once overwritten: this is a genuine data-loss defect class, not a cosmetic trade-off, hence DATA-001 at medium and the CONCERNS gate.
