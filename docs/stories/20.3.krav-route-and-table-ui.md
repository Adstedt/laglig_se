# Story 20.3: `/krav` Route & Workspace Krav Table UI

## Status

Draft

## Story

**As a** workspace user,
**I want** a `/krav` page that shows every kravpunkt in my workspace in a single filterable table,
**so that** I can triage compliance gaps, my own assignments, and evidence-missing kravpunkter in one click without opening dozens of law item modals.

## Story Context

**Existing System Integration:**

- Integrates with: `getWorkspaceRequirements` server action (Story 20.2); `<AssigneeEditor />` cell editor; the existing `LegalDocumentModal` + `KravpunkterChecklist` for deep-link edits; the left sidebar's **Efterlevnad** accordion
- Technology: Next.js 14 App Router (`app/(workspace)/...`), TanStack Table v8, `@tanstack/react-virtual`, SWR, Radix UI, Lucide icons, hard-coded Swedish copy
- Follows pattern:
  - Column definitions & virtualization from `components/features/document-list/document-list-table.tsx`
  - URL-param-persisted filter chips + "Rensa (N)" clear-all from `components/features/document-list/filter-bar.tsx`
  - Sortable headers from `components/features/document-list/sortable-header.tsx`
  - Assignee inline edit from `components/features/document-list/table-cell-editors/assignee-editor.tsx`
  - Route group: `(workspace)` — workspace resolved from cookie, no slug in URL
- Touch points:
  - `app/(workspace)/krav/page.tsx` (new — server component shell)
  - `components/features/krav/krav-page-content.tsx` (new — client component with table + filters)
  - `components/features/krav/krav-table.tsx` (new — TanStack table)
  - `components/features/krav/krav-filter-chips.tsx` (new — 4 preset chips + "Rensa (N)")
  - `components/features/krav/hooks/use-workspace-requirements.ts` (new — SWR hook calling Story 20.2's action)
  - `components/layout/left-sidebar.tsx` (add "Krav" subitem under Efterlevnad)
  - Reuse: `<AssigneeEditor />`, `<SortableHeader />`, `<LegalDocumentModal />`

**Motivation:**

Stories 20.1 and 20.2 deliver the data model and the query. This story is the surface users actually interact with — the product value (audit triage, "Mina krav", evidence gap visibility) only lands when `/krav` exists. Design alignment is non-negotiable here: the new page must feel like part of the existing document-list family, not a new design.

**Depends on:** Stories 20.1 and 20.2 must land first (the column, the resolver, and the aggregation action all need to exist and be stable).

## Acceptance Criteria

**Route & Navigation:**

1. New route `app/(workspace)/krav/page.tsx` renders the workspace krav overview. Workspace is resolved from the cookie (`ACTIVE_WORKSPACE_COOKIE`) via the same pattern as sibling `(workspace)` routes — no slug in the URL.
2. `components/layout/left-sidebar.tsx` includes a new `subItem { title: 'Krav', href: '/krav' }` under the existing **Efterlevnad** accordion, positioned between "Ändringar" and "Mallar" (or at the end — pick whichever reads most naturally; document the choice in the story's completion notes).
3. Active-route highlighting on the sidebar item works in both expanded and collapsed sidebar states, matching the existing `isActive(item.href)` convention.
4. Icon: `ListChecks` from `lucide-react` (falls back to `CheckSquare` if `ListChecks` is unavailable).

**Page Shell:**

5. Page header shows the title "Krav" (Swedish) and a one-line description: "Alla kravpunkter i din arbetsyta — filtrera efter luckor, mina krav eller krav som saknar bevis."
6. Below the header: filter chips row, then search input, then the table.
7. Empty state (no krav in the workspace at all): Swedish copy + secondary link to "Öppna Mina listor" pointing at `/laglistor`.
8. Filtered-empty state (current filter returns 0 rows): Swedish copy inviting the user to clear filters + the "Rensa (N)" button inline.
9. Loading state: skeleton rows matching the column count (reuse the skeleton pattern from the document list table or similar).
10. Error state: generic error boundary matching existing pages — title "Kunde inte ladda krav", retry button.

**Filter Chips:**

11. Four chips, left-to-right: **Alla**, **Luckor**, **Mina krav**, **Saknar bevis**.
12. Default active chip on first load: **Luckor** (reflects the primary JTBD of gap triage).
13. Exactly one chip is active at a time. Clicking the active chip does nothing (does not deselect).
14. Chip active state matches the visual style used in `filter-bar.tsx` (border/background treatment).
15. Each chip shows a badge with the current row count for that filter — numbers come from parallel lightweight count calls OR from the `total_count` field of the aggregation action (cheapest acceptable implementation).
16. A "Rensa (N)" button appears to the right of the chips ONLY when either a non-default chip is active OR the search has a non-empty value; `N` = number of non-default adjustments (default chip + empty search = no button).
17. Clicking "Rensa (N)" resets to chip=Luckor + empty search.

**URL State:**

18. The active filter and search term are reflected in URL query params: `?filter=all|gaps|mine|needs_evidence&search=...`.
19. Deep-linking to e.g. `/krav?filter=mine` loads directly with that preset active.
20. Omitting `filter` defaults to `gaps`.
21. URL updates use `router.replace` (not `push`) so filter changes don't pollute the back-button history.

**Search:**

22. Single search input above the table, placeholder "Sök kravpunkter...", debounced 300ms, persists to the `search` URL param and the aggregation action's `search` input.
23. Search is AND-combined with the active filter preset (handled in the action — this story just passes both through).

**Table Columns:**

24. Column order (left → right):
    1. Status (inline checkbox — `is_fulfilled` toggle)
    2. Kravpunkt (text; truncated to two lines with a tooltip for full text; width-flex)
    3. Lag (law name; clickable → opens `LegalDocumentModal` for the parent law item, scrolled to the krav)
    4. Laglista (law list name; plain text, secondary color)
    5. Ansvarig (reuse `<AssigneeEditor />` inline — with the `inherited` variant from Story 20.1 when `effective_assignee.is_inherited` is true)
    6. Bevis (icon badge: `ShieldCheck` green with count when linked; `ShieldAlert` amber when `bevis_required && count === 0`; muted dash when not required)
    7. Uppdaterad (relative date: "2 dagar sedan", tooltip with absolute timestamp)
25. Columns 2, 3, 5, 7 are sortable via `<SortableHeader />`; the sort corresponds directly to the `sort` input of `getWorkspaceRequirements`.
26. Default sort: `updated_at desc`.
27. Column widths, header padding, row height, and text sizes match `document-list-table.tsx` conventions.

**Virtualization:**

28. Virtualization auto-enables above 100 rows (same threshold as `document-list-table.tsx`). Below 100: normal rendering.

**Inline Edits:**

29. The status checkbox in column 1 calls `updateRequirement({ isFulfilled })` (existing action); row updates optimistically; SWR revalidates.
30. The assignee picker in column 5 calls `updateRequirement({ responsible_user_id })`; row updates optimistically; SWR revalidates. If the krav was previously inherited and the user picks a value, the row's `effective_assignee.is_inherited` flips to `false`.
31. There is NO inline edit for text, comment, evidence, or reorder on this page — those stay in the modal. The only edits here are status and assignee.

**Deep-link Modal:**

32. Clicking the "Lag" cell opens the existing `LegalDocumentModal` for the parent `LawListItem`.
33. The modal scrolls to (and briefly highlights) the specific kravpunkt row — either via an anchor/ref the modal already supports, or by passing a `focusRequirementId` prop (add to the modal if it doesn't exist; keep it optional to avoid breaking existing callers).
34. Closing the modal returns to `/krav` with no state loss (filters, scroll position, search intact).

**Performance:**

35. First meaningful render after initial navigation is under 500ms on a seeded workspace of ~500 krav; filter changes re-render under 200ms after the server action returns.
36. No N+1 network requests — the page issues exactly one aggregation call per filter/search/sort/page combination.

**Accessibility:**

37. Keyboard navigation: Tab between filter chips, Enter to activate; Arrow keys to move row focus; Enter on the "Lag" cell opens the modal; Space toggles the fulfilled checkbox.
38. Screen-reader: table has a descriptive `aria-label` ("Kravpunkter i arbetsytan"); chips are `role="tab"` within a `role="tablist"` equivalent, OR plain buttons with `aria-pressed` — whichever matches the existing `filter-bar.tsx` pattern.
39. Color is not the only signal for "bevis saknas" — icon shape differs as well (ShieldAlert vs ShieldCheck).

**Regression:**

40. No changes to `document-list-table.tsx`, `compliance-detail-table.tsx`, `legal-document-modal.tsx`, or sibling document-list components other than the optional `focusRequirementId` prop on the modal (AC 33). Verified by smoke test of the existing `/laglistor` flows.

## Dev Notes

**Design alignment principle:**

The `/krav` page is a new surface, but every primitive it uses must come from the existing design system:

| Concern | Reused from |
|---|---|
| Column definitions | `document-list-table.tsx` |
| Row virtualization | `@tanstack/react-virtual` with 100-row threshold |
| Sortable headers | `sortable-header.tsx` |
| Inline assignee edit | `assignee-editor.tsx` |
| Filter chips visual | tokens from `filter-bar.tsx` (border, active state, badge count) |
| URL-param filter state | `filter-bar.tsx` convention |
| Modal deep-link | `LegalDocumentModal` (with optional `focusRequirementId`) |
| Sidebar structure | `left-sidebar.tsx` Efterlevnad accordion |
| Icon library | `lucide-react` |
| Copy | hard-coded Swedish |

If you find yourself building a new primitive, stop and ask — it's almost certainly a reuse opportunity you missed.

**Workspace members prop threading:**

Don't fetch members per-cell. Fetch once in the page (server component → client component prop), pass down as `members: WorkspaceMemberOption[]` to `KravTable` → to each `AssigneeEditor`. Exact pattern used by `document-list-page-content.tsx` — mirror it.

**Why `router.replace` not `router.push`:**

Filter toggling is not "navigation" in the user's mental model — hitting back-button after toggling the filter three times should leave the page, not walk backwards through filter states. `replace` matches the behavior of `filter-bar.tsx`.

**Modal scroll-into-view:**

The existing modal renders `<KravpunkterChecklist />` inside an accordion. For the `focusRequirementId` feature, the checklist component needs to:
1. Expand its accordion section if collapsed
2. Scroll the matching row into view
3. Apply a brief highlight (e.g., a `bg-yellow-100/50` class that fades after 1.5s)

Confirm the current structure of `kravpunkter-checklist.tsx` before adding — the row element must have a stable id/ref target.

**What this story does NOT do:**

- No CSV/PDF export — deferred to a follow-up epic.
- No bulk actions (multi-select + bulk reassign/toggle).
- No grouping/accordion view.
- No change-indicator badges.
- No per-krav due dates or priority.
