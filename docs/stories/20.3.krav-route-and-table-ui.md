# Story 20.3: `/krav` Route & Workspace Krav Table UI

## Status

Approved

## Story

**As a** workspace user (compliance manager, responsible owner, or auditor),
**I want** a `/krav` page that shows every kravpunkt in my workspace in a single filterable table that reuses the existing document-list design language,
**so that** I can triage compliance gaps, my own assignments, and kravpunkter missing bevis in one click without opening dozens of law-item modals.

## Acceptance Criteria

### Route & Navigation

1. New route at `app/(workspace)/krav/page.tsx` renders the workspace krav overview. Workspace context inherits from the existing `(workspace)` layout (cookie-based, no slug in URL). Page is a **server component** that reads `searchParams: Promise<{ filter?: string; search?: string; sort?: string; dir?: string }>` per the Next.js 16 pattern used by every other route in this group, awaits it, and passes the parsed values to a `<KravPageContent />` client component. [Source: architecture/17-coding-standards.md#17.3 — "await params" Next.js 16 pattern; verified in `app/(workspace)/browse/eu/[type]/page.tsx:10, 20`]
2. `components/layout/left-sidebar.tsx` gains a new `subItem` under the existing **Efterlevnad** accordion (currently at lines 95–105 of that file). Insert `{ title: 'Krav', href: '/krav' }` **immediately after `{ title: 'Mina listor', href: '/laglistor' }`** (so the order becomes: Mina listor → Krav → Ändringar → Mallar). Rationale: "Krav" is a cross-list drill-down of "Mina listor", so adjacent placement matches the mental model.
3. Active-route highlighting uses the existing `isActive(item.href)` convention — no new logic required, matching `/laglistor/mallar` which already works as a subItem.
4. **No icon on the subItem** — the sidebar's `subItems` type is `{ title: string; href: string }[]` (line 79 of `left-sidebar.tsx`). The parent **Efterlevnad** entry already carries the `Scale` icon for the accordion group.

### Page Shell

5. Page header: title "Krav" (Swedish) + one-line description "Alla kravpunkter i din arbetsyta — filtrera efter luckor, mina krav eller krav som saknar bevis."
6. Layout below header (top to bottom): filter chips row, search input, table (in a single scrollable container).
7. **Empty-workspace state** (no krav at all, not just under the active filter): centered Swedish copy "Du har inga kravpunkter ännu." + a secondary link "Öppna Mina listor" pointing to `/laglistor`.
8. **Filtered-empty state** (current filter returns 0 rows but krav exist in the workspace): Swedish copy "Inga kravpunkter matchar de aktiva filtren." + the "Rensa (N)" button inline so the user can reset.
9. **Loading state**: 10 skeleton rows matching the table's column count and row height. Reuse the `<Skeleton />` component from `components/ui/skeleton.tsx` (shadcn/ui). [Source: architecture/17-coding-standards.md#17.3 — shadcn/ui mandated]
10. **Error state**: error boundary catches `getWorkspaceRequirements` failures, renders title "Kunde inte ladda krav" + a "Försök igen" button that calls SWR `mutate()` on the workspace-requirements key.

### Filter Chips

11. Four chips, left-to-right in this order: **Alla**, **Luckor**, **Mina krav**, **Saknar bevis**.
12. Default active chip on first load (when no `?filter=` present): **Luckor** — reflects the primary JTBD of gap triage per Epic 20 §Value Delivered.
13. Exactly one chip is active at a time. Clicking the active chip is a no-op (does not toggle to a "nothing active" state; `Alla` is the explicit "show everything" chip).
14. Chip visual state mirrors `filter-bar.tsx` tokens (border/background treatment for active vs inactive). Do NOT fork the component — the four-chip set here is different from the document-list's content-type / compliance filters, so `KravFilterChips` is a new pure component but uses the same Tailwind utility classes as `content-type-filter.tsx` and `compliance-filters.tsx`.
15. Each chip shows a badge with that preset's row count. Counts come from **`getWorkspaceRequirementCounts()`** — the sibling action shipped in Story 20.2. Single call on page mount, revalidated when any row's `isFulfilled` or `responsibleUserId` changes (SWR cross-key invalidation).
16. A **"Rensa (N)"** button appears to the right of the chips when the active state is anything other than (chip = Luckor AND search = ""). `N` counts the distinct non-default adjustments: 1 if only the filter differs, 1 if only search is non-empty, 2 if both.
17. Clicking "Rensa (N)" resets the URL params so filter defaults to `gaps` and search is cleared. Uses `router.replace('/krav', { scroll: false })`.

### URL State

18. Active filter, search, and sort are mirrored in URL query params: `?filter=all|gaps|mine|needs_evidence&search=...&sort=<field>&dir=asc|desc`. Default values are **omitted** from the URL (no `?filter=gaps` appears when on the default). **Graceful fallback:** if the URL carries an unknown value for `filter`, `sort`, or `dir` (e.g., a hand-crafted `?filter=banana`), the client silently coerces to the built-in default (`gaps` / `updated_at` / `desc`) on first render and strips the malformed param on the next `router.replace`. Never render an error page for bad URL params.
19. Deep-linking to e.g. `/krav?filter=mine` or `/krav?filter=needs_evidence&sort=law_name&dir=asc` loads with that state applied on first render (no flash of default state).
20. Writes use `router.replace(newUrl, { scroll: false })` — mirrors the established convention at `components/features/document-list/document-list-page-content.tsx:257` and `compliance-filters.tsx:103`. The `scroll: false` option is non-negotiable — without it, toggling a chip scrolls the page to the top.
21. URL reads use `useSearchParams()` from `next/navigation` inside `<KravPageContent />` (client component) — the server component at `app/(workspace)/krav/page.tsx` only uses the initial awaited `searchParams` for SSR hydration.

### Search

22. Single `<Input />` above the table (shadcn/ui primitive), placeholder **"Sök kravpunkter..."**, debounced 300ms before writing to the URL `search` param and triggering a refetch.
23. Search is AND-combined with the active filter preset (already handled by Story 20.2's server action — this story just passes both values through via the SWR hook's key).

### Table Columns

24. Column order (left → right):

    | # | Column | Content | Interaction |
    |---|---|---|---|
    | 1 | **Status** | checkbox reflecting `row.isFulfilled` | Inline toggle — calls `updateRequirement({ requirementId: row.id, isFulfilled: !row.isFulfilled })` (existing action from Story 17.16) |
    | 2 | **Kravpunkt** | `row.text`, truncated to 2 lines with Radix Tooltip showing full text | Non-interactive |
    | 3 | **Lag** | `row.lawName`, styled as a link | **Click opens `<LegalDocumentModal />`** with `listItemId={row.lawItemId}`, `focusField="kravpunkter"`, and the new `focusRequirementId={row.id}` prop (see AC 33) |
    | 4 | **Laglista** | `row.laglistaName`, `text-muted-foreground` | Non-interactive |
    | 5 | **Ansvarig** | `<AssigneeEditor />` inline, with `variant={row.effectiveAssignee.isInherited ? 'inherited' : 'direct'}` | Inline — calls `updateRequirement({ requirementId: row.id, responsibleUserId: newId })` (shipped via Story 20.1) |
    | 6 | **Bevis** | `ShieldCheck` (green) + `row.evidenceCount` when count > 0; `ShieldAlert` (amber) with tooltip "Saknar bevis" when `row.bevisRequired && row.evidenceCount === 0`; muted `—` otherwise | Non-interactive |
    | 7 | **Uppdaterad** | relative date from `row.updatedAt` via `date-fns` with the Swedish locale | Radix Tooltip shows absolute timestamp |

25. Columns **1 (Status), 3 (Lag), 4 (Laglista), 7 (Uppdaterad)** are sortable via `<SortableHeader />` from `components/features/document-list/sortable-header.tsx`. Mapping to Story 20.2's `sort.field` enum: Status → `'is_fulfilled'`, Lag → `'law_name'`, Laglista → `'laglista_name'`, Uppdaterad → `'updated_at'`. Columns **2 (Kravpunkt), 5 (Ansvarig), 6 (Bevis)** are NOT sortable — render their header labels without the sort-affordance chevron.
26. Default sort: `{ field: 'updated_at', direction: 'desc' }`.
27. Column widths, header padding, row height, and text sizes mirror `document-list-table.tsx` conventions. Pull the exact Tailwind classes from that file rather than introducing new spacing tokens.

### Virtualization

28. Virtualization auto-enables above **100 rows** (same threshold as `document-list-table.tsx:838`). Below 100: render normally via `useReactTable().getRowModel().rows.map(...)`. Above: `useVirtualizer` from `@tanstack/react-virtual` (already in the dependency tree per `document-list-table.tsx:12`).

### Pagination ("Ladda fler")

29. Cursor-based pagination per Story 20.2: the SWR hook calls `getWorkspaceRequirements({ ..., cursor, limit: 50 })`. When `data.nextCursor !== null`, render a **"Ladda fler"** button below the table that triggers a fetch with the stored `nextCursor` and appends results to the rendered list. This story does NOT implement infinite-scroll on viewport — an explicit button keeps control simple and matches the non-destructive read pattern.

### Inline Edits

30. **Status toggle** (col 1): optimistic update of `row.isFulfilled`, call `updateRequirement({ requirementId: row.id, isFulfilled })`, on error revert + show toast "Kunde inte uppdatera kravpunkt" (existing Swedish copy from `law-list-item-requirements.ts:340`).
31. **Assignee change** (col 5): optimistic update of `row.effectiveAssignee`, call `updateRequirement({ requirementId: row.id, responsibleUserId })`, on success re-compute `isInherited` locally (if a UUID was set, `isInherited` becomes `false`; if `null`, recompute via the parent law item's responsible). On error revert + show toast.
32. **No other inline edits** on this page. Text, comment, evidence linking, reorder, delete — all stay in the `<LegalDocumentModal />`. The table is a read + light-edit surface, not a full CRUD editor.

### Deep-link to Modal

33. The **"Lag"** cell click (column 3, AC 24) opens the existing `<LegalDocumentModal />` with:
    - `listItemId={row.lawItemId}`
    - `focusField="kravpunkter"` (existing prop at `legal-document-modal/index.tsx:52`)
    - `focusRequirementId={row.id}` — **new optional prop to add** (see Task 12). Default behaviour when the prop is absent stays unchanged, so existing callers (`document-list-table.tsx`, `compliance-detail-table.tsx`) are unaffected.
34. When `focusRequirementId` is set, `kravpunkter-checklist.tsx` must: (a) ensure its accordion section is open, (b) scroll the matching krav row into view via `scrollIntoView({ block: 'center', behavior: 'smooth' })`, (c) apply a 1.5-second highlight class (prescribed utility: `bg-accent/40` fading via `transition-colors`).
35. Closing the modal returns the user to `/krav` with filters, search, sort, scroll position, and "Ladda fler"-loaded rows **intact**. Render the modal as a sibling of the table inside `<KravPageContent />` (not as a modal route) so closing it does not trigger a router navigation.

### Performance

36. First meaningful render after initial navigation to `/krav` ≤ **500ms** on a seeded workspace of ~500 krav (below the virtualization threshold) and ≤ **800ms** above the threshold. [Source: architecture/22-performance-architecture.md#22.2.1 — Dynamic Page Navigation P50 300ms / P95 800ms]
37. Filter change re-renders under **200ms** after Story 20.2's action returns. [Source: architecture/22-performance-architecture.md#22.2.1 — Real-time Update target]
38. Exactly **one** aggregation call per `(filter, search, sort, cursor)` combination (SWR dedup with `keepPreviousData: true` to prevent skeleton flash during filter toggles). [Source: architecture/21-caching-and-data-fetching-strategy.md#3.1 — `keepPreviousData` is the established convention]

### Accessibility

39. **Keyboard navigation**: Tab between filter chips, Enter to activate; Arrow keys move row focus; Enter on the "Lag" cell opens the modal; Space toggles the Status checkbox; Enter on the "Ladda fler" button triggers the next page fetch.
40. **Screen-reader**: table has `aria-label="Kravpunkter i arbetsytan"`; chips are plain `<button>` with `aria-pressed={isActive}` (matching the pattern in `filter-bar.tsx` filter siblings, not a new `role="tablist"`); search input has an explicit `<Label />` (shadcn/ui).
41. Color is not the only signal for **Bevis** — the `ShieldAlert` vs `ShieldCheck` shapes differ so colorblind users can distinguish "missing" from "present" without relying on the amber/green hue.

### Regression

42. No changes to `document-list-table.tsx`, `compliance-detail-table.tsx`, `grouped-document-list-table.tsx`, `grouped-compliance-table.tsx`, `filter-bar.tsx`, or sibling document-list components — with **one intentional exception**: the optional `focusRequirementId` prop added to `<LegalDocumentModal />` and `<KravpunkterChecklist />` per AC 33–34. Verified by smoke test of `/laglistor` (all three views: table, grouped, compliance).
