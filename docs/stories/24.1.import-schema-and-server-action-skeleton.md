# Story 24.1: Import — schema + server-action skeleton

## Status

Draft

## Story

**As a** developer building the Epic 24 import pipeline,
**I want** the underlying Prisma models, Zod schemas, and server-action surface stubbed and tested,
**so that** Stories 24.2 (parser), 24.3 (matcher), 24.4 (review), 24.5 (admin queue), and 24.6 (skapa-rewire) can fill in their respective areas without redoing schema or action plumbing.

This is the **schema + server-action layer only** — no UI, no parsing, no matching. The story proves the data model can carry an import lifecycle (`UPLOADED → PARSED → MATCHING → AWAITING_REVIEW → COMMITTED | FAILED`), per-row staging with confidence scores, and a catalog-ingest-request queue, plus exposes eight server-action stubs that subsequent stories implement.

## Acceptance Criteria

### Schema

1. **New Prisma migration `<TIMESTAMP>_add_law_list_import/migration.sql`.** Generate via `pnpm prisma migrate dev --create-only --name add_law_list_import` to match the canonical `YYYYMMDDHHMMSS_` pattern. Reference example: `prisma/migrations/20260427120000_add_finding_closure_metadata/migration.sql` (additive-only precedent shape).

2. **Three new enums:**
   ```prisma
   enum ImportStatus {
     UPLOADED          // file received + rows parsed; awaiting matching
     MATCHING          // matcher in progress
     AWAITING_REVIEW   // matching complete; user must accept/reject rows
     COMMITTED         // user committed; rows written to LawList/LawListItem
     FAILED            // parser or matcher hit an unrecoverable error
   }

   enum RowMatchStatus {
     PENDING                  // not yet matched
     MATCHED_HIGH             // confidence ≥ 0.85 — auto-acceptable
     MATCHED_MEDIUM           // confidence 0.5–0.85 — user must confirm
     UNMATCHED                // confidence < 0.5 — no candidate met threshold
     ACCEPTED_BY_USER         // user accepted matched_doc_id
     REPLACED_BY_USER         // user picked a different candidate from list
     REJECTED_BY_USER         // user rejected the row entirely
     CATALOG_REQUEST_PENDING  // user clicked "Begär tillägg" — queued for ops
     CATALOG_REQUEST_FULFILLED // ops ingested; row auto-rematched
   }

   enum CatalogRequestStatus {
     PENDING
     FULFILLED
     REJECTED   // ops marked as duplicate / out of scope
   }
   ```

3. **`LawListImport` model** — workspace-scoped aggregate, one row per uploaded file:
   ```prisma
   model LawListImport {
     id                  String       @id @default(cuid())
     workspace_id        String
     created_by_user_id  String
     filename            String       // e.g., "notisum-export-2026.xlsx"
     source_type         String       // 'xlsx' | 'csv' | 'paste'
     status              ImportStatus @default(UPLOADED)
     row_count           Int          @default(0)
     column_mapping      Json         // {titel: 'col_2', sfs_nummer: 'col_1', ...}
     committed_law_list_id String?    // nullable until status=COMMITTED
     error_message       String?      // populated when status=FAILED
     created_at          DateTime     @default(now())
     updated_at          DateTime     @updatedAt
     committed_at        DateTime?

     workspace           Workspace    @relation(fields: [workspace_id], references: [id], onDelete: Cascade)
     created_by          User         @relation("ImportCreatedBy", fields: [created_by_user_id], references: [id], onDelete: Restrict)
     committed_law_list  LawList?     @relation(fields: [committed_law_list_id], references: [id], onDelete: SetNull)
     rows                LawListImportRow[]

     @@index([workspace_id, status])
     @@index([created_by_user_id])
   }
   ```

4. **`LawListImportRow` model** — per-row staging:
   ```prisma
   model LawListImportRow {
     id                  String         @id @default(cuid())
     import_id           String
     row_index           Int            // 0-based row order from source file
     source_titel        String?
     source_sfs_nummer   String?
     source_omrade       String?
     source_lagansvarig  String?
     source_kommentar    String?
     source_raw          Json           // full raw row in case mapping wrong
     match_status        RowMatchStatus @default(PENDING)
     matched_document_id String?
     confidence_score    Float?         // 0.0–1.0
     match_candidates    Json?          // [{document_id, score, title}, ...]
     match_reasoning     String?        // LLM rationale; null for fuzzy-only matches
     user_decided_at     DateTime?
     created_at          DateTime       @default(now())
     updated_at          DateTime       @updatedAt

     import              LawListImport  @relation(fields: [import_id], references: [id], onDelete: Cascade)
     matched_document    LegalDocument? @relation(fields: [matched_document_id], references: [id], onDelete: SetNull)
     catalog_request     CatalogIngestRequest?

     @@index([import_id, match_status])
     @@index([matched_document_id])
   }
   ```

5. **`CatalogIngestRequest` model** — ops queue for unmatched rows:
   ```prisma
   model CatalogIngestRequest {
     id                  String               @id @default(cuid())
     workspace_id        String
     import_row_id       String               @unique
     requested_by_user_id String
     handler_user_id     String?              // ops user who claimed it
     fulfilled_with_document_id String?       // LegalDocument.id once ingested
     admin_note          String?
     status              CatalogRequestStatus @default(PENDING)
     created_at          DateTime             @default(now())
     fulfilled_at        DateTime?
     rejected_at         DateTime?

     workspace           Workspace            @relation(fields: [workspace_id], references: [id], onDelete: Cascade)
     import_row          LawListImportRow     @relation(fields: [import_row_id], references: [id], onDelete: Cascade)
     requested_by        User                 @relation("CatalogRequestedBy", fields: [requested_by_user_id], references: [id], onDelete: Restrict)
     handler             User?                @relation("CatalogRequestHandler", fields: [handler_user_id], references: [id], onDelete: SetNull)
     fulfilled_with_document LegalDocument?   @relation(fields: [fulfilled_with_document_id], references: [id], onDelete: SetNull)

     @@index([workspace_id, status, created_at])
     @@index([handler_user_id])
   }
   ```

6. **Back-references on existing models:**
   - `Workspace`: add `law_list_imports LawListImport[]` and `catalog_ingest_requests CatalogIngestRequest[]`
   - `User`: add `imports_created LawListImport[] @relation("ImportCreatedBy")`, `catalog_requests_made CatalogIngestRequest[] @relation("CatalogRequestedBy")`, `catalog_requests_handled CatalogIngestRequest[] @relation("CatalogRequestHandler")`
   - `LegalDocument`: add `import_rows_matched LawListImportRow[]` and `catalog_requests_fulfilled CatalogIngestRequest[]`
   - `LawList`: add `import_committed_from LawListImport?` (back-reference; nullable since most law lists are not import-derived)

7. **RLS policies (Supabase) mirror the existing `LawList` pattern.** Workspace-scoped read/write for members; admin override for `CatalogIngestRequest` (ops needs cross-workspace access). **Canonical RLS reference: `prisma/migrations/20260430120000_enable_rls_public_tables/migration.sql`** — RLS policies in this project live as inline SQL appended to Prisma migrations (not separate Supabase migrations). Add the three policy blocks (`LawListImport`, `LawListImportRow`, `CatalogIngestRequest`) at the bottom of the same `add_law_list_import` migration file using the `ALTER TABLE ... ENABLE ROW LEVEL SECURITY;` + `CREATE POLICY ... USING (...)` pattern from that reference.

### Zod schemas

8. **New file `lib/validation/law-list-import.ts`** — exports five schemas:
   - `CreateImportInput` — `{ filename: string, source_type: 'xlsx'|'csv'|'paste' }`
   - `ParseImportInput` — `{ importId: string, fileBuffer: string }` (base64-encoded)
   - `RowDecisionInput` — discriminated union: `{type: 'accept', rowId} | {type: 'replace', rowId, candidateDocId} | {type: 'reject', rowId} | {type: 'request', rowId, note?}`
   - `CommitImportInput` — `{ importId: string, listName: string }`
   - `FulfillCatalogRequestInput` — `{ requestId: string, fulfilledWithDocumentId: string, adminNote?: string }`
   - All schemas use Zod with strict types matching the Prisma models. Example errors are Swedish-language to match existing convention.

**Source artefacts (read these before implementing):**

**Existing patterns to follow:**
- **Schema additive pattern** — see `prisma/schema.prisma` lines around `ComplianceFinding` (Story 21.7) and `OnboardingEvent` (Epic 25 B.0) — both added new tables without modifying existing columns. Follow that shape.
- **Server action with `withWorkspace`** — see `app/actions/compliance-audit-cycle.ts` for the `withWorkspace(cb, 'tasks:edit')` wrapper pattern + workspace-scoped query convention.
- **RLS policy** — see existing `LawList` and `LawListItem` policies as the template. Workspace-membership-based read; same workspace required for write.
- **Zod validation** — see `lib/validation/workspace.ts` and `lib/validation/law-list.ts` for naming + structure conventions.
- **Activity-log integration** — see `lib/activity/action-constants.ts` for entity-type-key + action-key conventions; `lib/activity/format-activity.ts` for Swedish label maps.

**Key decisions baked in:**
- `column_mapping` is `Json` (not normalised columns) because the parser may detect arbitrary headers; flexibility > strictness.
- `match_candidates` is `Json` (not a separate join table) because it's read-mostly per-row; persisting top-K candidates inline beats N+1 lookups.
- `RowMatchStatus` is intentionally rich (9 states, not 3) so subsequent stories don't need to extend the enum mid-flight.
- `CatalogIngestRequest.import_row_id` is `@unique` because we explicitly do NOT want multiple requests for the same source row — if a row's request is rejected, the user can update the row state to `REJECTED_BY_USER` instead.
