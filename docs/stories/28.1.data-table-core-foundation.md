## Spike outcome: expansion under virtualization — PROVEN

Flattened render list (detail rows as own virtual items with `measureElement`) works with semantic `<tr>` elements. Verified in Chrome against 1,000 rows: expand at top/middle/bottom of the 52,000px scroll range → variable detail heights measured (353/517/681/845px vs 200px estimate), **zero row overlaps**, scroll anchoring correct (expanding overscan rows above the viewport compensates scrollTop), no render loops. **No div-row fallback needed.**

Three traps discovered — all load-bearing for every future consumer:

1. **`estimateSize`/`getItemKey` MUST be identity-stable.** virtual-core keeps `options.getItemKey` in a memo whose `onChange` calls `notify()` (a setState) synchronously during render via `getVirtualItems()`. A fresh closure per render — inevitable when consumers pass inline prop literals — loops into React's "Too many re-renders". Fixed in the core by reading `renderItems` through a ref from `useCallback([], …)` closures; also stabilized upstream memo deps on feature *presence* (`Boolean(selection)`) instead of adapter identity.
2. **Virtualized rows must escape the table-row-group layout context** (the krav-table recipe): tbody `display: block`, rows `display: table; tableLayout: fixed` + explicit width on every cell — otherwise the table engine couples row heights and measurement never converges.
3. **Percentage widths are unusable on absolutely-positioned virtual rows** (the display:block tbody shrink-wraps, so `width: 100%` collapsed to 56px). Rows carry `rowWidth = liveTotalWidth` in pixels. Virtual detail rows must NOT use `colSpan` (spanning nonexistent columns in an isolated single-cell table collapses the cell) — a single cell with explicit width is the whole row.
