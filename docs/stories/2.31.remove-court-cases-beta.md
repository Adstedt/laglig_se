# Story 2.31: Remove Court Cases from Beta

## Status

Approved

## Story

**As a** product owner,
**I want** all court case functionality removed from the application for beta launch,
**so that** we ship a focused beta around SFS laws and agency regulations without half-baked court case content that could undermine product quality.

## Context & Dependencies

**Why this story exists:** Court cases (Domstolsverket API integration) were built early in Epic 2 (Story 2.3) as part of the "170K+ SEO pages" strategy. However, the sync cron job is already disabled, court case content quality hasn't been maintained, and for beta the core value proposition centers on SFS laws + agency regulations (föreskrifter). Rather than shipping incomplete court case pages, we remove them entirely and revisit post-beta when we can dedicate proper attention to content quality, citation linking, and the Domstolsverket API integration.

**Approach — "Remove UI + data, keep schema dormant":**
- Remove all user-facing court case routes, components, and integration points
- Remove the cron job route and dedicated utility code
- **Delete all court case data** from the database via a one-time cleanup script (safe cascade order)
- **Keep the Prisma schema intact** (no migration) — the `CourtCase` model and `COURT_CASE_*` enum values stay in the schema so we don't need a destructive migration and can re-enable later
- Remove dedicated scripts from `scripts/` directory

**Dependencies:** None — court cases are already dormant (sync cron disabled). This is a removal/cleanup story.

**What this unblocks:** Cleaner beta codebase, faster builds (fewer static pages generated), reduced confusion for beta users who might find empty/stale court case pages.

## Acceptance Criteria

1. No `/rattsfall` or `/rattsfall/**` routes are accessible (public or workspace) — requests to these paths return 404
2. Court case content types (`COURT_CASE_*`) do not appear in search results, catalogue browse, or any filter/dropdown UI
3. The "Rättsfall" section is removed from navigation, catalogue filters, and content type groupings
4. The `sync-court-cases` cron job route is deleted and removed from `JOB_REGISTRY`
5. Court case reference detection patterns (NJA, HFD ref, AD nr, MÖD, MIG) are removed from `detect-references.ts`
6. The "Citing Court Cases" component on law detail pages is removed, along with the court-case-aware `RelatedDocsPrefetcher`
7. Court case cached query functions and browse prewarming are removed
8. Court case email notification functions are removed from `cron-notifications.ts`
9. The Prisma schema (`CourtCase` model, `COURT_CASE_*` enum values) is **preserved** — no destructive migration
10. All court case data is **deleted** from the database — `LegalDocument` records with `COURT_CASE_*` content types and all cascading child records (CourtCase, CrossReference, ChangeEvent, etc.) are removed. **Only** court case documents are deleted; SFS, EU, and agency documents are untouched
11. The application builds, type-checks, and all tests pass after removal
12. The `domstolsverket.ts` external API client is deleted
13. All court-case-specific scripts in `scripts/` are deleted
14. The sitemap (`app/sitemap.ts`) no longer generates `/rattsfall/**` URLs
15. All test files referencing court case functionality are updated or deleted so the test suite passes cleanly
16. The data cleanup script uses a safe deletion order, runs inside a transaction, and logs counts before and after — providing a dry-run mode for verification before execution
17. Zero `LegalDocument` records with `COURT_CASE_*` content types remain after cleanup. Zero orphaned `ContentChunk` records for deleted documents remain

## Dev Notes

### Approach: Remove UI + Data, Keep Schema Dormant

Two principles guide this story:

1. **No Prisma schema migration.** The `CourtCase` model and 6 `COURT_CASE_*` enum values stay in `prisma/schema.prisma`. This avoids PostgreSQL enum removal complexity and keeps the door open for re-enabling court cases later via re-ingestion.

2. **Delete all court case data from the database.** The ~15,000 court case `LegalDocument` records and all child records are removed via a one-time cleanup script. This eliminates stale data, dead cross-reference links in SFS law pages, and orphaned content chunks. If court cases are re-enabled post-beta, the ingestion pipeline (preserved in git history) can re-populate the data.

Re-enabling later requires: restore routes/components from git, re-run Domstolsverket ingestion (~12 hours).

Story 2.3 (Ingest Court Cases) established:
- All court cases stored as `LegalDocument` with `content_type` set to one of the 6 `COURT_CASE_*` variants
- 1:1 relationship: `LegalDocument` → `CourtCase` (metadata table)

### Technical Constraints

- **No Prisma migration**: Do NOT run `prisma migrate dev` or create any migration. The schema file keeps the CourtCase model and enum values. Only application code and data are removed.
- **Build impact**: Removing the `/rattsfall/[court]/[id]` static generation (was pre-generating ~210 pages) will speed up builds.
- **Seed data**: `prisma/seed-multi-content.ts` creates a test `COURT_CASE_AD` document. This is acceptable to keep — it references valid schema types and only runs in dev/test environments.
- **`createNewRulingEvent()`**: This function in `lib/sync/change-detection.ts` creates `ChangeEvent` records with `ChangeType.NEW_RULING`. Since no court case sync is active, this is dead code. Remove the function. If `NEW_RULING` is referenced elsewhere, check before removing those references — but keep the enum value in the Prisma schema.

### Data Cleanup — Cascade Safety Reference

The cleanup script must delete in a specific order because two relations use `onDelete: Restrict` (will block parent deletion) and one has no FK (will orphan).

**Must delete EXPLICITLY before `LegalDocument` (no cascade):**

| Table | Reason | Expected count |
|---|---|---|
| `ContentChunk` | No FK to `LegalDocument` — uses `source_type` + `source_id` strings. Will orphan if not deleted first. | Varies |
| `LawListItem` | `onDelete: Restrict` — FK constraint will block `LegalDocument` deletion. | ~0 (court cases unlikely in user law lists) |
| `TemplateItem` | `onDelete: Restrict` — FK constraint will block `LegalDocument` deletion. | ~0 (court cases not in templates) |

**Auto-cascade from `LegalDocument` deletion (all `onDelete: Cascade`):**

| Table | Relation |
|---|---|
| `CourtCase` | 1:1 via `document_id` |
| `CrossReference` | Both `source_references` and `target_references` |
| `Amendment` | Both `base_amendments` and `amending_amendments` |
| `DocumentSubject` | via `document_id` |
| `DocumentVersion` | via `document_id` |
| `ChangeEvent` | via `document_id` |
| `LawSection` | via `document_id` |
| `LegislativeRef` | via `document_id` |
| `DocumentVisit` | via `document_id` |

**CRITICAL SAFETY**: The cleanup script WHERE clause must **only** match `content_type IN ('COURT_CASE_AD', 'COURT_CASE_HD', 'COURT_CASE_HOVR', 'COURT_CASE_HFD', 'COURT_CASE_MOD', 'COURT_CASE_MIG')`. Never use a wildcard or LIKE pattern. The script must assert the count of matched document IDs and log per-type counts before any deletion proceeds.
