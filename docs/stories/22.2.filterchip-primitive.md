# Story 22.2: `FilterChip` primitive + Tabs/FilterChip semantic split

## Status

Draft

## Story

**As a** developer working on workspace UI,
**I want** a single `FilterChip` primitive (with `pressed`, `count`, `icon` props and `aria-pressed` semantics) and a discipline that shadcn `Tabs` is reserved for view-switching only,
**so that** in-view filter controls share one rendering and one a11y contract, screen readers stop mis-announcing toggle buttons as tabs, and any future surface gets the chip primitive for free.

## Acceptance Criteria

### Functional

1. **`FilterChip` primitive created.** New file `components/ui/filter-chip.tsx` exports `<FilterChip>` rendering as `<button type="button" aria-pressed={pressed}>` (NOT `role="tab"`). Props:
   - `pressed: boolean` (controlled toggle state)
   - `onPressedChange?: (pressed: boolean) => void`
   - `count?: number` (renders inline count badge when provided)
   - `icon?: ReactNode` (optional leading icon slot)
   - `disabled?: boolean`
   - `children: ReactNode` (chip label — Swedish copy)
   - Standard `className` for caller overrides; merged via `cn()`.
   - Class strings pulled from `_prototypes/ui-alignment-prototype.html` §3 proposed FilterChip:
     - Base: `inline-flex items-center gap-1.5 rounded-full border px-3 py-1 text-xs font-medium transition-colors`
     - Pressed: `border-foreground bg-foreground text-background`
     - Unpressed: `border-border text-muted-foreground hover:border-foreground/40 hover:text-foreground`
     - Disabled: `opacity-50 cursor-not-allowed`
   - Count badge (when `count` is provided): inline `<span>` with `rounded-full px-1.5 py-0 text-[10px] font-semibold tabular-nums`. Pressed: `bg-background/20 text-background`. Unpressed: `bg-muted text-muted-foreground`.

2. **`FilterChipGroup` wrapper exported from same file.** `<FilterChipGroup label aria-label className>` renders `<div role="group" aria-label={label} className="flex flex-wrap gap-2">{children}</div>`. The `aria-label` is REQUIRED — chips inside the group describe what's being filtered (e.g., `aria-label="Filtrera kontroller efter status"`). No `role="tablist"` (this is the whole point of the split).

3. **Migrate Kontroller list filter chips.** In `components/features/compliance-audit/cycle-list/cycle-list-table.tsx` (line 81 area, `FILTERS` array around line 44), replace the inline chip render with `<FilterChipGroup label="Filtrera kontroller efter status">{...}</FilterChipGroup>` containing one `<FilterChip pressed count>` per status (`Aktiva`, `Slutförda`, `Förseglade`, `Arkiverade`, `Alla`). Counts continue to come from the existing FILTERS data. **Remove all `role="tab"` and `role="tablist"` markup from this file.**

4. **Migrate Findings tab filter chips.** In `components/features/compliance-audit/cycle-detail/cycle-findings-tab.tsx` (line 63 area), the file has TWO chip rows today:
   - **Row 1 (type filter):** `Alla` / `Avvikelse` / `Observation` / `Förbättringsförslag` → migrate to `<FilterChipGroup label="Filtrera findings efter typ">`. Counts ARE optionally added (currently absent on Findings — they're useful, derive from the filtered list state). If counts are non-trivial to compute in time for this story, ship without `count` prop and add a follow-up task; do NOT block on count derivation.
   - **Row 2 (state filter):** `Alla` / `Öppna` / `Stängda` → migrate to `<FilterChipGroup label="Filtrera findings efter status">`. Same count-optional rule.
   - **Remove all `role="tab"` markup from these two filter rows.** The `<Tabs>` for view-switching at the cycle-detail level (Items / Findings / Rapport / Aktivitet) is untouched — that stays as shadcn `Tabs`.

5. **Do NOT touch existing shadcn `<Tabs>` usages.** The view-switching tabs on `/laglistor`, `/tasks`, `/workspace/styrdokument`, and the cycle-detail page (Items / Findings / Rapport / Aktivitet) remain `TabsTrigger` from `components/ui/tabs.tsx`. This story explicitly preserves the semantic split: `Tabs` for views, `FilterChip` for filters.

6. **Search-result-zero check.** After migration, `grep -rE 'role="tab"' components/features/compliance-audit/` returns ONLY hits inside actual `<Tabs>` blocks (i.e., the cycle-detail `<TabsList>` for Items/Findings/Rapport/Aktivitet). No leftover `role="tab"` from filter-chip code.

### Test coverage

7. **`components/ui/__tests__/filter-chip.test.tsx` (NEW).** Cases:
   - Renders pressed and unpressed states with the expected class strings (asserted via `toHaveClass` rather than full snapshot to avoid noisy diffs on token tweaks).
   - `aria-pressed` attribute reflects `pressed` prop.
   - `onPressedChange(!pressed)` is invoked on click.
   - `count` prop renders the count badge with `tabular-nums`; absence of `count` renders no badge.
   - `disabled={true}` disables the button (`pointer-events-none` or native `disabled` attribute) and prevents `onPressedChange` from firing.
   - `icon` slot renders before the label text.
   - `<FilterChipGroup>` renders `role="group"` with the provided `aria-label`. **Asserts `role` is NOT `tablist`** (regression guard against future drift).

8. **`cycle-list-table.test.tsx` (EXTEND existing if present, NEW otherwise).** Cases (use Glob first to find existing test file):
   - Filter chip group renders 5 chips (`Aktiva` / `Slutförda` / `Förseglade` / `Arkiverade` / `Alla`) with correct counts.
   - Clicking a chip calls the existing filter handler with the matching status value.
   - `aria-pressed` reflects the active filter.
   - No `role="tab"` markup anywhere in rendered output.

9. **`cycle-findings-tab.test.tsx` (EXTEND existing if present, NEW otherwise).** Cases:
   - Two `FilterChipGroup`s render (type filter + state filter).
   - Selecting a type chip narrows the rendered finding list.
   - Selecting a state chip narrows the rendered finding list.
   - Type and state filters compose (both active narrows further).
   - No `role="tab"` markup in the filter row output (regression guard).

## Dev Notes

### Previous Story Insights

- **Story 22.1 (Badge tones + Priority alignment)** establishes the `<Badge tone variant>` pattern and the `lib/ui/` token-module convention. Story 22.2 follows the same pattern for FilterChip (primitive in `components/ui/`, no token module needed since the primitive only has two states — pressed / unpressed — and no semantic colour ramp).
- 22.1 also established the discipline that **dev should verify existing test file existence via Glob** before writing new tests, and prefer assertion-based tests over snapshot tests where possible. Same approach here.
- 22.1 sets the precedent that **acknowledged visual deltas** (intentional pill restyling) should be flagged in the PR description. 22.2 has fewer deltas (filter chips were already inconsistent across cycle-list and findings) — the alignment IS the change, no separate flag needed.

### Component Specifications

- **shadcn/ui is mandatory for all UI primitives** [Source: architecture/17-coding-standards.md#17.3]. `FilterChip` is a new primitive that follows the shadcn convention: `cva` for variants, `cn()` for class merging, `React.forwardRef` for ref forwarding, accessible-by-default via `<button>` + `aria-pressed` (Radix not required since this is a pure toggle button, no menu / dialog / popover semantics).
- **Custom domain components must compose shadcn primitives** [Source: architecture/17-coding-standards.md#17.3]. `cycle-list-table.tsx` and `cycle-findings-tab.tsx` are domain components that compose `<FilterChip>` rather than rendering their own chip markup.
- **Existing `cn` className helper** at `lib/utils/cn.ts` [Source: architecture/12-unified-project-structure.md#12.2] is used to merge variant + caller classes.
- **`Tabs` primitive** at `components/ui/tabs.tsx` (shadcn) is the canonical view-switcher and remains the only `role="tab"` carrier in workspace UI after this story.

### Tech Stack & Versions

- **TypeScript 5.5+** with strict mode [Source: architecture/17-coding-standards.md#17.2; architecture/3-tech-stack.md#3.1]. `FilterChip` props use explicit interface; `pressed` is a required boolean (no implicit `any`).
- **React 19** [Source: architecture/3-tech-stack.md#3.1]. `FilterChip` is a controlled component — no internal state, parent owns `pressed`. No `'use client'` needed unless the parent component already requires it (`cycle-list-table.tsx` and `cycle-findings-tab.tsx` are already client components per their existing handlers).
- **Tailwind CSS 3.4+** with shadcn/ui [Source: architecture/3-tech-stack.md#3.1]. Class strings sourced from prototype §3.
- **`cva` (class-variance-authority)** for variant config — convention shared with all `components/ui/*` primitives.

### Technical Constraints

- **Zero behaviour change.** Filter handlers, count derivation, list narrowing logic — all unchanged. Only the rendered chip markup differs.
- **Out of scope for 22.2 (handled by 22.3):** consolidation of toolbar layout (where the filter chips sit relative to search and primary action). This story migrates the chip primitive only; toolbar slot positioning is owned by Story 22.3.
- **Out of scope for 22.2 (handled by 22.1):** finding-row pills (severity, type, state badges INSIDE finding rows). Those are owned by Story 22.1's `<Badge>` migration. 22.2 touches only the FILTER chips at the top of the Findings tab.
- **Coordination with active Epic 21:** if Epic 21 stories land new filter rows on `cycle-items-tab.tsx` or elsewhere while 22.2 is in flight, target the new `<FilterChip>` API directly. Flag in PR review.
- **No new dependencies.** All work uses existing shadcn / Tailwind / cva infrastructure.
- **TypeScript strict mode:** `pressed` is `boolean`, not `boolean | undefined`. Forces parent to explicitly track filter state.

### Testing

- **Test file location:** Co-locate. `components/ui/__tests__/filter-chip.test.tsx` for the primitive; existing/new tests next to the migrated surfaces. [Source: architecture/12-unified-project-structure.md#12.1]
- **Test framework:** Vitest 1.4+ + React Testing Library 14.2+. [Source: architecture/3-tech-stack.md#3.1]
- **Patterns to follow:** assertion-based tests (`toHaveClass`, `toHaveAttribute('aria-pressed', 'true')`), user-interaction tests (`fireEvent.click`, `userEvent.click`).
- **Specific requirements:**
  - `aria-pressed` toggle assertion is the most important regression guard.
  - `role="tab"` ABSENCE assertion in filter-chip output is the second most important guard (the whole point of the split).
  - Helper: assert count badge appears iff `count` prop is provided.
  - No e2e additions; manual a11y pass acceptable for IV4.
