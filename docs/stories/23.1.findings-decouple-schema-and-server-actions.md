# Story 23.1: Findings — decouple from cycle (schema + server-action layer)

## Status

Draft

## Story

**As a** team member or KMA-samordnare working a Laglig workspace,
**I want** the underlying data model and server actions to support findings that exist outside an audit cycle (anchored to a `LawListItem` or `Requirement`), with workspace scoping, assignment, and threaded comments,
**so that** subsequent Epic 23 stories can ship a `/anmärkningar` registry, a shared `<FindingModal>`, and ad-hoc creation entry points without requiring further schema or server-action changes.

This is the **schema + server-action layer only** — no UI changes. Existing cycle-scoped flows (Story 21.7's `<FindingEditor>`, `<CycleFindingsTab>`, `<CycleItemRowDrawer>`) continue to work unchanged. The story proves the data model can carry both cycle-scoped and ad-hoc findings, and exposes the new read/mutation shapes Story 23.2 will consume.

## Acceptance Criteria

### Schema

1. **Prisma migration `<TIMESTAMP>_decouple_findings_from_cycle/migration.sql` (NEW).** Generate the timestamp via `pnpm prisma migrate dev --create-only --name decouple_findings_from_cycle` so it matches the canonical `YYYYMMDDHHMMSS_` pattern of recent migrations (e.g., `20260505105405_add_workspace_usage`). Do NOT hand-pick a truncated date. Five DDL operations applied as a single migration:
   1. `ALTER TABLE "compliance_findings" ALTER COLUMN "cycle_id" DROP NOT NULL;` — `cycle_id` becomes nullable.
   2. `ALTER TABLE "compliance_findings" ADD COLUMN "workspace_id" TEXT NOT NULL DEFAULT '';` followed by a backfill `UPDATE` (see AC 2) and finally `ALTER TABLE "compliance_findings" ALTER COLUMN "workspace_id" DROP DEFAULT;`. The temporary default lets the `NOT NULL` add succeed before backfill; dropping it post-backfill prevents future inserts from sliding through with empty strings.
   3. `ALTER TABLE "compliance_findings" ADD COLUMN "created_by_user_id" TEXT NOT NULL DEFAULT '';` + backfill (AC 2) + `DROP DEFAULT`.
   4. `ALTER TABLE "compliance_findings" ADD COLUMN "assignee_id" TEXT;` (nullable, no backfill needed).
   5. `ALTER TABLE "compliance_findings" ADD CONSTRAINT "compliance_findings_anchor_check" CHECK ("cycle_id" IS NOT NULL OR "law_list_item_id" IS NOT NULL OR "requirement_id" IS NOT NULL);` — every finding must have at least one anchor (cycle, law list item, or requirement). Existing rows always have `cycle_id` so this passes immediately for legacy data.
   6. Foreign-key constraints for the new columns:
      ```sql
      ALTER TABLE "compliance_findings" ADD CONSTRAINT "compliance_findings_workspace_id_fkey"
        FOREIGN KEY ("workspace_id") REFERENCES "workspaces"("id") ON DELETE CASCADE;
      ALTER TABLE "compliance_findings" ADD CONSTRAINT "compliance_findings_created_by_user_id_fkey"
        FOREIGN KEY ("created_by_user_id") REFERENCES "users"("id") ON DELETE RESTRICT;
      ALTER TABLE "compliance_findings" ADD CONSTRAINT "compliance_findings_assignee_id_fkey"
        FOREIGN KEY ("assignee_id") REFERENCES "users"("id") ON DELETE SET NULL;
      ```
   7. Indexes:
      ```sql
      CREATE INDEX "compliance_findings_workspace_id_closed_at_idx"
        ON "compliance_findings"("workspace_id", "closed_at");
      CREATE INDEX "compliance_findings_assignee_id_idx"
        ON "compliance_findings"("assignee_id");
      CREATE INDEX "compliance_findings_created_by_user_id_idx"
        ON "compliance_findings"("created_by_user_id");
      ```
      The composite `(workspace_id, closed_at)` index targets the registry's primary access pattern (Story 23.3): "open findings in this workspace, sorted by recency". Existing indexes (`cycle_id, type` / `law_list_item_id` / `requirement_id` / `corrective_action_task_id`) remain untouched.
   `[Source: prisma/schema.prisma#ComplianceFinding (current model, ~lines 2025–2063); prisma/migrations/20260427120000_add_finding_closure_metadata/migration.sql (additive-only precedent shape); architecture additive-schema principle]`

2. **Backfill (inline SQL within the migration file, between the column-add and DROP DEFAULT steps).**
   ```sql
   UPDATE "compliance_findings" f
   SET "workspace_id"       = c.workspace_id,
       "created_by_user_id" = c.created_by_user_id
   FROM "compliance_audit_cycles" c
   WHERE f.cycle_id = c.id;
   ```
   Every existing finding has a non-null `cycle_id`, and every cycle has a non-null `workspace_id` and `created_by_user_id` (verified at `prisma/schema.prisma#ComplianceAuditCycle`, line ~1979), so the join is total. **Rationale for `created_by_user_id` ← `cycle.created_by_user_id`:** in the pre-Epic-23 world, only the cycle creator (lead auditor) could file findings via `<FindingEditor>` against their own cycle — attributing the finding to that user is the historically accurate "filer of record". Documented inline as a `-- ` comment in the migration. *Verification:* the post-backfill counts must match (`SELECT COUNT(*) FROM compliance_findings WHERE workspace_id = '' OR created_by_user_id = '';` → 0). Add this as a `-- DO $$ BEGIN ASSERT ... END $$;` block at the migration tail to guard against partial-data edge cases. `[Source: prisma/schema.prisma#ComplianceAuditCycle (created_by_user_id NOT NULL); architecture deterministic-backfill principle]`

3. **`Comment` polymorphic extension (same migration file).** Add `finding_id` to the `comments` table:
   ```sql
   ALTER TABLE "comments" ADD COLUMN "finding_id" TEXT;
   ALTER TABLE "comments" ADD CONSTRAINT "comments_finding_id_fkey"
     FOREIGN KEY ("finding_id") REFERENCES "compliance_findings"("id") ON DELETE CASCADE;
   CREATE INDEX "comments_finding_id_idx" ON "comments"("finding_id");
   ```
   The existing one-of-three constraint at the application layer (`task_id | law_list_item_id`) extends to `task_id | law_list_item_id | finding_id`. **Decision: enforce at the application layer only**, NOT via DB CHECK constraint. Rationale: the existing two-parent polymorphism is enforced inline in the comment-creating server actions (`app/actions/task-modal.ts` around the `prisma.comment.create` call ~line 993; `app/actions/legal-document-modal.ts#createListItemComment` ~line 1107) — there is no central `comments.ts` action file, each parent owns its comment writes. Adding a CHECK now would break parity with `task_id` / `law_list_item_id` enforcement and is a follow-up of its own. Document the application-level invariant in the Zod schema for `addFindingComment` (AC 11). `[Source: prisma/schema.prisma#Comment (model definition); app/actions/task-modal.ts#~993 (task-comment write); app/actions/legal-document-modal.ts#createListItemComment (law-list-item comment write)]`

4. **`prisma/schema.prisma` updates (sync the model with the migration):**
   - In `model ComplianceFinding`:
     - `cycle_id String` → `cycle_id String?`
     - After `requirement_id String?`: add `workspace_id String`, `created_by_user_id String`, `assignee_id String?`
     - Existing `cycle ComplianceAuditCycle @relation(...)` → `cycle ComplianceAuditCycle? @relation(fields: [cycle_id], references: [id], onDelete: Cascade)` (nullable relation)
     - After the existing `closed_by` relation, add three new named relations + the comments back-reference:
       ```prisma
       workspace   Workspace          @relation(fields: [workspace_id], references: [id], onDelete: Cascade)
       created_by  User               @relation("FindingCreatedBy", fields: [created_by_user_id], references: [id], onDelete: Restrict)
       assignee    User?              @relation("FindingAssignee", fields: [assignee_id], references: [id], onDelete: SetNull)
       comments    Comment[]
       ```
       **Naming convention:** `User`-targeting relations on `ComplianceFinding` are NAMED (matches existing `FindingClosedBy` precedent) — needed because there are now multiple relations from `User` to `ComplianceFinding` (created_by, assignee, closed_by). The `Comment[]` back-reference is UNNAMED — Prisma disambiguates by FK column on the `Comment` side.
     - Add `@@index([workspace_id, closed_at])`, `@@index([assignee_id])`, `@@index([created_by_user_id])` to the model's index list.
   - In the `User` model: add `findings_created  ComplianceFinding[] @relation("FindingCreatedBy")` and `findings_assigned ComplianceFinding[] @relation("FindingAssignee")` to the existing relation list. The existing `findings_closed @relation("FindingClosedBy")` (Story 21.7) stays untouched.
   - In the `Workspace` model: add `compliance_findings ComplianceFinding[]` to the existing relation list.
   - In the `Comment` model: add `finding_id String?` (alongside the existing `task_id` and `law_list_item_id` polymorphic columns) and an UNNAMED relation: `finding ComplianceFinding? @relation(fields: [finding_id], references: [id], onDelete: Cascade)`. **Unnamed** matches the existing `task` and `law_list_item` relation pattern on `Comment` — Prisma disambiguates the multiple polymorphic parents by FK column name. Also add `@@index([finding_id])`.
   `[Source: prisma/schema.prisma#ComplianceFinding, #Comment, #User, #Workspace; existing FindingClosedBy named-relation precedent; existing unnamed Comment relation pattern]`

   **Type-level updates (catches TypeScript regressions):**
   - In `app/actions/compliance-finding.ts`, the local `LoadedFinding` type currently declares `cycle_id: string` (non-null) and `cycle: { id, status, law_list_id, name, lead_auditor_user_id }` (non-null). After this AC's schema change, both must become nullable: `cycle_id: string | null`, `cycle: { ... } | null`. The mapper at `mapRowToFindingRow` must handle the null branch when projecting `FindingRow.cycleId` and the new `FindingRow.cycle` field (see AC 6).
   - The existing `findFirst({ where: { id, cycle: { workspace_id } } })` query in `loadFindingScopedToWorkspace` (current implementation, see AC 6/Task 4) excludes ad-hoc findings by construction — that's part of why AC 6's helper rewrite is required.

   1. Zod parse (cycleId now optional).
   2. **If `cycleId` provided:** existing Story 21.7 logic runs verbatim (load cycle scoped to workspace, cross-link tenant validation against cycle's laglista). Workspace_id is set to the cycle's workspace_id. **Note:** Story 21.7 originally had an `assertCycleEditableUi` cycle-status gate; Story 21.27 deleted that gate and the helper itself — findings are unconditionally editable regardless of cycle state. ### Schema-design rationale

**Why `workspace_id` is denormalised on `ComplianceFinding`:** the registry's primary access pattern is "all findings in this workspace" — joining through `cycle.workspace_id` requires a non-null cycle, which ad-hoc findings won't have. Denormalising costs one column and one trivial backfill; the alternative (dual-path query that union-joins through cycle for cycle-scoped + reads ad-hoc directly) is fragile and slower. The composite index `(workspace_id, closed_at)` makes the registry's "open findings" query an index-only scan.

**Why `created_by_user_id` is required, not nullable:** every finding has a creator. The pre-Epic-23 world implicitly attributed via `cycle.created_by_user_id` (lead auditor created the cycle and the findings within). Making the column required with a deterministic backfill captures the historical truth without losing data, and prevents future "who filed this?" gaps.

**Why `assignee_id` is nullable:** ad-hoc findings filed by a non-KMA-samordnare may not know who should own follow-up. Allowing null lets the registry surface "Otilldelade" as a filter chip; assignment happens later via `assignFinding`. Cycle-scoped findings often have a clear owner (the lead auditor or item-responsible user) — but enforcing assignment on create would block low-friction observation-filing, which is the value of ad-hoc.

**Why CHECK constraint, not just application validation:** defence-in-depth. The application-layer check in `createFinding` (AC 5) is the user-facing one (Swedish error string). The DB CHECK is a backstop against (a) future server actions added without re-implementing the check, (b) migrations or scripts that modify rows directly, (c) data-fix queries run in production. Both layers fire on the same condition; only the app layer renders a friendly error.

### Migration safety

**Backfill on a 50-row table is trivial; on a 50,000-row table it's still fine** — Postgres handles batched UPDATE with the WHERE-EXISTS join in milliseconds. No need for chunked migration.

**The `DEFAULT '' + DROP DEFAULT` pattern is the canonical Postgres way** to add a NOT NULL column to a populated table. The empty-string default is a tombstone that the backfill UPDATE replaces; the post-DROP DEFAULT prevents future inserts from accidentally landing the empty string. The post-backfill assert (`workspace_id = '' OR created_by_user_id = ''` count = 0) catches any straggler rows.

**The CHECK constraint is added LAST in the migration sequence** — after all backfills complete. Adding it earlier would fail on legacy rows where the new columns are still empty strings (those rows still have `cycle_id` set, so the CHECK passes, but adding-then-backfilling-then-checking is cleaner sequencing for human readers of the migration file).

### Helper-rewrite contract (the C2 critical fix)

Story 21.27's lifecycle simplification removed `assertCycleEditableUi` from `compliance-finding.ts` so findings are unconditionally editable. But it left `loadFindingScopedToWorkspace` joining through `cycle.workspace_id`, which was correct in a world where every finding had a cycle. Story 23.1 makes `cycle_id` nullable — and the existing helper would silently return null for every ad-hoc finding without that rewrite.

The rewrite (AC 6 Part B) is small (one where-clause swap) but **carries every existing 21.7/21.8 mutation along** — `updateFinding`, `closeFinding`, `reopenFinding`, `spawnTaskForFinding`, and the read in `listFindingsForCycle`. Once the helper accepts ad-hoc findings, every consumer transparently does too. This is the single highest-leverage edit in the story.

### Permission semantics under the relaxed model

A user with `tasks:edit` in workspace W can:
- Create cycle-scoped findings in any cycle of W (existing 21.7 behaviour, unchanged)
- Create ad-hoc findings against any `LawListItem` or `Requirement` in any laglista of W (new in 23.1)
- Assign findings to any `WorkspaceMember` of W (new in 23.1)
- Comment on findings, including findings whose cycle is AVSLUTAD (new in 23.1; deviates from cycle-mutation gate, intentional per AC 11.3)

A user with only `activity:view` (typical AUDITOR) can:
- Read findings via `listFindingsForWorkspace` and `getFindingById` (existing OR-gate pattern)
- NOT create, assign, comment, anchor-change

PO has accepted reusing `tasks:edit` for v1 (Decision 3); a follow-up story can introduce per-feature `findings:create` / `findings:assign` / `findings:comment` if a customer asks for fine-grained control.
