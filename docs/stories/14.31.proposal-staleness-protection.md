## Story

**As a** user who may approve an agent proposal hours or days after it was made,
**I want** the system to refuse to apply a change when the underlying data has shifted since the proposal,
**so that** I never silently overwrite someone else's edit with a stale agent suggestion — the proposal is blocked and the agent is asked to re-read and propose afresh.

## Context

A `PendingAgentAction` is durable: the user can walk away and approve it tomorrow (that persistence is the whole point of Phase 5). But durability creates a hazard for **overwrite** actions. If the agent proposes "set kravpunkt X's text to Y" on Monday, and a colleague edits that kravpunkt's text on Tuesday, approving the proposal on Wednesday would clobber Tuesday's edit with Monday's assumptions.

This story adds a **freshness guard**: each overwrite-type tool snapshots the target entity's `updated_at` into `params.entity_version` at propose time (Stories 14.28 and 14.30 already do this; this story retrofits the rest and makes it a convention). At approval time, `approvePendingAction` re-reads the target's current `updated_at` and, if it differs from the snapshot, **blocks the dispatch**, leaves the row `PENDING`, and surfaces a Swedish "the data changed" message with a re-propose affordance.

Not every type needs this. **Create** (`CREATE_TASK`, `DRAFT_DOCUMENT`) and **append/link** (`ADD_TASK_COMMENT`, `ADD_OBLIGATION`, `LINK_*`) actions don't overwrite an existing field — for them, the dispatch's own existence/FK check is sufficient. The guard is scoped to types whose semantics are "replace the current value of an existing entity."

## Acceptance Criteria

### Convention & Snapshot Coverage

1. **No schema change.** `params.entity_version` (the target's `updated_at` ISO-8601 string at propose time) is a JSON convention inside the existing `PendingAgentAction.params`. Document the convention in a code comment at the top of `app/actions/pending-agent-actions.ts`.
2. A single source of truth `STALENESS_STRATEGY: Record<PendingAgentActionType, 'guarded' | 'unguarded'>` is defined (co-located with the dispatch). New action types added in future MUST add an entry (enforce via an exhaustive `Record` type so a missing key is a compile error).
3. **Retrofit snapshot population** to the overwrite-type tools once 14.23 has migrated them to the pending-action pattern. **Present-state caveat:** `assign-task.ts` does not exist yet (it is *created* by 14.23 Task 2.4), and `update-compliance-status.ts` exists today in its **legacy `execute: true` direct-write form** (Story 14.7c — no `PendingAgentAction`, no `params`); 14.23 Task 7.2 migrates it. **This story's edit** is: *after* that migration, add `params.entity_version` capture (read the target's current `updated_at` on the `execute: false` path) to both `assign-task.ts` (target `Task`) and `update-compliance-status.ts` (target `LawListItem`). (14.28 `update-requirement.ts` and 14.30 `transition-document-status.ts` capture it from the start.)

### Freshness Guard

4. New helper `resolveEntityVersion(action): Promise<string | null>` in `app/actions/pending-agent-actions.ts`. For `guarded` types it loads the target model by the id in `params` and returns `updated_at.toISOString()`; for `unguarded` types it returns `null` (skip). If a guarded type's target row no longer exists, it returns the sentinel `'__MISSING__'`. (The sentinel can never collide with a real value: `entity_version` is always an ISO-8601 `toISOString()` string, never `'__MISSING__'` — note this in a code comment.)
5. `approvePendingAction` calls the guard **before** dispatching, only for `guarded` types:
   - Read `current = resolveEntityVersion(action)`.
   - If `current === '__MISSING__'` → return `{ success: false, error: 'STALE', message: 'Underlaget finns inte längre. Be agenten läsa om och föreslå på nytt.' }`. Row stays `PENDING`.
   - If `current !== action.params.entity_version` → return `{ success: false, error: 'STALE', message: 'Datan har ändrats sedan förslaget gjordes. Be agenten läsa om och föreslå på nytt.' }`. Row stays `PENDING`; **dispatch does not run**.
   - If equal → proceed to dispatch as today.
6. The staleness block is **non-destructive**: the row remains `PENDING` (not `REJECTED`/`EXPIRED`), so the user can still reject it manually or re-propose. The `{ error: 'STALE' }` discriminator lets the client distinguish a freshness block from a dispatch failure.
7. **TOCTOU acknowledgement (v1 scope):** the read-compare-dispatch sequence is not a single DB transaction; a sub-second race remains theoretically possible. v1 targets the real-world case (hours/days-old proposals against drifted entities), which the `updated_at` comparison catches reliably. A fully transactional `WHERE updated_at = :snapshot` optimistic-lock is noted as a future hardening, not in scope. Document this in a code comment.

### Renderer — Stale State

8. The **staleness banner** is rendered **once in the shared `AgentActionCard` wrapper** (not duplicated across the four guarded-type renderers), keyed off the most recent `approvePendingAction` result being `error: 'STALE'`: an amber inline notice with the Swedish message (AC 5) + a "Be agenten uppdatera förslaget" button. Centralising it in the wrapper avoids four copies of the banner logic.
9. "Be agenten uppdatera förslaget" button behaviour (v1):
   - Calls `rejectPendingAction(id)` to retire the stale proposal, **then**
   - Triggers a re-propose request to the agent. `useChatInterface` exposes `sendMessage(content)` ([`lib/hooks/use-chat-interface.ts:260`](../../lib/hooks/use-chat-interface.ts)), but it is **owned in `chat-panel.tsx` and consumed down in `chat-message.tsx`** — it is not guaranteed in scope at the card render site. **Preferred path:** thread a single `onReproposeRequest(summary: string)` callback down from where `sendMessage` lives to the `AgentActionCard`, which the card invokes with a templated Swedish prompt (e.g., "Underlaget för '${summary}' har ändrats. Läs om det och föreslå på nytt om det fortfarande behövs."). **Fallback** (only if threading the callback proves impractical): reject the row + show a toast instructing the user to re-ask. Document which path was taken and why.
10. The banner reads the STALE state from the approve-call result (local component state), not from a persisted field — staleness is a transient property of the *current* data, not a stored status.

### Batch Card

11. `AgentActionBatchCard`'s "Godkänn alla" handler treats a `STALE` result like a skipped row: it stays `PENDING`, is **not** counted as approved, and the summary reflects it — "2 av 3 godkända — 1 inaktuell" / "1 av 3 godkända — 2 inaktuella" (pluralise: 1 → "inaktuell", >1 → "inaktuella"). The per-row staleness banner appears on the affected row.

### Non-Regression

12. Unguarded types (create/append/link) are entirely unaffected — no extra read, no guard, no behaviour change.
13. The `expirePendingActions` cron (time-based, 14.22) is independent and unchanged; a row can be both fresh-enough and not-yet-expired, stale-and-not-expired, etc. The two safety nets do not interact.
14. Existing single-action approvals for guarded types succeed normally when the entity has **not** drifted (the common case) — verified with the snapshot equal to current `updated_at`.

### Testing Requirements

- **Framework**: Vitest 1.4+ + React Testing Library. **Locations**: `tests/unit/` mirrors source tree. **Coverage**: 60–70%. [Source: architecture/3-tech-stack.md#3.1]
- **Guard**: matching snapshot → dispatch; drift → STALE + row PENDING + dispatch not called; missing target → STALE; unguarded type → no guard, no extra read.
- **Strategy map**: a compile-time exhaustiveness test (a `satisfies Record<PendingAgentActionType, ...>` assertion) so adding a future type without classifying it fails the build.
- **Renderer/batch**: banner appears on STALE; re-propose button rejects + appends; batch summary counts stale rows separately.

### Technical Constraints

- **Guard only overwrite types** — never add a read for create/append/link types (perf + no semantic need). [Source: this story's classification table]
- **Non-destructive on staleness** — the row stays `PENDING`; let the user decide. [Source: this story's design — distinct from `expirePendingActions` which sets `EXPIRED`]
- **`error: 'STALE'` discriminator** — must be a stable string the client switches on; do not fold it into a generic error message. [Source: architecture/18-error-handling-strategy.md#18.5]
- **Exhaustive strategy record** — a missing key must be a TypeScript error, forcing future types to declare guarded/unguarded. [Source: architecture/17-coding-standards.md#17.x type-safety conventions]
- **TOCTOU** — v1 is read-compare-dispatch (not a transactional optimistic lock); acceptable, documented. [Source: this story AC 7]
