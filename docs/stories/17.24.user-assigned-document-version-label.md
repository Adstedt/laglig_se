# Story 17.24: User-Assigned Document Version Label — Brownfield Addition

## Status

Draft

## Story

**As a** workspace member who manages governing documents (styrdokument),
**I want** to assign my own version designation (e.g. "1.0", "2.3", "Rev C") to a document when I import it and when I approve it,
**so that** Laglig reflects my organisation's own governance versioning — not just the system's internal edit counter — and the agent cites the version my colleagues actually recognise.

## Story Context

**The problem (and the core risk this story must resolve):**

Laglig already tracks a version per document via `WorkspaceDocumentVersion.version_number` — a **system-managed, monotonic integer** (always starts at 1, auto-increments on each saved/approved revision, drives the dual-pointer model and supersession). This is really an *edit/revision counter*, not the organisation's own version identity.

Users, however, think of "the version" as **their own governance label** ("1.0", "2.3", "Rev C", sometimes a date). Today that label only lives inside the document body — the agent surfaced it by *reading the content* (observed: agent reported `Version: 1.0 | Gäller från: 2026-06-15` from an imported GDPR policy), but it is not a structured field, so it can't be displayed, filtered, or cited reliably.

**The risk to manage explicitly:** if we add a user version label naively, the UI ends up showing **two competing "version N" numbers** on the same document (internal `version_number=3` vs author label "Version 1.0") — confusing and eroding trust. This story's design rules (see Dev Notes → "Two versions, one mental model") exist specifically to prevent that.

**Existing System Integration:**

- Integrates with: `WorkspaceDocument` + `WorkspaceDocumentVersion` (Story 17.1, dual-version model Story 17.16), the docx import flow (Story 17.5), the lifecycle/approval flow (Story 17.4), and the styrdokument RAG/citation pipeline (Stories 17.9b / 17.10b / 17.18 / 17.9d).
- Technology: Next.js server actions (`app/actions/documents.ts`), Prisma + Postgres (additive nullable column), Tiptap document model, pgvector chunk index (`ContentChunk` + `metadata` JSON), Zod validation (`lib/validation/documents.ts`).
- Follows pattern: `withWorkspace()` auth + `ActionResult<T>` server actions; per-version provenance already lives on `WorkspaceDocumentVersion` (`source`, `change_summary`, `approved_at/by`); chunk `metadata` already carries `version_number`, `status`, `tier`.
- Touch points: `importDocxDocument()` and `promoteDraftToApproved()` in `app/actions/documents.ts`; `import-document-dialog.tsx`; the approval modal/UI; document settings panel (optional); the reindex/citation path (chunk `metadata` + citation pill).

## Acceptance Criteria

**Data model (additive, nullable):**

1. Add an optional column `author_version_label String?` (max 50 chars) to `WorkspaceDocumentVersion` — a per-version, free-text governance label. Migration is **additive + nullable, no backfill**; existing rows stay `null`.
2. The field accepts free text (no enforced format) so "1.0", "2.3", "Rev C", "2024-03", etc. are all valid. It is **never** auto-derived from or synced to `version_number`.

**Two versions, one mental model (the anti-confusion rules):**

3. The UI MUST NOT present two competing "version N" numbers for the same document. The system `version_number` is surfaced to users as **"Revision"** (an edit-history counter) and ONLY in history/timeline/audit contexts; the author label is surfaced as **"Version"** in primary document chrome.
4. Where both must appear together (e.g. the revision history list), they are visually distinct and explicitly labelled, e.g. `Revision 3 · Version 2.3`. A revision with no author label shows just `Revision 3`.

**Entry point A — import / upload:**

5. The import dialog (`import-document-dialog.tsx`) gains an optional **"Versionsbeteckning"** input (free text, ≤50 chars, may be left blank).
6. The field is best-effort **pre-filled from the filename** via a small heuristic (e.g. `…v2.3.docx`, `… 2.3.docx`, `…_rev_c.docx` → suggested label); the user can edit or clear it. _(Optional stretch: also read docx core properties such as `cp:revision` / custom properties.)_
7. On import, the (possibly blank) label is stored on the created v1 `WorkspaceDocumentVersion.author_version_label`. Blank → `null` (no fabricated value).

**Entry point B — approval / promotion:**

8. When a user promotes a draft to APPROVED (`promoteDraftToApproved()`), the approval UI **requires** a version label for the version being approved — approval cannot complete with an empty label. The dialog displays the **last approved version for reference**, e.g. `Senast godkänd: Version 1.0 · Revision 1`.
9. The label is **not** auto-incremented (1.0 → 1.1 vs 2.0 is a human decision). The input may be pre-filled from this version's existing draft label if one exists; otherwise it starts empty with the previous approved label shown above it as context. If the entered label **equals the previous approved label**, a soft, non-blocking warning is shown (`Samma version som föregående — är det avsiktligt?`) — the user may still proceed.
10. Saving stores the label on the promoted version. It does **not** change `version_number`, the dual pointers, or supersession logic in any way.

**Editing later (optional / may defer to 17.24b):**

11. The author version label is editable from the document settings panel without requiring a re-approval, and the edit is persisted to the current relevant version.

**Display + citation (the payoff; splittable to 17.24b if scope must shrink):**

12. When present, the author version label is shown in the document header/detail UI alongside status (e.g. `Godkänd · Version 2.3`).
13. `author_version_label` is included in `ContentChunk.metadata` on (re)index, and the citation label **prefers it when present** — `[Källa: GDPR-policy 2.3]` / `[Utkast: GDPR-policy 2.3]` — falling back to `v{version_number}` when the label is `null`. Existing citations for label-less documents are unchanged.

**Integration / quality:**

14. Import and approval flows behave exactly as today when no label is provided (the `null` path is the default and is fully exercised).
15. No change to `version_number` sequencing, the dual-pointer (`current_approved_version_id` / `current_draft_version_id`) model, supersession (`superseded_at`), or de-indexing of SUPERSEDED/ARCHIVED docs.
16. Tests cover: filename pre-fill heuristic, store-on-import, store-on-approve, edit-later, citation preference + `null` fallback, and the unchanged label-less path.

## Dev Notes

### Two versions, one mental model (READ FIRST — this is the point of the story)

| Concept | Field | Owner | Where shown | Label in UI |
| --- | --- | --- | --- | --- |
| Edit/revision counter | `WorkspaceDocumentVersion.version_number` | System (monotonic, append-only) | History / timeline / audit only | **"Revision {n}"** |
| Governance version | `WorkspaceDocumentVersion.author_version_label` (new) | User (free text, optional) | Primary chrome, header, citations | **"Version {label}"** |

Hard rules:
- Never render two "version N" numbers competing for the same document. Internal counter → "Revision"; author label → "Version".
- The author label is **never** derived from, validated against, or kept in sync with `version_number`. They are independent by design.
- Per-version storage (not per-document) so each approved revision keeps the label it shipped with, labels travel through re-uploads, and history reads correctly.

### What already exists (do NOT re-implement)

- `WorkspaceDocumentVersion` already carries per-version provenance: `source` (TIPTAP|IMPORT|AGENT), `change_summary`, `approved_at`/`approved_by`, `superseded_at`, `storage_path`. Add the label alongside these.
- Dual-pointer model (Story 17.16): `current_approved_version_id` / `current_draft_version_id`; `version_number` is computed via `MAX(version_number)+1`. **Leave all of this untouched.**
- Indexing is automatic (`after()` → `indexWorkspaceDocument`), dual-tier (DRAFT/APPROVED), debounced reindex (60s) + cron sweep; chunk `metadata` already holds `version_number`, `status`, `tier`. Add the label to that metadata; status flips use `updateChunkMetadata()` (cheap UPDATE, no re-embed).
- Citations render `[Källa: X v1]` (approved) / `[Utkast: X v1]` (draft) today — the fallback when no author label exists must stay exactly this.

### File locations

- `prisma/schema.prisma` — add column to `WorkspaceDocumentVersion` (~2117–2159).
- `app/actions/documents.ts` — `importDocxDocument()` (~2175–2317), `promoteDraftToApproved()` (~1638–1777), optional settings-edit action.
- `lib/validation/documents.ts` — add `authorVersionLabel` to import/promote/settings schemas.
- `lib/documents/guess-version-label.ts` — new helper (Task 2).
- `components/features/documents/import-document-dialog.tsx` — import field.
- Approval modal/UI + document settings panel — confirm exact components in `components/features/documents/`.
- `lib/chunks/workspace-document-reindex.ts` + `lib/chunks/sync-workspace-chunks.ts` — metadata; citation pill render (confirm file; Stories 17.9d / 17.18).

### Migrations are manual

Do not run the migration. After editing `schema.prisma`, hand the user the command (e.g. `pnpm prisma migrate dev --name add_author_version_label`) and confirm none is needed elsewhere.

## Risk and Compatibility Check

**Minimal Risk Assessment:**
- **Primary Risk:** UI confusion between the internal counter and the author label (two "version" numbers). **Mitigation:** the naming discipline in AC 3–4 / Dev Notes (counter = "Revision", label = "Version"); per-version storage; label never synced to `version_number`.
- **Secondary Risk:** citation regression for existing (label-less) documents. **Mitigation:** strict fallback to `v{version_number}`; explicit regression test (AC 16).
- **Rollback:** the column is additive + nullable; feature reads default to `null`. Revert the UI/action/citation changes and the column can stay harmlessly (or be dropped) — no data migration to unwind.

**Compatibility Verification:**
- [ ] No breaking changes to existing server-action signatures (new param is optional).
- [ ] Database change is additive only (nullable column, no backfill).
- [ ] UI changes follow existing dialog/settings patterns (Story 17.5 import dialog, 17.4 settings panel).
- [ ] Performance impact negligible (one nullable column; metadata is a JSON merge with no extra embedding cost).
