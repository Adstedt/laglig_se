## Story

**As a** user drafting a styrdokument from the global compliance chat (not from a specific kravpunkt/task),
**I want** the draft to still be linkable to the right law list item(s) and task(s),
**so that** an agent-authored document doesn't land orphaned just because I wasn't inside a specific entity's modal.

## Context — the gap (verified in code during 14.24 smoke testing)

For a `contextLinks` entry to wire up on approval, the agent needs a real **`LawListItem.id`** (workspace law-list entry) or **`Task.id`**. Today, from a global chat, it has neither:

- **`search_laws` returns `LegalDocument` ids** (`r.sourceId` at `lib/agent/tools/search-laws.ts:64`) — the SFS law itself, a *different entity* from a per-workspace `LawListItem`. Putting one in `contextLinks` as `LIST_ITEM` → `linkDocumentToListItem` FK lookup fails → partial-link error.
- **No tool returns `Task.id`** at all.
- **The chat context only surfaces title/SFS** to the model (`lib/agent/system-prompt.ts:260–264`), not the entity id — so even in a LAW/TASK chat the model can't read the id (14.24 works around this by auto-deriving server-side from `writeContext.contextId`, which only exists for LAW/TASK context chats).

Net: from a **global** chat, approved drafts are created **unlinked** (best case) or with a **failed link** (if the model guesses). The auto-link value prop only works from the kravpunkt/task modal.

## Approach options (for SM/PO + Architect to decide)

**Option A — Agent id-resolution tools (model-driven).**
Add read tools so the agent can resolve workspace ids before drafting:
- `find_law_list_items(query)` → returns `[{ listItemId, title, documentNumber }]` for the workspace's law list.
- `find_tasks(query)` → returns `[{ taskId, title }]`.
The agent calls these, then populates `contextLinks`. **Pro:** fully conversational ("link it to the fire-safety task"). **Con:** another model round-trip + the model must pick the right match; also fixes `create_task.relatedDocumentId`, which has the *same* latent gap.

**Option B — Link picker on the draft card (user-driven). ★ likely simplest + most reliable.**
Add a "+ Länka till" affordance to the `DraftDocumentApprovalRenderer` (reuse `EntityPickerDialog` / the existing settings-panel picker, Story 17.14) so the **user** picks the task/kravpunkt to link, writing valid ids into `params.contextLinks` via `updatePendingActionParams`. **Pro:** no model-id problem at all, deterministic, reuses shipped UI. **Con:** manual (but it's a review surface anyway).

**Option C — Both.** Picker for reliability (B) + resolution tools (A) for the conversational path. Probably the eventual end-state.

**Recommendation:** start with **B** (deterministic, reuses 17.14, also the natural fit for a review card), and add **A** only if users want fully hands-free linking. Note A also repairs `create_task`'s `relatedDocumentId` gap, so it has leverage beyond drafts.

## Acceptance Criteria (outline — refine in SM pass)

1. From a global chat, a user can link an approved/pending draft to one or more `LawListItem`s and/or `Task`s using valid workspace ids.
2. (Option B) The draft card exposes "+ Länka till" → entity picker (reuse `EntityPickerDialog` / settings-panel picker); selections write valid ids into `params.contextLinks` via `updatePendingActionParams`; chips render + dismiss as today.
3. (Option A, if chosen) `find_law_list_items` / `find_tasks` read tools return workspace-scoped ids; agent populates `contextLinks` from them; `create_task.relatedDocumentId` can use the same resolver.
4. Approval dispatch wires the links exactly as 14.24 (positional `linkDocumentToTask` / `linkDocumentToListItem`, partial-failure handling).
5. The 14.24 context auto-derive (LAW/TASK modal) is unchanged and composes with whatever this story adds.
6. Workspace isolation: resolved/picked ids must belong to the current workspace (the existing pickers + server actions enforce this — verify).

## Dev Notes

- **14.24 already auto-derives from `writeContext.contextType/contextId`** for LAW/TASK context chats — this story only addresses the GLOBAL case. Don't duplicate the context path.
- **`create_task.relatedDocumentId` shares the gap** — it's described as "LawListItem ID" but the agent has no resolver. Option A would fix both; worth scoping together.
- **Picker substrate:** `components/features/documents/document-settings-panel.tsx` already has a "Länka till" + picker; Story 17.14 generalises it to `EntityPickerDialog`. Reuse, don't rebuild.
- **Dispatch is done** — `approvePendingAction` `DRAFT_DOCUMENT` branch already iterates `contextLinks` with positional link actions + `partialLinkErrors`. This story only needs to *populate* valid links.
