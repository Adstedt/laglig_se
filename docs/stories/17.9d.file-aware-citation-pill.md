# Story 17.9d: File-aware Citation Pill — Snippet Sidebar + Open-in-Preview for USER_FILE Sources

## Status

Draft

> 📌 **Why this exists (created 2026-05-22):** [[17.9c]] shipped the `search_workspace_files` tool and made `[Källa: <filnamn>]` markers **resolve** to a citation pill (AC 7) — that part works. But the pill's *click-through* still renders through the legal-document path: clicking a file citation opens the sidebar with **legal copy** ("Fullständig lagtext visas i lagläsaren") and is a **dead-end** (no snippet, no link), because the whole citation UX is gated on `source.path` (a chapter/§ concept files don't have). This story makes the citation **file-aware**: the cited passage shows inline, and a button opens the actual document. UX polish surfaced during 17.9c's manual smoke test — deliberately kept **out** of 17.9c (which is Done/PASS) to avoid bolting UX onto a clean story.

## Story

**As a** user reading an AI answer grounded in my workspace's uploaded files,
**I want** clicking a `[Källa: <filnamn>]` pill to show me the exact passage it used and let me open the document,
**so that** I can verify and trace the answer to its source — the same way SFS citations let me jump to the law text.

## Story Context

**Existing System Integration:**

- **Producer (Story 17.9c):** `search_workspace_files` (`lib/agent/tools/search-workspace-files.ts`) already returns per result `fileId` (= chunk `sourceId`), `filename`, `category`, `snippet`, `citationKey` (= filename). The citations branch added in 17.9c (`extractSourcesFromToolResult` in `lib/ai/citations.ts`) currently maps only `filename → SourceInfo` (`documentNumber`/`title` = filename, `snippet`) — it **drops `fileId` and `category`**. This story carries them through.
- **`SourceInfo`** (`lib/ai/citations.ts:15`): `{ documentNumber: string, title, snippet, slug, path, anchorId, url? }`. No source-kind discriminator today — a file (no `url`, no `path`, no `slug`) is indistinguishable from a legal doc-level fallback, which is why the renderer treats it as a law.
- **Citation pill** (`components/features/ai-chat/citation-pill.tsx`): snippet is gated on `source.path` — `isChunkResolved = !!source.path` (`:110`); the click payload sends `snippet: isChunkResolved ? source.snippet : ''` (`:140`); the hover card sets `description = isChunkLevel ? source.snippet : null` (`:190-191`). Files have `path: null` → snippet suppressed in **both** surfaces.
- **Citation detail sidebar** (`components/features/ai-chat/details/citation-detail.tsx`): two branches only — web source (`:54`) and "DB source" (`:88`), the latter hardcoded for laws: snippet-or-`"Fullständig lagtext visas i lagläsaren"` (`:104`) + a `"Visa i lagläsaren"` link built from `slug` (`:110`). **No `USER_FILE` branch.**
- **Detail payload type** `CitationDetailData` (`lib/ai/chat-detail-context.tsx:23`) + `ChatDetailItem` union (`:128`) — the shape passed to `openDetail`.
- **File preview (reuse target):** `QuickPreview` (`components/features/files/quick-preview.tsx:36`) — a self-contained modal overlay. Signed-URL action `getFileDownloadUrl(fileId)` (`app/actions/files.ts:1291`, 1h URL).

**Out of scope:** `WORKSPACE_DOCUMENT`/styrdokument citations (that's **Story 17.10**, which adds the styrdokument search consumer — the `kind` discriminator added here should be designed so 17.10 adds a `'workspace_document'` kind cleanly, not duplicated). Any change to legal (SFS) or web citation rendering beyond the additive discriminator. Changing the `search_workspace_files` tool's result shape (it already returns `fileId`/`category`).

## Acceptance Criteria

### Carry the file identity through

1. `SourceInfo` (`citations.ts:15`) gains optional, additive fields: `kind?: 'file'` (absence = today's legal/web behavior, inferred as now from `url`/`path`), `fileId?: string`, `category?: string | null`. The 17.9c `search_workspace_files` branch in `extractSourcesFromToolResult` sets `kind: 'file'`, `fileId` (from `item.fileId`), and `category` (from `item.category`) **in addition to** the existing filename/snippet mapping. Legal/web branches are untouched.
2. `CitationDetailData` (`chat-detail-context.tsx:23`) carries the file fields it needs to render + open the preview: add optional `fileId?: string`, `category?: string | null`, `kind?: 'file'`. `citation-pill.tsx`'s `openDetail` call (`:134`) threads them from the resolved `SourceInfo`.

### Show the cited passage (the core fix)

3. **A file citation's snippet is no longer suppressed.** In `citation-pill.tsx`, widen the snippet condition so a `kind === 'file'` source shows `source.snippet` in **both** the hover card (`:190-191`) and the sidebar payload (`:140`). The existing legal behavior (doc-level entries without `path` suppress the snippet) must be preserved for non-file sources.
4. `citation-detail.tsx` gains a **file branch** (sits before/within the DB branch, keyed off `kind === 'file'` / presence of `fileId`): renders **filename (title) + category label + the snippet** in the "Källa" card. It must **not** show the `"Fullständig lagtext visas i lagläsaren"` copy and must **not** render the `"Visa i lagläsaren"` link.

### Open the document

5. The file branch shows an **"Öppna i förhandsvisning"** action that opens the file preview for `fileId`:
   - **Preferred:** mount `QuickPreview` (`quick-preview.tsx:36`) — requires a `WorkspaceFileWithLinks` (fetch the file by `fileId` via a files action/hook) and `getFileUrl` wired to `getFileDownloadUrl`.
   - **Acceptable v1 fallback** (if embedding `QuickPreview` in the chat tree exceeds the single-session budget — **DEC for Dev**): an **"Öppna filen"** button calling `getFileDownloadUrl(fileId)` (`files.ts:1291`) and opening the signed URL in a new tab. Either satisfies "open the document"; document the choice in the Dev Agent Record.

### Non-regression

6. **Legal (SFS) and web citations are behaviourally unchanged.** The snippet-gate widening (AC 3) must not surface doc-level snippets for legal sources that previously suppressed them, and the new sidebar branch must not intercept legal/web sources. Verify against the existing `tests/unit/lib/ai/citations.test.ts` + a render check of the legal path.

### Quality

7. Unit tests: (a) `extractSourcesFromToolResult` sets `kind:'file'` + `fileId` + `category` for `search_workspace_files` results; (b) the file branch in `citation-detail.tsx` renders filename/category/snippet and **omits** the lagläsaren copy + lagläsaren link; (c) a non-file (legal) source still suppresses its doc-level snippet (AC 6 guard). `pnpm typecheck` + `pnpm eslint` clean on changed files. Manual smoke: click a file citation → snippet shows → "Öppna i förhandsvisning" opens the document.

## Dev Notes

- **Discriminator design (for 17.10):** model `kind` as an open string union (`'file'` now; `'workspace_document'` later) rather than a boolean — Story 17.10 adds styrdokument citations and will want its own branch + affordance (open in the styrdokument editor, not Filer preview). Keep the legal/web inference (`url`→web, `path`/`slug`→legal) as the default when `kind` is absent so nothing legacy needs touching.
- **Why the snippet was blank (root cause):** the citation UX equates "has a snippet worth showing" with "is a chunk-level legal hit" via `!!source.path`. Files are chunk-level but have no legal `path`, so they fall through to the doc-level (no-snippet) treatment. The fix is to treat `kind==='file'` as snippet-bearing.
- **QuickPreview reuse caveat:** `QuickPreview` expects a full `WorkspaceFileWithLinks` + a `files[]` array (for prev/next nav) + a `getFileUrl` resolver. From a citation we only have `fileId`. The preferred path fetches the file by id; the fallback (signed-URL new tab) avoids that wiring entirely. Pick based on the single-session budget — both meet AC 5.
- **No schema, no new deps, no server-action changes** (beyond *calling* the existing `getFileDownloadUrl`). Frontend-only.

### Testing

Vitest. Extend `tests/unit/lib/ai/citations.test.ts` for the `kind:'file'` mapping (AC 7a + the AC 6 legal-suppression guard). Add a component/render test for the `citation-detail.tsx` file branch (AC 7b) — React Testing Library, mirroring existing detail-panel tests if present. Manual click-through is the acceptance check for the preview wiring (AC 5).

## Risk and Compatibility

- **Primary risk:** the snippet-gate widening (AC 3) accidentally changes legal citation rendering (surfacing doc-level summaries as if they were the cited section). **Mitigation:** gate the new behavior strictly on `kind==='file'`; AC 6 + the AC 7c test guard legal suppression. **Rollback:** revert the 3-file frontend diff; pills fall back to today's resolve-but-dead-end behavior (no data loss, no schema).
- **Compatibility:** no API/schema changes; additive optional `SourceInfo`/`CitationDetailData` fields; UI follows the existing citation-detail/QuickPreview patterns; negligible perf impact (one extra file fetch only on click, only for the preview path).
