# Story 8.22: Use SectionChange Table for AI Context Instead of changed_sections JSON

## Status

Draft

## Story

**As a** compliance officer using the AI chat to review amendments,
**I want** the AI to have accurate section-level change data in its context,
**so that** it can reference exactly which paragraphs were added, amended, or repealed.

## Context & Dependencies

### Existing System Integration

- **`lib/agent/system-prompt.ts`** (lines 268-271) — reads `changed_sections` (unreliable JSON field on ChangeEvent, populated by regex parsing) to build the system prompt context for assessment conversations
- **`lib/agent/tools/get-change-details.ts`** (lines 56-83) — already reads from `section_changes` table via AmendmentDocument join, returning rich `affectedSections` with chapter, section, changeType, description, oldText, newText
- **`changed_sections`** on ChangeEvent is a `Json?` field populated by `findChangedSections()` in `lib/sync/section-parser.ts` — regex-based, often returns empty/null
- **`section_changes`** table has the real parsed data from amendment PDFs — populated by `discover-sfs-amendments` with chapter, section number, change type, content

### The Problem

The system prompt uses `changed_sections` (regex-based, often null) while the tool already uses `section_changes` (LLM-parsed, reliable). This means the AI's initial context for an assessment conversation often lacks section details, even when they're available in the database.

### Scope

1. System prompt reads from `section_changes` table instead of `changed_sections` JSON
2. Remove redundant `changedSections` field from `get-change-details` tool response (it already returns the richer `affectedSections`)
3. Do NOT drop the `changed_sections` column from schema — that's a separate migration story

## Acceptance Criteria

1. System prompt context for assessment conversations includes section-level details from `section_changes` table (via AmendmentDocument join on `amendment_sfs`)
2. Section details in the prompt show: chapter, section number, and change type (AMENDED/REPEALED/NEW) — not full text content (too large for prompt context)
3. When no `section_changes` exist for an amendment, the prompt gracefully omits the section line (same as current behavior when `changed_sections` is null)
4. `get-change-details` tool response no longer includes the redundant `changedSections` field — `affectedSections` is the single source
5. Existing assessment flow and AI chat behavior unchanged
6. All existing tests pass, `pnpm build` clean, `tsc` clean

### Key Pattern (from get-change-details.ts:56-83)

```typescript
if (changeEvent.amendment_sfs) {
  const amendmentDoc = await prisma.amendmentDocument.findFirst({
    where: { sfs_number: changeEvent.amendment_sfs },
  })
  if (amendmentDoc) {
    const sections = await prisma.sectionChange.findMany({
      where: { amendment_id: amendmentDoc.id },
      orderBy: { sort_order: 'asc' },
      select: { chapter: true, section: true, change_type: true },
    })
  }
}
```
