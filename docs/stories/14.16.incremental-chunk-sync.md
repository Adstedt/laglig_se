# Story 14.16: Incremental Chunk Sync with Content Hashing

## Status

Draft

## Story

**As a** platform operator managing RAG infrastructure costs,
**I want** the chunk pipeline to only re-process chunks whose content actually changed,
**so that** amendments to large laws (e.g., Miljöbalken with 580 chunks) don't trigger 580 Haiku + 580 embedding calls when only 5 paragraphs changed.

## Context & Dependencies

### The Problem

When `sync-sfs-updates` applies an amendment to a base law, it updates the `LegalDocument`'s content fields. The `process-chunks` cron then detects `updated_at > max_chunk_created` and calls `syncDocumentChunks()`, which:

1. **Deletes ALL existing chunks** for the document
2. **Re-chunks the entire document** from scratch
3. **Generates context prefixes** (Claude Haiku call per chunk)
4. **Generates embeddings** (OpenAI call per chunk)

For a large law like Miljöbalken (~580 chunks), a single paragraph amendment costs:
- ~580 Haiku API calls (context prefixes)
- ~580 OpenAI embedding calls
- ~10 minutes of processing time
- The entire document is unsearchable during re-processing (chunks deleted before new ones created)

### Existing System Integration

- **`lib/chunks/sync-document-chunks.ts`** — `syncDocumentChunks()` does atomic delete-all + create-all (lines 106-113)
- **`lib/chunks/chunk-document.ts`** — `chunkDocument()` produces chunks with `path` (e.g., "kap.2/§3"), `content`, and `contextual_header`
- **`lib/chunks/generate-context-prefixes.ts`** — Haiku call per chunk
- **`lib/chunks/embed-chunks.ts`** — `generateEmbeddingsBatch()` for OpenAI embeddings
- **`app/api/cron/process-chunks/route.ts`** — cron that finds stale docs and calls `syncDocumentChunks()`
- **`content_chunks` table** — has `content` (text), `embedding` (vector), `content_prefix`, `path`, `contextual_header`

### The Solution

Add a content hash per chunk. On re-sync:
1. Chunk the document as before (produces new chunk list)
2. Compare each new chunk's content hash against the existing chunk at the same `path`
3. **Unchanged chunks** — keep as-is (no Haiku, no embedding)
4. **Changed/new chunks** — generate prefix + embedding
5. **Removed chunks** (path no longer exists) — delete

This turns a 580-chunk full rebuild into a 5-10 chunk incremental update for a typical amendment.

## Acceptance Criteria

1. `content_chunks` table gets a new `content_hash` column (`String`, SHA-256 hex of the chunk `content` text)
2. Existing chunks are backfilled with content hashes (migration or backfill script)
3. `syncDocumentChunks()` compares new chunks against existing by `(source_id, path)` + `content_hash`
4. Chunks with matching path + hash are skipped (no Haiku call, no embedding call, no DB write)
5. Chunks with matching path but different hash are updated (new content, new prefix, new embedding)
6. New chunks (path doesn't exist yet) are created with prefix + embedding
7. Chunks with paths that no longer exist in the new chunk list are deleted
8. The document remains searchable during incremental sync (no delete-all step)
9. Full re-chunk can be forced via a `force` parameter (bypasses hash comparison)
10. `process-chunks` cron stats track: `chunksSkipped` (unchanged), `chunksUpdated`, `chunksCreated`, `chunksDeleted`
11. Existing `syncDocumentChunks()` API contract unchanged — callers don't need modification
12. All existing tests pass, `pnpm build` clean, `tsc` clean

### Content Hash

```typescript
import { createHash } from 'crypto'

export function contentHash(text: string): string {
  return createHash('sha256').update(text).digest('hex')
}
```

### Cost Estimate

A typical amendment changes 3-10 paragraphs. For Miljöbalken (580 chunks):
- **Before:** 580 Haiku + 580 embedding calls (~$0.50, ~10 min)
- **After:** 5-10 Haiku + 5-10 embedding calls (~$0.005, ~30 sec)
- **Savings:** ~99% per amendment to large laws
