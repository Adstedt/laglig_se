# Story 14.16: Incremental Chunk Sync with Content Hashing

## Status

Draft

## Story

**As a** platform operator managing RAG infrastructure costs,
**I want** the chunk pipeline to only re-process chunks whose content actually changed,
**so that** amendments to large laws (e.g., Miljöbalken with ~580 chunks) don't re-contextualize and re-embed all 580 chunks when only 5 paragraphs changed.

## Context & Dependencies

### The Problem

When `sync-sfs-updates` applies an amendment to a base law, it updates the `LegalDocument`'s content fields. The `process-chunks` cron then detects `ld.updated_at > cc.max_chunk_created` and calls `syncDocumentChunks()`, which today:

1. **Deletes ALL existing chunks** for the document, then **creates all new chunks** in one transaction ([`lib/chunks/sync-document-chunks.ts:106-113`](../../lib/chunks/sync-document-chunks.ts))
2. **Re-generates context prefixes** for every chunk via one batched Claude Haiku call (`generateContextPrefixes`, [`sync-document-chunks.ts:163`](../../lib/chunks/sync-document-chunks.ts))
3. **Re-embeds every chunk** via OpenAI in batches of 100 (`generateEmbeddingsBatch`, [`sync-document-chunks.ts:186-207`](../../lib/chunks/sync-document-chunks.ts))
4. **Writes every embedding** with a per-chunk raw-SQL `UPDATE content_chunks SET embedding = …::vector`

So a single-paragraph amendment to Miljöbalken (~580 chunks) re-contextualizes 580 chunks (full-document Haiku tokens + 580 prefix outputs), re-embeds 580 chunks (~6 OpenAI batch calls), and runs ~580 + ~580 DB writes. It also blanks the document mid-sync: the delete-all runs before the create-all, so the document is briefly unsearchable.

> **Cost-framing correction (verified against code):** prefix generation is **one** Haiku call per document (batched), and embeddings are `ceil(N/100)` OpenAI batch calls — **not** "580 + 580 API calls." The real cost driver is **tokens processed and DB writes**, both of which scale with the number of chunks touched. The win below is "process 5 chunks instead of 580," not "5 calls instead of 580 calls."

### Existing System Integration (all verified against current code)

- **`lib/chunks/sync-document-chunks.ts`** — `syncDocumentChunks(documentId: string): Promise<SyncResult>`. Does atomic delete-all + create-all (`lines 106-113`), then calls private `generateEmbeddingsForDocument()` (`lines 147-210`) which generates prefixes + embeddings for **all** chunks of the document.
- **`lib/chunks/chunk-document.ts`** — `chunkDocument(input): ChunkInput[]`. Each `ChunkInput` has `source_type: 'LEGAL_DOCUMENT'`, `source_id`, `workspace_id: null`, `path` (e.g. `"kap.2/§3"`), `contextual_header`, `content`, `content_role`, `token_count`, `metadata`.
- **`lib/chunks/generate-context-prefixes.ts`** — `generateContextPrefixes(document, chunks): Promise<Map<string, string>>`. ONE Haiku call (`HAIKU_MODEL = 'claude-haiku-4-5-20251001'`) for the whole document; returns a `Map` keyed by chunk **`path`**.
- **`lib/chunks/embed-chunks.ts`** — `generateEmbeddingsBatch(items): Promise<BatchEmbeddingResult>` (OpenAI `text-embedding-3-small`), plus `vectorToString(embedding)`. Embedding input is `{ text: content, contextPrefix, contextualHeader }`.
- **`app/api/cron/process-chunks/route.ts`** — finds stale docs via `ld.updated_at > cc.max_chunk_created` (`line ~98`), calls `syncDocumentChunks(doc.id)` (`line ~130`), aggregates a `ProcessStats` object (`lines 28-37`).
- **`content_chunks` table** (`prisma/schema.prisma` model `ContentChunk`, `lines ~1923-1942`) — columns: `id`, `source_type`, `source_id`, `workspace_id`, `path`, `contextual_header`, `content` (`@db.Text`), `content_role`, **`context_prefix`** (`@db.Text`, nullable), `embedding` (`Unsupported("vector(1536)")`), `token_count`, `metadata`, `created_at`, `updated_at`. **There is no `content_hash` column today** (one exists on `WorkspaceFile`, but not on `ContentChunk`).

### The Solution

Add a `content_hash` per chunk. On re-sync (the new default path):

1. Chunk the document as before (`chunkDocument()` → new chunk list).
2. Load existing chunks for the document; build a map `path → { id, content_hash }`.
3. For each new chunk, compute its hash and compare against the existing chunk at the same `path`:
   - **Unchanged** (path exists, hash matches) — leave the row untouched (no Haiku, no embedding, no DB write). Bulk of chunks.
   - **Changed** (path exists, hash differs) — replace: delete old row, create new row, then generate prefix + embedding for it.
   - **New** (path not in map) — create row, then generate prefix + embedding.
   - **Removed** (existing path not in the new list) — delete row.
4. Only the changed/new chunks are passed to `generateContextPrefixes` (full document markdown still supplied as context) and to `generateEmbeddingsBatch`.

This turns a 580-chunk full rebuild into a 5–10-chunk incremental update for a typical amendment, and the document stays searchable (the ~570 unchanged rows are never deleted).

### Decisions & Known Limitations (read before implementing)

1. **Hash input = `content` + `contextual_header`, not `content` alone.** The embedding is generated from `content`, `context_prefix`, and `contextual_header` ([`sync-document-chunks.ts:188-192`](../../lib/chunks/sync-document-chunks.ts)). `contextual_header` is deterministic from document structure (chapter/section titles). Hashing both means a structural rename that changes a header re-embeds the affected chunks; hashing `content` only would leave a stale embedding. Do **not** include `context_prefix` in the hash — it is LLM-derived and would never be stable.
2. **Unchanged chunks keep their old `context_prefix` even if a neighbor changed.** A prefix situates a chunk within the surrounding document, so an amendment to a neighboring paragraph *could* in principle make an unchanged chunk's prefix slightly stale. We accept this for v1: the unchanged paragraph's own text is byte-identical, and re-contextualizing the whole document would defeat the purpose. `force: true` regenerates everything when a full refresh is wanted.
3. **`path` is assumed unique within a document.** The existing prefix map is already keyed by `path` (`prefixes.get(chunk.path)`), so this invariant exists today. The incremental map relies on it too. Add a defensive check (Task 2) so a duplicate-path document fails loudly rather than silently mis-syncing.
4. **Migration is applied manually.** Per project practice, the dev creates the Prisma migration file but does **not** run it; hand the apply command to the operator.

## Acceptance Criteria

1. `ContentChunk` gets a new `content_hash String?` column (nullable; SHA-256 hex). A shared utility computes the hash over `content + "\n" + contextual_header` (documented as the canonical hash input).
2. Existing chunks are backfilled with `content_hash` via a one-off script (compute from each row's current `content` + `contextual_header`).
3. `syncDocumentChunks()` (default, non-`force` path) loads existing chunks and builds a `path → { id, content_hash }` map for the document being synced.
4. Chunks with matching `path` **and** matching `content_hash` are skipped entirely — no Haiku call, no embedding call, no DB write.
5. Chunks with matching `path` but **different** `content_hash` are replaced (old row deleted, new row created) and get a fresh prefix + embedding.
6. New chunks (`path` not in the map) are created and get a prefix + embedding.
7. Existing chunks whose `path` is no longer in the new chunk list are deleted.
8. The document remains searchable during an incremental sync — there is **no** delete-all step; unchanged rows are never touched. The targeted delete + create run in a single `prisma.$transaction`.
9. A full re-chunk can be forced via `syncDocumentChunks(documentId, { force: true })`, which preserves today's exact behavior (delete-all + create-all + embed-all). The signature change is additive — the second argument is optional.
10. `SyncResult` and `process-chunks`'s `ProcessStats` gain `chunksSkipped` and `chunksUpdated` (alongside existing `chunksCreated`, `chunksDeleted`, `chunksEmbedded`); the cron logs/aggregates them.
11. The `syncDocumentChunks()` contract for existing callers is unchanged — `syncDocumentChunks(id)` with no options still works and existing callers need no modification.
12. New rows always have `content_hash` populated on write (incremental and `force` paths both set it). All existing tests pass; `pnpm build`, `pnpm tsc`, and `pnpm lint` are clean.

### Content Hash Utility

```typescript
// lib/chunks/content-hash.ts
import { createHash } from 'crypto'

/**
 * Canonical chunk content hash. Covers BOTH inputs the dev controls
 * deterministically for the embedding (content + contextual_header);
 * intentionally excludes the LLM-derived context_prefix.
 */
export function contentHash(content: string, contextualHeader: string): string {
  return createHash('sha256').update(`${content}\n${contextualHeader}`).digest('hex')
}
```

### Cost Estimate (corrected)

A typical amendment changes 3–10 paragraphs. For Miljöbalken (~580 chunks):
- **Before (every sync):** 1 Haiku call sized for all 580 chunks' prefixes + ~6 OpenAI embedding batches (580 items) + ~580 prefix `UPDATE`s + ~580 embedding `UPDATE`s.
- **After (incremental):** 1 Haiku call sized for ~5–10 prefixes (full doc still supplied as context) + 1 OpenAI batch (~5–10 items) + ~5–10 prefix `UPDATE`s + ~5–10 embedding `UPDATE`s; ~570 rows untouched.
- **Net:** roughly an order-of-magnitude reduction in Haiku output tokens, embedding tokens, and DB writes per amendment to a large law — and no mid-sync searchability gap. `force: true` restores the full path when needed.
