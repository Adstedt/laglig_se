# Story 14.7c: Write / Action Tools

## Status

Draft

## Story

**As a** compliance partner agent,
**I want** tools to create tasks, update compliance status, save assessments, and add context notes on behalf of the user,
**so that** I can take actions that help the user manage their compliance work вЂ” always with user confirmation before execution.

## Context & Dependencies

This is the third of three stories split from the original 14.7 (Agent Tool Definitions). It adds four write tools that change state in the application. These are the highest-risk tools вЂ” they follow a strict **proposal-before-execution** pattern where the agent first presents what it intends to do and only executes after user confirmation.

**Key existing infrastructure:**

- **Tool infrastructure** (Story 14.7a): Shared types (`WriteToolResponse`, `ToolError`, `ToolMeta`), utility helpers (`wrapWriteToolResponse`, `wrapToolError`), factory function pattern, chat route already wired. This story adds tools to the existing registry.
- **Existing server actions**: Task CRUD (`createTask()` in `app/actions/tasks.ts`) вЂ” `create_task` wraps this directly. [Source: app/actions/tasks.ts]
- **LawListItem model**: `compliance_status`, `business_context`, `compliance_actions` fields вЂ” `update_compliance_status` and `add_context_note` write to these directly via Prisma. [Source: prisma/schema.prisma]

**Depends on:** 14.7a (tool infrastructure, types, factory pattern, chat route wiring)

**Deferred models:** `ChangeAssessment` (Story 14.10 вЂ” not yet built). The `save_assessment` tool defines its full parameter schema now so the interface is ready, but returns a stub response noting the model doesn't exist yet. It will be fully implemented when 14.10 creates the model.

**Depended on by:** 14.9 (system prompt references write tool confirmation behavior), 14.10 (assessment flow calls `save_assessment` and `update_compliance_status`), 14.11 (chat UI surfaces confirmation prompts)

**Design principle (Anthropic guide):** Human-in-the-loop for all state-changing actions. Read-only operations run freely; write operations explain what will happen, wait for confirmation, then execute.

## Acceptance Criteria

### Write Tools

1. `create_task` вЂ” create a workspace task linked to a law. Parameters: `title` (string), `description` (string, optional), `relatedDocumentId` (optional), `priority` (optional: `LOW`|`MEDIUM`|`HIGH`|`CRITICAL`), `execute` (boolean, default false). Wraps existing `createTask()` from `app/actions/tasks.ts`. When `execute` is false (or omitted), returns action proposal with `confirmation_required: true`. When `execute` is true, performs the action and returns the result.
2. `update_compliance_status` вЂ” update a LawListItem's compliance status. Parameters: `lawListItemId` (string), `newStatus` (enum: `EJ_PABORJAD`|`PAGAENDE`|`UPPFYLLD`|`EJ_UPPFYLLD`|`EJ_TILLAMPLIG`), `reason` (string), `execute` (boolean, default false). When `execute` is false, returns action proposal with `confirmation_required: true` including a preview showing old в†’ new status. When `execute` is true, performs the update.
3. `save_assessment` вЂ” persist a change assessment. Parameters: `changeEventId` (string), `lawListItemId` (string), `impactLevel` (`HIGH`|`MEDIUM`|`LOW`|`NONE`), `analysis` (string), `recommendations` (string), `execute` (boolean, default false). When `execute` is false, returns action proposal. (Note: `ChangeAssessment` model deferred вЂ” this tool is a stub until Story 14.10. Returns `{ confirmation_required: false, _note: "ChangeAssessment-modellen skapas i Story 14.10", _meta: ... }`)
4. `add_context_note` вЂ” add a note explaining why a law matters to this company. Parameters: `lawListItemId` (string), `note` (string), `execute` (boolean, default false). Updates `LawListItem.business_context` field. When `execute` is false, returns action proposal with `confirmation_required: true` showing the note text. When `execute` is true, performs the update.

### Confirmation Pattern

5. All write tools implement the `execute` flag pattern: `execute: false` (default) returns a structured proposal (`{ confirmation_required: true, action, params, preview, _meta }`), `execute: true` performs the action
6. Proposal preview text is in Swedish (e.g., `"Skapa uppgift: 'Uppdatera kemikaliefГ¶rteckning' med prioritet HГ¶g"`)
7. The LLM presents the proposal to the user naturally and calls the tool again with `execute: true` only after the user confirms

### Quality Standards (inherited from 14.7a)

8. Tool descriptions written as comprehensive docstrings вЂ” purpose, when to use, what it returns, confirmation behavior explained
9. Response shapes follow `WriteToolResponse<T>` with `_meta: { tool, executionTimeMs, resultCount }` pattern
10. Error responses include specific, actionable guidance in Swedish (e.g., validate `lawListItemId` exists before proposing action, return clear error if not found)
11. Tools registered in the `createAgentTools` factory function вЂ” added to the existing tool registry in `lib/agent/tools/index.ts`
12. At least 6 unit tests covering: parameter validation, confirmation flag on first call, successful execution on `execute: true`, error on invalid ID, preview text format, stub behavior for `save_assessment`

## Dev Notes

### Write Tool Confirmation Pattern (Critical)

Write tools do NOT execute on first call. They return a proposal:

```typescript
// Tool returns this to the LLM:
{
  confirmation_required: true,
  action: 'create_task',
  params: { title: 'Uppdatera kemikaliefГ¶rteckning', priority: 'HIGH' },
  preview: 'Skapa uppgift: "Uppdatera kemikaliefГ¶rteckning" med prioritet HГ¶g',
  _meta: { tool: 'create_task', executionTimeMs: 5, resultCount: 0 }
}
```

The LLM presents this to the user: "Jag fГ¶reslГҐr att vi skapar uppgiften 'Uppdatera kemikaliefГ¶rteckning' med hГ¶g prioritet. Ska jag gГ¶ra det?"

When the user confirms, the LLM calls the tool again with `execute: true`, and the tool runs the actual server action:

```typescript
execute: async ({ title, description, priority, execute }) => {
  const startTime = performance.now()
  if (!execute) {
    return wrapWriteToolResponse('create_task', 'create_task', { title, priority },
      `Skapa uppgift: "${title}" med prioritet ${priority}`, startTime)
  }
  // Execute the action
  const result = await createTask({ title, description, priority, workspaceId, columnId })
  return wrapToolResponse('create_task', { taskId: result.id, title }, startTime)
}
```

### Workspace Scoping for Write Tools

All write tools must validate that the target entity belongs to the current workspace before proposing or executing an action. For example:

```typescript
// Validate LawListItem belongs to workspace
const item = await prisma.lawListItem.findFirst({
  where: {
    id: lawListItemId,
    law_list: { workspace_id: workspaceId },
  },
  include: { law_list: true, document: { select: { title: true, document_number: true } } },
})
if (!item) {
  return wrapToolError('update_compliance_status',
    'Laglistposten hittades inte',
    'Kontrollera att ID:t Г¤r korrekt och att posten tillhГ¶r den aktiva arbetsytan.',
    startTime)
}
```
