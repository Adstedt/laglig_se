# Story 17.10: Agent Tools — Search, Read & List Workspace Documents (Styrdokument)

## Status

**Approved** — re-validation refresh 2026-05-22 (Sarah, PO). Was a stale Draft (2026-03-30, written before [[17.9b]]/[[17.9c]]). Cites verified against current code; DEC-1 (keep search separate) + DEC-2 (cite by title) resolved. validate-story-draft **GO**. Ready for SM/dev pickup.

> **Why now:** Story 17.9b (Done) indexes authored styrdokument as `WORKSPACE_DOCUMENT` chunks — but ships **no consumer**. This story adds the agent's read side so it can search/read/list a workspace's policies & rutiner, mirroring what [[17.9c]] did for uploaded files (`search_workspace_files`). **This story keeps styrdokument search SEPARATE from file search** (DEC-1) — distinct result shapes and citation labels.

## Story

**As an** AI compliance agent,
**I want** to search, read, and list a workspace's authored styrdokument,
**so that** I can give advice grounded in the organization's actual policies, procedures, and risk assessments — never another tenant's.

## Story Context

**Existing System Integration:**

- **RAG (Story 17.9b, Done):** authored styrdokument are chunked + embedded as `WORKSPACE_DOCUMENT` (workspace-scoped). `retrieveContext(query, workspaceId, { sourceTypes: ['WORKSPACE_DOCUMENT'], topK })` (`lib/agent/retrieval.ts:25`, `= ANY(...::"SourceType"[])` at `:172`). Chunk `metadata` carries `{ title, document_type, status, content_hash }`; `contextual_header` = `"<title> (<TYPE>)"`.
- **Parallel precedent (Story 17.9c):** `createSearchWorkspaceFilesTool` (`lib/agent/tools/search-workspace-files.ts`) — `USER_FILE` search, registered at `lib/agent/tools/index.ts:72`. **DEC-1: this story does NOT touch or merge with it** — styrdokument get their own tool because the result shapes (files: filename+category; styrdokument: title+type+status) and citation labels differ.
- **Models (anti-hallucination — there is NO `Document` model):** `WorkspaceDocument` (`schema.prisma:1948`, `prisma.workspaceDocument`) + `WorkspaceDocumentVersion` (`:1993`). Enums: **`WorkspaceDocumentType` (`:1923`)**, **`WorkspaceDocumentStatus` (`:1934`)**. ⚠️ Do NOT use `DocumentType` (doesn't exist) or `DocumentStatus` (`:604` — exists but is the **SFS/LegalDocument** enum, wrong namespace; it compiles, so the mistake is silent).
- **Tool pattern:** mirror `search-laws.ts`, `search-workspace-files.ts` (search), `get-document-details.ts` (read). Same `tool()` shape, `inputSchema: zodSchema(...)`, `z` from `'zod/v4'`, `wrapToolResponse`/`wrapToolError`/`truncateMarkdown` from `./utils`, `startTime` telemetry.
- **Registry:** `createAgentTools(workspaceId, userId, context?)` (`lib/agent/tools/index.ts:62`) currently registers **15** tools. Read-only tools take **`workspaceId` only** (no `userId`/`writeContext` — that's for write tools).
- **Citations:** `extractSourcesFromToolResult` lives in **`lib/ai/citations.ts:54`** (NOT `app/api/chat/route.ts` — the route only *calls* it at `:410` and `:597`). Add a `search_workspace_documents` branch alongside the `search_workspace_files` branch (`citations.ts:179`).
- **Content source:** read the current version's **`content_html`** and convert via `htmlToMarkdown` (`lib/transforms/html-to-markdown.ts:43`). Do **NOT** use `WorkspaceDocumentVersion.extracted_text` (`schema.prisma:2002`) — it's tag-stripped flat plaintext (`extractPlaintext`, `documents.ts:41`), no headings.
- **System prompt:** `lib/agent/system-prompt.md`. ⚠️ Token budget is **already >4000** (the `system-prompt.test.ts` guardrail, flagged by 17.9c) — keep the three tools' guidance terse.

**Out of scope:** any change to `search_workspace_files` / `search_laws`; merging file + styrdokument search (DEC-1 = separate); write tools (covered by Epic 14 / Story 17.11).

## Acceptance Criteria

### `search_workspace_documents` tool

1. New tool `search_workspace_documents` in `lib/agent/tools/search-workspace-documents.ts` via a `createSearchWorkspaceDocumentsTool(workspaceId)` factory (closure-captured `workspaceId`, mirroring `search-workspace-files.ts`).
2. Input: `query` (string, required, Swedish), `limit` (number, 1–10, default 5). Optional `document_type` (`WorkspaceDocumentType`) + `status` (`WorkspaceDocumentStatus`) filters are **nice-to-have post-filters** on the returned chunks' `metadata` (retrieval itself filters only by `sourceTypes` + workspace) — include only if clean; otherwise defer.
3. Calls `retrieveContext(query, workspaceId, { sourceTypes: ['WORKSPACE_DOCUMENT'], topK: limit })`. **Never** widens to `USER_FILE` or `LEGAL_DOCUMENT`.
4. Returns per result: `documentId` (= `sourceId`), `title` + `documentType` + `status` (from chunk `metadata`), `snippet` (= chunk content), `relevanceScore` (rounded), `citationKey` (= `title`). **Note:** `version_number` is **not** in chunk metadata → not returned by search; `get_workspace_document` provides it.
5. Empty-result path returns a `wrapToolError` with a Swedish "inga resultat" message + rephrase hint (mirror `search-workspace-files.ts`).
6. Cap at **10** results (enforce via `limit` max).

### `get_workspace_document` tool

7. New tool `get_workspace_document` in `lib/agent/tools/get-workspace-document.ts` via `createGetWorkspaceDocumentTool(workspaceId)`.
8. Input: `document_id` (string, required).
9. Returns: `title`, `document_type`, `status`, `current_version_number`, `document_number`, **`content` = `htmlToMarkdown(current_version.content_html)`** (NOT `extracted_text`), `review_date`, `approved_by`, `approved_at`, linked tasks (id + title via `task_links`), linked law-list items (id + law title/number via `list_item_links`).
10. Content truncated to first 20,000 chars via `truncateMarkdown` (`./utils`) with a Swedish note: `"[Trunkerat — dokumentet har {N} tecken totalt]"`.
11. **Workspace-scoped:** `findFirst({ where: { id, workspace_id } })`; a doc from another workspace → `wrapToolError` (not found). The closure `workspaceId` is the only tenant source.

### `list_workspace_documents` tool

12. New tool `list_workspace_documents` in `lib/agent/tools/list-workspace-documents.ts` via `createListWorkspaceDocumentsTool(workspaceId)`.
13. Input: optional `document_type` (`WorkspaceDocumentType`), `status` (`WorkspaceDocumentStatus`), `linked_list_item_id`, `linked_task_id`. Type/status are direct Prisma `where` clauses; the two `linked_*` filters go through the `list_item_links` / `task_links` relations (**confirm the join-table FK field names against `schema.prisma` — do not assume**).
14. Returns: `title`, `document_type`, `status`, `version_count` (`_count: { versions: true }`), `last_updated` (`updated_at`), `created_by` name (via `creator` relation).
15. Max **25** results, ordered by `updated_at desc`.
16. Workspace-scoped (`where: { workspace_id }`).

### Registration & prompt

17. All three tools registered in `createAgentTools()` (`index.ts:62`) alongside the existing **15** tools, each constructed with **`workspaceId` only** (read-only). Existing tools unchanged.
18. System prompt (`system-prompt.md`) updated **tersely** (token budget >4000): when to prefer `search_workspace_documents` (authored policies/rutiner/risk assessments) vs `search_workspace_files` (uploaded files) vs `search_laws`; `get_workspace_document` to read full content; `list_workspace_documents` to enumerate docs for a law/task. Cite styrdokument as `[Källa: <titel>]`.

Import any new icon (e.g. `Files`) — **verify it's exported by `lucide-react`** before use, else reuse `FileText`.

### Citations

20. `extractSourcesFromToolResult` (`citations.ts:54`) gains a `search_workspace_documents` branch (alongside `search_workspace_files` at `:179`) that maps each result → a `SourceInfo` keyed by its `citationKey`. **DEC-2: citation key = the document `title`** (human-readable, consistent with `[Källa: <filename>]` for files). Carry the title in `SourceInfo.documentNumber` (keeps it a non-optional `string` — same low-risk approach 17.9c used for files). **No `app/api/chat/route.ts` change** (it already calls the extractor).
21. **Collision disambiguation (QA CITE-002):** title-as-key is first-wins via `set()`; same-title styrdokument collide. Carry `documentId` in the result so resolution is unambiguous, and append a short id suffix to the `citationKey`/display only when a title collision is detected within the result set.

## Dev Notes

- **Anti-hallucination:** `WorkspaceDocument`/`prisma.workspaceDocument` (`schema:1948`), `WorkspaceDocumentVersion` (`:1993`), `WorkspaceDocumentType` (`:1923`), `WorkspaceDocumentStatus` (`:1934`). No `Document` model; do not use `DocumentType`/`DocumentStatus` (the latter exists at `:604` but is SFS-namespace).
- **Tool idioms (from `search-workspace-files.ts`):** `tool({ description, inputSchema: zodSchema(schema), execute })`; `z` from `'zod/v4'`; `wrapToolResponse`/`wrapToolError`/`truncateMarkdown` from `'./utils'`; closure-captured `workspaceId`; `startTime = Date.now()` telemetry.
- **Why `content_html`→markdown for the read tool:** consistency with 17.9b's indexing (it chunks from `htmlToMarkdown(content_html)`), and `extracted_text` is lossy plaintext. Structured markdown gives the agent headings → better comprehension + citation context.
- **Two genuinely different content shapes (don't merge):** files = filename + `FileCategory`; styrdokument = title + `WorkspaceDocumentType` + `WorkspaceDocumentStatus` + version. DEC-1 keeps the search tools separate; the system prompt disambiguates when to use which.
- **Citation pipeline:** `extractSourcesFromToolResult` (`citations.ts:54`) → consumed by `resolveSource`. `SourceInfo.documentNumber` is non-optional `string` (`citations.ts:16`); carry the title there (mirrors the files branch carrying filename). A bare title label fails the legal-citation regexes and resolves via the title-keyed `sourceMap` entry.
- **Search can't return version_number** (not in chunk metadata) — only `get`/`list` can. Don't promise it in search results.
