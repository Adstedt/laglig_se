# Story 17.15: Unified Content Library

## Status

Draft

## Story

**As a** workspace member,
**I want** one place to find all my compliance content — both documents I've authored and files I've uploaded,
**so that** I don't have to think about which section something belongs to when I need to find, link, or manage my materials.

## Story Context

**Existing System Integration:**

- Replaces: Two separate sidebar items ("Dokument" → `/workspace/documents` and "Mina filer" → `/documents`) with one unified "Innehåll" item
- Merges: `DocumentBrowserPage` (flat table, status/type filters, versioning) and `DocumentsBrowser` (folder tree, file cards, upload/preview)
- Technology: React, shadcn/ui, Next.js App Router, Prisma server actions
- Preserves: All existing data models (`WorkspaceDocument`, `WorkspaceFile`, folder hierarchy) — this is a UI-only merge, no schema changes
- Touch points: Left sidebar, workspace routing, document/file server actions

**Motivation:**

Users currently face a split between "Dokument" (authored compliance documents with versioning and lifecycle) and "Mina filer" (uploaded files organized in folders). From a compliance perspective, both are "material that supports our obligations." A user looking for their risk assessment PDF doesn't know if it was uploaded or authored in-app — they just want to find it. The current split forces users to understand our internal data architecture.

## Acceptance Criteria

**Unified Navigation:**

1. Single sidebar item "Innehåll" (icon: `Library` or `FolderOpen`) replaces both "Dokument" and "Mina filer" under the "Arbete" section
2. Route: `/workspace/content` — both `/workspace/documents` and `/documents` redirect here (preserve bookmarks)
3. Existing document editor route `/workspace/documents/[documentId]/edit` unchanged

**Unified Table View:**

4. One table showing both authored documents and uploaded files, sorted by last updated (default)
5. Each row shows: icon (document vs file type), title/filename, content type label, source badge ("Skapat" / "Uppladdat"), status (for documents) or category (for files), last updated, creator/uploader
6. Clicking a document row → navigates to document editor (`/workspace/documents/{id}/edit`)
7. Clicking a file row → opens file preview panel (same as current "Mina filer" behavior)

**Unified Actions:**

8. "Skapa dokument" button → opens existing `CreateDocumentDialog`
9. "Ladda upp fil" button → opens existing file upload flow
10. "Importera .docx" button → opens existing `ImportDocumentDialog`
11. Row actions dropdown adapts per item type: documents get (Öppna, Exportera Word, Exportera PDF, Arkivera); files get (Ladda ner, Kategorisera, Ta bort)

**Filtering & Search:**

12. Combined search across document titles and filenames
13. "Källa" (Source) filter: "Alla" | "Skapade dokument" | "Uppladdade filer"
14. "Typ" filter: shows document types (Policy, Riskbedömning, etc.) when source is documents; shows file categories (Bevis, Avtal, etc.) when source is files; shows both when "Alla"
15. "Status" filter: only visible/active when source includes documents (DRAFT, IN_REVIEW, APPROVED, etc.)
16. Clear all filters button

**Sorting:**

17. Sortable columns: title, last updated, created date
18. Default sort: last updated (descending)

**Pagination:**

19. Cursor-based pagination with "Ladda fler" button, 25 items per page
20. Consistent pagination across both content types

**Folder Access (Preserved):**

21. Optional "Mappar" toggle/view that shows the file folder tree for users who organize files in folders — appears as a collapsible sidebar panel on the content page
22. Selecting a folder filters the table to show only files in that folder (documents always visible unless filtered out by source)

**Integration Requirements:**

23. All existing document linking (Story 17.12) works unchanged — linked documents/files open correctly from task modal and legal document modal
24. All existing document management features (versioning, lifecycle, export) accessible from the unified view
25. All existing file management features (upload, categorize, preview, folder organization) accessible from the unified view

**Quality Requirements:**

26. Page loads within 1 second with up to 200 documents + 200 files
27. No data model or schema changes — pure frontend consolidation

## Dev Notes

**Unified content item type:**

```typescript
type ContentItem =
  | {
      kind: 'document'
      id: string
      title: string
      documentType: string
      status: string
      versionNumber: number
      createdBy: string
      creatorName: string
      updatedAt: Date
      createdAt: Date
      reviewDate: Date | null
      documentNumber: string | null
    }
  | {
      kind: 'file'
      id: string
      title: string // filename
      mimeType: string
      fileSize: number
      category: string
      folderId: string | null
      createdBy: string
      creatorName: string
      updatedAt: Date
      createdAt: Date
    }
```

**Merging query results:**

The unified query needs to fetch from two tables and merge. Approach:
- Fetch documents and files in parallel (`Promise.all`)
- Map each to `ContentItem` discriminated union
- Sort the merged array by the requested column
- Apply cursor-based pagination on the merged result

Note: True cursor-based pagination across two tables is complex. Pragmatic approach: fetch `take` items from each table, merge, sort, slice to `take`. This over-fetches slightly but is simple and correct for the expected data volume (< 500 items per workspace).
