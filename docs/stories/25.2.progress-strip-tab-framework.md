# Story 25.2: Progress strip + tab framework (B.2 shell)

## Status

Done

## Story

**As a** new user who just clicked "Generera ny lista" (or in B.4+, "Importera"),
**I want** the first-run modal to stay open with a live progress strip at the top + a tab bar below where I can read tutorial content while my list is being built,
**so that** the 2–5 minute LLM wait stops feeling like a dead spinner and turns into a productive onboarding moment — and I can minimise the modal whenever I'm done reading without cancelling the background job.

This is the **B.2 wedge** of Epic 25: it ships the **TutorialStep shell** (progress strip + tab bar + empty tab-body container) without any tab content. Tab content is B.3. Done states are B.4. The FAB + tutorial-only mode is B.6. B.2's sole job is to land the chrome and the data wiring so B.3/B.4/B.6 can plug into a known surface.

Concretely: B.2 adds a new `<TutorialStep>` mode that the modal transitions into after a successful Generera click (and, later in B.4, after a successful Importera commit). The step renders (a) a `<ProgressStrip>` driven by the same SWR key as the dashboard's existing `<LawListGenerationProgress>` banner (single source of truth — confirmed in arch §4 and §6.3); (b) a tab bar with six placeholder tabs (no content, just labels + active-tab underline + "1 av 6" counter + horizontal scroll on narrow viewports); (c) an empty tab-body container; (d) a "Minimera" affordance that calls the existing `minimiseFirstRunModal()` action and closes the modal — the dashboard banner takes over after close (same handoff B.1 already wired via `router.refresh() + router.push('/dashboard')`).

This unlocks the "tutorial-while-waiting" UX the architecture has been pointing at since the brief. The user finally sees something happen during generation instead of staring at a spinner.

## Acceptance Criteria

### State machine + mounting

1. **`<FirstRunModal>` gains a fourth `step` value: `'tutorial'`.** The existing union type `Step = 'path-choice' | 'template-pick' | 'import-upload'` extends to `Step = 'path-choice' | 'template-pick' | 'import-upload' | 'tutorial'`. Adding a new value is a payload-shape extension only — no schema change, no migration.
2. **Generera success transitions the modal to `step='tutorial'` instead of closing it.** Currently in B.1, `handleGenerate` in `path-choice-step.tsx` calls `runDismissAction(minimiseFirstRunModal)` → `onClose()` → `router.refresh() + router.push('/dashboard')` (closes the modal and lets the dashboard banner take over). B.2 changes this: on `res.ok` (200) the modal transitions to the new `'tutorial'` step instead. On 409 (already-in-progress soft success) it also transitions to `'tutorial'` (the user lands in the right place regardless of whether they kicked off the run or just observed an existing one). On error: existing behavior preserved (toast + modal stays on path-choice). **`minimiseFirstRunModal()` is no longer called on Generera success in B.2** — it gets called later when the user hits Minimera in the tutorial step (AC 14) or when generation completes and done state ships in B.4.
3. **Importera success also transitions to `step='tutorial'` if the user wants to stay (B.2 deferral).** In B.1, `handleImportSuccess` in `first-run-modal.tsx` minimises + closes + routes to `/laglistor/skapa/{importId}/granska`. B.2 keeps this B.1 behaviour as-is — the import path does NOT transition to the tutorial step in B.2. (Rationale: import already has a substantive post-success surface — the review page. Adding tutorial-while-waiting there would conflict with the review flow. B.4's `done-import-step` will explicitly land the import path's "done" surface.) Document this scope decision in Completion Notes.
4. **The path-choice step's `data-onboarding-focus-target` moves from the Generera card to the new tutorial-step's first focusable element on transition.** When the user clicks Generera and the modal transitions to `'tutorial'`, the existing `useEffect` in `<FirstRunModal>` (the one that queries `[data-onboarding-focus-target="true"]` on step change, see `first-run-modal.tsx:90-106`) should find the active tab button. Default focus target: the first tab button (matches the AC 32 pattern from Story 25.1 where each step focuses its first interactive element).

### Progress strip — `<ProgressStrip>` component

7. **New file: `components/features/onboarding-modal/progress-strip.tsx`.** Client component. Mirrors the existing `<LawListGenerationProgress>` data wiring (SWR key `'/api/workspace/generation-status'`, fetcher returns `GenerationStatusResponse` — see `components/features/dashboard/law-list-generation-progress.tsx:42-58`) but renders a different visual: prototype frame ③ fidelity. **Same SWR key as the banner so both surfaces share one cache** (per arch §4: "The modal's progress strip reuses the same SWR key").
8. **`<ProgressStrip>` props:**
    - `initialStatus?: GenerationStatus | null` — same prop the banner takes; lets the modal pass server-derived initial status to avoid an empty-state flash.
    - No other props in B.2. Tab list + active tab are owned by the parent `<TutorialStep>`, not the strip.
9. **Visual structure (prototype frame ③ — see `_prototypes/onboarding-tutorial-modal.html:783-869`):**
    - **Top row:** `<Loader2 className="h-4 w-4 animate-spin" />` icon + "Vi skapar er personliga laglista" title in `font-safiro text-[15px]`
    - **Subhead:** "Du kan stänga rutan — vi fortsätter i bakgrunden och notifierar när det är klart." in `text-[12.5px] text-muted-foreground mb-3`
    - **Step trail (horizontal chevron-separated chips):** for each `step` in the SWR-fetched `progress` array, render a pill. Steps with `status='done'` get a `<Check className="h-3 w-3" />` + label in `text-foreground`. Step with `status='active'` gets a `bg-primary text-primary-foreground` rounded-md pill with a `h-1.5 w-1.5 rounded-full bg-primary-foreground` pulse-dot (animated, see AC 10). Steps with `status='pending'` get just the label in `text-muted-foreground`. Chevrons (`<ChevronRight className="h-3 w-3" />`) between every pair.
    - **Shimmer progress bar:** `h-1 w-full rounded-full` with an indeterminate-progress shimmer animation (see AC 11). Track is `bg-muted`; fill is a gradient that translates left-to-right continuously.
    - **Bottom row (font-mono):** `"Steg X av Y"` on left, `"N rader"` on right (Y = `progress.length`; N from the SWR response's `data?.itemCount`). See AC 12 for the count-source caveat.
10. **Pulse-dot animation:** add a `.pulse-dot` utility under the existing `@layer utilities` block in `app/globals.css` (next to `.font-safiro`). Animation: `@keyframes pulseDot { 0%, 100% { opacity: 1 } 50% { opacity: 0.4 } }`, `.pulse-dot { animation: pulseDot 1.4s ease-in-out infinite }`. Reuses the same pattern the prototype defines at `:213-228`. Apply only to the active step's dot.
11. **Shimmer animation:** add a `.shimmer-track` / `.shimmer-track .fill` utility under the same `@layer utilities`. Animation: `@keyframes shimmer { 0% { transform: translateX(-100%) } 100% { transform: translateX(200%) } }`, `.shimmer-track { @apply bg-muted overflow-hidden relative }`, `.shimmer-track .fill { @apply absolute inset-y-0 left-0 w-1/3 bg-gradient-to-r from-transparent via-foreground/20 to-transparent; animation: shimmer 2.4s ease-in-out infinite }`. Reuses the prototype pattern at `:184-211`. The shimmer is intentionally indeterminate (no progress percentage) — the actual progress is shown via the step-trail chips, not the bar.
12. **`rader` count comes from the SWR response's `data?.itemCount`** (where `data` IS the `GenerationStatusResponse` object, NOT nested under `.data` — verified against `components/features/dashboard/law-list-generation-progress.tsx:34-40` interface definition). **The API does NOT expose an "expected total rows" field**, so the prototype's `"N / M rader"` syntax (e.g., "47 / 120 rader") cannot be implemented in B.2 — the `/M` denominator has no data source. **B.2 scope decision:** render `"N rader"` only (no denominator) when `data?.itemCount` is a number; render nothing (omit the row entirely) when `data?.itemCount` is null/undefined. Do NOT block B.2 on adding a total-count field to the API — that's a Story 16.4 follow-up. The "Steg X av Y" left-side counter is independent of itemCount and renders from `progress.length`.
13. **`<ProgressStrip>` hides itself when status is non-progress.** If SWR returns `status === 'completed'` or `'failed'` or `null`, the strip returns `null`. The done state is B.4's concern (`done-generate-step.tsx` will own the celebration UI). For B.2: if generation finishes while the user is reading tutorial tabs, the strip just disappears — the modal stays open with the tabs intact until the user clicks Minimera. (B.4 will replace this disappear-quietly behavior with an explicit transition to `done_generate` mode.)

### Sub-step transitions back to path-choice

22. **No "← Tillbaka" affordance on the tutorial step.** Unlike template-pick + import-upload (which render a Tillbaka button per Story 25.1 AC 13-14), the tutorial step does NOT let the user go back to path-choice. Once a generation is in flight, going back would be misleading — the work is already committed. The only ways out are: (a) Minimera (modal closes, banner shows progress on dashboard, generation continues), or (b) waiting for completion (B.4 will land the auto-transition to `done_generate`). Document this design decision in Completion Notes — it's a deliberate UX choice that diverges from the path-choice sub-steps' reversible model.

### Visual / a11y

26. **The active tab is focusable + activatable via keyboard.** `<button>` element (not a `<div role="button">`), Tab key navigates between tabs, Enter/Space activates. Active-tab styling uses `focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2` so keyboard users can see focus position. The "Ny" chip on the AI-agenten tab is inside the button, so it's part of the focusable target.
27. **Tab bar is `role="tablist"`, tabs are `role="tab"`, tab body is `role="tabpanel"` with `aria-labelledby` pointing at the active tab's id.** Standard ARIA tabs pattern (https://www.w3.org/WAI/ARIA/apg/patterns/tabs/). The active tab has `aria-selected="true"` and `tabIndex={0}`; inactive tabs have `aria-selected="false"` and `tabIndex={-1}`. Roving tabindex (left/right arrows move focus between tabs without changing selection — that's an "ARIA-recommended" interaction). **B.2 minimum:** ARIA attributes correct, keyboard nav via Tab works; roving tabindex with arrow keys is nice-to-have, can defer to B.3 if it adds too much code.
28. **`<ProgressStrip>` uses `aria-live="polite"` on the wrapper** so screen reader users hear progress updates as steps complete. The shimmer bar gets `aria-hidden="true"` (purely decorative).
29. **The Minimera button has an accessible label "Minimera"** (visible text, no additional aria-label needed). The Minimize2 icon is `aria-hidden="true"`.

### Key decisions baked into this story

- **Tutorial step is NOT reversible.** No "← Tillbaka" affordance back to path-choice. Once generation is in flight, going back is misleading because the path is already committed. Only ways out: Minimera (close to banner) or wait for done state (B.4). This diverges from template-pick + import-upload which ARE reversible. See AC 22.
- **Import path doesn't get B.2 tutorial-step treatment.** Currently Importera success routes to `/laglistor/skapa/{importId}/granska` (the review surface). Adding a tutorial-while-waiting step there would conflict with the review flow. B.4 will land `done-import-step.tsx` as the import path's explicit done surface. See AC 3.
- **Tab body is a placeholder ("Innehåll kommer snart") in B.2.** Real content is B.3's job — 6 separate tab files under `tutorial-tabs/` with mini-previews of actual product surfaces. B.2 just needs the navigation chrome to land so B.3 has somewhere to plug content in. The `min-h-[280px]` on the body container prevents height shifts when B.3 lands. See AC 17.
- **Minimera lives at the bottom-right of the step body, NOT inside the progress strip.** Prototype puts it inside the strip (frame ③), but that couples Minimera to strip visibility (strip hides on completion per AC 13, but Minimera must still be reachable). Bottom-right is the cleanest decoupling. See AC 19.
- **`modal_dismissed` event still records `from_state: 'path_choice'` even when fired from the tutorial step.** Changing this requires a server-action signature change (accept a from_state param). Out of scope for B.2 — funnel analysts will need to know the limitation until B.6 reworks this. See AC 25.
- **Tab IDs are short kebab-case matching arch §6.1's `tab-{id}.tsx` filename convention** — so B.3's six tab files (`tab-laglista.tsx`, `tab-kravpunkter.tsx`, `tab-uppgifter.tsx`, `tab-kontroller.tsx`, `tab-lagandringar.tsx`, `tab-ai-agent.tsx`) map 1:1 to B.2's telemetry IDs (`'laglista'`, `'kravpunkter'`, etc.). Stable across B.2 → B.3 → B.6 means `tab_viewed` analytics have consistent values across the epic. See AC 23.
- **Roving tabindex (arrow keys to move between tabs) is nice-to-have, not required.** Standard ARIA tabs pattern recommends it but Tab key navigation + Enter activation covers the keyboard a11y minimum. B.2 ships the ARIA attributes correctly + Tab nav; arrow-key roving can land in B.3 if it doesn't add too much code. See AC 27.
- **No FAB in B.2.** The corner FAB ships in B.6. Until then, "Minimera" just means "close + dashboard banner takes over" — exactly what B.1's Generera handoff already does. The Minimera button visual signals to the user "you can dismiss this without losing the work" — even without the FAB, the dashboard banner provides the persistent progress indicator.

AC 6 forbids changing the modal shell's `px-8` layout. **Iterative polish baked in post-staging-review (Change Log v0.3 — details there):**

9. **`generate-law-list` route now returns 202 immediately via `after()`** (architectural — Story 16.4 territory but unblocked B.2). The route previously awaited the 2-5 min LLM job synchronously, which left the modal hanging on a stalled fetch. ### Refactoring Performed

- **File**: `components/features/onboarding-modal/tutorial-step.tsx`
  - **Change**: Added `ArrowLeft` / `ArrowRight` / `Home` / `End` keyboard handlers to the tab buttons via a `handleTabKeyDown` function; introduced a `tabRefs` record to delegate focus to the newly-selected tab after re-render; promoted the magic string `'laglista'` to a named `DEFAULT_TAB_ID` constant.
  - **Why**: The implementation used `tabIndex={isActive ? 0 : -1}` (proper ARIA tabs roving) **without** arrow-key handlers — meaning inactive tabs were unreachable via Tab key alone. This violated AC 26's explicit minimum ("Tab key navigates between tabs"). Keyboard-only users would have been stuck on the default tab `laglista` with no way to reach the other five.
  - **How**: Implements the W3C ARIA Tabs "automatic activation" pattern (https://www.w3.org/WAI/ARIA/apg/patterns/tabs/). Arrow keys move selection + focus; Home/End jump to first/last; selection wraps. This satisfies both AC 26 (minimum keyboard nav works) and AC 27's nice-to-have (proper roving with arrow keys) without deferring to B.3. Selection still calls `recordTabViewed` so telemetry is consistent across mouse, Tab, and arrow-key paths.

- **File**: `tests/unit/components/features/onboarding-modal/tutorial-step.test.tsx`
  - **Change**: Added 3 new tests: "ArrowRight moves selection to the next tab and fires tab_viewed", "ArrowLeft wraps from the first tab to the last", "End moves selection to the last tab; Home returns to the first".
  - **Why**: Locks in the QA refactor against regressions. Without these tests a future developer could re-introduce `tabIndex={-1}` without the arrow handlers.
  - **How**: Uses `fireEvent.keyDown` with `key: 'ArrowRight'` etc., then asserts on `aria-selected` + the last `recordTabViewed` call. Same mock pattern as the existing tests.

- **File**: `tests/unit/components/features/onboarding-modal/progress-strip.test.tsx`
  - **Change**: Added "renders exactly one .pulse-dot element on the active step".
  - **Why**: AC 35 explicitly calls out "Active step gets the pulse-dot" but the original test suite only asserted on labels — the pulse-dot rendering was untested. A future refactor could silently strip it.
  - **How**: `container.querySelectorAll('.pulse-dot')` with `length === 1` assertion (one active step in the mock SWR data → exactly one pulse-dot).
