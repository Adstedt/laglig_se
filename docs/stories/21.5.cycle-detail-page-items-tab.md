# Story 21.5: Cycle detail page — items list + inline bedömning editor

## Status

Done

## Story

**As a** responsible user assigned to cycle items (KMA-samordnare, lead auditor, or member acting on a cycle),
**I want** a list view of all `ComplianceAuditItem` rows in a cycle with inline `efterlevnadsbedomning` + `motivering` editors and per-row sign-off,
**so that** I can work through the cycle efficiently without opening a modal for each item — bedömning, justification, and sign-off all happen in-place, with progress visible in the header.

## Acceptance Criteria

### Functional

1. **Route + auth gate + tab shell.** A new Next.js 16 App Router page at `app/(workspace)/laglistor/kontroller/[cycleId]/page.tsx` (Server Component) gates access via the same OR-permission pattern used by `getCycleById` in Story 21.2 (`activity:view || tasks:edit`). Unauthorised users redirect to `/laglistor`; not-found cycles (including wrong-workspace) redirect to `/laglistor/kontroller` with the standard generic Swedish error toast `'Kontrollen hittades inte'`. The page renders a four-tab shell using the existing shadcn `Tabs` primitive (`components/ui/tabs.tsx`): **Items (default)**, **Findings**, **Rapport**, **Aktivitet**. Only the **Items** tab is implemented in this story — **Findings** renders a neutral placeholder `'Hanteras i Story 21.7'`, **Rapport** renders `'Hanteras i Story 21.11–21.12'`, **Aktivitet** renders `'Hanteras i Story 21.13'`. Tab selection is URL-hash based (`#items | #findings | #rapport | #aktivitet`) so deep-links into specific tabs work; default `#items` when no hash present.

2. **Detail header (above tabs).** Renders:
   - Title: cycle `name` (text-2xl, semibold).
   - Breadcrumb: `Laglistor / Kontroller / {cycle.name}` using the existing inline breadcrumb pattern from `/laglistor/kontroller/skapa` (Story 21.4 Task 4.4).
   - Metadata chips row (small muted text): `{lawList.name}` · `{auditType === 'INTERN' ? 'Intern revision' : 'Extern revision'}` · `{scheduledStart}–{scheduledEnd}` (date-fns Swedish locale `d MMM yyyy`) · lead auditor name (with avatar).
   - **Status badge** (AC 3).
   - **Progress cluster** (AC 7).
   - On the right: an **"Åtgärder"** dropdown (placeholder — disabled for now; later stories populate Complete / Seal / Archive).

3. **Cycle-status badge.** Inline badge next to the title. Swedish copy + Tailwind colour tokens (no new design system):
   - `PLANERAD` → `Planerad` (gray).
   - `PAGAENDE` → `Pågående` (blue).
   - `AVSLUTAD` → `Avslutad` (amber).
   - `SEALED` → `Förseglad` (emerald).
   - `ARKIVERAD` → `Arkiverad` (slate).
   Implement as a local `CycleStatusBadge` component under `components/features/compliance-audit/cycle-detail/cycle-status-badge.tsx`. Reuses Tailwind classes from the existing `COMPLIANCE_STATUS_OPTIONS` palette in `components/features/document-list/table-cell-editors/compliance-status-editor.tsx:26-64` (variant pattern already established in the codebase).

4. **Items tab — row set + read model.**

Read permission: same OR-gate (`activity:view || tasks:edit`). Tenant-isolated by joining through `cycle.workspace_id = ctx.workspaceId` (mirrors `getCycleById`'s check in `app/actions/compliance-audit-cycle.ts:383-395`).
   - The action returns rows sorted by `(law_list_item.group.position asc NULLS LAST, law_list_item.position asc, id asc)` — mirrors the canonical grouped ordering already used by `compliance-detail-table`. A minor-but-material detail for UX consistency.

5. **Items tab — table columns, virtualisation, and display rules.** Renders a table with these columns (left → right):
   1. **Lag** — `lawTitle` with `lawDocumentNumber` below it in muted text. Clicking the title opens the source law in a new tab (`target="_blank"`) to `/laglistor?law=${lawListItemId}` (the existing query-param pattern that opens the `LegalDocumentModal` — see `components/features/document-list/legal-document-modal/index.tsx` for the query-param contract).
   2. **Nuvarande status** — read-only `ComplianceStatusBadge` for the **source** `LawListItem.compliance_status` (purely informational — this is the state of the underlying laglista at read time, NOT the cycle's bedömning). Uses the exact colour map + Swedish labels from `COMPLIANCE_STATUS_OPTIONS` in `compliance-status-editor.tsx:26-64`. No click-to-edit here.
   3. **Bedömning** — inline `<ItemBedomningSelect>` editor (AC 6). Select with four variants: `UPPFYLLD` (green), `DELVIS` (blue), `EJ_UPPFYLLD` (red), `EJ_TILLAMPLIG` (gray, strikethrough). Nullable — the select also has an empty "—" option that clears the bedömning; the default unresolved state after materialisation for `PAGAENDE` / `EJ_PABORJAD` source items (see Story 21.4 AC 9 mapping).
   4. **Motivering** — `<ItemMotiveringEditor>` click-to-edit textarea (AC 6).
   5. **Ansvarig** — displays the `responsible_user` of the source `LawListItem` if present, else `—`. Read-only in this story (source field, not a cycle-item field). If the source user is no longer a workspace member, render `—`.
   6. **Signerad** — displays `{signedOffAt}` (date + time, Swedish locale) + `{signedOffBy.name}` below when signed, `—` otherwise. A per-row `<ItemSignOffButton>` (AC 6) renders in this column when the cycle is editable and `signedOffAt === null`.
   7. **Chevron** — row-expand toggle (AC 5 row expansion → drawer).
   - **Virtualisation.** When `rows.length > 100`, use `@tanstack/react-virtual`'s `useVirtualizer` with `estimateSize: 72`, `overscan: 5` — mirrors the exact pattern at `compliance-detail-table.tsx:103-112`. Below 100 rows, render without virtualisation. The container is scroll-bounded to `max-h-[calc(100vh-18rem)]` so the header + tabs stay pinned; rationale matches the existing table.
   - **Empty state.** If the cycle has zero items, render `'Kontrollen har inga poster.'` (paranoid — materialisation AC 7 of Story 21.4 rejects empty scope, but the defensive render keeps the UI resilient).

6. **Items tab — inline editors + sign-off.** New client components under `components/features/compliance-audit/item-bedomning-editor/`:
   - **`ItemBedomningSelect.tsx`** — inline `<Select>` (shadcn) bound to `efterlevnadsbedomning`. Saves **on change** (optimistic UI) via the new server action `updateItemBedomning(itemId, { efterlevnadsbedomning: EfterlevnadsBedomning | null })` in `app/actions/compliance-audit-item.ts` (AC 10). Shows a compact loading spinner (reuse pattern from `compliance-status-editor.tsx:77-84`) and a brief error toast on failure (use the existing toast helper — grep for `useToast` in the codebase; existing pattern in `kravpunkter-checklist.tsx`). Must be disabled when cycle is read-only (AC 8).
   - **`ItemMotiveringEditor.tsx`** — click-to-edit pattern: renders the current text truncated to 2 lines by default; clicking switches to a `<Textarea>` with autosize. Saves **on blur** OR after `2000ms` of idle (debounced via the already-installed `use-debounce` package's `useDebouncedCallback` — same as `compliance-detail-table.tsx:88`). Optimistic UI. `maxLength: 5000` with client + server validation. Disabled when cycle is read-only.
   - **`ItemSignOffButton.tsx`** — small primary button `'Signera'` (Swedish). Idempotent: calling `signOffItem(itemId)` on an already-signed item returns `{ success: true }` with a no-op (per AC 6 of epic). Confirmation dialog NOT required (sign-off is reversible pre-`AVSLUTAD`; Story 21.6 introduces the lifecycle lock). Disabled when cycle is read-only OR `signedOffAt !== null`. **Unsigning** is an **affordance-only** in this story: display an `<X>` icon-button with tooltip `'Ångra signering'` next to the signed-at timestamp. Clicking it calls `unsignOffItem(itemId)` (AC 10). This is allowed when cycle is `PLANERAD` (impossible — no items exist yet) or `PAGAENDE`; blocked when cycle is `AVSLUTAD` / `SEALED` / `ARKIVERAD` (handled by AC 8 read-only gate + server-side check). Unsigning in Story 21.6 will be restricted further (pre-complete only); the `AVSLUTAD` block is defence-in-depth handled by the existing gate here — 21.6 will narrow the UI affordance.

7. **Progress cluster — header.** Header renders two progress metrics (calculated client-side from the same `getCycleItemsForCycle` result; no extra fetch):
   - `'{bedomdaCount} av {total} bedömda'` — `bedomdaCount = items.filter(i => i.efterlevnadsbedomning !== null).length`.
   - `'{signeradeCount} av {total} signerade'` — `signeradeCount = items.filter(i => i.signedOffAt !== null).length`.
   Rendered as two stacked lines with muted-foreground label + prominent count. Each metric is a clickable button: **click-to-jump** scrolls to the first item that's either unbedömd (metric 1) or unsignerad (metric 2), with a brief highlight flash on the target row. Use `document.querySelector(\`[data-cycle-item-id="${firstId}"]\`)?.scrollIntoView({ behavior: 'smooth', block: 'center' })`; highlight via a CSS class applied for 1.5s. If every item is already bedömd / signerad, the button is disabled with tooltip `'Alla poster {bedömda|signerade}'`.

8. **Read-only mode for sealed / archived cycles.** When `cycle.status` is `SEALED` or `ARKIVERAD`, the **entire Items tab** renders with all editors disabled:
   - `<ItemBedomningSelect>` shows current value as a plain read-only badge (not a select).
   - `<ItemMotiveringEditor>` shows current text as a read-only paragraph (no click-to-edit).
   - `<ItemSignOffButton>` + unsign `X` are hidden (not merely disabled).
   - A banner above the table: `'Denna kontroll är förseglad ([truncated seal_hash]). Läsbehörighet endast.'` (SEALED) or `'Denna kontroll är arkiverad. Läsbehörighet endast.'` (ARKIVERAD). The banner is dismissable per-session (stored in `sessionStorage` under `laglig:cycle-readonly-banner-dismissed:${cycleId}`) — matches the kravpunkter legacy-banner precedent in memory.
   - A boolean prop `readOnly` threads from the page shell → `<CycleDetailPage>` → `<CycleItemsTab>` → each row component (same threading pattern as `complianceReadOnly` documented in `MEMORY.md`).
   - **Defence-in-depth.** The read-only gate is the UI-layer defence. Story 21.10 adds the server-layer `assertCycleEditable` guard; until then, the four new mutation server actions (`updateItemBedomning`, `updateItemMotivering`, `signOffItem`, `unsignOffItem`) all check `cycle.status !== 'SEALED' && cycle.status !== 'ARKIVERAD'` explicitly and return a structured error otherwise. Add a `// TODO(21.10)` marker at each action's callback entry (mirrors the convention in Story 21.4's `materialiseCycleItems`).

9. **Row expansion — drawer.** Clicking the row's chevron expands a drawer below the row (not a separate side-sheet; inline accordion expansion — matches the 17.17 kravpunkter row accordion UX in `compliance-detail-table.tsx`). The drawer contains three read-mostly sections:
   - **Kravpunkter (from snapshot).** Renders `kravpunkterSnapshot.requirements` from Story 21.4's frozen snapshot as a compact, read-only list: each requirement's `text`, `comment` (if set), and a small fulfilment indicator (check / empty circle) based on the snapshot's `is_fulfilled` flag. The snapshot is **frozen** — no editor affordance. Header text: `'Kravpunkter (snapshot {frozen_at})'` (date-fns Swedish format).
   - **Länkade artefakter (current, live).** Renders `<LinkedArtifactsPanel listItemId={lawListItemId} readOnly />` — the existing Story 17.18 component, passed `readOnly` so no add/remove affordances are exposed from within the cycle. This surfaces the live (not frozen) artifact set that's currently linked to the source `LawListItem`; the cycle's frozen evidence capture is Story 21.9's concern. **Error isolation:** the panel owns its own SWR fetch and renders its own skeleton + error UI. Do NOT wrap it in a parent-level error boundary here — a panel-fetch failure must NOT bubble up as a page-level error or collapse the row drawer. The existing component already handles its own failure states.
   - **Findings för denna post.** Placeholder row `'Hanteras i Story 21.7'` — no fetch, no component yet.
   Only one row's drawer is open at a time (single-expand accordion semantics); expanding row B auto-collapses row A. Keyboard: `Enter` on a focused row toggles its drawer; `Escape` collapses the currently-expanded drawer.

10. **New server actions in `app/actions/compliance-audit-item.ts`.** New file, first creation of this module. Exports:
    - `getCycleItemsForCycle(cycleId: string): Promise<ActionResult<{ items: CycleItemRow[]; cycle: CyclePartial }>>` — read. OR-permission gate. Returns the items + a minimal cycle partial (`id, status, name, sealHash`) so the page can gate read-only in one round-trip.
    - `updateItemBedomning(input: { itemId: string; efterlevnadsbedomning: EfterlevnadsBedomning | null }): Promise<ActionResult<{ item: CycleItemRow }>>` — mutation. `tasks:edit`. Sets `reviewed_at = now()`, `reviewed_by_user_id = ctx.userId` on every change. Blocked on SEALED/ARKIVERAD with structured error. Logs activity `cycle_item_bedomning_updated` with `{ old: prev, new: next }`.
    - `updateItemMotivering(input: { itemId: string; motivering: string | null }): Promise<ActionResult<{ item: CycleItemRow }>>` — mutation. `tasks:edit`. Sets `reviewed_at = now()`, `reviewed_by_user_id = ctx.userId` on every save. Blocked on SEALED/ARKIVERAD. `motivering.length <= 5000` (Zod). Logs activity `cycle_item_motivering_updated` with `{ old_length, new_length }` only (not the full text — AC 11 log-volume guard).
    - `signOffItem(itemId: string): Promise<ActionResult<{ item: CycleItemRow }>>` — mutation. `tasks:edit`. Idempotent: if `signed_off_at !== null`, returns success without write. Sets `signed_off_at = now()` and `signed_off_by_user_id = ctx.userId`. Blocked on SEALED/ARKIVERAD. Additionally blocked when `efterlevnadsbedomning === null` — server-side structured error `'Ange bedömning innan signering'`; UI enforces the same via disabled button + tooltip. Logs activity `cycle_item_signed_off`.
    - `unsignOffItem(itemId: string): Promise<ActionResult<{ item: CycleItemRow }>>` — mutation. `tasks:edit`. Reverse of `signOffItem`. Sets `signed_off_at = null`, `signed_off_by_user_id = null`. Blocked on SEALED/ARKIVERAD. Logs activity `cycle_item_unsigned`.
    - All actions use `withWorkspace(cb, <perm>)`, workspace-scope via a cycle-join (the item's parent cycle must belong to `ctx.workspaceId`); not-found / wrong-workspace returns the generic `'Kontrollposten hittades inte'`. Every mutation calls `revalidatePath('/laglistor/kontroller/' + cycleId)` after success — path revalidation does not invalidate the main `/laglistor` view.
    - Every mutation returns the **full updated `CycleItemRow`** (not just `{ success: true }`) so the SWR cache can set-in-place without a refetch. Optimistic-UI callers still reconcile against the returned row on success.

11. **Activity logging.** Every mutation in AC 10 writes exactly one `ActivityLog` row via the shared `logActivity(workspaceId, userId, entityType, entityId, action, oldValue?, newValue?)` helper at `@/lib/services/activity-logger.ts:8-28` with:
    - `entityType = 'compliance_audit_item'`
    - `entityId = item.id`
    - `action` strings: `'cycle_item_bedomning_updated'`, `'cycle_item_motivering_updated'`, `'cycle_item_signed_off'`, `'cycle_item_unsigned'`.
    - **Log volume guard (NFR / memory — activity-log debouncing from Story 14).** `updateItemMotivering` is called on debounce (2s idle / blur), so per-save is already bounded; do NOT add an additional per-keystroke log path. For bedömning and sign-off, one-log-per-action is correct. Verify behaviour in unit tests (AC 13).
    - Story 21.13 owns the Swedish label map for these actions in the global `/workspace/activity` feed; this story does NOT add label entries (surfaced in activity feed will be untranslated until 21.13 ships — acceptable).

Virtualisation must keep DOM-focused rows present (use `useVirtualizer`'s `measureElement` + focus-retention pattern already established in `compliance-detail-table.tsx`; verify the existing behaviour rather than re-invent).

### Read-model shape (CycleItemRow) — why this is the right cut

The read model joins three tables (ComplianceAuditItem, LawListItem → document + group + responsible_user, User twice for reviewed_by/signed_off_by) in one Prisma call. Rationale:

- **Single round-trip.** A plain `findMany` with `include` resolves in one query. Splitting into per-row lookups would be N+1.
- **Shape-for-render.** The UI needs all of: law title, document number, group name, source compliance status, bedömning, motivering, review metadata, sign-off metadata, responsible-user display, and the kravpunkter snapshot. Returning `CycleItemRow` ready-to-render lets the tab avoid any client-side joining.
- **Frozen kravpunkter snapshot rides along.** The snapshot is stored as a JSON column on `ComplianceAuditItem`. Returning it in the same call is free (one extra JSON deserialisation per row); the row-drawer render reads it directly without a secondary fetch.
- **Tenant isolation via `cycle.workspace_id` join.** The `where: { cycle: { id, workspace_id } }` filter uses the existing FK from `ComplianceAuditItem.cycle_id → ComplianceAuditCycle.id` + the cycle's own `workspace_id`. Prisma translates this to a single `EXISTS` subquery — tenant-safe without a second round-trip.

Trade-off: the response payload is larger per row (~1.5KB typical, ~3KB for items with many kravpunkter in the snapshot). For a 500-item cycle, total payload is ~1MB — acceptable for an authenticated SSR response; the payload is cached by SWR for 30s between refetches.

### Read-only gate — three layers

Sealed-cycle immutability is enforced on three independent layers:

1. **UI layer (this story).** The `readOnly` prop threads from the page shell → header → items tab → row components. Editors render non-interactive variants; sign-off buttons are hidden; the banner informs the user.
2. **Server-action layer (this story, defence-in-depth).** Each of the four mutation actions checks `assertCycleEditableUi(cycle.status)` at entry and returns a structured error on SEALED/ARKIVERAD. Even a direct call from a malicious client cannot mutate — the UI layer is not the sole defence.
3. **Guard library (Story 21.10, future).** `lib/compliance-audit/cycle-guards.ts#assertCycleEditable(tx, cycleId)` will replace this story's inline `assertCycleEditableUi` check with a transaction-participating guard. Each mutation action gets a `// TODO(21.10)` marker at the check point so Story 21.10's dev can find + replace them.
