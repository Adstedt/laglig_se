## Status

## Story

**As an** HR manager,
**I want** to manage uploaded agreements and assign them to employees,
**so that** the right agreement applies to each person.

## Acceptance Criteria

1. Agreement list (the 7.5 `KollektivavtalManager`, both mounts) shows: Namn, Typ, Uppladdad, Status, **antal tilldelade anställda** (7.5's `listCollectiveAgreements` already returns the count — surface it), plus **Redigera** (namn/typ/giltighetsperiod) and **Ta bort** actions for `employees:manage`.
2. **Assign in bulk** — per-agreement "Tilldela" action with two targeting modes, mirroring Fortnox "Avtal för löner":
   - **by Personaltyp**: e.g. the ARB agreement to ALL employees with `personel_type = ARB`
   - **by EmployeeGroup**: e.g. the whole "Lager" enhet
   Preview of affected count before confirming ("Tilldelar X anställda"); overwrites existing assignments for the targeted employees (deliberate — reassignment is the Fortnox behavior); optimistic-safe via `revalidatePath`. (Individual assignment already exists in the Personalkort — unchanged.)
3. **Delete an agreement** (confirmation dialog stating consequences) →
   - its `ContentChunk` rows removed (`source_type COLLECTIVE_AGREEMENT`, `source_id = agreement.id`, workspace-verified BEFORE the delete) — no longer retrievable
   - employees auto-unassign via the existing FK `onDelete: SetNull`
   - **the backing `WorkspaceFile` + storage object are deleted too** (SM/PO decision: 7.5's root-folder filename dedupe would otherwise block re-uploading the same PDF — a verified trap; the file was created by the CA flow and belongs to it)
   - **profile-sync honesty**: if no agreements remain → `has_collective_agreement = false` + `collective_agreement_name = null`; if the deleted agreement's name matches the profile name → repoint to a remaining agreement's name; same scoped-write + completeness-recompute path as 7.5 (never the `workspace:settings`-gated action)
4. **Carried findings closed** (from 7.4b/7.5/7.5b gates):
   - **UX-ADMIN-001**: Settings "Kollektivavtal" tab renders only when `canAccessSettings && canViewEmployees` — ADMIN (no employees permission) no longer sees a permanently broken tab
   - **DUP-001**: extract a shared storage-upload helper used by BOTH `files.ts#uploadFile` and `collective-agreements.ts` (quota, hash, mime allowlist, `upsert:false`, dedupe — one implementation)
   - **STATE-001**: `employee-list-table.tsx` column-state handlers → functional `setColumnState(prev => …)`
   - **TEST-002**: test the `workspaceId === null` persistence-skip path
   - **LOG-001**: fix the stale "truncated at 200k chars" log string in the extract-files cron route

### Code Quality Assessment

Deep review on the destructive path, moderate elsewhere, per the risk profile. The implementation is disciplined and the two dev-flagged judgment calls were both made correctly (verdicts below).

**Delete cascade (AC 3) — the deep target, verified line by line** (`app/actions/collective-agreements.ts#deleteCollectiveAgreement`):

- Verify-then-act: `findFirst { id, workspace_id }` before anything; a cross-workspace id returns "hittades inte" with zero side effects (test-pinned: no chunk/agreement/file/storage call).
- ONE interactive `$transaction` covers chunks → agreement → file row → profile sync, in exactly the Dev Notes order (order test-pinned via `invocationCallOrder`).
- Chunk `deleteMany` filter carries `source_type` AND `source_id` AND `workspace_id` — defense in depth as specified, mirroring the files.ts USER_FILE precedent.
- Backing-file lookup is itself workspace-verified; a foreign file row is provably left alone (test-pinned) while the agreement still deletes.
- Adversarial FK audit (mine, beyond the story): every relation onto `WorkspaceFile` (`FileTaskLink`, `FileListItemLink`, `RequirementEvidenceLink`, folder children) is `onDelete: Cascade`, and `CollectiveAgreement.workspace_file_id` is `SetNull` — but the agreement is deleted *before* the file row, so no ordering hazard either way. The in-tx file-row delete cannot hit a restrict trap even when the user linked the AVTAL file to tasks/law-list items in Filer.
- Storage removal is post-commit, double-guarded (both the `removeError` result and a thrown error are caught + logged), and a remove failure still returns success (test-pinned).
- Profile-honesty matrix: all three branches test-pinned — delete-last → `has_collective_agreement=false` + name null + completeness recompute; delete-the-named → repoint to first remaining (name-asc, `take: 1`, same order as the list) with the flag provably NOT flipped; delete-other → zero profile writes. The no-op guard (`if (!data) return`) also avoids gratuitous writes when the profile was already honest.

**Verdict 1 — in-tx profile sync (dev-flagged deviation): ACCEPTED, and I judge it the better design.** For a destructive operation, consistency beats availability: a profile-write failure rolls the whole delete back to a fully consistent, *retriable* state. The fail-safe alternative would commit the delete and leave a lying `CompanyProfile` (flag true, name pointing at a deleted agreement) with no reconciliation path — exactly the dishonesty AC 3 exists to prevent. The asymmetry with 7.5's upload sync is principled, not accidental: on upload the primary artifact already exists and the sync is additive best-effort (rolling back a successful upload over a profile hiccup would destroy user work); on delete nothing is lost by retrying. The failure mode is also near-theoretical — a scoped two-field update on a row in the same database. 7.5's upload path keeps its fail-safe semantics untouched, as claimed.

**Verdict 2 — inactive employees included in bulk assign (dev-flagged): ACCEPTED as implemented, with a PO confirmation logged (BULK-INACTIVE-001, low).** Fortnox "Avtal för löner" assigns wholesale per personaltyp; the preview counts inactive employees too, so the guardrail never lies; and I verified the dialog copy nowhere implies "active only" ("Tilldela X till flera anställda samtidigt.
