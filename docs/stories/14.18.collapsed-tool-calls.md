## Story

**As a** user reviewing an AI assessment or chat response,
**I want** completed tool calls to be collapsed into a single compact summary line,
**so that** the conversation stays focused on the AI's analysis rather than the tool-call mechanics.

## Context

Today `components/features/ai-chat/chat-message.tsx` renders each tool call as its own `ToolCallRow` — status icon, tool icon, Swedish label, optional detail text. An assessment response typically invokes 3-5 tools in sequence (`get_company_context`, `search_laws` ×1–3, `get_change_details`, `suggest_followups`), producing a vertical stack of 4-6 rows before the assistant's actual prose appears. On the change-assessment view this pushes the assessment text below the fold on a 13" laptop.

Claude Desktop solves this by collapsing *completed* tool calls into one line:

```
✓ Hämtade företagskontext · Sökte i lagdatabasen (3) · Hämtade ändringsdetaljer
```

Running or errored calls stay expanded; only the successful, finished block collapses. The collapsed line is expandable to reveal the per-call rows.

The codebase already contains a partial precedent: `SearchToolGroup` in `chat-message.tsx` collapses consecutive `search_laws` calls into `"Sökte i lagdatabasen — N sökningar"` with an optional "Visa sökfrågor" toggle. This story generalizes that pattern across all tool types.

## Acceptance Criteria

### Collapsed Summary Row

1. When an assistant message contains two or more tool calls that are **all in `output-available` state**, they render as a single "completed-tools summary row" in place of the per-call `ToolCallRow`s.
2. The summary row shows:
   - A single green check icon (same styling as the existing `ToolCallRow` done state)
   - Dot-separated (`·`) list of `doneLabel` values from `TOOL_CONFIG` in the order the calls occurred
   - When the same tool is called multiple times in a row, the label appears once with a trailing count in parentheses: `"Sökte i lagdatabasen (3)"` (preserves the existing `SearchToolGroup` UX for parallel searches)
3. Tools marked `hidden: true` in `TOOL_CONFIG` (currently `suggest_followups`) are excluded from both the label list and the count.
4. If a tool has no entry in `TOOL_CONFIG`, fall back to the raw `toolName` (matches current `ToolCallRow` fallback behavior).

### Mixed-State Handling

5. Tool calls in `input-streaming` / `input-available` / running state (`state !== 'output-available' && state !== 'output-error'`) render **expanded** as individual `ToolCallRow`s with the existing animated loader. They do not collapse until they complete.
6. Tool calls in `output-error` state render **expanded** as individual `ToolCallRow`s with the existing destructive styling. Errored calls never collapse, even after peer calls complete.
7. In a mixed block (some done, some still running), the completed calls that are *before* the first non-done call may collapse; running/errored calls remain expanded below them. Implementation-wise this means grouping is done in contiguous same-state runs rather than one monolithic group per message.

### Expand / Collapse Interaction

8. The collapsed summary row has an inline expand affordance (chevron + "Visa detaljer" / "Dölj detaljer" toggle, mirroring `SearchToolGroup`'s "Visa sökfrågor" pattern).
9. Expanding reveals the per-tool `ToolCallRow`s beneath the summary line using the existing row component (no visual changes to the expanded rows themselves).
10. Collapse/expand state is component-local (`useState`) — not persisted across re-renders triggered by new streaming tokens in sibling parts, and not shared across messages.

### Clickability Into Sidebar

11. Each dot-separated label in the collapsed summary row is individually clickable when the corresponding tool call's output has `sidebarHint === 'open' | 'suggest'` AND `chatDetail !== null`. Clicking opens the sidebar using the existing `buildDetailItem()` routing in `chat-message.tsx` (re-exported/shared, not duplicated).
12. Labels without a clickable target render as plain text (no hover affordance). The green check icon applies to the row as a whole and is not itself clickable.
13. The active label (matching `chatDetail.activeDetail?.id`) gets the existing `ring-1 ring-primary/30` active-state treatment, applied to the individual label chip rather than the whole row.

### Parity With SearchToolGroup

14. The generalized grouping logic **replaces** the current `SearchToolGroup` / `groupSearchParts` implementation. A sequence of `search_laws` calls still collapses to `"Sökte i lagdatabasen (3)"` and still opens the merged search-results sidebar detail when clicked — behavior is preserved, code path is unified.
15. The "Visa sökfrågor" sub-toggle (listing individual queries under a `search_laws` group) is dropped in favor of the generalized "Visa detaljer" expander that reveals the per-call rows. Individual queries are visible via each expanded `ToolCallRow`'s `detail` field (already shows `"query"`).

### Streaming Transition

16. During streaming, when the last tool call transitions from `input-streaming` → `output-available`, the group re-renders as collapsed on the next React update. No animation is required for v1 — an instant swap is acceptable. (Animated expand→collapse transition is explicitly out of scope; capture as a follow-up if desired.)
17. After all tool calls in a contiguous block are done and `isStreaming === false`, the group remains collapsed unless the user expands it.

### Non-Regression

18. Single-tool-call blocks (only one tool call in the message, or contiguous block of length 1) continue to render as a standalone `ToolCallRow` — no collapse, no summary wrapper. Collapsing a single row into a "1 tool" summary adds no value.
19. The citation pill rendering, `ReasoningBlock`, `TextBlock`, `step-start` handling, and user-message branch in `ChatMessage` are untouched.
20. The `suggest_followups` hidden-tool behavior remains unchanged (still not rendered at all, whether in the summary line or expanded).
21. Write-preview auto-open behavior (`sidebarHint === 'open'` → `buildDetailItem()` → `chatDetail.openDetail()` fired after a 100ms debounce in `ToolCallRow`) must still trigger when the tool call is inside a collapsed group. Auto-open is keyed to the individual tool's result, not to the group's render state.

## Tasks / Subtasks

### Task 1: Refactor grouping logic (AC: 1, 5–7, 14)

- [ ] 1.1: In `components/features/ai-chat/chat-message.tsx`, replace `groupSearchParts()` with a generalized `groupCompletedToolParts(parts)` that produces `RenderItem` entries. The new union:
  ```typescript
  type RenderItem =
    | { kind: 'part'; part: any; index: number }
    | { kind: 'tool-group'; items: Array<{ part: any; index: number }> }
  ```
- [ ] 1.2: Walk `parts` and accumulate a "group candidate" while the next part is `isToolUIPart(part)` AND `state === 'output-available'` AND the tool is not `hidden: true` in `TOOL_CONFIG`. Flush the candidate on any non-tool part, any non-done tool state, or end-of-array.
- [ ] 1.3: A flushed candidate with length ≥ 2 emits `{ kind: 'tool-group', items }`. Length 1 emits a regular `{ kind: 'part' }` so single calls still use plain `ToolCallRow` (AC 18).
- [ ] 1.4: Hidden tools (`TOOL_CONFIG[name]?.hidden === true`) are filtered **before** grouping — same as the current `ChatMessage` render path already skips them — so they neither appear nor interrupt a group.
- [ ] 1.5: Running (`input-*` / `output-*` pending) and errored (`output-error`) tool parts flush the current group and emit `{ kind: 'part' }` — these render as standalone `ToolCallRow`s per AC 5–6.

### Task 2: Build `CollapsedToolGroup` component (AC: 1–4, 8–13, 15)

- [ ] 2.1: Create a new `CollapsedToolGroup` component in `chat-message.tsx` (co-located like `SearchToolGroup`/`ToolCallRow`; keep all chat-message UI in one file for consistency with the existing module).
- [ ] 2.2: Props: `{ items: Array<{ part: any; index: number }> }`. Extract per-tool info using the existing `extractToolPartInfo()` helper.
- [ ] 2.3: Compute the label sequence: walk `items`, coalesce runs of the same `toolName` into `{ toolName, count, toolCallIds: string[], outputs: unknown[] }`. Render each as a span with the `TOOL_CONFIG[toolName].doneLabel` (fallback: `toolName`) and `count > 1 ? ' (' + count + ')' : ''`.
- [ ] 2.4: Each label span becomes a `<button>` when the coalesced group is clickable (per AC 11). Clickability rule: the first `output` in the run has `_meta?.sidebarHint` in `{'open','suggest'}` AND `chatDetail !== null`. For `search_laws` runs, reuse the existing `SearchToolGroup` merging logic — build `mergedData = outputs.filter(...)` and call `chatDetail.openDetail({ type: 'tool-result', id: 'tool-group-{firstIndex}-{toolName}', toolName, data: mergedData })`. For single-call runs, call `buildDetailItem(toolCallId, toolName, output)` — extract `buildDetailItem` from `ToolCallRow`'s inner scope to module scope so both components can reuse it (required anyway for AC 11).
- [ ] 2.5: Separator: render `·` (U+00B7) between label spans with `text-muted-foreground/50` spacing.
- [ ] 2.6: Active-state ring on individual label chips when `chatDetail.activeDetail?.id` matches the chip's detail id (AC 13). Use the existing `ring-1 ring-primary/30` utility.
- [ ] 2.7: Render the existing row container styling (`rounded-md bg-muted/15 py-1 px-2.5 flex items-center gap-1.5`) with a single `Check` icon at the start. No per-tool icon in the collapsed row (keep the line compact).
- [ ] 2.8: Below the summary row, render an expand toggle (chevron + `Visa detaljer` / `Dölj detaljer`) mirroring `SearchToolGroup`'s toggle styling. Local `useState` for expansion (AC 10).
- [ ] 2.9: When expanded, map `items` to the existing `ToolCallRow` component — passing `toolCallId`, `toolName`, `state`, `detail = getToolDetail(toolName, input)`, and `output`. This guarantees the expanded view is byte-identical to today's per-row rendering (AC 9, 15).

### Rendering pattern to follow

Per CLAUDE.md memory: "SDK best practice — render `message.parts` sequentially; each part type gets its own component. No grouping/reclassification logic." The `groupCompletedToolParts` helper is a **visual** grouping on top of the sequential render, not a reclassification of parts — it preserves order and only wraps contiguous same-state runs.

### Constraints

- [Source: architecture/17-coding-standards.md#17.2 TypeScript Standards] — strict mode is enforced; avoid `any` where possible. The existing `groupSearchParts` uses `any` for the `part` type because the `UIMessage` parts union is complex. Keep the same pattern for `groupCompletedToolParts` to avoid churn — do not attempt a type refactor in this story.
- [Source: architecture/17-coding-standards.md#17.3 React/Next.js Standards] — `chat-message.tsx` is a client component (`'use client'`). All new sub-components remain client components by inheritance.
- [Source: architecture/17-coding-standards.md#17.3] — use shadcn/ui primitives for interactive UI. The label chips are plain `<button>`s styled with Tailwind (matching the existing `ToolCallRow` / `SearchToolGroup` convention); no new shadcn component is needed.
- Accessibility: the collapsed summary row and each clickable label must be reachable via keyboard (native `<button>` takes care of this). Add `aria-label` on each label button describing the tool (e.g., `aria-label="Visa resultat: Hämtade företagskontext"`) since the visual label alone may be ambiguous for screen readers grouped with the adjacent icon.
