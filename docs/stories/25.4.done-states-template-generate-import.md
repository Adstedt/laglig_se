# Story 25.4: Done states for template / generate / import paths (B.4)

## Status

Ready for Review

## Story

**As a** new user who picked a path in the first-run modal (Mall / Generera / Importera) and watched the work complete,
**I want** the modal to land me on a celebratory "klart" surface that summarises what was created and points me at the obvious next step (open my list, review the matchings, keep reading the tutorial),
**so that** the modal closes the loop instead of disappearing into a dashboard banner or silently dropping me onto a granska page with no orientation — and so I can decide between "ta mig till listan" and "låt mig läsa klart" without losing context.

This is the **B.4 wedge** of Epic 25: three `<DoneXStep>` components + the transition logic that lands users on them. B.0–B.3 built the gate, the modal shell, the progress strip + tab framework, and the six tab content panels. What's missing is the **finish line**: today, generate path silently drops the progress strip when status flips to `completed` (per Story 25.2 AC 13 — explicit B.4 deferral); import path immediately closes the modal and pushes to `/laglistor/skapa/{importId}/granska` (per Story 25.1's `handleImportSuccess`); template path immediately closes the modal and pushes to `/laglistor/mallar/{slug}` for the user to click "Använd denna mall" themselves. None of those handoffs use the modal's own real estate to celebrate or summarise — they just exit.

Concretely: B.4 adds three new `Step` values (`'done-generate' | 'done-import' | 'done-template'`) to `<FirstRunModal>`'s step union, three new step component files under `components/features/onboarding-modal/`, auto-transition logic from `tutorial` → `done-generate` when SWR sees `status='completed'`, a pivot of the template path from "route out to /mallar/{slug}" to "call `adoptTemplate` inline and show done-template", and a pivot of the import path from "immediate route" to "show done-import first, then route on CTA click". Each done step has its primary navigate-away CTA ("Visa min laglista →" / "Granska matchningar →") and a secondary "Fortsätt utforska" link that closes the modal back to the dashboard for now — full `tutorial_only` mode rendering is B.6 scope.

Failure paths get matching error cards: generate failed → "Försök igen" CTA that re-fires `POST /api/workspace/generate-law-list`; import matching failed → "Skapa supportärende" CTA (or routes back to the upload step for retry, dev's call per AC 14); template apply failed → "Försök igen" CTA that re-calls `adoptTemplate`.

### Done-import step — `<DoneImportStep>`

9. **Auto-transition `tutorial` → `done-generate`** lives in `<TutorialStep>` (or in `<FirstRunModal>` if cleaner — dev's call; tutorial-step already owns the SWR-shared progress data via `<ProgressStrip>`). Recommended approach: lift the transition into `<FirstRunModal>` by reading `initialStatus` and the SWR cache directly in the parent so the parent can swap `step` state. Implementation sketch (in `<FirstRunModal>`):
    ```ts
    // Re-use the same SWR key as ProgressStrip so we hit the warm cache.
    // GenerationStatusResponse interface (post-25.3 v0.5) now includes
    // `startedAt: string | null` — required for the duration copy hook in AC 6.
    const { data } = useSWR<GenerationStatusResponse>(
      '/api/workspace/generation-status',
      fetcher,
      {
        refreshInterval: (latest) => {
          const s = latest?.status ?? initialStatus
          return s === 'pending' || s === 'in_progress' ? 3000 : 0
        },
        revalidateOnFocus: false,
      }
    )
    const generationStatus = data?.status ?? initialStatus ?? null

    useEffect(() => {
      if (step === 'tutorial' && generationStatus === 'completed') {
        setStep('done-generate')
        return
      }
      if (step === 'tutorial' && generationStatus === 'failed') {
        setStep('done-generate') // showing the failure card variant — see AC 14
      }
    }, [step, generationStatus])
    ```
    Pass `data?.itemCount ?? null`, `data?.groups ?? null`, and `data?.startedAt ?? null` into `<DoneGenerateStep>` as the `itemCount` / `groups` / `startedAt` props. The shared SWR cache means `<ProgressStrip>` and the parent both render from one fetch. **GenerationStatusResponse shape note:** v0.5 added `startedAt: string | null` to the API response (verifiable at `app/api/workspace/generation-status/route.ts:134-135`). If you re-declare the interface locally in `first-run-modal.tsx` instead of importing from `progress-strip.tsx`, include the `startedAt` field; otherwise the SWR fetcher's type widens to `unknown` and TypeScript loses the prop.

### Owner decisions ack'd (v0.4)

The two architectural pivots in this story were explicitly ack'd by the owner on 2026-05-17 before Dev started — both are real behaviour changes to shipped 25.1 code, not bugfixes, and both are architecturally required by the epic spec:

1. **Template path: route-out → inline `adoptTemplate` + done-template.** Picking a template card in the first-run modal now invokes `adoptTemplate({templateSlug})` server-side inside the modal (showing an inline spinner during the typically-<2s wait), then transitions to `<DoneTemplateStep>` with the apply result. Replaces the 25.1 behaviour of closing the modal + routing to `/laglistor/mallar/{slug}` where the user clicked "Använd denna mall" themselves. Trade-off ack'd: one fewer "review before apply" confirmation moment; users who pick the wrong template have no in-flow review step.
2. **Import path: instant-route → done-import → route on CTA.** Successful import now lands users on `<DoneImportStep>` (confidence breakdown + "Granska matchningar →" primary CTA) before routing to `/laglistor/skapa/{importId}/granska`. Replaces the 25.1 behaviour of closing modal + jumping directly to /granska. Trade-off ack'd: one extra click before the user hits the review surface; mitigated by the celebratory + summary content that justifies the extra step.

A third smaller decision was ack'd same day:

3. **"Fortsätt utforska" secondary CTA rendered DISABLED with "Kommer snart" tooltip in B.4.** The architecture spec routes this CTA to `tutorial_only` mode (modal stays open with tabs only, progress strip hidden) — but that render is B.6 scope. ### Key decisions baked into this story

- **Three new step components instead of one switch-based "done step".** Reasoning: each done step has different props (itemCount/groups for generate; counts for import; listName/listId for template), different visual structure (success ring vs. confidence breakdown vs. lightweight handoff), and different CTAs. A single component with mode props would balloon to 200+ lines of conditional JSX — three files at ~100 lines each is cleaner. Matches the per-step organization of the rest of the modal (`path-choice-step.tsx`, `template-pick-step.tsx`, `import-upload-step.tsx`, `tutorial-step.tsx`).
- **Auto-transition logic lives in `<FirstRunModal>`, not in `<TutorialStep>`.** Reasoning: tutorial-step only knows about generate (no import context). Lifting the SWR fetch into the parent gives the parent visibility into all three completion signals and keeps the step transitions centralized. Trade-off: one extra SWR hook in the parent, but it shares the cache with `<ProgressStrip>` so there's no double-fetch.
- **Template path now applies inline instead of routing to /mallar/{slug}.** Reasoning: the architecture spec (epic line 207, arch §6.2 mode transitions) treats template apply as synchronous with an immediate done-template-step. The previous "route to detail page → user clicks adopt" pattern was 25.1's interim — B.4 closes the loop. Cost: a one-line change in `<TemplatePickStep>` (replace `router.push(/laglistor/mallar/${slug})` with `onTemplateApplied(result)`) + the parent owns the done-template transition.
- **Import path now shows done-import inside the modal before routing to /granska.** Reasoning: same architectural close-the-loop pattern. The 25.1 immediate-route was interim; B.4 lands the celebratory + summary surface the epic spec describes.
- **Secondary "Fortsätt utforska" CTA is rendered DISABLED in B.4 with a "Kommer snart" tooltip (owner decision 2026-05-17, v0.4).** Reasoning: `tutorial_only` mode render is B.6 scope. A button labelled "keep exploring" that actually closes the modal — the v0.1/v0.2/v0.3 draft proposed this — is misleading. Disabling with an honest "Kommer snart" tooltip is better product positioning for one release. The `onKeepExploring` callback prop is still plumbed through all three done components so B.6 enables the button with a one-line per-component swap (drop `disabled`, drop the tooltip wrapper, restore `onClick={onKeepExploring}`). The `done_cta_clicked` payload union includes `'keep_exploring'` for the same forward-compat reason — no telemetry-taxonomy change needed in B.6. Failure-mode secondary CTAs ("Stäng guiden" on AC 19/20) stay enabled — closing on failure is a legitimate user action, not a deferred B.6 surface.
- **Failure modes use the same component with a `mode` prop, not separate FailedGenerate/FailedImport components.** Reasoning: the failure UI is structurally close to the success UI (same headline-subheadline-CTA layout, just different icon + copy + CTA labels) — same component with a discriminator avoids file proliferation. Trade-off: the component's prop shape is slightly fuzzier (loose-prop pattern with optional callbacks) — accepted because the alternative (discriminated union) generates more boilerplate.
- **`done_shown` + `done_cta_clicked` event types added inline + architecture doc updated in same story (Task 11).** Reasoning: `event_type` is `String` in the OnboardingEvent schema, so new values don't require migration. v0.1 of this story deferred the architecture-doc edit to a separate ticket; v0.2 (PO polish) brought it back in-scope as Task 11 since the cost is ~5 minutes and the doc-loses-fidelity-on-drift risk is real (per S5 in the v0.1 validation report).
- **No `<ErrorBoundary>` wrap on the done steps — accepted risk.** Mirrors Story 25.3's same decision for the tab content components. The done steps are static-leaning JSX with one server-action call (template apply) and one router call (CTA handlers). Runtime-crash surface is small; per-step render tests + StrictMode guards catch the "renders without crashing" case automatically. If real-world crashes materialize, future hardening story can wrap the step body.
- **`getImport(importId)` reused for import counts (or client-side reduce as fallback).** Reasoning: avoid writing a new query if the existing one already returns the data. The fallback (client-side count over the rows array) is a one-liner with no downside since the rows are already in memory by the time matching completes.

**Known design limitation — done-state surfaces are one-shots (NOT a bug, but worth documenting):**

Discovered during the generate-path smoke test on 2026-05-17. The auto-transition from `tutorial → done-generate` requires the modal to be mounted when SWR sees `status === 'completed'`. If the modal closes before that moment (dev-mode Fast Refresh during a code edit, browser navigation away from `/dashboard`, or any cause that triggers `<HemPage>` to re-evaluate `onboardingState.firstRunOpen`), the done-generate surface never renders — the user falls back to the dashboard `<LawListGenerationProgress>` `CompletedCard` banner, which is the pre-B.4 success surface.

**Why this is design-intent, not a bug:**

- `getOnboardingState`'s `firstRunOpen` guard requires `law_list_generation_status === null` — by design, the modal is "first-run only" and once the user picks Generera, status flips to `'pending'` → `'in_progress'` and firstRunOpen permanently becomes false on subsequent server-renders.
- The architecture's intended re-entry pattern is **Story B.6's corner FAB + `tutorial_only` mode** — not a firstRunOpen re-mount. The FAB will let users re-open the modal post-dismissal at `tutorial_only` (tabs only, progress strip hidden) — but **not** at the done-generate / done-import / done-template surfaces. Those done states are by-design one-shots in the auto-transition path.
- The dashboard banner (`<LawListGenerationProgress>` CompletedCard) is the existing safety net: it already shows the substantive info (itemCount + group breakdown) that done-generate adds. What's lost when the modal closes mid-generation is only the "Klart på Xm Ys" duration line (new in B.4), the sage success ring (decorative), and the "Visa min laglista" CTA (banner has equivalent button).

**Why we are NOT applying a "loosened firstRunOpen" fix in B.4:**

Considered but rejected. Making `firstRunOpen=true` when `first_run_dismissed_at IS NULL AND status IN ('pending','in_progress','completed','failed')` would re-mount the modal on subsequent /dashboard visits and let the auto-transition fire. But:

1. It would create dual re-entry paths (firstRunOpen-driven AND FAB-driven from B.6) that will conflict when B.6 lands and have to be unwound.
2. The done-state being a one-shot is consistent with the architecture's celebratory-moment framing.
3. The dashboard banner is the existing safety net for the substantive info.

**Flagged for B.6:** when implementing the FAB + tutorial_only mode, consider whether FAB re-entry should be smart enough to land on done-* if the user dismissed during that step (would require persisting `last_modal_step` or similar). Out of scope here.
