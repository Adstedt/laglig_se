# Story 14.10: Change Acknowledgement Flow

## Status

Ready for Review (QA fixes applied — SEC-001 resolved, re-review requested)

## Story

As a workspace member tracking laws in my law list, I want the compliance partner to guide me through understanding and acknowledging changes to my tracked documents, so that I can make an informed assessment and document whether a change affects my company.

## Context & Dependencies

**Why this story exists:** This is the primary value delivery surface for change monitoring. When a user receives a change notification — via the Hem dashboard smart cards, the Changes tab, or a law list item indicator — they need to understand what changed, whether it affects them, and record their assessment. The compliance partner agent (already operational) guides them through this using the existing chat interface with change-specific context injection.

**Key architecture principle:** This story composes existing infrastructure. The unified chat route (`/api/chat`), `ChatPanel` component, `useChatInterface` hook, tool suite, RAG pipeline, and citation system are all operational from Stories 14.7a-c, 14.8, 14.9, and 14.11. This story adds a new context type (`'change'`), a resolution/acknowledgement UI, and the data model to persist assessments.

**Builds on:**
- Story 14.11 (Hem dashboard chat — smart cards showing unacknowledged changes, `ChatPanel`, `useChatInterface`, streaming message rendering)
- Story 14.7a-c (agent tools — `search_laws`, `get_change_details`, `get_company_context`, `save_assessment` stub, write confirmation pattern)
- Story 14.8 (RAG retrieval pipeline)
- Story 14.9 (system prompt with dynamic context injection via `buildSystemPrompt`)
- Story 14.4 (WorkspaceProfile for company context)
- Story 8.1 (Changes tab with `tab=changes` URL param, `ChangeRow` click navigation)

**Depended on by:** Nothing currently — this is a leaf story.

**Future enhancements (out of scope):**
- Enriching `get_change_details` to fetch full amendment document content for deeper analysis
- Web search tool for referenced propositions (propositioner) in amendment documents
- Enriched company context (law lists, completed tasks, uploaded evidence)
- BM25 keyword search complementing vector search (Story 14.13)

## Acceptance Criteria

### Data Model

1. Prisma model `ChangeAssessment` with fields: id (cuid), changeEventId (String), lawListItemId (String), workspaceId (String), status (AssessmentStatus enum), impactLevel (ImpactLevel enum), aiAnalysis (Text?), aiRecommendations (Json?), userNotes (Text?), assessedBy (String, userId), assessedAt (DateTime), createdAt, updatedAt
2. AssessmentStatus enum: REVIEWED, ACTION_REQUIRED, NOT_APPLICABLE, DEFERRED
3. ImpactLevel enum: HIGH, MEDIUM, LOW, NONE
4. Relations: ChangeAssessment -> ChangeEvent (add `change_assessments` relation field to ChangeEvent model), ChangeAssessment -> LawListItem, ChangeAssessment -> Workspace
5. ComplianceStatusLog model: id, lawListItemId, previousStatus (ComplianceStatus), newStatus (ComplianceStatus), changedBy (userId), reason (String?), changeAssessmentId (nullable), changedAt (DateTime). Uses existing `ComplianceStatus` enum values: `EJ_PABORJAD`, `PAGAENDE`, `UPPFYLLD`, `EJ_UPPFYLLD`, `EJ_TILLAMPLIG`
6. Add `CHANGE` value to existing `ChatContextType` Prisma enum (currently: `GLOBAL`, `TASK`, `LAW`)
7. Migration runs cleanly on fresh and existing databases

### Context Injection — `'change'` Context Type

8. `ChatContextType` extended with `'change'` (in `useChatInterface` TS type and Prisma `ChatContextType` enum)
9. When `contextType='change'` and `contextId=changeEventId`: chat route loads ChangeEvent + related amendment data and injects change-specific context into the system prompt via `buildSystemPrompt`
10. System prompt change context includes: amendment SFS number, change type, AI summary, affected sections list, base law title and SFS number, and effective date if known
11. Agent opens with contextual guidance acknowledging the specific change (driven by system prompt, not hardcoded UI text)

### Entry Points

12. **Hem smart cards**: Unacknowledged changes card on `/hem` (`components/features/dashboard/context-cards.tsx`) opens chat with `contextType='change'` and the relevant `contextId`. Currently sends a chat prompt — wire to set context instead.
13. **Changes tab**: ChangeRow click (Story 8.1, `components/features/changes/changes-tab.tsx`) opens chat panel/navigates with change context pre-loaded
14. **Law list item indicator**: Change badge click (`components/features/changes/change-indicator.tsx`) on a tracked law opens chat with change context

### Resolution / Acknowledgement UI

15. When in `contextType='change'` mode, the chat panel shows a resolution footer/sidebar with: AssessmentStatus selector (Granskad, Atgard kravs, Ej tillamplig, Uppskjuten) and ImpactLevel selector (Hog, Medel, Lag, Ingen)
16. Optional user notes textarea in resolution UI
17. Save button creates/updates `ChangeAssessment` record, updates `LawListItem.last_change_acknowledged_at` and `last_change_acknowledged_by`, and logs `ComplianceStatusLog` if compliance status changed
18. Visual feedback on save success; assessment status persists across sessions

### `save_assessment` Tool Completion

19. Complete the existing `save_assessment` stub tool in `lib/agent/tools/` to persist `ChangeAssessment` to the database (currently returns stub response)
20. Tool follows existing write confirmation pattern: `execute: false` → preview, `execute: true` → persist
21. Agent can also trigger assessment save conversationally (via the tool), not only via the resolution UI

### Effective Date Display (folded from Story 8.10)

22. Shared utility `getEffectiveDateBadge(date: Date | null)` returns `{ text, variant }` with Swedish labels: future (amber "Trader i kraft om N dagar"), today (red), recent past (green "Tradde i kraft"), unknown (gray "Ikrafttradadedatum okant")
23. Resolve effective date from chain: `ChangeEvent.amendment_sfs` -> `AmendmentDocument.effective_date` -> `LegalDocument.effective_date` -> null
24. Effective date badge shown in change context (system prompt context and optionally in the resolution UI)
25. Changes tab cards (Story 8.1) show effective date badge — import shared utility

### Changes Tab Status Update

26. Changes tab queries `ChangeAssessment` status for each change event
27. Replace hardcoded "Ny" badge with actual assessment status badge (color-coded: REVIEWED green, ACTION_REQUIRED red, NOT_APPLICABLE gray, DEFERRED blue, no assessment amber "Ny")

### Testing

28. At least 15 unit tests covering: ChangeAssessment CRUD server actions, ComplianceStatusLog creation, context injection for change type, effective date badge utility, effective date resolution, save_assessment tool persistence, resolution UI rendering

The following are **already operational** and should be reused, not duplicated:

- **Chat route**: `app/api/chat/route.ts` — unified route accepting `contextType` + `contextId` in request body. Already loads company profile, builds system prompt, wires all 8 tools, handles citations, rate limiting, streaming.

## Post-QA UX Iteration (2026-03-10)

### Overview

After initial implementation passed QA, a comprehensive UX iteration was performed to improve the change assessment flow end-to-end. The changes span entry points, chat persistence, assessment component behavior, and completion states.

### 1. Changes Tab — Modal Instead of Page Navigation

**Problem:** `ChangeRow` navigated to `/laglistor/andringar/[id]` — a route that was never created, resulting in a 404.

**Solution:** Clicking a change row now opens a `ChangeAssessmentModal` dialog containing `ChatPanel` with `contextType='change'`. No new route needed.

- `ChangeRow` calls `onSelect(change)` callback instead of `router.push()`
- `ChangesTab` manages `selectedChange` state and renders the modal
- Modal shows: change type badge + law title header, full chat with change context, inline assessment

**Files:**
- `components/features/changes/change-assessment-modal.tsx` — Created
- `components/features/changes/change-row.tsx` — Removed `useRouter`, changed to `onSelect` callback
- `components/features/changes/changes-tab.tsx` — Added `selectedChange` state, renders modal

### 2. Hem Page — Change Picker + Dedicated Assessment View

**Problem:** Amendments card on /hem sent a generic prompt ("Visa mina väntande ändringar") which produced a high-level overview — not actionable for reviewing specific changes.

**Solution:** Three-step flow: card click → change picker dialog → dedicated assessment view.

**Flow:**
1. Click "N nya lagändringar" card → `ChangePicker` dialog opens listing all pending changes
2. Select a specific change → `/hem` transitions from home state to `ChangeAssessmentView`
3. Dedicated chat view with: back button, change info header, chat with `contextType='change'`, inline assessment
4. Click "Tillbaka" → returns to /hem home state

**Architecture:** `HemPage` manages `activeChange` state. When set, renders `ChangeAssessmentView` instead of `HemChat`. The change picker is a controlled `Dialog` triggered by state in `HemChat`.

**Files:**
- `components/features/dashboard/change-picker.tsx` — Created. Controlled dialog listing pending changes, lazy-fetches on open via `getUnacknowledgedChanges()`
- `components/features/dashboard/change-assessment-view.tsx` — Created. Full /hem chat view for reviewing a single change (mirrors HemChat conversation layout)
- `components/features/dashboard/hem-page.tsx` — Added `activeChange` state, conditional rendering between HemChat and ChangeAssessmentView
- `components/features/dashboard/hem-chat.tsx` — Added `onSelectChange` prop, `showChangePicker` state, renders ChangePicker
- `components/features/dashboard/context-cards.tsx` — `onChangeClick` triggers picker instead of navigating

### 3. Inline Assessment Component

**Problem:** `AssessmentResolution` was rendered as a fixed footer below the chat — felt disconnected from the conversation.

**Solution:** Assessment now renders inline inside the message scroll area, as part of the conversation flow.

- `ChatMessageList` gained a `footer` prop — optional ReactNode rendered after all messages inside the scroll area
- `ChatPanel` passes `AssessmentResolution` as the `footer` instead of rendering it as a fixed element
- Assessment only appears after Lexa completes her first reply (`hasCompletedReply` check — requires at least one assistant message and not currently streaming)

**Files:**
- `components/features/ai-chat/chat-message-list.tsx` — Added `footer` prop

## Dev Notes — Follow-up Question Chips (`suggest_followups` tool)

### Tool Architecture

The `suggest_followups` tool (`lib/agent/tools/suggest-followups.ts`) is a lightweight passthrough — it performs no database operations, no external calls, and no side effects. Its sole purpose is to structure the agent's follow-up question output via `wrapToolResponse` so the UI can extract it from the tool result. The tool is registered in `createAgentTools()` (`lib/agent/tools/index.ts`) without workspace or user params since it needs neither.

The input schema uses Zod v4 (`z.array(...).min(2).max(4)`) with each question having a `text` (required) and `category` (optional enum: `action`, `deadline`, `context`, `clarification`). The category is available for future UI grouping but is not currently used in the chip rendering.

### Critical Prompt Engineering Lesson — Tool Call Ordering with `streamText`

The AI SDK `streamText` execution model is: **tool calls first, then text output, then done**. Once the model begins writing text, the stream ends. There is NO opportunity for tool calls after text output.

This caused multiple failed attempts:

1. **First attempt** — Prompt said "write your assessment, then call suggest_followups." The tool was never called because the stream ended after the text block was written.
2. **Second attempt** — Stronger Swedish mandatory language ("OBLIGATORISKT", "MASTE"). Same result — the fundamental execution model cannot be overridden by prompt wording.
3. **Working solution** — Restructured `ASSESSMENT_WORKFLOW` in `lib/agent/system-prompt.ts` to group ALL tool calls (`get_company_context` + `search_laws` + `suggest_followups`) into a phase BEFORE the text output. The workflow now has two explicit phases: "Verktygsanrop (gor ALLA verktygsanrop INNAN du skriver bedomningstexten)" followed by "Bedomningstext (skriv EFTER att alla verktygsanrop ar klara)."

**General rule for this codebase**: Any tool that should be called as part of a response MUST be called before the final text block, not after. This applies to all current and future tools used with `streamText`.

The prompt also includes an exception: "Undantag: anropa INTE suggest_followups om du behover stalla en direkt fraga till anvandaren" — this prevents the tool from firing when the agent needs a clarifying question answered before it can assess.

### Hidden Tool Pattern in `TOOL_CONFIG`

`TOOL_CONFIG` in `components/features/ai-chat/chat-message.tsx` maps tool names to Swedish labels, icons, and display state. A `hidden?: boolean` field was added to the config type. When `config?.hidden` is true, the tool part is skipped in the message parts render loop (`return null`), so no `ToolCallRow` appears.

The `suggest_followups` entry is:
```ts
suggest_followups: {
  label: 'Forbereder uppfoljningsfragor',
  doneLabel: 'Forberedde uppfoljningsfragor',
  icon: MessageCircleQuestion,
  hidden: true,
}
```

The tool result is not rendered inline in the message — it is extracted separately by `useFollowupChips` and rendered as chip buttons elsewhere. The config entry exists so the tool is "known" to the system (for logging, future visibility toggles, etc.) without being visible in the message stream.

### Shared Hook: `useFollowupChips`

`lib/hooks/use-followup-chips.ts` extracts follow-up questions from the `suggest_followups` tool result. It returns `{ questions, dismiss }` where `questions` is either an array of `FollowupQuestion` or `null`.

**Dismissal conditions** (returns `null`):
- `dismissed` state is true (set by calling `dismiss()`)
- `hasCompletedReply` is false (agent is still streaming)
- User has sent more than 1 message (only shows after the initial auto-sent assessment request)
- No `suggest_followups` tool result found in the first assistant message, or the tool result state is not `output-available`

**Data extraction path**: `firstAssistantMessage.parts` -> find tool part where `toolName === 'suggest_followups'` and `state === 'output-available'` -> cast result to `{ data?: FollowupQuestion[] }` -> return `result.data`.

The hook uses `useMemo` over `[messages, hasCompletedReply, dismissed]` to avoid unnecessary recomputation.

### UI Component: `FollowupChips`

`components/features/ai-chat/followup-chips.tsx` renders pill-shaped buttons with:
- `animate-in fade-in slide-in-from-bottom-2 duration-300` CSS animation on mount
- A header line "Vill du ga djupare?" with a `MessageCircleQuestion` icon
- Each question rendered as a `rounded-full` button with border, hover states, and `text-xs` sizing
- `onSelect(question.text)` callback on click

### Wiring Into Both Views

Both `ChatPanel` (`components/features/ai-chat/chat-panel.tsx`) and `ChangeAssessmentView` (`components/features/dashboard/change-assessment-view.tsx`) follow the same pattern:

1. Compute `hasCompletedReply`: `!isLoading && messages.some(m => m.role === 'assistant')`
2. Call `useFollowupChips(messages, hasCompletedReply)` to get `{ questions, dismiss }`
3. Wrap `sendMessage` in `handleSendMessage` that calls `dismissFollowups()` first, then sends
4. Wrap chip click in `handleFollowupClick` that calls `dismissFollowups()` then `sendMessage(question)`
5. Render `FollowupChips` inside the `ChatMessageList` footer slot, after `AssessmentResolution`

This means chips appear inline in the scroll area, below the assessment form, and above the scroll anchor — not as a floating overlay.

### Phase 2 Roadmap

The tool schema and UI are designed for future enrichment without breaking changes. Potential improvements:
- **Historical context injection**: Pass past assessments, related tasks, or change history into the prompt so `suggest_followups` generates questions informed by what the user has already reviewed
- **Category-based styling**: The `category` field (`action`, `deadline`, `context`, `clarification`) is captured but not yet used for visual differentiation in chips
- **Persistent suggestions**: Currently chips are ephemeral (dismissed on any user message). A future version could persist them as part of the assessment record

All of these only require richer context injection into the system prompt or minor UI changes — the tool schema and extraction hook remain stable.
