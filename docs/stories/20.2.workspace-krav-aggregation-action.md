# Story 20.2: Workspace Krav Aggregation Server Action

## Status

Draft

## Story

**As a** workspace user (compliance manager, responsible owner, or auditor),
**I want** a single server action that returns every kravpunkt in my workspace with the right filters applied (all / gaps / mine / needs evidence),
**so that** the new `/krav` page can render a cross-laglista overview without N+1 queries or client-side filtering gymnastics.

## Story Context

**Existing System Integration:**

- Integrates with: `LawListItemRequirement`, `LawListItem`, `LawList`, `RequirementEvidenceLink` Prisma models; the shared `resolveEffectiveAssignee` helper from Story 20.1
- Technology: Prisma, Next.js server actions, `withWorkspace(fn, perm)` auth/tenant wrapper, cookie-resolved active workspace
- Follows pattern: existing workspace-scoped aggregation actions (e.g., `getDocumentListItems`, `getComplianceOverview`) — same auth wrapper, same pagination shape, same SWR-friendly return type
- Touch points:
  - `app/actions/law-list-item-requirements.ts` (or a new sibling `app/actions/workspace-requirements.ts`)
  - Shared resolver from Story 20.1
  - `prisma/schema.prisma` — no changes (reads only)

**Motivation:**

Today there is no workspace-level read of kravpunkter — callers must fetch every `LawListItem` in every `LawList` and then drill into `getRequirementsForListItem` per item. That's impractical for the filter presets we want on `/krav` (especially "needs_evidence" which needs a join on `RequirementEvidenceLink`, and "mine" which depends on effective-assignee resolution). This story ships the one query that makes Story 20.3 possible, and codifies the four filter presets so the UI doesn't own that business logic.

**Depends on:** Story 20.1 must land first (column exists and resolver is exported).

## Acceptance Criteria

**Action Shape:**

1. A new server action `getWorkspaceRequirements(input)` is exported from `app/actions/law-list-item-requirements.ts` (or a sibling file under `app/actions/`).
2. Input type:
   ```ts
   type GetWorkspaceRequirementsInput = {
     filter: 'all' | 'gaps' | 'mine' | 'needs_evidence'
     search?: string                // free-text, matches against krav.text (case-insensitive, trimmed)
     sort?: {
       field: 'updated_at' | 'law_name' | 'laglista_name' | 'is_fulfilled'
       direction: 'asc' | 'desc'
     }
     page?: { cursor?: string; take?: number } // keyset pagination; default take = 50, max 200
   }
   ```
3. Return type:
   ```ts
   type WorkspaceRequirementRow = {
     krav_id: string
     text: string
     comment: string | null
     is_fulfilled: boolean
     bevis_required: boolean
     effective_assignee: { user_id: string | null; is_inherited: boolean }
     evidence_count: number
     law_item_id: string
     law_id: string
     law_name: string
     laglista_id: string
     laglista_name: string
     updated_at: Date
   }
   type GetWorkspaceRequirementsResult = {
     rows: WorkspaceRequirementRow[]
     total_count: number       // matches current filter+search, for header chip badges
     next_cursor: string | null
   }
   ```

**Auth & Scoping:**

4. Wrapped in `withWorkspace(fn, perm)` using the same permission level as existing krav read actions (likely `workspace:read` — match the existing convention).
5. Workspace is resolved from the cookie context (no workspace slug in input).
6. Returns `{ rows: [], total_count: 0, next_cursor: null }` — never throws — when the workspace has no krav.

**Filter Semantics:**

7. `filter: 'all'` — every krav belonging to any `LawListItem` in any `LawList` of the active workspace.
8. `filter: 'gaps'` — `is_fulfilled = false`.
9. `filter: 'mine'` — `effective_assignee.user_id = current_user_id`, where effective assignee is resolved via `resolveEffectiveAssignee` (direct override OR inherited from the parent law item). Implementation note: either (a) compute effective assignee in SQL as `COALESCE(krav.responsible_user_id, list_item.responsible_user_id)` inside the query, or (b) fetch candidates and filter in JS — prefer (a) for performance.
10. `filter: 'needs_evidence'` — `bevis_required = true AND evidence_count = 0`. `evidence_count` is a count of related `RequirementEvidenceLink` rows for that krav.

**Search & Sort:**

11. `search` applies case-insensitive `contains` against `krav.text`, AND-combined with the active filter.
12. Default sort: `updated_at desc`. All four sort fields work both ascending and descending.
13. Sort is deterministic — a secondary `krav_id asc` tiebreaker is included so pagination is stable.

**Evidence Count:**

14. `evidence_count` is correct for every row — counted via `RequirementEvidenceLink` rows with the matching `requirement_id`. Single query (no N+1); use Prisma `_count` or a raw SQL aggregate.

**Effective Assignee:**

15. `effective_assignee` uses the `resolveEffectiveAssignee` helper from Story 20.1 — the ONLY place effective-assignee logic lives. No duplication inside this action.
16. For SQL-filter-path efficiency on `mine`, the `COALESCE` expression in the WHERE clause must produce the same result as the helper (asserted by test).

**Pagination:**

17. Keyset pagination uses the `(updated_at, krav_id)` pair as the cursor, base64-encoded, to match existing conventions.
18. `next_cursor` is `null` when there are no more rows under the current filter+search.
19. `take` defaults to 50, clamps to a max of 200.

**Performance:**

20. Page load on a seeded workspace of ~500 krav across ~50 laws returns in under 300ms on the baseline dev DB.
21. No N+1: `law_name`, `laglista_name`, `evidence_count`, `list_item.responsible_user_id` are all resolved in a bounded number of queries (ideally one).

## Dev Notes

**Resolver parity (the primary-risk guard):**

Epic 20 calls out modal/table drift on effective assignee as the primary risk. This action uses a SQL `COALESCE`-style `OR` in the `mine` branch for performance — but the canonical truth remains `resolveEffectiveAssignee` from Story 20.1. Task 5's parity test locks these in agreement. If the SQL approach ever diverges (e.g., nullability edge cases), the test breaks loudly rather than silently returning wrong "Mina krav" sets.

**Why keyset, not offset:**

The `/krav` page will be the kind of surface users keep open while editing — `LIMIT/OFFSET` pagination gets inconsistent when rows are updated between pages. Keyset on `(updated_at desc, krav_id asc)` stays stable even under concurrent writes and matches existing workspace-scoped list patterns in this codebase.

**Performance budget:**

The target workspace has ≤500 krav for the foreseeable horizon. A single `findMany` with `include` for `list_item.law_list + list_item.law + _count.evidence_links` is the entire plan — Prisma issues it as one SQL statement via LEFT JOINs and a correlated count. If profiling ever shows this is tight, the fallback is a raw SQL query — but we don't pre-optimize.

**Relation name verification:**

Before writing the query, run `grep "law_list\|law_list_item\|list_item.law" prisma/schema.prisma` to confirm exact relation names. Paths referenced above (`list_item.law_list.name`, `list_item.law.name`) are plausible but unverified.

**Out of scope:**

- No writes. CRUD stays in the existing krav actions.
- No route handler, no SWR hook — Story 20.3 owns that.

### Testing

- **Location:** colocate with the action file — e.g., `app/actions/workspace-requirements.test.ts` — following existing patterns for other workspace-scoped actions in `app/actions/*.test.ts`.
- **Framework:** Vitest + a test Prisma client pointing at a scratch DB (match existing test harness — check `vitest.config.ts` / test setup file).
- **Fixtures:** seed via existing factory helpers if present; otherwise inline `prisma.create` calls in `beforeEach`.
- **Patterns to match:** look at how existing aggregation actions are tested (`getDocumentListItems`, `getComplianceOverview`) and mirror their setup/teardown style.
- **Must pass:** all 12 sub-tests enumerated in Task 12, plus a smoke test asserting `withWorkspace` rejection when called without a workspace cookie.
