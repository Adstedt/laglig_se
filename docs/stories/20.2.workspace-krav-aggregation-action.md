# Story 20.2: Workspace Krav Aggregation Server Action

## Status

Approved

## Story

**As a** workspace user (compliance manager, responsible owner, or auditor),
**I want** a single server action that returns every kravpunkt in my workspace with the right filter applied (all / gaps / mine / needs evidence) plus a sibling action for chip-badge counts,
**so that** the new `/krav` page in Story 20.3 can render a cross-laglista overview without N+1 queries, client-side filtering gymnastics, or inventing new data-fetching patterns.

## Acceptance Criteria

### Primary aggregation action

1. A new server action `getWorkspaceRequirements(input)` is exported from `app/actions/workspace-requirements.ts` (new sibling file to `law-list-item-requirements.ts`, created because the new action introduces workspace-scoped joins and types that don't belong inside the krav-CRUD file).
2. Input type (Zod-validated, camelCase at the API boundary per the convention established in Story 20.1):
   ```ts
   type GetWorkspaceRequirementsInput = {
     filter: 'all' | 'gaps' | 'mine' | 'needs_evidence'
     search?: string
     sort?: {
       field: 'updated_at' | 'law_name' | 'laglista_name' | 'is_fulfilled'
       direction: 'asc' | 'desc'
     }
     cursor?: string // krav id of the last row on the previous page
     limit?: number  // default 50, clamp to max 200
   }
   ```
3. Return type:
   ```ts
   export interface WorkspaceRequirementRow {
     id: string
     text: string
     comment: string | null
     isFulfilled: boolean
     bevisRequired: boolean
     responsibleUserId: string | null
     effectiveAssignee: { userId: string | null; isInherited: boolean }
     evidenceCount: number
     lawItemId: string
     lawId: string
     lawName: string
     laglistaId: string
     laglistaName: string
     updatedAt: Date
   }
   type Result = ActionResult<{
     items: WorkspaceRequirementRow[]
     nextCursor: string | null
   }>
   ```
4. Action is wrapped in `withWorkspace(async (ctx) => { ... })` from `@/lib/auth/workspace-context`. Workspace ID is resolved from the cookie context via `ctx.workspaceId`; current user ID from `ctx.userId`. Unauthorized callers receive the same rejection shape as every other `withWorkspace`-wrapped action.
5. When the workspace has no kravpunkter matching the active filter, the action returns `{ success: true, data: { items: [], nextCursor: null } }` — never throws.

### Filter semantics

6. `filter: 'all'` — every krav whose parent `LawListItem` belongs to a `LawList` in the active workspace. No additional `where` beyond the workspace scope.
7. `filter: 'gaps'` — adds `is_fulfilled: false` to the `where`.
8. `filter: 'mine'` — returns krav whose **effective assignee** equals `ctx.userId`. Implementation uses a SQL-level `OR` clause mirroring `resolveEffectiveAssignee` from Story 20.1:
   ```ts
   where: {
     OR: [
       { responsible_user_id: ctx.userId },
       {
         responsible_user_id: null,
         list_item: { responsible_user_id: ctx.userId },
       },
     ],
   }
   ```
   The action MUST include a unit test asserting this filter's result set is identical to applying `resolveEffectiveAssignee` in JavaScript across a seeded fixture (parity guarantee — see Task 10).
9. `filter: 'needs_evidence'` — adds `bevis_required: true, evidence_links: { none: {} }` to the `where`. Uses the existing `RequirementEvidenceLink` relation introduced by Epic 17 / Story 17.18.

### Search, sort, pagination

10. `search` (trimmed, non-empty): adds `text: { contains: search, mode: 'insensitive' }` to the `where`, AND-combined with the active filter.
11. `sort` default: `{ field: 'updated_at', direction: 'desc' }`. All four sort fields must be honoured; `law_name` maps to `list_item.document.title`, `laglista_name` maps to `list_item.law_list.name`. Exact field name resolution is part of Task 2 — confirm against `prisma/schema.prisma` and `docs/architecture/9-database-schema.md#9.6.1` before writing the query.
12. Pagination is **cursor-based using the krav `id`**, mirroring `getWorkspaceActivity` (`app/actions/workspace-activity.ts:80–94`):
    ```ts
    take: limit + 1,
    ...(cursor ? { cursor: { id: cursor }, skip: 1 } : {}),
    ```
    Fetch `limit + 1`, slice to `limit`, set `nextCursor = items[items.length - 1]?.id ?? null` when the sliced count indicates more pages exist. `nextCursor` is `null` when there are no more rows.
13. `limit` defaults to `50`; the Zod schema clamps it to `z.number().int().min(1).max(200)`.

### Evidence count & effective assignee

14. `evidenceCount` is sourced via Prisma's `_count` relation aggregation (`_count: { select: { evidence_links: true } }`) — no N+1 loop. The parent-workspace-scope query must resolve `evidenceCount`, `list_item.document.title`, `list_item.document.id`, `list_item.law_list.id`, `list_item.law_list.name`, `list_item.responsible_user_id` in a single Prisma statement.
15. `effectiveAssignee` is populated by calling the exported `resolveEffectiveAssignee(krav, listItem)` helper from `app/actions/law-list-item-requirements.ts` (shipped in Story 20.1) on every row after fetch. This is the **single source of truth** for effective-assignee logic across the modal (Story 20.1) and the workspace overview (this story) — it is the primary-risk guard called out in Epic 20.

### Sibling counts action

16. A sibling server action `getWorkspaceRequirementCounts()` is exported from the same file (`app/actions/workspace-requirements.ts`). Input: none. Output: `ActionResult<{ all: number; gaps: number; mine: number; needs_evidence: number }>`. Powers Story 20.3's filter chip badges — `search` does not apply here (counts reflect the unfiltered-by-search view of each preset).
17. `getWorkspaceRequirementCounts()` runs the four `count` queries via `Promise.all` and is wrapped in the same `withWorkspace` as above.

### Performance

18. P95 execution time ≤ **300ms** for `getWorkspaceRequirements` against a seeded workspace of ~500 krav across ~50 laws on the baseline dev DB. [Source: architecture/22-performance-architecture.md#22.2.1 — API Response (Database) P95 target]
19. Total Prisma queries per call: **≤ 3** (one `findMany` with `include` + joined `_count`; the sibling counts action issues four `count` queries in parallel, still within the §22.2.3 "≤ 10 database queries per request" budget).
20. The API payload for a default page (limit 50) stays under **100KB** on realistic data. [Source: architecture/22-performance-architecture.md#22.2.3]

**Shared where-clause builder (prescribed design):**

```ts
import type { Prisma } from '@prisma/client'

function buildRequirementWhere(
  ctx: { workspaceId: string; userId: string },
  filter: GetWorkspaceRequirementsInput['filter'],
  search?: string,
): Prisma.LawListItemRequirementWhereInput {
  const workspaceScope: Prisma.LawListItemWhereInput = {
    law_list: { workspace_id: ctx.workspaceId },
  }

  const base: Prisma.LawListItemRequirementWhereInput = {
    list_item: workspaceScope,
    ...(search ? { text: { contains: search, mode: 'insensitive' } } : {}),
  }

  switch (filter) {
    case 'all':
      return base
    case 'gaps':
      return { ...base, is_fulfilled: false }
    case 'mine':
      return {
        ...base,
        OR: [
          { responsible_user_id: ctx.userId },
          {
            responsible_user_id: null,
            list_item: {
              ...workspaceScope,
              responsible_user_id: ctx.userId,
            },
          },
        ],
      }
    case 'needs_evidence':
      return {
        ...base,
        bevis_required: true,
        evidence_links: { none: {} },
      }
  }
}
```

Notes on this design:
- `workspaceScope` is typed as `Prisma.LawListItemWhereInput` and reused cleanly in the `mine` branch's nested list-item filter — no `as object` cast, no risk of `exactOptionalPropertyTypes` / `strictNullChecks` complaints.
- `search` is spread conditionally via `...(search ? { ... } : {})` so the `Prisma.LawListItemRequirementWhereInput` never carries an `undefined` property (important under `exactOptionalPropertyTypes`). [Source: architecture/17-coding-standards.md#17.2]
- The exhaustive `switch` with the typed enum prevents silent drift if a new filter is added later.

This helper is the single definition both Tasks 3 and 4 consume.

### Project Structure Notes

- Epic 20 references `withWorkspace(fn, perm)` as if a permission argument is passed. Story 20.1 and the codebase confirm the wrapper is invoked as `withWorkspace(async (ctx) => { ... })` with no permission argument; permission checks, when needed, happen inline inside the callback (see `lib/rbac.ts` per [Source: architecture/11-backend-architecture.md#11.4.3]). For this story, **read access is scoped by workspace membership alone** — consistent with `getWorkspaceActivity` and `getDocumentListItems`, both of which rely on `withWorkspace` alone without an extra permission check. If a future permission model (e.g., AUDITOR read-only) is introduced, the action can be extended without changing the signature.
- The new file `app/actions/workspace-requirements.ts` was chosen over co-locating in `law-list-item-requirements.ts` because: (1) this is a cross-workspace aggregation, not per-list-item CRUD; (2) the new types (`WorkspaceRequirementRow`, input/output shapes) are sizeable and would bloat the existing file; (3) it matches the precedent of `workspace-activity.ts` being separate from the entity-CRUD actions it aggregates over.
- The PO draft (v0.1) specified base64-encoded composite `(updated_at, krav_id)` keyset cursors. The codebase does NOT use that pattern. This SM rewrite aligns to the simpler `id`-cursor pattern established by `getWorkspaceActivity` (Story 6.10). Cursor stability under concurrent writes is preserved because `id` is unique and `orderBy` always includes a deterministic tiebreaker (Task 3).
