# Story 24.2: Import — file upload + parser + paste fallback

## Status

Draft

## Story

**As a** workspace member who already has a curated law list in Excel, CSV, or copy-paste form,
**I want** to upload my file (or paste raw rows) at the start of an import flow and see Laglig automatically detect which columns mean what (titel, SFS-nummer, område, lagansvarig, kommentar),
**so that** my source data lands in the import pipeline as structured rows ready for matching, without me manually mapping every column.

This is the **upload + parsing layer only** — the raw rows land in `LawListImport.status = UPLOADED` with `LawListImportRow` rows persisted, but **no matching is run** (Story 24.3) and **no review UI exists** (Story 24.4). The story validates that we can ingest the five real-world source types (Notisum export, Lex.nu export, consultant Excel, internal spreadsheet, paste input) into normalized rows.

## Acceptance Criteria

### Dependencies

1. **Add `xlsx` (SheetJS) and `papaparse` to `package.json`.** Both MIT-licensed. Use latest stable versions. Run `pnpm install` and commit `pnpm-lock.yaml`. **Decision:** SheetJS community edition is sufficient — we don't need the Pro/streaming variant for sub-5MB files.

### Parser library

2. **New file `lib/import/parser.ts`** — exports four pure functions:
   ```ts
   export function parseExcel(buffer: Buffer | ArrayBuffer): ParsedRow[]
   export function parseCsv(text: string): ParsedRow[]
   export function parsePaste(text: string): ParsedRow[]
   export function detectColumns(rows: ParsedRow[]): ColumnMapping
   ```
   - `ParsedRow` shape: `{ index: number, raw: Record<string, string>, columns: string[] }` — `raw` is the source row keyed by detected column name; `columns` is the header order.
   - `ColumnMapping` shape: `{ titel: string | null, sfs_nummer: string | null, omrade: string | null, lagansvarig: string | null, kommentar: string | null, _confidence: Record<string, number> }` — values are source column names; `_confidence` is per-mapping 0.0–1.0 score from the heuristic.

3. **`parseExcel` reads the first sheet, first 1000 data rows.** Header is row 1. If the workbook has multiple sheets, log a warning + only process the first. Hard cap: 1000 rows ignored beyond row 1001 (returned in a `truncated: true` flag — extend `ParsedRow[]` return to a `{rows, truncated}` shape if needed).

4. **`parseCsv` auto-detects delimiter** (`,` / `;` / `\t`). Use Papa Parse's `delimiter: ''` auto-detect. Trim header whitespace. Handle UTF-8 BOM stripping (Excel-exported CSVs).

5. **`parsePaste` handles two paste shapes:**
   - **TSV** (Excel copy/paste): tab-separated, newline-separated rows, header in row 1
   - **Single column** (titles only): one item per line, no delimiter, header inferred as just "titel"
   Sniff TSV via tab-character presence on row 1. Fallback = single-column.

6. **`detectColumns` heuristic** — for each canonical field (titel/sfs_nummer/omrade/lagansvarig/kommentar), score each source column on three signals.

    **Worked example:** given a Notisum-export header row `["SFS-nr", "Lagens namn", "Rättsområde", "Egen status", "Kommentar"]` and 10 sample data rows, `detectColumns` should return:
    ```ts
    {
      titel: "Lagens namn",        // header regex /lag|namn/ → 0.5; content matches Swedish-law pattern → +0.4 = 0.9
      sfs_nummer: "SFS-nr",        // header regex /sfs/ → 0.5; content matches /\d{4}:\d+/ → +0.5 = 1.0
      omrade: "Rättsområde",       // header regex /rättsområde/ → 0.5; short content → +0.2 = 0.7
      lagansvarig: null,           // no candidate scored above 0
      kommentar: "Kommentar",      // header regex /kommentar/ → 0.5; long-content threshold not always met → 0.5
      _confidence: { titel: 0.9, sfs_nummer: 1.0, omrade: 0.7, lagansvarig: 0, kommentar: 0.5 }
    }
    ```
   - **Header name match** (regex per field):
     - titel: `/^(titel|namn|lag|lagstiftning|title|name)$/i` → 0.5
     - sfs_nummer: `/^(sfs|sfs[-_ ]?(nummer|nr|id)|nummer|kod)$/i` → 0.5
     - omrade: `/^(område|omrade|kategori|rättsområde|category|area)$/i` → 0.5
     - lagansvarig: `/^(lagansvarig|ansvarig|owner|responsible)$/i` → 0.5
     - kommentar: `/^(kommentar|note|notes|anteckning|description)$/i` → 0.5
   - **Content match** (sample first 10 rows of each column):
     - titel: > 50% of cells contain a Swedish-law-like phrase (e.g., contains "lag" / "förordning" / "förordningen" / "föreskrift" / "kungörelse" or matches `/^[A-ZÅÄÖ]/`) → +0.4
     - sfs_nummer: > 50% of cells match `/\b\d{4}:\d{1,5}\b/` → +0.5 (high signal — SFS pattern is unambiguous)
     - omrade: > 50% of cells are short (≤ 30 chars) AND non-numeric → +0.2
     - lagansvarig: > 30% of cells contain a name pattern (capitalized first/last) or email-like → +0.3
     - kommentar: > 30% of cells are >50 chars → +0.2
   - Pick the highest-scoring column per canonical field. Tie-break: leftmost column wins.
   - Confidence = clamped sum of header + content score (max 1.0).
   - Threshold: if all candidates scored 0, leave the field as `null` — user will be asked to map manually in 24.4 (a future enhancement; not in this story).

### Server actions

7. **Implement `createImport`** (skeletoned in 24.1):
   ```ts
   createImport({ filename, source_type })
   → returns { success: true, data: { importId } }
   → creates `LawListImport` row with status=UPLOADED, row_count=0, column_mapping={}, blank everything else
   ```
   Records activity-log: `law_list_import.created`.

8. **Implement `parseImportFile`** (skeletoned in 24.1):
   ```ts
   parseImportFile({ importId, fileBuffer })  // fileBuffer base64-encoded
   → returns { success: true, data: { rowCount, columnMapping } }
   ```
   Steps:
   - Verify import exists, belongs to ctx.workspaceId, status=UPLOADED
   - Decode base64 buffer
   - Dispatch on `import.source_type`: 'xlsx' → `parseExcel`, 'csv' → `parseCsv`, 'paste' → `parsePaste(buffer.toString('utf8'))`
   - Run `detectColumns` on parsed rows
   - In a Prisma transaction:
     - Insert all parsed rows into `LawListImportRow` (use `prisma.lawListImportRow.createMany` for bulk; `skipDuplicates: false`)
     - Update `LawListImport.row_count` and `column_mapping`
     - Status remains `UPLOADED` (not `MATCHING` — that flips in 24.3 when matching fires)
   - Return row count + detected mapping

9. **Error handling.**
   - Unparseable file → set `status=FAILED`, `error_message='Vi kunde inte läsa filen. Se till att den är .xlsx, .xls eller .csv och under 5 MB.'`, return error response.
   - Empty file (0 rows) → `status=FAILED`, `error_message='Filen verkar vara tom — vi hittade inga rader att importera.'`
   - More than 1000 rows → still succeed but return `truncated: true` in payload + `error_message='Vi importerade de första 1000 raderna. Kontakta oss om du behöver importera fler.'` (set as a warning, not a failure — let user proceed).

### Upload step UI

10. **New file `components/features/onboarding-modal/import-upload-step.tsx`** — client component used by Story B.1's import path AND by Story 24.6's `/laglistor/skapa` rewire. Reuses prototype frame ②b styling.

11. **UI elements:**
    - Header: "Ladda upp er befintliga lista" + sub-copy
    - Toggle: `Fil` / `Klistra in` (radix Tabs primitive, the small inline variant)
    - Fil tab: hand-rolled dropzone using HTML5 drag-and-drop events (`onDragOver`, `onDragLeave`, `onDrop`) layered on a `<div>` with a hidden `<input type="file" accept=".xlsx,.xls,.csv">` for click-to-select fallback. Visual: `Cloud-upload` lucide icon, "Dra & släpp eller välj från dator", ".xlsx, .xls, .csv — max 5 MB". **No `components/ui/dropzone.tsx` exists in this repo** — do not import one; build inline. ~30 lines of component code.
    - Klistra in tab: textarea (large, monospace, "Klistra in raderna här — en lag per rad eller kopiera direkt från Excel")
    - "Vi känner igen dessa kolumner" hint card with neutral pills (Lag/Titel, SFS-nummer, Område, Lagansvarig, Kommentar) — shown on Fil tab only
    - "Ladda ner mall (.xlsx)" link to `/templates/laglista-import-mall.xlsx` (static asset added to `public/templates/`)
    - Submit CTA: "Importera" (disabled until file selected or paste >0 chars)

12. **Submit flow:**
    - Reads file as ArrayBuffer (drag-drop) or string (paste)
    - Calls `createImport({filename, source_type})` server action → gets `importId`
    - Base64-encodes the buffer, calls `parseImportFile({importId, fileBuffer})`
    - On success: `router.push(\`/laglistor/skapa/${importId}/granska\`)` using `useRouter` from `'next/navigation'` (NOT `'next/router'` — App Router only). Route exists as a stub in 24.4; for now just navigate to it — 404 is acceptable for this story since 24.4 builds the page.
    - On error: surface `error_message` via `toast.error` (sonner)

13. **Permission gate.** Only `tasks:edit` workspace members can upload. Server action enforces this via `withWorkspace(cb, 'tasks:edit')`; UI doesn't need to gate explicitly because the path-choice modal (B.0) already gates on first-run.

    - `consultant-excel-sample.xlsx` — ~50 rows, columns: Lagstiftning, Kategori, Lagansvarig, Anteckning (no SFS column — title-only matching test)

### Compatibility & out-of-scope guards

17. **No matching logic in this story.** `LawListImport.status` stays at `UPLOADED` after `parseImportFile`. Story 24.3 flips to `MATCHING`.

18. **No review UI.** The `/laglistor/skapa/[importId]/granska` route is a 404 after 24.2 ships (24.4 builds it). UI redirects to it but the user will see a Next.js 404 — acceptable since 24.4 lands before this is user-facing.

19. **Existing wizard onboarding flow unchanged.** This story does not touch `app/onboarding/_components/`.

20. **Bundle-size check.** `xlsx` is ~600 KB minified — acceptable for the import flow but should not bloat dashboard JS. **Verification method:** add `@next/bundle-analyzer` if not present, run `ANALYZE=true pnpm build`, inspect `.next/analyze/client.html` to confirm `xlsx` does NOT appear in any chunk loaded by `/dashboard` or `/laglistor` non-import paths. The parser must be imported only from server-action code (`app/actions/law-list-import.ts` is server-only by `'use server'` directive), never from a client component. Add an ESLint `no-restricted-imports` rule scoping `xlsx` to `app/actions/**` and `lib/import/**` if leakage is detected.

**Existing patterns to follow:**
- **File upload pattern** — see `components/features/files/file-upload.tsx` (if exists in this repo) or the `WorkspaceFile` upload flow for chunked-upload pattern. For Epic 24 we do NOT need chunking — files are ≤ 5 MB and base64-over-server-action is fine.
- **Server-action progress reporting** — keep it simple; this story is sub-second on real files. No polling needed.
- **Tabs primitive** — `components/ui/tabs.tsx`, the small inline variant used in `chat-message.tsx` for sub-tab navigation.
- **Toast errors** — `sonner` `toast.error(...)` per existing convention in `hem-page.tsx`.
- **Activity-log integration** — see Story 24.1 AC 11 for the entity type + action constant; add the `created` action here.

**Key decisions baked in:**
- **No Office 365 / Microsoft Graph integration** — out of scope. We accept the file as a plain upload, not an OAuth-linked Excel pull.
- **base64-over-server-action** is the upload mechanism — works around Next.js Server Actions FormData edge cases for binary uploads. Decoded server-side. Limit enforced via Zod schema (max 5 MB ≈ 6.7 MB base64).
- **Paste tab is intentionally simple** — no column detection beyond title-only or TSV. Users with complex pastes are told to save as Excel first.
- **No live column-mapping override UI in this story** — if `detectColumns` mismaps, the user goes through 24.4's review with possibly weird mappings. A future enhancement would let users override mapping pre-matching; defer until customer feedback warrants.

**Out of scope explicitly:**
- Matching logic (Story 24.3)
- Review UI (Story 24.4)
- Manual column-mapping override
- Multi-sheet Excel handling beyond the first sheet
