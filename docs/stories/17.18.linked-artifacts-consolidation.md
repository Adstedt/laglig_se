# Story 17.18: Linked Artifacts Consolidation & Bevis-Required Semantics

## Status

Ready for Review

## Story

**As a** workspace member working in the legal document modal,
**I want** a single consolidated "Länkade filer & dokument" surface that unifies every way an artifact can be linked to a law list item (direct, via kravpunkter, via tasks), and a way to explicitly mark kravpunkter that do *not* require evidence,
**so that** I (and our AI agent) can see everything relevant to a law in one deduped place — and the "missing evidence" signal becomes meaningful instead of noisy.

## Acceptance Criteria

### A. Activity pane cleanup — logs only

1. `components/features/document-list/legal-document-modal/activity-tabs.tsx` is reduced to three tabs: **Alla · Kommentarer · Historik**. The `Uppgifter`, `Bevis`, and `Dokument` tabs are removed. (User explicitly named `Bevis` and `Dokument` in scoping; `Uppgifter` is included here because the `TasksAccordion` in `left-panel.tsx` is the canonical task surface and the Activity-tab variant duplicated it — confirmed in PO validation as a logical extension of the "Activity = logs only" principle. If the user prefers a tighter scope, drop the `Uppgifter` removal back into a follow-up story.)
2. The `EvidenceTab` and `DocumentsTab` components (`evidence-tab.tsx`, `documents-tab.tsx`) are deleted — their functionality is subsumed by the new consolidated panel (see section B). The `TasksTab` component (`tasks-tab.tsx`) is also deleted; the TasksAccordion in `left-panel.tsx` remains the single surface for tasks (no behavior loss — `TasksTab` duplicated it).
3. No regression in the three retained tabs: `ActivityFeed` (Alla), `CommentsTab` (Kommentarer), `HistoryTab` (Historik) render and function as before.

### B. Consolidated "Länkade filer & dokument" panel (left panel accordion)

4. A new accordion section **"Länkade filer & dokument"** is added in `left-panel.tsx`, placed between `TasksAccordion` and `ActivityTabs`. Header shows total unique count, e.g. `Länkade filer & dokument (7)`.
5. A new server action `getLinkedArtifactsForListItem(listItemId)` lives at `app/actions/linked-artifacts.ts`, UNIONs all five link-pathway tables for the list item, deduplicates by artifact identity (`(kind, id)` where `kind ∈ {'file','document'}`), and returns a flat `LinkedArtifact[]` with **back-reference metadata** describing every way the artifact is linked:
   ```ts
   type LinkedArtifact = {
     kind: 'file' | 'document'
     id: string
     // File fields (when kind='file')
     filename?: string
     mimeType?: string | null
     fileSize?: number
     // Document fields (when kind='document')
     title?: string
     documentType?: string  // e.g. 'POLICY', 'RISK_ASSESSMENT'
     status?: string        // e.g. 'DRAFT', 'APPROVED'
     versionNumber?: number
     // Back-references (at least one is always non-empty — otherwise the artifact wouldn't be in the list)
     directLink: boolean               // linked via FileListItemLink or WorkspaceDocumentListItemLink
     requirements: Array<{ id: string; text: string }>  // linked via RequirementEvidenceLink — list of kravpunkter this is bevis for
     tasks: Array<{ id: string; title: string }>        // linked via FileTaskLink or WorkspaceDocumentTaskLink where task.list_item_id = this
   }
   ```
   Each artifact appears **exactly once** in the array regardless of how many pathways connect it.
6. Artifact rows visually distinguish files vs styrdokument:
   - **Files:** `Paperclip` icon (existing `getFileIcon()` from `components/features/files/file-dropzone.tsx`), filename, filesize, neutral muted styling.
   - **Styrdokument:** `FileText` icon (filled/primary tint), document title, `DOCUMENT_TYPE_LABELS` badge (from `documents-tab.tsx` — move constant to a shared location), `DocumentStatusBadge` (existing component), version number `v{n}`.
   Type difference must be scannable at a glance (not only via hover/tooltip).
7. Each artifact row shows its back-reference badges inline — compact chips such as:
   - `Direktlänkad` (when `directLink=true`)
   - `Bevis · {requirement.text}` (one chip per kravpunkt in `requirements`; text truncated to ~24 chars with full text on tooltip)
   - `Uppgift · {task.title}` (one chip per task; same truncation)
   If the chip count would exceed 3 inline, collapse overflow under `+N fler` with a popover listing the rest.
8. A **filter bar** above the list offers: `Alla` (default) · `Bevis` (only artifacts with non-empty `requirements`) · `Direktlänkade` (only `directLink=true`) · `Via uppgifter` (only non-empty `tasks`). Filters are additive-OR within the active filter (i.e., selecting "Bevis" shows any artifact where `requirements` is non-empty, regardless of other links).
9. A **type toggle** (checkbox pair or segmented control) with independently toggleable `Filer` and `Styrdokument` (both on by default) filters by artifact `kind`.
10. Empty states:
    - Zero linked artifacts overall: `"Inga länkade filer eller dokument än"` + two ghost buttons (`Ladda upp fil`, `Länka styrdokument`) that open the existing `FilePickerModal` / `DocumentPickerModal` and link on selection via the direct pathway.
    - Zero results after filter: `"Inga matchande artefakter"` + `Återställ filter` link.
11. Row actions (visible on hover or always on mobile):
    - **Files:** download, unlink-from-this-list-item (only when `directLink=true`), open file detail.
    - **Styrdokument:** open document editor link (existing `href={/workspace/styrdokument/${doc.id}/edit}` pattern), unlink (only when `directLink=true`).
    - The unlink affordance is hidden/disabled when the artifact is linked **only** through kravpunkter or tasks — in that case a tooltip directs the user to remove the link at its source ("Länkad via kravpunkt X — ta bort länken där"). This prevents accidental loss of context and matches the item-first mental model: removing here would require deciding *which* pathway to sever.
12. Add actions at the top of the panel: `Ladda upp fil` (uploads + direct-links via existing `uploadFileAndLinkToListItem`), `Länka fil` (opens `FilePickerModal`, links via `linkFilesToListItem`), `Länka styrdokument` (opens `DocumentPickerModal`, links via `linkDocumentToListItem`). These all create *direct* links — the kravpunkter/task pathways keep their existing in-context add UX (no change to those surfaces).
13. Data-fetching: use SWR with cache key `linked-artifacts:${listItemId}`. Mutations in any of the existing surfaces (kravpunkter checklist evidence link/unlink, task file/document links) must invalidate this cache so the panel stays live. Use `mutate('linked-artifacts:${listItemId}')` at the end of those existing server actions (or within the existing revalidation paths).

## Dev Notes

### Provenance

This story continues the post-epic-17 kravpunkter-and-linking thread started by Stories 17.16 and 17.17. It is **not** in the original Epic 17 PRD (which ended at 13 stories, 17.1–17.13). It was scoped in a PO/SM validation conversation on 2026-04-14 responding to user feedback that document/evidence linking had become fragmented across multiple surfaces (Activity tabs, right-panel Bevis card, kravpunkter chips) and that "missing bevis" was not always a valid compliance signal. The story consolidates the artifact surface (Option C: item-first, deduplicated with back-references) and introduces explicit `bevis_required` semantics per kravpunkt so the gap signal becomes meaningful. No PM re-scoping required — organic extension, confirmed via this session's validation dialogue.

Note on story numbering: the label "17_18" appears in the migration name `20260414000000_story_17_18_business_context_attribution` **from Story 17.17's v1.6 change log entry** — that was an internal plan-mode label that got folded back into 17.17, not a filed story. This is the first **filed** 17.18. A later timestamp (`20260414120000`) is used for the new migration to avoid collision and to make the chronological order clear.

- **Workspace isolation:** the new `getLinkedArtifactsForListItem` MUST verify the list item belongs to the caller's active workspace — use the same `withWorkspace()` helper every other read action uses.

- **Security:** unlinking from the consolidated panel only works for direct links; indirect links are hidden/disabled. This is deliberate per AC 11 — prevents accidental context loss.
