## Story

**As a** vård- och omsorgs compliance owner (and the catalog itself),
**I want** Socialstyrelsen's consolidated föreskrifter (the legacy **SOSFS** + the shared **HSLF-FS** series) ingested as first-class `AGENCY_REGULATION` documents,
**so that** they resolve in laglistor, kravpunkter, search and RAG, link from marketing pages, and feed future amendment notifications through the existing pipeline — closing the single biggest catalog gap for the vård-omsorg segment (e.g. `SOSFS 2011:9`, the ledningssystem-föreskrift that is the backbone of every vård laglista).

## Scope note

This story is **Phase 1 — ingestion** (get the consolidated set into the catalog, correctly attributed, with amendment *metadata* captured). **Forward amendment monitoring** (emit `ChangeEvent`s when a consolidated page changes) is **Phase 2**, drafted as a separate story (see AC 9 / Future Work). **Per-version historik reconstruction from amendment PDFs is explicitly out of scope** and deferred indefinitely (the consolidation already merges them; the PDFs remain available to backfill later). [Decision recorded in-session 2026-06-15.]

### Ingestion (HTML intake — new front-end, existing pipeline tail)

3. A registry of the consolidated föreskrifter is produced via a **headless enumeration** of the consolidated listing (the listing is JS-rendered; a plain `fetch` under-counts — verified: static HTML showed ~55 of an estimated ~76). Each entry carries `{ documentNumber, title, sourceUrl, regulatoryBody, agencyPrefix }`. [Source: in-session investigation 2026-06-15; robots.txt allows `/kunskapsstod-och-regler/`, disallows `/sok/`, `/_api/*`, `/*.xml$`]
4. An **HTML-intake ingester** ingests each consolidated page through the existing agency pipeline tail, proven end-to-end on SOSFS 2011:9 (`scripts/test-ingest-sosfs.ts`):
   - fetch page (individual pages are **server-rendered** — plain GET works) → cheerio-extract `#main-content` (strip nav/menu/breadcrumb chrome)
   - **LLM-normalize** extracted HTML → canonical `<article class="legal-document">` (the HTML→HTML analog of the AFS PDF→HTML step), reusing `AGENCY_REGULATION_SYSTEM_PROMPT`/`AGENCY_DEFAULT_MODEL`. [Source: lib/agency/agency-regulation-prompt.ts:16,294]
   - `validateLlmOutput(raw, docNumber)` → `htmlToMarkdown` / `htmlToPlainText` / `parseCanonicalHtml({documentType:'AGENCY_REGULATION'})`. [Source: lib/sfs/llm-output-validator.ts:68; lib/transforms/html-to-markdown.ts:43,503; lib/transforms/canonical-html-parser.ts:46]
   - **linkify** via `buildSlugMap()` + `linkifyHtmlContent(html, slugMap, docNumber)`, store the linkified result as `html_content` (also strips the stray `<html><body>` wrapper). [Source: lib/linkify/build-slug-map.ts:25; lib/linkify/linkify-html.ts:83]
   - upsert `LegalDocument` keyed on `document_number` (`AGENCY_REGULATION`, `ACTIVE`, slug via `generateAgencySlug`). [Source: scripts/ingest-agency-pdfs.ts:330-357; lib/agency/agency-pdf-registry.ts:2470]
   - idempotent (upsert), `--skip-existing`/`--force`/`--filter`/`--dry-run` flags consistent with `ingest-agency-pdfs.ts`. [Source: scripts/ingest-agency-pdfs.ts:9-17]
   - **Fail-soft:** a doc that truncates (`stop_reason === 'max_tokens'`) or fails `validateLlmOutput` is recorded to a **failures manifest** (docNumber + reason) and the batch **continues** — one bad doc never aborts the run. [Source: scripts/ingest-agency-pdfs.ts:247-251 (max_tokens warn); lib/sfs/llm-output-validator.ts:68]

4a. **Large/failed-doc handling is DEFERRED to an end-of-run remediation pass, not pre-built** (PO decision 2026-06-15): the bulk run completes regardless; only **if the failures manifest is non-empty** do we remediate the affected docs — first option being chunked / per-chapter LLM normalization (the AFS `callChunked` pattern). If the failed set is large or non-trivial, remediation splits to a follow-up story. Rationale: the test doc (28 §) was small; let real failures reveal which (if any) HSLF-FS docs actually exceed the 64k output ceiling rather than engineer chunking speculatively. [Source: scripts/ingest-afs-regulations.ts (callClaudeChunked / per-chapter split precedent)]
5. After upsert, each doc is **chunked + context-prefixed + embedded** via `syncDocumentChunks(doc.id)` (it gates on `content_type ∈ {SFS_LAW, AGENCY_REGULATION}` and fires prefixes+embeddings when `markdown_content` is present), and its **`search_vector`** is set (raw SQL; note `legal_documents.id` is `text`, **no `::uuid` cast**). [Source: lib/chunks/sync-document-chunks.ts:22,118; scripts/backfill-search-vectors.ts:45]

### Attribution + provenance

6. **Multi-publisher attribution:** `regulatory_body` is resolved **per-document** for HSLF-FS (shared series across ~8 agencies; publisher is NOT derivable from the prefix). For this story scope, all targets are Socialstyrelsen-issued; rows from other co-signatories are out of scope (they live on other authority sites). [Source: in-session investigation; epic intent]
7. **Amendment lineage metadata captured** at ingest into `metadata`: amendment numbers + dates from the page's amendment cards, `consolidated_through` (the "Ändrad: t.o.m. …"), and the **authoritative grund-/ändrings-PDF URLs**. The current extractor returns 0 PDF links because the cards sit outside `#main-content` — **broaden the selector (or headless-render) to capture them.** No amendment-PDF *content* is parsed/ingested (deferred). [Source: scripts/test-ingest-sosfs.ts (PDF-provenance gap noted); in-session decision]

### Surfacing + completeness

Ingested föreskrifter resolve everywhere a catalog doc is expected, with **no schema/route changes** beyond AC1: render at `/foreskrifter/<slug>`; resolve in marketing `relatedCatalogLaws` by `document_number` (ACTIVE exact-match) so vård-omsorg pages' SOSFS/HSLF-FS links resolve and the build's `marketing-link-warnings.txt` stays empty; addable to laglistor via `LawListItem.document_id`. ### Source anatomy (Socialstyrelsen konsoliderade föreskrifter)

- **Consolidated text lives only in HTML** on the page (Episerver/Optimizely CMS, server-rendered, entity-encoded). There is **no single consolidated PDF** — PDFs are the per-instrument diff chain (grund + each ändring). → ingest the HTML for the body; reference the PDFs for provenance. [Verified in-session]
- **Listing page is JS-rendered** (enumeration needs headless); **individual pages are server-rendered** (plain GET works for ingest). [Verified: `curl` got the body for SOSFS 2011:9; listing static fetch under-counted.]
- **No RSS/API/usable sitemap**; `robots.txt` allows the content path, disallows `/sok/`, `/_api/*`, `/*.xml$`. → discovery is scrape-based.
- Primary key = document number (`SOSFS 2011:9` / `HSLF-FS 2022:30`); slugs are NOT algorithmically derivable (must crawl/curate).
- HSLF-FS is a **shared series across ~8 agencies** → publisher per-document; this story = Socialstyrelsen-issued only.

### Data model (verified against `main`, NOT the stale epic docs)

- Agency regs are `LegalDocument` rows, `content_type = AGENCY_REGULATION` (the enum already exists; **no `AgencyRegulation` satellite model** — the epic doc proposing one is stale). [Source: prisma/schema.prisma#LegalDocument, #ContentType]
- Full text stored as `html_content` (SSOT, linkified), `markdown_content` + `full_text` (derived), `json_content` (canonical). Authority/consolidation lived only in `metadata` until this story's columns. [Source: prisma/schema.prisma:528-540]
- `document_number` is the unique join key; `lib/import/document-number.ts:78` already parses `SOSFS`/`HSLF-FS`. Catalog links match exact `document_number`, `ACTIVE` only. [Source: lib/marketing/catalog-link.ts:35-102]
- Chunks/embeddings live in `content_chunks` (`ContentChunk`), `source_type='LEGAL_DOCUMENT'`, per-chunk `embedding vector(1536)` (OpenAI text-embedding-3-small) + LLM `context_prefix` (Haiku). The whole-doc `LegalDocument.embedding` column is unused. [Source: lib/chunks/embed-chunks.ts:11; lib/chunks/sync-document-chunks.ts]

### Reuse map (build on the generalized agency-PDF ingester)

`scripts/ingest-agency-pdfs.ts` + `lib/agency/agency-pdf-registry.ts` (Story 9.2) already generalize the AFS flow for MSBFS/NFS/etc. The SOSFS ingester is that flow with the **PDF intake swapped for HTML intake** (fetch+cheerio+LLM-normalize instead of PDF→Claude); everything from `validateLlmOutput` onward is identical. `contentHash` change-detection already exists in `AgencyPdfMetadata` for Phase 2. [Source: scripts/ingest-agency-pdfs.ts; lib/agency/agency-pdf-registry.ts]

### Monitoring fit (Phase 2 — design only here)

Downstream is source-agnostic: a global `ChangeEvent` (`AGENCY_REGULATION`/`AMENDMENT`) flows through notify (`resolveAffectedRecipients` is `document_id`-only) and `ChangeAssessment` unchanged. The SFS-coupled pieces (discover crawler, the `amendment_documents.sfs_number` enrich join, `SFS_LAW`-gated historik) do NOT apply — Phase 2 builds a Socialstyrelsen-specific detector that emits the ChangeEvent; forward diffs come free from comparing the new consolidated text to the prior ingest snapshot (like `sync-sfs-updates`). [Source: in-session pipeline map; app/api/cron/notify-amendment-changes; lib/notifications/recipient-resolution.ts]
