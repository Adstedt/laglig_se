## Story

**As** the compliance agent (and especially the read-mostly AUDITOR persona and the upcoming `prepare_audit` / `ledningens_genomgang` skills),
**I want** to discover the workspace's lagefterlevnadskontroll cycles (`list_cycles`), read one cycle's full state with lazy handles to its items/findings/reports/linked tasks (`get_cycle`), read one finding's full state including closure metadata (`get_finding`), and link an agent-proposed corrective task back to its finding (`create_task` + `findingId`),
**so that** I can answer the single most important auditor question — *"visa er senaste lagefterlevnadskontroll"* — from the workspace's actual cycle history (aggregate coverage across rolling partial cycles, open findings, `law_change_cutoff_date` diffs), and so that corrective tasks I propose in audit-prep triage register as the finding's åtgärd instead of becoming orphans.

### `list_cycles` (discovery — the A3 entry point)

5. New `'read'` tool `list_cycles` in `lib/agent/tools/list-cycles.ts`, factory `createListCyclesTool(workspaceId)`. Input (all optional): `{ status?: 'PLANERAD'|'PAGAENDE'|'AVSLUTAD', auditType?: 'INTERN'|'EXTERN', lawListId?: string, completedAfter?: string (ISO date — filters sealed_at >= date), limit?: number (1-50, default 20) }`. Workspace-scoped `where: { workspace_id, deleted_at: null, …filters }`. Returns the **true `count`** (separate `count` query on the same `where`) + a capped list, ordered `sealed_at desc` (nulls last) then `scheduled_start desc`. Tool description states the positive-zero rule ("0 kontroller är ett viktigt fynd — rapportera det tydligt, det är exakt vad en revisor frågar efter") and steers follow-up to `get_cycle`. [Source: `lib/agent/tools/list-bevis-gaps.ts:18-101` (count + capped-list + description conventions)]
6. Each row carries enough for the skill to evaluate **aggregate coverage across rolling partial cycles without opening each one** (Journey A3: four quarterly `groups`-scoped cycles that tile the list ARE current coverage — the summary must expose kind + counts so the skill can see the tiling):

   - `scheduledStart`, `scheduledEnd`, `completedAt` (= `sealed_at`, null unless AVSLUTAD)

   - `progress: { itemCount, reviewedCount }` and `findings: { openCount, closedCount }` (open = `closed_at: null`)
   [Source: `prisma/schema.prisma:2281` (scope_definition shape comment), `:2276-2316`]
7. 8. New `'read'` tool `get_cycle` in `lib/agent/tools/get-cycle.ts`, factory `createGetCycleTool(workspaceId)`. Input `{ cycleId: string }` — **required** (no CYCLE chat context exists until Story 29.5; description steers "hitta id via list_cycles"). Workspace-scoped `findFirst({ where: { id, workspace_id, deleted_at: null } })`; miss → `wrapToolError` steering to `list_cycles`.

10. Returns **neighbour collections as capped rows/handles** (lazy edges, true totals alongside):

## Dev Notes

### Why this story exists (epic context the dev must keep)

- **A3 — the ISO move (aggregate coverage):** `prepare_audit` must answer *"do the AVSLUTAD INTERN cycles of the last 12 months **together** cover the scope?"* — aggregate, not single-cycle. Rolling kontroller (`scope_definition: groups/items`, e.g. a quarter of the list per quarterly cycle) are a **diligence pattern**, and four partial cycles that tile the list ARE current coverage. That is why AC 6 puts the `scopeSummary` (kind + group/item counts + materialised item count) on **every `list_cycles` row**: the skill evaluates tiling from the list alone, without N× `get_cycle` calls. [Source: epic Journey A row A3; Stories list item 1]
- **A6 — the findingId delta:** triage pairs every open avvikelse with an action; where no corrective task exists the agent proposes one "**linked back via `Task.compliance_finding_id`**". Without the delta, agent-proposed corrective tasks are orphans that never register as the finding's åtgärd in the kontroller UI. [Source: epic Journey A row A6; Stories list item 1]
- **Finding traversal is cycle-first:** `cycle_id` is the finding's required FK today, the law item only its optional pointer — A3 costs three hops (`list_cycles` → `get_cycle` findingRows → `get_finding`), no law-list walk. [Source: epic Stories list item 1]
- **Awareness model:** readers exist so skills **aggregate approved judgments, never re-derive them** — the 19.4 caps deliberately stop one chat turn from pretending to have deep-read a year of history. Do not weaken the caps "to be helpful". [Source: epic "Awareness & depth model"]
- **AUDITOR persona:** the external auditor's free seat runs the read phases only; all three tools land in the read tier so role narrowing (19.5) does the degradation automatically. [Source: epic Journey D; Personas]

### Epic 23 tolerance rule (CRITICAL — encode, don't skip)

Epic 23 (planned; Story 23.1 Draft at `docs/stories/23.1.findings-decouple-schema-and-server-actions.md`) will make `ComplianceFinding.cycle_id` **nullable**, add `workspace_id`/`created_by_user_id`/`assignee_id` columns, and add a CHECK constraint that every finding anchors to a cycle OR law item OR requirement. **29.1 lands first**, so per the epic's coordination note: *build readers tolerant of the coming nullable `cycle_id` — treat the cycle handle as optional from day one.* Concretely:

1. `get_finding`'s output interface types `cycle: ContextHandle | null` and the shaping code runs `finding.cycle ? handle : null` at runtime (even though Prisma currently types the relation non-null — do not lean on that type).
2. The dispatch link step (AC 16b) creates the `ComplianceCycleTaskLink` row **only when the finding's `cycle_id` is non-null** — after Epic 23, an ad-hoc finding has no cycle to link.
3. `get_finding`'s workspace scoping (`cycle: { workspace_id }`) is the **only** available path today; leave a `// Epic 23:` comment noting the where-clause should move to the finding's own `workspace_id` column once 23.1 lands (23.1 owns that migration; do NOT pre-build it).
4. Do not add an `assigneeName` field yet — it doesn't exist pre-23; the output contract must not contain speculative fields.

  workspace_id: workspaceId,
  deleted_at: null,
  ...(status && { status }),
  ...(auditType && { audit_type: auditType }),
  ...(lawListId && { law_list_id: lawListId }),

const [count, rows] = await Promise.all([
  prisma.complianceAuditCycle.count({ where }),
  prisma.complianceAuditCycle.findMany({
    where, take,
    orderBy: [{ sealed_at: { sort: 'desc', nulls: 'last' } }, { scheduled_start: 'desc' }],
    include: {
      law_list: { select: { name: true } },
      _count: { select: { items: true, findings: true } },
    },
  }),
])
const ids = rows.map((r) => r.id)
const [reviewed, open] = await Promise.all([
  prisma.complianceAuditItem.groupBy({ by: ['cycle_id'], where: { cycle_id: { in: ids }, reviewed_at: { not: null } }, _count: { _all: true } }),
  prisma.complianceFinding.groupBy({ by: ['cycle_id'], where: { cycle_id: { in: ids }, closed_at: null }, _count: { _all: true } }),
