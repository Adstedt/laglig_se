# Story 25.5: Feedback form — in-scope MVP (B.5)

## Status

Ready for Review

## Story

**As a** Laglig.se user (encountering the modal on first run, or re-opening the tutorial later via FAB / Hjälp) who has an opinion about *anything in the product* — onboarding, the laglista, kravpunkter, lagändringar, the AI agent, missing features, pricing, copy, bugs,
**I want** a one-screen feedback form right inside the modal — thumbs up/down + an optional free-text message + an optional reply-to email — that I can submit in a single click,
**so that** I can tell the team what works or what doesn't without leaving what I'm doing, hunting for a support address, or filing a ticket — and so the product team gets unfiltered, in-the-moment product signal from the moment Epic 25 ships.

**Scope framing (clarified 2026-05-18):** the form captures **general product feedback**, not feedback *about* the onboarding modal specifically. The modal is just the **first surface** that hosts it — future stories may surface the same form (or a deep-link to it) from the settings page, the dashboard footer, or a "Skicka feedback" link in the help menu, all routing through the same server action. The payload carries a `source` discriminator so future surfaces can be filtered (`source: 'onboarding_modal_feedback_tab'` for B.5).

This is the **B.5 wedge** of Epic 25: a single `<FeedbackStep>` component rendered as the **7th (rightmost) tab** in the tutorial-tabs strip, plus one new server action (`submitProductFeedback`) that writes one `OnboardingEvent` row AND emails dev@laglig.se via the existing Resend transport (`lib/email/email-service`). The pattern mirrors Story 24.5's internal-ops email pattern (`app/actions/law-list-import.ts:810-862` → `CatalogRequestReceivedInternalEmail`): event row is the durable record; email is a convenience nudge; email failure does NOT block the user submit.

**No new schema** — reuses `OnboardingEvent` from Story 25.0 with `event_type='feedback_submitted'` (already in the `§5.2` event-shape table). **No admin dashboard** — the dev@laglig.se inbox is the surface (PO decision 2026-05-13). **No new permission scope** — `withWorkspace(cb)` covers it, workspace-self-write only.

Concretely: B.5 adds (1) one new TutorialTabId value `'feedback'`, (2) a 7th entry to the `TABS` array in `tutorial-step.tsx` (rightmost, `MessageCircle` icon, **no "Ny" chip** — quiet entry per arch §14), (3) a new `<FeedbackStep>` component file colocated with the other top-level step components, (4) the `'feedback'` entry added to `TUTORIAL_TAB_COMPONENTS` in `tutorial-tabs/index.ts`, (5) the `submitProductFeedback` server action appended to `app/actions/onboarding-modal.ts`, (6) a new React Email component `ProductFeedbackInternalEmail` under `emails/`, and (7) tests for the component + the action.

The form is reachable in both `tutorial` mode (during/after generation, which is the only mode B.5 ships against — B.6 adds `tutorial_only`) and will be reachable in `tutorial_only` mode once B.6 lands the FAB / Hjälp re-entry — same component, same tab strip position, no extra wiring needed in B.5.

### Server action — `submitProductFeedback`

18. **New export from `app/actions/onboarding-modal.ts`** (append to the existing file; do NOT create a new action file — keeps related actions colocated, even though the action's name is product-wide; the modal file remains the right home for B.5 since that's the only caller.    
    ```ts
    export async function submitProductFeedback(input: {
      sentiment: 'positive' | 'negative'
      message?: string
      email?: string
    }): Promise<ActionResult>
    ```
    where `ActionResult = { ok: true } | { ok: false; error: string }` (the existing type alias at `onboarding-modal.ts:26` — reuse, do not redefine).

    **Naming note — diverges from arch §7.1.** The architecture doc (line 415) names this `submitOnboardingFeedback`. Story 25.5 renames to `submitProductFeedback` to honestly reflect the broader scope clarified 2026-05-18 (see Story section "Scope framing" + Owner-ack'd decision 6). 19. **Server-side validation** (before any write):
    - `sentiment` must be `'positive' | 'negative'`. If not, return `{ ok: false, error: 'Välj en tumme upp eller ner.' }`.
    - `message`, if provided, is sliced to 500 chars server-side (defence-in-depth against the client cap in AC 13).
    - `email`, if provided after `.trim()`, must satisfy a simple regex `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`. If non-empty AND invalid, return `{ ok: false, error: 'Ogiltig e-postadress.' }`. If empty, fall through and use session email as fallback (see step 21).

20. **Wrapped in `withWorkspace(cb)`** with NO permission argument (workspace-self-write only — matches the existing four actions in this file). Throws propagate from `withWorkspace` for unauthenticated callers; the action only returns `{ok: false}` for validation / data errors.

21. **Implementation sequence (mirrors the file's existing fail-safe pattern at `onboarding-modal.ts:60-105`):**
    1. **Validate input** (AC 19). Bail with `{ok: false, error}` if invalid.
    2. **Fetch user.email + user.name** via `prisma.user.findUnique({ where: { id: ctx.userId }, select: { email: true, name: true } })`. `ctx.workspaceName` is already on the `WorkspaceContext` (see `lib/auth/workspace-context.ts:23-31`) — no extra query needed for workspace name.
    3. **Resolve final email** for the email body's reply-to: `const replyToEmail = (input.email?.trim() || user.email) ?? null`. (User existence is guaranteed by `withWorkspace`'s session check, so `user.email` is non-null in practice; the `?? null` is paranoia.)
    4. **Write the `OnboardingEvent` row** via the existing `writeOnboardingEvent(ctx.workspaceId, ctx.userId, 'feedback_submitted', { sentiment, message: trimmedMessage ?? null, email: replyToEmail, source: 'onboarding_modal_feedback_tab' })` helper (`onboarding-modal.ts:32-54`). This is the durable record. The helper itself is fail-safe — it logs `[ONBOARDING_EVENT_WRITE_FAIL]` on failure and never throws. **Do not await separately** — `await writeOnboardingEvent(...)` matches the existing call sites. **`source` is a forward-compat discriminator** so future surfaces (a "Skicka feedback" link in settings, the help menu, the dashboard footer) can write `feedback_submitted` rows tagged with their own origin string and the team can filter by `payload->>'source'` (per Owner-ack'd decision 7).

Wrapped in `try/catch` — on failure, log `[PRODUCT_FEEDBACK_EMAIL_FAIL]` and swallow the error (mirrors `app/actions/law-list-import.ts:842-862` pattern). User-facing result is still `{ok: true}` — they see "Tack" regardless of email transport state.
    6. **Return `{ ok: true }`.**

### Owner-ack'd decisions (v0.1 draft)

1. **`<FeedbackStep>` takes no props (no `defaultEmail` from session) — server fallback instead.** Decision rationale: threading a default email through 4 layers of components to populate one form field would require widening the uniform `Record<TutorialTabId, ComponentType>` map signature, which breaks the parallel structure of the other 6 tab components for a single string. The server action falls back to `session.user.email` when the email field is empty, and the placeholder text tells the user this happens ("Din e-post (valfritt — vi använder din inloggning som standard)"). UX-wise: the user can still type a different reply-to if they want; a blank field "just works" silently. This is a deliberate simplification from arch §14's "prefilled from session.user.email, editable" wording. **Reversible if needed:** future hardening can add the prop drilling in <30 min if the team decides the prefill UX matters more than the prop-drilling cost.

2. **Custom radiogroup instead of pulling in `@radix-ui/react-radio-group`.** The project's `components/ui/` directory has no `radio-group.tsx` — only `toggle.tsx`, `toggle-group.tsx`, and `fulfilled-toggle.tsx`. Pulling in a new Radix dependency for a single form (two-option toggle that conceptually fits the toggle primitive) adds bundle size and maintenance burden. AC 11 + AC 25 + AC 26 implement the WAI-ARIA radio pattern by hand using `role="radio"` + `aria-checked` + roving tabindex — same pattern `<TutorialStep>` uses for its own tab bar (`tutorial-step.tsx:90-112`). Tested-by-precedent.

3. **`emails/product-feedback-internal.tsx` lives at the top level of `emails/`, not under `emails/components/`.** `emails/components/` is for SHARED layout primitives (`laglig-email-layout.tsx`, `email-icons.ts`) — individual email templates live one level up (see `emails/catalog-request-received-internal.tsx`, `emails/import-review-ready.tsx`, etc.). B.5's email is a template, not a primitive.

4. **No client-side email-format validation beyond `type="email"`.** The arch §14 doesn't prescribe validation strictness. ### Key decisions baked into this story

- **One server action, not three** (one for write, one for email, one for orchestration). Reasoning: the orchestration is trivial (validate → write event → send email → return) and all three steps share the same `ctx` from `withWorkspace`. Splitting would create three action files for one feature.
- **Reuse the existing `writeOnboardingEvent` helper instead of inlining the prisma.onboardingEvent.create call.** Reasoning: the helper handles the fail-safe try/catch and the standard `[ONBOARDING_EVENT_WRITE_FAIL]` log prefix. Inlining would duplicate that.
- **`payload.email` stores the resolved reply-to (user-entered OR session fallback)**, not the raw user input. Reasoning: the email row in the DB IS the durable record — if the email transport fails later and the team has to backfill replies, they need the actual address that was used as reply-to, not the empty string the user typed. Slightly less faithful to the raw input but more useful operationally.
- **`message` is trimmed before being stored AND trimmed-empty becomes `null`.** Reasoning: a message of `"   \n  "` is not meaningful feedback; storing it as `null` keeps queries on `payload->>'message' IS NOT NULL` honest. Client AC 13 doesn't trim (the textarea preserves whitespace as the user types), but the server action does.
- **No rate-limit at v1** (per arch §7.1 "Why no rate-limit at v1"). The feedback form is naturally throttled by the manual click + one-form-per-tab UX. If we see spam from authenticated workspace users, add a per-workspace cooldown in a follow-up — out of scope for B.5.
- **`feedback_submitted` event row is written BEFORE the email send** (not after, not in parallel). Reasoning: per arch §7.1 "Why event-write before email-send" — the DB row is the durable record; if we sent email first and the DB write failed, the email would arrive with no audit trail. Reverse failure mode (row written, email lost) is recoverable.
- **RIGHT column context card is not just decorative** — it carries the privacy note ("Vi sparar bara rummet du tillhör, din e-post och meddelandet. Inga tabsfingeravtryck eller analytics-IDs.") which is the right surface for that disclosure given B.5 sends data outside the user's workspace context. PO can edit the copy at any time without code change to the form itself.
