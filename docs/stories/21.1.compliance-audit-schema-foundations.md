# Story 21.1: Compliance-audit schema foundations

## Status

Draft

## Story

**As a** backend developer,
**I want** the Prisma schema to define all five compliance-audit-cycle models plus their enums with proper relations, and the `canonicalize` dependency added to `package.json`,
**so that** subsequent Epic 21 stories can build server actions and UI against a stable data model, a deterministic seal-hash foundation is in place, and the existing Laglig schema is not destructively modified.

## Acceptance Criteria

1. `ComplianceAuditCycle` model defined in `prisma/schema.prisma` with: `id`, `workspace_id`, `law_list_id`, `name`, `scope_definition` (Json), `audit_type` (enum `AuditType`: INTERN | EXTERN), `scheduled_start`, `scheduled_end`, `law_change_cutoff_date`, `status` (enum `ComplianceCycleStatus`: PLANERAD | PAGAENDE | AVSLUTAD | SEALED | ARKIVERAD), `lead_auditor_user_id`, `sealed_at`, `sealed_by_user_id`, `seal_hash`, `created_at`, `updated_at`, `deleted_at`, `created_by_user_id`.
2. `ComplianceAuditItem` model with: `id`, `cycle_id`, `law_list_item_id`, `efterlevnadsbedomning` (enum `EfterlevnadsBedomning`: UPPFYLLD | DELVIS | EJ_UPPFYLLD | EJ_TILLAMPLIG, nullable), `motivering` (text, nullable), `reviewed_at`, `reviewed_by_user_id`, `signed_off_at`, `signed_off_by_user_id`, `kravpunkter_snapshot` (Json, nullable), `created_at`, `updated_at`.
3. `ComplianceFinding` model with: `id`, `cycle_id`, `law_list_item_id` (nullable), `requirement_id` (nullable), `type` (enum `FindingType`: AVVIKELSE | OBSERVATION | FORBATTRING), `severity` (enum `FindingSeverity`: MAJOR | MINOR, nullable), `title`, `description`, `root_cause` (nullable), `corrective_action_task_id` (nullable FK to `Task`), `due_date` (nullable), `closed_at`, `closed_by_user_id`, `created_at`, `updated_at`.
4. `ComplianceEvidenceSnapshot` model with: `id`, `cycle_id`, `law_list_item_id` (nullable), `requirement_id` (nullable), `evidence_kind` (enum `EvidenceKind`: FILE | DOCUMENT), `evidence_file_id` (nullable FK to `WorkspaceFile`), `evidence_document_id` (nullable FK to `WorkspaceDocument`), `evidence_sha256`, `captured_at`. A database-level CHECK constraint enforces exactly one of `evidence_file_id` / `evidence_document_id` is non-null (XOR).
5. `ComplianceAuditReport` model with: `id`, `cycle_id`, `report_kind` (enum `ReportKind`: COMPLETE | SEALED), `generated_at`, `pdf_storage_path` (nullable), `html_storage_path` (nullable), `manifest` (Json).
6. Nullable column `compliance_finding_id UUID NULL` added to the existing `Task` model as a back-reference; indexed `@@index([compliance_finding_id])`. No other `Task` field changes.
7. All new models use workspace scoping where applicable (`workspace_id` + FK to `Workspace`) and follow existing Laglig naming + index conventions (snake_case DB, camelCase Prisma field names, `@db.Uuid`, `DateTime` with `@default(now())` or `@updatedAt`, explicit `@@map("...")` for table names matching snake_case plural convention observed in existing schema).
8. Composite and single-column indexes added to support expected queries: `compliance_audit_cycles(workspace_id, status)`, `compliance_audit_items(cycle_id, signed_off_at)`, `compliance_findings(cycle_id, type)`, `compliance_evidence_snapshots(cycle_id)`, `compliance_audit_reports(cycle_id, report_kind)`, `tasks(compliance_finding_id)`.
9. `prisma migrate dev --name add_compliance_audit_cycle` runs clean against a fresh DB; `prisma generate` produces typed clients for all new models; inspecting the generated migration SQL shows only `CREATE TABLE`, `CREATE TYPE`, `CREATE INDEX`, and a single `ALTER TABLE tasks ADD COLUMN compliance_finding_id UUID NULL` — zero `DROP`, zero `ALTER COLUMN`, zero `RENAME` on any existing object.
10. `canonicalize` npm package (RFC 8785 JSON Canonicalization Scheme) added to `package.json` dependencies as an **exact-version** pin (no caret, no tilde). Runs `pnpm install` clean; license-check CI step passes (`canonicalize` is MIT).
11. Seed / fixture helper `tests/fixtures/compliance-audit.ts` exported with a `createTestCycle(overrides?)` utility that builds a cycle + materialised items in a test DB — used by subsequent stories' tests.

### Integration Verification

- **IV1:** Existing Prisma models (`LawList`, `LawListItem`, `LawListItemRequirement`, `RequirementEvidenceLink`, `Task`, `ActivityLog`, `User`, `Workspace`, `WorkspaceFile`, `WorkspaceDocument`) remain untouched in field definitions. Only new relation arrays are added where a new child model points at them (e.g., `Task` gains `ComplianceFinding?` back-relation + the single nullable column; other existing models gain only inward-pointing relation arrays like `cycles ComplianceAuditCycle[]`).
- **IV2:** `pnpm build` passes clean after migration + `prisma generate`. `npx tsc --noEmit` passes clean. `pnpm test` runs all existing tests and they pass unchanged.
- **IV3:** Visual diff of the generated Prisma migration SQL contains only: additive `CREATE TABLE` statements, `CREATE TYPE` statements for new enums, `CREATE INDEX` statements, and a single `ALTER TABLE public.tasks ADD COLUMN compliance_finding_id UUID NULL`. The migration file is reviewed by a second pair of eyes before merge.

### Rollback

- **DB:** `prisma migrate resolve --rolled-back 20260423_add_compliance_audit_cycle` followed by manual DROP of the 5 new tables, 6 new enum types, and `ALTER TABLE tasks DROP COLUMN compliance_finding_id` (in reverse FK dependency order). Safe because no existing row references any new structure.
- **Code:** `git revert` the single Story 21.1 commit; `pnpm install` restores `package.json` without `canonicalize`.
- **Trigger:** rollback only if `prisma migrate deploy` fails in staging or if the migration causes unexpected lock contention on `tasks` in staging.

### XOR CHECK constraint — mirror existing pattern

Before generating the migration, **inspect `prisma/schema.prisma` for how `RequirementEvidenceLink` enforces its file-XOR-document constraint.** Two possibilities:

1. There is an existing raw-SQL migration with `ALTER TABLE ... ADD CONSTRAINT ... CHECK (...)`. Replicate that exact idiom for `compliance_evidence_snapshots`.
2. The existing code enforces XOR only at the application layer (no DB constraint). In that case, ADD a DB-level constraint for `compliance_evidence_snapshots` anyway — this is a strict-mode improvement justified by audit integrity, and the existing table can be left as-is (scope discipline for this story).

Target statement (to be appended to the generated migration SQL file):

```sql
ALTER TABLE compliance_evidence_snapshots
  ADD CONSTRAINT compliance_evidence_snapshots_xor_check
  CHECK (
    (evidence_file_id IS NOT NULL AND evidence_document_id IS NULL)
    OR
    (evidence_file_id IS NULL AND evidence_document_id IS NOT NULL)
  );
```

### Technical constraints

- **Prisma version:** 5.12+ (existing) — no upgrade required. [Source: architecture/3-tech-stack.md — Prisma row]
- **PostgreSQL:** Supabase PG 15.1 — supports all constraint types used here. [Source: architecture/3-tech-stack.md — Database row]
- **Additive-only schema:** no DROP, no RENAME, no ALTER COLUMN of any existing field. Enforced by AC9 + IV3. [Source: architecture/epic-21-lagefterlevnadskontroll.md §2.3, §4.3]
- **License policy:** `canonicalize` is MIT — complies with policy §3.6. Exact-version pin is required to prevent canonical-JSON drift across `npm install` runs (future seal-hash determinism depends on it). [Source: architecture/epic-21-lagefterlevnadskontroll.md §6.2, §3.6]
- **FK onDelete semantics summary:**
  - Cycle children (items, findings, evidence snapshots, reports): `Cascade` — deleting a cycle removes its children.
  - Source references (`LawList`, `LawListItem` from cycle/item): `Restrict` — you cannot delete a law or laglista while any cycle references it.
  - Optional parent references (findings → item/requirement, snapshots → file/document, etc.): `SetNull` — preserves the finding/snapshot history even if the referenced entity is removed.
  - The evidence-file-deletion BLOCK policy (Architecture §6.3) is a Server Action layer concern introduced in Story 21.9, NOT a DB FK concern. Do not over-constrain at the DB level here.
