# Story 17.17: Styrdokument Table + Doc Page Dual-Version UX

## Status

Draft

> 🆕 **Second of three in `docs/prd/epic-17-addendum-dual-version-visibility.md`.** Sits on top of Story 17.16's data-model foundation (dual pointers + draft sub-status + per-version audit timestamps). Surfaces the new model in the user-facing UI: composite status badge in the styrdokument table for the dual state, doc page header that preserves approved metadata throughout draft windows, editor banner that clarifies the draft-replaces-approved relationship, and a discard-draft button wired to 17.16's new server action. Story 17.18 (agent reads + citations) ships on top of this UX foundation. Story 17.11c (agent auto-branch on APPROVED) follows separately.

## Story

**As a** compliance officer (or any user reading a styrdokument),
**I want** the styrdokument table and doc page to show me — at a glance — both the currently-effective approved version AND any draft in progress, with clear audit metadata throughout,
**so that** I never lose sight of "what's in force right now" during revision windows, and so the dual state is operationally obvious without me having to dig through version history.

## Story Context

**Existing System Integration (load-bearing — story 17.17 is brownfield-UX-additive):**

- **17.16 ships the data foundation.** `WorkspaceDocument.current_approved_version_id` + `current_draft_version_id` + `draft_status` are populated by the backfill migration and maintained by the refactored server actions (`createDraftFromApproved`, `saveDocumentVersion`, `promoteDraftToApproved`, `discardDraft`). `WorkspaceDocumentVersion.approved_at` / `approved_by` / `superseded_at` carry per-version audit data. Story 17.17 reads these fields to render the new UI; it does NOT change server-action behavior or schema.
- **Styrdokument table:** `components/features/documents/document-table.tsx` renders the doc list. Today it shows a single `<DocumentStatusBadge>` per row, sourced from the doc's `status` enum. The new composite badge is a render-time concern (compose two badges + a separator) — no schema or data layer change required.
- **Badge primitive:** `components/features/documents/document-status-badge.tsx` wraps `components/ui/badge.tsx`. Epic 22 (UI Primitives Alignment) ships an extended `Badge` `tone × variant` matrix; this story uses the existing tone tokens (or coordinates naming if 22.x is mid-flight) for the new composite render.
- **Doc page header:** `components/features/documents/...` (specific filename to be confirmed in dev) — renders title, status badge, and `approved_by`/`approved_at` metadata today. Under Model A's `createDraftFromApproved`, that metadata went blank during draft windows. With 17.16 preserving it, the header simply renders it correctly; this story adds a secondary "Utkast pågår" indicator for the dual state.
- **Editor surface:** Tiptap-based editor in `components/features/documents/editor/...`. Story 17.17 adds a banner above the editor area that explains the draft-vs-approved relationship and surfaces the discard-draft action.
- **Filter chips:** likely live in `components/features/documents/document-filters.tsx` (per the file list in `app/(workspace)/workspace/styrdokument/page.tsx`). Today's "Visa godkända" filter chip matches `status = 'APPROVED'`. Under the new model, an APPROVED-with-draft-pending doc is still operationally approved — the filter must include those rows.
- **Promote-draft action:** Story 17.16 introduces `promoteDraftToApproved` as the explicit atomic transition. Today, approval happens via `updateDocumentStatus({newStatus: 'APPROVED'})`. This story surfaces a "Godkänn utkast" button in the editor (and the doc page header) when `draft_status = 'IN_REVIEW'` that calls the new server action. The legacy `updateDocumentStatus` path remains for never-had-a-draft promotions and is unchanged in this story.
- **17.10b citation contract:** `[Källa: X]` for APPROVED status, `[Utkast: X]` for DRAFT/IN_REVIEW status. Story 17.18 refactors this to the new model; this story does NOT touch the agent's read/citation layer.
- **17.11 + 17.11b approval cards:** when the agent's `update_document` / `add_document_section` lands a new version via the dispatch path, the version is now written to the draft pointer (per 17.16 AC 5). The "Visa mer" canvas in the renderers reads `oldSectionContentJson` / `newSectionContentJson` from the proposal params — those snapshots are unaffected by Model B. The "Öppna dokument" link in the APPROVED-state card already routes to the editor; this story makes sure the editor lands the user on the draft when one is in progress.

**Reuse from existing infrastructure:**
- `components/ui/badge.tsx` (or Epic 22's extended variant) for the composite render.
- The existing approval-card frame chrome (`ActionRendererFrame`) for the editor banner — NO, scratch that — the editor banner is its own surface, not an approval card. Use a lighter banner primitive (e.g. an `Alert`-style box with `lucide-react` icon).
- `lucide-react` icons: `GitBranch` for the draft indicator, `Trash2` for discard, `CheckCircle2` for promote.

**Out of scope (deferred to sibling stories or follow-ups):**
- Agent read tools + citation grammar — Story 17.18.
- Agent auto-branch on APPROVED (`add_document_section` / `update_document` transparently creating a draft) — Story 17.11c.
- Diff viewer between approved and draft (would be nice; not in this epic — Story 17.x follow-up).
- "Show effective version on date X" historical query UI — uses the per-version `approved_at` Story 17.16 ships, but the UI for it is a follow-up.
- Approval workflow polish (assignees, comments) — out of scope; existing `updateDocumentStatus` chain handles the simple case.

## Acceptance Criteria

### Composite status badge (table)

1. **`document-status-badge.tsx` (or a new `dual-status-badge.tsx` wrapper) renders the composite for the dual state:**
   - When the doc has `current_approved_version_id` set AND `current_draft_version_id` set: render `Godkänd v{N} · Utkast v{N+1} pågår` (where `{N}` is the approved version_number and `{N+1}` is the draft version_number). Both pieces are styled `Badge` instances; the separator is a non-breaking middot. Status tone follows existing conventions: approved = `success` (or current shipped tone for APPROVED); draft = `neutral` (or current shipped tone for DRAFT).
   - When the doc has ONLY `current_approved_version_id`: render today's APPROVED badge (unchanged).
   - When the doc has ONLY `current_draft_version_id` AND `draft_status = 'DRAFT'`: render today's DRAFT badge (unchanged).
   - When the doc has ONLY `current_draft_version_id` AND `draft_status = 'IN_REVIEW'`: render today's IN_REVIEW badge (unchanged).
   - When the doc is `status = 'ARCHIVED'` or `status = 'SUPERSEDED'`: render today's badge (unchanged).
   - Composite badge is screen-reader-friendly: an `aria-label` reads `"Godkänd version 3, utkast version 4 pågår"` (full Swedish, not abbreviated) so non-sighted users get the complete state.
   - **CP-001:** the badge NEVER renders raw identifiers (no `documentId`, no `versionId`) — only natural Swedish + version numbers.

2. **Table column update (`document-table.tsx`):** the existing status column uses the new dual-aware badge (no column header rename needed). Sort behavior on the status column unchanged. Row visual height stays consistent (composite badge stacks within the existing cell padding without forcing a row-height bump).

### Filter chips + list semantics

3. **`document-filters.tsx` (or equivalent) — "Visa godkända" filter:** matches rows where `current_approved_version_id IS NOT NULL`. Under the new model this includes APPROVED-with-draft-pending rows. The filter pill copy stays `"Godkända"` (no UI rename); behavior change is invisible to the user except that they no longer see policies vanish from the "Godkända" filter mid-revision.

4. **"Visa utkast" filter:** matches rows where `current_draft_version_id IS NOT NULL`. Under the new model this includes APPROVED-with-draft-pending rows (the same row appears under both "Godkända" AND "Utkast" filters — by design, because the doc IS both at once).

5. **"Senast ändrad" column:** sourced from a derived `last_meaningful_change_at` value: `MAX(current_draft_version.created_at, current_approved_version.approved_at, document.updated_at)`. So a stable APPROVED doc shows its last approval date; an in-revision doc shows its most recent draft save. Existing column header copy unchanged.

### Doc page header

6. **Doc page header (`components/features/documents/document-header.tsx` or per dev's locator):**
   - When the doc has `current_approved_version_id` set AND `current_draft_version_id` set: render the composite badge (per AC 1) PLUS a secondary metadata row: `Godkänd v{N} av {approver.name} den {approved_at, sv-locale-formatted}` from `current_approved_version`'s per-version timestamps (Story 17.16 AC 2). Under the composite badge, render a smaller indicator: `Utkast v{N+1} pågår — senast ändrat den {current_draft_version.created_at}`.
   - When only `current_approved_version_id` set: render today's header (approved metadata + standard badge — unchanged from current behavior, modulo 17.16 fixing the metadata-wipe bug).
   - When only `current_draft_version_id` set AND `draft_status = 'DRAFT'`: render today's DRAFT header.
   - When `draft_status = 'IN_REVIEW'`: render the IN_REVIEW header PLUS — if `current_approved_version_id` is also set — the approved metadata indicator below (so approvers know what they're replacing).
   - **CP-001 holds:** no raw IDs in the header copy.

7. **"Godkänn utkast" / "Skicka för granskning" actions in the header:** when the user has `documents:approve` (or `tasks:edit` fallback per 17.16 AC 6) AND `draft_status = 'IN_REVIEW'`, surface a primary "Godkänn utkast" button that calls `promoteDraftToApproved(documentId)` (17.16 AC 6). When `draft_status = 'DRAFT'`, surface a "Skicka för granskning" button that updates `draft_status` to `IN_REVIEW` (small new wrapper around `updateDocumentStatus`, or new server action — dev's call). When no draft is in progress AND the doc is APPROVED, today's header is unchanged.

### Editor banner

8. **Editor banner (`components/features/documents/editor/...`):** when the editor opens a doc with `current_draft_version_id` set AND `current_approved_version_id` set, render a banner above the editor area:
   - Copy: `Du redigerar Utkast v{N+1} som ersätter Godkänd v{N} efter godkännande.` ({N} = approved version number)
   - Icon: `lucide-react` `GitBranch`.
   - Visual: tinted-amber background, slate-text-on-amber, soft border. Non-dismissable (always visible while editing a draft).
   - Right side of banner: "Förkasta utkast" button (icon `Trash2`, ghost variant) — opens a confirmation `AlertDialog` (`"Vill ni förkasta utkast v{N+1}? Den godkända versionen v{N} förblir gällande."`) and on confirm calls `discardDraft(documentId)` (17.16 AC 7). On success, navigate back to the doc page or close the editor (dev's call based on the existing UX flow).
9. **No banner when the doc has only one pointer set** (never-approved DRAFT — banner-less, today's editor experience; or stable APPROVED — editor shouldn't even be reachable in the new model without first branching a draft).

### Approval card "Öppna dokument" link routing

10. **`update-document-renderer.tsx` + `add-document-section-renderer.tsx` (Stories 17.11 / 17.11b):** the "Öppna dokument" link in the APPROVED state of the cards uses `result_ref.documentId` to route to `/workspace/styrdokument/{documentId}/edit`. Under the new model, that URL must land the user on the draft version's editor view (since the agent's change was applied to the draft pointer). Verify that the editor route (`app/(workspace)/workspace/styrdokument/[documentId]/edit/page.tsx`) reads `current_draft_version_id` first when present — should already work transparently via 17.16's deprecated alias, but confirm. No code change expected if the alias works; this AC is a verification checkpoint.

### Accessibility + i18n

11. **Composite badge `aria-label`:** full Swedish (`"Godkänd version 3, utkast version 4 pågår"`), not abbreviated. Tested via RTL `getByRole('status', { name: /Godkänd version 3, utkast version 4 pågår/ })`.

12. **Color contrast:** composite badge meets WCAG AA contrast for both the approved and draft tone pills against the table row background (both dark and light themes). Verified in Storybook (if used) OR via a contrast-ratio snapshot in tests.

### Non-regression

13. **Existing single-state badge behavior is preserved.** Docs in pure APPROVED, pure DRAFT, pure IN_REVIEW, ARCHIVED, SUPERSEDED states render exactly as today (no visible UX change). RTL tests covering today's badge variants stay green. Existing Storybook/visual-regression snapshots for the table stay valid for the non-dual cases.

14. **Existing keyboard navigation in the table is unchanged.** The composite badge does not introduce new focusable elements (it's a render-time composition of static badges, not interactive).

15. **No new server-side queries.** All new render logic reads fields already populated by Story 17.16's schema; no additional Prisma queries in the table or page loaders. (Performance regression risk: zero.)

## Dev Notes

- **Composite badge styling reference** — the visual treatment should follow the same neutral-with-accent pattern Epic 22's `Badge` matrix establishes. If 22.x has not yet shipped the `tone` axis when this story enters dev, use today's shipped tones (which `document-status-badge.tsx` already maps from `WorkspaceDocumentStatus` enum). Don't block on 22.x; the composite is a render composition, not a new variant.
- **Why the table column header copy doesn't change** — "Status" + "Senast ändrad" still describe what the column shows; the meaning of "Senast ändrad" widens slightly (now also reflects draft activity for in-revision docs), but that's the user's mental model anyway ("when did this doc last change in any meaningful way"). Renaming the column would create churn for zero clarity gain.
- **`last_meaningful_change_at` derivation** — keep this as a render-time computation in the table loader query (not a new persisted column). `MAX(current_draft_version.created_at, current_approved_version.approved_at, document.updated_at)` is cheap with the existing FK joins. If profiling shows it's a hot path, a follow-up story can add a denormalized column.
- **Editor banner copy is FROZEN.** *"Du redigerar Utkast v{N+1} som ersätter Godkänd v{N} efter godkännande."* This is the authoritative phrasing — please don't paraphrase during dev. The copy is consistent with how the system prompt teaches the agent to talk about drafts ("ett pågående utkast föreslår…", "ännu inte godkänt") and with the doc-page header text.
- **"Förkasta utkast" confirmation dialog is REQUIRED** — discard is destructive (the draft version row stays in history per 17.16 AC 7, but the user no longer sees it without going into version history). Confirmation prevents fat-finger losses.
- **Approval-card link verification (Task 6)** — this is bureaucratic but important. The agent's `add_document_section` / `update_document` approval cards link to `/workspace/styrdokument/{documentId}/edit` post-approve. If the editor route loads `current_version_id` (the deprecated alias) it will land on the draft (since 17.16's `saveDocumentVersion` advances the alias to point at the draft). That's the right behavior. If the editor route is later refactored to explicitly load `current_approved_version_id` (Story 17.18 or a cleanup story), this AC re-opens. Document the assumption clearly so future refactors don't break it.
- **"Skicka för granskning" wrapper** — under the new model, this just flips `draft_status: DRAFT → IN_REVIEW`. The simplest implementation is a new `submitDraftForReview(documentId)` server action that does the single field update + ActivityLog row. Dev can decide whether to bundle this into 17.16 (would be a small AC addition to that story) or add it inline in 17.17 as a tiny helper. Either is fine; flag the decision in dev notes.
- **Diff viewer is OUT OF SCOPE** — please don't build a "compare approved vs draft" UI in this story. It's a great follow-up but materially expands scope. The composite badge + header indicator + editor banner together give the user enough operational awareness without a diff.

## Risk and Compatibility Check

**Primary Risk:** The composite badge introduces a visual element that doesn't exist today. Edge cases (very long version numbers like `v123`, very narrow table columns at small viewports, RTL languages — not currently a concern but worth noting) could cause layout breakage in the table row.

**Mitigation:**
- Composite render uses `whitespace-nowrap` on each badge half and `truncate` on the surrounding cell — single-line guarantee at table density.
- Storybook (or test fixtures) cover extreme version numbers and narrow viewports.
- At very narrow viewports (mobile), the composite collapses to the approved badge only (with a small "+" indicator or icon to signal draft pending) — keeps the row legible. **Note for dev:** confirm this narrow-viewport treatment with PO before shipping; the default in this story spec is "always show composite" with the collapse as a stretch goal.

**Rollback:**
- All UI changes; rollback is straightforward (revert the components). No data migration tied to this story.
- If the composite badge causes visual regressions in production, feature-flag it (`dual_version_badge_v1`) and fall back to the single-status badge until the issue is resolved. The underlying schema + data layer remains correct because 17.16 owns those.

**Compatibility Verification:**
- [ ] No breaking changes to existing component APIs (`DocumentStatusBadge` keeps its existing prop shape; the dual variant is an internal render decision based on the doc's pointer state).
- [ ] Database changes — none in this story.
- [ ] UI changes follow existing design patterns (`Badge` primitive from `components/ui/`, Epic 22's tone matrix where applicable).
- [ ] Performance impact is negligible (no new DB queries; render-time composition only).
