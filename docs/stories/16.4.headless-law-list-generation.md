# Story 16.4: Headless Law List Generation Skill

## Status

Draft

## Dependencies

- **Story 16.3** (contextual onboarding questions) — activity flags must be populated on CompanyProfile for the agent to use
- **Story 16.1** (auto-login) — user must reach dashboard without verification gate
- **Epic 14** (agent tools) — `createAgentTools()`, `search_laws`, `get_company_context` must exist (Done)
- **Epic 12** (templates) — template data provides quality floor for curated content (Done)

## Story

**As a** newly onboarded user,
**I want** the system to automatically generate a personalized law list for my company based on my business profile,
**so that** I land on a populated dashboard with relevant laws, monitoring, and compliance tracking from day one.

## Acceptance Criteria

### Headless Agent Skill

1. New server-side skill at `lib/agent/skills/generate-law-list.ts` using `generateText()` from AI SDK (not `streamText()` — no chat UI). This establishes the `lib/agent/skills/` directory as a new architectural pattern for headless agent capabilities.
2. Uses existing `createAgentTools(workspaceId, userId)` tool registry — same `search_laws`, `get_company_context` tools as the chat agent. Skill function signature: `generateLawList(workspaceId: string, userId: string): Promise<GenerateLawListResult>`
3. New tool `add_laws_to_list` added to the tool set for this skill — bulk-adds laws to a LawList with grouped sections and commentary. Uses Prisma directly (NOT the `addDocumentToList` server action, which requires cookie-based `withWorkspace` context unavailable in headless execution).
4. System prompt instructs the agent to: search for relevant laws across all regulatory areas, evaluate applicability based on company profile, group by area, and add with personalized commentary
5. Step limit: `stopWhen: stepCountIs(20)` (NOT `maxSteps` which does not exist in the AI SDK) — allows 10-15 search passes + add operations
6. Model: `anthropic('claude-sonnet-4-6')` (balance of quality and cost)
7. Skill returns: `{ listId, itemCount, groups, tokensUsed, durationMs }`

### Trigger & Execution

8. Triggered server-side after `createWorkspace()` completes successfully — call the generation API route from the server action itself (not fire-and-forget from client). If the API call fails to fire, set status to `'pending'` so the dashboard retry logic (AC 22) picks it up.
9. Runs asynchronously (non-blocking) — user is redirected to dashboard immediately
10. New field `Workspace.law_list_generation_status`: `'pending' | 'in_progress' | 'completed' | 'failed'`
11. Status updated at each phase: `pending` (workspace created) → `in_progress` (skill starts) → `completed`/`failed`
12. If skill fails (LLM error, timeout), status set to `'failed'` with error logged — user can still use empty dashboard and add laws manually
13. Generation endpoint is idempotent: if `law_list_generation_status` is already `'in_progress'`, return early without starting a second run

### Law List Output

14. Creates a default `LawList` named "Er laglista" with `is_default: true`
15. Items grouped into `LawListGroup` sections: "Arbetsratt", "Bolagsratt", "Skatt", "Miljo", "Dataskydd", etc.
16. Each `LawListItem` includes:
    - `document_id` — FK to existing `LegalDocument` (agent must search and find real documents)
    - `commentary` — Personalized one-liner: "Galler er som arbetsgivare med 12 anstallda inom restaurang"
    - `source: 'ONBOARDING'` (exists in Prisma `LawListItemSource` enum; note: the Zod `LawListItemSourceEnum` in `lib/validation/document-list.ts` is missing `TEMPLATE` — pre-existing gap, not this story's concern)
    - `compliance_status: 'EJ_PABORJAD'`
    - `position` — ordered within group
17. Target: 40-80 laws depending on company complexity
18. No duplicate documents in the list (unique constraint on `[law_list_id, document_id]`)

### Dashboard Progress Indicator

19. Dashboard shows generation progress when `law_list_generation_status` is `'pending'` or `'in_progress'`
20. Progress component: card with Loader2 spinner + "Vi skapar er personliga laglista..." message
21. Polls `GET /api/workspace/generation-status` every 3 seconds (use `useSWR` with `refreshInterval: 3000` for robust polling with automatic cleanup, or `useEffect` + `setInterval` with proper stale-closure handling)
22. On load, if status is `'pending'` (trigger never fired or failed), re-fire the generation POST as a retry
23. On completion: progress card replaced with law list summary card, smooth transition
24. On failure: show "Laglistan kunde inte skapas automatiskt. Du kan skapa din lista manuellt." with link to template catalog

### Cost & Performance

25. Target cost: <$0.25 per generation (Sonnet, ~10-15 tool calls). Raw estimate ~$0.24; prompt caching and reduced tool calls may bring this lower.
26. Target duration: <60 seconds
27. Log token usage and duration to Sentry as custom breadcrumbs/transactions for monitoring

### Manual Re-generation

28. Button in Settings or Law List page: "Generera om laglista" — re-runs the skill with current company profile
29. Shows confirmation dialog: "Detta ersatter din nuvarande laglista. Forandringar du gjort behalls inte."
30. Re-generation uses the same skill, creates a new list (old one archived as "Er laglista (arkiverad {date})", not deleted)

## Dev Notes

### Headless Agent Pattern

This is the first production use of `generateText()` in the app. The chat route uses `streamText()`. Both share the same tool registry. The simulation scripts (`scripts/tmp-simulate-full-agent.ts`) demonstrate the exact pattern:

```typescript
import { generateText, stepCountIs } from 'ai'
import { anthropic } from '@ai-sdk/anthropic'
import { createAgentTools } from '@/lib/agent/tools'

const result = await generateText({
  model: anthropic('claude-sonnet-4-6'),
  system: skillPrompt,
  messages: [{ role: 'user', content: 'Build a comprehensive law list.' }],
  tools: { ...createAgentTools(workspaceId, userId), add_laws_to_list: addLawsToListTool },
  stopWhen: stepCountIs(20),
})

// Access results:
// result.text — final response
// result.steps — array of tool invocations
// result.usage — { inputTokens, outputTokens }
```

### Skills Directory Pattern

`lib/agent/skills/` is a new directory establishing the pattern for headless agent capabilities. A skill is:
- **Input:** workspace context (workspaceId, userId)
- **Output:** typed result object
- **Internals:** system prompt + tools + `generateText()` + `stopWhen`
- **Reusable:** same brain, same tools, different prompt and trigger

Future skills follow the same contract: `lib/agent/skills/assess-law-change.ts`, `lib/agent/skills/generate-compliance-report.ts`, etc.

### Why Prisma Directly (Not Server Action)

The existing `addDocumentToList()` at `app/actions/document-list.ts` is a `'use server'` action that calls `withWorkspace()`. This reads cookies to resolve the workspace context, which is NOT available in a headless API route execution. The `add_laws_to_list` tool must use Prisma directly with the `workspaceId` passed explicitly.

### Tool Response Pattern

Read tools in the codebase use `wrapToolResponse()` from `lib/agent/tools/utils.ts`. Write tools have a two-step confirm pattern (`execute: false` → preview → `execute: true`). For the headless skill, `add_laws_to_list` should execute directly (no confirmation needed — it's a background job, not interactive).

### Zod/Prisma Enum Note

The Prisma `LawListItemSource` enum includes `ONBOARDING`, `MANUAL`, `IMPORT`, `TEMPLATE`. The Zod `LawListItemSourceEnum` in `lib/validation/document-list.ts` only has `['ONBOARDING', 'MANUAL', 'IMPORT']` — `TEMPLATE` is missing. This is a pre-existing gap. The `add_laws_to_list` tool uses Prisma directly (not the Zod schema), so it can use `ONBOARDING` without issue.

### Cost Estimation

Based on simulation scripts and Sonnet pricing ($3/1M input, $15/1M output):
- `get_company_context`: ~500 tokens
- `search_laws` (per call, ~10 calls): ~2K input + ~1K output each
- `add_laws_to_list` (per call, ~3 calls): ~1K tokens each
- System prompt + orchestration: ~5K tokens
- **Total estimate: ~30K input ($0.09) + ~15K output ($0.23) = ~$0.32**
- With prompt caching (repeated system prompt): ~$0.20-0.25
- AC target: <$0.25 (achievable with optimization)
