## Story

**As a** user proposing a task via the compliance partner,
**I want** the proposal to mirror the full task object вЂ” status, ansvarig, fГ¶rfallodatum, and a link to the law list item I'm discussing вЂ” with sensible defaults already filled in,
**so that** the approved task is ready-to-go and matches what I'd create by hand in the task modal, without post-creation editing.

## Context

14.22 ships the minimal CREATE_TASK proposal (title / description / priority). The actual task object (and the task modal) carry more: **Status** (board column), **Ansvarig** (assignee), **FГ¶rfallodatum** (due date), and links to law list items / kravpunkter. This story enriches the proposal card to mirror that object so the agent proposes a complete, ready task.

The data layer is already there вЂ” `createTask` accepts every field вЂ” so this is mostly (a) tool schema fields, (b) renderer UI mirroring the modal's controls, and (c) the agent pre-filling from context. Several fields get **deterministic smart defaults** from signals we already have (the current chat context for the kravlГ¤nk, the linked item's `responsible_user_id` for ansvarig), so the agent doesn't need to "guess" вЂ” the heavier "match the most relevant person by expertise/workload" reasoning is deferred to Epic 19.

## Acceptance Criteria

1. The `create_task` tool schema + the `CREATE_TASK` proposal `params` are extended beyond 14.22's `{ title, description, relatedDocumentId, priority }`: add `columnId?` (Status), `assigneeId?` (Ansvarig), `dueDate?` (FГ¶rfallodatum). `relatedDocumentId` already exists (the kravlГ¤nk в†’ `LawListItem`). All four extra fields are **optional**. `createTask` ([`app/actions/tasks.ts:1164`](../../app/actions/tasks.ts)) already accepts every one вЂ” no new dispatch wiring; the `approvePendingAction` `CREATE_TASK` branch maps the extra params straight through (`relatedDocumentId` в†’ `linkedListItemIds: [id]`).
2. **Smart defaults from chat context (no extra round-trip):**
   - **KravlГ¤nk** вЂ” when the chat context resolves to a `LawListItem` (`contextType` law/change + a `contextId`), default `relatedDocumentId` to that `contextId`. The `AgentToolContext` (`contextType` + `contextId`) is already threaded into `create_task` from `app/api/chat/route.ts` (Story 14.22). Not obligatory вЂ” clearable.
   - **Ansvarig** вЂ” default `assigneeId` to the linked item's `responsible_user_id` (`LawListItem.responsible_user_id`, `prisma/schema.prisma:381`; or `LawListItemRequirement.responsible_user_id` for kravpunkt-scoped chats, Story 20.1) when a kravlГ¤nk is set; otherwise unassigned ("Ingen tilldelad").
   - **Status** вЂ” default to the workspace's first board column ("Att gГ¶ra"); `createTask` already does this when `columnId` is omitted.
3. **Agent awareness** вЂ” new read tool `list_workspace_members` returning workspace members (`{ id, name, role }[]`) + board columns (`{ id, name, position }[]`), workspace-scoped (member data via `getWorkspaceMembers()` at `app/actions/document-list.ts:1151`), registered in `createAgentTools`. Lets the agent *propose* Ansvarig + Status by name. Tool description steers: only suggest an assignee on a clear signal (the kravlГ¤nk's responsible person, or an explicit user request); otherwise leave unassigned. (Without this tool the fields still function as user-fill controls.)
4. **Renderer mirrors the task-modal controls** вЂ” re-implement (don't import) so they drive `params` edits, matching the modal's visual treatment. **The modal's controls are inline JSX in `details-box.tsx` wired to live mutation actions, NOT extracted standalone components** вЂ” there is no drop-in `<StatusSelect>` / `<MemberPicker>` / `<DatePicker>`. Reference treatments to mirror: Status column `<Select>` (`details-box.tsx:229`), Ansvarig member `<Select>` (`details-box.tsx:263`), FГ¶rfallodatum `<Popover>`+`<Calendar>` (`details-box.tsx:323-355`). Prioritet already uses the canonical `getPriorityBadgeProps` + tone-aware Badge pill (14.22 polish). Lay out as the modal's compact "Detaljer" rows (Status / Ansvarig / FГ¶rfallodatum / Prioritet) so the inline card stays scannable.
5. **KravlГ¤nk display** вЂ” when `relatedDocumentId` is set, the card shows the linked law list item as a dismissable chip ("Kopplad till: <titel>"); dismissing clears the link. Resolve the title from the `LawListItem`.
6. Editing any new field debounces 500ms в†’ `updatePendingActionParams` (same pattern as 14.22's title/description/priority).
7. `APPROVED` state surfaces the resolved values (assignee name, status, due date, linked kravpunkt) on the read-only summary, matching the created task.
8. **Non-regression:** a minimal proposal (`{ title, description, priority }`, no status/assignee/due/link) behaves exactly as 14.22 вЂ” the new fields are additive + optional. This rich-create assignment is **distinct from the `ASSIGN_TASK` action type** (14.23): rich-create assigns *at creation*, `ASSIGN_TASK` reassigns an *existing* task вЂ” both write `Task.assignee_id`, different moments.

## Dev Notes

- **`createTask` already accepts the full field set** вЂ” `{ title, description?, columnId?, assigneeId?, dueDate?, priority?, linkedListItemIds? }` at [`app/actions/tasks.ts:1164`](../../app/actions/tasks.ts). The rich proposal is additive at the data layer: extend the tool schema + `params` + renderer; the dispatch maps params в†’ these args 1:1 (`relatedDocumentId` в†’ `linkedListItemIds: [id]`, the 14.22 mapping).
- **Context is already threaded** вЂ” `AgentToolContext` (`{ assistantMessageId, contextType, contextId }`) reaches `create_task` via `createAgentTools` (Story 14.22). Use `contextId` for the kravlГ¤nk default when `contextType` is a law/change context.
- **Assignee-default signal** вЂ” `LawListItem.responsible_user_id` (`prisma/schema.prisma:381`); finer-grained `LawListItemRequirement.responsible_user_id` (Story 20.1) for kravpunkt-scoped chats.
- **Mirror the task-modal controls (re-implement, don't import)** вЂ” the modal's status / assignee / date controls are inline JSX in `components/features/tasks/task-modal/details-box.tsx` bound to live mutation actions, NOT extracted components (`right-panel-rail.tsx` is read-only display). Re-implement against the same visual treatment to drive `params` edits. Member list: workspace-scoped `getWorkspaceMembers()` (`app/actions/document-list.ts:1151`); no shared `<MemberPicker>` exists. Priority already uses `getPriorityBadgeProps` + the tone-aware `Badge`.
- **Distinct from `ASSIGN_TASK` (14.23)** вЂ” assignment *at creation* (here) vs post-hoc reassignment (14.23); both write `Task.assignee_id`.
- **"Most relevant member" reasoning** beyond the deterministic `responsible_user_id` default (expertise / workload / history) is **Epic 19** scope вЂ” out of scope here.
