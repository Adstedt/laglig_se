# Story 16.6a: Generation cost instrumentation — measure before optimising

## Status

Draft

## Story

**As a** product owner watching Anthropic API spend per workspace signup,
**I want** the `generate-law-list` agent skill to record per-step token usage and per-run cost telemetry,
**so that** the next optimisation story (16.6b) is grounded in real measured data instead of speculation about which tokens are actually expensive.

This story is a deliberate **read-and-write-instrument-only** scope, split out of the original Story 16.6 draft after PO validation flagged that the optimisation ACs (tool-result caching, step-cap, search-filter) rested on fabricated payload-size estimates. Before changing how the agent runs, we need ground truth on:

- How many input tokens does each step actually consume?
- How many of those are cached (via the v1 caching shipped in commit `2aaa3e6`)?
- How does the cumulative input-token cost grow across the 20-step loop?
- What proportion of total spend is `input` vs `output` vs `cached_input`?
- How many tool calls does a typical run actually make, and which tool's results are biggest in practice?

Once 16.6a lands and a few generations have been captured, 16.6b will draft optimisation ACs against the **real distribution** — not against speculation.

**Cost context that prompted this:** owner observed $45 × 2 = $90 of Anthropic spend on 2026-05-18-19 across 3-4 dev generation runs. The v1 caching commit (`2aaa3e6`) added a single cache breakpoint on `SYSTEM_PROMPT` as a 5-line zero-risk fix — estimated 30-40% reduction but never measured. This story closes that measurement gap and unlocks the next optimisation pass.

**No agent-loop behavioural changes in scope.** This story only adds:
1. Per-step structured logging inside the existing `onStepFinish` hook
2. One `OnboardingEvent` row per successful generation summarising total token usage
3. A read-only aggregate script for ops/spend tracking

If any AC drifts toward "change how the agent runs" — pause and move that into 16.6b.

## Acceptance Criteria

### Per-step logging (read of existing data, no behavioural change)

1. **`onStepFinish` in `lib/agent/skills/generate-law-list.ts:250-263`** is extended to capture per-step token usage from `event.usage`. AI SDK 6.x exposes the fields below on each step:
    - `inputTokens` — total prompt tokens sent (includes any not-yet-cached system + tool defs + accumulated message context)
    - `outputTokens` — model completion tokens
    - `cachedInputTokens` — tokens served from cache (Anthropic prompt cache hits)
    - `reasoningTokens` (if present — Sonnet 4.6 may emit reasoning blocks)
2. **Structured per-step log** written via `console.log` with a stable prefix (`[GEN_COST_STEP]`) so a `grep` against Vercel logs (or local dev terminal) surfaces the breakdown. JSON payload shape:
    ```ts
    {
      tag: 'GEN_COST_STEP',
      workspaceId: string,
      stepIndex: number,         // 0-based
      toolCalls: string[],       // names of tools invoked this step (event.toolCalls.map(t => t.toolName))
      inputTokens: number,
      cachedInputTokens: number,
      outputTokens: number,
      reasoningTokens?: number,
    }
    ```
3. **Do NOT swallow log failures.** If `console.log` is missing fields (e.g., the SDK version doesn't expose `cachedInputTokens` yet), still log what's available and write `null` for absent fields. Document any missing field in Completion Notes so 16.6b knows what's available.

### Per-run telemetry (best-effort `OnboardingEvent`)

4. **New `OnboardingEvent.event_type = 'generation_cost'`** written ONCE per successful generation, in the success-path block **AFTER** `const lawList = await prisma.lawList.findFirst({...})` **and BEFORE** `const durationMs = Date.now() - startTime` (grep anchors avoid line-ref drift — currently around lines 271-283 post v1 caching commit `2aaa3e6`, but exact line numbers will shift again as the file evolves). Payload shape:
    ```ts
    {
      list_id: string | null,            // generated list ID if any
      item_count: number,                 // result.itemCount
      groups: string[],                   // result.groups
      duration_ms: number,                // result.durationMs
      total_input_tokens: number,         // result.totalUsage.inputTokens
      total_output_tokens: number,        // result.totalUsage.outputTokens
      total_cached_input_tokens: number,  // cache reads — result.totalUsage.cachedInputTokens (else 0)
      total_cache_write_tokens: number,   // cache writes — see Task 1 for SDK field-name verification
      reasoning_tokens: number,           // result.totalUsage.reasoningTokens (else 0) — N3: useful for diagnosing extended-thinking cost
      step_count: number,                 // number of steps the agent loop took (from event tracking)
      estimated_cost_usd: number,         // computed via estimateCostUsd from lib/usage/cost-estimator.ts (see AC 5)
    }
    ```
5. **Reuse the existing cost-estimator primitive — DO NOT duplicate pricing inline.** `lib/usage/cost-estimator.ts:30-69` is the canonical pricing source for this codebase (shipped Story 14.27 for `ChatUsageEvent`). It already has Sonnet 4.6 + Opus 4.x + Haiku 4.5 + gpt-4-turbo rates with correct cache-read AND cache-write handling. Import and use:

    ```ts
    import { estimateCostUsd } from '@/lib/usage/cost-estimator'

    const estimated_cost_usd = estimateCostUsd({
      model: 'claude-sonnet-4-6',
      inputTokens: result.totalUsage.inputTokens ?? 0,
      outputTokens: result.totalUsage.outputTokens ?? 0,
      cacheReadInputTokens: result.totalUsage.cachedInputTokens ?? 0,
      cacheWriteInputTokens: /* see Task 1 — SDK field-name verification */ 0,
      reasoningTokens: result.totalUsage.reasoningTokens ?? 0,
    })
    ```

    Important: the cost-estimator correctly accounts for cache **writes** at 1.25× input rate ($3.75/MTok for Sonnet 4.6). A naive cache-read-only formula would systematically under-report cost since the FIRST run of each generation pays full cache-write cost. If we duplicate pricing here, the two formulas will drift and the diagnostic data will be unreliable.
6. **Fire-and-forget pattern.** Telemetry write wrapped in try/catch; logs `[GEN_COST_EVENT_WRITE_FAIL]` on error; never re-thrown. Mirrors the existing `recordOnboardingEvent` fail-safe pattern at `app/actions/onboarding-modal.ts:34-50`.
7. **Reuse `recordOnboardingEvent`** from `app/actions/onboarding-modal.ts`. No new server action needed — the `event_type` field is `String` and accepts any string value.

### Aggregate script

8. **New dev/ops script: `scripts/aggregate-generation-cost.ts`** (mirrors the pattern of `scripts/peek-nordviken-status.ts` — read-only diagnostic). Uses the SAME `estimateCostUsd` primitive from `lib/usage/cost-estimator.ts` as AC 5 so per-run cost numbers stay consistent across both surfaces (per-run row write AND aggregate report). Output: a per-day summary printed to stdout:
    ```
    Generation cost — last 30 days
    -------------------------------------------------------------------------
    Date         Runs  Avg input  Avg cached  Avg output  Total $   Avg $/run
    2026-05-19    4    420K       180K        35K         $42.40    $10.60
    2026-05-18    3    415K       175K        38K         $32.10    $10.70
    ...
    Totals       7    —          —           —           $74.50    —
    ```
    The `Total $` and `Avg $/run` columns are derived from each row's pre-computed `estimated_cost_usd` field (written by AC 5 at OnboardingEvent-write time). The script does NOT re-compute cost from raw tokens — that would double-implement pricing and risk drift. Sum the `estimated_cost_usd` field directly.
9. **Read-only.** Script must NOT write to the DB. Uses `prisma.onboardingEvent.findMany` with `event_type: 'generation_cost'` filter. Groups in JS by `created_at::date`.
10. **Date range arg.** Defaults to last 30 days; accepts `--days=N` for custom range. Lazy implementation: process.argv parsing, no commander/yargs.

### Data collection + findings

11. **Capture 3-5 real generation runs in dev** post-merge. Completion Notes **MUST** include a permanent artifact in one of two forms:
    - **(a) Inline table** in Completion Notes with one row per run, columns:
      `[run_id | total_input_tokens | cached_read_tokens | cache_write_tokens | output_tokens | step_count | est_cost_usd | duration_ms]`
    - **(b) Saved log file** at `docs/qa/findings/16.6a-cost-data.md` (or similar) attached to the repo, referenced inline in Completion Notes with a brief summary.

    Plus narrative summary covering:
    - Average tokens per step across the run (peak step, lowest step)
    - Total input vs output vs cache-read vs cache-write breakdown
    - Identified cost-driver tokens (which step + which tool calls are the biggest contributors)
    - Anthropic cache hit ratio (cache_reads / total_input — high ratio = v1 caching is working)
    - Per-run estimated cost spread across the 3-5 runs

12. **Recommend ACs for 16.6b in Completion Notes** using this structured format. List the top 2-3 cost-reduction levers ordered by `potential_saving_pct × (1 / implementation_effort)`. For each lever:

    - **Lever**: short name (e.g., "Cache `get_template_laws` result on tool-result message")
    - **Potential saving**: `<percentage>` grounded in 16.6a's captured data. Cite the specific token number that supports the estimate (e.g., "saves ~120K tokens per run on average across 5 runs measured = ~36% input reduction")
    - **Files affected**: paths
    - **Implementation effort**: rough estimate (hours / half-day / full day)
    - **Risk callouts**: anything that could regress agent quality or break tests
    - **Pre-implementation design discussions**: any open questions that should be resolved before drafting 16.6b's ACs

    Explicit "avoid speculation" rule: every percentage estimate must be backed by a captured-data citation. No "this should save ~50%" without a number from the runs.

### Guard against scope creep

13. **NO changes to** `stopWhen: stepCountIs(20)`, the system prompt, tool definitions, the `search_laws` tool, the agent loop architecture, or the response shape. Any change that affects what the agent does is out of scope and moved to 16.6b.
14. **NO new dependencies.** Uses `console.log`, existing Prisma client, existing `recordOnboardingEvent`. No analytics SDK, no logger library.

### Why this story exists (PO validation history)

The original Story 16.6 draft (now deleted) attempted to scope tool-result caching + step-cap + search-filter ACs in one shot, with estimated "70-80% reduction" targets. PO validation flagged three critical issues:

- M1: `search_laws` already returns ≤20 results (default 5), not "30-50". The "filter to top 10" AC was solving a problem that doesn't exist.
- M2: `get_template_laws` returns ~2-8K tokens per call, not "~30K". Order-of-magnitude error.
- M3: The cumulative-token diagram (Step 15 ≈ 410K input) was speculation, not measurement.

Story 16.6a (this story) closes the measurement gap. 16.6b will be drafted *after* 16.6a's data lands, with ACs grounded in real numbers.

### Existing patterns to follow

- **`recordOnboardingEvent(eventType, payload)`** from `@/app/actions/onboarding-modal` — fire-and-forget telemetry helper. Don't `await` in critical path; wrap in try/catch with `[GEN_COST_EVENT_WRITE_FAIL]` log on error.
- **Structured `console.log` with a stable tag prefix** — the codebase doesn't use a logger library yet. `[GEN_COST_STEP]` is parseable via grep against Vercel logs.
- **Dev script structure** — `scripts/peek-nordviken-status.ts:1-20` shows the dotenv + Prisma init pattern. Mirror it exactly.
- **No-thrown-telemetry rule** — the same fail-safe pattern lives at `app/actions/onboarding-modal.ts:34-50` and `lib/email/email-service.ts`. New telemetry writes must follow this — never break the user-facing path.

### Key decisions baked into this story

- **Instrument first, optimise later.** Three lines of unsupported cost claims got into the original 16.6 because there was no measurement loop. 16.6a fixes that *before* code changes are even on the table.
- **OnboardingEvent, not a new table.** Generation-cost telemetry is low-volume (one row per signup) and reuses the existing event-table infrastructure. A dedicated `LLMUsageEvent` table is premature.
- **Reuse `lib/usage/cost-estimator.ts`, NOT inline pricing constants.** PO validation v0.2 flagged that v0.1's "hard-coded pricing in the skill file" suggestion was wrong — the cost-estimator already exists (shipped Story 14.27 for `ChatUsageEvent`) and is the canonical pricing source for this codebase. Reusing it avoids drift, accounts for cache writes correctly (1.25× input — v0.1's formula missed this and would have under-reported cost ~20% per run on first cache writes), and handles model-name fallback if Sonnet 4.6 gets superseded.
- **No model swap, no behavioural change.** The whole point of 16.6a is to NOT change agent behaviour. Any drift in this direction → move into 16.6b.
- **`console.log` instead of a logger.** Vercel captures stdout; `[GEN_COST_STEP]` prefix is greppable. A logger library is an unnecessary dep for a 1-2 line change.
- **Estimated cost is "best-effort", not authoritative.** Anthropic Console is the source of truth. The aggregate script + `estimated_cost_usd` field are for fast in-dev visibility, not billing reconciliation.

### Out of scope explicitly

- **Anything that changes how the agent runs** — caching breakpoints beyond v1, step-cap reduction, `search_laws` filter, prompt edits, model swap, tool changes. All these are 16.6b territory.
- **Production cost dashboard** — `scripts/aggregate-generation-cost.ts` is sufficient. An admin UI page is a future story.
- **Cross-loop caching** — Anthropic ephemeral cache is 5-min TTL per request; sharing across separate generations requires the persistent cache, deferred.
- **Per-tool token attribution** — measuring "how many tokens does each individual tool's result add to context" would require more granular instrumentation than the AI SDK exposes today. The per-step log captures total step input, which is sufficient signal for 16.6b.
- **Embedding costs** — `lib/agent/retrieval.ts` may use embeddings (cheap, separate from generation). Out of scope here; if 16.6a's data points at retrieval as a non-trivial cost, capture separately.
