# Story 21.14: Permissions — `audit:seal` scope + AUDITOR read-access threading

## Status

Done

## Story

**As a** workspace owner or admin,
**I want** granular control over who can seal cycles (role-based `audit:seal` with a runtime lead-auditor override) and explicit coverage that AUDITOR role can read cycle data without gaining any edit ability,
**so that** segregation of duties for sealing is enforceable, the permission matrix stays honest, and the later `sealCycle` action (Story 21.9) can land with its authorization check already built and tested.

## Acceptance Criteria

5. **Runtime authorization helper `lib/compliance-audit/authorization.ts`.** Create a new file `lib/compliance-audit/authorization.ts` exporting two named functions:

   ```ts
   export async function isLeadAuditor(
     prismaClient: PrismaClient | Prisma.TransactionClient,
     args: { userId: string; cycleId: string; workspaceId: string }
   ): Promise<boolean>
   ```
   Resolves true if there is a `ComplianceAuditCycle` row matching `id = args.cycleId AND workspace_id = args.workspaceId AND lead_auditor_user_id = args.userId AND deleted_at = null`. Uses `findFirst({ select: { id: true } })` (idiom: minimal select, boolean coerce). Accepts either the base `prisma` client OR a transaction client — mirrors `assertCycleEditable(tx, cycleId)` pattern from architecture §5.1 so 21.9's `sealCycle` can call the helper inside its seal transaction without re-reading from a separate connection.

   ```ts
   export async function canSealCycle(
     prismaClient: PrismaClient | Prisma.TransactionClient,
     args: { role: WorkspaceRole; userId: string; cycleId: string; workspaceId: string }
   ): Promise<boolean>
   ```
   Resolves true if EITHER `canSealAuditCycle(args.role)` (role has the scope) OR `isLeadAuditor(prismaClient, ...)` (runtime override). Short-circuits on the role check — no DB hit when the role already has `audit:seal`. `[Source: architecture/epic-21-lagefterlevnadskontroll.md §5.1 (cycle-guards.ts signature pattern) + §12.2 (lead-auditor runtime override)]`

### Security Review

**PASS.** Authorization surface verified across four defence layers:

1. **Default-deny:** `audit:seal` granted only to OWNER + ADMIN. MEMBER / HR_MANAGER / AUDITOR blocked by role. Locked down by per-role `hasPermission('audit:seal')` tests.

Forged client-side state cannot grant seal authority — the server-side `canSealCycle` is the authoritative check (which Story 21.9 will call inside the seal transaction).
