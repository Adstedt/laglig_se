## Story

**As** the compliance agent platform,
**I want** a file-based skill convention (`lib/agent/skills/<name>/SKILL.md` + companions) and a pure loader library that lists, loads, and reads the tool-whitelist + context-mapping from those files,
**so that** Story 19.7 can wire skills into the agent (prompt injection, `activate_skill`, registry narrowing) against a **proven, tested loader** instead of designing the file format and the integration simultaneously.

The loader API below is shaped to fit those consumers (see Dev Notes "Designed-for consumers").

## Acceptance Criteria

### Directory convention

1. A **skill is a subdirectory of `lib/agent/skills/`** containing `SKILL.md` (**required**) + optional `PROCEDURE.md`, `STYLE.md`, `CRITERIA.md`, and an `examples/` dir. `SKILL.md` = YAML frontmatter + markdown body. **Frontmatter schema (Zod-validated by the loader):**
   - `name: string` — kebab-case; MUST equal the directory name.
   - `description: string` — one-line, English with Swedish domain terms quoted (consumed by 19.7's available-skills block).
   - `contextTypes: ('global'|'task'|'law'|'change')[]` — which chat contexts this skill is *primary* for (`[]` = never auto-primary). Drives `getPrimarySkillForContext` (19.7 consumes it).
   - `tools: string[]` — the tool whitelist (tool names the skill's procedure needs; 19.7 consumes it for registry narrowing). Not validated against the live `ToolName` union in 19.6 (the loader is registry-agnostic — keeps it dependency-free; 19.7 cross-checks when it wires the filter).
   [Source: PRD 19.6 + conceptual model lines 41-43]
2. **Loader scan rules:** include only subdirectories that contain a `SKILL.md`; **ignore loose files** in `lib/agent/skills/` (e.g. `generate-law-list.ts`); **skip `_`-prefixed directories** (authoring templates/examples are not real skills). A subdir whose `SKILL.md` frontmatter fails the Zod schema, or whose `name` ≠ dirname, is **skipped with a `console.warn`** (never throws — one malformed skill must not break the loader for the others).

### Loader library (`lib/agent/skill-loader.ts`)

3. New `lib/agent/skill-loader.ts` — **pure functions, no agent/prisma/runtime deps** — each accepting an **optional `baseDir`** (default `lib/agent/skills/`, so tests pass a fixtures dir):
   - `listSkills(baseDir?): SkillMeta[]` — scan + parse + validate frontmatter (gray-matter + Zod) → `SkillMeta = { name; description; contextTypes; tools }`. Cached per resolved `baseDir` (skills are deploy-static; mirror the `BASE_PROMPT` read-once pattern at `system-prompt.ts:44`).
   - `loadSkill(name, baseDir?): string | null` — assemble the body: `SKILL.md` body + `PROCEDURE.md` + `STYLE.md` + `CRITERIA.md`, each under a `## ` section header and **only when the file exists**; `null` if the skill dir / `SKILL.md` is missing.
   - `getSkillToolWhitelist(name, baseDir?): string[]` — the `tools` frontmatter (`[]` for none / unknown skill).
   - `getPrimarySkillForContext(contextType, baseDir?): string | null` — the skill `name` whose `contextTypes` includes the given context; `null` when none match. If >1 match, return the first by sorted name + `console.warn` (deterministic).
   - Export the `SkillMeta` type + the frontmatter Zod schema (so 19.7 can reuse it).

### Authoring template + tests

4. Ship `lib/agent/skills/_template/` (skipped by the loader): a `SKILL.md` with fully-commented example frontmatter + stub `PROCEDURE.md` / `STYLE.md` / `CRITERIA.md`, plus `lib/agent/skills/README.md` documenting the convention (the file set, the frontmatter fields, the `_`-prefix rule, bilingual expectation per principle #5).
5. **Unit tests** against a `tests/fixtures/skills/` dir (passed as `baseDir`): a valid skill (`contextTypes:['change']`, `tools:['get_change_details','search_laws']`); a `contextTypes:[]` skill; a `_skip` dir (assert skipped); a loose `notaskill.ts` (assert ignored); a malformed-frontmatter dir (assert warn+skip, others still load); a `name`≠dirname dir (assert warn+skip).

### Scope guards

6. **No agent integration in 19.6 — nothing imports the loader yet.** Do NOT touch `buildSystemPrompt`, `createAgentTools`, the chat route, or add `activate_skill`; do NOT migrate `ASSESSMENT_WORKFLOW`. The loader is a standalone library validated entirely by its unit tests → **zero production surface, zero behaviour change** (a `tsc`/`eslint`/full-suite pass is the only footprint). No schema change / no migration.
7. **Out of scope (→ 19.7):** `buildSystemPrompt` skill injection + `<available_skills>`; the `activate_skill` meta-tool; `createAgentTools` `activeSkills` + `ALWAYS_AVAILABLE` registry narrowing; route wiring; `assess_change`/`gap_analysis`; the `ASSESSMENT_WORKFLOW` migration; the CI bilingual-lint. All land with their first real consumers in 19.7.
