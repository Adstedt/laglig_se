# Story 19.8: `draft_styrdokument` skill (type-aware) + Swedish template library seed

## Status

Done

## Story

**As a** compliance consultant using the Laglig agent,
**I want** the agent to author each kind of styrdokument with the canonical Swedish structure, tone, and quality checks for *that document type* (a riskbedömning gets a risk matrix, a handlingsplan gets ansvarig/klart-datum rows, a checklista gets verifiable yes/no points),
**so that** an agent-drafted document arrives shaped like the document a domain expert would write — not a generic prose blob — and the quality gate rejects type-wrong drafts before they reach my approval card.

## Acceptance Criteria

1. A `draft_styrdokument` skill exists at `lib/agent/skills/draft_styrdokument/` with `SKILL.md` (frontmatter: `name: draft_styrdokument`, English `description` with Swedish domain terms quoted, `contextTypes: []` — activation-only, never auto-primary, like `gap_analysis`; `tools` whitelist), plus `PROCEDURE.md`, `STYLE.md`, `CRITERIA.md`. The skill loads through the existing `lib/agent/skill-loader.ts` with no validation warnings and is reachable via `activate_skill('draft_styrdokument')`.

2. `PROCEDURE.md` is an English step-by-step playbook for drafting a styrdokument: resolve `docType` → search for an existing document of that type/topic → pull company + law context → **consult the per-type module for that `docType`** → draft the type-canonical structure → run the type-specific quality checks → call `draft_styrdokument` → (optionally) propose a review task / kravpunkt link. `STYLE.md` carries the cross-cutting Swedish tone + KP-001 (verifiable-obligation framing) + GR-001 (citation-grounding) written once. `CRITERIA.md` carries cross-cutting guardrails: must-cite relevant law, currency, and **CP-001 (no internal identifiers / status enums leaked into prose)**.

3. Per-type modules exist at `lib/agent/skills/draft_styrdokument/types/*.md` — exactly one per `WorkspaceDocumentType`: `policy.md`, `risk_assessment.md`, `action_plan.md`, `procedure.md`, `instruction.md`, `checklist.md`, `report.md`, `other.md`. Each module holds the type-specific **STRUCTURE** (canonical Swedish skeleton), **per-type STYLE** (verb form / perspective — Policy declarative "vi ska"; Instruktion imperative "utför"; Checklista verifierbar "är genomförd"), and **per-type CRITERIA** (machine-checkable invariants).

4. The per-type module for the resolved `docType` is **surfaced into the agent's context** when the skill is in play (see Dev Notes "Design decision: surfacing the type module" — implement the chosen option). The agent must be able to read the structure/style/criteria for the specific `docType` it is drafting; it must NOT need the seven irrelevant modules dumped verbatim for every draft.

5. The generic content-quality gate in `lib/agent/tools/draft-styrdokument.ts` (today: `blocks.length >= 3 && hasHeading`) is upgraded to a **type-aware gate keyed on the existing `docType` arg**. At minimum, the gate enforces these machine-checkable invariants and returns a `wrapToolError` (persisting nothing) when violated:
   - `RISK_ASSESSMENT` → the Tiptap doc MUST contain a `table` node (the riskmatris).
   - `ACTION_PLAN` → the doc MUST contain a `table` node (the åtgärdsrader).
   - `CHECKLIST` → the doc MUST contain a **structured enumeration**: either a `table` node (the kontrollpunkts-tabell — this is what the seeded `Checklista` template uses) OR a `bulletList`/`orderedList` with ≥3 items. **Do NOT require `taskList` — it is not in the editor schema** (see Dev Notes "Editor Tiptap schema").
   - All other types → retain the current `≥3 blocks + ≥1 heading` baseline.
   The existing baseline gate (and its current error message) remains the floor for every type; the type rules are additive.

6. The error returned on a type-gate failure names the missing element in Swedish so the agent can self-correct (e.g. *"En riskbedömning måste innehålla en riskmatris (tabell) med riskkälla, sannolikhet och konsekvens."*), not a generic "too short" message.

7. The Swedish template library seed (`scripts/seed-document-templates.ts` + `lib/documents/template-content.ts`) is extended to cover **≥1 active template per `WorkspaceDocumentType`**. The five existing templates (Arbetsmiljöpolicy/POLICY, Riskbedömning/RISK_ASSESSMENT, Handlingsplan/ACTION_PLAN, Rutin/PROCEDURE, Checklista/CHECKLIST) are retained; new templates are added for the uncovered types **INSTRUCTION, REPORT, OTHER**, plus the GDPR-track minimums called out in the PRD: **Dataskyddspolicy** (POLICY), **Incidenthanteringsrutin** (PROCEDURE), and a **personuppgiftsbiträdes-/leverantörspolicy** (POLICY). The RISK_ASSESSMENT and ACTION_PLAN template `content_json` each include the table scaffold so a template-seeded draft passes the new type gate. Seeding stays idempotent (deterministic UUIDs, `--dry-run`/`--force` honoured).

8. The no-regression harness `tests/unit/agent/tools/registry-narrowing.test.ts` asserts the safety invariant for this activation-only skill: **every tool in `getSkillToolWhitelist('draft_styrdokument')` is a member of `ALWAYS_AVAILABLE`** (so the skill is reachable and complete even though the route never re-narrows on mid-conversation `activate_skill` — see Dev Notes "Reachability constraint"). New unit tests cover: the type-aware gate (accept + reject per RISK_ASSESSMENT / ACTION_PLAN / CHECKLIST; baseline types unaffected), the skill loads/validates via the loader, and the type-module surfacing mechanism. `tsc` and `eslint` clean; full agent unit suite green.

9. No regression to the existing `draft_styrdokument` tool behaviour for the baseline types: a valid POLICY/PROCEDURE/INSTRUCTION/REPORT/OTHER draft that passed before still passes, still produces a `DRAFT_DOCUMENT` `PendingAgentAction`, still strips the duplicate leading title heading, still sanitizes dangerous link hrefs, and still resolves/auto-derives context links exactly as today.

### Skill convention + loader (how a skill is assembled)

- A **skill is a subdirectory** of `lib/agent/skills/` containing a required `SKILL.md` (YAML frontmatter + body) plus optional `PROCEDURE.md` / `STYLE.md` / `CRITERIA.md`. `_`-prefixed dirs and loose files are ignored. Frontmatter that fails validation, or whose `name` ≠ directory name, is skipped with a `console.warn` (one bad skill never breaks the rest). [Source: lib/agent/skills/README.md; lib/agent/skill-loader.ts]
- Frontmatter schema (`skillFrontmatterSchema`): `name` (must equal dir), `description`, `contextTypes: ('global'|'task'|'law'|'change')[]` (default `[]`), `tools: string[]` (default `[]`). [Source: lib/agent/skill-loader.ts]
- `loadSkill(name)` assembles: SKILL.md body, then each present companion under a `## Procedure` / `## Style` / `## Criteria` header — in that order. **It does NOT currently descend into subdirectories** → the `types/*.md` modules will not be picked up without the Task 3 change. [Source: lib/agent/skill-loader.ts `COMPANIONS` + `loadSkill`]
- `getPrimarySkillForContext(contextType)` injects a skill's full body into `buildSystemPrompt` only when `contextTypes` includes the chat's context. With `contextTypes: []`, `draft_styrdokument` is **never auto-injected** — it surfaces only via the `activate_skill` tool, whose result is `loadSkill(name)`. The agent is told it exists through the `<available_skills>` list in the system prompt. [Source: lib/agent/system-prompt.ts (lines 304–313); lib/agent/tools/activate-skill.ts]
- Bilingual rule (principle #5): `PROCEDURE.md` English; `STYLE.md` exemplars Swedish; `description` English with Swedish domain terms quoted. [Source: lib/agent/skills/README.md]

### Reachability constraint (activation-only skill) — read before choosing the design

`draft_styrdokument` is `contextTypes: []` → it is **never** a primary skill, so `getPrimarySkillForContext` never returns it and the route's `activeSkills = primarySkill ? [primarySkill] : []` never contains it. The agent reaches the skill only via the `activate_skill` tool mid-conversation — and **the route does NOT re-narrow the tool registry on mid-conversation activation** (`app/api/chat/route.ts` comment: "Mid-conversation activate_skill → activeSkills re-injection stays deferred"). Consequence: **a tool gated behind this skill's whitelist would be unreachable.** Therefore every tool the skill needs MUST already be in `ALWAYS_AVAILABLE` (all eight candidates are — they are reads + the ad-hoc `create_task`/`draft_styrdokument` writes). The SKILL.md `tools` whitelist is effectively *documentation of intent* here, not an access grant. This is also why Option B's reader tool (if chosen) must be `ALWAYS_AVAILABLE`, not skill-gated. [Source: lib/agent/tools/index.ts (`ALWAYS_AVAILABLE`/`SKILL_GATED_TOOLS`); app/api/chat/route.ts (lines ~305–321)]

### Design decision: surfacing the type module (AC 4) — **resolve before coding**

The agent must see the structure/style/criteria for *one* `docType`, not all eight. Two viable options — **Option A is recommended**:

- **Option A (recommended) — generic loader extension.** Teach `loadSkill()` to append, after the three companions, a `## Type modules` section containing each present `types/*.md` under `### Type: <stem>`. Pros: zero new tools, "skills are files" preserved, flows through `activate_skill` automatically, deploy-static + cache-friendly. Con: all eight modules enter context on activation — acceptable because the modules are short skeletons and only load when the skill is explicitly activated (not on every chat). Keep the SKILL.md `tools` whitelist unchanged.
- **Option B — lazy per-type reader tool.** Add a tiny `read_styrdokument_type(docType)` tool that returns one module; add it to the whitelist + narrowing harness. Pros: only the needed module enters context. Cons: a whole new tool + registry entry + decision-log wrapping for one file read; more surface, more tests, and it must be gated/whitelisted correctly.

**Recommendation:** Option A unless the eight-module context size proves material at review. Whichever is chosen, the lookup key is `docType.toLowerCase()` → `types/<stem>.md`, which is why Task 2 fixes the filenames to the enum stems.

### Data models / schema constraint (affects AC 7)

- `WorkspaceDocumentTemplate` columns: `id`, `name`, `description?`, `document_type` (`WorkspaceDocumentType`), `content_json` (Json), `is_active`, `sort_order`, timestamps. **There is NO `metadata` column.** [Source: prisma/schema.prisma (model `WorkspaceDocumentTemplate`, ~line 2161)]
- The PRD asks templates to carry `metadata.applicable_law_ids[]` + `metadata.docType`. `docType` already exists as `document_type`. **`applicable_law_ids` has no home without a schema migration.** Decision for this story: **descope `applicable_law_ids`** — the skill matches user intent → type via `document_type` + `name`/`description`, which is sufficient for v1. Adding a `metadata` JSON column (and law-id wiring) is a clean follow-up. If the PO wants it now, it requires a Prisma migration, which is applied **manually by the user** — flag it, do not run it. [Source: docs/prd/epic-19-agent-partner-skills-subagents.md (19.8 layer 3); memory: migrations-run-manually]
- `WorkspaceDocumentType` enum values (exact): `POLICY`, `RISK_ASSESSMENT`, `ACTION_PLAN`, `PROCEDURE`, `INSTRUCTION`, `CHECKLIST`, `REPORT`, `OTHER`. [Source: prisma/schema.prisma]
- Existing seed coverage (`lib/documents/template-content.ts`, Story 17.7): Arbetsmiljöpolicy/POLICY, Riskbedömning/RISK_ASSESSMENT, Handlingsplan/ACTION_PLAN, Rutin/PROCEDURE, Checklista/CHECKLIST. **Uncovered: INSTRUCTION, REPORT, OTHER.** Seed script upserts by deterministic UUID (`TEMPLATE_IDS.*`), honours `--dry-run`/`--force`. [Source: scripts/seed-document-templates.ts; lib/documents/template-content.ts]

### Editor Tiptap schema (exact node-type strings the gate matches on)

The gate (Task 4) keys on Tiptap node `type` strings, so they must match the WorkspaceDocument editor's actual schema. The editor (`components/ui/rich-text-editor.tsx`) loads **`StarterKit` + `@tiptap/extension-table` (Table/TableRow/TableCell/TableHeader) + Link/Underline/Placeholder/Image**. Resulting node types relevant to the gate:

- Tables: `table` (rows `tableRow`, cells `tableCell`/`tableHeader`). ✅ in schema.
- Lists: `bulletList`, `orderedList` (items `listItem`) — from StarterKit. ✅ in schema.
- **`taskList`/`taskItem` are NOT loaded** — there is no `@tiptap/extension-task-list` import. A draft emitting `taskList` would not render. **The gate must NOT accept `taskList`.** [Source: components/ui/rich-text-editor.tsx (extensions list, lines 8–18, 111–148)]
- The seed builds tables via a `table(rows)` helper emitting `{ type: 'table', content: [tableRow…] }`. [Source: lib/documents/template-content.ts (helpers, ~lines 26–53)]

### The existing tool (what NOT to break — AC 9)

`lib/agent/tools/draft-styrdokument.ts` (Story 14.24): always proposes a `DRAFT_DOCUMENT` `PendingAgentAction` (no `execute:true` path — inline approval only). Pipeline: `normalizeTiptapDoc` (accepts doc-node / bare array / JSON-string) → **baseline gate `blocks.length >= 3 && hasHeading`** (returns `wrapToolError`, persists nothing on fail) → `stripLeadingTitleHeading` → `sanitizeLinkHrefs` (drops `javascript:`/`data:`/`vbscript:` hrefs, keeps text) → resolve/auto-derive `contextLinks` (drops ids that don't resolve; auto-derives from `context.contextId` in LAW/TASK chats) → `createPendingActionRow(..., 'DRAFT_DOCUMENT', params)` → `wrapWriteToolResponse` envelope with `{ pendingActionId }`. The tool already receives `docType` (enum literal array `DOC_TYPES`) — the type-aware gate keys off it with **no schema/signature change**. Insert the type checks immediately after the baseline gate so baseline behaviour for non-table/list types is identical. [Source: lib/agent/tools/draft-styrdokument.ts]
