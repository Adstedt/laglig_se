# Story 20.1: Per-Krav Assignee Model & Modal Picker

## Status

Done

## Story

**As a** compliance manager or responsible owner,
**I want** each kravpunkt to optionally carry its own `Ansvarig` that falls back to the parent law item's ansvarig when unset,
**so that** I can delegate an individual kravpunkt to the right person without changing who owns the whole law.

## Acceptance Criteria

1. `LawListItemRequirement` has a new nullable column `responsible_user_id String?` with a `User` relation (`@relation("RequirementResponsible", ...)`, `onDelete: SetNull`) and a single-column index `@@index([responsible_user_id])`; Prisma migration applies cleanly to dev with no data loss.
2. The `User` model gains a reciprocal relation field (e.g., `requirements_responsible_for LawListItemRequirement[] @relation("RequirementResponsible")`).
3. A pure helper `resolveEffectiveAssignee(krav, listItem)` is exported from `app/actions/law-list-item-requirements.ts` and returns `{ userId: string | null; isInherited: boolean }` covering three branches: krav override, inherited from parent, both null.
4. `createRequirement` accepts an optional `responsibleUserId: string | null` field in its Zod input schema (camelCase at the API boundary).
5. `updateRequirement` accepts `responsibleUserId: string | null | undefined` — `undefined` = unchanged, `null` = clear override, UUID = set override. The existing `.refine()` "at least one field" guard extends to include this field.
6. `getRequirementsForListItem` returns each krav with `responsibleUserId: string | null` plus an `effectiveAssignee: { userId, isInherited }` object populated via the shared resolver, and includes the parent list item's `responsible_user_id` in its select shape.
7. Assignee value changes write an `ActivityLog` row via the existing `logActivity` helper (`@/lib/services/activity-logger`) with **`entity_type = 'requirement'`** (matches the convention used by every other `logActivity` call in `app/actions/law-list-item-requirements.ts`, e.g., at lines 191, 308, 372, 600, 678) and **`action = 'requirement_assignee_updated'`** (follows the file's `requirement_<field>_updated` pattern, e.g., `requirement_comment_updated`, `requirement_text_updated`). `old_value` / `new_value` contain the user IDs (or `null` when no override).
8. All three mutating actions (`create`/`update`/`assignee-change`) preserve workspace-scoped access: the existing `withWorkspace(...)` wrapper from `@/lib/auth/workspace-context` is invoked with the same permission level current krav mutations use. Unauthorized callers get the same rejection shape as before.
9. `components/features/document-list/legal-document-modal/kravpunkter-checklist.tsx` renders an `<AssigneeEditor />` per kravpunkt row, prefilled with `effectiveAssignee.userId`. Position: **between the evidence-count / "Saknar bevis" badge and the `CollapsibleTrigger` expand chevron** (i.e., just before the row's expand+delete cluster at `kravpunkter-checklist.tsx` lines 460–510). The picker must not occupy horizontal space when hidden on mobile — reuse the existing avatar-only trigger size (~7×7).
10. When `effectiveAssignee.isInherited === true`, the picker's avatar trigger uses a visually muted/dimmed variant and a Radix Tooltip saying "Ärvd från lagansvarig".
11. Selecting a user from the picker calls `updateRequirement({ responsibleUserId: newId })`; the SWR cache for `getRequirementsForListItem` revalidates so the row reflects the change and `isInherited` flips to `false`.
12. The `<AssigneeEditor />` dropdown contains a conditional **"Återställ till lagansvarig"** item, visible only when `effectiveAssignee.isInherited === false` (i.e., the krav currently has a direct override). It renders **below the "Ej tilldelad" option** with a visual separator (Radix `<DropdownMenuSeparator />` or the Select equivalent already in use) so it is clearly distinct from the members list. Selecting it calls `updateRequirement({ responsibleUserId: null })` and the row returns to the inherited visual state (dimmed avatar + tooltip). **No new kebab/menu button is introduced to the row** — the existing row layout (drag → checkbox → text → evidence badge → assignee → chevron → delete X) is preserved.
13. The workspace members list is fetched once at the parent modal level and passed into the checklist as a `members: WorkspaceMemberOption[]` prop. No per-row fetching.
14. All UI copy is hard-coded Swedish (e.g., "Ärvd från lagansvarig", "Återställ till lagansvarig"), matching existing modal copy conventions.
15. Existing kravpunkter CRUD (create, edit text, toggle fulfilled, toggle bevis_required, add comment, reorder, link/unlink evidence, delete) works unchanged in the modal. Existing callers of `createRequirement`/`updateRequirement` that don't pass `responsibleUserId` compile and behave identically.

The existing `updateRequirement` used a nested ternary to pick one action string; that pattern would silently attribute an assignee change to a "text update" if any other field was also updated.
