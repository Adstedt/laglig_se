# Story 3.15: Scope AI chat history to the authoring user — Brownfield Addition

## Status

Done

## Story

**As a** workspace member using the AI chat,
**I want** my chat history (questions, answers, archived conversations) to be visible only to me,
**so that** exploratory compliance questions stay private to me and I cannot accidentally read, delete, or clear a colleague's chats.

## Story Context

**Existing System Integration:**

- **Integrates with:** `chat_messages` persistence layer (Epic 3 — Stories 3.3 built the UI + persistence, 3.10 added history/session management, 14.11 added conversation archive/search on top).
- **Technology:** Next.js server actions, Prisma, PostgreSQL. Zod input validation. SWR-cached consumers.
- **Follows pattern:** Workspace-scoped-then-user-scoped read guards used elsewhere in the codebase for user-private artefacts (e.g., `notification_preferences` scoped by `user_id + workspace_id`; agent decision logs scoped by `user_id` within a workspace). No new pattern invented.
- **Touch points in `app/actions/ai-chat.ts`:** 8 query sites require a `user_id` filter addition — one per read/mutation server action, plus the cursor-resolver `findFirst` inside `getChatHistory` (line 172). The 7 server actions: `getChatHistory`, `clearChatHistory`, `archiveConversation`, `getConversationHistory` (raw SQL), `loadConversation`, `searchConversations`, `deleteChatMessage`. Additional read used by search: `prisma.chatMessage.count` inside `searchConversations`.
- **Write actions require no change.** `saveChatMessage` (line 83) and `saveChatMessages` (line 120) already stamp `user_id: ctx.userId` on every insert — they are correct by construction and explicitly excluded from the filter additions.

## Acceptance Criteria

### Functional (privacy guarantees)

1. **User A cannot see User B's chats.** For every context (`GLOBAL`, `TASK`, `LAW`, `CHANGE`), `getChatHistory` returns only messages where `user_id = ctx.userId`. Verified via unit test asserting two users in the same workspace who have each written messages in the same task / law / change / global context see only their own rows.
   - **Cursor-resolver isolation (prevents timing oracle).** The `findFirst` cursor-lookup inside `getChatHistory` (line 172) ALSO filters by `user_id = ctx.userId`. If User A passes a pagination cursor whose UUID points to one of User B's messages, the cursor resolver returns `null` and User A gets page-1 behaviour — never leaks the cursor message's timestamp.

2. **User A cannot delete, clear, or archive User B's messages.** The three mutating server actions all filter by `user_id = ctx.userId`:
   - `deleteChatMessage(messageId)` — `findFirst` gate verifies `workspace_id` AND `user_id`; returns `'Message not found'` (not a permissions error — do not leak existence) if the message belongs to another user.
   - `clearChatHistory({contextType, contextId})` — `deleteMany.where` includes `user_id`.
   - `archiveConversation()` — `updateMany.where` includes `user_id`, so archiving only tags the caller's active GLOBAL messages with the new `conversation_id`. Two users clicking "archive" in the same workspace produce two independent archived conversations.

3. **Existing pre-story chat rows remain visible only to their original author.** No backfill. No migration of existing data. Messages stay attached to their original `user_id`; they simply stop appearing for other users. This is the desired outcome — no data is destroyed, history that the authoring user stored is preserved for them.

4. **AUDITOR role loses cross-user chat read access but retains own-chat access.** Confirmed acceptable per 2026-04-24 scope decision: the `AUDITOR` role (introduced in Story 21.14) exists for compliance-audit cycle / item / finding oversight, not chat transcript review. Any existing AUDITOR users see only their own chats post-story. **Permission interplay (verified at `lib/auth/permissions.ts:116`):** AUDITOR retains the `'ai:chat'` permission — the role's grant reads `AUDITOR: ['activity:view', 'ai:chat', 'read']` and is regression-pinned by Story 21.14 AC 10. The `'ai:chat'` permission grants the capability to write and read AI chat; the new `user_id` filter added by this story constrains what is visible. The two compose correctly without mutual change — no edit to `lib/auth/permissions.ts` or `lib/api/require-permission.ts` is required. Add an explicit Dev Note documenting that if a future "audit chat review" feature is needed it requires a deliberate new permission scope (e.g., `chat:audit`) and a separate read surface — not a relaxation of this story's filters.

5. **`searchConversations` returns only the caller's matches.** Both the `findMany` (finding matching rows) AND the per-conversation `count` query inside the `Promise.all` filter by `user_id = ctx.userId`.

6. **`getConversationHistory` raw SQL includes `user_id` filter.** The three SQL references — outer `WHERE`, inner `SELECT content … WHERE cm2.conversation_id = cm.conversation_id AND cm2.role = 'USER'` (first-message snippet), and the `GROUP BY` grouping — all constrain by `cm.user_id = ${ctx.userId}` on the outer query; the inner snippet subquery inherits the scope because it is keyed by `conversation_id`, and archived conversations are now themselves per-user (per AC 2). Add a `AND cm2.user_id = ${ctx.userId}` to the inner subquery defensively even though it is redundant given AC 2.

### Data-layer

## Dev Notes

### Background / Why now

Audit of `app/actions/ai-chat.ts` (2026-04-24) found that every read/write/delete/search/archive query filters on `workspace_id + context_type + context_id` only — `user_id` is written to every row but never used as a filter. Effective behaviour today:

- Any workspace member can read every other member's AI chat history across all four contexts (`GLOBAL`, `TASK`, `LAW`, `CHANGE`).
- `deleteChatMessage`, `clearChatHistory`, and `archiveConversation` operate on the entire workspace's history, so any member can destroy or re-scope a colleague's conversations.
- `searchConversations` surfaces matches from every member's transcripts.

For a compliance/legal product this is a privacy concern (exploratory questions — "are we violating X?", "what happens if we miss this deadline?" — should not be visible to colleagues) and a data-integrity foot-gun (one-click "clear history" wipes the whole workspace).

**Scope decision (2026-04-24):** fully per-user across all four contexts. A shared-knowledge surface ("pin to workspace" / "share this conversation") is deferred as a deliberate future feature, not a default.

### Filter placement convention

All eight filter additions (7 server actions + 1 cursor resolver — see Task 2) sit next to the existing `workspace_id` filter in the same `where` object — do NOT introduce a helper wrapper or refactor. The goal of this story is the minimum-surface-area privacy fix. The existing patterns already require every mutation to declare both scopes explicitly; this story aligns chat reads with that practice.

Example shape (illustrative):

```ts
// Before
where: {
  workspace_id: ctx.workspaceId,
  context_type: validated.contextType,
  context_id: validated.contextId ?? null,
  ...(validated.contextType === 'GLOBAL' ? { conversation_id: null } : {}),
}

// After
where: {
  workspace_id: ctx.workspaceId,
  user_id: ctx.userId,
  context_type: validated.contextType,
  context_id: validated.contextId ?? null,
  ...(validated.contextType === 'GLOBAL' ? { conversation_id: null } : {}),
}
```

### Raw SQL change in `getConversationHistory`

The `$queryRaw` block needs the `user_id` comparison added in three places for clarity (outer `WHERE`, the subquery, and a redundant belt-and-braces check). Prisma's tagged template interpolation handles the parameter safely. Example:

```sql
SELECT
  cm.conversation_id,
  (
    SELECT content FROM chat_messages cm2
    WHERE cm2.conversation_id = cm.conversation_id
      AND cm2.user_id = ${ctx.userId}
      AND cm2.role = 'USER'
    ORDER BY cm2.created_at ASC
    LIMIT 1
  ) as first_message,
  MIN(cm.created_at) as created_at,
  COUNT(*) as message_count
FROM chat_messages cm
WHERE cm.workspace_id = ${ctx.workspaceId}
  AND cm.user_id = ${ctx.userId}
  AND cm.context_type = 'GLOBAL'
  AND cm.conversation_id IS NOT NULL
GROUP BY cm.conversation_id
ORDER BY MIN(cm.created_at) DESC
LIMIT 20
```

### AUDITOR role clarification (forward-compatibility note)

The `AUDITOR` role (Story 21.14) exists for compliance-audit oversight, not AI chat review. This story does not add any cross-user chat visibility for AUDITOR. If a future product requirement surfaces "auditors must be able to review team AI chat transcripts" it MUST be implemented as a deliberate new read surface gated by a new permission scope (suggested: `chat:audit`) with an explicit UI indicator that the viewer is in cross-user mode — not by relaxing the `user_id` filter in these server actions.

### Index rationale

Current indexes are optimised for workspace-wide reads. After this story, every non-trivial read filters on `(workspace_id, user_id, …)`, so the two composite indexes become the correct leading edges. The standalone `@@index([user_id])` becomes redundant because the new composite indexes are a prefix match for user-only scans (Postgres can use an index on `(workspace_id, user_id, …)` for `WHERE user_id = …` only if it seeks via a skip-scan, which Postgres doesn't support — BUT the only place a naked `user_id` scan is needed is the user-deletion cascade, which is already governed by the `onDelete: Cascade` FK constraint and does not need a dedicated index). If query plans on the user-cascade path regress in practice, the `@@index([user_id])` can be reintroduced in a follow-up migration — not in scope here.

**Informational note (not a concern, no action required):** for pre-story-deploy workspaces that had shared chat history, the `archiveConversation` action from before this story deployed may have assigned a single `conversation_id` to rows from multiple users (because the prior `updateMany` filter was workspace-only). - Each user sees the conversation in their own history **only if they contributed at least one message to it**
- `message_count` in the conversation list reflects each user's own count (via the new `cm.user_id` filter in the outer WHERE)
- `first_message` snippet shows each user's own first USER-role message (via the new `cm2.user_id` filter in the subquery); if the user contributed zero USER-role messages but has some ASSISTANT-role messages, the snippet is empty (mapped to `''` via `c.first_message ?? ''`)

This is correct privacy behaviour — no cross-user leak — but may produce mildly surprising UX on pre-existing mixed-user conversations (e.g. a conversation appearing with an empty snippet). No fix required; the story's AC 3 explicitly accepts this ("existing pre-story rows remain visible only to their original author").
