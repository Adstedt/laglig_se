# Story 6.9: Enable Threaded Comments on Law List Item Modal

## Status

Done

## Story

**As a** compliance team member,
**I want** to add threaded comments on law list items,
**so that** I can discuss compliance status, interpretations, and action items directly in context with my colleagues.

## Context & Background

Threaded comments are already fully implemented for the **task modal** (Story 6.6), including:
- 3-level nested threading with reply/edit/delete
- @mentions with autocomplete
- Comment input with avatar + Cmd+Enter
- Activity feed integration

The Prisma `Comment` model already has a `law_list_item_id` foreign key, and the law list item modal already has an activity tab structure with a disabled `CommentsTab` placeholder (Story 6.3).

This story enables the same commenting capability on law list items by reusing the existing task modal comment components and creating parallel server actions scoped to `law_list_item_id`.

## Acceptance Criteria

1. Comments tab in law list item modal displays threaded comments (same UX as task modal)
2. Users can create root comments on a law list item (content: 1–5000 characters)
3. Users can reply to comments (max 3 levels: root → reply → reply-to-reply)
4. Users can edit their own comments ("Redigerad" indicator shown)
5. Users can delete their own comments (with confirmation, cascades to replies)
6. Comments appear in the "Alla" activity feed alongside existing activity types
7. Comments are scoped to workspace — users only see comments from their workspace
8. Empty state shows "Inga kommentarer ännu" with prompt to add one
9. @mentions: typing @ searches workspace members, mentioned users stored in `mentions` array

### Component Extraction Decision

The task modal comment components (`threaded-comments.tsx`, `comment-input.tsx`) are currently tightly coupled to task modal actions. Two approaches:

1. **Extract to shared** (recommended): Move to `components/features/comments/` with a generic `onSubmit`/`onEdit`/`onDelete` prop interface, used by both task modal and law list item modal.
2. **Duplicate with adaptation**: Copy and adapt for law list item context. Faster but creates maintenance debt.

### Depth Calculation Pattern

The existing `createComment()` in `task-modal.ts` calculates depth as follows:
- Root comment (no `parentId`): `depth = 0`
- Reply: `depth = parent.depth + 1`
- Reply allowed only when `depth < 2` (enforces 3-level max: 0, 1, 2)

Replicate this logic in the new law list item comment actions.

### @Mentions Extraction Pattern

The existing `createComment()` in `task-modal.ts` extracts mentions as follows:
- Regex-match `@username` patterns from comment content
- Store matched usernames in the `mentions` string array field on the `Comment` model
- The `CommentInput` component provides autocomplete UI — this comes for free when reusing/extracting the component

### Activity Logging Pattern

The existing task modal logs comment activity via `prisma.activityLog.create()`:
- `entity_type: 'comment'`
- `entity_id`: the new comment's ID
- `action: 'created'` (or `'updated'` / `'deleted'`)
- Related entity context: for law list item comments, set `old_value` / `new_value` JSON to include `{ law_list_item_id }` so the activity feed can filter by parent entity

This is how comments appear in the "Alla" activity tab (Task 3).

### Data Model Decision: Polymorphic FKs (Keep As-Is)

The `Comment` model currently uses two nullable FKs (`task_id`, `law_list_item_id`) to identify the parent entity. For this story, **keep this pattern** — it works correctly for the two current entity types and requires no migration.

A future story (pre-Epic 14.7 agent tools) should refactor to a generic `entity_type` + `entity_id` pattern when the compliance agent needs cross-entity comment search. That refactor is out of scope here — see backlog note below.

> **Backlog:** "Refactor Comment model to generic entity pattern" — migrate `task_id`/`law_list_item_id` to `entity_type` + `entity_id` columns. Prerequisite for agent tool comment search across entity types. Schedule before Story 14.7.
