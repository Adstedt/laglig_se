## Story

**As** the compliance agent operating in a LAW, CHANGE, or GLOBAL chat,
**I want** a reliable way to obtain the active `lawListItemId` ‚Äî and to look up law-list items and tasks by name when no item is in scope,
**so that** the law-item write tools work without the user pasting an internal id, and the entity-readers (19.4) have an entry node to traverse from.

## Acceptance Criteria

### ID threading (LAW + CHANGE ‚Üí the agent)

1. **Client ‚Üí route.** `useChatInterface` accepts an optional `lawListItemId` and includes it in the chat transport body alongside the existing `contextType`/`contextId` (same body path those use today). Wired at the two mounts where an item is in scope:
   - **LAW** (`legal-document-modal/ai-chat-panel.tsx`): pass `lawListItemId={listItemId}` (it already passes `contextId={listItemId}` at `:77` ‚Äî make the id explicit rather than relying on `contextId` overloading).
   - **CHANGE** (`chat-panel.tsx` already receives `lawListItemId` at `:68`, fed by `change-assessment-modal.tsx:106`): thread it into the `useChatInterface` body so the route receives it.
2. **Route resolves one `lawListItemId`** (`app/api/chat/route.ts`, near the body parse `:181-188`): prefer the explicit `body.lawListItemId`; **fallback** to `contextId` when `contextType === 'law'` and no explicit id (so the existing LAW mount keeps working with zero client change). Thread the resolved value into **both** `buildSystemPrompt(...)` (`:235-238`) and `createAgentTools(...)` context (`:274-280`).
3. **System-prompt LAW block** (`system-prompt.ts:261-265`): `SystemPromptOptions` gains `lawListItemId?: string` (`:23-35`); when present in a LAW context, the block appends the id so the agent can use it ‚Äî e.g. `Aktiv laglistpost-ID: <id>. Anv√§nd detta ID f√∂r add_obligation / update_compliance_status / add_context_note.` **Inject the line ONLY when a `lawListItemId` is resolved** ‚Äî `ai-chat-panel.tsx:19` `listItemId?` is `undefined` when the user is browsing a law that is *not* in the workspace's list, so the block must omit the line (never print `Aktiv laglistpost-ID: undefined`) and let the agent fall back to `search_law_list_items`. Additive; no other block changes. (SF-2)
4. **Tool-context default.** `AgentToolContext` (`tools/index.ts:59-66`) and `PendingActionToolContext` (`pending-action.ts:20-26`) gain `lawListItemId?: string`, populated by the route from the resolved value (AC 2).
5. **Harden the 3 shipped write tools** (`add_obligation`, `add_context_note`, `update_compliance_status`): make their `lawListItemId` input **optional**, and inside `execute` default it to `context.lawListItemId` when the arg is absent/empty. [Source: `add-obligation.ts:19-22,57-61` ‚Äî the others mirror this shape]

### Discovery tools (GLOBAL entry ‚Äî `app/actions/search.ts` is corpus-only)

6. **`search_law_list_items(query)`** ‚Äî new `'read'` tool in `lib/agent/tools/search-law-list-items.ts` (`createSearchLawListItemsTool(workspaceId)`), built like `search-workspace-files.ts`. Workspace-scoped Prisma query: `LawListItem` joined to its `document`, filtered by `law_list.workspace_id === workspaceId`, **case-insensitive `contains`** match on `document.title` OR `document.document_number` (`mode: 'insensitive'`). Returns up to `limit` (default 5, max 20) rows of `{ lawListItemId, title, sfsNumber, complianceStatus, listName }` (from `LawListItem.id`, `document.title`, `document.document_number`, `LawListItem.compliance_status`, `law_list.name`). **(SF-1) Include `listName`** because `@@unique([law_list_id, document_id])` (`schema:437`) allows the same SFS to appear once per list ‚Äî a multi-list workspace ("Huvudlista", "Lager Malm√∂", ‚Ä¶) can return the same law as several distinct items, and the list name is what lets the agent/user disambiguate *which* item to write to (rather than silently deduping and writing to the wrong list). **Guardrail in the description:** this tool resolves *ids*, not citable law text ‚Äî the agent must still use `search_laws` for `[K√§lla: ‚Ä¶]` citations. [Source: schema `LawListItem:360` (`document_id:363`, `compliance_status:382`, `law_list_id:362`), `LawList:318` (`workspace_id:320`), `LegalDocument` title/document_number]
7. **`search_tasks(query)`** ‚Äî new `'read'` tool in `lib/agent/tools/search-tasks.ts`. Workspace-scoped `Task` query: `where: { workspace_id, title: { contains: query, mode: 'insensitive' } }`, return `{ taskId, title, columnName }` (resolve `column.name` via the `TaskColumn` relation, `:1186`). `limit` default 5, max 20. [Source: schema `Task:1159` (`workspace_id:1161`, `title:1163`, `column_id:1162`)]
8. **Registry + governance + UI** for both new tools: add to the `ToolName` union + `TOOL_REGISTRY_POLICY` as **`'read'`** (compile-enforced) + register in `createAgentTools` (auto-wrapped by the 19.5 decision-log loop, kept for AUDITOR); add `TOOL_CONFIG` entries in `chat-message.tsx` (`search_law_list_items` ‚Üí "S√∂ker i laglistan"/"S√∂kte i laglistan"; `search_tasks` ‚Üí "S√∂ker bland uppgifter"/"S√∂kte bland uppgifter"); add a steering bullet in `system-prompt.md` (use these to find an item/task by name when none is in scope, then call the readers/write tools with the returned id). [Source: `tools/index.ts:74-109,134-164`; `chat-message.tsx:76-101`; `system-prompt.md:87-97`]
