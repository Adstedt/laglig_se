# Story 17.11b: Agent Tool — Add New Section to Existing Documents (Approval-Gated)

## Status

Done

> 🆕 **Brownfield addition (PO sized 2026-06-03, Sarah).** Surfaced by live smoke-testing 17.11 (PASS gate 2026-06-03 10:58). Filling a UX cliff: when the user asks the agent to *add* a section to an existing styrdokument, today's only options are (a) full doc rewrite via `draft_styrdokument` — disproportionate and destructive of incremental audit trail, or (b) tell the user to manually add the heading in the editor — defeats the agent. Quinn (QA) recommended scoping a follow-up; user signed off on **ADD-only** for this story (REMOVE / RENAME / REORDER tracked separately when actually needed). 17.11's gate stays PASS — 17.11b is **additive**, not a re-scope.

## Story

**As an** AI compliance agent,
**I want** to propose adding a NEW section to an existing workspace document (subject to user approval),
**so that** I can help users grow a policy/procedure incrementally — adding what's missing — without re-drafting the whole document and without silently mutating an approved document.

## Acceptance Criteria

### Schema

1. `PendingAgentActionType` extended with **`ADD_DOCUMENT_SECTION`**, appended after `UPDATE_DOCUMENT` (the most recent addition, Story 17.11 — **`schema.prisma:1625`**). Pattern is **additive append** (`ALTER TYPE ... ADD VALUE`; generate `--create-only` and hand the SQL to the user — never auto-apply, per the project's manual-migration workflow). No new tables. Follows the same migration template as 17.11's `20260603100000_add_update_document_pending_type/migration.sql`. Pick a migration directory timestamp **after `20260603100000`** (e.g. `20260604100000_*` or current date — Prisma orders by timestamp prefix).
2. `params` JSON shape for `ADD_DOCUMENT_SECTION`:
   ```ts
   {
     documentId: string
     documentTitle: string                // CP-001: renderer copy (carried by tool, mirrors 17.11 UX-001 fix)
     newSectionHeading: string             // the heading TEXT for the new section (e.g. "Syfte")
     newSectionLevel: 1|2|3|4|5|6          // heading level (h1..h6)
     newSectionContentJson: TiptapNode[]   // Tiptap body nodes for the new section
     position: InsertPosition              // see below
     changeSummary: string                 // for the version history
     entity_version: string                // ISO-8601 UTC, document.updated_at — same as 17.11
   }

   type InsertPosition =
     | { at: 'start' }
     | { at: 'end' }
     | { at: 'after'; heading: string }    // case-insensitive
     | { at: 'before'; heading: string }   // case-insensitive
   ```

### Tool — `add_document_section`

3. New tool `lib/agent/tools/add-document-section.ts` (`add_document_section`). Zod schema:
   ```ts
   {
     document_id: string (uuid)
     new_section_heading: string (min 1, max 200)
     new_section_level: number (int, 1..6)
     new_section_content: any        // accepts array | doc-node | single-node | JSON string — same shape-tolerance as 17.11's update_document
     change_summary: string (min 1, max 500)
     position: discriminated union with `at: 'start'|'end'|'after'|'before'` + optional `heading: string` (required for after/before)
   }
   ```
   **No `execute` param** — propose-only, mirrors 14.23/17.11 pattern. Registered with policy `'write'` in `lib/agent/tools/index.ts` (slot alphabetically — `add_document_section` lands before `add_obligation`).
4. **Guards (at tool time):**
   - Document is in the caller's workspace.
   - Status is **DRAFT or IN_REVIEW** (reject APPROVED/SUPERSEDED/ARCHIVED via `wrapToolError` — guide to `createDraftFromApproved`, mirrors 17.11 AC 4).
   - Document has a current version (`current_version.content_json.content` is an array).
   - `new_section_heading` does **NOT** already exist in the document (case-insensitive) — if it does, surface `"Avsnittet \"{heading}\" finns redan i dokumentet. Använd update_document för att redigera ett befintligt avsnitt."` (no pending row). Reuses `hasSection` from 17.11's helper.
   - `new_section_content` normalizes to a non-empty `TiptapNode[]` (no inserting empty sections — surface `"Det nya avsnittet är tomt. Ange minst en stycke text."`).
   - For `position.at === 'after'` or `'before'`: `position.heading` exists in the document (case-insensitive). If not → `"Rubriken \"{position.heading}\" finns inte i dokumentet — kan inte positionera det nya avsnittet."` (no pending row).
   - On any failure → `ToolError`, no pending row.
5. On success: create a `PendingAgentAction(ADD_DOCUMENT_SECTION)` via `createPendingActionRow` and return a `WriteToolResponse` whose preview shows the new heading + position context + a short content excerpt (e.g. `Nytt avsnitt: "Syfte" (h2, sist i dokumentet) — Syftet med policyn är att…`). `wrapWriteToolResponse` inherits `_meta.sidebarHint: 'none'` (post-197B-S1).

### Renderer

6. New renderer `components/features/ai-chat/agent-action-renderers/add-document-section-renderer.tsx`, **built on `ActionRendererFrame`** (no hand-rolled chrome). **Primary template: 17.11's `update-document-renderer.tsx`**. The diff card degenerates cleanly for an insert:
   - **PENDING:** document title + new heading badge (with level chip, e.g. `h2`) + position context line (`"Läggs till sist i dokumentet"` / `"Läggs till efter \"Inledning\""` / etc.) + change summary + a content preview (excerpt, line-clamp-3). "Visa mer" opens the chat detail panel by **reusing the existing `'document-update'` `ChatDetailItem` variant** with `oldSectionContentJson: []` — the existing `document-update-detail.tsx` renders the `(tom)` placeholder for the empty "Nuvarande" side automatically (verified at `document-update-detail.tsx:18-21`). **No new ChatDetailItem variant, no new detail component, no `chat-detail-sidebar.tsx` switch change required.** Controls: "Godkänn" / "Avvisa".
   - **APPROVED:** title + link to the document at its new version (with `v{N}` chip — same as 17.11).
   - **REJECTED / EXPIRED:** standard muted variants from the frame.
   - **CP-001 (formatting guardrail):** renderer copy NEVER exposes internal identifiers — no raw `documentId`, no `pendingActionId`, no `new_section_heading` rendered as a code-style/quoted identifier. Use the document **title** and heading **text** as natural Swedish.
   - **Sidebar header note:** the existing `chat-detail-sidebar.tsx` case for `'document-update'` renders subtitle as `Ändring: {sectionHeading}`. For an ADD this reads slightly less precise than ideal but is not wrong (the new heading IS the subject of the change). Acceptable per the brownfield-additive principle; if a future story wants tighter copy it can extend `DocumentUpdateDetailData` with an optional `kind?: 'update' | 'add'` discriminator. **Out of scope for 17.11b — keep the sidebar untouched.**
7. Register the renderer in **both** exported maps in `components/features/ai-chat/agent-action-card.tsx` — `RENDERERS[ADD_DOCUMENT_SECTION]` and `ACTION_TYPE_LABELS[ADD_DOCUMENT_SECTION]` (suggested label: `'Lägg till avsnitt'` — mirrors 17.11's `'Uppdatera dokument'` and 14.30's `'Ändra dokumentstatus'` pattern).

### Approval dispatch

8. `approvePendingAction` (`app/actions/pending-agent-actions.ts`) gains an `ADD_DOCUMENT_SECTION` branch. Flow mirrors 17.11's `UPDATE_DOCUMENT` branch:
   - Re-read the live document (with `current_version.content_json`).
   - **Re-assert the DRAFT/IN_REVIEW status guard** (state may be stale).
   - **Re-assert that `newSectionHeading` doesn't exist** (heading may have been added manually since proposal — refuses with `"Avsnittet \"{heading}\" finns redan i dokumentet."` to avoid duplicate headings).
   - For `position.at ∈ {'after','before'}`: re-assert the target heading exists.
   - Call `addSection()` (AC 9) to produce the **full** updated document.
   - Generate `contentHtml` via the existing `lib/documents/tiptap-to-html.ts` walker.
   - Call existing `saveDocumentVersion(documentId, fullContentJson, changeSummary, undefined, generatedHtml)` — same as 17.11. Auto-reindex via `after()` is inherited.
   - **AC 10 stamp:** patch the auto-created `document_version_saved` activity log row's `new_value` with `{ by: 'agent', pendingActionId, operation: 'add_section' }` — mirrors 17.11 AC 10 (`findFirst` by `entity_id` + JSON-path `version_number`, then `update`). Fire-and-forget try/catch.
   - On any guard/dispatch failure → row stays PENDING with a clear Swedish error.
   - `revalidatePaths`: `['/workspace/styrdokument', '/workspace/styrdokument/{documentId}/edit']` (same as 17.11).
9. Pure `addSection(contentJson, headingText, level, contentNodes, position)` utility in `lib/documents/update-document-section.ts` (extending the existing file — keeps section-mutation logic in one place). Returns the full updated doc (does not mutate input). Position resolution:
   - `{ at: 'start' }` → insert at index 0
   - `{ at: 'end' }` → append
   - `{ at: 'after', heading }` → insert after the located section (= after the located heading AND its body up to next same-or-higher heading; **directly calls the existing `findSectionBounds(...)` private helper in the same file** — `findSectionBounds` is module-private at `update-document-section.ts:61` and stays that way; `addSection` co-locates so it can use it without an export)
   - `{ at: 'before', heading }` → insert at the start index of the located section
   - The new heading node is constructed as `{ type: 'heading', attrs: { level }, content: [{ type: 'text', text: headingText }] }`, followed by the body nodes.
   - Throws **new `SectionAlreadyExistsError`** (introduced by 17.11b alongside the existing `SectionNotFoundError`) if `headingText` already exists (case-insensitive). Reuses the existing `SectionNotFoundError` when `position.heading` is required and missing. Both errors live in `update-document-section.ts` and follow the same shape as `SectionNotFoundError`.

### Activity log + audit

10. `ActivityLog` on add uses `entity_type: 'workspace_document'` (mirrors 17.11 AC 10). `saveDocumentVersion` already writes `document_version_saved` with `version_number` + `change_summary`. Dispatch enriches `new_value` with `{ ..., by: 'agent', pendingActionId, operation: 'add_section' }` — the `operation` key distinguishes ADD from UPDATE for downstream audit consumers and supersedes the "future hardening" REL-001 trade-off path noted in 17.11's gate.

### Security / non-regression

11. Workspace scoping and the DRAFT/IN_REVIEW guard are enforced **both** at tool time (AC 4) **and** at dispatch (AC 8) — never trust the proposal-time snapshot. A role lacking `tasks:edit` cannot approve (inherited gate).
12. 17.11's `update_document`, 14.24's `draft_styrdokument`, and all other existing agent tools/renderers are **behaviourally unchanged** (additive only). Full unit suite must remain green post-change.

### Prompt

13. `lib/agent/system-prompt.md` updated (restart dev server to load): when to **add** a new section (`add_document_section`) vs **edit** an existing one (`update_document`) vs **draft a new document** (`draft_styrdokument`). Three-way discriminator. Concrete examples:

    - *"Uppdatera Syfte-avsnittet"* → `update_document`

Follow the 14.28/14.29/14.30/17.11 precedent — chip labels must include both PENDING + DONE labels per the 1428-001 owner-smoke finding.
