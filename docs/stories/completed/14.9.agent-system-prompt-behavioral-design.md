# Story 14.9: Agent System Prompt & Behavioral Design

## Status

## Story

**As a** platform operator,
**I want** the compliance partner to behave consistently, transparently, and safely,
**so that** users trust its guidance and it doesn't take inappropriate actions.

## Context & Dependencies

The system prompt is the agent's "personality and rules." Following Anthropic's [Prompting Best Practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices) and [Effective Context Engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents) guidance, the agent must: cite sources, never fabricate legal text, distinguish fact from advice, show its reasoning, and pause before state-changing actions. The prompt is in Swedish.

This story replaces the placeholder `buildSystemPrompt()` function in `app/api/chat/route.ts` (lines 161-186) with a comprehensive, tested prompt builder in a dedicated module. The chat route will be updated to import and use the new builder.

**Key existing infrastructure:**

- **Chat route** (Story 14.7a): `app/api/chat/route.ts` has a minimal `buildSystemPrompt()` placeholder at lines 161-186 that returns a basic Swedish instruction string. Accepts `contextType`, `contextId`, `title`, `sfsNumber`. This story replaces it with the comprehensive prompt from `lib/agent/system-prompt.ts`.
- **8 agent tools** (Stories 14.7a-c): `createAgentTools(workspaceId)` in `lib/agent/tools/index.ts` returns: `search_laws`, `get_document_details`, `get_change_details` (14.7a), `get_company_context` (14.7b), `create_task`, `update_compliance_status`, `save_assessment`, `add_context_note` (14.7c). Write tools use `execute` flag confirmation pattern. The system prompt must reference all 8 tools by name and explain when/how to use each.
- **RAG pipeline** (Story 14.8): `retrieveContext()` in `lib/agent/retrieval.ts` handles two-stage vector+rerank retrieval. `assembleAgentContext()` in `lib/agent/context-assembly.ts` formats chunks with Swedish `--- Källa: ... ---` headers. `detectLegalReferences()` extracts SFS/section references. The system prompt instructs the agent on retrieval strategy (when to search, how to reformulate).
- **Company context tool** (Story 14.7b): `get_company_context` returns `profileComplete: false` when profile fields are NULL (pre-14.4 settings UI, based on 80% completeness threshold). The system prompt should instruct the agent to call this tool at conversation start and nudge users to complete their profile when incomplete.
- **CompanyProfile model**: Already exists in `prisma/schema.prisma` (lines 160-184). Fields: `company_name`, `org_number`, `sni_code`, `industry_label`, `employee_count_range`, `certifications`, `compliance_maturity`, `activity_flags`, `profile_completeness`, etc. Many fields NULL until 14.4/14.5 provide UI.

## Acceptance Criteria

### System Prompt

1. System prompt builder `buildSystemPrompt(options?)` exported from `lib/agent/system-prompt.ts`. Options include: `companyContext` (formatted string from CompanyProfile), `contextType` ('global' | 'task' | 'law'), `contextId`, `title`, `sfsNumber`
2. Prompt written in Swedish — agent communicates in Swedish by default
3. Role definition: "Du är en compliance-partner som hjälper användare att förstå och hantera lagändringar"
4. Knowledge boundary: "Du baserar dina svar på de lagar och dokument som finns i systemet. Om du inte hittar relevant information, säg det tydligt."
5. Citation requirement: always reference specific document numbers, chapters, and sections when quoting law text — format: `[Källa: SFS 1977:1160, Kap 2, 3 §]`
6. Distinction between fact and advice: use "Lagen säger..." for factual statements, "Ni bör överväga..." for advisory recommendations
7. Company context section injected when profile data is available — formatted from CompanyProfile fields
8. Tool usage instructions for all 8 registered tools — explains when to use each, read-only vs write behavior

### Behavioral Guardrails

9. Never fabricate legal text — only quote from retrieved chunks
10. Never invent requirements that aren't in the law
11. When uncertain, explicitly say: "Jag är inte säker om detta — jag rekommenderar att ni konsulterar en jurist"
12. Legal disclaimer: "Jag ger vägledning baserat på lagtext, men detta ersätter inte juridisk rådgivning"

### Transparency Protocol

13. Function `formatTransparencyBlock(searchResults)` exported from `lib/agent/transparency.ts` — generates a "Jag sökte i..." block showing which documents were searched and how many relevant sections were found
14. Function `formatSourceCitation(source)` exported from `lib/agent/transparency.ts` — formats inline source references in `[Källa: SFS 1977:1160, Kap 2, 3 §]` format. Supports LegalDocument, UserFile, and ChangeEvent source types.

### Human-in-the-Loop

15. Write tool instructions in prompt: "Innan du utför en åtgärd (skapar uppgift, ändrar status), beskriv vad du planerar göra och invänta användarens godkännande. Anropa verktyget med execute: false först."
16. Read-only tool instructions: "Du kan fritt söka i lagar och dokument utan att fråga först"

### Fallback Behavior

17. When company profile is incomplete: agent instructed to say "Jag ser att er företagsprofil saknar [fält]. Vill ni berätta mer om er verksamhet så att jag kan ge mer relevanta råd?"
18. When no relevant chunks found: agent instructed to say "Jag hittade ingen relevant information i vårt register om detta. Kan ni formulera frågan annorlunda, eller vill ni att jag söker bredare?"

### Chat Route Integration

19. Replace the placeholder `buildSystemPrompt()` in `app/api/chat/route.ts` with import from `lib/agent/system-prompt.ts`. Wire company context by querying CompanyProfile via Prisma using `workspaceId` from session (same pattern as `get_company_context` tool — direct Prisma, not server actions).
20. Function `formatCompanyContext(profile)` exported from `lib/agent/system-prompt.ts` — formats a CompanyProfile record into the Swedish context string for prompt injection

### Testing

21. At least 8 unit tests covering: prompt construction with/without company context, prompt construction with contextType variants, transparency block formatting, source citation formatting for different source types, company context formatting

### Anthropic Prompt Engineering Guidance (Key Principles for This Story)

The following principles from Anthropic's official documentation directly inform how the system prompt should be written. The dev agent should read and apply these:

**From [Prompting Best Practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices):**

1. **Use XML tags to structure the prompt** — Wrap each section in descriptive tags (`<role>`, `<guardrails>`, `<tool_guidance>`, etc.). "XML tags help Claude parse complex prompts unambiguously, especially when your prompt mixes instructions, context, examples, and variable inputs."
2. **Give context/motivation behind instructions** — Don't just say "never fabricate"; explain *why*: "Felaktiga juridiska råd kan ha allvarliga konsekvenser." Claude generalizes better when it understands the reasoning.
3. **Claude 4.6 tool triggering calibration** — "Claude Opus 4.5 and Claude Opus 4.6 are more responsive to the system prompt than previous models. If your prompts were designed to reduce undertriggering on tools, these models may now overtrigger. The fix is to dial back aggressive language." Use "Use this tool when..." not "You MUST always use this tool...".
4. **Tell Claude what to do instead of what not to do** — Instead of "Don't give legal advice", say "Ge vägledning baserat på lagtext och hänvisa alltid till källa."
5. **Be specific about the desired output** — The prompt should describe the expected communication style (prose, not bullet points; formal Swedish "ni"; cite sources inline).

**From [Effective Context Engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents):**

1. **Context is a finite resource** — Every token in the system prompt depletes the model's "attention budget." Curate the smallest possible set of high-signal tokens. Don't repeat information already in tool descriptions (the model reads those automatically).
2. **Find the "right altitude"** — Between overly brittle hardcoded logic ("always search before answering every question") and vague guidance ("be helpful"). The sweet spot: "Sök i lagdatabasen när användaren frågar om regler, skyldigheter eller krav" — specific enough to guide, flexible enough for the model's heuristics.
3. **Dynamic context retrieval over static loading** — Instruct the agent to *call tools to discover context* (e.g., `get_company_context`) rather than pre-loading everything into the system prompt. This mirrors how humans work — they look things up as needed.
4. **Tools should be self-contained** — Tool descriptions already define parameters, types, and behavior. The system prompt should focus on *strategy* (when to use which tool, how to combine them) not repeat schema details.
5. **Few-shot examples as canonical demonstrations** — If space permits, one brief example interaction showing expected behavior (search → cite → distinguish fact/advice) can be worth more than paragraphs of rules.

### Prompt Token Budget

Target ~800-1200 tokens for the base prompt (without company context). Company context adds ~100-200 tokens when present. Total should stay well under 2000 tokens to leave room for tool results and conversation in the model's context window (128K for GPT-4 Turbo, 200K for Claude).

**Context engineering rule of thumb:** Every added token has diminishing marginal returns. If a piece of information is already in a tool description, don't repeat it in the system prompt. The prompt's job is *behavioral strategy* — the tools handle their own documentation.

### Prompt Anti-Patterns to Avoid

Per Anthropic docs, these patterns cause problems with Claude 4.6:

- **Over-prompting tool use** — "You MUST ALWAYS call search_laws before answering ANY question" → causes overtriggering. Instead: "Sök i lagdatabasen när du behöver verifiera eller citera lagtext."
- **Repeating tool descriptions** — The model reads tool descriptions from the SDK's tool definitions. Don't copy-paste parameter lists into the prompt. Focus on *when* and *why* to use tools.
- **Excessive ALL-CAPS emphasis** — Claude 4.6 is more responsive to system prompts. Normal language works. "CRITICAL: NEVER" can be replaced with clear, motivated instructions.
- **Negative instructions without alternatives** — "Don't give legal advice" is less effective than "Ge vägledning baserat på lagtext och hänvisa alltid till juridisk rådgivning för formella beslut."
- **Static context that should be dynamic** — Don't bake in lists of tracked laws or compliance statuses. The agent should call `get_company_context` to get current data.
