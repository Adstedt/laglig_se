# Story 22.10: Unified "+ Skapa" Creation Menu

## Status

Done

## Story

**As a** workspace user,
**I want** the global "+ Skapa" button in the top bar to open a menu of all top-level creation surfaces (task, kontroll, laglista, dokument) instead of only opening the task dialog,
**so that** I can create any primary entity from one consistent affordance regardless of which page I am on, without hunting for per-page create buttons.

## Story Context

**Existing System Integration:**

- **Modifies:** `components/layout/header.tsx` (the single host of the "+ Skapa" button — Story 6.7)
- **Wires to (no rebuild):** `components/features/tasks/create-task-modal.tsx`, `components/features/document-list/manage-list-modal.tsx`, `components/features/documents/create-document-dialog.tsx`, route `/laglistor/kontroller/skapa`
- **Technology:** React, shadcn/ui (`DropdownMenu`), `lucide-react` icons, NextAuth session for role
- **Permission gating:** `lib/auth/permissions.ts` — `hasPermission(role, permission)` already in use across the app
- **Touch points:** `components/layout/header.tsx` only — page-side dialog mounts in Laglistor / Styrdokument pages remain untouched

**Motivation:**

Today the global "+ Skapa" button (`components/layout/header.tsx:65-72`) is hard-wired to open `CreateTaskModal`. The button label "+ Skapa" semantically promises generic creation, but the only thing it creates is a task. Users who want to create a kontroll, a laglista, or a dokument must navigate to that surface and find a per-page create button — the inverse of the affordance the top-bar button suggests.

A discovery conversation considered three patterns:
1. Always-menu (current proposal)
2. Context-aware default + caret split-button
3. Command palette / Cmd+K

Option 1 was chosen because the app has neutral routes (`/hem`, `/settings`, `/profil`, admin) where no entity is the obvious default, and a per-route default-mapping adds maintenance burden for marginal speed gain. The added click on the common task-create path (currently the only path) is offset by uniform behaviour everywhere else and is the pattern Linear/GitHub use on neutral surfaces.

The existing `Ctrl+Shift+T` global shortcut for quick task creation (registered in `header.tsx:43-45` via `useGlobalKeyboardShortcuts`) remains the fast path for power users and is preserved unchanged.

## Acceptance Criteria

**Menu structure & behaviour:**

1. The "+ Skapa" button in `Header` opens a `DropdownMenu` (from `components/ui/dropdown-menu.tsx`) instead of directly opening `CreateTaskModal`.
2. The menu lists the following items in this order, each with a Lucide icon and Swedish label:
   - **Uppgift** (icon: `CheckSquare`) — opens `CreateTaskModal`
   - **Kontroll** (icon: `ClipboardCheck`) — navigates to `/laglistor/kontroller/skapa`
   - **Laglista** (icon: `ListChecks`) — opens `ManageListModal` with `mode='create'`
   - **Dokument** (icon: `FileText`) — opens `CreateDocumentDialog`
3. Each menu item, when selected, triggers the corresponding dialog/route within ~50ms — the menu closes before or simultaneously with the dialog opening (no overlap glitch).
4. Behaviour is identical across every workspace route — no per-page default action, no per-page menu reordering.
5. The existing `Ctrl+Shift+T` keyboard shortcut still opens `CreateTaskModal` directly (does NOT open the menu first).

**Permission gating:**

6. Each menu item is hidden (not disabled) for roles lacking the corresponding permission, using `hasPermission(role, permission)` from `lib/auth/permissions.ts`:
   - Uppgift → `'tasks:edit'`
   - Kontroll → `'tasks:edit'`
   - Laglista → `'lists:create'`
   - Dokument → `'documents:add'`
7. If the user has zero create permissions (e.g., the `AUDITOR` role), the "+ Skapa" button itself is hidden (not disabled). No empty menu shall ever render.
8. Permission checks read the role from the existing session/user prop already passed into `Header` (no new server action calls).

**Accessibility & keyboard:**

9. The menu is fully keyboard accessible via Radix `DropdownMenu` defaults: `Enter`/`Space` to open, `↑`/`↓` to navigate, `Enter` to activate, `Esc` to close, focus returns to the trigger button on close.
10. The trigger button has `aria-haspopup="menu"` and `aria-expanded` reflecting open state (Radix provides this automatically when `DropdownMenuTrigger` wraps the button).
11. Each menu item exposes its accessible name in Swedish (the visible label is sufficient).

**Mobile:**

12. On viewports `<md` (Tailwind's `md` breakpoint), the dropdown still renders as a Radix `DropdownMenu` — no separate `Sheet` fallback. Radix `DropdownMenu` is touch-compatible and anchors correctly in narrow viewports. Confirm by manual smoke at 375px width.

**Integration constraints:**

13. The four target dialogs/routes are wired to the existing implementations — no new authoring UIs, no copy of any existing dialog, no schema or server-action changes.
14. Page-side mounts of `ManageListModal` (in `document-list-page-content.tsx`) and `CreateDocumentDialog` (in `document-browser-page.tsx`) are NOT removed — they remain in place for in-page create buttons (e.g., "Nytt dokument" on the Styrdokument toolbar).
15. The global menu's **Laglista** item offers the same template-chooser experience as the page-side `/laglistor` create flow when published templates exist; falls back to the blank form when no published templates are available. Templates are fetched once at workspace-layout level via the existing `getPublishedTemplates()` helper (`lib/db/queries/template-catalog.ts:36`) and threaded into `Header` — no new server query is introduced.
16. Two simultaneous mounts of the same dialog (one in `Header`, one in the page) is acceptable; only one is open at a time because each is controlled by independent `useState`. This trade-off (vs. lifting state to a global store) is intentional for minimum-risk brownfield wiring.
17. `CreateTaskModal`'s `onTaskCreated` callback in `Header` continues to call `router.refresh()` (existing behaviour from Story 6.7) — preserve verbatim.

## Out of Scope

The following are deliberately deferred. Each entry includes the trigger condition that would justify revisiting:

- **Command-palette / Cmd+K search interface** (option 3 from discovery). Revisit when the menu grows past ~6 items or when usage telemetry shows users hunting through the dropdown for a specific item.
- **Per-route default behaviour / split-button with caret** (option 2 from discovery). Revisit only if the always-menu pattern produces measured friction on the task-create path that outweighs the cross-route uniformity benefit.
- **New creation surfaces.** Only the four entities above are wired. Excluded from v1:
  - **Filer** — file upload is contextual (always scoped to a folder, task, or list-item), not a top-level standalone surface. The existing `FileDropzone` is embedded in those parent surfaces.
  - **Anteckningar** — no such entity exists in the data model.
  - **Kommentar** — comments are always entity-scoped (task, kravpunkt, finding); never top-level.
- **Lifting dialog state to a global store / context provider.** Two-mount-points strategy is preferred for minimum-risk brownfield wiring (see Dev Notes architectural decision). Revisit if cross-page dialog coordination is needed (e.g., AI chat opens a create-laglista dialog on the user's behalf).
- **Telemetry on which menu items are clicked.** Deliberately deferred — not "by omission". The deferral is acceptable because (a) the initial item ordering is informed by qualitative judgement that's reasonable to validate later with real data, and (b) reordering via subsequent story is cheap. **Revisit trigger:** when product wants to defend or change menu ordering with data, add an event emit at the four `onSelect` handlers (`menu_create_clicked` with `entity` dimension) routed through the same instrumentation surface that hosts `ChatUsageEvent` (memory: `lib/usage/cost-estimator.ts` neighbourhood).

### Architectural decision: two-mount-points strategy

Three options were considered for how the global menu invokes dialogs that already exist on a page:

1. **Two mount points** (this story): `Header` mounts its own instance of each dialog; the page also mounts its own. Each pair is controlled by independent `useState`. Simple, zero changes to existing pages, two cheap idle dialogs in DOM tree per workspace surface.
2. **Lift state to a global store**: One Zustand store (e.g., `useCreateMenuStore`) owns `taskModalOpen`, `listModalOpen`, etc. Both `Header` and the page subscribe and either can open. Cleaner semantically, but requires touching every page-side mount and decoupling internal dialog state from local component state.
3. **Lift dialogs to layout level**: Move all dialog mounts into `app/(workspace)/layout.tsx` and remove from pages. Pages dispatch open events. Most invasive.

**Choice: option 1.** The cost is duplicated dialog mounts when both `Header` and a relevant page are rendered (e.g., `/styrdokument` has two `CreateDocumentDialog` instances in the tree at any moment). Idle dialogs render essentially nothing (Radix Dialog only mounts portal content when `open=true`), so memory cost is negligible. This isolates the story to `Header` and avoids regression risk in three otherwise-unrelated pages. If a future need (e.g., open create-laglista from the AI chat) emerges, option 2 is the natural follow-up — but YAGNI for now.

### Architectural decision: hide vs. disable for permissions

Discussed and chosen: **hide**. Rationale:

- AUDITOR is a read-mostly role; surfacing creation entry points they can't use is noise.
- The empty-menu edge case (AUDITOR opens menu → sees nothing) is avoided by hiding the trigger entirely when zero permissions remain.
- Disabled+tooltip is more discoverable for users transitioning between roles, but role transitions are rare and the affordance reappears immediately on next session refresh.
- Server-side permission checks remain authoritative — the wizard route at `/laglistor/kontroller/skapa` and the relevant server actions enforce permissions independently. UI hide is guidance, not security.

If future product feedback shows confused users (especially MEMBER → ADMIN transitions), revisit and switch to disable+tooltip with copy "Kontakta din administratör för åtkomst" or similar.

### Why no `Sheet` fallback on mobile

Radix `DropdownMenu` portals to `document.body` and uses Floating-UI for positioning — it works correctly at 375px width and is touch-friendly. The existing app pattern for mobile branching is the `md:hidden` Tailwind utility (`header.tsx:54` uses this for the mobile menu toggle), and `mobile-sidebar.tsx` uses `Sheet` because the *sidebar itself* is a different component on mobile (slide-out drawer vs. always-visible left rail). The "+ Skapa" menu has no such structural difference — it's a popover from a button on both viewports. Adding a Sheet fallback would create two code paths for identical content.

**Existing mobile-visible `DropdownMenu` precedent:** `components/features/compliance-audit/cycle-detail/cycle-actions-dropdown.tsx` already uses `<DropdownMenu>` with `<DropdownMenuContent align="end">` on cycle-detail surfaces that render at mobile widths without a Sheet fallback. The same configuration is reused here — see lines 92-99 of that file for the canonical setup.

### Rationale for menu-item ordering

Order: **Uppgift → Kontroll → Laglista → Dokument**.

- Uppgift first because it's the highest-frequency creation (preserves muscle memory for users who currently click "+ Skapa" expecting a task).
- Kontroll second because it's the differentiator workflow for the compliance product.
- Laglista third (less frequent — most workspaces create law lists infrequently after onboarding).
- Dokument fourth (also infrequent for most users).

If usage telemetry later shows a different distribution, reorder. Out of scope for this story (AC 21).
