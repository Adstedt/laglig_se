# Story 7.5: Upload kollektivavtal (Settings + HR area) + ingest into RAG

## Status

Done

## Story

**As an** HR manager,
**I want** to upload our kollektivavtal from either Settings or the HR/employee area,
**so that** I can add it wherever I happen to be and the AI can reason over it.

## Acceptance Criteria

1. **Two entry points, one shared component:** the upload/manage UI is reachable from a **Kollektivavtal** surface in Settings (own tab, gated like the company-profile tab) **and** from the HR area (replace the Personalkort's disabled "Ladda upp kollektivavtal (kommer snart)" affordance from 7.3 with the real flow). Both mount the **same** component and server action — no duplicated logic. Gated `employees:manage`.
2. Upload PDF (reuse the `WorkspaceFile` upload path — category `AVTAL`, max 25 MB, `application/pdf`) with fields: **Namn** (e.g. "Byggnads Kollektivavtal 2024"), **Typ** (Arbetare → `ARB` / Tjänstemän → `TJM` / Övrigt → null), **giltighetsperiod** (optional from/to — requires two new nullable columns, see migration).
3. `SourceType` enum extended with `COLLECTIVE_AGREEMENT`; the existing extract→chunk→embed pipeline writes `ContentChunk` rows with `source_type = COLLECTIVE_AGREEMENT`, `source_id = <CollectiveAgreement.id>`, and `workspace_id` set (workspace-isolated).
4. Ingestion status surfaced to the user (**Väntar → Bearbetas → Klart / Misslyckades**) driven by `CollectiveAgreementStatus` (`PENDING → PROCESSING → READY / FAILED`), updated by the pipeline.
5. Chunks carry a `contextual_header` identifying the **agreement + section** (not just the filename) so citations are meaningful.
6. Migration authored (**NOT applied** — handed to the user): `ALTER TYPE "SourceType" ADD VALUE 'COLLECTIVE_AGREEMENT'` + additive nullable `effective_from`/`effective_to` date columns on `collective_agreements`.

### Verified pipeline facts (the seam)

### Code Quality Assessment

Deep review (security-critical isolation + migration + cron seam). This is a disciplined implementation of the epic's riskiest story. The three calls I scrutinized hardest all hold up:

1. **Workspace isolation (the epic's PRIMARY RISK) — structurally closed.** Every `COLLECTIVE_AGREEMENT` chunk write funnels through `syncWorkspaceChunks` → `chunkCollectiveAgreement`, and BOTH layers hard-throw on a falsy `workspace_id` before any DB write. `WorkspaceChunkInput.workspace_id` is a required (non-nullable) string, so the `createMany` insert cannot omit it. The only call site (`indexExtractedFile`) sources `workspace_id` from `WorkspaceFile.workspace_id` — a non-nullable column. A codebase-wide grep confirms no other writer of `source_type COLLECTIVE_AGREEMENT` exists. Defense in depth is real, not decorative: the agreement lookup pairs `workspace_file_id` WITH `workspace_id`, and status writes are `updateMany` with the workspace filter inside the WHERE. **Verdict: an agreement chunk can neither land with null `workspace_id` nor be attributed to a foreign workspace through any path in this diff.** (And until 7.7, retrieval doesn't span CA at all — read-side exposure is zero today; verified `lib/agent/retrieval.ts`, `context-assembly.ts`, and all `lib/agent/tools/*` are CA-free.)
