# Story 21.8: Auto-spawn corrective-action Task on AVVIKELSE

## Status

Done

## Story

**As a** lead auditor creating an avvikelse in a Lagefterlevnadskontroll cycle,
**I want** the system to automatically create a linked corrective-action `Task` in the workspace task system and keep the finding ↔ task lifecycle in sync (spawn on create, complete task on finding-close, notify lead auditor on task-close),
**so that** every deviation is tracked in the existing Kanban / assignment / reminder infrastructure instead of as a disconnected free-text note, and the audit cycle's corrective-action loop closes naturally without manual bookkeeping.

## Acceptance Criteria

### Functional

1. **Spawner helper `lib/compliance-audit/task-spawner.ts` (NEW).** Exports a single async function:
   ```ts
   export async function spawnCorrectiveActionTask(
     tx: Prisma.TransactionClient,
     args: {
       workspaceId: string
       cycleId: string
       cycleName: string
       leadAuditorUserId: string
       itemResponsibleUserId: string | null
       finding: {
         id: string
         title: string
         description: string
         dueDate: Date | null
       }
       createdByUserId: string
     }
   ): Promise<{ taskId: string; columnId: string }>
   ```
   The helper runs entirely on the provided `TransactionClient` (participates in the caller's tx — mirrors the `assertCycleEditable(tx, cycleId)` signature pattern called out in architecture §5.1). It:
   1. Resolves the target column: the workspace's lowest-`position` `TaskColumn` that is NOT `is_done` (the "Att göra" default, position 0 — see `app/actions/tasks.ts:580-586` for the default-column seeding). If no such column exists, create one via `tx.taskColumn.create` with the existing default shape (`name: 'Att göra', color: '#6b7280', position: 0, is_default: true, is_done: false`) — a defensive fallback for degenerate workspaces. If the workspace has TaskColumns but none with `is_done=false`, throw `new Error('Ingen öppen uppgiftskolumn i arbetsytan')` — caller surfaces as structured error. **PO v0.2 note (NH-3):** the "only done columns exist" branch is a true defensive edge — every workspace auto-seeds `['Att göra', 'Pågående', 'Klar']` on first `getTaskColumns` call (see `app/actions/tasks.ts:580-622`) and `deleteTaskColumn` blocks deletion of columns with `is_default: true` (see `app/actions/tasks.ts:1596-1599`), which includes the auto-seeded non-done "Att göra" column. The throw is a last-line guardrail, not an expected user-facing path.
   2. Computes position: `(max(position) in target column) + 1`, default `0` when empty.
   3. Creates the Task with the epic AC 1 field map — verbatim:
      - `title` = `args.finding.title` (max 200 chars, inherits finding's Zod-enforced cap from Story 21.7).
      - `description` = `'Korrigerande åtgärd för avvikelse: ' + args.finding.description`. **Truncation rule:** the Task.description column is `@db.Text` (no hard limit in Postgres) so no truncation required. Leave the full finding description inline. If finding description is 5000 chars, resulting task description is ~5040 chars — fine.
      - `due_date` = `args.finding.dueDate` (nullable passthrough).
      - `assignee_id` = `args.itemResponsibleUserId ?? args.leadAuditorUserId` (epic AC 1 explicit fallback — item responsible if finding is item-linked and the item has a responsible_user_id; otherwise the cycle's lead_auditor_user_id, which is always populated per Story 21.1 schema).
      - `priority` = `'HIGH'` — **opinionated default** since AVVIKELSE findings are by definition deviations requiring prompt action. Aligns with the existing `TaskPriority` enum; not a new enum value.
      - `column_id` = resolved column from step 1.
      - `position` = computed from step 2.
      - `created_by` = `args.createdByUserId`.
      - `compliance_finding_id` = `args.finding.id` (back-reference populated on create — symmetric with the `corrective_action_task_id` write the caller issues on the finding).
   4. **PO v0.5 — creates the M:N cycle link row in the same tx.** After the `Task` is created, call `tx.complianceCycleTaskLink.create({ data: { task_id: newTask.id, cycle_id: args.cycleId } })`. The link row is what makes the task appear on the right-rail "Länkade kontroller" card (AC 9) AND what the future cycle-detail "Uppgifter" tab (Story 21.15) reads from. The `compliance_finding_id` FK on Task stays populated too — it's the 1:1 reference to the specific finding that spawned this task. Both fields co-exist and carry different semantic weight.
   5. Returns `{ taskId, columnId }`. Does NOT log activity — caller owns the `finding_task_spawned` log entry so the entity_type is `compliance_finding` (the spawning cause), not `task` (the side-effect).
   6. Does NOT invalidate any caches or call `revalidatePath` — caller handles that too. Helper is pure-transactional.

5. **Extend `closeFinding` in `app/actions/compliance-finding.ts` (MODIFIED — Story 21.7 file).** After Story 21.7's step 5 (the `complianceFinding.update` that sets `closed_at` + `closed_by_user_id`) and BEFORE step 6 (activity log for `finding_closed`), insert a task-completion step:
   ```ts
   // Epic 21.8: mark linked corrective-action task completed if still open —
   // but ONLY when the close path was the happy path, NOT the manual-override path.
   //
   // PO v0.3 gate-review addition: Story 21.7 closeFinding reaches step 5 via two
   // distinct paths (see 21.7 AC 5 step 3):
   //   (a) Happy path — non-AVVIKELSE, OR AVVIKELSE with the linked task already completed.
   //   (b) Manual-override path — AVVIKELSE + linked task still open + user supplied `closeReason`.
   // In path (b) the user has EXPLICITLY said "close this finding despite the corrective task
   // being incomplete". Auto-completing the task would silently invert that intent. Guard
   // the task-complete branch on `input.closeReason == null` so only the happy path triggers
   // the side-effect.
   if (finding.corrective_action_task_id && parsed.data.closeReason == null) {
     const linkedTask = await tx.task.findFirst({
       where: { id: finding.corrective_action_task_id, workspace_id: ctx.workspaceId },
       select: { id: true, completed_at: true, column_id: true },
     })
     if (linkedTask && linkedTask.completed_at === null) {
       const doneColumn = await tx.taskColumn.findFirst({
         where: { workspace_id: ctx.workspaceId, is_done: true },
         orderBy: { position: 'asc' }, // first is_done column (typically "Klar")
         select: { id: true },
       })
       if (doneColumn) {
         const maxPosition = await tx.task.aggregate({
           where: { column_id: doneColumn.id },
           _max: { position: true },
         })
         await tx.task.update({
           where: { id: linkedTask.id },
           data: {
             column_id: doneColumn.id,
             position: (maxPosition._max.position ?? -1) + 1,
             completed_at: new Date(),
           },
         })
         // Log `finding_task_completed` (entity_type compliance_finding, payload carries task_id + column_id)
         await logActivity(ctx.workspaceId, ctx.userId, 'compliance_finding', finding.id, 'finding_task_completed', null, { task_id: linkedTask.id, new_column_id: doneColumn.id })
       }
       // If no `is_done` column exists (degenerate workspace), skip silently — log a warning via console.warn. User can complete the task manually.
     }
     // If linkedTask is null (soft-deleted / FK stale), skip silently — log already occurred via onDelete: SetNull.
   }
   // If closeReason IS set (manual override), the task stays in its current state.
   // The finding is closed with manual_override=true in its own activity log (Story 21.7 AC 5 step 6);
   // the corrective task remains the user's responsibility to close manually via the Kanban UI.
   ```
   **Transaction boundary:** this block should run inside the same tx as the `closeFinding` update so that a failed task update rolls back the finding close. Story 21.7's closeFinding does NOT currently wrap in `$transaction` — for AC 5, Story 21.8 MUST wrap the finding update + task update together. Small diff: promote the two writes + one read to `prisma.$transaction(async (tx) => { ... })`. The existing `loadFindingScopedToWorkspace` call + `assertCycleEditableUi` check happen BEFORE the tx (read-only). `[Source: Story 21.7 AC 5 closeFinding body (manual-override path); docs/qa/gates/21.7-findings-create-edit-close.yml PROD-001 context; app/actions/tasks.ts:514-521 (task status+completed_at update pattern)]`

6. **Task-completion notification hook — `lib/compliance-audit/notify-finding-task-completed.ts` (NEW).** Export single async function:
   ```ts
   export async function notifyIfFindingTaskCompleted(args: {
     taskId: string
     workspaceId: string
     actorUserId: string

   ```
   Fire-and-forget — caller does NOT await the result, and exceptions are swallowed with `console.error` (mirrors the pattern at `app/actions/tasks.ts:1275-1278` for `createTaskNotification` error handling). Body:
   1. Load the task with `compliance_finding_id` + linked finding + cycle in one query:
      ```ts
      const task = await prisma.task.findFirst({
        where: { id: args.taskId, workspace_id: args.workspaceId },
        select: {
          id: true,
          title: true,
          completed_at: true,
          compliance_finding_id: true,
          compliance_finding: {
            select: {
              id: true,
              closed_at: true,
              cycle: { select: { id: true, name: true, lead_auditor_user_id: true } },
            },
          },
        },
      })
      ```
   2. Early-exit when any of the following hold (all silent, no error): task null; `compliance_finding_id` null; `compliance_finding` null (soft-deleted); `compliance_finding.closed_at !== null` (finding already closed — no point notifying); `task.completed_at === null` (task not actually completed — defensive against caller misuse).
   3. **Dedup gate (activity-log based — authoritative path, no Notification-table query):** check the ActivityLog for any `finding_task_completion_notified` row emitted against this finding in the last 24h:
      ```ts
      const recentNotification = await prisma.activityLog.findFirst({
        where: {
          workspace_id: args.workspaceId,
          entity_type: 'compliance_finding',
          entity_id: task.compliance_finding.id,
          action: 'finding_task_completion_notified',
          created_at: { gte: new Date(Date.now() - 24 * 60 * 60 * 1000) },
        },
        select: { id: true },
      })
      if (recentNotification) return // dedup — no second notification within 24h for this finding
      ```
      The activity log IS the canonical audit trail for this loop (step 5 writes to it), so gating on its own prior entries is both natural and single-source-of-truth. We deliberately avoid querying the Notification table because (a) it requires a per-notification-type index + composite filter that the current schema doesn't optimise for, and (b) Notification rows can be user-deleted which would defeat the dedup. **PO v0.2 decision (SF-1): this is the single authoritative dedup path — do NOT add a Notification-table query fallback.**
   4. Create the Notification row matching the actual `prisma/schema.prisma:1266-1292` column shape (**PO v0.2 fix — Critical #1**). The Notification model has NO `recipient_user_id`, NO `actor_user_id`, and NO `link` column — the recipient lives in `user_id`, actor context lives in the `body` text only (matches the `task-notifications.ts:343-345` precedent), and deep-links are derived by the frontend resolver from `entity_type` + `entity_id`:
      ```ts
      await prisma.notification.create({
        data: {
          workspace_id: args.workspaceId,
          user_id: cycle.lead_auditor_user_id,   // recipient — schema field is `user_id`
          type: NotificationType.FINDING_READY_TO_CLOSE,
          title: 'Uppgift slutförd för avvikelse',
          body: `Uppgiften "${task.title}" är slutförd. Avvikelsen kan nu stängas i kontrollen "${cycle.name}".`,
          entity_type: 'compliance_finding',
          entity_id: task.compliance_finding.id,
        },
      })
      ```
      **Actor context:** `args.actorUserId` is NOT persisted on the Notification row (schema doesn't support it). If product wants the actor named in the notification body, prefix the body with the actor's display name — load via `prisma.user.findUnique({ where: { id: args.actorUserId }, select: { name: true, email: true } })` and prepend as `${actor.name ?? actor.email} slutförde uppgiften...`. For MVP minimalism, skip the actor-name lookup — the Swedish body copy above is sufficient. The actor IS persisted on the ActivityLog row (step 5), which is the auditable record.

7. **Task-completion hooks wired into all three task-status paths.** Three existing actions invoke `notifyIfFindingTaskCompleted` immediately after their task update succeeds (fire-and-forget, error-swallowing) — matches the existing `createTaskNotification(...).catch(...)` pattern.

   **Transition guard (PO v0.2 — SF-2 precise wording):** fire the hook for a task if and only if BOTH hold:
   1. `previousTask.completed_at === null` (the task was NOT completed before this mutation), AND
   2. `targetColumn.is_done === true` (the mutation is moving the task to a done column — which is what sets `completed_at = new Date()` in the same update per the existing `task-modal.ts:519` idiom).

   Why both are required: `completed_at` is a **derived** field set by the action body based on the target column's `is_done` flag (see `task-modal.ts:519` + `tasks.ts:670`). A task can be "re-completed" (moved within done columns) without being a genuine completion transition — guard #1 prevents re-notification. A task can also be moved to a non-done column (reopening) — guard #2 prevents notifying on reopen.

   - **`updateTaskStatusColumn` at `app/actions/task-modal.ts:482-566`** — after line 521 `prisma.task.update` and before the existing `createTaskNotification(STATUS_CHANGED, ...)` call at :540. Both guards are trivially available: `task.completed_at` from the pre-update findFirst at line 491, `column.is_done` from the parallel findFirst at line 495. Fire only when the AND of the two guards holds.
   - **`updateTaskStatus` at `app/actions/tasks.ts:639-687`** — after line 672 `prisma.task.update`. Extend the pre-update findFirst at line 647 to `select: { completed_at: true }` (or `include` — whichever matches the existing shape). Fire only when `task.completed_at === null && column.is_done === true`.
   - **`updateTasksBulk` at `app/actions/tasks.ts:694-774`** — after line 756 `prisma.task.updateMany`. `updateMany` does NOT return per-row pre/post state, so reuse the pre-update `tasks` array already fetched at line 717 (which has `completed_at` implicitly via `findMany` default select — **verify the select projection at Task 6.3; if narrowed, extend to include `completed_at`**). Filter `taskIds` to those where `task.completed_at === null` in the pre-fetch, THEN gate on `updates.columnId` being set AND the target column's `is_done === true`. Fire the hook for each surviving task id via `Promise.all(filtered.map(id => notifyIfFindingTaskCompleted({ taskId: id, ... }).catch(console.error)))`.     model ComplianceCycleTaskLink {

No separate copy file for this story — the strings are inline at their single-use sites (notification template + chip component). `[Source: architecture/epic-21-lagefterlevnadskontroll.md §2.2]`

### Technical constraints

- **TypeScript strict + `exactOptionalPropertyTypes`** — The new `TaskDetails.complianceFinding` field is explicit `| null` (not `| undefined`). Optional fields on the helper arg objects use `| null` where the underlying DB column is nullable. Mirrors Story 21.7's existing `FindingRow` null-first shape.

- **Prisma `$transaction` + TransactionClient signature** — `spawnCorrectiveActionTask` takes `Prisma.TransactionClient` explicitly. This is the same signature as `isLeadAuditor` from Story 21.14 AC 5 and matches the forthcoming `assertCycleEditable(tx, cycleId)` from Story 21.10. ### Deviations from epic

1. **`FINDING_READY_TO_CLOSE` NotificationType enum value introduced (AC 12).** The epic's AC 4 says "triggers a notification" without naming the NotificationType. A new enum value is the cleanest integration with the existing Notification model + `buildNotificationText` switch. Additive migration, zero backward-compat risk. Alternative considered: use `STATUS_CHANGED` — rejected because the audience (lead auditor, not task participants) and semantics (finding-level, not task-level) don't match.

2. **`deleteFinding` action + confirmation dialog deferred (epic AC 5).** The epic's AC 5 says "Deleting the finding does NOT auto-delete the task... a confirmation dialog explains this." Story 21.7 did NOT implement `deleteFinding` (out of its AC list); Story 21.8's focus is task-lifecycle sync, not finding-delete UX. The schema-level `onDelete: SetNull` on `Task.compliance_finding_id` already provides the data-integrity half of AC 5. The user-facing confirmation dialog is deferred until a dedicated story owns `deleteFinding`. **Carry-forward note for the future delete story:** when `deleteFinding` ships, its confirmation dialog must warn "This task will remain in the workspace with no link back to this finding" in Swedish. The activity-log entry `finding_deleted` will need a formatter + category mapping — add via the three-file carry-forward convention.

3. **Task list card "Från kontroll" chip + filter chip deferred (epic AC 2).** Epic AC 2 says "optional 'Från kontroll [cycle name]' badge / filter chip" — the "optional" qualifier justifies the deferral. MVP ships the modal-only chip (AC 9), which is the minimum user-facing affordance. If after user feedback the card-level chip proves valuable, a post-MVP story adds it (low-effort — reuse the existing chip component).

4. **Task priority defaulted to HIGH for corrective actions (AC 1 step 3).** The epic doesn't specify a default priority. Story 21.8 opinionates: AVVIKELSE → HIGH by definition (a deviation requiring attention). User can downgrade manually via the existing priority editor. If product disagrees, change the single constant in `task-spawner.ts:priority: 'HIGH'` to the chosen default.

5. **Finding type change (OBSERVATION ↔ AVVIKELSE) does NOT retroactively spawn / clean up tasks (AC 11).** Story 21.7's `updateFinding` allows changing the type; Story 21.8 does NOT hook into that path. Rationale: adding a spawn-on-type-change branch creates subtle invariants (what if the finding already had a task from a previous spawn? what if the user intended to cancel the task by changing type?). Keep creation-time semantics only — user can manually create a task for a retroactively-upgraded OBSERVATION→AVVIKELSE if needed. Documented inline.

6. **Notification dedup window of 24h (AC 6 step 3) not in the epic.** A defensive anti-spam measure. Without dedup, a user who closes + reopens a task repeatedly would notify the lead auditor each time. 24h is a conservative default matching the existing `task-notifications.ts` dedup pattern.

7. **Activity log action strings named with `finding_` prefix (AC 10).** Mirrors Story 21.7's naming convention (`finding_created`, `finding_closed`). Alternative `task_` prefix was considered — rejected because the entity_type is `compliance_finding`, so the prefix reflects where the log row is rooted.

All deviations are additive and small; none changes the epic's intent. Document each in the story's Change Log when landing.
