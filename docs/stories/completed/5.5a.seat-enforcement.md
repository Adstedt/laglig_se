# Story 5.5a: Seat Enforcement

## Story

**As a** workspace owner,
**I want** the platform to block adding more users than my plan allows,
**so that** I'm prompted to upgrade or buy add-on seats before paying for capacity I don't have.

## Acceptance Criteria

1. **Prisma migration adds `Workspace.trial_picked_tier SubscriptionTier?`** (nullable). Migration runs cleanly on existing data; existing rows have NULL.
2. **`lib/usage/limits.ts` exists** with the full skeleton from the parent spec — `TIER_LIMITS`, `TEAM_ADDON_PER_SEAT`, `getEffectiveLimits()`, `isUnlimited()`, AI-quota helpers (helpers used by 5.5c, but the full module lands here so the file isn't half-built across two stories).
3. **Invite-create path is gated:** `POST /api/workspace/invitations` rejects with HTTP 402 + `code: 'SEAT_LIMIT_REACHED'` when current seat count + pending invitations >= `getEffectiveLimits(workspace, addonSeats).users`. Pending invitations count toward the limit so a workspace can't fire 10 invites in parallel and break the cap on accept.
4. **Invite-accept path is gated:** `acceptInvitation()` server action re-checks the seat limit at accept time (race protection against tier downgrades or other accepts landing first). Rejects with the same shape if the workspace has filled while the invitation was pending. Stale invite is marked `EXPIRED`.
5. **Add-on seats are correctly counted:** `getEffectiveLimits(workspace, addonSeatCount)` returns the right `users` value. `addonSeatCount` is computed by counting active Stripe Subscription Items on the workspace's subscription whose `price.id !== STRIPE_TEAM_PRICE_ID` (i.e., quantity beyond the Team base Price).
6. **Trial workspaces honor `trial_picked_tier`:** a workspace with `subscription_tier='TRIAL'` and `trial_picked_tier='TEAM'` allows 3 users (Team limit), not 1 (Solo fallback).
7. **UI surfaces seat usage:** `/settings?tab=team` shows "X / N platser används". The invite modal shows "Du har använt X / N platser. Bjud in en till?" and disables the submit button when at limit, with a CTA to upgrade or buy add-on seats.
8. **Employees are NOT enforced** in any code path. Comment in `lib/usage/limits.ts` next to the `employees` field explicitly states "deferred — HR module not yet built". No `/api/employees` gate, no employee count in the team tab.
9. **Error response shape** (HTTP 402) on both create + accept paths:
   ```json
   {
     "error": "Platsgräns uppnådd",
     "message": "Din plan tillåter X platser. Uppgradera till Team eller köp en extra plats.",
     "code": "SEAT_LIMIT_REACHED",
     "currentSeats": 1,
     "limit": 1,
     "tier": "SOLO"
   }
   ```
10. **Enterprise tier bypasses all seat checks** — `getEffectiveLimits()` returns `users: null`, callers correctly interpret as unlimited.
11. **Test coverage:** unit tests for `getEffectiveLimits()` resolution; integration tests for both invite paths blocking at limit; E2E for the invite-modal CTA flow.

### Add-on seat counting

Team add-on seats live on the Stripe subscription as additional `SubscriptionItem` rows. The base Team Price ID is `env.STRIPE_TEAM_PRICE_ID`. Any other Price ID on the subscription with `recurring.interval='month'` is an add-on seat (each `quantity: 1`). Story 5.6 (Add-On Purchase System, backlog) will formalize this — for v1 just count items where `price.id !== STRIPE_TEAM_PRICE_ID` and sum quantities.

> **⚠️ Cross-reference with Story 5.6:** This counting logic assumes Story 5.6 will model add-on seats as **additional `SubscriptionItem` rows on the same subscription, with distinct Price IDs**. If 5.6 ships first with a different shape (e.g., a single `quantity` increment on the base Price, or an `addon_seats` column on `Workspace`), revise `countActiveAddonSeats()` to match. Validate against the 5.6 design doc before implementing this function.

```typescript
// lib/usage/seats.ts (sketch — not full implementation)
import { stripe } from '@/lib/stripe/config'
import { env } from '@/lib/env'

export async function countActiveAddonSeats(workspaceId: string): Promise<number> {
  const workspace = await prisma.workspace.findUnique({
    where: { id: workspaceId },
    select: { stripe_subscription_id: true, subscription_tier: true },
  })
  if (!workspace?.stripe_subscription_id) return 0
  if (workspace.subscription_tier !== 'TEAM') return 0

  const sub = await stripe.subscriptions.retrieve(workspace.stripe_subscription_id, {
    expand: ['items.data.price'],
  })

  return sub.items.data
    .filter((item) => item.price.id !== env.STRIPE_TEAM_PRICE_ID)
    .reduce((sum, item) => sum + (item.quantity ?? 0), 0)
}
```

### Pending-invitation race

Counting pending invitations toward the seat limit prevents this attack/oversight:

1. Solo workspace (1 user, owner) fires 5 invites in 5 seconds (each invite-create currently passes the 1-user check)
2. All 5 invitees accept simultaneously
3. Workspace ends up with 6 users despite paying for 1 seat

Counting `WorkspaceInvitation.status='PENDING'` toward the seat tally fixes this at create time. The accept-time re-check (AC 4) handles the rare case where the invitation was created before a downgrade flipped the tier limit.

**Latency budget:** seat check is O(1) DB query (`COUNT` on `WorkspaceMember` + `WorkspaceInvitation`) plus a Stripe API call for add-on seats. Stripe Subscription retrieval is ~100-200ms — acceptable on the invite path which isn't a hot path.

**Caching scope (important):** if multiple seat checks fire in one HTTP request (rare but possible — e.g., team tab page + concurrent modal data fetch), use a **request-scoped memo** (e.g., a `Map` declared inside the route handler / server action call, NOT module-scope). Do NOT import 5.5c's process-wide `seat-cache.ts` here — that cache is tuned for the chat hot path's repeated reads of the same workspace's seat count over many minutes. The invite-create path's seat count needs to reflect any concurrent invitation-create in the same request, so per-request memoization is the right scope.
