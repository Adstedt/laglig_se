# Story 25.3: Six tutorial tab content panels (B.3)

## Status

Done

## Story

**As a** new user reading the onboarding tutorial while my laglista generates,
**I want** each of the 6 tutorial tabs to show real product-feature previews + explanatory copy instead of the "Innehåll kommer snart" placeholder,
**so that** the 2–5 minute LLM wait becomes a guided learning moment where I understand what Laglig.se actually does (laws, kravpunkter, tasks, kontroller, change events, the AI agent) before I ever touch the live product.

This is the **B.3 wedge** of Epic 25: it fills in the six tab content panels that the B.2 tutorial-step shell left as `<p>Innehåll kommer snart</p>` placeholders. B.2 owns the tab bar, progress strip, telemetry wiring, and Minimera handoff — none of those change here. B.3 is **pure content authoring + light orchestration**: six new component files under `tutorial-tabs/`, a barrel + lookup map, and a small swap inside `<TutorialStep>` to render the active tab's component instead of the placeholder.

Concretely: B.3 ports each tab's copy + realistic mini-preview verbatim from `_prototypes/onboarding-tutorial-modal.html` (frame ③ panels 1-6, lines 919–2717) into six React components. Each panel is a **2-column grid** — left col (`col-span-2`) is the explainer copy + 3-item benefits list + "Läs mer i hjälpcentret" link; right col (`col-span-3`) is a realistic-looking mock of the actual product surface (laglistor table, kravpunkter checklist, mini Kanban, kontroller cycle, change card, AI chat). All static — no live data, no API calls, no live components borrowed from the real product (the prototype's mocks are deliberately copied as new self-contained JSX so we don't accidentally mount the real `<DocumentListTable>` etc. with stub data).

This unlocks the "tutorial-while-waiting" UX the architecture has been pointing at since the brief. Users now spend the 2–5 min LLM wait scrolling through six rich previews of what they're about to use, instead of staring at a placeholder.

### Accessibility

18. **Each tab component's outermost element is a semantic container** (`<div>` is fine; no need for `<section>` since the parent's tabpanel role already provides region semantics).
19. **Heading hierarchy** inside each panel starts at `<h3>` (NOT h1/h2) because the modal's `<DialogTitle>` is the page's primary heading per Radix Dialog semantics. The eyebrow ("Översikt" / "Bevisföring" / etc.) is a styled `<div>` or `<p>`, NOT a heading.
20. **Decorative icons inside the mock previews carry `aria-hidden="true"`** (the Lucide icons in the table toolbar, group header chevrons, file chips, etc. are purely visual — screen readers should skip them and read the labels instead).
21. **Mock avatars in the previews use `aria-hidden="true"`** + visible text fallback (e.g., initials). They're not real users; treating them as accessible images would confuse screen reader users.
22. **The "Läs mer i hjälpcentret" link's external-link indicator** (`ArrowUpRight` icon) carries `aria-hidden="true"` and the link's accessible name comes from its visible text ("Läs mer i hjälpcentret"). Don't add `aria-label="opens in new tab"` — the link is non-functional in B.3 and adding aria-label about new-tab behavior would be misleading. **Visual states** (per prototype `:996-1005`): default `text-foreground` + `font-medium`, hover `hover:underline underline-offset-4`, focus uses the global `focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2` (already applied by the app's base styles when using `<Link>` / `<button>` — no extra class needed).
23. **No `aria-disabled` on the decorative buttons inside previews.** They're meant to look interactive (per AC 13) — adding `aria-disabled` would announce them as disabled to screen reader users, contradicting their visual treatment. Use plain `<button type="button">` with no handler; users who tab to them get focus but no action.

### Telemetry (no changes)

25. **No analytics for clicks on the decorative buttons/links inside previews.** They're visual only — instrumenting them would inflate the event log with non-actions.

### Existing patterns to follow

- **shadcn/ui first, but raw HTML is fine for static preview mocks.** The tab content panels are not interactive surfaces — they're visual mocks of other product surfaces. The architecture's coding standards (`docs/architecture/17-coding-standards.md:73-153`) require shadcn for interactive components; static visual mocks of OTHER components are exempt because they're not real UI elements. Use plain `<div>` / `<span>` / `<button type="button">` (no handler) for the mock previews. The "Läs mer i hjälpcentret" link MAY use Next.js `<Link href="#">` per AC 13 if you want lint-clean (but `<a href="#">` is also acceptable since it's not a real route).
- **Lucide Icons** for all icons. The prototype uses `<i data-lucide="…">` which is the CDN-loader convention; the React equivalent is `import { Layers } from 'lucide-react'` + `<Layers className="h-3 w-3" aria-hidden="true" />`. Sizes in the prototype:
  - 3w/3h: most inline icons in benefits lists, table toolbars
  - 3.5w/3.5h: link-decoration arrows (ArrowUpRight, etc.)
  - 4w/4h: larger header icons (Bell on Lagändringar panel)
- **Tone tokens for the preview chips/pills.** After Task 0 lands the 8 `--tone-*-soft-*` variables in `app/globals.css`, use Tailwind arbitrary-value syntax with `hsl(var(--…))` wrappers (the variables store HSL triples, not full color values). Common patterns:
  - Info pill: `bg-[hsl(var(--tone-info-soft-bg))] text-[hsl(var(--tone-info-soft-fg))]`
  - Success pill (Efterlevs): `bg-[hsl(var(--tone-success-soft-bg))] text-[hsl(var(--tone-success-soft-fg))]`
  - Warning pill (Avviker / Saknar bevis): `bg-[hsl(var(--tone-warning-soft-bg))] text-[hsl(var(--tone-warning-soft-fg))]`
  - Danger pill (Hög påverkan / removed-in-diff): `bg-[hsl(var(--tone-danger-soft-bg))] text-[hsl(var(--tone-danger-soft-fg))]`
  - Neutral pill: `bg-muted text-muted-foreground`
  - Sage tint for benefits-list icon containers: `bg-section-sage` (defined at `app/globals.css:231-233`). **NOTE**: the prototype uses `bg-sage` as a shortcut in its hand-rolled Tailwind config — the app does NOT have that alias, only `bg-section-sage`.
  - Warm tint background: `bg-section-warm/60` (defined at `app/globals.css:228-230`). Prototype's `bg-warm/60` → app's `bg-section-warm/60`.
- **font-safiro for h3 headings inside panels** (per AC 10) — class already in `app/globals.css` `@layer utilities` from Story 25.0.
- **No live data, no live components.** Reasoning: importing real components like `<DocumentListTable>` would require feeding them workspace context, mock workspace ids, etc. — and any one of those imports could mount real Prisma queries, real React Query providers, or real auth context. Static JSX with hardcoded mock content avoids that entire class of bug. Visual fidelity is achieved by matching styles + layout, not by reusing the real components.

### Key decisions baked into this story

- **One file per tab** (not one file with all 6 inlined). Reasoning: each panel is 50–150 lines of JSX. A single file would be 600+ lines and unwieldy for review. Co-location under `tutorial-tabs/` mirrors the architecture's prescribed structure at §6.1 line 196-203 and makes B.3a/B.3b splittability trivial if velocity demands (per Epic PRD line 186-188).
- **Lookup map (`TUTORIAL_TAB_COMPONENTS`) instead of switch statement.** Reasoning: makes the parent component thin (one map lookup instead of 6 switch cases), gives B.5 (feedback tab) a clean extension point (add `'feedback': FeedbackStep` to the map), and lets tests stub the entire map in one mock call.
- **Static JSX for previews, NOT real components.** Per the "Existing patterns" section above. Reasoning: avoids the entire class of "real component needs real context" bugs that would otherwise have to be solved with extensive mocking.
- **Hybrid grounding — prototype layout + real-app chrome (AC 12.5 + Task 0.5).** The prototype was authored before all 6 real surfaces shipped; the real app may have drifted from the prototype's chrome details (column headers, chip colors, spacing, typography). Dev runs a per-tab audit before building — opens each real surface in a browser, documents drift, and biases the mock toward the real app where they diverge. This avoids the "tutorial shows X, real surface shows Y" cognitive dissonance that would happen if Dev ported the prototype verbatim. Acceptable trade-off: the previews will drift again as the real surfaces evolve — but that's a maintenance concern, not a launch blocker; future re-grounding can happen incrementally as the rest of the app evolves.
- **No mobile responsive treatment in B.3** (per AC 11). The modal is desktop-first at `max-w-[760px]`; on mobile the modal already scrolls horizontally per existing Radix Dialog behavior. Optimizing the tab panels for narrow viewports is deferred — would require a separate UX pass and is not in the Epic 25 scope.
- **Decorative buttons inside previews have NO `aria-disabled`** (per AC 23). Reasoning: they're meant to LOOK interactive (that's the whole point of a realistic mock) — announcing them as disabled to screen readers would contradict the visual treatment. The trade-off: a screen reader user could tab to them and press Enter with no action. Mitigated by the fact that they're inside a tutorial-step that itself isn't a primary workflow (user is reading, not doing).
- **No `aria-label="opens in new tab"` on the "Läs mer i hjälpcentret" link** (per AC 22). The link is non-functional in B.3 — adding that label would be misleading. When the real hjälpcenter routes ship (future story), the link can be wired and the aria-label added then.
- **Tab components take NO props.** Reasoning: they're pure-presentation content modules. If the parent ever needs to inject context (e.g., user's company name into copy), that's a future refactor; for B.3 the copy is fully static per the Epic PRD line 184 ("All six panels static — no live data, just realistic-feeling mock content").
- **No `<ErrorBoundary>` wrap around the tab body — accepted risk.** Reasoning: the tab components are static JSX with zero logic, no hooks, no external data fetching, no derived computations. The runtime-crash surface is essentially "a typo in JSX that fails to compile" — which would be caught at build time, not at render. If a panel does somehow crash at runtime, React would crash the entire modal (`<FirstRunModal>` has no error boundary either). Mitigation: keep each panel small (~50-150 lines) and well-tested via the per-tab render tests in Task 4 — that catches the "renders without crashing" case automatically. A future hardening story can wrap the tab body in an `<ErrorBoundary>` with a "Något gick fel — ladda om sidan" fallback if real-world crashes ever materialize.
