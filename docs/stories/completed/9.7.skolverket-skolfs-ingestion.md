# Story 9.7: Skolverket (SKOLFS) FГ¶reskrifter Ingestion вҖ” Valid + Consolidated via the SKOLFS API + Monitoring Hooks

## Status

Done

## Story

**As a** kommun (municipal) compliance owner вҖ” and the catalog itself,
**I want** Skolverket's in-force **SKOLFS** fГ¶rfattningar (the gГӨllande grundfГ¶rfattningar + their published consolidated versions) ingested as first-class `AGENCY_REGULATION` documents,
**so that** they resolve in laglistor, kravpunkter, search and RAG, link from marketing pages, and feed amendment notifications through the existing pipeline вҖ” closing the single biggest catalog gap for the **kommun** segment (lГӨroplaner, kurs-/ГӨmnesplaner, betygskriterier, statsbidrag, behГ¶righet/legitimation), since municipalities are *huvudmГӨn* for the bulk of Swedish public schooling and SKOLFS carries the binding rules they operate against.

## Scope note

This story is **Phase 1 вҖ” ingestion**: get the **gГӨllande** (in-force) SKOLFS set into the catalog, correctly attributed per issuer, with amendment + upcoming-change *metadata* captured as the baseline for monitoring. **Forward amendment monitoring** (emit `ChangeEvent`s when SKOLFS publishes/repeals/amends) is **Phase 2 вҖ” Story 9.8** (drafted alongside this). **Per-version historik reconstruction from amendment PDFs is explicitly out of scope** (the consolidation already merges them; the PDFs remain available to backfill later). This mirrors the 9.5/9.6 split for Socialstyrelsen. [Decision recorded in-session 2026-06-25.]

### Discovery (API enumeration вҖ” replaces the headless-scrape of 9.5)

3. **`scripts/enumerate-skolverket-foreskrifter.ts`** enumerates via the **public JSON API** (no headless browser, no auth) вҖ” `GET https://skolfs.skolverket.se/api/statute?size=20000` returns all **5,421** docs вҖ” and applies the verified **gГӨllande corpus rule**:
   - **Keep** `validity == "VALID"` AND `documentType вҲҲ {GRUNDFORFATTNING, ALLMANNA_RAD_OVRIGT}` вҶ’ **~1,400** readable docs (verified 1,390 + 10).
   - For each kept base, **prefer the consolidated PDF** (`SENASTE_LYDELSE`) when a `VALID` one exists for that `baseSkolfsNumber` (216 unique bases), **else the base PDF** (1,174). **Dedup consolidated-by-base** вҖ” exactly one base (SKOLFS 2011:22) lists its consolidated entry twice; pick one PDF per law. [Source: verified `validity Г— documentType` cross-tab + orphan/zombie/dup checks 2026-06-25]
   - **Exclude** (do NOT ingest as docs): all `EXPIRED` (3,173 вҖ” repealed/superseded, incl. 563 superseded consolidated snapshots), all `VALID ANDRINGSFORFATTNING` (582 вҖ” amendment *acts*, consumed as change events in 9.8), all `UPCOMING` (49 вҖ” not in force; surfaced via 9.8's upcoming feed). Verified: **0 orphan consolidated, 0 zombie (valid consolidated of an expired base).**
4. Each registry entry carries `{ documentNumber: "SKOLFS YYYY:N", title (newline/whitespace-cleaned вҖ” API titles contain embedded `\n`), documentType, validity, isConsolidated, baseSkolfsNumber, pdfType вҲҲ {GRUNDFORFATTNING, SENASTE_LYDELSE, ALLMANNA_RAD_OVRIGT}, pdfUrl, sourceUrl, issuedBy, regulatoryBody, agencyPrefix: "SKOLFS", decisionDate, effectiveDate, promulgationDate, latestChange, latestChangeBySkolfsNo, amendmentChain[], upcoming[] }`, persisted to `data/skolverket-foreskrifter-registry.json` (mirrors the `data/socialstyrelsen-foreskrifter-registry.json` shape). `pdfUrl` is `вҖҰ/api/document/{pdfType}/{number}/pdf` (verified `200 application/pdf`). [Source: 9.5 registry pattern; verified PDF endpoint]

### Ingestion (PDF intake вҖ” reuse the existing agency-PDF pipeline tail)

6. **`scripts/ingest-skolverket-foreskrifter.ts`** ingests each registry entry through the **PDF intake path** (SKOLFS content is **PDF-only вҖ” there is no HTML body endpoint**, verified `вҖҰ/html вҶ’ 404`). **Reuse boundary (verified by SM 2026-06-25):** `scripts/ingest-agency-pdfs.ts` `processDocument` covers **download вҶ’ Claude вҶ’ validate вҶ’ derive вҶ’ linkify вҶ’ upsert and then RETURNS at ~line 362** вҖ” it does **NOT** chunk/embed or set `search_vector`. So reuse it for that front half, and take the **chunk/embed/FTS tail from the 9.5 ingester** (AC 7). The model for the whole flow is the 9.5 Socialstyrelsen ingester (registry-driven, fail-soft, standard flags), with PDF intake swapped in for HTML intake:
   - download the PDF to `data/skolfs-pdfs/` (a fetch step or folded into enumerate), then **PDF вҶ’ Claude** (base64 `document` block) вҶ’ semantic `<article class="legal-document">` using `AGENCY_REGULATION_SYSTEM_PROMPT` / `AGENCY_DEFAULT_MODEL`. [Source: scripts/ingest-agency-pdfs.ts:201-225; lib/agency/agency-regulation-prompt.ts]
   - `validateLlmOutput(raw, docNumber)` вҶ’ `htmlToMarkdown` / `htmlToPlainText` / `parseCanonicalHtml({documentType:'AGENCY_REGULATION'})`. [Source: scripts/ingest-agency-pdfs.ts:296-303]
   - **linkify** via `buildSlugMap()` + `linkifyHtmlContent(html, slugMap, docNumber)` вҶ’ store linkified result as `html_content`. [Source: scripts/ingest-agency-pdfs.ts:304-309]
   - upsert `LegalDocument` keyed on `document_number` (`AGENCY_REGULATION`, `ACTIVE`, slug via `generateAgencySlug`). [Source: scripts/ingest-agency-pdfs.ts:330-357]
   - idempotent (upsert); standard flags `--dry-run/--limit/--filter/--skip-existing/--force`; respects the budget ceiling with running cost + rate-limit between calls. **Note:** `BUDGET_CEILING_USD` in `ingest-agency-pdfs.ts` is a **hardcoded `const = 15`** (line 67) вҖ” raise/parameterize it for the bulk run (see AC 12). [Source: scripts/ingest-agency-pdfs.ts:65-67,83-129,617-659]
   - **Fail-soft:** a doc that truncates (`stop_reason === 'max_tokens'`) or fails `validateLlmOutput` is recorded to a **failures manifest** (`data/skolverket-ingest-failures.json`, docNumber + reason) and the batch **continues** вҖ” one bad doc never aborts the run. [Source: scripts/ingest-agency-pdfs.ts:247-251; 9.5 AC4]
6a. **Large-doc handling** (the lГӨroplaner are the heavy case вҖ” e.g. the gymnasieskolan consolidated PDF is 265 KB / hundreds of pages, `issuedBy: Regeringen`, 12 amendments): route to **chunked-by-chapter normalization** when extracted HTML exceeds ~55 KB OR a single pass returns `max_tokens`; stitch chapters into one `<article>`. The working implementations are **`scripts/ingest-socialstyrelsen-foreskrifter.ts` (chunked mode вҖ” `LARGE_HTML_THRESHOLD` / "CHUNKED MODE", ~lines 195вҖ“312)** and **`scripts/ingest-afs-regulations.ts` (`callClaudeChunked`)** вҖ” port one of those, NOT `ingest-agency-pdfs.ts` (which has no chunked path). For documents too large/visual for reliable LLM extraction (heavy tables/matrices), fall back to **`stubOnly`** (metadata + external PDF link, no LLM body) вҖ” exactly as ADR-S/RID-S are handled. [Source: scripts/ingest-socialstyrelsen-foreskrifter.ts chunked mode; scripts/ingest-afs-regulations.ts callClaudeChunked; scripts/ingest-agency-pdfs.ts:567-615 (`stubOnly` block); lib/agency/agency-pdf-registry.ts `stubOnly`]
7. After upsert, each doc is **chunked + context-prefixed + embedded** via `syncDocumentChunks(doc.id)` (gates on `content_type вҲҲ {SFS_LAW, AGENCY_REGULATION}`; fires context-prefixes (Haiku) + embeddings (OpenAI `text-embedding-3-small`, 1536-d) when `markdown_content` present), and its **`search_vector`** is set (raw SQL; `legal_documents.id` is `text` вҖ” **no `::uuid` cast**). **This tail is NOT in `ingest-agency-pdfs.ts`** вҖ” copy it from the 9.5 ingester, which calls `syncDocumentChunks(doc.id)` then the FTS `UPDATE` immediately after upsert. [Source: scripts/ingest-socialstyrelsen-foreskrifter.ts:346-351; lib/chunks/sync-document-chunks.ts:22,118; scripts/backfill-search-vectors.ts:45; 9.5 AC5]

| Field | Meaning | 9.8 diff signal it feeds |
