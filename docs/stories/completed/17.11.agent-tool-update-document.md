## Story

**As an** AI compliance agent,
**I want** to propose a section-level edit to an existing workspace document (subject to user approval),
**so that** I can help users keep a policy/procedure current without re-drafting the whole document — while never silently mutating an approved document.

## Story Context

**Existing System Integration:**

- **Create is already shipped (14.24).** Out of scope here. This story is **update-only**.
- **Approval pattern (14.22/14.23/14.24/14.28/14.30):** new edits flow through a `PendingAgentAction` (`action_type: UPDATE_DOCUMENT`) → approval-card renderer → `approvePendingAction` dispatch. **No `execute:true` path** (removed per 14.23). The closest in-flight precedent is **14.28 `UPDATE_REQUIREMENT`** — same shape (overwrite-an-existing-field via a diff card), now Done with the renderer + dispatch branch live.
- **Models:** `WorkspaceDocument` / `WorkspaceDocumentVersion` (Epic 17, `WorkspaceDocument*` prefix). New content saved as a new `WorkspaceDocumentVersion`. **Source enum (`WorkspaceDocumentVersionSource`) now includes `TIPTAP`, `IMPORT`, `AGENT`** (`schema.prisma:2007`) — `AGENT` was added but is currently unused in the codebase; **this story uses `source: 'TIPTAP'` to match 14.24's actual dispatch** (which calls `createDocument`, which hardcodes `'TIPTAP'`) — agent authorship is tracked via `created_by` + the activity-log `new_value` stamp (AC 10), not via the enum.
- **Reuse:** `ActionRendererFrame` + `useDebouncedParamsChange` (14.23 "spine & whisper" card chrome, 4 states); `createPendingActionRow` helper; the `approvePendingAction` dispatch switch + its `tasks:edit` gate (SEC-001); the read-only **chat detail panel** canvas seam (`document-draft`-style `ChatDetailItem`, 14.24 AC 11a) for the before/after preview.
- **Content format:** agent emits the replacement section as **Tiptap JSON** (matches `draft_styrdokument`).
- **17.10b indexing contract (NEW since v2.0):** DRAFT and IN_REVIEW workspace documents are now indexed and queryable by the agent (with `[Utkast:]` citations). **`saveDocumentVersion` already calls `indexWorkspaceDocument` via `after()`** (`app/actions/documents.ts:~668`), so an `update_document` dispatch auto-triggers a re-embed — no explicit `markWorkspaceDocumentDirty` call needed in the dispatch branch.

**Out of scope:** creating new documents (14.24's `draft_styrdokument`); whole-document rewrites; markdown conversion; the read/list tools (Story 17.10); the legal `Document`/`DocumentVersion` (SFS) models.

## Acceptance Criteria

### Schema

1. `PendingAgentActionType` extended with **`UPDATE_DOCUMENT`**, appended after `TRANSITION_DOCUMENT_STATUS` (the most recent addition, Story 14.30 — `schema.prisma:1618`). Pattern is **additive append** (`ALTER TYPE ... ADD VALUE`; generate `--create-only` and hand the SQL to the user — never auto-apply, per the project's manual-migration workflow). No new tables.
2. `params` JSON shape for `UPDATE_DOCUMENT`: `{ documentId, sectionHeading, oldSectionContentJson (Tiptap), newSectionContentJson (Tiptap), changeSummary, entity_version: string }`. Two captured-at-propose-time snapshots:
   - **`oldSectionContentJson`** is captured at propose time by extracting the current section's contents via the same `updateSection`/section-locator helper used by the dispatch (AC 9) — locate the heading, capture the body up to the next same-or-higher-level heading. This lets the renderer show the diff without re-fetching the document AND preserves an honest "what the agent was comparing to" view even if the document drifts before approval. **Mirrors 14.28's `oldSnapshot` pattern** (see `components/features/ai-chat/agent-action-renderers/update-requirement-renderer.tsx:37, 56, 73, 88`).
   - **`entity_version: string`** is the document version snapshot (ISO-8601 UTC, from `document.updated_at.toISOString()` — mirrors 14.28's tool at `lib/agent/tools/update-requirement.ts:188`). Populated by the tool, consumed by 14.31's staleness guard at approval time. 17.11 itself does not implement the staleness check; it only ensures the snapshot is captured so 14.31's retrofit picks it up cleanly (14.31's `STALENESS_STRATEGY` enumerates `UPDATE_DOCUMENT` as a guarded type).

### Tool — `update_document`

3. New tool `lib/agent/tools/update-document.ts` (`update_document`). Zod schema: `document_id` (string), `section_heading` (string — heading text to locate), `updated_content` (**Tiptap JSON** for the replacement section), `change_summary` (string). **No `execute` param** — the tool only ever proposes a pending row. Registered with policy `'write'` in `lib/agent/tools/index.ts` (slot in alphabetically; the existing write neighbours are `update_compliance_status`, `update_requirement`, `transition_document_status`).
4. **Guards (at tool time):** target document is in the caller's workspace; status is **DRAFT or IN_REVIEW** (reject APPROVED/SUPERSEDED/ARCHIVED via `wrapToolError` — guide the user to branch a new editable version first via the existing **`createDraftFromApproved`** action (`app/actions/documents.ts:823`)); `section_heading` exists in the document; and `newSectionContentJson` is **not deep-equal** to the resolved current section body (no-op edit rejected at propose time — surfaces a `"Inga ändringar att föreslå"` error via `wrapToolError`, no pending row created). On any failure → `ToolError`, no pending row.
5. On success: create a `PendingAgentAction(UPDATE_DOCUMENT)` via `createPendingActionRow` and return a `WriteToolResponse` whose preview shows the **section heading + a short before/after excerpt**. `wrapWriteToolResponse` is used as-is; its `_meta.sidebarHint` is already `'none'` (post-197B-S1, 2026-05-27) — the inline card is the surface, no detail-sidebar autopops.

### Renderer

6. New renderer `components/features/ai-chat/agent-action-renderers/update-document-renderer.tsx`, **built on `ActionRendererFrame`** (do not hand-roll chrome/states). **Primary template: `update-requirement-renderer.tsx`** (14.28, 225 lines) — closer pattern for "edit an existing entity's field via a diff card" than 14.24's `draft-document-renderer.tsx` (which is a create card). Borrow the chat-detail-panel before/after preview pattern from 14.24. **Diff display reads from `params.oldSectionContentJson` vs `params.newSectionContentJson` — no re-fetch** (AC 2 captures both snapshots at propose time):
   - **PENDING:** document title + target section heading + a compact before→after excerpt (from the two `params` snapshots); "Visa mer" opens the full read-only before/after in the **chat detail panel** (canvas seam, per 14.24 AC 11a) — not a modal. Controls: "Godkänn" / "Avvisa".
   - **APPROVED:** title + link to the document at its new version.
   - **REJECTED / EXPIRED:** standard muted variants.
   - **CP-001 (formatting guardrail):** renderer copy NEVER exposes internal identifiers — no raw `documentId`, no `pendingActionId`, no `section_heading` rendered as a code-style/quoted identifier. Use the document **title** and section **heading text** as natural Swedish language.
7. Register the renderer in **both** exported maps in `components/features/ai-chat/agent-action-card.tsx` — `RENDERERS[UPDATE_DOCUMENT]` (line ~56) and `ACTION_TYPE_LABELS[UPDATE_DOCUMENT]` (line ~39, suggested label: `'Uppdatera dokument'` — mirrors 14.28's `'Ändra kravpunkt'` / 14.30's `'Ändra dokumentstatus'` pattern).

### Approval dispatch

8. `approvePendingAction` (`app/actions/pending-agent-actions.ts`) gains an `UPDATE_DOCUMENT` branch (inherits the existing `tasks:edit` gate). Flow: re-read the current `content_json` → **re-assert the DRAFT/IN_REVIEW status guard** (state may be stale since proposal) → `updateSection()` (AC 9) to produce the **full** updated document → call the existing **`saveDocumentVersion(documentId, contentJson, changeSummary?, title?, contentHtml?)`** (`app/actions/documents.ts:590`), which appends a new `WorkspaceDocumentVersion` (`source: 'TIPTAP'` by enum default), bumps `current_version_id`/`current_version_number`, writes a `document_version_saved` ActivityLog row, **AND triggers `indexWorkspaceDocument` via `after()`** so 17.10b chunks stay current — all in one `$transaction`. ⚠️ **`saveDocumentVersion` takes the WHOLE document `contentJson`, not a section** — pass the full `updateSection` output. It also expects `contentHtml`; generate it from the Tiptap JSON (e.g. `generateHTML(json, extensions)`) so `extracted_text` derives correctly. **Note:** `saveDocumentVersion` does NOT itself enforce the DRAFT/IN_REVIEW guard — the dispatch branch must (AC 11). **Reindex is automatic** — no explicit `markWorkspaceDocumentDirty` call needed. **The downstream `syncWorkspaceChunks` is content-hash-gated** — a repeated update with identical content is a no-op at the chunk level. Tests should assert `indexWorkspaceDocument` was invoked but should NOT assert chunk-level mutation on no-content-change paths (the AC 4 no-op-edit guard already blocks identical re-proposes at tool time; the hash-gate is the belt-and-suspenders backstop).
9. Pure section-replacement utility `lib/documents/update-document-section.ts` → `updateSection(contentJson, sectionHeading, newContentNodes)`: locate the heading (case-insensitive), replace from that heading to the next heading of **same-or-higher level** (preserving the heading, replacing the body), return the full updated doc. Heading-not-found → informative error.
10. `ActivityLog` on update uses `entity_type: 'workspace_document'` (the live convention — **not** `'document'`, which is the SFS legal entity). `saveDocumentVersion` (AC 8) **already writes** a `document_version_saved` row, so an agent update is auditable for free. To distinguish *agent* edits from human ones, stamp agent authorship in the row's `new_value` (e.g. `{ ..., by: 'agent', pendingActionId }`) rather than minting a redundant action label. Only add a distinct `document_updated_by_agent` action label (in `lib/activity/action-constants.ts`) if product wants it surfaced separately in the feed.

### Security / non-regression

11. Workspace scoping and the DRAFT/IN_REVIEW guard are enforced **both** at tool time (AC 4) **and** at dispatch (AC 8) — never trust the proposal-time snapshot. A role lacking `tasks:edit` cannot approve (inherited gate).
12. `draft_styrdokument` and all existing agent tools/renderers are behaviourally unchanged (additive).

### Prompt

13. `lib/agent/system-prompt.md` updated (restart dev server to load): when to **update** an existing document (`update_document`) vs **draft a new one** (`draft_styrdokument`); provide the replacement section as Tiptap JSON; cannot edit Approved documents.
14. `TOOL_CONFIG` (`chat-message.tsx`) gains an `update_document` entry (`label`/`doneLabel`/`icon`, e.g. "Uppdaterar dokument" / "Föreslog ändring" / `FileEdit`). Follow the 14.28/14.29/14.30 precedent — chip labels were caught in their owner smokes (1428-001 + 14.30 process finding) and must include both PENDING + DONE labels.

The DRAFT/IN_REVIEW chunks reindex automatically — `[Utkast:]` citations stay accurate post-edit.
- **`WorkspaceDocumentVersionSource` enum (NEW value flagged for v2.2):** `schema.prisma:2007` (was `:1997` in v2.0). Values: `TIPTAP`, `IMPORT`, **`AGENT`** (the `AGENT` value was added after v2.0 but **is currently unused** by 14.24's actual dispatch — `createDocument` at `documents.ts:62` hardcodes `'TIPTAP'`). For consistency with 14.24, **17.11's dispatch also uses `'TIPTAP'`** (or accepts the column default, which is `TIPTAP`). Agent authorship is tracked via `created_by` + the activity `new_value` stamp (AC 10), mirroring 14.24's choice. If a future story consolidates on `AGENT` for agent-written versions, both 14.24 and 17.11 should flip together — outside this story's scope.
- **`wrapWriteToolResponse` 197B-S1 fix (2026-05-27):** the helper's `_meta.sidebarHint` default is now `'none'` (was `'open'`). 17.11's tool inherits this — no detail-sidebar autopops on the write-tool response envelope; the inline approval card is the surface. (`lib/agent/tools/utils.ts:49-66`.)
- **14.31 staleness coupling:** 14.31 (Approved, blockers cleared 2026-05-28) retrofits an `entity_version` snapshot guard across all overwrite types — `UPDATE_DOCUMENT` is on its guarded list. 17.11 just needs to populate `params.entity_version` (a `WorkspaceDocument.updated_at` snapshot) at propose time per AC 2 — 14.31's dispatch-side check will pick it up cleanly when 14.31 ships.
- **Why approval-gated (not atomic):** document mutations go through `PendingAgentAction` since 14.23; an agent editing a live document must be reviewable. The `execute=false/true` two-phase pattern from the original 17.11 draft is deprecated.
