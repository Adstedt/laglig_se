# Story 8.21: ChangeEvent Deduplication & Unique Constraint

## Status

Done

## Story

**As a** compliance officer reviewing pending law changes,
**I want** each amendment to appear exactly once in the assessment picker,
**so that** I can trust the change list is accurate and don't waste time on duplicate entries.

## Context & Dependencies

### The Problem

Investigation on 2026-03-30 revealed **242 duplicate ChangeEvent groups** in the database (380+ excess rows). For 2026 amendments specifically: 396 total events but only 258 unique — **138 are duplicates**. Some amendments appear up to 5 times (e.g., SFS 2025:1309 × 5).

Users see duplicate entries in the assessment picker dialog (`ChangePicker`), which erodes trust in the system and creates confusion about what needs to be reviewed.

### Root Cause Analysis

**Two distinct duplication patterns were identified:**

#### Pattern A: Repeated text-change detection (144 groups — 59%)

`sync-sfs-updates` (04:30 UTC) detects base law text changes from the Riksdagen API. Each time the text has changed since last sync, it calls `detectChanges()` which creates a new `ChangeEvent`. However, Riksdagen rolls out consolidated text changes incrementally over days/weeks while the API's `undertitel` (which provides `amendment_sfs`) remains the same.

**Result:** Multiple `ChangeEvent` rows with the same `(document_id, amendment_sfs)` but different `detected_at` dates and different diffs. Each represents a real text change, but they all reference the same amendment.

**Example — SFS 2025:1309 (× 5):**

| detected_at | diff_len | Source |
|---|---|---|
| 2025-12-11 | 17,556 | sync-sfs-updates |
| 2025-12-16 | 1,602 | sync-sfs-updates |
| 2026-01-09 | 28,022 | sync-sfs-updates |
| 2026-01-23 | 50,016 | sync-sfs-updates |
| 2026-03-12 | 50,016 | sync-sfs-updates |

#### Pattern B: Cross-cron overlap (98 groups — 41%)

`discover-sfs-amendments` (02:00 UTC) creates a barebones `ChangeEvent` when it discovers and processes a new amendment PDF (no diff, no `previous_version_id`). Later, `sync-sfs-updates` independently detects the same amendment via Riksdagen text changes and creates a rich event (with diff + version archive).

**Example — SFS 2026:146 (× 2):**

| detected_at | Source | Has diff? |
|---|---|---|
| 2026-03-04 | sync-sfs-updates | Yes (50,016 bytes) |
| 2026-03-05 | discover-sfs-amendments | No |

#### Why the existing dedup didn't catch these

A dedup check was added to `detectChanges()` on **2026-03-10** (commit `d668ab5`). All 242 duplicate groups pre-date this fix. The dedup correctly prevents **new** duplicates, but the old ones remain in the database. Additionally, there is no DB-level constraint to serve as a safety net.

### Source of Truth

`discover-sfs-amendments` (svenskforfattningssamling.se) is the **authoritative source** for amendment events. It directly crawls the official publication site and processes the actual amendment PDF. `sync-sfs-updates` provides supplementary diff data by comparing base law text versions — valuable enrichment, but secondary.

The correct model: **one ChangeEvent per `(document_id, amendment_sfs)` pair**. One published amendment = one event for the user to assess. Text-change diffs from `sync-sfs-updates` should enrich the existing event, not create new ones.

## Acceptance Criteria

### Database Cleanup

1. A cleanup script identifies all duplicate `ChangeEvent` groups where `(document_id, amendment_sfs)` has more than one row (amendment_sfs IS NOT NULL) — the script MUST only touch rows where `amendment_sfs IS NOT NULL`; rows with NULL amendment_sfs (NEW_LAW, METADATA_UPDATE, etc.) are never deleted
2. For each duplicate group, exactly one row is kept — the "best" row, selected by: prefer the row with the longest `diff_summary` (richest diff data); if tied, prefer the row with a non-null `previous_version_id`; if still tied, keep the earliest `detected_at`
3. Before deleting any duplicate row, check if it has associated `ChangeAssessment` records (`onDelete: Cascade` on the FK would destroy them). If any assessments reference a row being deleted, reassign their `change_event_id` to the kept row first, then delete. Log any reassignments.
4. All other rows in the group are deleted
5. The cleanup is idempotent — running it twice produces no changes on the second run
6. The cleanup logs: groups found, rows deleted, rows kept, assessments reassigned
7. After cleanup, run the duplicate-finding query again and assert 0 groups remain

### Unique Partial Index

8. A Prisma migration adds a unique partial index on `change_events (document_id, amendment_sfs) WHERE amendment_sfs IS NOT NULL` — this allows multiple events with `amendment_sfs = NULL` (e.g., metadata updates) while enforcing one-per-amendment for named amendments
9. The cleanup script (AC 1-7) MUST be run and verified (0 duplicate groups) BEFORE applying the migration — the migration only contains the `CREATE UNIQUE INDEX`, not the cleanup logic
10. The migration is reversible (down migration drops the index)

### Cron Hardening — sync-sfs-updates

11. `detectChanges()` in `lib/sync/change-detection.ts` already has dedup logic (added 2026-03-10) — verify it correctly finds and enriches an existing event when `(document_id, amendment_sfs)` already exists, rather than creating a new row
12. When the dedup finds an existing event, it updates: `diff_summary` (replace with latest diff), `previous_version_id` (update to latest archived version), `changed_sections` (replace if non-empty) — the existing enrichment logic already does this, verify no regression
13. When `amendmentSfs` is `null` (unknown amendment), `detectChanges()` still creates a new event — this is correct behavior, no change needed

### Cron Hardening — discover-sfs-amendments

14. The ChangeEvent creation in `discover-sfs-amendments` already has a dedup check — verify it correctly skips creation when a matching event exists from `sync-sfs-updates`
15. When `discover-sfs-amendments` finds an existing event (created by sync-sfs-updates), it should NOT skip silently — instead, it should update the existing event's `notification_sent` to `false` if it was `true`, ensuring the notification pipeline re-evaluates it (defensive: ensures discover's authoritative detection isn't lost)

### Backfill Phase Hardening

16. Phase 3 of `discover-sfs-amendments` (backfill missing ChangeEvents) runs OUTSIDE a transaction — this is safe now that the unique index exists (INSERT will fail on conflict), but should be updated to wrap the `create` in a try/catch that catches the unique constraint violation and logs it (Prisma `upsert` doesn't work with partial unique indexes — use `$queryRaw` with `ON CONFLICT DO NOTHING` or catch `PrismaClientKnownRequestError` with code `P2002`)

### Assessment Picker Display

17. The `ChangePicker` component displays the `listName` for each entry so that users can distinguish entries when the same document legitimately appears in multiple law lists (the query returns one row per ChangeEvent × LawList, which is correct — a change must be assessed per list). Note: `ChangeRow` in `changes-tab.tsx` already shows `listName` — only the picker needs this fix.

### Regression Safety

18. All existing tests pass (`vitest run`)
19. `pnpm build` passes with zero errors
20. `npx tsc --noEmit` passes
21. The `getUnacknowledgedChanges()` query continues to return correct results (no duplicate rows for same `(document_id, amendment_sfs)` within a single law list)

### Key Technical Details

**Partial unique index (PostgreSQL):**
```sql
CREATE UNIQUE INDEX change_events_document_amendment_unique
ON change_events (document_id, amendment_sfs)
WHERE amendment_sfs IS NOT NULL;
```

Prisma doesn't support partial unique indexes in the schema — must use raw SQL in migration. The `@@unique` directive doesn't support `WHERE` clauses. The index allows:
- Multiple events with `amendment_sfs = NULL` for the same document (metadata changes, unknown amendments)
- Exactly one event per `(document_id, amendment_sfs)` for named amendments

**Cleanup row selection logic:**
```sql
-- CRITICAL: WHERE amendment_sfs IS NOT NULL on the outer DELETE
-- Without it, ALL rows with NULL amendment_sfs would be deleted
-- (NEW_LAW, METADATA_UPDATE events would be destroyed)
DELETE FROM change_events
WHERE amendment_sfs IS NOT NULL
  AND id NOT IN (
    SELECT DISTINCT ON (document_id, amendment_sfs) id
    FROM change_events
    WHERE amendment_sfs IS NOT NULL
    ORDER BY document_id, amendment_sfs,
      LENGTH(COALESCE(diff_summary, '')) DESC,
      (previous_version_id IS NOT NULL) DESC,
      detected_at ASC
  );
```

**Before running the DELETE, reassign orphaned assessments:**
```sql
-- Find assessments that reference duplicate rows being deleted
-- and reassign them to the kept row
UPDATE change_assessments ca
SET change_event_id = keeper.id
FROM change_events ce
JOIN (
  SELECT DISTINCT ON (document_id, amendment_sfs) id, document_id, amendment_sfs
  FROM change_events
  WHERE amendment_sfs IS NOT NULL
  ORDER BY document_id, amendment_sfs,
    LENGTH(COALESCE(diff_summary, '')) DESC,
    (previous_version_id IS NOT NULL) DESC,
    detected_at ASC
) keeper ON keeper.document_id = ce.document_id AND keeper.amendment_sfs = ce.amendment_sfs
WHERE ca.change_event_id = ce.id
  AND ce.id != keeper.id;
```

**Existing dedup in detectChanges() (lib/sync/change-detection.ts:216-251):**
The dedup checks `findFirst({ where: { document_id, amendment_sfs } })`. If found, it updates the existing event with new diff/version data and returns it. This is correct behavior — the unique index serves as a safety net if the application-level dedup ever fails.

### Database Impact

- **Current state:** 242 duplicate groups, ~380+ excess rows out of ~2,800+ total change_events
- **After cleanup:** ~380 rows deleted, unique index prevents future duplicates
- **CASCADING DELETE WARNING:** `ChangeAssessment` has `onDelete: Cascade` on `change_event_id` (`schema.prisma:1486`). Deleting a ChangeEvent will silently destroy any associated assessments. The cleanup script MUST reassign assessment FKs to the kept row BEFORE deleting duplicates. This is unlikely to affect many rows (assessment is a recent feature) but must be handled defensively.
