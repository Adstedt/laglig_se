## Dev Notes

### The Two-Version Problem (critical design constraint)

A naive `createDraftFromApproved` â†’ `saveDocumentVersion` chain creates **TWO** version rows for ONE agent edit:
- Step 1: `createDraftFromApproved` creates `v(N+1)` with content = approved `v(N)` (clone). Sets `current_draft_version_id = v(N+1)`.
- Step 2: `saveDocumentVersion(documentId, editedContent, ...)` sees `current_draft_version_id != null` â†’ routes to Path A â†’ creates `v(N+2)` with the edit.

The user would see the version history as `v3 â†’ v4 (clone) â†’ v5 (edit)` for what they perceive as ONE action. This is wrong on UX grounds (confusing version history) AND on data grounds (every reindex of v4 burns embedding cost on identical content).

**Fix:** a new server action `createDraftFromApprovedWithEdit` that writes ONE version row containing the EDITED content directly. AC 1 specifies the shape. The action mirrors `createDraftFromApproved`'s `$transaction` body (alias-freeze, draft pointer set, `document_draft_created` log row) but with `content_json = editedContentJson` instead of cloning + adds the `document_version_saved` log row that `saveDocumentVersion` writes.

### Why not refactor `saveDocumentVersion` Path C instead?

Path C's refusal (`saveDocumentVersion` line 753-762) is **load-bearing for the editor autosave path** â€” without it, an editor autosave race against an APPROVED doc would silently mutate it. 17.11c's branch+write is a *semantically different* operation (it intentionally creates a new draft + writes), so it belongs in a sibling action rather than a Path D inside `saveDocumentVersion`. Keeps the autosave guard tight.

### Why `createDraftFromApprovedWithEdit` instead of inlining in dispatch

Two reasons:
1. **Testability.** The transaction body is non-trivial (twin ActivityLog rows + alias-freeze invariant). A dedicated server action is unit-testable in isolation with mocked Prisma â€” the dispatch tests then assert "calls createDraftFromApprovedWithEdit with right args" rather than re-asserting the whole transaction.
2. **Future reuse.** When the canvas streaming-edit feature lands (future story), it'll need exactly this server-action shape too. Inlining the logic in dispatch would force that future work to either extract or re-implement.

## Risk and Compatibility Check

**Primary Risk:** the new server action's twin-ActivityLog write could diverge from `createDraftFromApproved`'s single-row pattern in a way that confuses audit consumers (a `document_draft_created` row + a same-timestamp `document_version_saved` row with the same `version_number` â€” is that one event or two?).

**Mitigation:**
- AC 14 stamps the `document_draft_created` row with `by:'agent'` AND `operation: 'auto_branch_then_<...>'` â€” gives consumers an explicit signal that this row pairs with a same-version `document_version_saved`.
- Test the audit-stream shape against the existing 17.16 ActivityLog tests as part of Task 1.
- Document the new pairing pattern in the server-action JSDoc.

**Rollback:**
- Pure code revert. No schema, no migration, no enum value to retire.
- Pending rows created post-17.11c that carry `creates_draft: true` would gracefully default to `creates_draft: false` after revert (the new dispatch routing fork wouldn't fire) â€” but those rows would then refuse at dispatch because the writeable predicate has reverted too.
