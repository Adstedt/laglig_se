## Dev Notes

**âš  Cross-cutting Epic 24 conventions (ratified 2026-05-07 per Story 24.1 QA gate `docs/qa/gates/24.1-import-schema-and-server-action-skeleton.yml`):**

1. **Activity-log action keys use dot notation as a deliberate Epic 24 namespace.** Examples already in this story's AC 113-116: `catalog_request.fulfilled`, `catalog_request.rejected`. The dot is a semantic separator; do NOT rewrite to snake_case.
2. **RLS is ENABLE-only â€” no `CREATE POLICY` blocks.** Important for this story: the original AC 7 of Story 24.1 mentioned an "admin override" CREATE POLICY for cross-workspace access on `CatalogIngestRequest`. **That policy is NOT needed and should NOT be added.** Admin actions in this story already get cross-workspace access automatically because Prisma connects as the postgres role (BYPASSRLS) â€” admin queries via `prisma.catalogIngestRequest.findMany({})` (no workspace filter) will return rows from all workspaces by default. The `getAdminSession()` check at the top of each server action is the only authorization gate. Workspace isolation for non-admin paths is enforced by `withWorkspace` + workspace_id filters in user-facing actions, not by RLS.
3. **Primary keys use `@default(uuid())` not `@default(cuid())`.** Not directly applicable (no new models in this story).

These three points were spec-text vs project-pattern divergences in the original Epic 24 stories; PO ratified the project-pattern interpretation. They are NOT to be re-litigated in this story's QA review.

**Key decisions baked in:**
- **Sort by age ascending** â€” surface SLA-breach risks first. Within same age bucket, stable sort by import_id avoids visual jumping when new rows arrive.
- **No real-time push** â€” ops checks the queue periodically. SWR pip refresh = 60s. Consider Pusher / Realtime if the volume warrants; for now polling is fine.
- **Fulfilment requires legal_document_id** â€” ops must have actually ingested the document first. The catalog-requests queue is downstream of ingestion, not coupled to it. Document IDs are public-ish in admin views.
- **Rejection doesn't auto-flip the import row** â€” keeps user agency intact. They might still want to manually accept an ambiguous match instead of letting the rejection stand.
- **Idempotent fulfilment** â€” re-fulfilling the same request is a no-op + no double-email. Critical for ops who might double-click.
- **Activity log scope** â€” workspace activity log surfaces the fulfilment/rejection on the originating workspace's timeline, not just admin-side. So users see "KatalogtillĂ¤gg hanterat" alongside their own actions.

**Out of scope explicitly:**
- Real-time push (queue uses SWR polling)
- Bulk-fulfilment (one at a time for now)
- Auto-fulfilment heuristics (no "match by SFS-number to existing doc" automation)
- Cross-workspace duplicate detection (if two workspaces request the same source row, show two requests; ops can reject the duplicate)
- Re-ingestion workflow (if a fulfilled doc later gets revoked, no auto-cascading; manual cleanup)
