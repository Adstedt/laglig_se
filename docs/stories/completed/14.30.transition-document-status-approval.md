## Story

**As a** user managing a styrdokument's lifecycle,
**I want** the agent to propose a status transition (submit for review, supersede, archive) as a reviewable card — but never to mark a document approved itself,
**so that** the agent can move documents along the workflow while the approval decision stays a deliberate, accountable human act.

## Context

Styrdokument move through a lifecycle ladder: `DRAFT → IN_REVIEW → APPROVED → SUPERSEDED → ARCHIVED` (Story 17.4). The agent will often be the one that knows a document is ready to advance ("det här utkastet är klart för granskning") or that an approved policy is now obsolete ("den här policyn har ersatts — arkivera den"). Letting it *propose* the transition closes the loop after a draft (14.24) or an assessment.

The hard rule: **the agent cannot move a document to `APPROVED`.** Approval is the point where a human takes responsibility for the document's content — separation of duties. The guard is enforced **server-side in the dispatch** (authoritative), with the tool refusing it up front as a first line of defence. Every other transition on the ladder is proposable, subject to the existing `VALID_STATUS_TRANSITIONS` validity check.

6. On `execute: false` (valid, non-APPROVED transition): capture `oldStatus` + the document's `updated_at` (stored as `params.entity_version` per AC 2), create the `PendingAgentAction` row, return the envelope with `data: { pendingActionId }`.
7. Tool description (Swedish): use to submit a draft for review (`DRAFT → IN_REVIEW`), send a document back to draft, supersede or archive a document. Explicitly: **never** propose `APPROVED` — approval is a manual human step in the styrdokument UI.
8. `execute: true` is **not** supported (returns a `ToolError`).
