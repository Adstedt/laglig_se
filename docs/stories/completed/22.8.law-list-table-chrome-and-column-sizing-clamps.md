# Story 22.8: Law list table chrome alignment + column-sizing clamps

## Status

Done

## Story

**As a** user resizing columns on the law list (`/laglistor`, both flat and grouped views, both `viewMode='table'` and `viewMode='compliance'`),
**I want** the chrome columns (row select, drag handle, type icon) to hold their declared widths regardless of how I resize the content columns, AND I want resizable columns to respect their declared `minSize`/`maxSize` bounds (no "drag Dokument to infinite width"),
**so that** the law list feels exactly like the Tasks Lista table — same chrome geometry, same resize ergonomics — and persisted column widths can never fall outside their bounds.

## Background

After Story 22.7 brought Tasks Lista to a polished state, a side-by-side comparison with the law list (`DocumentListTable`) showed three problems:
1. Chrome columns squashed proportionally when other columns grew (table-fixed + `w-full` redistribution)
2. The Dokument (title) column had no `minSize`/`maxSize` — could be dragged to arbitrary width
3. Stale `columnSizing` in localStorage from before bounds existed rendered columns out of bounds, even after declared bounds were added (TanStack `header.getSize()` doesn't auto-clamp on read)

The compliance view (`viewMode='compliance'`) renders a different table (`ComplianceDetailTable`) which had the same issues. This story brings BOTH to 1:1 parity with Tasks Lista.

## Acceptance Criteria

### Functional — chrome (mirror Tasks Lista)

1. **Chrome columns bumped:** select 40 → 56, dragHandle 40 → 56, type/contentType 60 → 72. All three set `enableResizing: false` and `size === minSize === maxSize` (locked).

2. **Type column pinned.** `'type'` (DocumentListTable) and `'contentType'` (ComplianceDetailTable) added to `PINNED_COLUMN_IDS` so they render as plain `<TableHead>`, not `<DraggableColumnHeader>`. No drag-reorder on the icon column.

3. **Icons centered in chrome cells.** Type-icon cell wraps the colored icon box in `flex items-center justify-center`. DragHandle cell uses `flex w-full items-center justify-center` on the inner button. Wider chrome columns now visibly "breathe" around their icons (16-20px each side).

4. **Header padding overrides:**
   - select: `pl-6 pr-2` (24px / 8px) — gives the row checkbox visual breathing room from the table's left edge
   - dragHandle: `px-2`
   - type/contentType: `pl-5 pr-2` — places the "Typ" header label at exactly the same x-coordinate as the icon's left edge below it

### Functional — sizing (clamp on commit AND read)

5. **`minSize`/`maxSize` declared on every resizable column.** Title (Dokument) bound to `[150, 600]` — matches Tasks Lista's title bounds. Other columns' size values normalized so `size >= minSize` (was logically broken in places where `size < minSize`).

6. **`getColumnWidth` clamps on read** (both files). Non-resize branch returns `Math.max(minSize, Math.min(maxSize, defaultSize))`. Catches stale localStorage AND TanStack's onEnd commit that doesn't always honor bounds.

7. **`liveTotalWidth` + spacer cell pattern** (both files). Sum of all visible column widths (live-aware during drag); applied as `<Table style={{ minWidth: liveTotalWidth }}>`. A final `<TableHead aria-hidden="true" className="p-0" />` and matching `<TableCell aria-hidden="true" className="p-0" />` per row absorb leftover horizontal space — chrome columns no longer inflate proportionally under `table-fixed` when column-sum < container width.

8. **`COLUMN_SIZE_BOUNDS` + `clampColumnSizing`** (both files). Module-level hardcoded width-bounds map keyed by column ID; `clampColumnSizing(sizing)` clamps every column to its bounds. Wired into `handleColumnSizingChange` BEFORE persisting — localStorage physically cannot hold an out-of-bounds width.

9. **Empty-state colSpan increment.** In-table empty fallbacks use `colSpan={columns.length + 1}` to also span the spacer column.

10. **Sticky title column preserved.** Existing `header.id === 'title' && 'sticky left-0 bg-background z-10'` behavior unchanged — chrome paddings are additive.

### Functional — store migration

11. **`document-list-store` v3 → v4 migration.** Resets BOTH `columnSizing` and `complianceColumnSizing` to `{}` on next hydration. Existing users with persisted out-of-bounds widths (from before bounds existed) get a clean slate.

### Functional — group view alignment

12. **`group-table-section.tsx` outer wrapper drops horizontal padding.** Was `<div className="px-3 sm:px-4 pb-3 sm:pb-4">`; becomes `<div className="pb-3 sm:pb-4">`. The 12-16px of horizontal padding around each group's table was offsetting the chrome strip ~16px to the right of where it sits in the flat law list and Tasks Lista. Removing it brings grouped chrome to 1:1 absolute geometry parity.

## Dev Notes

### Why hardcoded `COLUMN_SIZE_BOUNDS` vs reading from columns array

The `handleColumnSizingChange` callback is defined BEFORE the `columns` useMemo in the render order (other handlers depend on the column-sizing state, so the state setup has to come first). A closure over `columns` would read it at call-time (post-init) but accessing it in the `useCallback` deps array would hit TDZ. Cleanest fix: hardcode bounds at module level. They mirror what's declared on each `columnDef` — duplication accepted because (a) it's the source of truth for the persistence layer, (b) deviation from columnDef bounds would be caught immediately by visual smoke.

### Why clamp on commit AND on read

Belt-and-suspenders. TanStack v8 in `'onEnd'` resize mode commits `startSize + deltaOffset` to `columnSizing` — empirically not always clamped to columnDef bounds (depends on which TanStack release). Plus we have a population of users with `columnSizing` already persisted in localStorage from before bounds were declared. Clamp-on-commit handles new resizes; clamp-on-read handles stale state. Either alone leaves a gap.

### Why the spacer cell

`table-fixed` with `width: 100%` (the shadcn `Table` primitive's default `w-full`) distributes any excess width PROPORTIONALLY across all columns when column-sum < container. That means tiny chrome columns (40px) inflate to ~60px when the table has unused horizontal space — unacceptable for "rock-steady chrome."

Adding a column WITHOUT explicit width (the spacer) at the end means the browser puts ALL the slack there. Chrome columns hold their declared widths. The spacer's own width adjusts as the user resizes other columns.

The `<table style={{ minWidth: liveTotalWidth }}>` keeps the table at LEAST as wide as the column-sum (so it overflows the container and triggers `overflow-x-auto` when needed) while letting `w-full` handle the "table at least fills container" case.
