# Story 26.3: Catalog-Link Primitive (`<CatalogLawList>` Live Data Path)

## Status

Done

## Story

**As a** reader of a marketing page (e.g. `/branscher/bygg`),
**I want** the "laws that apply" list to link to real, verified pages in Laglig's public catalog (`/lagar/*`, `/foreskrifter/*`, `/eu/*`) with the catalog's own titles,
**so that** marketing pages demonstrate the product's actual depth (the unique-defensible move: no competitor can link into a 47k-document catalog) and dead links never ship.

## Acceptance Criteria

### Shared URL mapping

1. The ContentType → public-URL mapping currently private to `scripts/generate-sitemaps.ts` (`getUrlPath`) is **extracted** to `lib/catalog/public-url.ts` as `getPublicUrlPath(contentType, slug): string | null`, keeping the `_exhaustive: never` arm. The sitemap script imports it (single source of truth — resolver and sitemap can never drift).
2. **Epic correction encoded:** the mapping covers ALL public kinds — `SFS_LAW` → `/lagar/[slug]`, `SFS_AMENDMENT` → `/lagar/andringar/[slug]`, **`AGENCY_REGULATION` → `/foreskrifter/[slug]`** (the epic's AC text only named `/lagar` + `/eu`, but AFS/MSBFS links on industry pages are agency regulations), `EU_REGULATION` → `/eu/forordningar/[slug]`, `EU_DIRECTIVE` → `/eu/direktiv/[slug]`. Court-case types map to `null` (no public page) → resolver treats as unmatched.

### Resolver

3. `lib/marketing/catalog-link.ts` exports `resolveCatalogLinks(entries: CatalogLawEntry[]): Promise<ResolvedCatalogLink[]>` where `ResolvedCatalogLink = { title: string; href: string | null; status: 'matched' | 'unmatched'; contentType?: ContentType }`. Input type is the existing `catalogLawEntrySchema` shape from 26.1 (`{ documentNumber?, slug?, title?, anchor? }`) — frontmatter written in 26.1's format works unchanged.
4. Resolution is **one batched Prisma query per call** (no N+1): single `findMany` over `LegalDocument` with `OR` on the collected `document_number` and `slug` values, `status: 'ACTIVE'`, `select: { document_number, slug, title, content_type }`, via the `@/lib/prisma` singleton. Matching precedence per entry: `documentNumber` exact match first, then `slug`.
5. Matched entries: `href` from `getPublicUrlPath` (+ `#anchor` appended when the entry has one); `title` = **DB title** (catalog is the authority), falling back to the frontmatter `title` only if the DB title is empty. Unmatched entries (no DB row, or content type with no public URL): `href: null`, `status: 'unmatched'`, `title` = frontmatter title (or the raw documentNumber/slug as last resort).
6. Every unmatched entry logs `console.warn('[CATALOG_LINK_UNMATCHED] …')` naming the entry and the page context when provided (optional `context?: string` param) — editorial's signal to fix the MDX.

### Component

7. `<CatalogLawList>` (26.1 stub) swaps to the live path: async Server Component that calls `resolveCatalogLinks` internally. **Prop contract unchanged** (`entries`, optional `heading`) — templates and MDX from 26.1 need zero edits. Matched entries render as links (next/link) with the existing styled-list treatment; unmatched render as plain muted text (no href), exactly like the stub's fallback.
8. New optional `groupBy?: 'content_type'` prop: groups matched entries under Swedish headings — `Lagar & förordningar` (SFS_LAW + SFS_AMENDMENT), `Föreskrifter` (AGENCY_REGULATION), `EU-regler` (EU_REGULATION + EU_DIRECTIVE) — unmatched entries last under no heading. Ungrouped remains the default.

### Build-time dead-link warnings

9. `scripts/generate-sitemaps.ts` extended: after sitemap generation it walks every published marketing MDX, parses `relatedCatalogLaws` frontmatter (gray-matter — already used by the loader), resolves all entries against the DB, and writes `.next/marketing-link-warnings.txt` (one line per unmatched entry: file + entry identifier) plus a console summary (`[generate-sitemaps] N unmatched catalog links — see .next/marketing-link-warnings.txt`). Zero unmatched → file still written, empty (CI can always cat it). **Warnings, not errors** — an unmatched link must not fail the deploy (the page renders plain text gracefully).

### Quality gates

10. Unit tests (mocked prisma): match by documentNumber, match by slug, documentNumber precedence over slug, anchor appended, DB-title precedence, unmatched (no row), unmatched (court-case content type → null URL), single-query batching (prisma mock called exactly once for n entries), empty-entries short-circuit (no query).
11. Component test: matched entries render hrefs from the resolver, unmatched render as non-link text, `groupBy="content_type"` renders the three Swedish group headings in order (mock `resolveCatalogLinks`).
12. `getPublicUrlPath` unit test: every public ContentType maps to its route shape; court-case types return null (pin the mapping the sitemap also relies on).
13. No regression: sitemap output unchanged for catalog URLs (the extraction is move-only), 26.1 marketing tests still green, `pnpm lint && pnpm typecheck && pnpm test` green, `pnpm tsx scripts/generate-sitemaps.ts` runs end-to-end locally (warnings file produced).

## Dev Notes

### What 26.1 already fixed in place (do not re-design)

- `catalogLawEntrySchema` (`lib/marketing/frontmatter-schemas.ts`): `{ documentNumber?, slug?, title?, anchor? }` with a refine requiring documentNumber or slug. **This story must consume exactly this type** — `CatalogLawEntry` is exported.
- `<CatalogLawList>` stub (`components/marketing/sections/catalog-law-list.tsx`): styled `<ul>` card list, slug-bearing entries get best-effort `/lagar/${slug}` links, others plain text. Its doc comment explicitly promises "Story 26.3 replaces the inner resolution … the prop contract below is already 26.3's contract." Keep heading default `'Lagar och föreskrifter som berör området'`.
- Templates pass `entries={fm.relatedCatalogLaws}` (`components/marketing/templates/base-page-template.tsx`) — untouched by this story.

### Database facts (verified in prisma/schema.prisma, lines ~514–568)

- Model `LegalDocument`: `document_number String @unique` (format `"SFS 1977:1160"`, also `"NJA 2023 s. 45"` for court cases), `title String`, `slug String @unique`, `content_type ContentType`, `status DocumentStatus @default(ACTIVE)`. `@@index([document_number])` exists — the OR query is index-backed on both arms.
- Prisma singleton: `import { prisma } from '@/lib/prisma'` (NOT `lib/db/prisma` — the architecture doc's tree is stale here). `scripts/generate-sitemaps.ts` instantiates its own `PrismaClient` — keep that pattern inside the script; the resolver uses the singleton.
- Query only ACTIVE documents — repealed laws shouldn't be marketed. [Source: scripts/generate-sitemaps.ts — same filter]

### URL mapping (single source of truth after Task 1)

From `scripts/generate-sitemaps.ts` `getUrlPath` (verbatim move): SFS_LAW → `/lagar/`, SFS_AMENDMENT → `/lagar/andringar/`, AGENCY_REGULATION → `/foreskrifter/`, EU_REGULATION → `/eu/forordningar/`, EU_DIRECTIVE → `/eu/direktiv/`, six COURT_CASE_* types → null, `_exhaustive: never` default arm (a future ContentType addition becomes a TS error, not a silent 404). **Do not reimplement — move.**

### Why batching matters (AC 4)

Industry pages carry 10–20 catalog links (epic 26.4: "links to ~15 AFS + PBL + miljöbalk laws"). Pages are SSG'd, so resolution cost is build-time — but 24 pages × 15 sequential queries is still a slow build and a pool-contention risk (this repo has seen Supabase pooler flakiness under parallel build workers — see 26.1 Debug Log). One `findMany` with `OR: [{ document_number: { in: [...] } }, { slug: { in: [...] } }]` per page is the shape.

### Rendering notes

- Component stays a Server Component (async) — Prisma in RSC at build time is the established pattern (`app/(public)/lagar/[id]/page.tsx` does it at request/ISR time).
- Use `next/link` for matched hrefs (the stub used `<a>` — upgrade while you're in there; 26.2 set the precedent in `FeatureLink`).
- Unmatched styling: keep the stub's muted plain-text row so a half-fixed page degrades invisibly to readers.
- Group headings (AC 8) use the section-label idiom: Safiro `font-medium`, small, muted — see `SubLabel` in `components/features/landing-v3/section-label.tsx` for the exact treatment.

### Build-warnings design (AC 9)

The script already: connects prisma, walks `content/marketing/` (26.1's `getMarketingUrls`), and has `MARKETING_KINDS`. Add a step after sitemap writes: for each published MDX, `matter(readFileSync(...)).data.relatedCatalogLaws ?? []` → collect → resolve with the script's own prisma client (one batched query for ALL pages' entries combined is fine — a few hundred max) → diff matched/unmatched → write warnings file. Do NOT import `lib/marketing/catalog-link.ts` into the script (it binds the `@/lib/prisma` singleton with Next-specific env loading); share only the pure URL mapper. Frontmatter here doesn't need Zod re-validation — `next build` already validated it minutes earlier in the same pipeline.
