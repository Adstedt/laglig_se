# Story 22.6: `<DatePicker>` primitive + native `<input type="date">` sweep

## Status

Done

## Story

**As a** user filling out forms across the workspace (cycle wizard, search filters, admin error log, catalogue filters),
**I want** every date input to render as the same Popover + Calendar pattern used in Tasks and Document modals,
**so that** I'm never surprised by the OS-default date picker chrome (different on every browser/OS) appearing in the middle of a Swedish-language workspace.

## Background

Five surfaces still rendered native `<input type="date">` controls — visually inconsistent with the rest of the app (which already used Popover + shadcn `Calendar` + outline Button trigger in ~10 places via inline duplication). This story extracts a canonical primitive and migrates every native-input holdout.

## Acceptance Criteria

### Functional

1. **`<DatePicker>` primitive created** at `components/ui/date-picker.tsx`. Props:
   - `value: Date | null`
   - `onChange: (date: Date | null) => void`
   - `placeholder?: string` (default `'Välj datum'`)
   - `id?`, `disabled?`, `invalid?`, `ariaDescribedBy?`, `clearable?` (default `true`), `align?: 'start' | 'center' | 'end'` (default `'start'`), `className?`
   - Uses Popover + shadcn `<Calendar mode="single">` + outline `<Button>` trigger
   - Swedish output formatting: `format(value, 'd MMMM yyyy', { locale: sv })`
   - When `clearable && value`, popover footer renders a `<Button variant="ghost" size="sm">Rensa datum</Button>`

2. **ISO ↔ Date helpers co-located** on the primitive (exported alongside `<DatePicker>`):
   - `parseISODate(s: string | null | undefined): Date | null` — accepts `null` so `error-filters.tsx`'s `string | null` shape works without casting
   - `toISODate(d: Date | null): string`

3. **Cycle creation wizard** (`components/features/compliance-audit/cycle-creation-wizard/CycleMetadataStep.tsx`) — three native date fields (Startdatum, Slutdatum, Brytdatum) replaced with `<DatePicker>`. All three set `clearable={false}` (required fields per `canProceed` check).

4. **Search results filter** (`components/features/search/search-filters.tsx`) — Från / Till date-range fields. State stays `string | undefined` (cleared = `undefined`); `onChange` converts `Date | null` → `undefined` when cleared.

5. **Catalogue filters** (`components/features/catalogue/catalogue-filters.tsx`) — same Från/Till pattern, identical state shape (`string | undefined`).

6. **Admin error filters** (`components/admin/error-filters.tsx`) — Från/Till. State is `string | null` (cleared = `null`); `parseISODate` accepts null directly. **Bonus cleanup:** raw `<input type="date">` with hand-rolled `h-9 rounded-md border bg-background px-3 text-sm` classes net-deleted in favour of the primitive.

### Non-functional

7. **`exactOptionalPropertyTypes: true`** compatibility. All optional props on `DatePickerProps` use `T | undefined` form so `<DatePicker ariaDescribedBy={cond ? 'id' : undefined}>` works without TS error.

8. **Trade-off acknowledged:** native `<input type="date">` allowed typed entry (`2026-05-01`); the popover-only `<DatePicker>` does not. This matches every other date input already in the app (task-creation-form, due-date-editor, etc.). If a power-user typed-entry need surfaces later, it's a primitive-only enhancement (add an inner masked `<Input>` next to the calendar trigger).
