# Story 21.4: Cycle creation wizard + materialisation

## Status

Done

## Story

**As a** KMA-samordnare,
**I want** a guided three-step wizard that captures cycle metadata + scope and then materialises the item set,
**so that** a new compliance audit cycle transitions from `PLANERAD` to `PAGAENDE` with a frozen set of `ComplianceAuditItem` rows ready for bedömning by responsible users (Story 21.5 onward).

## Acceptance Criteria

### Functional

1. **Route and auth gate.** A new Next.js 16 App Router page at `app/(workspace)/laglistor/kontroller/skapa/page.tsx` renders the wizard. The page is a Server Component that gates access via the existing `withWorkspace(..., 'tasks:edit')` pattern (mirrors `app/(workspace)/laglistor/page.tsx`); unauthorised users are redirected to `/laglistor`. The server-component shell hydrates the client-side wizard with a server-fetched list of candidate laglistor (via `getDocumentLists`, AC 2) and workspace members (via `getWorkspaceMembers`, AC 2) as props — no client-side secondary fetch for these lookup lists.

2. **Step 1 — Metadata capture.** Renders a form with the following fields, all validated on blur + on `Nästa` click:
   - `name: string` — required, 1–200 chars, matches `CreateCycleSchema.name` bounds from Story 21.2 (`app/actions/compliance-audit-cycle.ts:98`). Placeholder: `'T.ex. Årlig miljörevision 2026'`.
   - `lawListId: string (uuid)` — single-select dropdown populated from `getDocumentLists()` (`app/actions/document-list.ts:163`), filtered server-side to the active workspace. Required. Ordering: default-list first, then oldest-created first (matches the action's existing `orderBy`). If the workspace has zero law lists, the wizard renders a blocking empty-state ("Skapa först en laglista för att kunna skapa en kontroll") with a link to `/laglistor`.
   - `auditType: 'INTERN' | 'EXTERN'` — segmented toggle control using the already-installed `components/ui/toggle-group.tsx` (shadcn ToggleGroup) in `type="single"` mode. Default `INTERN`. Swedish labels: `Intern revision` / `Extern revision`. **Do NOT** add the shadcn `radio-group` primitive — it is not currently installed and ToggleGroup gives a cleaner segmented-control UX for this binary mutually-exclusive choice (no new dependencies, matches the existing stack).
   - `scheduledStart: Date` — required. Use existing shadcn `Input type="date"` — no new date-picker library.
   - `scheduledEnd: Date` — required. Validation: `scheduledEnd >= scheduledStart` on blur; on violation, show inline Swedish error `'Slutdatum måste vara lika med eller efter startdatum'` (matches the exact string from `app/actions/compliance-audit-cycle.ts:107`).
   - `lawChangeCutoffDate: Date` — required. Helper copy below the field (Swedish): `'Lagändringar publicerade efter detta datum ingår inte i kontrollens omfattning.'`
   - `leadAuditorUserId: string (uuid)` — single-select dropdown populated from `getWorkspaceMembers()` (`app/actions/document-list.ts:1100`). Required. Display format: `{name || email}` with avatar — use the installed shadcn `Avatar` primitive at `components/ui/avatar.tsx` following the exact pattern in `components/features/settings/team-tab.tsx` (`<Avatar>` + `<AvatarImage src={avatarUrl} />` + `<AvatarFallback>{initials}</AvatarFallback>`). **Include AUDITOR-role members in the dropdown** — consistent with Story 21.14's runtime `lead_auditor_user_id` override design (AUDITORs can lead audits via the runtime override even without default `tasks:edit`; filtering them out in the UI pre-empts that pattern and would create drift between the allowed-set in the UI and what `createCycle` will actually accept). `getWorkspaceMembers()` already returns all members regardless of role — do not add a role filter.
   All fields map 1:1 to `createCycle`'s input shape; no transformation needed between step-1 state and the `createCycle` call in step 3.

3. **Step 2 — Scope selection.** Renders `<ScopeSelector>` from Story 21.3 (`components/features/compliance-audit/scope-selector/ScopeSelector.tsx`), passing the Step-1 `lawListId` as its `listId` prop. The wizard is the `onChange` consumer and stores the emitted `ScopeDefinition` in the step-2 state slot. `Nästa` is disabled until the emitted `ScopeDefinition` is non-null (i.e., at least one item is selected). If the user navigates back to Step 1 and changes `lawListId`, Step 2 state is cleared (the previously-emitted `ScopeDefinition` becomes invalid for the new list). Because `ScopeSelector` is controlled via `onChange` only (per Story 21.3 AC 1), re-entering Step 2 after the list change re-mounts the component and shows an empty selection.

4. **Step 3 — Confirm.** Renders a read-only summary:
   - **Omfattning:** `{listName} — {scope summary}` where `{scope summary}` follows the same copy rules as `ScopeSelector`'s live summary (AC 7 of 21.3) — `'Alla N lagar'`, `'{N} lagar i {M} grupper'`, or `'{N} valda lagar'`. Reuse the `formatScopeSummary` helper exported from `ScopeSelector`'s `__private` bundle to stay DRY (see Dev Notes "DRY scope-summary helper").
   - **Metadata:** all Step-1 fields displayed in a small 2-column key/value grid with Swedish labels.
   - **Primary action:** a `<Button>` labelled `'Skapa kontroll'`. Clicking triggers the submit flow (AC 5 below). While in-flight, the button is in a `disabled` + spinner state; all `Tillbaka` buttons are also disabled so the user cannot navigate away mid-submit.
   - **Secondary action:** `Tillbaka` button returns to Step 2.

5. **Submit flow — two-server-action sequence.** On `Skapa kontroll`:
   1. Call `createCycle(metadata + scopeDefinition)` (existing Story 21.2 action at `app/actions/compliance-audit-cycle.ts:182`). On `{ success: false }`, surface the error inline in Step 3 and halt — the cycle was not created; the user can retry.
   2. On `createCycle` success, extract `cycleId` from the response and call `materialiseCycleItems(cycleId)` (new action, this story — see AC 6). On `{ success: false }`, surface the error inline in Step 3 with copy `'Kontrollen skapades men materialiseringen misslyckades. Försök igen.'` + a `Försök igen` button that re-calls only `materialiseCycleItems(cycleId)` — the cycle row remains in `PLANERAD`. Do NOT call `createCycle` a second time (it would produce a duplicate cycle).
   3. **Stale-scope escape hatch.** If `materialiseCycleItems` returns specifically the error `'Omfattningen matchar inga lagar'` (empty-scope edge case — AC 7; occurs when every item in the selected scope was deleted between ScopeSelector emission and materialisation), the wizard renders a **secondary** `Ta bort och börja om` button alongside `Försök igen`. Clicking `Ta bort och börja om` calls the existing `softDeleteCycle(cycleId)` Server Action (Story 21.2 AC 6 — allowed because the cycle is still in `PLANERAD`), and on success resets the wizard to Step 2 with the same `lawListId` pre-filled in Step 1 (metadata is preserved; only scope is cleared). The user re-selects scope and re-submits — which creates a **new** cycle via `createCycle`. Do NOT attempt to re-scope the existing cycle; scope is fixed-at-creation per FR4. If `softDeleteCycle` itself fails, surface its error inline and keep the `Försök igen` / `Ta bort och börja om` options available — the user is not stuck.
   4. On `materialiseCycleItems` success, the client performs `router.push('/laglistor/kontroller/' + cycleId)` via the existing `useRouter` from `next/navigation`. (The target route is defined by Story 21.5; until 21.5 ships, this redirect lands on a `not-found` — acceptable, documented in Dev Notes "Redirect target dependency".)

6. **`materialiseCycleItems(cycleId)` — new Server Action.** A new file `app/actions/compliance-audit-cycle.ts` appends one exported function (or if file length becomes unwieldy, split into `app/actions/compliance-audit-materialise.ts` — see Dev Notes "File colocation vs split" for the decision). Signature:
   ```ts
   export async function materialiseCycleItems(
     cycleId: string
   ): Promise<ActionResult<{ itemCount: number }>>
   ```
   Behaviour:
   - Wrap in `withWorkspace(cb, 'tasks:edit')`.
   - Validate `cycleId: uuid` via a module-local Zod schema (`MaterialiseCycleItemsSchema`). On invalid → structured error `'Valideringsfel'` without any Prisma call.
   - Load cycle scoped to workspace (reuse `loadCycleScopedToWorkspace` helper from Story 21.2). Not-found → `'Kontrollen hittades inte'`.
   - If `cycle.status !== 'PLANERAD'` → return `{ success: false, error: 'Kontrollen har redan materialiserats eller är inte i status Planerad' }`. Idempotency guard: this makes a second call on the same cycle a no-op from the caller's perspective (returns a structured error, no mutation).
   - Resolve `cycle.scope_definition` (Prisma `Json`) into a concrete list of `LawListItem` ids. See AC 7 below for the resolution logic. If the resolved set is empty, return `{ success: false, error: 'Omfattningen matchar inga lagar' }` — this should not happen under normal UI flow because `ScopeSelector` guarantees at least one item (Story 21.3 AC 6), but defence-in-depth is required.
   - Freeze kravpunkter per item: for each resolved `LawListItem`, fetch its current `LawListItemRequirement[]` and store them as a JSON snapshot on the corresponding `ComplianceAuditItem.kravpunkter_snapshot` column. Snapshot shape defined in AC 9 below.
   - Perform the materialisation + cycle-status transition inside a single `prisma.$transaction(...)`: `createMany` all `ComplianceAuditItem` rows, then `update` the cycle row to `status: 'PAGAENDE'` and set `updated_at` (auto via `@updatedAt`). On any failure inside the tx, the full tx rolls back — cycle stays in `PLANERAD`, zero items written.
   - After tx success, call `logActivity(ctx.workspaceId, ctx.userId, 'compliance_audit_cycle', cycleId, 'cycle_materialised', null, { itemCount })`.
   - Call `revalidatePath('/laglistor/kontroller')` exactly once.
   - Return `{ success: true, data: { itemCount } }`.
   - Outer try/catch with `console.error` + generic Swedish `'Kunde inte materialisera kontrollen'`.

7. **Scope resolution logic.** Pure function `resolveScopeToItemIds(scope: ScopeDefinition, lawListId: string, tx: Prisma.TransactionClient)` — called inside the materialisation transaction, returning `string[]` of `LawListItem.id` values:
   - `{ kind: 'all' }` → `tx.lawListItem.findMany({ where: { law_list_id: lawListId }, select: { id: true } })`.
   - `{ kind: 'groups', groupIds }` → `tx.lawListItem.findMany({ where: { law_list_id: lawListId, group_id: { in: groupIds } }, select: { id: true } })`. Groups in `groupIds` that no longer contain any items (edge case — group emptied between ScopeSelector emission and materialisation) simply contribute zero ids; no error.
   - `{ kind: 'items', itemIds }` → `tx.lawListItem.findMany({ where: { law_list_id: lawListId, id: { in: itemIds } }, select: { id: true } })`. If any `itemIds` no longer exist in the list (edge case — item deleted between ScopeSelector emission and materialisation), they are silently excluded and the resolved set is a subset. **Surface a structured warning in the response** if `resolved.length < requested.length`: `{ success: true, data: { itemCount, warnings: ['N valda lagar kunde inte materialiseras (borttagna eller inte längre i laglistan)'] } }`. Extend `ActionResult<T>` to support an optional `warnings?: string[]` field for this story. The wizard surfaces warnings inline after redirect (or in the submit confirmation toast); not blocking.

8. **Atomic materialisation — Prisma transaction shape.**
   ```ts
   await prisma.$transaction(async (tx) => {
     const itemIds = await resolveScopeToItemIds(scope, cycle.law_list_id, tx)
     if (itemIds.length === 0) throw new Error('EMPTY_SCOPE') // caught by outer, maps to error result
     const itemsWithSnapshots = await Promise.all(
       itemIds.map(async (lawListItemId) => ({
         cycle_id: cycleId,
         law_list_item_id: lawListItemId,
         efterlevnadsbedomning: mapComplianceStatus(<fetched>), // AC 9
         kravpunkter_snapshot: <frozen requirements JSON>,       // AC 9
       }))
     )
     await tx.complianceAuditItem.createMany({ data: itemsWithSnapshots })
     await tx.complianceAuditCycle.update({
       where: { id: cycleId },
       data: { status: 'PAGAENDE' },
     })
   }, { timeout: 10_000 }) // 10s tx timeout for 500-item upper bound (NFR1 / IV2)
   ```
   The `timeout: 10_000` explicit setting protects against runaway transactions on pathological inputs and keeps the action within Vercel's 300s serverless ceiling with comfortable headroom. Prisma's default tx timeout is 5000ms; we raise it for the 500-item worst case. (Benchmark: IV2 targets p95 < 3s.)

9. **Item row pre-population — status + kravpunkter.** Per PRD FR3 + AC 5 of the epic:
   - **`efterlevnadsbedomning` mapping** from the source `LawListItem.compliance_status`:
     - `UPPFYLLD` → `UPPFYLLD`
     - `EJ_UPPFYLLD` → `EJ_UPPFYLLD`
     - `EJ_TILLAMPLIG` → `EJ_TILLAMPLIG`
     - `PAGAENDE` → `null` (no initial judgement; reviewer decides)
     - `EJ_PABORJAD` → `null` (ditto)
     Implement as a pure module-local function `mapComplianceStatusToBedomning(status: ComplianceStatus): EfterlevnadsBedomning | null`.
   - **`kravpunkter_snapshot`** — JSON shape (frozen at materialisation time):
     ```ts
     type KravpunkterSnapshot = {
       frozen_at: string // ISO 8601, written at materialisation time
       requirements: Array<{
         id: string          // source requirement.id (for traceability)
         text: string
         comment: string | null
         is_fulfilled: boolean
         bevis_required: boolean
         position: number
         responsible_user_id: string | null
         created_by: string
       }>
     }
     ```
     The snapshot is a **point-in-time copy** — subsequent edits to the source `LawListItemRequirement` rows do NOT mutate this snapshot (Story 21.4's frozen-scope guarantee, per FR4). The snapshot is stored in the Prisma `Json` column `ComplianceAuditItem.kravpunkter_snapshot` (nullable — we set it unconditionally at materialisation; `null` only for cycles created before this story, which should not exist in prod). See Dev Notes "Snapshot shape rationale" for why `id` is included.
   - Other `ComplianceAuditItem` fields at materialisation time: `motivering = null`, `reviewed_at/by = null`, `signed_off_at/by = null` (all defaulted by Prisma).

Rationale: duplicate prevention at DB level; app layer enforces idempotency via the `PLANERAD` guard (AC 6), but a DB-level constraint is a belt-and-braces guarantee. ### Double-submit / concurrency protection — three layers

The materialisation path is protected against double-submit and race conditions by a deliberate three-layer defence:

1. **Client-side `submit: 'submitting'` state (Task 3.6)** — while the two-action sequence is in-flight, the primary button is `disabled`; a double-click, Enter key repeat, or browser back-forward-refresh cycle cannot fire a second submit.
2. **Server-side `PLANERAD` status guard in `materialiseCycleItems` (AC 6)** — a second call after a successful first will find `cycle.status === 'PAGAENDE'` and return a structured error without any writes.
3. **DB-level `@@unique([cycle_id, law_list_item_id])` (AC 10)** — in the pathological case of two parallel `materialiseCycleItems` calls both reading the cycle while still in `PLANERAD` (tiny race window), the second tx's `createMany` hits a unique-constraint violation and the second tx rolls back. First writer wins; no duplicate items.

Each layer is independent and cheap. The combination is belt-and-braces but each layer is the primary defence under a different threat model — 1 handles UI accidents, 2 handles business-logic correctness, 3 handles genuine concurrency. No single layer is redundant.

### Snapshot shape rationale

The `kravpunkter_snapshot.requirements[*]` array includes the source `requirement.id` (not just the text). Rationale:

- Traceability: the cycle detail view (Story 21.5) may want to correlate snapshot entries back to the source requirement to show "changes since snapshot" diffs, or to surface the source requirement's current state alongside the frozen snapshot.
- Reproducibility: the seal hash (Story 21.9) canonicalizes the cycle + all items + snapshots — including `id` makes the canonical input unambiguous.
- Cost: a UUID string adds ~36 bytes per requirement — negligible. For 500 items × 10 requirements average = 180KB total per cycle snapshot column — well within Postgres `jsonb` comfort.

### Code Quality Assessment

**Strong implementation with one real quality issue caught during review.** The wizard's state machine is a clean `useReducer`-based discriminated union; the `materialiseCycleItems` Server Action mirrors Story 21.2's established pattern faithfully (Zod + `withWorkspace('tasks:edit')` + `loadCycleScopedToWorkspace` helper reuse + `logActivity` + `revalidatePath`); the snapshot shape is thoughtfully designed (includes source `id` for traceability); the `TODO(21.10)` marker is correctly placed for the future immutability guard; the three-layer concurrency defence (client submit lock + server PLANERAD guard + DB `@@unique`) is documented and tested.

**What the 4251-test unit suite caught:** all 13 ACs, the SF-1 escape hatch, the bedömning mapping, tx rollback, idempotency, permission denied, cross-tenant isolation, subset-warning populatation, and the p95 perf microtest. All green on Dev's first push.

**What the unit suite MISSED and this review caught:** three material issues that the mocked-Prisma boundary concealed:

1. **N+1 query pattern in materialisation.** `buildKravpunkterSnapshot` was called per-id inside `Promise.all`; inside Prisma's interactive `$transaction`, concurrent `await`s serialise on the single tx connection. For 500 items at ~20ms per round-trip that's ~10s — exceeding NFR1 <3s. The p95 microtest did not catch it because mocks resolve instantly. **Fixed in review** — batched into one `findMany({ where: { list_item_id: { in: resolvedIds } } })` then grouped in-memory, preserving position-asc order.
2. **Zod `limit.max` caller-contract mismatch.** Story 21.3's `ScopeSelector` has always called `getDocumentListItems({ listId, page: 1, limit: 500 })` — deliberately, to match Epic 21 NFR1's 500-item ceiling. But `GetDocumentListItemsSchema` at `lib/validation/document-list.ts:241` capped `limit` at 100. The action returned `{ success: false, error: 'Number must be less than or equal to 100' }` → SWR threw → live wizard showed `"Kunde inte hämta lagar"`. Both Stories 21.3 + 21.4 test suites missed it because both mocked the server action entirely. **Fixed in review** — raised max to 500 with an inline comment explaining the 21.3 + 21.4 contract.
3. **Layout fighting page scroll.** Dev's first pass wrapped ScopeSelector in `max-h-[60vh] overflow-y-auto` + sticky action row; this exposed the underlying ScopeSelector item-row bug (flex children without `min-w-0` → long titles overflow horizontally) on real 68-law lists. **Fixed in review** — removed the scroll cap (natural page scroll restored); item row now uses `flex-1 min-w-0 break-words` so long titles wrap to 2 lines; widened the wizard page to `max-w-4xl` for Step 2 data density.

Three minor polish items addressed as cleanup:
- Swedish singular grammar on the subset warning: `1 vald lag` (wrong) → `1 valt dokument` (correct neuter inflection).
- Dead code: `formatCycleScopeSummaryEmpty` exported but never consumed — removed.
