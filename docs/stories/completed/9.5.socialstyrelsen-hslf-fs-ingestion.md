## Story

**As a** vård- och omsorgs compliance owner (and the catalog itself),
**I want** Socialstyrelsen's consolidated föreskrifter (the legacy **SOSFS** + the shared **HSLF-FS** series) ingested as first-class `AGENCY_REGULATION` documents,
**so that** they resolve in laglistor, kravpunkter, search and RAG, link from marketing pages, and feed future amendment notifications through the existing pipeline — closing the single biggest catalog gap for the vård-omsorg segment (e.g. `SOSFS 2011:9`, the ledningssystem-föreskrift that is the backbone of every vård laglista).

## Scope note

This story is **Phase 1 — ingestion** (get the consolidated set into the catalog, correctly attributed, with amendment *metadata* captured). **Forward amendment monitoring** (emit `ChangeEvent`s when a consolidated page changes) is **Phase 2**, drafted as a separate story (see AC 9 / Future Work). **Per-version historik reconstruction from amendment PDFs is explicitly out of scope** and deferred indefinitely (the consolidation already merges them; the PDFs remain available to backfill later). ### Ingestion (HTML intake — new front-end, existing pipeline tail)

[Source: in-session investigation 2026-06-15; robots.txt allows `/kunskapsstod-och-regler/`, disallows `/sok/`, `/_api/*`, `/*.xml$`]
4. An **HTML-intake ingester** ingests each consolidated page through the existing agency pipeline tail, proven end-to-end on SOSFS 2011:9 (`scripts/test-ingest-sosfs.ts`):
   - fetch page (individual pages are **server-rendered** — plain GET works) → cheerio-extract `#main-content` (strip nav/menu/breadcrumb chrome)
   - **LLM-normalize** extracted HTML → canonical `<article class="legal-document">` (the HTML→HTML analog of the AFS PDF→HTML step), reusing `AGENCY_REGULATION_SYSTEM_PROMPT`/`AGENCY_DEFAULT_MODEL`. [Source: lib/agency/agency-regulation-prompt.ts:16,294]
   - `validateLlmOutput(raw, docNumber)` → `htmlToMarkdown` / `htmlToPlainText` / `parseCanonicalHtml({documentType:'AGENCY_REGULATION'})`. [Source: lib/sfs/llm-output-validator.ts:68; lib/transforms/html-to-markdown.ts:43,503; lib/transforms/canonical-html-parser.ts:46]
   - **linkify** via `buildSlugMap()` + `linkifyHtmlContent(html, slugMap, docNumber)`, store the linkified result as `html_content` (also strips the stray `<html><body>` wrapper). [Source: lib/linkify/build-slug-map.ts:25; lib/linkify/linkify-html.ts:83]
   - upsert `LegalDocument` keyed on `document_number` (`AGENCY_REGULATION`, `ACTIVE`, slug via `generateAgencySlug`). [Source: scripts/ingest-agency-pdfs.ts:330-357; lib/agency/agency-pdf-registry.ts:2470]
   - idempotent (upsert), `--skip-existing`/`--force`/`--filter`/`--dry-run` flags consistent with `ingest-agency-pdfs.ts`. [Source: scripts/ingest-agency-pdfs.ts:9-17]
   - **Fail-soft:** a doc that truncates (`stop_reason === 'max_tokens'`) or fails `validateLlmOutput` is recorded to a **failures manifest** (docNumber + reason) and the batch **continues** — one bad doc never aborts the run. [Source: scripts/ingest-agency-pdfs.ts:247-251 (max_tokens warn); lib/sfs/llm-output-validator.ts:68]

4a. **Large/failed-doc handling is DEFERRED to an end-of-run remediation pass, not pre-built** (PO decision 2026-06-15): the bulk run completes regardless; only **if the failures manifest is non-empty** do we remediate the affected docs — first option being chunked / per-chapter LLM normalization (the AFS `callChunked` pattern). If the failed set is large or non-trivial, remediation splits to a follow-up story. Rationale: the test doc (28 §) was small; let real failures reveal which (if any) HSLF-FS docs actually exceed the 64k output ceiling rather than engineer chunking speculatively. [Source: scripts/ingest-afs-regulations.ts (callClaudeChunked / per-chapter split precedent)]
5. After upsert, each doc is **chunked + context-prefixed + embedded** via `syncDocumentChunks(doc.id)` (it gates on `content_type ∈ {SFS_LAW, AGENCY_REGULATION}` and fires prefixes+embeddings when `markdown_content` is present), and its **`search_vector`** is set (raw SQL; note `legal_documents.id` is `text`, **no `::uuid` cast**). [Source: lib/chunks/sync-document-chunks.ts:22,118; scripts/backfill-search-vectors.ts:45]

### Source anatomy (Socialstyrelsen konsoliderade föreskrifter)

- **Consolidated text lives only in HTML** on the page (Episerver/Optimizely CMS, server-rendered, entity-encoded). There is **no single consolidated PDF** — PDFs are the per-instrument diff chain (grund + each ändring). → ingest the HTML for the body; reference the PDFs for provenance. [Verified in-session]
- **Listing page is JS-rendered** (enumeration needs headless); **individual pages are server-rendered** (plain GET works for ingest). [Verified: `curl` got the body for SOSFS 2011:9; listing static fetch under-counted.]
- **No RSS/API/usable sitemap**; `robots.txt` allows the content path, disallows `/sok/`, `/_api/*`, `/*.xml$`. → discovery is scrape-based.
- Primary key = document number (`SOSFS 2011:9` / `HSLF-FS 2022:30`); slugs are NOT algorithmically derivable (must crawl/curate).
- HSLF-FS is a **shared series across ~8 agencies** → publisher per-document; this story = Socialstyrelsen-issued only.

### Data model (verified against `main`, NOT the stale epic docs)

### Plan — oversized doc remediation (SOSFS 2009:30)

**Problem:** 66 KB text / 93 KB extracted HTML. A single normalize call (`AGENCY_MAX_TOKENS.standard` = 64k output) can't emit the full canonical HTML → the stream terminates. Affects any future föreskrift of this size.

**Fix — chunked/per-chapter normalization (port the AFS `callClaudeChunked` split-and-stitch):**
1. **Trigger:** route to the chunked path when extracted `contentHtml` exceeds a threshold (~60 KB) OR a single-pass attempt returns `stop_reason === 'max_tokens'` / terminates.
2. **Split:** segment the extracted HTML at chapter (`kap.`) boundaries (Episerver renders `<h2>N kap. …</h2>`); for non-chaptered docs, split by § ranges.
3. **Normalize each segment** with the existing `AGENCY_REGULATION` prompt, emitting one `<section class="kapitel">` per chapter (consistent `{articleId}_K{n}_P{m}` anchors), not a full `<article>` wrapper.
4. **Stitch:** concatenate the chapter sections inside one `<article class="legal-document" id={articleId}>` + the lovhead/title, then run the unchanged tail (`validateLlmOutput` → md/plaintext/json → linkify → upsert → `syncDocumentChunks` → search_vector).
5. **Reuse:** the AFS ingester already does exactly this for >100-page PDFs (`splitPdfIntoChunks` + stitch by stripping inner `<article>` tags) — port that logic, swapping PDF-page-split for HTML-chapter-split.

**Effort:** small–medium; isolated to the ingester's normalize step. Belongs in the large-doc remediation follow-up (or fold into Story 9.6). Until done, `SOSFS 2009:30` stays in the failures manifest — re-run `--skip-existing` after to pick it up.
