# Story 6.11: Task Notifications

## Status

Done

## Story

**As a** workspace member,
**I want** to receive in-app and email notifications for task-related activities,
**so that** I stay informed about assignments, deadlines, comments, and status changes without having to manually check the task board.

## Acceptance Criteria

**Notification Triggers:**

1. When a task is assigned to me (via `createTask` with assigneeId or `updateTaskAssignee`), I receive a TASK_ASSIGNED notification
2. When a task I'm assigned to is due within 3 days, I receive a TASK_DUE_SOON notification (daily cron check)
3. When a task I'm assigned to is overdue and not completed, I receive a TASK_OVERDUE notification (daily cron check)
4. When someone comments on a task I created or am assigned to, I receive a COMMENT_ADDED notification (not for my own comments)
5. When I am @mentioned in a task comment, I receive a MENTION notification (using existing `mentions[]` extraction)
6. When a task I created changes status column, I receive a STATUS_CHANGED notification (not triggered by my own changes)
7. A weekly digest email summarizes tasks due this week: "Du har N uppgifter att slutfora denna vecka" (Sunday 18:00 CET)

**Notification Link Navigation:**

8. Clicking a task notification in the bell dropdown navigates to `/tasks?task={taskId}` and opens the task modal
9. The `/api/notifications` route enriches task notifications with the correct `link_url`

**Email Templates:**

10. Email notifications use React Email templates matching the existing `LagligEmailLayout` style
11. Each task email includes an action button deep-linking to the task
12. Email sending respects the user's `email_enabled` global setting and per-type preference toggles

**Notification Preferences UI:**

13. The notification settings tab (`notifications-tab.tsx`) replaces the current disabled placeholder with functional controls
14. Users can toggle each task notification type on/off (7 task types)
15. Preferences are persisted to the `NotificationPreference` table via server action `updateNotificationPreferences`
16. Default: all task notification types enabled

**Deduplication & Self-exclusion:**

17. A user does not receive notifications for their own actions (assigning to self, own comments, own status changes)
18. Duplicate notifications are not created for the same entity + type + user combination within a 24-hour window
19. TASK_DUE_SOON and TASK_OVERDUE crons skip tasks that already have a notification of the same type for the same user in the last 24 hours

**Retention:**

20. Notifications older than 30 days are cleaned up by the daily cron job (piggyback on the deadline cron)

### Pattern to Follow

Model task notifications after `lib/notifications/amendment-notifications.ts`:

1. Receive entity ID + action context
2. Map action to `NotificationType`
3. Resolve recipients (simpler than amendment — just assignee/creator/mentioned users, not full workspace traversal via law lists)
4. Check preference via `shouldSendEmail(userId, workspaceId, type)`
5. Create `Notification` record with `entity_type: 'task'`, `entity_id: taskId`
6. Optionally send email (respecting `email_enabled` + per-type preference)

[Source: architecture/6-components.md#6.4.6]

### Cron Pattern

Follow existing cron patterns in `app/api/cron/notify-amendment-changes/route.ts`:
- `maxDuration = 300` (5 min Vercel limit)
- 30-second buffer timeout before `maxDuration`
- Vercel cron auth header check (see snippet below)
- Log stats (created, skipped, errors)
- Send admin summary email on completion
- Register schedules in `vercel.json`

  }
  // ... cron logic
