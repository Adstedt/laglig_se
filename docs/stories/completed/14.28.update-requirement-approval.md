## Story

**As a** user whose compliance partner proposes an edit to a kravpunkt,
**I want** the proposed change rendered inline as a field-by-field old‚Üínew diff I can review, tweak, and approve,
**so that** requirement edits flow through the same persistent, auditable review path as every other agent action ‚Äî never silently applied.

## Context

The agent can already *read* kravpunkter (via the entity-read tools and the legal-document context), but it has no way to *propose changes* to them. Updating a requirement is one of the most common assessment outcomes: "den h√§r kravpunkten √§r uppfylld ‚Äî markera den och l√§gg till en notis", or "sk√§rp formuleringen", or "den h√§r beh√∂ver bevis". Today the user must do this by hand in the kravpunkter inline editor (Stories 17.16 / 17.17).

This story adds an `update_requirement` write tool that creates an `UPDATE_REQUIREMENT` `PendingAgentAction`. The renderer shows a **field-by-field old‚Üínew diff** so the user sees exactly what will change (e.g., `Uppfylld: Nej ‚Üí Ja`, plus an edited `text`) before approving. On approval the dispatch calls the existing `updateRequirement` server action ‚Äî no new write path into the kravpunkter domain.

Scope is the four **compliance fields** the agent reasons about: `text`, `is_fulfilled`, `comment`, `bevis_required`. Reassigning `responsible_user_id` is explicitly **out of scope** (that's an assignment concern, closer to `ASSIGN_TASK` semantics ‚Äî keep the tool focused).

## Acceptance Criteria

### Schema

1. `PendingAgentActionType` enum is extended with `UPDATE_REQUIREMENT`. Migration is additive (enum value addition only).
2. No new tables. `PendingAgentAction.params` JSON shape for this type:
   ```
   {
     requirementId: string,
     patch: { text?: string, isFulfilled?: boolean, comment?: string | null, bevisRequired?: boolean },
     oldSnapshot: { text: string, isFulfilled: boolean, comment: string | null, bevisRequired: boolean },
     entity_version: string   // requirement.updated_at ISO string at propose time (forward-compat with 14.31)
   }
   ```
3. `patch` contains **only** the fields the agent intends to change; absent keys mean "unchanged". `oldSnapshot` captures the full prior state of the same four fields so the renderer can diff without an extra fetch.

### Tool

4. New tool `lib/agent/tools/update-requirement.ts`. Zod schema mirrors `UpdateRequirementSchema` ([`app/actions/law-list-item-requirements.ts:76`](../../app/actions/law-list-item-requirements.ts)) so invalid input fails at tool time with a Swedish message rather than as a raw `ZodError` at dispatch: `{ requirementId: z.string().uuid(), text?: z.string().min(1).max(500), isFulfilled?: z.boolean(), comment?: z.string().max(2000).nullable(), bevisRequired?: z.boolean(), execute: boolean }`. `comment: null` is valid (clears the comment). Validation: at least one of the four mutable fields must be present (matches the server schema's `.refine`), else `ToolError` ("Ingen √§ndring angiven ‚Äî minst ett f√§lt m√•ste anges.").
5. On `execute: false`: load the current `LawListItemRequirement` (ownership-scoped), snapshot its `text` / `is_fulfilled` / `comment` / `bevis_required` into `params.oldSnapshot`, capture `updated_at` into `params.entity_version`, build `params.patch` from the supplied fields, create the `PendingAgentAction` row, and return the `WriteToolResponse` envelope with `data: { pendingActionId }`.
6. Tool description (Swedish) guides the agent: use when the user asks to mark a kravpunkt fulfilled/unfulfilled, refine the requirement text, toggle "bevis kr√§vs", or set the requirement comment. Explicitly note: do **not** use this to reassign ownership; do not propose a no-op (identical values).
7. `execute: true` is **not** supported (inline approval is the only finalization path ‚Äî matches the 14.23 pattern). If called with `execute: true`, return a `ToolError`.

### Renderer

8. New component `components/features/ai-chat/agent-action-renderers/update-requirement-renderer.tsx`, wrapping the shared `renderer-frame.tsx` and **registered in the `RENDERERS` map at `components/features/ai-chat/agent-action-card.tsx:50`** (add `UPDATE_REQUIREMENT: UpdateRequirementRenderer`) **plus a `UPDATE_REQUIREMENT: '√Ñndra kravpunkt'` entry in the label map at `agent-action-card.tsx:40`** ‚Äî and the batch card's compact path (14.23). **Closest templates:** `update-compliance-status-renderer.tsx` (already renders an old‚Üínew transition ‚Äî the nearest analog to this diff) and `add-obligation-renderer.tsx` (same kravpunkt field vocabulary: text `Textarea` + `bevisRequired` `Switch` mirroring the 17.16 inline editor). This story adds the `is_fulfilled` switch, the `comment` field, and the multi-field diff layout.
9. `PENDING` state ‚Äî **diff layout**: a header ("√Ñndra kravpunkt") + one row per *changed* field (derived by comparing `params.patch` against `params.oldSnapshot`):
   - `text` ‚Üí old text (muted/strikethrough) above new text.
   - `is_fulfilled` ‚Üí `Uppfylld: Nej ‚Üí Ja` (or reverse) as a badge transition.
   - `bevis_required` ‚Üí `Bevis kr√§vs: Nej ‚Üí Ja` (or reverse).
   - `comment` ‚Üí old comment (muted) ‚Üí new comment; `null` / empty / whitespace-only rendered as "(tom)". A change that only differs by surrounding whitespace (e.g. `"text"` ‚Üí `"text  "`) is treated as no-change and suppressed from the diff.
   The proposed **new** values are editable inline before approval (text `Textarea`, fulfilled `Switch`, bevis `Switch`, comment `Textarea`). Edits debounce 500ms and call `updatePendingActionParams` (updating `params.patch`).
10. `APPROVED` state: read-only "Kravpunkt uppdaterad" summary + a link to the parent list item in the laglistor surface (resolve from the requirement's `list_item_id`). Styling matches the other APPROVED renderers.
11. `REJECTED` / `EXPIRED` states: standard muted read-only variants per 14.22's contract. (No `IN_EDITOR` state ‚Äî that's specific to `DRAFT_DOCUMENT`.)
12. `compact?: boolean` variant for the batch card: collapsed summary line "√Ñndra kravpunkt: ${n} f√§lt" expandable to the full diff.

### Approval Dispatch

13. `approvePendingAction` dispatch gains an `UPDATE_REQUIREMENT` branch ‚Üí calls `updateRequirement(requirementId, patch)` at [`app/actions/law-list-item-requirements.ts:278`](../../app/actions/law-list-item-requirements.ts). The `params.patch` keys map 1:1 to that action's `updates` object (`text`, `isFulfilled`, `bevisRequired`, `comment`).
14. `result_ref = { requirementId }`. On dispatch failure the row stays `PENDING` and the action returns `{ success: false, error }` (same pattern as 14.22).
15. `revalidatePath('/laglistor')`. Note: the kravpunkter UI (Stories 17.16 / 17.17) reads via the client SWR key `list-item-requirements:${listItemId}`; that surface refreshes on its next mount/fetch. Live cross-surface sync from the chat approval is **best-effort** for v1 (mirrors the linked-artifacts cross-cache note in MEMORY.md) ‚Äî do not add a server-side client-cache poke.

### Staleness Coordination (14.31)

16. `params.entity_version` (the requirement's `updated_at` at propose time) is captured by the tool (AC 5) and carried through unchanged. This story does **not** add the re-read comparison ‚Äî that's 14.31's cross-cutting retrofit. The field exists from day one so 14.31 needs no schema change.

### Non-Regression

18. The batch card (14.23) correctly includes `UPDATE_REQUIREMENT` rows in per-row + "Godk√§nn alla" flows.
19. Reasoning blocks (14.20), tool-call collapsing (14.18), citation pills (14.16), and chat streaming continue to render correctly.
