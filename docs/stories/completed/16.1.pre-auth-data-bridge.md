## Story

**As a** new user who explored Laglig.se's landing page before signing up,
**I want** my company information to carry through signup and email verification into the onboarding wizard,
**so that** I don't re-enter information I already provided and my path to a populated dashboard is as fast as possible.

## Acceptance Criteria

### Pre-Auth Data Persistence

1. Client-side `OnboardingStore` utility at `lib/onboarding/onboarding-store.ts` — saves, loads, and clears onboarding data from localStorage
2. Data shape includes: org number, company name, legal form, address fields, SNI code, industry label, website URL, business description, inferred activity flags, company summary, and BolagsAPI enrichment fields
3. Data expires after 24 hours (TTL check on read) — prevents stale data from old sessions
4. Stored under key `laglig_onboarding_data` with timestamp
5. All localStorage access wrapped in try/catch — graceful degradation in private browsing or when storage is full

### Signup Page Bridge

6. Signup page (`_signup-form.tsx`) reads `org`, `url`, `flags`, and `summary` query parameters on mount (set by Story 16.2's landing page CTA)
7. If `org` param is present, save to localStorage via OnboardingStore with parsed flags and summary
8. Show a subtle confirmation badge below the signup form: "Vi har din företagsinformation redo" — only visible when onboarding data is stored
9. Standard signup flow — no changes to auth actions, NextAuth config, or email verification

### Onboarding Wizard Pre-fill

10. Company info step reads OnboardingStore on mount
11. If stored data exists: pre-fill org number field, which triggers the existing `useCompanyLookup()` watcher for a fresh BolagsAPI lookup
12. Pre-fill company name and other fields from stored data as immediate fallback while BolagsAPI lookup loads
13. When BolagsAPI lookup succeeds, its data overwrites stored values (fresh data wins) — user sees auto-filled form with "Hämtat från Bolagsverket" badge
14. If stored `companySummary` is present, use it as the initial `businessDescription` value (overrides BolagsAPI legal boilerplate via `setValue`)
15. If stored `inferredFlags` are present, pass them through wizard state to the activity questions step (Story 16.3) as `defaultValues`
16. User reviews pre-filled form and clicks "Nästa" — no auto-advance (user should confirm their data)

### Cleanup

17. After `createWorkspace()` succeeds in the onboarding wizard, call `clearOnboardingData()` before redirect
18. OnboardingStore 24h TTL handles eventual cleanup for abandoned flows

### Standard Auth (Unchanged)

19. No changes to `signupAction()` in `app/actions/auth.ts`
20. No changes to auth callback route (`app/auth/callback/route.ts`)
21. No changes to NextAuth configuration (`app/api/auth/[...nextauth]/route.ts`)
22. Email verification remains standard and required before login
23. Existing workspace guard continues to redirect users without workspaces to `/onboarding`

## Dev Notes

### Architecture: Why localStorage

The data bridge between pre-auth (landing page) and post-auth (onboarding wizard) must survive:
1. Page navigation (landing → signup → verification email → login → onboarding)
2. Browser refresh during signup/verification flow
3. Asynchronous email verification (user may verify minutes or hours later)

Options considered:
- **localStorage** — simple, survives all above, no server-side state, 24h TTL prevents staleness
- **URL query params** — lost during auth flow (verification link → `/auth/callback` → `/dashboard` → `/onboarding` — params don't carry through the redirect chain)
- **Cookies** — 4KB size limit, sent on every request, complicates server-side rendering
- **Server-side session** — requires authenticated user (doesn't exist pre-signup), adds complexity

**Limitation:** If user verifies email on a different device, localStorage data is lost. Acceptable — they go through standard onboarding and enter org number manually (current behavior). The pre-fill is a happy-path optimization, not a requirement.

### Current Onboarding Flow (Unchanged)

```
Signup → Supabase verification email → Click link → /auth/callback
  → exchangeCodeForSession → redirect to /dashboard
    → Workspace guard: no workspace → redirect to /onboarding
      → OnboardingWizard: CompanyInfoStep → ConfirmStep → createWorkspace() → /dashboard
```

This flow is UNCHANGED. What 16.1 adds is pre-filling CompanyInfoStep from localStorage, so the user sees their company info already filled in and just confirms.

### Existing Pre-fill Mechanism

`company-info-step.tsx` already watches `orgNumber` via `useCompanyLookup()` hook. When `orgNumber` changes:
1. Debounced lookup fires against BolagsAPI
2. On success, `useEffect` calls `setValue()` for all enrichment fields
3. "Hämtat från Bolagsverket" badge appears

Setting `orgNumber` from localStorage triggers this same flow automatically. The pre-fill from localStorage seeds the initial value — BolagsAPI provides authoritative data.

### useSearchParams and Suspense

`useSearchParams()` in Next.js requires the component to be wrapped in a `<Suspense>` boundary, otherwise it opts the entire route into client-side rendering. Check if `app/(auth)/signup/page.tsx` already wraps `<SignupForm />` in Suspense. If not, add one:

```typescript
// app/(auth)/signup/page.tsx
import { Suspense } from 'react'

export default function SignupPage() {
  return (
    <Suspense fallback={null}>
      <SignupForm />
    </Suspense>
  )
}
```

### companySummary vs businessDescription Timing

The BolagsAPI auto-fill effect sets `businessDescription` to the BolagsAPI legal text. The `companySummary` from the landing page LLM analysis (Story 16.2) is a better user-facing default. To ensure the summary isn't overwritten by the BolagsAPI effect:

```typescript
// company-info-step.tsx
const storedData = useRef(getOnboardingData())
const [lookupComplete, setLookupComplete] = useState(false)

// After BolagsAPI auto-fill completes (existing useEffect that sets enrichment fields):
useEffect(() => {
  if (data) {
    // ... existing setValue calls ...
    setLookupComplete(true)
  }
}, [data])

// Override businessDescription with stored summary AFTER lookup completes
useEffect(() => {
  if (lookupComplete && storedData.current?.companySummary) {
    setValue('businessDescription', storedData.current.companySummary)
  }
}, [lookupComplete])
```
