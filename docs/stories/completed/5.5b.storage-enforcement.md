# Story 5.5b: Storage Enforcement

## Status

Done

## Story

**As a** workspace owner,
**I want** the platform to block file uploads when my workspace would exceed its storage allowance,
**so that** I'm prompted to upgrade or clean up before silently overflowing into unbilled disk usage.

## Acceptance Criteria

1. **`assertWithinStorageQuota(workspaceId, incomingBytes)` exists** at `lib/usage/storage.ts`:
   - Sums `WorkspaceFile.file_size WHERE workspace_id = ? AND is_folder = false` (no soft-delete column on this model — see Dev Notes for the rationale)
   - Compares `currentBytes + incomingBytes` to `getEffectiveLimits(workspace).storageGB * 1_073_741_824`
   - Throws `StorageQuotaExceededError` (with `currentBytes`, `incomingBytes`, `limitBytes`, `tier` fields) when over
   - Returns `{ warning?: ... }` for soft warn at 80% (non-blocking, surfaced to caller for UI)
2. **Upload entry is gated:** `app/actions/files.ts` upload action calls `assertWithinStorageQuota()` BEFORE writing to Supabase Storage. If the check throws, no Storage object is created (no orphan files).
3. **Error response shape** (HTTP 413 Payload Too Large):
   ```json
   {
     "error": "Lagringsgräns uppnådd",
     "message": "Du har använt 0,9 GB av 1 GB. Filen kunde inte laddas upp. Frigör utrymme eller uppgradera planen.",
     "code": "STORAGE_QUOTA_EXCEEDED",
     "currentBytes": 966367641,
     "incomingBytes": 73741824,
     "limitBytes": 1073741824,
     "tier": "SOLO"
   }
   ```
4. **Storage usage is recomputed on every upload + every delete** — there's no denormalized counter for v1 (small workspaces, cheap aggregate over indexed `workspace_id`; revisit if scale demands). Counter pattern is reserved for AI tokens where the chat hot path needs O(1).
5. **Soft warn at 80%** — when `(currentBytes + incomingBytes) / limitBytes >= 0.80`, the upload succeeds but the response includes a warning payload the UI can surface as a toast or banner.
6. **API contract for storage meter UI:**
   - Server action `getStorageUsage(workspaceId)` returns `{ usedBytes: number, limitBytes: number | null, percentUsed: number, tier: string }`
   - Returns `limitBytes: null` for Enterprise (unlimited)
   - Used by Epic 17 file management surface to render the storage meter (placement decided by Epic 17 — this story only ships the data API)
7. **Enterprise tier has a generous 100 GB default**, NOT truly unlimited storage in v1. `getEffectiveLimits()` returns `storageGB: 100` per parent spec. Enterprise customers needing more get a sales-led `Workspace.storage_gb_override` field (out of scope for 5.5b — separate future story). The check still runs for Enterprise; just at a much higher cap. UI should show "100 GB" not "Obegränsat" for Enterprise storage until the override field exists.
8. **Trial workspaces honor `trial_picked_tier`** — picks up from `getEffectiveLimits()` without additional code in this story.
9. **Existing files are NOT retroactively gated** — if a workspace already exceeds its limit at the time this story ships (unlikely given small Story 5.4 footprint), users can still read/delete existing files. New uploads are blocked until under limit.
10. **Test coverage:** unit tests for `assertWithinStorageQuota()` math; integration tests for upload-blocking; smoke test that delete frees space.

### Why no denormalized counter for storage (v1)

For AI tokens (5.5c), the chat hot path runs hundreds of times a day per active user — a `SUM` query each turn would be wasteful. Storage uploads are a much lower-frequency event (typically a few per workspace per week) and the `WorkspaceFile` table is small. A `SELECT SUM(file_size) FROM workspace_files WHERE workspace_id = ? AND is_folder = false` is fast enough that we don't need denormalization — `WorkspaceFile` already has `@@index([workspace_id])` (`prisma/schema.prisma:1265`), and per-workspace file counts are bounded by storage limits anyway. **Revisit if a workspace has > 10K files** (unlikely for a B2B compliance product where uploads are policy docs and PDFs).

### Index — already in place

The existing `@@index([workspace_id])` on `WorkspaceFile` (`prisma/schema.prisma:1265`) is sufficient for the aggregate query. **No new index needed.** The `is_folder` filter is selective (folders are a small fraction of rows) and PostgreSQL handles the additional condition efficiently within the workspace_id scan.

### No soft-delete on WorkspaceFile

`WorkspaceFile` has no `deleted_at` column — deletes are hard-deletes. This means the storage aggregate is always live: a deleted file's bytes immediately drop out of the SUM. No need to filter on a soft-delete flag. If a future story introduces soft-delete (e.g., trash/restore), update this query to add `deleted_at IS NULL` then.

### `assertWithinStorageQuota` sketch

```typescript
// lib/usage/storage.ts
import { prisma } from '@/lib/prisma'
import { getEffectiveLimits, isUnlimited } from './limits'

const BYTES_PER_GB = 1_073_741_824 // 1024^3

export class StorageQuotaExceededError extends Error {
  constructor(
    public currentBytes: number,
    public incomingBytes: number,
    public limitBytes: number,
    public tier: string
  ) {
    super(`Storage quota exceeded: ${currentBytes + incomingBytes}/${limitBytes} bytes`)
  }
}

export async function assertWithinStorageQuota(
  workspaceId: string,
  incomingBytes: number
): Promise<{ warning?: { percentUsed: number; limitBytes: number } }> {
  const workspace = await prisma.workspace.findUniqueOrThrow({
    where: { id: workspaceId },
    select: { subscription_tier: true, trial_picked_tier: true },
  })

  const limits = getEffectiveLimits(workspace)
  if (isUnlimited(limits.storageGB)) return {}

  const limitBytes = limits.storageGB * BYTES_PER_GB

  const usage = await prisma.workspaceFile.aggregate({
    where: { workspace_id: workspaceId, is_folder: false },
    _sum: { file_size: true },
  })
  const currentBytes = Number(usage._sum.file_size ?? 0)
  const projectedBytes = currentBytes + incomingBytes

  if (projectedBytes > limitBytes) {
    throw new StorageQuotaExceededError(
      currentBytes,
      incomingBytes,
      limitBytes,
      workspace.trial_picked_tier ?? workspace.subscription_tier
    )
  }

  const percentUsed = projectedBytes / limitBytes
  if (percentUsed >= 0.80) {
    return {
      warning: { percentUsed, limitBytes },
    }
  }

  return {}
}
```

### Error message localization

Swedish numbers use comma decimals + space thousands separator. `Intl.NumberFormat('sv-SE', { maximumFractionDigits: 1 })` for "1,2 GB" / "850 MB" / "0,9 GB". Format helper in `lib/usage/storage.ts`:

```typescript
export function formatBytesSwedish(bytes: number): string {
  if (bytes >= BYTES_PER_GB) {
    return new Intl.NumberFormat('sv-SE', { maximumFractionDigits: 1 }).format(
      bytes / BYTES_PER_GB
    ) + ' GB'
  }
  const mb = bytes / (1024 * 1024)
  return new Intl.NumberFormat('sv-SE', { maximumFractionDigits: 0 }).format(mb) + ' MB'
}
```
