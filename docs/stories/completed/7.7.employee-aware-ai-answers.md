## Story

**As a** user,
**I want** the AI to answer employee-specific questions grounded in the full law & regulation corpus plus our kollektivavtal,
**so that** I get accurate, personalized compliance guidance вЂ” for whatever law or fГ¶reskrift applies, not just LAS.

## Acceptance Criteria

1. A user can bring an employee into chat context via an **employee pill/selector** in the chat input (net-new UI вЂ” no context-picker precedent exists; model on the attachment-chip pattern). Selector visible/usable only for roles with `employees:view`; removable pill; the selected `employeeId` rides the transport body per send.
2. The agent receives a compact **`<employee_context>` block** in the system prompt (mirror `formatCompanyContext` exactly): namn, anstГ¤llningsform, anstГ¤llningsdatum, personaltyp, sysselsГ¤ttningsgrad, tilldelat kollektivavtal (name), aktiv/inaktiv. **Explicitly omitted вЂ” asserted by test: personnummer (any form), email, phone, address.** No employee selected в†’ no block, zero behavior change.
3. **Retrieval spans the whole corpus**: a new `search_collective_agreements` agent tool (`sourceTypes: ['COLLECTIVE_AGREEMENT']`, mirroring `search-workspace-files.ts` вЂ” one sourceType per tool, `search_laws` NOT widened) joins the always-available tools. All legal search stays untouched; the model chooses instruments freely вЂ” nothing hardcoded to LAS.
4. **Multi-avtal bias (user requirement):** retrieval gains a `sourceId?: string` filter (`RetrievalOptions` + one WHERE line, same shape as the existing sourceType filter). When an employee with an assigned agreement is in context, the CA tool **hard-filters to that agreement's chunks**; no employee (or unassigned) в†’ all workspace agreements. Workspace isolation clause unchanged.
5. **Citations visually distinguish** Lag/FГ¶reskrift vs **Kollektivavtal** vs Egen fil: `SourceInfo` gains a `collectiveAgreementId` discriminator; `lib/ai/citations.ts#extractSourcesFromToolResult` gets a CA branch (mirror the workspace-files branch); `citation-pill.tsx` gets a CA variant вЂ” distinct icon + "Kollektivavtal" labeling, CTA opening the agreement's PDF via the existing file QuickPreview (`workspace_file_id`). Citation names the instrument (the contextual header already carries "Avtal (Kollektivavtal) > avsnitt").
6. **Scoped grounding eval** (epic AC amended вЂ” the referenced 19.13 harness is an unbuilt Draft): `scripts/eval-ca-grounding.ts` (modeled on `scripts/benchmark-retrieval.ts`) runs the epic's reference queries against `retrieveContext` and asserts (a) CA chunks retrieved are workspace-correct and agreement-correct under bias, (b) a cross-workspace probe returns **zero** foreign CA chunks, (c) legal-corpus retrieval still returns relevant instruments across в‰Ґ2 legal areas (uppsГ¤gningstid/LAS, semester/Semesterlagen, arbetstid/ATL). Retrieval-layer eval only; the full LLM answer-grounding harness remains Story 19.13.

### Design decisions (SM/PO, pre-made)

- **No new `ChatContextType` member** вЂ” the employee is an attribute of the turn (body field), not a context type; GLOBAL/LAW mounts keep working with an employee pill added.
- **One sourceType per tool** is the codebase invariant (`search-workspace-files.ts:8` "never widen the legal search") вЂ” hence the new tool, not a widened `search_laws`.
- **Hard filter, not boost**, for the assigned agreement (per user's multi-avtal requirement): reranker offers no per-source boost hook, and "their agreement, not the other one" is a correctness property, not a preference.
- **Epic AC amendment**: "reuse 19.13 harness" в†’ scoped retrieval-layer eval script (19.13 doesn't exist; building the full answer-eval framework here would triple the story). Cross-workspace leakage assertion = the epic's Risk-Mitigation check, now executable.
- **CA content readability in chat**: agreement text is workspace content like files вЂ” retrieval tools don't add per-permission gating (consistent with `search_workspace_files`); the `employees:view` gate protects the employee *pill/context*, not the agreement text.

### Traps

1. **Never trust the client's employeeId**: the route re-fetches under `withWorkspace`/permission check вЂ” a crafted id from another workspace must return null в†’ no block, no bias (test it).
2. **`formatEmployeeContext` must omit PII by construction** (allowlist of fields, not blocklist) вЂ” tests assert absence of personnummer/email/phone/address.
3. **Inert-path byte-equivalence**: no employeeId в†’ identical system prompt, identical tools except the CA tool with `sourceId: undefined` (which must behave exactly like today's absence of the tool for legal queries вЂ” the tool existing is fine; it simply searches all workspace agreements when invoked).
4. Label strings for enums in the context block: `lib/` may not import from `components/` вЂ” extract or duplicate the few Swedish labels.
5. The eval script needs a live DB вЂ” author + document, don't execute here.
6. Decimal serialization does not apply server-side (route works with Prisma types directly; `full_time_equivalent.toNumber()` or string-format at the block boundary).

### Out of scope

Full LLM answer-grounding harness (19.13), drag-employee-into-chat (epic marks drag optional вЂ” pill only), the Personalkort modal's `renderChat` panel (optional in epic; skip), any schema change.
