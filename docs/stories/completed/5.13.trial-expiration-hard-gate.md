# Story 5.13: Trial Expiration Hard Gate + Stripe Checkout Conversion

## Status

Done

## Story

**As a** product owner,
**I want** trial workspaces to be hard-gated 15 days after signup with a one-click Stripe Checkout conversion path tier-aware to the tier the user picked at onboarding,
**so that** trial users either convert to a paid subscription or lose access — turning evaluation interest into paying customers and bounding free-tier COGS within a fixed window.

## Context & Dependencies

- **Replaces (functionally) Story 4.10 (Draft, never implemented).** 4.10 was drafted Nov 2025 against a hypothetical schema before Stripe shipped. Its camelCase field names (`trialEndsAt`, `subscriptionTier='trial'`), `node-cron` scheduler, custom marketing-style trial-expired page, and EUR pricing all conflict with what 5.4 / 5.5\* / 5.12 actually built. 4.10 is retained as historical context (status flipped to `Superseded by 5.13`); new work goes here.
- **Builds on Story 5.4 (Stripe billing — Done 2026-05-05):**
  - Existing `/api/billing/checkout` accepts `{tier: 'SOLO' | 'TEAM'}`, creates a Checkout Session with `subscription_data.metadata`, returns the redirect URL. The 5.13 conversion panel POSTs to it unchanged.
  - Existing `checkout.session.completed` webhook flips `subscription_tier`, sets `stripe_subscription_id`, clears `trial_ends_at`. The TRIAL_EXPIRED gate naturally lifts — no webhook changes required.
  - `WorkspaceAccessError` + `requirePermissionWithContext` + 402-with-redirect is the precedent for the new TRIAL_EXPIRED gate (mirrors the existing `PAYMENT_PAST_DUE` plumbing).
  - `/settings/billing` (BillingDashboard) is the destination for the gated redirect — already wired, just needs a `?reason=trial_expired` panel.
- **Builds on Story 5.5 family (Done 2026-05-05):** `trial_picked_tier` column persists what the user picked at signup. The conversion prompt uses this to pre-select the right TierCard so conversion is one-click.
- **Builds on Story 5.12 (Done 2026-05-05):** `<TierCard>` (`components/features/billing/tier-card.tsx`) + `lib/billing/tier-display.ts` are the reusable atoms for the trial-expired conversion panel — no new tile components needed. Enterprise-inquiry pattern (`enterprise_inquiry_at` + `trial_picked_tier='TEAM'`) carries through unchanged at trial end.
- **Coordinates with Story 4.9 (Draft — welcome email sequence; not yet implemented):** email-ownership split documented in both directions. **4.9 owns nurture** (Days 1, 3, 7, and a final pre-expiry conversion-push email which 4.9's draft AC currently lists as Day 12 but should be re-numbered to Day 13 when 4.9 is implemented to align with this story's 15-day trial). **5.13 owns lifecycle transactional** (Day 15 trial-ended, Day 45 workspace-paused). No production emails to migrate — 4.9 has never been built.
- **Depended on by Story 5.5z (production cutover, Draft):** production launch cannot ship without trial-end enforcement. 5.13 is a hard prerequisite — recommend pinning this in 5.5z's checklist.
- **Pricing source of truth (unchanged):** `project_tier_pricing_v1.md` memory + `lib/billing/tier-display.ts` (Solo 499 SEK, Team 1 299 SEK, Enterprise custom).
- **Out of scope:** Stripe metered overage (Story 5.5d, deferred); add-on seat purchase UI (Story 5.6, backlog); trial extension via support tooling (manual DB op for MVP); admin dashboard for trial-conversion cohorts (Story 5.10); 4.9's nurture sequence implementation (separate story); pre-trial-end nurture beyond the in-app banner (4.9 territory).

## Acceptance Criteria

1. **Trial duration constant.** New file `lib/billing/trial-config.ts` exports `TRIAL_DURATION_DAYS = 15` and `TRIAL_GRACE_PAUSE_DAYS = 30` and `TRIAL_GRACE_DELETE_DAYS = 60` (days post-trial-end before status PAUSED / DELETED respectively). `createWorkspace` (`app/actions/workspace.ts`) imports + uses the duration constant. All 7 user-facing "14 dagar" mentions (see "Surfaces affected" above) updated to "15 dagar".

2. **New `WorkspaceAccessError` code `TRIAL_EXPIRED`** added to `lib/auth/workspace-context.ts` `WorkspaceErrorCode` union. The gate is computed when:
   - `subscription_tier === 'TRIAL'` AND
   - `trial_ends_at !== null && trial_ends_at < now()` AND
   - `stripe_subscription_id === null` (defense-in-depth — converted users have this set; the webhook also clears `trial_ends_at`, but checking both prevents any race-window holes).

3. **Gate evaluation in BOTH the DB-fetch and Redis cache-hit paths** of `getWorkspaceContextInternal()`. The cached payload is extended with `trialEndsAt` (ISO string), `subscriptionTier`, and `stripeSubscriptionId` (mirror existing `paymentGracePeriodEndsAt` from 5.4). Cache TTL (5 min) is well inside the cron's 24h window — a freshly-expired trial cannot stay accessible for more than 5 minutes after its `trial_ends_at`.

4. **`requirePermission` + `requirePermissionWithContext` mapping.** Both helpers translate `TRIAL_EXPIRED` → HTTP 402 with body `{ code: 'TRIAL_EXPIRED', redirectTo: '/settings/billing?reason=trial_expired' }`. Mirror the existing `PAYMENT_PAST_DUE` handling in all 3 require-permission code paths.

5. **`/settings/billing?reason=trial_expired` conversion panel.** `BillingDashboard` detects the query param and renders a prominent panel above the existing tile grid:
   - Title: **"Din provperiod är slut"**
   - Body: *"Aktivera din prenumeration för att fortsätta använda Laglig.se. Din data är kvar — välj en plan för att låsa upp åtkomsten igen."*
   - Three `<TierCard>`s with the workspace's `trial_picked_tier` rendered FIRST in tab order and visually emphasised (`recommended` prop). Solo + Team buttons label "Aktivera prenumeration" + post to existing `/api/billing/checkout` (no endpoint changes). Enterprise tile shows "Boka samtal" link to `https://cal.com/laglig/sales` (consistent with 5.4 + 5.12 precedent).
   - Footer (small print): *"Din workspace pausas efter 30 dagar och raderas efter 60 dagar utan aktivering. Aktivera nu för att behålla all data."* — state-machine transparency.

6. **Daily cron `/api/cron/expire-trials`** — runs at **00:30 UTC** (offset from `cleanup-workspaces` 06:00 UTC and `cleanup-invitations` 02:00 UTC to spread load). Auth: `Bearer ${env.CRON_SECRET}` (existing pattern from `cleanup-workspaces`). Wrapped in `lib/admin/job-logger` for ops dashboard visibility. `force-dynamic`. `maxDuration = 60`. Registered in `vercel.json`.

7. **Cron action 1 — Trial-end notification (idempotent).** For each workspace where `subscription_tier='TRIAL' AND trial_ends_at < now() AND trial_expired_notified_at IS NULL AND stripe_subscription_id IS NULL`:
   - Set `trial_expired_notified_at = now()` (idempotency lock — guarantees one email per trial, even on cron retry / partial failure).
   - Send Day-15 trial-ended email via Resend (`emails/trial-ended.tsx`).
   - Write `ActivityLog` entry with action `trial_expired`.
   - **For Enterprise inquirers** (`enterprise_inquiry_at IS NOT NULL`): additionally send a sales re-ping email to `env.SALES_NOTIFICATION_EMAIL` ("Trial expired for Enterprise inquirer — please follow up"). Reuse / extend `emails/enterprise-inquiry.tsx` template or add `emails/enterprise-inquiry-trial-ended.tsx`.

8. **Cron action 2 — Status PAUSED at +30 days post-trial-end.** For each workspace where `subscription_tier='TRIAL' AND status='ACTIVE' AND trial_ends_at < now() - INTERVAL '30 days' AND stripe_subscription_id IS NULL`:
   - `status = 'PAUSED'`, `paused_at = now()`.
   - Send `emails/workspace-paused.tsx` ("Din workspace har pausats — sista chans att aktivera").
   - Write `ActivityLog` entry with action `trial_paused`.

9. **Cron action 3 — Status DELETED at +60 days post-trial-end.** For each workspace where `subscription_tier='TRIAL' AND status='PAUSED' AND trial_ends_at < now() - INTERVAL '60 days' AND stripe_subscription_id IS NULL`:
   - `status = 'DELETED'`, `deleted_at = now()`.
   - No email at this stage (the Day-45 paused-email already warned + the existing `cleanup-workspaces` cron will hard-delete 30 days after `deleted_at`).
   - Write `ActivityLog` entry with action `trial_workspace_deleted`.

10. **Cron handler order:** action 1 (notify) → action 2 (pause) → action 3 (delete), each in a separate `prisma.findMany` + per-row update. Per-row failures are caught + logged via the existing `job-logger` with `itemsFailed` increment so one bad workspace doesn't block the rest of the run.

11. **Schema column `trial_expired_notified_at DateTime?` on Workspace** (single new nullable column). NULL on existing rows; populated by cron action 1. No index — daily-cron query volume is low enough that a sequential scan on `subscription_tier + trial_ends_at` (existing partial index opportunity) is fine. Migration is **operator-applied** per `project_prisma_manual_migration_workflow.md` precedent (Stories 5.4, 5.12).

12. **In-app countdown banner** — `components/features/billing/trial-countdown-banner.tsx`. Bottom-right corner card, fixed position. Renders ONLY when `subscription_tier='TRIAL' AND days_until_trial_ends ∈ {1, 2}` (Days 13 + 14 of a 15-day trial). Dismissable per session via `sessionStorage` key `laglig:trial-countdown-banner-dismissed`. Click → `/settings/billing`. Mounted in the workspace layout (server component reads workspace state, passes props to client banner — omitted entirely if not in trial / not in last 2 days, so zero render cost for paid workspaces).

    **Accessibility + responsive requirements:**
    - Container: `role="status" aria-live="polite"` so screen readers announce on appear without stealing focus.
    - Dismiss button: keyboard-focusable, labelled `aria-label="Stäng"`, returns focus to previously-focused element on close.
    - CTA button: real `<button>` / `<Link>` (not div+onClick) for keyboard + screen-reader navigation.
    - Mobile (<640px viewport): repositioned above any bottom nav bar (e.g. `bottom: env(safe-area-inset-bottom) + 80px`), full-width minus 16px gutter, OR hidden entirely if a workspace-layout bottom-nav exists. Verify against actual mobile workspace layout when implementing.
    - Dark mode: respect existing theme tokens, do not hardcode colors.

13. **Email templates (3 new + 1 sales re-ping)** — all React Email, Swedish copy, mirroring the structure + style register of `emails/payment-failed.tsx` (Story 5.4) and `emails/enterprise-inquiry.tsx` (Story 5.12):
    - `emails/trial-ended.tsx` — Subject *"Din provperiod är slut — aktivera för att fortsätta"*. Body: trial ended, data preserved, choose plan to continue. CTA *"Aktivera prenumeration"* → `${NEXT_PUBLIC_APP_URL}/settings/billing?reason=trial_expired&utm_source=email&utm_medium=trial&utm_campaign=trial_ended`.
    - `emails/workspace-paused.tsx` — Subject *"Din workspace har pausats"*. Body: workspace paused, data retained 30 more days, last chance to reactivate. CTA *"Återaktivera workspace"* → `${NEXT_PUBLIC_APP_URL}/settings/billing?reason=trial_expired&utm_source=email&utm_medium=trial&utm_campaign=workspace_paused`.
    - `emails/enterprise-inquiry-trial-ended.tsx` — Internal sales notification. Subject *"Enterprise-lead: trial slut — följ upp"*. Body includes workspace name, owner email (mailto), captured company info, days since `enterprise_inquiry_at`, and a reminder to follow up.
    - **Sender address:** match the established convention. Trial-ended + workspace-paused are billing-critical → `from: 'no-reply'` (per FROM_ADDRESSES doc; same as 5.4's `payment-failed.tsx`). Sales re-ping → `from: 'notifications'` (internal, not customer-facing — same as 5.12's `enterprise-inquiry.tsx`).

14. **Email-send fail-safe pattern.** Cron wraps each email send in try/catch that logs `[TRIAL_EXPIRED_EMAIL_FAIL]` / `[WORKSPACE_PAUSED_EMAIL_FAIL]` / `[ENTERPRISE_INQUIRY_REPING_FAIL]` via `console.error` and continues — never re-throws. Email failure must NEVER prevent the status-machine transitions (idempotency timestamp + status update happen first, email last). Mirrors Story 14.27's `[CHAT_USAGE_EVENT_WRITE_FAIL]` and Story 5.12's `[ENTERPRISE_INQUIRY_EMAIL_FAIL]` patterns.

15. **Enterprise inquiry handling — same hard gate.** Workspaces with `enterprise_inquiry_at IS NOT NULL` are gated identically (per product decision 2026-05-06). The trial-ended email body acknowledges the inquiry: *"Vi har inte hunnit prata med dig än om Enterprise — vill du komma igång med Team så länge? Du kan när som helst byta plan när vi pratats vid."* Sales re-ping email fires alongside (per AC 7). 5.12's framing — Team-during-trial, Enterprise as sales-led — extends naturally past day 15.

16. **ActivityLog action constants** added to `lib/activity/action-constants.ts`:
    - `trial_expired` — fires from cron action 1
    - `trial_paused` — fires from cron action 2
    - `trial_workspace_deleted` — fires from cron action 3
    - `trial_converted` — fires from existing Stripe `checkout.session.completed` webhook handler (small addition there for completeness — single `logActivity(...)` call after the workspace update)
    
    Swedish labels in `lib/activity/categories.ts` + `format-activity.ts`. Entity type stays `workspace` (no new entity type needed). Cron-fired writes use `workspace.owner_id` as `user_id` (ActivityLog requires non-null actor — see "ActivityLog write pattern" in Dev Notes).

19. **Stripe webhook recovers paused-from-trial-expiry workspaces on conversion.** When a user converts AFTER 5.13's cron has flipped `status='PAUSED'` (Day 45+ window), the existing `checkout.session.completed` handler in `app/api/webhooks/stripe/route.ts` MUST also flip `status='ACTIVE'` + clear `paused_at` if the workspace was previously paused due to trial expiry. Without this, the paying customer remains locked out by the existing PAUSED gate. Recovery condition: `workspace.status === 'PAUSED' && workspace.paused_at !== null && workspace.subscription_tier === 'TRIAL'` (the trial-tier check distinguishes trial-expiry-pause from billing-failure-pause and explicit user-pause). New ActivityLog action `workspace_reactivated_from_trial_pause` written alongside.

## Dev Notes

### Why this story exists

The platform has all the conversion plumbing — `/api/billing/checkout`, `<TierCard>`, `trial_picked_tier`, BillingDashboard, Stripe webhook lifecycle handling — but no trigger that says "trial just ended, force the user to convert or stop." Today every trial workspace has `trial_ends_at` set on signup and then... nothing. The user keeps full access indefinitely, COGS keeps accruing, no Stripe revenue ever lands. 5.13 closes the loop by adding the daily cron + access gate + conversion-funneled UI.

### Why hard gate (not soft) — and why no grace window before the gate

The user's product decision (2026-05-06) is unambiguous: *"I absolutely want to gate them upon trial completion. In order for them to continue using our product, they must convert after 15 days through this Stripe checkout to activate their subscription."*

The gate fires the moment `trial_ends_at < now` (effectively the cron's 00:30 UTC tick on Day 16). There is no soft window — login still works (otherwise users can't reach the conversion path), but every workspace route except `/settings/billing` and `/api/billing/*` redirects to the conversion panel until the user converts.

**State machine post-trial-end:**

```
Day 15 (trial_ends_at)
  ↓
Day 16 cron 00:30 UTC: gate active, email sent, banner unnecessary (already gated)
  status=ACTIVE, subscription_tier=TRIAL, trial_ends_at < now
  ↓
+30 days (Day 45): status=PAUSED, paused_at=now, "paused" email sent
  ↓
+30 more days (Day 75): status=DELETED, deleted_at=now, no email
  ↓
+30 more days (Day 105): existing cleanup-workspaces cron hard-deletes
```

Total: 90 days from trial-end to physical delete. User can convert at any point in that window and the gate lifts immediately (Stripe webhook clears `trial_ends_at` + sets `stripe_subscription_id`).

### Cron handler shape

```typescript
// app/api/cron/expire-trials/route.ts — mirrors cleanup-workspaces/route.ts
import { NextResponse } from 'next/server'
import { startJobRun, completeJobRun, failJobRun } from '@/lib/admin/job-logger'
import {
  notifyExpiredTrials,
  pauseAbandonedTrials,
  deleteAbandonedTrials,
} from '@/lib/billing/trial-expiration-cron'

export const dynamic = 'force-dynamic'
export const maxDuration = 60

export async function GET(request: Request) {
  // Bearer auth (same as cleanup-workspaces)
  const authHeader = request.headers.get('authorization')
  const cronSecret = process.env.CRON_SECRET
  if (process.env.NODE_ENV === 'production' && cronSecret) {
    if (authHeader !== `Bearer ${cronSecret}`) {
      return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
    }
  }

  const triggeredBy = request.headers.get('x-triggered-by') || 'cron'
  let runId: string | undefined
  try { runId = await startJobRun('expire-trials', triggeredBy) } catch {}

  try {
    const notifyResult = await notifyExpiredTrials()
    const pauseResult = await pauseAbandonedTrials()
    const deleteResult = await deleteAbandonedTrials()

    const totalProcessed = notifyResult.processed + pauseResult.processed + deleteResult.processed
    const totalFailed = notifyResult.failed + pauseResult.failed + deleteResult.failed

    if (runId) await completeJobRun(runId, { itemsProcessed: totalProcessed, itemsFailed: totalFailed })

    return NextResponse.json({
      success: true,
      notify: notifyResult,
      pause: pauseResult,
      delete: deleteResult,
      timestamp: new Date().toISOString(),
    })
  } catch (error) {
    if (runId) await failJobRun(runId, error instanceof Error ? error : new Error(String(error)))
    return NextResponse.json(
      { success: false, error: error instanceof Error ? error.message : 'Unknown error' },
      { status: 500 }
    )
  }
}
```

### Smoke-test session 2026-05-06 — discoveries + post-implementation refactors

Live local-dev smoke testing surfaced 5 issues that were architecturally significant. All resolved before QA hand-off; documented here so the QA agent + future readers understand the actual shipped behaviour vs the spec's initial sketch.

#### 1. Gate semantics: redirect() instead of throw

**Spec said:** `assertTrialNotExpired` throws `WorkspaceAccessError('TRIAL_EXPIRED')` → caller catches + redirects.

**What broke:** in Next.js App Router, layouts and pages render concurrently. When `/dashboard` rendered, BOTH the `(workspace)/layout.tsx` AND `app/(workspace)/dashboard/page.tsx` called `getWorkspaceContext()` directly. The page-level throw raced the layout-level catch — the page won, propagating the WorkspaceAccessError to the error boundary as a 500. Even the layout's redirect couldn't save it.

**Fix:** `assertTrialNotExpired` and `assertNotPastDue` now call Next.js `redirect('/settings?tab=billing&reason=trial_expired')` directly. `redirect()` throws a special `NEXT_REDIRECT` exception that Next.js intercepts at the framework level — every caller (page, layout, server action, API route) gets short-circuited automatically with a 307 response. No try/catch needed downstream.

The original `WorkspaceAccessError` class + `TRIAL_EXPIRED` / `PAYMENT_PAST_DUE` codes are RETAINED for backwards compatibility and for the `requirePermission` API-route response shape — but the gate ASSERTIONS no longer use them.

#### 2. NEXT_REDIRECT propagation via passthroughRedirect helper

**What broke:** many existing server actions wrap their logic in generic `try { ... } catch (err) { console.error; return { success: false } }` patterns. Those catches swallow `NEXT_REDIRECT` instead of re-throwing it, breaking the redirect chain AND flooding dev console with NEXT_REDIRECT errors. Affected during smoke test: `getCompanyProfile`, `getTaskColumns`, `getDocumentLists`, `getOrCreateDefaultList`, `getUnacknowledgedChanges`, `getUnacknowledgedChangeCount`.

**Fix:** new helper `lib/auth/passthrough-redirect.ts` exports `passthroughRedirect(error)` which re-throws the error if `isRedirectError(error)` is true (using `next/dist/client/components/redirect-error`). Applied to the 6 noisy server actions found during smoke testing — each now starts its catch block with `passthroughRedirect(error)` before generic logging.

**Future maintenance rule:** any NEW server action that calls `getWorkspaceContext()` (directly or via `withWorkspace`) inside a try/catch MUST start its catch with `passthroughRedirect(error)`. There's no automatic enforcement — if you see "NEXT_REDIRECT" appearing in server-action error logs, that's the symptom.

#### 3. Bypass-billing-gates pattern for the conversion surface

**What broke:** the `/settings` page IS the conversion surface — when the user is gated, they need to reach it. But `/settings/page.tsx` calls `getWorkspaceContext()` which triggers the gate's redirect → infinite redirect loop to itself.

**Fix:** new `getWorkspaceContextBypassBillingGates()` export from `lib/auth/workspace-context.ts` (cached separately via React `cache()`). Calls `getWorkspaceContextInternal({ skipBillingGates: true })` which skips both `assertNotPastDue` AND `assertTrialNotExpired`. The 4 surfaces that opt in:

- `app/(workspace)/settings/page.tsx`
- `app/api/billing/checkout/route.ts` (via new `requirePermissionForBilling` helper)
- `app/api/billing/portal/route.ts` (same)
- `app/api/billing/invoices/route.ts` (same)
- `app/api/workspace/context/route.ts` ← **discovered during smoke test** — without bypass here, the client `useWorkspace` hook fetches `/api/workspace/context`, gets a redirect-followed-to-HTML response, JSON parse fails, role defaults to `MEMBER`, and the Fakturering tab is hidden because `canAccessBilling = hasPermission('MEMBER', 'workspace:billing') === false`. User can't see the conversion CTA. **Critical fix.**

The layout's `getWorkspaceContextSafe()` also pathname-detects (`/settings` or `/api/billing/`) and uses the bypass version on those paths — see `app/(workspace)/layout.tsx`.

#### 4. Webhook cache invalidation — Redis + Next.js data cache

**What broke:** after a successful Stripe Checkout, the webhook updated the DB correctly (`subscription_tier=TEAM, status=ACTIVE, stripe_subscription_id=sub_xxx, trial_ends_at=NULL`), but the user STILL saw the conversion panel + paused banner for up to 5 minutes. Two stale-cache layers:

1. **Redis auth-context cache** (`auth:context:${email}:${workspaceId}`, TTL 300s) — populated by `getWorkspaceContextInternal` cache-write branch. Held the pre-conversion state.
2. **Next.js `unstable_cache`** for `getWorkspaceData` in `app/(workspace)/settings/page.tsx` (TTL 300s, tagged `workspace-${workspaceId}`). Held the pre-conversion `subscription_tier` and `trial_ends_at`.

**Fix:** new `invalidateWorkspaceAuthContextCache(workspaceId)` export from `workspace-context.ts` deletes both cache key variants (`auth:context:${email}:${workspaceId}` AND `auth:context:${email}:default`) for every workspace member email. Wired into ALL 5 webhook event handlers in `app/api/webhooks/stripe/route.ts`:

- `checkout.session.completed` — gate lifts immediately
- `customer.subscription.updated` — tier change visible immediately
- `customer.subscription.deleted` — paused banner appears immediately
- `invoice.payment_failed` — past-due gate fires immediately
- `invoice.payment_succeeded` — past-due gate clears immediately

Plus `revalidatePath('/settings')` in the same 5 handlers to invalidate the Next.js data cache so the Prenumeration card shows fresh tier + billing date immediately.

**Both cache invalidations are safe to call multiple times** (Redis `del` of non-existent keys is a no-op; `revalidatePath` is idempotent).

#### 5. Settings page falls back via safeWorkspaceFetch helper

**What broke:** even with `getWorkspaceContextBypassBillingGates` on the settings PAGE, server actions called inside `Promise.all` (`getCompanyProfile`, `getTaskColumns`) still use the GATED context internally. When gated, those actions throw NEXT_REDIRECT through `passthroughRedirect`, which would redirect the page away from the conversion surface (loop).

**Fix:** new `safeWorkspaceFetch<T>(p, fallback)` helper inside `app/(workspace)/settings/page.tsx` catches `isRedirectError` and returns the fallback (`null` for company profile, `{success: false}` for task columns). The Company tab + Workflow tab show empty/placeholder state when gated; user can still reach the Fakturering tab to convert.

#### 6. UI deduplication: hide UsageWidget + tier-picker grid when conversion panel is shown

**What broke:** the BillingDashboard renders `UsageWidget` + a "Välj nivå" tier-picker grid by default. With the conversion panel ALSO showing, the page had duplicate UI: panel at top, then "Användning" widget showing "Kunde inte hämta användningsdata" (gated server action), then ANOTHER tier picker grid below.

**Fix:** wrapped both in `{!showTrialExpiredPanel && (<UsageWidget />)}` and `{!showTrialExpiredPanel && (<Card>...PAID_TIERS...</Card>)}` in `billing-dashboard.tsx`. The conversion panel becomes the single, focused upgrade surface for gated users. The Plan Status card + Invoices section stay (legitimate information).
