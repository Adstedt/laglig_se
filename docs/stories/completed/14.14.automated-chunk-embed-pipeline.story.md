# Story 14.14: Automated Document Normalization, Chunking & Embedding Pipeline

## Status

Done

## Story

**As a** system operator,
**I want** new and updated legal documents to be automatically normalized, chunked, and embedded after ingestion,
**so that** the AI agent's RAG search always includes the latest content without manual script intervention.

## Acceptance Criteria

1. `sync-sfs` derives `json_content` (via `parseCanonicalHtml`) and `markdown_content` (via `htmlToMarkdown`) for every newly created `LegalDocument`, in the same transaction as the `html_content` write. Derivation failures are logged but do not fail the sync.
2. `sync-sfs-updates` derives `json_content` and `markdown_content` for every updated `LegalDocument`, in the same transaction as the `html_content` update. Derivation failures are logged but do not fail the sync.
3. A new cron route `/api/cron/process-chunks` exists that finds documents needing chunking (new or updated since last chunk) and calls `syncDocumentChunks()` for each, with batch limit and timeout protection.
4. The `process-chunks` cron is instrumented with `startJobRun`/`completeJobRun`/`failJobRun`, added to `JOB_REGISTRY` and `vercel.json`, and scheduled at 14:00 and 18:00 UTC.
5. After chunking, chunks missing `context_prefix` or `embedding` are processed (Haiku for prefixes, OpenAI `text-embedding-3-small` for embeddings).
6. A one-time backfill script derives `json_content` and `markdown_content` for existing `LegalDocument` records that have `html_content` but are missing these fields.
7. The cron job logs documents processed, chunks created/updated, embeddings generated, and failures. The daily ops digest includes chunk processing stats.

### Chunking Strategy (Three Tiers)

1. **Tier 1 (preferred)**: Paragraf-level from `json_content`. Each Â§ = one chunk. Path: `kap{N}.Â§{M}`. Requires `json_content` with populated `chapters.paragrafer`.
2. **Tier 2**: Non-Â§ content (transition provisions, preamble, appendices). Separate chunks.
3. **Tier 3 (fallback)**: Markdown paragraph-merge when JSON has 0 paragrafer. Path: `md.chunk{N}`. Uses `markdown_content` or `htmlToPlainText(html_content)`.

**Decision**: Amendments (`SFS_AMENDMENT`) are NOT chunked â€” intentionally excluded per architecture decision. Only `SFS_LAW` and `AGENCY_REGULATION` are chunked. Amendment content (typically 1-5K chars) is fetched at query time via `base_law_sfs` lookup, avoiding ~34K duplicate chunks since consolidated law text already reflects amendment changes.

[Source: `docs/architecture/chunking-strategy.md#three-tier-strategy`, `docs/architecture/chunking-strategy.md#amendment-context-query-time-strategy`]

### Embedding Pipeline

Two phases per document:
1. **Context prefix** (Claude Haiku): One API call per document, returns short semantic context per chunk. Stored in `ContentChunk.context_prefix`. Cost: ~$0.001-0.005/doc.
2. **Embedding** (OpenAI `text-embedding-3-small`): Batch of up to 100 chunks. Input = `contextual_header + context_prefix + content`. Stored in `ContentChunk.embedding` (1536-dim). Cost: ~$0.001/doc.

[Source: `docs/architecture/chunking-strategy.md#llm-context-prefix`, `lib/chunks/embed-chunks.ts`, `lib/chunks/generate-context-prefixes.ts`]

### Code Quality Assessment

**Overall: Strong implementation.** The derivation pattern is clean, correctly placed (after normalization, before linkification), and follows the existing reference implementation (`discover-sfs-amendments`). The new `process-chunks` cron route is well-structured with established patterns (auth, timeout buffer, job logger, batch limits). The backfill script follows the proven `bulk-sync-chunks.ts` cursor-based pattern. No security concerns, no architectural violations.

**Key observations:**

1. **Derivation order is correct** â€” derives from `processedHtml`/`normalizedHtml` (post-normalization, pre-linkification). This ensures json/markdown represent clean canonical HTML, not linkified HTML with `<a>` tags.

2. **Transaction compliance** â€” AC 1 (sync-sfs): derivation happens before the transaction, but the derived values are included in the `create()` call inside the transaction. Atomicity preserved. AC 2 (sync-sfs-updates): derivation happens inside the transaction block. Both correct.

3. **`JSON.parse(JSON.stringify())` for Prisma compatibility** â€” pragmatic workaround for the `CanonicalDocumentJson â†’ InputJsonValue` type mismatch. The reference implementation avoids this because it passes through `createLegalDocumentFromAmendment()` which handles casting internally. Acceptable trade-off here since these are inline calls.

4. **Raw SQL query in process-chunks** â€” well-constructed, parameterized via Prisma tagged template (no injection risk), correctly filters content types and identifies both no-chunk and stale-chunk documents.

5. **Minor: `markdown_content` asymmetry on failure** â€” In the update path, if derivation fails: `json_content` gets `undefined` (field untouched, old value preserved), but `markdown_content` gets `null` (old value cleared). This is defensible since the HTML has changed and stale markdown is arguably worse than no markdown. The system self-heals via the next `process-chunks` run. Not a blocker.

6. **Minor: Backfill script type cast** â€” `documentType: docType as 'SFS_LAW'` masks the `AGENCY_REGULATION` variant. Runtime behavior is correct, but the cast suppresses a useful type check.
