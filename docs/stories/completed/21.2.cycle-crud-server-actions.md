# Story 21.2: Cycle CRUD server actions

## Status

Done

## Story

**As a** KMA-samordnare,
**I want** server actions to create, list, read, update, and soft-delete compliance audit cycles,
**so that** I can manage the lifecycle of cycles programmatically and from the UI without materialising items prematurely (Story 21.4 handles the `PLANERAD → PAGAENDE` materialisation transition).

## Acceptance Criteria

1. `app/actions/compliance-audit-cycle.ts` is created and exposes the following named async server-action exports (no default export, `'use server'` at file top):
   - `createCycle(input)` — creates a cycle in `status: PLANERAD`.
   - `listCyclesForWorkspace(filters?)` — returns cycles for the active workspace, newest first, with optional filters (see AC 9).
   - `getCycleById(cycleId)` — returns the cycle + minimal relation summary (lead auditor display, law list name, item count).
   - `updateCycleMetadata(cycleId, updates)` — partial update of cycle metadata fields only (see AC 5).
   - `softDeleteCycle(cycleId)` — sets `deleted_at` only if the cycle is currently in `PLANERAD`; blocked with a structured error otherwise.
2. All actions call `withWorkspace(callback, requiredPermission)` from `@/lib/auth/workspace-context`. Mutations (`createCycle`, `updateCycleMetadata`, `softDeleteCycle`) require `'tasks:edit'`. Reads (`listCyclesForWorkspace`, `getCycleById`) use `'read'` at the `withWorkspace` entry, AND explicitly check inside the callback that `ctx.hasPermission('activity:view') || ctx.hasPermission('tasks:edit')` — failing the OR-check returns a structured permission-denied result without hitting Prisma. (Rationale: AUDITOR has `activity:view`; all editing roles have `tasks:edit`. The explicit OR-check matches FR16 and the epic's stated read gate.)
3. Every action validates its input via a dedicated Zod schema declared at file top. Invalid input returns `{ success: false, error: <first-issue-message> }` without performing any Prisma call. Schemas:
   - `CreateCycleSchema`: `workspaceId` is taken from `ctx` (NOT from input — prevents cross-workspace writes); input carries `lawListId: uuid`, `name: string (1..200)`, `auditType: enum('INTERN','EXTERN')`, `scheduledStart: ISO date`, `scheduledEnd: ISO date`, `lawChangeCutoffDate: ISO date`, `leadAuditorUserId: uuid`, and `scopeDefinition` — a Zod **discriminated union** on `kind` matching the Json shape declared in `prisma/schema.prisma:1861` and `docs/architecture/epic-21-lagefterlevnadskontroll.md §4.1`:

     ```ts
     scopeDefinition: z.discriminatedUnion('kind', [
       z.object({ kind: z.literal('all') }),
       z.object({ kind: z.literal('groups'), groupIds: z.array(z.string().uuid()).min(1) }),
       z.object({ kind: z.literal('items'),  itemIds:  z.array(z.string().uuid()).min(1) }),
     ])
     ```

     Add a `.refine` that `scheduledEnd >= scheduledStart`. Rationale for requiring `scopeDefinition` at creation time: Story 21.4's wizard calls `createCycle` + `materialiseCycleItems` inside a single Prisma transaction (21.4 AC 4); `materialiseCycleItems` reads `scope_definition` from the freshly-created row (21.4 AC 5). Scope therefore must land on the cycle at create-time. Per FR4, scope is fixed for the cycle's lifetime — re-scoping is a dedicated future "Rescope" action, not a metadata edit — so `scopeDefinition` is deliberately excluded from `UpdateCycleMetadataSchema` below.
   - `UpdateCycleMetadataSchema`: `cycleId: uuid` + optional partials of the editable fields (name, auditType, scheduledStart, scheduledEnd, lawChangeCutoffDate, leadAuditorUserId). At least one field must be present (`.refine` that the partial is non-empty). `scope_definition`, `status`, `seal_*`, and `law_list_id` are NOT editable via this action — scope is fixed-at-creation per FR4 (re-scoping is a dedicated future action), status flows via dedicated lifecycle actions (Stories 21.6 / 21.9), seal fields only by `sealCycle`.
   - `GetCycleByIdSchema` / `ListFiltersSchema` / `SoftDeleteSchema`: see AC 9 / AC 10 / AC 6 respectively.
4. `createCycle` persists the cycle in `status: PLANERAD`, with `created_by_user_id = ctx.userId`, `workspace_id = ctx.workspaceId`, and `scope_definition` set to the validated `scopeDefinition` input (serialised to the Prisma `Json` column as-is). **Does NOT** create any `ComplianceAuditItem` rows — materialisation (resolving `scope_definition` into concrete items + kravpunkter snapshots) is explicitly out of scope for this story and belongs to Story 21.4's `materialiseCycleItems` action, which reads `scope_definition` from the cycle row written here. Returns `{ success: true, data: { cycle } }` with the created cycle shape.
5. `updateCycleMetadata` only mutates the six fields listed in `UpdateCycleMetadataSchema`. Additionally:
   - Validates `leadAuditorUserId` (if present) is an active member of `ctx.workspaceId` via `prisma.workspaceMember.findFirst({ where: { workspace_id, user_id }})` — mirrors the pattern in `law-list-item-requirements.ts:191-205`. If not a member, return a structured error; no write.
   - Validates `scheduledEnd >= scheduledStart` when either is being updated (use current stored value for the un-updated side).
6. `softDeleteCycle`:
   - Loads the cycle (scoped to `ctx.workspaceId`). Not-found / wrong-workspace → structured error.
   - If `status !== 'PLANERAD'` → return `{ success: false, error: 'Kontrollen kan bara tas bort i status Planerad' }`. Do NOT write.
   - Otherwise set `deleted_at = new Date()`. (Prisma soft delete — existing Laglig pattern uses a nullable `deleted_at` column; `listCyclesForWorkspace` MUST filter `deleted_at: null` by default.)
7. Every successful mutation writes a single `ActivityLog` row via the shared `logActivity(workspaceId, userId, entityType, entityId, action, oldValue?, newValue?)` helper at `@/lib/services/activity-logger.ts`:
   - `entityType`: `'compliance_audit_cycle'`
   - `entityId`: the cycle id
   - `action`: `'cycle_created'` | `'cycle_metadata_updated'` | `'cycle_soft_deleted'`
   - `oldValue` / `newValue`: JSON-serialisable snapshots of the changed fields (for `create`, only `newValue`; for `update`, the before-and-after of the mutated fields only; for `soft_delete`, only `oldValue: { status }`).
8. Every successful mutation calls `revalidatePath('/laglistor/kontroller')` exactly once, at the very end of the action (after the DB write and `logActivity`). `/laglistor` itself is NOT revalidated — cycle mutations have no effect on that page (NFR5).
9. `listCyclesForWorkspace(filters?)` accepts optional filter object validated by a Zod schema:
   - `status?: ComplianceCycleStatus | ComplianceCycleStatus[]`
   - `leadAuditorUserId?: uuid`
   - `lawListId?: uuid`
   - `includeDeleted?: boolean` (default `false`)
   - Pagination: `take?: number (1..100, default 50)`, `cursor?: uuid`. Cursor-based pagination mirroring `getLawsPaginated` at `docs/architecture/11-backend-architecture.md §11.7.1` (which uses `take: 51` to detect `hasMore` — use `take + 1` here for the same technique). The `default 50` + `max 100` bounds are an engineering choice consistent with existing list endpoints; no arch rule prescribes the exact number.
   Returned payload: `{ success: true, data: { cycles: [...], nextCursor: string | null } }`. Each cycle row includes: `id, name, status, auditType, scheduledStart, scheduledEnd, leadAuditor: { id, name }, lawList: { id, name }, itemCount: number, createdAt, updatedAt`. `itemCount` is fetched via `_count: { select: { items: true } }` on the relation to avoid N+1. Orders by `created_at DESC` — consistent with the newest-first convention used across existing list endpoints (see `getLawsPaginated` at `docs/architecture/11-backend-architecture.md §11.7.1`).
10. `getCycleById(cycleId)` returns `{ success: false, error: 'Kontrollen hittades inte' }` if the cycle does not exist OR belongs to a different workspace (tenant-isolation check MUST join through `workspace_id = ctx.workspaceId`, not trust the input id). On success returns the full cycle record + `leadAuditor`, `createdBy`, `sealedBy` display names and the cycle's item count. Does NOT expand the full items/findings/reports/snapshots — those load lazily in Story 21.5+.
11. Unit tests in `app/actions/__tests__/compliance-audit-cycle.test.ts` (Vitest) cover, at minimum:
    - **createCycle — happy path**: valid input → cycle persisted in `PLANERAD`, `ActivityLog` row written, `revalidatePath` called once with `/laglistor/kontroller`.
    - **createCycle — validation errors**: missing required field / non-uuid / `scheduledEnd < scheduledStart` / invalid `scopeDefinition.kind` / `scopeDefinition = { kind: 'groups', groupIds: [] }` (empty array fails the `.min(1)`) all return `{ success: false, error }` without any Prisma call.
    - **createCycle — all three scopeDefinition shapes happy path**: `{ kind: 'all' }`, `{ kind: 'groups', groupIds: [...] }`, `{ kind: 'items', itemIds: [...] }` each persist verbatim into `scope_definition` — assert on the `prisma.complianceAuditCycle.create` call's `data.scope_definition` argument for all three.
    - **createCycle — permission denied**: a `ctx.hasPermission('tasks:edit')` returning false → `WorkspaceAccessError` (thrown by `withWorkspace`) → caller's try/catch returns `{ success: false, error }`.
    - **createCycle — cross-tenant lead-auditor**: if `leadAuditorUserId` is not a member of `ctx.workspaceId` (`workspaceMember.findFirst` returns null) → structured error; no `complianceAuditCycle.create` call.
    - **createCycle — duplicate name allowed**: two cycles in the same workspace + law list with identical `name` both succeed (no uniqueness constraint exists; AC 7 of the epic explicitly calls this out).
    - **listCyclesForWorkspace**: filters applied, `deleted_at: null` by default, pagination cursor round-trips.
    - **getCycleById — not found / wrong workspace**: returns the same generic error message in both cases (do not leak existence).
    - **updateCycleMetadata**: partial update persists only the provided fields; non-member `leadAuditorUserId` blocked; empty update rejected by Zod.
    - **softDeleteCycle — allowed on PLANERAD**: sets `deleted_at`, logs activity, revalidates path.
    - **softDeleteCycle — blocked on non-PLANERAD**: all of `PAGAENDE | AVSLUTAD | SEALED | ARKIVERAD` return a structured error, zero DB mutation, zero ActivityLog write.
    - **softDeleteCycle — already-deleted idempotency**: calling soft-delete on a cycle whose `deleted_at` is non-null returns `{ success: false, error: 'Kontrollen är redan borttagen' }` (kept distinct from the status-block error for log clarity).
    Mock `prisma`, `@/lib/auth/workspace-context`, `@/lib/services/activity-logger`, and `next/cache` following the exact pattern at `app/actions/__tests__/law-list-item-requirements.test.ts:24-63`. Use RFC 4122 v4 UUIDs for fixture ids (as `law-list-item-requirements.test.ts:70-76`).

### Read-permission OR-check pattern (AC 2)

The epic specifies reads require `activity:view` OR `tasks:edit`. The existing `withWorkspace` helper accepts a single `Permission` argument and throws `WorkspaceAccessError` if the user lacks it. There are two compatible approaches:

1. **Chosen approach (explicit OR-check inside the callback):** call `withWorkspace(callback, 'read')` to enforce session + workspace + base-workspace-member gate, then inside the callback assert `if (!(ctx.hasPermission('activity:view') || ctx.hasPermission('tasks:edit'))) return { success: false, error: 'Behörighet saknas' }`. Rationale: preserves the existing `withWorkspace` API surface, makes the OR-logic explicit at the call site, and matches the epic AC verbatim. Coverage matrix:
   - OWNER: tasks:edit ✓ → pass
   - ADMIN: tasks:edit ✓ → pass
   - HR_MANAGER: tasks:edit ✓ → pass
   - MEMBER: tasks:edit ✓ → pass
   - AUDITOR: activity:view ✓ → pass
   - (Any future role with only `'read'` and neither: blocked — desired.)
2. Rejected: `withWorkspace(callback, 'activity:view')` would block MEMBER + HR_MANAGER; `withWorkspace(callback, 'tasks:edit')` would block AUDITOR. Neither alone satisfies FR16 + the epic AC.
