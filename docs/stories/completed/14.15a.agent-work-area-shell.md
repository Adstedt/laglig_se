## Story

**As a** compliance manager chatting with the agent on the Hem page,
**I want** to click inline items in AI responses to open an interactive detail panel alongside the conversation,
**so that** I can inspect full context, see structured tool results, and interact with agent suggestions without losing my place in the chat.

## Vision: The Agent Work Area

The guiding principle is the same one that makes Cursor and Claude Code effective: **the conversation is the primary workspace, not a Q&A log.** Everything the user needs to understand and act on what the AI said should be reachable without leaving.

In Cursor, the AI mentions a file and the diff is right there. In Claude Code, tool results expand inline. The conversation and the artifacts it references live side by side.

This story builds the **shell** — the extensible sidebar infrastructure and its first concrete use (citation detail). Story 14.15b adds interactive agent surfaces (assessment cards, task forms, etc.) on top of this shell.

The sidebar is the agent's canvas. Every tool result, every suggestion, every actionable item can render here as structured UI rather than flat text in chat bubbles. New detail types compound — each one makes the workspace more complete.

## Context

### Two-Tier Interaction Model

| Tier | Trigger | Inline Items | Behavior |
|------|---------|-------------|----------|
| **Preview + sidebar** | Hover = quick peek, Click = full sidebar | Citations, law text references | Hover card stays as-is; click promotes to sidebar with full section text |
| **Direct sidebar** | Click = open sidebar immediately | Tasks, law lists, assessments, tool results | No hover preview — the detail panel is the primary interaction |

### Architecture Decision: Sidebar vs Tool Result Rendering

Not all tool results belong in the sidebar. The distinction:

- **Inline in chat**: Read-only search results, status confirmations, simple text responses — the agent's "thinking out loud"
- **Promoted to sidebar**: Structured data that benefits from space and interactivity — document details, write-action previews, assessment forms, comparison tables

This story adds a `sidebarRenderable` flag concept to tool response types (Story 14.15b implements the actual rendering).

## Acceptance Criteria

### Core Sidebar Shell

1. A `ChatDetailSidebar` component renders as a right-side panel within the Hem chat conversation layout. Width is proportional: `w-[35%]` with `min-w-[320px]` and `max-w-[480px]`, giving forms enough room without dominating the chat.
2. When no detail is open, the sidebar is hidden (width 0) and the chat occupies the full area as today.
3. When a detail is opened, the sidebar animates in (`transition-all duration-300`) and the chat area shrinks, with its `max-w-3xl` content naturally re-centering in the remaining space.
4. The sidebar has a header with the item title/type, a breadcrumb (e.g., "Arbetsmiljölagen › Kap 2 › 3 §"), and a close button (X). Clicking close or pressing Escape dismisses it.
5. Only one detail can be open at a time — opening a new item replaces the current sidebar content with a crossfade transition.

### State Management

6. A `ChatDetailContext` (React context) provides `openDetail(item)`, `closeDetail()`, and `activeDetail` to any descendant component within the chat.
7. The context supports a discriminated union of detail types: `citation`, `tool-result` (extensible in 14.15b with `task`, `assessment`, `document`, `law-list`).
8. Opening a detail with the same type and identifier as the currently open one closes the sidebar (toggle behavior).
9. Detail type rendering uses a static `switch` on `item.type` — future types are added by extending the discriminated union and adding a case in the sidebar component.

### Citation Integration (First Concrete Detail Type)

10. `CitationPillInline` retains its existing hover card behavior unchanged.
11. Clicking a resolved citation pill calls `openDetail({ type: 'citation', data: sourceData })` to open the sidebar.
12. The citation sidebar view shows: document title, document number (SFS/regulation number), full section text (from `resolveSource()` snippet or fetched), contextual header path, and a "Visa i lagläsaren" link to the document reader page.
13. The currently-open citation pill receives a visual active state (ring highlight or background color).

### Tool Result Sidebar Rendering (Infrastructure Only)

14. When a tool result part in the message stream has structured data, the `ToolCallRow` component in `chat-message.tsx` renders as clickable — clicking opens the sidebar with the tool result data.
15. A `ToolResultDetail` component renders the raw structured data in a readable card layout (key-value pairs, nested objects as sections). Tool result data is rendered as text only — no `dangerouslySetInnerHTML` or raw HTML rendering. This is the generic fallback — Story 14.15b adds specialized renderers per tool type.
16. Tool results from `search_laws` show as a list of document cards in the sidebar (title, document number, relevance score, snippet).

### Accessibility

17. The sidebar has `role="complementary"` and `aria-label="Detaljpanel"`.
18. When the sidebar opens, focus moves to the close button. When it closes, focus returns to the element that triggered it (the clicked citation pill or tool result row).
19. The sidebar is keyboard-navigable: Tab through interactive elements, Escape to close.

### Responsive Behavior

20. On screens below `lg` (mobile/tablet), clicking an inline item opens a bottom `Sheet` (shadcn/ui) instead of a sidebar. Sidebar mode requires a minimum viewport width of 1024px.
21. The sidebar does not render when `HemChat` is in `panel` mode (right sidebar on non-Hem pages) — inline items fall back to their default behavior (hover cards, links).

### Non-Regression

22. Existing hover card behavior on citation pills is unchanged.
23. Chat input, message list scrolling, and streaming continue to work correctly when the sidebar is open.
24. Conversation history, export, and new conversation flows are unaffected.
25. The `AssessmentResolution` component in `ChatPanel` (currently rendered as `footer` in `ChatMessageList`) continues to work — Story 14.15b relocates it to the sidebar.

## Dev Notes

### Existing Chat Layout Architecture

**`hem-chat.tsx`** currently uses a flex column layout with `max-w-3xl` centered content. The outer container needs to become a flex row:

```tsx
// Current (simplified)
<div className="flex flex-col h-full">
  <div className="max-w-3xl mx-auto w-full">
    {/* messages + input */}
  </div>
</div>

// New
<div className="flex flex-row h-full">
  <div className="flex-1 min-w-0 flex flex-col">
    <div className="max-w-3xl mx-auto w-full">
      {/* messages + input */}
    </div>
  </div>
  <ChatDetailSidebar />  {/* w-0 or w-[35%] min-w-[320px] max-w-[480px] */}
</div>
```

The `flex-1 min-w-0` on the chat area naturally absorbs the space change. The `max-w-3xl` re-centers automatically.

### Citation Source Data Shape

From `resolveSource()` in `lib/ai/citations.ts`, the data available to the sidebar:

```typescript
interface SourceInfo {
  title: string           // "Arbetsmiljölagen"
  snippet: string         // Section text excerpt
  documentNumber: string  // "SFS 1977:1160"
  slug: string           // URL slug for document reader
  anchorId?: string      // Section anchor (e.g., "K2P3")
  path?: string          // Contextual path "Kap 2 › 3 §"
}
```

This is passed from `CitationSourceContext` (provider wraps message rendering in chat route response via `citationSources` metadata).

### Tool Result Access Pattern

In `chat-message.tsx`, the `ToolCallRow` component already receives the tool call part from the AI SDK:

```typescript
// UIMessage.parts includes:
{ type: 'tool-invocation', toolInvocationId, toolName, args, state, output }
```

When `state === 'output-available'`, `output` contains the structured tool response (matching `ToolResponse<T>` shape from `lib/agent/tools/types.ts`). The sidebar can render this data.

### Animation Approach

```tsx
<div className={cn(
  "shrink-0 border-l border-border overflow-hidden transition-all duration-300",
  isOpen ? "w-[35%] min-w-[320px] max-w-[480px]" : "w-0"
)}>
  <div className="w-[35%] min-w-[320px] max-w-[480px] h-full overflow-y-auto">
    {/* Content always rendered at full width, outer container clips via overflow-hidden */}
  </div>
</div>
```

Inner content is always at full width — the outer container clips via `overflow-hidden`. This avoids content reflow during animation.

### Context Provider Placement

The `ChatDetailProvider` wraps the conversation area in `hem-chat.tsx` only when `mode='full'`. It does NOT wrap the `ChatPanel` used in sidebar mode — sidebar mode uses default click behavior (links, hover cards).

```tsx
// hem-chat.tsx
if (mode === 'panel') return <ChatPanel ... />  // no detail provider

return (
  <ChatDetailProvider>
    <div className="flex flex-row h-full">
      {/* chat + sidebar */}
    </div>
  </ChatDetailProvider>
)
```

## Risk Assessment

### Implementation Risks

- **Primary Risk**: Layout shift breaking chat scroll position when sidebar opens
- **Mitigation**: The scroll container stays within the chat flex-1 area; only the available width changes. Test that auto-scroll-to-bottom still works during streaming.
- **Verification**: Open sidebar mid-stream and confirm messages continue appending correctly.

- **Secondary Risk**: Citation pill click conflicting with hover card behavior
- **Mitigation**: Hover and click are independent events — hover shows HoverCard (existing), click opens sidebar (new). The HoverCard's `openDelay` prevents accidental hover on click.

### Rollback Plan

- Revert the `hem-chat.tsx` layout change — the sidebar is purely additive
- Citation pills' onClick can be removed independently of the hover behavior
- The `ChatDetailContext` is self-contained and has no side effects on existing components

## QA Results

### Review Date: 2026-03-26

### Reviewed By: Quinn (Test Architect)

### Code Quality Assessment

**Overall: Good** — Clean, well-structured implementation that follows project patterns closely. The discriminated union + static switch pattern is extensible, the context/hook API is minimal and correct, and the detail views are well-composed. A few issues identified below, none blocking.

**Architecture & Design:**
- Context design is sound — `useChatDetailSafe()` is a pragmatic addition for components in mixed provider/non-provider contexts
- The `renderToolRowContent()` inner function in `ToolCallRow` is declared inside the component after a conditional return — this works but the function is re-created on every render. Low impact since `ToolCallRow` is lightweight.
- Inner div in sidebar uses `w-[35vw]` (viewport width) while the outer container uses `w-[35%]` (parent %). These are not equivalent — on a deeply nested flex layout the inner content width may not match the outer container. Should use the same unit on both.

**Concerns identified:**
1. **Width unit mismatch (medium):** `chat-detail-sidebar.tsx:141` uses `w-[35vw]` for the inner div, but the outer `<aside>` uses `w-[35%]`. These will diverge when the chat is not full-viewport-width (e.g., inside a workspace shell with a left sidebar). The inner div should match the outer: use `w-full` since it's inside the clipped overflow container.
2. **Focus return race condition (low):** In `openDetail`, focus return happens inside `setActiveDetail` updater function. Calling `triggerRef.current.focus()` during a state updater is a side effect in render — React doesn't guarantee this runs at the right time. Should move the focus call to `closeDetail` only (which already handles it), and remove the duplicate from the toggle path in `openDetail`.
3. **Missing test for hover card non-interference (AC 22):** Test 8.4 "Test citation pill hover card still works independently" is listed in the story but the sidebar test file doesn't include a direct test for hover card behavior. The sidebar tests verify sidebar functionality but don't confirm hover cards still work. This is acceptable since the `CitationPillInline` tests would cover that, but there are no existing tests for that component either.
4. **Crossfade implementation (low):** The crossfade uses `contentKey` increment + `fadingIn` boolean with a 100ms delay. On rapid successive opens, the key increments multiple times but `fadingIn` may not reset cleanly due to the timeout. The `useEffect` cleanup handles the timer, so it's safe, but the visual effect may be jerky on rapid switches.

### Refactoring Performed

- **File**: `components/features/ai-chat/chat-detail-sidebar.tsx`
  - **Change**: Changed inner div from `w-[35vw]` to `w-full`
  - **Why**: The inner div is inside an overflow-hidden container that already has the correct width. Using `w-[35vw]` (viewport-relative) would be incorrect when the chat is inside a workspace shell with a left sidebar (the 35vw would include the sidebar width). `w-full` fills the clipped parent correctly.
  - **How**: Single class change, no functional impact.
