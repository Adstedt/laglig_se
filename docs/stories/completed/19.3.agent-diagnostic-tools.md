## Acceptance Criteria

### Shared shape & guards

1. **Four new `'read'` tools** in `lib/agent/tools/`: `list-bevis-gaps.ts`, `list-unassessed-changes.ts`, `list-overdue.ts`, `list-stale-documents.ts`. Each factory takes **only `createXxxTool(workspaceId)`** — these are workspace-wide aggregates, so (unlike the 19.4 readers) they need **no tool context** and take **no required input** (an optional `{ limit?: number }` is allowed, clamped). Workspace-scoped via the closure `workspaceId` + existing relations. **No approval flow** — pure reads.
2. **Every tool returns `{ count, <rows> }`** where `count` is the **true total** and `<rows>` is a **capped** list (default ≈20, at the Prisma `take` level per the 19.4 SF-1 rule — see AC 5 for the one reasoned exception). For the three `findMany` tools (bevis/overdue/stale), `count` is a **separate `prisma.<model>.count({ where })` using the same `where` minus `take`** — never `rows.length` (which would cap at 20). For `list_unassessed_changes` only, `count = rows.length` (the reused core returns the full, naturally-bounded set — AC 5). This lets the agent say "ni har 47 bevisluckor (visar 20)" without loading all rows.
3. **Names, not raw user IDs** (brief + 14.29): any `responsible_user` / `assignee` is resolved to a **name** via the shared `userName` / `userNameOrNull` helper (`lib/agent/tools/reader-utils.ts`); raw user ids are never emitted. Enum values surface as **canonical Swedish labels** via the existing `reader-utils` label helpers (`complianceStatusLabel`, `priorityLabel`, + a new `workspaceDocumentStatusLabel`) — never the raw enum (the CP-001 rule from 19.4). Long free-text (kravpunkt text, change summary) truncated via `shortText`.

### `list_bevis_gaps`

4. `list_bevis_gaps` returns kravpunkter that **require bevis but have none**, reusing the canonical definition (do NOT invent a new one): `{ list_item: { law_list: { workspace_id } }, bevis_required: true, evidence_links: { none: {} } }` — the `needs_evidence` clause from `buildRequirementWhere` (`lib/requirements/query-builders.ts:67`). **Inline this `where`** — do NOT import `buildRequirementWhere` (its signature requires a `userId` the `'needs_evidence'` branch never reads; cite `query-builders.ts:67` as the canonical definition for parity). Shape: `{ count, gaps: [{ requirementId, text, lawListItemId, lawName, sfsNumber, responsibleName }] }`, capped at the `take` level (≈20), `orderBy: { created_at: 'desc' }` (newest gaps first, deterministic). `lawName`/`sfsNumber` come from `list_item.document.{title, document_number}`; `responsibleName` resolves the requirement's `responsible_user` (falling back to the parent `list_item.responsible_user`) to a name via `userNameOrNull`. [Source: schema `LawListItemRequirement:452` (`responsible_user_id:461`, `created_at`), `RequirementEvidenceLink:479`; `query-builders.ts:67`]

### `list_unassessed_changes`

5. `list_unassessed_changes` returns detected lagändringar on the workspace's tracked laws that **have no `ChangeAssessment`** and post-date the item's `last_change_acknowledged_at` floor — reusing the existing logic verbatim. **(SF-A)** Extract a `workspaceId`-parameterized core `loadUnacknowledgedChanges(workspaceId)` from `getUnacknowledgedChanges()` (`app/actions/change-events.ts:31`, which is `withWorkspace`-wrapped → derives the workspace from `cookies()`/session); the action keeps its public signature and **delegates** to the core (behaviour unchanged for the existing dashboard caller). The **tool calls the core directly with its closure `workspaceId`** — no session dependency inside the streaming tool loop (the exact failure mode 19.4 SF-A avoided). Shape: `{ count, changes: [{ changeEventId, lawListItemId, lawName, sfsNumber, changeType, amendmentSfs, summary, detectedAt, priority }] }` (map `documentTitle → lawName`, `documentNumber → sfsNumber`, `aiSummary → summary` from the core's `UnacknowledgedChange` shape). [Source verified: `change-events.ts:31-104` + `UnacknowledgedChange` in `lib/changes/change-utils.ts`]
   - **(PO note, v0.2) Count semantics — match the existing dashboard, don't double-mislead.** The reused query emits **one row per (ChangeEvent × LawList)** intentionally (`change-events.ts:127-130` — same law in two lists = two work items), so `count = rows.length` counts the same lagändring once per list it appears in. This is the *Changes-tab* semantic and is correct to inherit — but the **tool description + the agent's Swedish phrasing must frame the count as "obedömda ändringar att hantera" (work items), NOT "N distinct lagändringar"**, or the agent will overstate distinct changes for a law tracked in multiple lists. Add that one-line framing to the tool description. (De-duping by `changeEventId` is an explicit non-goal — it would diverge from the dashboard count.)

### `list_overdue`

6. `list_overdue` returns tasks past their `due_date` that are **not done**: `prisma.task.findMany({ where: { workspace_id, due_date: { not: null, lt: now }, column: { is_done: false } }, take: 20, orderBy: { due_date: 'asc' } })` (oldest-overdue first). Shape: `{ count, tasks: [{ taskId, title, dueDate, daysOverdue, assigneeName, status }] }` where `status` = `column.name`, `dueDate` = `isoDate(due_date)`, and `daysOverdue` = `Math.floor((Date.now() - due_date.getTime()) / 86_400_000)` (whole days, server time; always ≥ 1 given the `lt: now` filter). "Not done" uses the canonical `TaskColumn.is_done` flag, **not** `completed_at`. Scope = **tasks** (the entity with a first-class due-date + done concept); law-list-item `due_date` overdue is a noted follow-up, not in v1. [Source: schema `Task:1159` (`due_date:1169`), `TaskColumn:1138` (`is_done:1145`)]

### `list_stale_documents`

7. `list_stale_documents` returns styrdokument **past their review date**: `prisma.workspaceDocument.findMany({ where: { workspace_id, review_date: { not: null, lt: now }, status: { notIn: ['SUPERSEDED', 'ARCHIVED'] } }, take: 20, orderBy: { review_date: 'asc' } })`. Shape: `{ count, documents: [{ documentId, title, documentType, status, reviewDate, daysOverdue }] }` (`reviewDate` = `isoDate(review_date)`; `daysOverdue` computed as in AC 6 from `review_date`). **Excluding `SUPERSEDED`/`ARCHIVED`** (no point flagging a retired doc for review) is **PO-ruled final (v0.2)** — review-overdue does not surface retired docs. `status` surfaces as a Swedish label via `workspaceDocumentStatusLabel`; `documentType` is the raw enum (no label map exists — acceptable, it's a coarse category, not user-facing copy). [Source: schema `WorkspaceDocument:1996` (`review_date:2014`, `status:2001`); `WorkspaceDocumentStatus` enum]

### Pinned implementation shapes

**Output interfaces (the tool `data` payloads):**
```ts
interface BevisGapsResult {
  count: number   // true total
  gaps: { requirementId: string; text: string; lawListItemId: string
          lawName: string; sfsNumber: string | null; responsibleName: string | null }[]
}
interface UnassessedChangesResult {
  count: number
  changes: { changeEventId: string; lawListItemId: string; lawName: string
             sfsNumber: string | null; changeType: string; amendmentSfs: string | null
             summary: string | null; detectedAt: string; priority: string }[]
}
interface OverdueResult {
  count: number
  tasks: { taskId: string; title: string; dueDate: string; daysOverdue: number
           assigneeName: string | null; status: string | null }[]
}
interface StaleDocumentsResult {
  count: number
  documents: { documentId: string; title: string; documentType: string
               status: string | null; reviewDate: string; daysOverdue: number }[]
}
```

**Caps decision (SF-1 + one exception):** `list_bevis_gaps`, `list_overdue`, `list_stale_documents` cap at the Prisma `take` level (≈20) with a **separate `count`** for the true total — never load all rows. `list_unassessed_changes` is the **reasoned exception**: it reuses the shipped `loadUnacknowledgedChanges` core (raw SQL, no `LIMIT`), so it loads all unacknowledged rows then `count = rows.length` + `slice(0, 20)` for display. This is safe because the set is **naturally bounded** — it can only be as large as the number of detected amendments on the workspace's tracked laws since their acknowledgement floor (typically tens). Do **not** rewrite the raw SQL to add a LIMIT (it would diverge from the dashboard caller and complicate the `count`).

**`limit` input:** optional `z.object({ limit: z.number().int().min(1).max(50).optional() })`; default 20. Clamp; never unbounded.

### Lazy/diagnostic boundary

These tools are **aggregate reads**, not entity-readers — they return *rows with drill-in ids*, not `ContextHandle` trees. Each row deliberately carries the id the agent needs to follow up (`lawListItemId` → `get_law_list_item`; `taskId` → `get_task`; `changeEventId` → `get_change_details`; `documentId` → `get_document_details`). Do **not** hydrate neighbours here — that's the 19.4 readers' job. Depth = 0 (the agent traverses on demand).

### Workspace-isolation note

`list_bevis_gaps` / `list_overdue` / `list_stale_documents` scope by the closure `workspaceId` in the `where` (no session). `list_unassessed_changes` uses the SF-A core (`loadUnacknowledgedChanges(workspaceId)`) — the raw SQL filters `WHERE ll.workspace_id = ${workspaceId}`, so isolation is on the passed id, not the session. Confirm the extracted core takes `workspaceId` as a parameter (not `ctx`) and that `getUnacknowledgedChanges` passes `ctx.workspaceId` through.
