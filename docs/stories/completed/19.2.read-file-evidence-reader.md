> 📌 **Why this exists:** [[19.1]] gave the chat the ability to *attach* a file and have the agent read it **in that turn** (the file is auto-injected into the user message as a Claude content block). But the agent still cannot reach back and read a file it did **not** just receive — a bevis-PDF linked to a kravpunkt months ago, a styrdokument-attachment, or an attachment from an earlier turn. This story adds the tool that closes that gap: `read_file(fileId)` reads **any** `WorkspaceFile` in the workspace and returns it to the model as native content blocks (PDF/image) or extracted text — the same routing 19.1 uses, but driven by an explicit agent tool call instead of an inbound attachment. It answers the PRD's litmus question: *"can the agent read evidence linked to a kravpunkt?"* — yes, via one code path whether the file arrived as a chat attachment or was uploaded to bevis long ago.

## Story

**As a** workspace member (or AUDITOR) using the compliance chat,
**I want** the agent to be able to open and read any of my workspace's uploaded files on demand — not just files I attached in the current message,
**so that** it can reason over the actual content of bevis, policies, and avtal that are already linked to my compliance state, instead of only the files I happen to re-attach.

## Context

After 19.1, reading a file the agent **did not just receive** is impossible: chat attachments are merged into the *user message* (`build-model-messages.ts`), so they're visible only for the turn they arrive on, and only `search_workspace_files` (semantic *snippets*, not the full file) touches stored files. There is no tool that takes a `fileId` and returns the *whole* file as model-readable content.

The conversion logic already exists — 19.1's `attachmentsToContent` does workspace-scoped lookup, size/MIME routing, Supabase download, base64, and on-demand extraction. The gap is purely **packaging**: expose that routing as an agent tool whose result is delivered to the model as a `tool_result` (a different content shape than a user-message part — see Dev Notes), under the role-filter + decision-log governance 19.5 established.

## Acceptance Criteria

### The tool

1. New tool `read_file` in `lib/agent/tools/read-file.ts`, created via a `createReadFileTool(workspaceId)` factory (closure-captures `workspaceId`, mirroring `createSearchWorkspaceFilesTool` / `createGetCompanyContextTool`). Built with `tool({...})` + `zodSchema` from `ai`, `z` from `'zod/v4'`. [Source: `lib/agent/tools/search-workspace-files.ts:18-48`; `get-document-details.ts:6-7,62-70`]
2. Input schema: `{ fileId: string }` with a Swedish `.describe()` (`'WorkspaceFile-id för filen som ska läsas (t.ex. från search_workspace_files-träffens fileId)'`). No other inputs in v1.
3. **Workspace isolation (hard).** The tool resolves the file via `prisma.workspaceFile.findFirst({ where: { id: fileId, workspace_id: workspaceId } })` — the **same isolation pattern** as 19.1 (`attachments-to-content.ts:153`). A missing file or a cross-workspace id returns a Swedish `wrapToolError` (`'Filen hittades inte.'` + guidance to use `search_workspace_files`) — **never** a thrown 403 and never another workspace's bytes. (Unlike 19.1's converter, which degrades to a *placeholder* because it runs inside an auto-built user message, an explicit tool call returns a proper `ToolError` so the model can recover.)
4. **Routing — identical thresholds to 19.1** (reused from the shared core, AC 9):
   - **PDF ≤ 10 MB** → native `document` block (base64) returned to the model — the agent reads the PDF directly (page layout, tables, scanned text).
   - **Image (PNG/JPEG/GIF) ≤ 5 MB** → native `image` block (base64).
   - **PDF > 10 MB, DOCX, XLSX, PPTX, and any other text-path file** → `extracted_text` prefixed `[Fil: <filnamn>]\n`, with **on-demand extraction** (run `extractFile` synchronously if the cron hasn't populated `extracted_text` yet) — reusing 19.1's `resolveText`. If extraction still yields nothing → a text note `[Fil: <filnamn> — innehåll ej tillgängligt]`.
   - **Image > 5 MB** → text note `[Bild: <filnamn> — för stor för direkt analys]`.
   - **Folder** (`is_folder = true`) → `wrapToolError` (`'Detta är en mapp, inte en fil.'`).
5. **Model delivery via `toModelOutput` (AC 12 — verify in the Task-1 spike).** Native PDF/image blocks cannot be returned through a tool's default JSON/text `execute` result — they must be delivered through the AI-SDK tool hook that converts a tool result into `tool_result` content parts. The tool's `toModelOutput` returns `{ type: 'content', value: [...] }` whose parts are the **tool-result** content shape (`{type:'text'|'image-data'|'file-data', …}`) — **distinct** from 19.1's user-message parts (`TextPart|ImagePart|FilePart`). [Source: `@ai-sdk/anthropic/dist/index.mjs:1756-1820` — provider maps `image-data`→image, `file-data`+`application/pdf`→document; ai runtime `tool.toModelOutput` at `ai/dist/index.mjs:1482`]

### Lean `execute` return (keeps base64 out of logs/history)

6. The tool's **`execute`** returns a **lean JSON envelope** (no base64, no full text) via `wrapToolResponse('read_file', {...}, startTime)`:
   ```
   { fileId, filename, mimeType, sizeBytes, contentKind: 'pdf'|'image'|'text'|'unavailable', citationKey, textPreview? }
   ```
   The heavy bytes/text are produced **only** inside `toModelOutput`, never in the `execute` return. **Rationale:** 19.5's `wrapWithDecisionLog` wraps `execute` and persists its return as `AgentDecisionLog.output_json` (and it's the value the chat-history tool-result renders) — a multi-MB base64 blob there would bloat the audit log, the DB, and the client. `toModelOutput` output is *not* logged by 19.5 (only `execute` is wrapped). [Source: `lib/agent/tools/index.ts:111-124,166-175`; 19.5 AC 9-11]
7. **Citation stub.** `citationKey = filename` — consistent with `search_workspace_files` / the `[Källa: <filename>]` USER_FILE rendering. The tool description instructs the agent to cite content it reads as `[Källa: <filnamn>]`. (Richer per-page/section citation anchors are an explicit non-goal for v1 — a follow-up.) [Source: `lib/agent/tools/search-workspace-files.ts:82-85`]

### Shared conversion core (DRY with 19.1)

8. Extract 19.1's routing internals into a shared module (e.g. `lib/agent/file-content.ts`) exporting a single resolver, e.g.
   ```ts
   resolveFileForReading(fileId, workspaceId): Promise<
     | { kind: 'pdf'|'image'; file: WorkspaceFileLite; bytes: Buffer; mediaType: string }
     | { kind: 'text'; file: WorkspaceFileLite; text: string }
     | { kind: 'unavailable'; file: WorkspaceFileLite | null; reason: 'not_found'|'folder'|'oversized_image'|'no_content' }
   >
   ```
   plus the shared constants (`PDF_INLINE_MAX`, `IMAGE_INLINE_MAX`, `IMAGE_MIMES`, `PDF_MIME`), `downloadBytes`, and `resolveText` (on-demand extraction + best-effort cache write). [Source: `lib/agent/attachments-to-content.ts:26-141`]
9. **Refactor `attachments-to-content.ts` to consume the shared core** — its public API (`attachmentsToContent(fileIds, workspaceId)`) and behaviour are **unchanged**; only the internals move. **All 10 existing converter tests must still pass** (`tests/unit/agent/attachments-to-content.test.ts`). `read-file.ts` maps the same resolver output to the tool-result shape. Net effect: download/route/extract logic lives once; the two callers differ only in **output mapping** (user-message part vs tool-result part). [Source: `docs/stories/completed/19.1.chat-attachment-upload.md` testing section]

### Registry, governance & UI (the 19.5 + 14.x seams)

### Why a tool result is shaped differently than a 19.1 attachment

19.1 attachments are merged into a **user message** (`build-model-messages.ts`) and use `ai`'s user-content parts (`TextPart | ImagePart | FilePart`, e.g. `{type:'file', mediaType, filename, data}` — `attachments-to-content.ts:112-117`). A **tool result** is a different position in the conversation and uses the **tool-result** content shape (`{type:'image-data'|'file-data'|'text', …}`). The shared core (AC 8) returns a neutral `{kind, bytes|text, mediaType}` so each caller maps to its own shape — this is exactly the "shares the conversion logic with 19.1" the PRD asks for, scoped to the **routing core**, not the output mapping.

### Data models

**No new Prisma models, no migration.** `read_file` reads the existing `WorkspaceFile` (id, filename, mime_type, file_size, storage_path, extracted_text, extraction_status, is_folder, workspace_id — the same `select` 19.1 uses, `attachments-to-content.ts:155-165`). The on-demand extraction path may best-effort `update` `extracted_text`/`extraction_status`/`extracted_at` (already done in `resolveText`, `:84-91`). No `CHAT_ATTACHMENT`-specific handling needed — `read_file` reads files of **any** `FileCategory` (bevis, OVRIGT, CHAT_ATTACHMENT, …); category only governs Filer/RAG visibility (19.1), not readability.

### Performance / design notes

- **Double-read trade-off (v1, accept).** `execute` reads only DB scalars (mime/size → `contentKind`), and `toModelOutput` does the heavy Supabase download + (text path) extraction. For the media path this means the bytes are fetched once, in `toModelOutput`. If a future profiling shows redundant work, cache the resolved bytes in a closure-scoped `Map<toolCallId, …>` populated in `execute` and drained in `toModelOutput` — deferred; v1 keeps it simple. 
- **On-demand extraction latency** — same caveat as 19.1: reading a not-yet-extracted DOCX/large-PDF runs a synchronous extraction inside the tool call (adds latency to that step). Acceptable vs. "innehåll ej tillgängligt"; reuses 19.1's best-effort cache so a re-read is fast.
- **`web_search`-style gap reminder** — like 19.5's WS-001, anything injected outside `createAgentTools` would skip decision-logging; `read_file` is **in** the factory, so it is logged. No gap here.
