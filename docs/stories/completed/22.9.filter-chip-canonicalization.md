# Story 22.9: Filter-chip canonicalization (3 bespoke chips → canonical FilterChip)

## Status

Done

## Story

**As a** developer working on workspace filter UIs,
**I want** the three remaining bespoke filter-chip implementations replaced with the canonical `<FilterChip>` + `<FilterChipGroup>` primitives shipped in Story 22.2,
**so that** every workspace surface uses the same chip semantics (`aria-pressed` toggle button), the same density (`py-1.5 text-sm`), and the same active-state visual (`bg-primary text-primary-foreground`) — and nobody can re-roll a fourth bespoke chip implementation when adding a new filter surface.

## Background

Story 22.2 introduced `<FilterChip>` and migrated two surfaces (`cycle-list-table.tsx`, `cycle-findings-tab.tsx`). A follow-on audit (`_prototypes/epic-22-consistency-audit.html` §A) found three more surfaces with bespoke chip implementations:
- `content-type-filter.tsx` (Laglistor type filter — `Alla / Lagar / Förordningar / ...`)
- `krav-filter-chips.tsx` (`/krav` preset filter — `Alla / Luckor / Mina krav / Saknar bevis`)
- `template-catalog-client.tsx` (Mallar domain filter — `Alla / Arbetsrätt / Miljö & Bygg / ...`)

All three used near-identical inline patterns (`rounded-full border px-3 py-1.5 text-sm`) but with subtly different active states (`bg-primary` in 2 of 3, but the canonical `<FilterChip>` was using `bg-foreground` — neutral inversion). Story 22.9 unifies on the bespoke (Laglistor) active state since Laglistor is the gold-standard surface; the canonical `<FilterChip>` was re-themed to match.

## Acceptance Criteria

### Functional — re-theme the canonical primitive

1. **`<FilterChip>` re-themed.** `components/ui/filter-chip.tsx`:
   - Active state: `border-primary bg-primary text-primary-foreground` (was `border-foreground bg-foreground text-background`)
   - Inactive state: `border-border bg-background text-foreground hover:bg-muted` (was `border-border text-muted-foreground hover:border-foreground/40 hover:text-foreground`)
   - Density: `px-3 py-1.5 text-sm` (was `px-3 py-1 text-xs`) — looser, more legible
   - Count badge: `inline-flex items-center justify-center rounded-full text-[10px] font-medium px-1.5 min-w-[1.25rem] h-5 tabular-nums` (was tighter)
   - Existing callers (Kontroller `cycle-list-table`, Findings `cycle-findings-tab`) absorb the new visual automatically — they were the canonical-thin variant; they're now the unified looser variant.

### Functional — migrate the three bespoke implementations

2. **`content-type-filter.tsx`** — replace inline `<button>` map with `<FilterChipGroup aria-label="Filtrera dokument efter typ">` containing `<FilterChip pressed onPressedChange>` per content-type group. `cn` import dropped (no longer needed). `getActiveGroupId` helper preserved.

3. **`krav-filter-chips.tsx`** — replace inline `<button>` + `<span>` count-badge pattern with `<FilterChipGroup aria-label="Filtrera kravpunkter" className="items-center">` containing `<FilterChip pressed count onPressedChange>`. Existing `Rensa (N)` button (rendered alongside chips when state differs from default) preserved; `cn` import dropped.

4. **`template-catalog-client.tsx`** — replace the domain-filter button map with `<FilterChipGroup aria-label="Filtrera mallar efter område">` + `<FilterChip>` per domain. The "source toggle" (`Laglig standardmallar` / `Community`) at the top of the page is NOT migrated — that's a tab-like view-switcher, not a filter chip group; semantic difference per Story 22.2's `Tabs` vs `FilterChip` split.

### Non-functional

5. **`aria-pressed` semantics.** All three migrated surfaces now announce as toggle buttons in screen readers (`aria-pressed="true|false"`), not as tabs (`role="tab"`). No surface uses `role="tab"` outside an actual `<Tabs>` group.

6. **Pressed/unpressed state visual is identical** across all five `<FilterChip>` callers (the original two from 22.2 + the three migrated here). Side-by-side comparison: `/laglistor/kontroller`, `/krav`, `/laglistor/mallar` all render chips with the same height, padding, font size, active bg, count badge size, hover behavior.

## Dev Notes

### Why re-theme rather than migrate Laglistor TO the canonical?

The user's design call (validated during the audit prototype review): Laglistor is the gold-standard surface; its visual conventions are what other surfaces should adopt, not the other way around. Re-theming the primitive is a 5-line className change in `filter-chip.tsx`; consolidating bespoke implementations comes for free after that. Inversely, migrating Laglistor to the canonical-thin variant would have been more migration churn for no real win.

The previous "two chip families running side-by-side" diagnosis (canonical neutral inversion vs bespoke primary accent) was based on `bg-foreground` vs `bg-primary` token names — but in this app's theme both resolve to the same near-white in dark mode and same near-black in light mode. The actual visible drift was the **density** (`py-1` vs `py-1.5`, `text-xs` vs `text-sm`) and the **inactive treatment** (muted text + brighten-on-hover vs full-strength text + darken-bg-on-hover). The re-theme aligns on the looser, more legible variant.
