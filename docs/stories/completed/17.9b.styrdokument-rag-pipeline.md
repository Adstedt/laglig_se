## Story

**As a** system component,
**I want** authored styrdokument (`WorkspaceDocument`) chunked + embedded into the vector pipeline (workspace-scoped, `WORKSPACE_DOCUMENT`) **when they reach `APPROVED`, and removed when they leave it**,
**so that** the agent can semantically search a workspace's *authoritative, currently-published* styrdokument alongside its uploaded files — and never surface a draft or retired policy as authoritative.

## Acceptance Criteria

1. **Chunk shape + reuse.** Styrdokument chunks: `source_type = WORKSPACE_DOCUMENT` (enum value already exists — `schema.prisma:1825`, no enum migration), `source_id = WorkspaceDocument.id`, `workspace_id` non-null. Reuse 17.9's `syncWorkspaceChunks` + the `sourceTypes` retrieval option (`retrieval.ts:25`, `:172`). Net-new code is **thin** — the private paragraph-merge core `chunkWorkspaceMarkdown(args)` (`chunk-workspace-document.ts:107`) is **already parameterised by `sourceType`/`contextualHeader`/`metadata`**, so: (a) a `chunkWorkspaceDocument(...)` wrapper that mirrors `chunkUserFile` (`:77`) — builds metadata + header, delegates to `chunkWorkspaceMarkdown` with `sourceType: 'WORKSPACE_DOCUMENT'` (no algorithm fork); (b) the `WORKSPACE_DOCUMENT` branch in `syncWorkspaceChunks` — which **today throws** `unsupportedSource()` (`sync-workspace-chunks.ts:213-217`); (c) **widen `SyncWorkspaceMeta`** (`sync-workspace-chunks.ts:23-29`) — currently file-shaped (`filename`/`category`/`content_hash`) — to carry styrdokument meta (`title`/`document_type`/`status`), e.g. via a discriminated union on `sourceType` or optional fields.

2. **Content source = `content_html`, NOT `extracted_text`.** `WorkspaceDocumentVersion.extracted_text` (`schema.prisma:2002`) is **flat plaintext** (`extractPlaintext` at `documents.ts:41` strips all tags → no headings to split on). **The HTML→markdown conversion happens in the trigger/caller layer** (mirroring the file path, where `chunkUserFile` receives already-converted markdown and `syncWorkspaceChunks` takes `markdown: string | null` as a param): the trigger calls `htmlToMarkdown(content_html)` (`lib/transforms/html-to-markdown.ts:43`, has an empty-input guard) and passes the result as the `markdown` arg to `syncWorkspaceChunks`. `chunkWorkspaceDocument` therefore takes **markdown**, uniform with `chunkUserFile` — do **not** put `htmlToMarkdown` inside the chunk function (that would diverge the two chunkers' signatures). `htmlToMarkdown` preserves `<h1>/<h2>` → the existing heading-aware `mergeParagraphs` (`:176`) gives section breaks for free.

3. **Reliable trigger (NOT fire-and-forget).** Route every index / de-index through `waitUntil` (or a cron/queue) — consistent with 17.8/17.9's rule that Vercel doesn't reliably run post-response work. Do **not** use bare `void sync(...)`.

4. **Index / de-index is driven by STATUS TRANSITIONS, not content writes** (because AC 5 gates on `APPROVED`). The live lifecycle is the gated state machine in `VALID_STATUS_TRANSITIONS` (`lib/validation/documents.ts:58`): `DRAFT → IN_REVIEW → APPROVED → {SUPERSEDED, ARCHIVED}`; `ARCHIVED` terminal. Wire:
   - **INDEX** on `updateDocumentStatus` (`documents.ts:345`) transition **`IN_REVIEW → APPROVED`** (`:375`): chunk + embed the **current version's** `content_html`.
   - **DE-INDEX** (delete all `WORKSPACE_DOCUMENT` chunks for the doc — the existing primitive: call `syncWorkspaceChunks(id, 'WORKSPACE_DOCUMENT', wsId, null, meta)`, which clears chunks on empty content, `sync-workspace-chunks.ts:77-82`) on:
     - `updateDocumentStatus` **`APPROVED → SUPERSEDED`** and **`APPROVED → ARCHIVED`** (`:378`), and
     - `createDraftFromApproved` (`documents.ts:755`), which flips **`APPROVED → DRAFT`** (verified at `:812`).
   - **DO NOTHING** on the content-write paths while the doc is non-APPROVED: `saveDocumentVersion` (`:537`, preserves status), `restoreDocumentVersion` (`:664`), `importDocxDocument` (`:941`, creates DRAFT), `autosaveDocument` (`:478`). Their content becomes indexable only when the doc is later approved — captured by the INDEX trigger above (it embeds the then-current version).
   - **Edge case RESOLVED (DEC-2):** the index **cannot** drift stale. APPROVED docs are **read-only in the editor** — `isDocumentEditable(status)` returns `true` for **`DRAFT` only** (`lib/utils/document-editability.ts:10`, locked by `tests/unit/lib/utils/document-editability.test.ts`), and `document-editor.tsx` gates the editor (`setEditable`, `:158`) plus the autosave/save handlers on that `editable` flag (`:111`, `:382`, `:485`). So `autosaveDocument`/`saveDocumentVersion` can only run while the doc is `DRAFT`; the sole route to edit an APPROVED doc is `createDraftFromApproved` (→ DRAFT, already an AC-4 de-index trigger). Content is frozen on leaving DRAFT (even `IN_REVIEW` is read-only), and the INDEX trigger embeds the then-current content at approval. **No re-index-on-save is needed** — the status-transition model is sufficient.

5. **Indexable status set = `APPROVED` only** (DEC-1, resolved). Index only `WorkspaceDocumentStatus.APPROVED` (`schema.prisma:1934`). **Never** index `DRAFT`, `IN_REVIEW`, `SUPERSEDED`, or `ARCHIVED` — surfacing unapproved (incl. in-review) or retired policy as authoritative is exactly the risk this gate prevents.

6. **Lifecycle cleanup is the AC-4 de-index hooks — no new delete action needed** (DEC-3, resolved). The two existing `prisma.workspaceDocument.deleteMany` sites (`pending-agent-actions.ts:448`, `:668`) delete **DRAFT / `IN_EDITOR` agent-draft rejections only** (`:439`, `:660`) — under AC 5 these were **never indexed**, so they require **no** chunk cleanup. There is **no hard-delete pathway for an `APPROVED` doc** today, so de-indexing is fully covered by AC 4's `APPROVED → {SUPERSEDED, ARCHIVED, DRAFT}` hooks. **Defensive note for future work:** any new hard-delete that can remove an *indexable* doc must also call the de-index primitive.

7. **Per-doc content hash gates re-embed.** Compute a per-document hash (sha256 of `content_html`) and store it in chunk `metadata` (mirrors 17.9's `content_hash` field, `sync-workspace-chunks.ts:27`). **REL-001 is already fixed in the shared `syncWorkspaceChunks`** (skip only when hash matches **and** zero null embeddings, `:95-114`) — 17.9b inherits it for free; **do NOT re-implement**. This AC only requires threading a hash so the dedupe gate engages.

8. **Metadata + header.** Chunk `metadata` carries `{ title, document_type, status, content_hash }`; `contextual_header` = `title` + `WorkspaceDocumentType` label (`schema.prisma:1923`).

## Dev Notes

- **Model names (anti-hallucination):** the styrdokument models are `WorkspaceDocument` (`schema.prisma:1948`, `prisma.workspaceDocument`) and `WorkspaceDocumentVersion` (`:1993`, `prisma.workspaceDocumentVersion`). There is no plain `Document` model. (A `DocumentVersion` model + `DocumentStatus` enum DO exist — `schema.prisma:1081` / `:604` — but under the SFS/LegalDocument namespace, unrelated to styrdokument; do not use them here.)
- **Schema cites (refreshed 2026-05-22 against current `feat/ai-agent`):** `SourceType` enum `:1820` / `WORKSPACE_DOCUMENT` value `:1825`; `WorkspaceDocumentType` `:1923`; `WorkspaceDocumentStatus` `:1934`; `WorkspaceDocument` `:1948`; `WorkspaceDocumentVersion` `:1993`; `content_html` `:2001` / `extracted_text` `:2002`. Lifecycle: `VALID_STATUS_TRANSITIONS` at `lib/validation/documents.ts:58`.
- **Why AC 4 is status-driven, not content-driven:** the file path (17.9) re-indexes on content writes because files have no approval lifecycle. Styrdokument do — and AC 5 only indexes `APPROVED`. So the events that cross the indexable boundary are **status transitions** (`updateDocumentStatus`, `createDraftFromApproved`), not content saves. Content edited while DRAFT is captured at approval time (the INDEX trigger embeds the then-current version). This is the single biggest correction from the original draft, whose AC 4 wired re-index onto DRAFT content-saves and mislabelled `createDraftFromApproved` as "re-index" when it actually de-indexes.
- **De-index primitive already exists:** `syncWorkspaceChunks(id, 'WORKSPACE_DOCUMENT', wsId, null, meta)` deletes all chunks for the source when content is empty/null (`sync-workspace-chunks.ts:77-82`). No separate delete helper needed.
- **Content is immutable above DRAFT (DEC-2, why the status-only model is safe):** `isDocumentEditable` (`lib/utils/document-editability.ts:10`) returns `true` for `DRAFT` only; `document-editor.tsx` (`:111`, `:158`) makes the editor read-only otherwise. There is no UI path to mutate content while `IN_REVIEW`/`APPROVED`/`SUPERSEDED`/`ARCHIVED`, so the embedded content can never silently diverge from the approved content — the INDEX trigger always sees the final approved version.
- **Layering (verified against live code — get this right):** the HTML→markdown step lives in the **trigger**, not the chunker. Data flow mirrors the file path exactly: `trigger → htmlToMarkdown(content_html) → syncWorkspaceChunks(…, markdown, meta) → chunkWorkspaceDocument({…, markdown}) → chunkWorkspaceMarkdown(…)`. `chunkWorkspaceMarkdown` (`chunk-workspace-document.ts:107`) is already `sourceType`-parameterised and `chunkUserFile` already takes pre-converted markdown — so `chunkWorkspaceDocument` is a ~15-line wrapper, and the only sync-module change beyond the branch is **widening `SyncWorkspaceMeta`** (file-shaped today) to carry `title`/`document_type`/`status`.
- **Net-new vs. reused:** reuses `syncWorkspaceChunks` (incl. REL-001 fix), `chunkWorkspaceMarkdown`/`mergeParagraphs` paragraph-merge core, `generateEmbeddingsBatch`, `retrieveContext` + `sourceTypes`, `htmlToMarkdown` (`lib/transforms/html-to-markdown.ts:43`). Net-new: `chunkWorkspaceDocument` wrapper, the `WORKSPACE_DOCUMENT` sync branch, the `SyncWorkspaceMeta` widening, the per-doc hash, and the status-transition trigger wiring.
