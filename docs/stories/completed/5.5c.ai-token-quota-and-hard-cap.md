# Story 5.5c: AI Token Quota & Hard Cap

## Status

Done

## Story

**As a** product owner,
**I want** the AI chat endpoint to enforce a per-workspace monthly token quota with a runaway-protection hard cap,
**so that** unit economics stay positive even on heavy users until Stripe metered overage (5.5d) calibrates the per-token price from real telemetry.

## Acceptance Criteria

1. **`WorkspaceUsage` Prisma model exists** per the parent spec — `BigInt tokens_used_this_period`, `BigInt storage_bytes`, `period_started_at`, `updated_at`. Migration runs cleanly. One row per workspace, created on first chat turn.
2. **`lib/usage/check.ts` exists** with `assertWithinTokenQuota(workspaceId)` per the parent spec. Returns `{ warning?: ... }` on soft warn or void; throws `TokenQuotaExceededError` when at or above the 2× hard cap.
3. **Chat endpoint quota check is wired:** `app/api/chat/route.ts` calls `assertWithinTokenQuota` BEFORE invoking `streamText`. On `TokenQuotaExceededError`, returns HTTP 402 with the exact JSON shape from the parent spec (key fields: `error`, `message`, `code: 'AI_TOKEN_QUOTA_EXCEEDED'`, `usage`, `limit`, `hardCap`, `tier`).
4. **Token counter increment** happens inside the existing `onFinish` callback in `app/api/chat/route.ts`, in **the same Prisma transaction** as the `ChatUsageEvent` write. The transaction rolls back together if either side fails — no drift.
5. **Counter tracks total billed tokens** = sum of (input + output + cache-read + cache-write) tokens for the turn. Cache-read tokens count even though they're cheap — they still consume the user's quota (the user benefits from the prompt being cached). The cost-rate difference is accounted for in `lib/usage/cost-estimator.ts` (Story 14.27) and irrelevant for quota math. **Implementation note:** the AI SDK v6 fields on the `onFinish` `totalUsage` object are `inputTokens`, `outputTokens`, `cachedInputTokens` (cache reads), and `cacheCreationInputTokens` (cache writes — top-level OR nested in `inputTokenDetails` per Anthropic). The existing `app/api/chat/route.ts` already locally renames these to `cacheReadInputTokens` and `cacheWriteInputTokens` (lines 242-260) — reuse the existing local variables, do NOT re-derive from `totalUsage` directly to avoid the `cachedInputTokens` vs `cacheReadInputTokens` naming collision.
6. **Soft warn at 80% included quota** — `assertWithinTokenQuota` returns `{ warning: { usage, limit, approxQueries } }` when `tokens_used_this_period >= 0.80 × limit && < 2.0 × limit`. The chat route surfaces this to the client via response headers or stream metadata (implementer's choice — document the contract).
7. **Hard cap = 2× included quota** (option B for v1):
   - Solo: 6 000 000 tokens
   - Team (3 users base): 18 000 000 tokens
   - Each Team add-on seat: +3 000 000 tokens
   - Enterprise: no cap (returns void from quota check)
8. **Billing-cycle-anchored reset (Pattern A)** — `tokens_used_this_period` resets to 0 when a workspace's Stripe `current_period_end` advances. Implemented by extending the existing `customer.subscription.updated` webhook handler in `app/api/webhooks/stripe/route.ts` (Story 5.4 already populates `current_period_end` on this event; this story ALSO zeros the counter when the period advances). The reset and the `current_period_end` write happen in the same Prisma transaction so they cannot drift.
   - **Trial workspaces:** `WorkspaceUsage.period_started_at` is initialized to `workspace.created_at` on first chat turn (the upsert path in AC 4 uses this when creating the row). Throughout the trial the counter accumulates against the picked-tier limits. When the user converts (Stripe Checkout → first subscription created), the `customer.subscription.updated` webhook fires with `current_period_end` going from NULL → future date. The handler detects the advance and resets — the trial-conversion charge moment IS the first reset event.
   - **Plan upgrade or downgrade mid-period:** `current_period_end` does NOT advance; counter is NOT reset. User's existing consumption carries through to the new tier; if the new tier's cap is lower, they may immediately be over.
   - **Cancellation:** counter unchanged; subscription stays active until `current_period_end`. Re-subscription after lapse fires a new `subscription.updated` and resets normally.
9. **Defensive safety cron** at `/api/cron/reset-usage-safety` runs weekly (`0 4 * * 0` = Sunday 04:00 UTC, ~05:00 SE winter / 06:00 SE summer). Resets `WorkspaceUsage` rows where `period_started_at < NOW() - INTERVAL '35 days'` to catch workspaces whose webhook reset failed. 35 days = monthly + 5-day grace so it never overlaps an in-flight billing cycle. Idempotent. Auth via `Bearer ${env.CRON_SECRET}`. **Most workspaces never hit this cron** — it's a backstop.
10. **`vercel.json` registers the safety cron** alongside the existing entries.
11. **Billing dashboard usage widget** lives on `/settings?tab=billing`. Shows for each workspace:
    - Token usage: `"X / Y miljoner tokens använda denna månad (≈ Z AI-frågor kvar)"` with a progress bar
    - Storage usage: `"A / B GB använt"` (uses `getStorageUsage()` from 5.5b)
    - Seat usage: `"M / N platser används"` (uses 5.5a's helper)
    - Reset hint: `"Förnyas {next_period_start_date}"` — formatted via `new Intl.DateTimeFormat('sv-SE', { dateStyle: 'long' }).format(new Date(workspace.current_period_end))` for output like "5 juni 2026". For trial workspaces (no `current_period_end`), show `"Förnyas vid första betalning"` instead.
12. **Latency budget:** `assertWithinTokenQuota` adds **< 30ms p95 on cache hit; < 250ms p99 on cache miss** to the chat request. Implementer must include a benchmark or measurement note in the dev record. The check is one indexed `findUnique` on `WorkspaceUsage` + one on `Workspace` + the cached `addonSeatCount` lookup (cache-hit: ~5ms Redis round-trip; cache-miss: 100-200ms Stripe API).
13. **Add-on seat caching uses Upstash Redis** (already a project dep — `@upstash/redis` in `package.json`), NOT in-memory `Map`. Rationale: Vercel serverless runs many cold instances; an in-memory cache would have a low hit rate because each instance has its own Map. Redis gives a true shared cache across instances, with the same 5-minute TTL semantics. Implementation: `await redis.get(seatCountKey)` / `await redis.set(seatCountKey, count, { ex: 300 })`. Network cost: ~5ms per chat turn (well within latency budget). **Cache invalidation is required**, not optional — the `customer.subscription.updated` webhook MUST call `invalidateSeatCache(workspaceId)` (`redis.del(...)`) so add-on seat purchases reflect immediately rather than waiting up to 5 minutes.
14. **Existing `ChatUsageEvent` rows are NOT migrated to populate counters** — the counter starts at 0 for all workspaces on rollout. Existing usage history stays in `ChatUsageEvent` for telemetry/admin views. From 5.5c rollout forward, the counter and `ChatUsageEvent` move in lockstep.
15. **Test coverage:** unit + integration + E2E per Testing section. Latency benchmark required.

### Why same-transaction increment

If the counter increment lives in a separate query AFTER the `ChatUsageEvent` write, you get this drift scenario:

1. Turn N completes, `ChatUsageEvent` row written (tx 1)
2. Counter increment fires (tx 2), DB connection drops mid-flight
3. Counter is short by N's tokens; `ChatUsageEvent` says otherwise
4. Reconciliation requires a periodic counter rebuild from `ChatUsageEvent` (extra cron, extra ops surface)

Same-transaction (`prisma.$transaction([eventWrite, counterUpsert])`) eliminates this entirely. Both succeed or both roll back. The `counterUpsert` is `prisma.workspaceUsage.upsert` so the row exists after the first turn.

### Counter upsert pattern

The existing chat route at `app/api/chat/route.ts:238-290` already destructures `totalUsage` from `onFinish` and computes `cacheReadInputTokens` / `cacheWriteInputTokens` as local variables (lines 242-260) — those handle the AI-SDK-vs-Anthropic-naming nuance. **Reuse those local variables**; don't re-read from `totalUsage` directly:

```typescript
// Inside onFinish callback in app/api/chat/route.ts — extending existing logic.
// totalUsage, cacheReadInputTokens, cacheWriteInputTokens are already in scope
// from the existing ChatUsageEvent code (lines 242-260).
const totalTokens =
  totalUsage.inputTokens +
  totalUsage.outputTokens +
  cacheReadInputTokens +
  cacheWriteInputTokens

// Read workspace.created_at once for the upsert's create branch (trial workspaces
// need period_started_at anchored to workspace creation, not "now") — see AC 8 trial section.
const workspace = await prisma.workspace.findUniqueOrThrow({
  where: { id: workspaceId },
  select: { created_at: true },
})

await prisma.$transaction([
  prisma.chatUsageEvent.create({ data: { /* existing */ } }),
  prisma.workspaceUsage.upsert({
    where: { workspace_id: workspaceId },
    create: {
      workspace_id: workspaceId,
      tokens_used_this_period: BigInt(totalTokens),
      period_started_at: workspace.created_at,  // anchor to workspace creation for trials
    },
    update: {
      tokens_used_this_period: { increment: BigInt(totalTokens) },
    },
  }),
])
```

For optimisation: if the `findUniqueOrThrow` for `created_at` shows up as latency in benchmarking, denormalise the value into `getWorkspaceContext()` (which the chat route already calls) and pass it through. v1: keep it explicit and simple.

### Soft-warn surface contract

The soft-warn payload needs to reach the client UI so it can render a toast/banner. Two viable patterns:

**Option A — Response headers** (simpler):
```
X-AI-Usage-Warning: { "usage": 2400000, "limit": 3000000, "approxQueriesRemaining": 20 }
```
Client middleware reads the header and dispatches a toast.

**Option B — Stream metadata** (richer, harder):
Pre-emit a metadata frame at the start of the stream before the model output begins. Requires stream-protocol changes.

**Pick Option A for v1.** Simple, doesn't disturb the existing chat streaming contract, and 80% warnings don't need to be in-stream.

### Add-on seat caching detail (Redis-backed)

The reason for caching: every chat turn calls `assertWithinTokenQuota`, which calls `getEffectiveLimits(workspace, addonSeatCount)`, which (without cache) calls `countActiveAddonSeats(workspaceId)` — and that's a Stripe API call (~100-200ms). Adding 100-200ms to every chat turn is unacceptable.

**Why Redis, not in-memory:** Vercel functions run on multiple cold-started instances. An in-memory `Map` cache would only help within a single instance — every cold start and every cross-instance request would miss. With Laglig's existing Upstash Redis (`@upstash/redis` in `package.json`, used by `@upstash/ratelimit`), we get a true shared cache for ~5ms of network overhead per chat turn. Net latency win: 100-200ms saved on cache hit, every hit, across all instances.

> **Note on the add-on shape dependency:** The `countActiveAddonSeats()` implementation lives in 5.5a's `lib/usage/seats.ts`. Story 5.6 (Add-On Purchase System, backlog) will formalize the add-on subscription shape, and 5.5a includes a cross-reference flagging this. **5.5c's cache wrapper is transparent to the underlying shape** — if 5.6 reshapes how add-on seats are stored on the Stripe subscription, 5.5a's function updates and this wrapper inherits the new behavior automatically. No changes needed here.

```typescript
// lib/usage/seat-cache.ts
import { Redis } from '@upstash/redis'
import { countActiveAddonSeats } from './seats'

const redis = Redis.fromEnv()
const TTL_SECONDS = 5 * 60
const key = (workspaceId: string) => `addon-seats:${workspaceId}`

export async function getCachedAddonSeatCount(workspaceId: string): Promise<number> {
  const cached = await redis.get<number>(key(workspaceId))
  if (cached !== null) return cached

  const fresh = await countActiveAddonSeats(workspaceId)
  await redis.set(key(workspaceId), fresh, { ex: TTL_SECONDS })
  return fresh
}

export async function invalidateSeatCache(workspaceId: string): Promise<void> {
  await redis.del(key(workspaceId))
}
```

**Cache invalidation is required.** The `customer.subscription.updated` webhook handler (which is already being extended for AC 8 reset logic) MUST also call `invalidateSeatCache(workspaceId)` immediately after the workspace+usage transaction commits. This way:
- Add-on seat purchases reflect immediately in the chat quota check (not after up to 5 min)
- Plan changes between Solo/Team/Enterprise (which change `addonSeatCount` semantics) reflect immediately
- The 5-min TTL is the safety net for missed webhooks, not the primary refresh mechanism

### Reset trigger — Pattern A primary (webhook) + safety cron

The parent spec ([Story 5.5](./5.5.usage-limits-per-tier.md), section "Reset trigger — Pattern A") has the full webhook-handler sketch and safety-cron sketch. Three things to remember when implementing:

1. **The webhook reset and the workspace `current_period_end` write happen in the same `prisma.$transaction`.** They must commit together — if the workspace gets a new period_end but the counter doesn't reset, the user has stale "used tokens" against a fresh period (effectively shrinking their next month's quota by the residual).
2. **`current_period_end` advancing covers both cases:** the trial→paid conversion (NULL → future date) and a normal billing anniversary (future date → later future date). The existing comparison `newPeriodEnd > existing.current_period_end || existing.current_period_end === null` handles both.
3. **The first `WorkspaceUsage` row for a trial workspace is created on the first chat turn, not at workspace creation.** The `onFinish` upsert's `create` branch sets `period_started_at = workspace.created_at` so the trial period is correctly anchored from creation, not from when the user first sends a message. This is important if a user signs up and waits a week before chatting — their trial usage counter still represents the full trial window.

### Usage widget data action

```typescript
// app/actions/usage.ts
import { getWorkspaceContext } from '@/lib/auth/workspace-context'
import { prisma } from '@/lib/prisma'
import { getEffectiveLimits, isUnlimited, tokensToApproxQueries } from '@/lib/usage/limits'
import { getStorageUsage } from '@/lib/usage/storage' // from 5.5b
import { getCachedAddonSeatCount } from '@/lib/usage/seat-cache'

export async function getWorkspaceUsageSummary() {
  const ctx = await getWorkspaceContext()
  const [workspace, usage, addonSeats, members, storage] = await Promise.all([
    prisma.workspace.findUniqueOrThrow({
      where: { id: ctx.workspaceId },
      select: { subscription_tier: true, trial_picked_tier: true },
    }),
    prisma.workspaceUsage.findUnique({ where: { workspace_id: ctx.workspaceId } }),
    getCachedAddonSeatCount(ctx.workspaceId),
    prisma.workspaceMember.count({ where: { workspace_id: ctx.workspaceId } }),
    getStorageUsage(ctx.workspaceId),
  ])

  const limits = getEffectiveLimits(workspace, addonSeats)
  const tokensUsed = Number(usage?.tokens_used_this_period ?? 0n)
  const tokenLimit = limits.aiTokensPerMonth

  return {
    tokens: {
      used: tokensUsed,
      limit: tokenLimit,
      percentUsed: isUnlimited(tokenLimit) ? 0 : tokensUsed / tokenLimit,
      approxQueriesRemaining: isUnlimited(tokenLimit)
        ? null
        : Math.max(0, tokensToApproxQueries(tokenLimit - tokensUsed)),
    },
    storage,
    seats: {
      used: members,
      limit: limits.users,
    },
    periodStart: usage?.period_started_at ?? null,
  }
}
```

### Latency considerations

The chat-route quota check is in the hot path. Measured cost:
- `prisma.workspace.findUniqueOrThrow` (indexed PK) — ~5-10ms
- `prisma.workspaceUsage.findUnique` (indexed `workspace_id`) — ~5-10ms
- `getCachedAddonSeatCount` (Redis hit) — ~5ms
- `getCachedAddonSeatCount` (cache miss → Stripe API + Redis write) — ~150ms
- Pure helper math — < 1ms
- **Cache hit p95 target: < 30ms** ✅
- **Cache miss p99 target: < 250ms** (rare — bounded by add-on subscription events, not workspace count)

Cache miss frequency: with webhook-driven invalidation, misses happen on (a) first chat after a subscription change for that workspace, (b) every 5 min if the safety TTL fires before any subscription event. For an active workspace, expect ~1 miss per ~hundreds of chat turns. The 100-200ms hit is bounded and not user-visible (chat operations are already multi-second).

If observability shows the cache miss rate is high enough to matter (e.g., > 5% of chat turns), revisit: (a) longer TTL (10-15 min); (b) move to a webhook-only invalidation model (TTL = ∞, only invalidated by subscription events).
