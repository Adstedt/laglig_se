> 📌 **Why this exists:** the agent can *act* on the workspace but is **blind to the company's own compliance state**. Its only company-specific reads are `search_workspace_files` (file snippets) and `get_company_context` (scalar profile) — it has **zero** read capability over the relational graph hanging off a `LawListItem`: status, kravpunkter/bevis, business context, past amendment assessments, status history, linked tasks and documents. This is the gap that produced the live miss where the agent said *"inga dokument är direkt kopplade"* while two were linked (it fell back to semantic search). 19.4 adds the **lazy-traversal read tier** — `get_law_list_item`, `get_task`, `list_linked_artifacts` — so the agent reads a node's own state plus *typed handles to its neighbours* and follows edges on demand. **The graph already exists** (Prisma relations); this story *exposes* edges, it does not model new data. It is the flagship of the knowledge-traversal track (19.4a shipped the entry points; 19.4b extends to Epic 21 cycles).

## Story

**As** the compliance agent (and especially the read-mostly AUDITOR persona),
**I want** to read a law-list item's / task's own state plus typed handles to its neighbours, and traverse to linked artifacts on demand,
**so that** I answer "är vi compliant / vad saknas?" from the company's *actual* posture (kravpunkter, bevis gaps, prior assessments, linked docs) — and my write proposals reflect current state instead of guessing.

## Acceptance Criteria

### Shared traversal grammar

1. **`ContextHandle`** added to `lib/agent/tools/types.ts` (one definition, used by every reader so the agent learns one grammar): `interface ContextHandle { id: string; type: 'law_item' | 'task' | 'document' | 'file' | 'requirement' | 'change_assessment' | 'finding'; label: string; count?: number }`. Neighbours are returned as **handles, not fully-hydrated objects** (lazy model). [Source: brief "Standardised handle shape"]
2. **Hard caps (not soft)** on every node's own collections so a pathological item can't blow the context window: comments capped at **≈10** (newest), requirements/assessments/status-log entries capped (≈10 / last ≈5), all long text fields **plain-text + truncated** via the existing `truncateMarkdown` helper (`utils.ts:84`) — never raw HTML. **(SF-1) Apply the caps at the Prisma `take` level** (with `orderBy` newest-first), not by post-fetch slicing — so an item with 500 comments/status-logs never loads them all into memory. [Source: brief "Hard caps"]
3. **Names, not raw user IDs** (brief + 14.29 no-mentions): every reader resolves `responsible_user` / `assignee` / comment author / `changed_by` to the User's **name**; raw user ids are never emitted. [Source: brief "names not IDs"]

### Lazy vs eager (the decision, from the brief)

Eager (return the whole one-hop neighbourhood hydrated) is **rejected** — it dumps a novel into context and the graph's cycles make "load everything" unbounded. Lazy (node fields + `ContextHandle` neighbours, agent follows edges) is the model: the `stepCountIs(10)` loop budget becomes the **traversal budget**. Prefer a *rich enough single reader* (one call suffices for most proposals) over raising the step cap.

### Workspace-isolation note (list_linked_artifacts)

`getLinkedArtifactsForListItem` derives the workspace from the **session** (`withWorkspace`), not a passed `workspaceId`. In the chat request the session === the tool's closure `workspaceId` (the route resolved both from the same `getWorkspaceContext`), so they match. **At implementation, confirm the action rejects a `listItemId` from another workspace** (it `findFirst`s the item + its `law_list.workspace_id`); if it doesn't hard-reject, add an explicit `workspace_id` check in the wrapper before returning. `get_law_list_item`/`get_task` use the explicit-closure-`workspaceId` `findFirst` pattern (cleaner) — no session dependency.
