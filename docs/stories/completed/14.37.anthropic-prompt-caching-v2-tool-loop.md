# Story 14.37: Anthropic Prompt Caching v2 — Tool-Loop Conversation Caching

## Status

## Story

**As a** product owner responsible for chat unit economics,
**I want** the Anthropic chat path to cache the growing tool-loop conversation (not just the system prompt) and to keep the cached system prefix byte-stable across turns,
**so that** multi-step agent turns stop re-billing their accumulated tool results at full input price — raising cache hit rate and cutting per-turn cost enough to make our usage tiers viable.

## Acceptance Criteria

1. **Tool-loop message caching (Fix #1):** On the Anthropic path, a *moving* `cacheControl: { type: 'ephemeral' }` breakpoint is applied to the **last message** on every internal agent step (via `streamText`'s `prepareStep` callback), in addition to the existing system+tools breakpoint from Story 14.26. The growing `tool_use`/`tool_result` history is therefore read from cache on steps 2…N instead of re-billed at full input price.
2. **Frozen cached system prefix (Fix #2):** Per-turn-volatile content — specifically `pendingActionsBlock` — no longer sits *inside* the cached system block. The cached prefix (BASE_PROMPT + company context + skills + reasoning guidance) is byte-stable across a workspace's turns; the volatile block is emitted **after** the cache breakpoint (a separate uncached system segment, or folded into the latest user message).
3. **OpenAI path untouched:** When `modelProvider !== 'anthropic'`, no `cacheControl` and no `prepareStep` caching logic is applied — the OpenAI/`gpt-4-turbo` fallback behaves exactly as before.
4. **Breakpoint budget respected:** No request emits more than Anthropic's max of **4** `cache_control` breakpoints (current: 1 for system+tools; this story adds 1 moving message breakpoint = 2 total).
5. **No regression to 14.26:** The existing system+tools caching continues to work (system prompt + tool schemas still cache-hit within a turn).
6. **Measured improvement (re-scoped v0.4 — within-turn only):** This story's win is the **multi-step (within-turn) loop**. Acceptance bar, verified via `chat_usage_events` + per-step logging:
   - **Within-turn caching engages:** on a multi-step turn, per-step `cache_read_input_tokens` climbs across steps (each step reads the prior step's write). ✅ *Verified in smoke test.*
   - **Comparable multi-step turns cost materially less** vs. the documented baseline (measured **~35–40%** lower: 4-step ~$0.18 → ~$0.11; 5–6-step ~$0.22 → ~$0.16). ✅
   - **Aggregate hit rate improves** from the ~42% baseline (measured **~48%**). ✅
   - **Out of scope (moved to Story 14.38):** the ≥65% target and cross-turn history reuse. Cross-turn caching does **not** land in this story — every turn restarts at the ~27K system block because history is text-projected (see Dev Notes / 14.38). That is the deliberate next-story lever, not a 14.37 failure.
7. **No functional regression:** Chat behavior, multi-step tool use, web-search citations, and the inline agent-action approval cards are unchanged.

### Root cause (confirmed in code)
- `app/api/chat/route.ts` (~lines 355–366) builds `systemMessage` with `cacheControl` on the Anthropic path → system + tools cached. **The `messages` array has no breakpoint.** In a multi-step tool loop (`stopWhen: stepCountIs(10)`, observed avg 3–4 steps), the AI SDK appends each step's `tool_use`/`tool_result` blocks internally and re-sends the whole growing array; without a moving breakpoint those tokens are billed at **full input price every step**. With avg ~3.2 steps and large tool results (search hits, law text, change diffs), this re-billing is the dominant input cost → ~30–40% hit rate.
- `lib/agent/system-prompt.ts` (`buildSystemPrompt`) injects `pendingActionsBlock` into the **middle** of the single cached system string (after company context, before the large skill bodies). When pending actions change between turns, the entire system prefix changes → cache write instead of read, invalidating everything after it (including the large skills). This degrades caching in exactly the action-proposing agentic chats.

### Fix #1 — `prepareStep` (the dominant lever)
- **Confirmed supported** in the installed stack: `ai@6.0.50`, `@ai-sdk/anthropic@3.0.23`. The AI SDK exposes `prepareStep` on `streamText`/`generateText`; the official cookbook (`content/cookbook/05-node/90-dynamic-prompt-caching.mdx`) uses it to mark the last message with `providerOptions.anthropic.cacheControl = { type: 'ephemeral' }` on each step — the canonical moving-breakpoint pattern.
- Helper shape (adapt the cookbook `addCacheControlToMessages`): map messages, set `providerOptions.anthropic.cacheControl` on `index === messages.length - 1` only; apply only for Anthropic.
- Wire as: `streamText({ ..., prepareStep: ({ messages }) => ({ messages: withLastMessageCached(messages) }) })` on the Anthropic path. Keep OpenAI path with no `prepareStep`.
- **Breakpoint-placement caveat:** `prepareStep` receives the full message array for the *upcoming* step, so marking the last element correctly tracks internally-appended `tool_result` messages as the loop grows. Watch the interaction with `attachmentBlocks` (Story 19.1) — those merge into the **last user message** at turn start; the moving breakpoint must still resolve to whatever block is genuinely last on each step, not a stale reference to the original user turn. Verify the merged-attachment turn still caches correctly.

### Fix #2 — keep the cached prefix frozen
- Goal: cached system block = stable content only; no per-turn-volatile string inside the `cacheControl`-marked block.
- **Chosen approach — structured system parts (mandated):** the route passes `system` as an array of text parts — a cached part (BASE_PROMPT + company + skills + reasoning) with `cacheControl`, followed by an **uncached** part containing `pendingActionsBlock`. Requires `buildSystemPrompt` to return the stable prefix and the volatile block separately (e.g. `{ cachedPrompt, volatileSuffix }`). This is the caching-correct option and preserves the block's "system authority" framing.
- **Fallback only (do not use unless the above is unworkable):** injecting `pendingActionsBlock` into the latest user message. Simpler, but it shifts the block out of the system role — acceptable in principle (it is workflow-state context, not a spoofable instruction) but it changes framing and is therefore not the default. If the dev must fall back to this, flag it in Completion Notes for PO review.

### Constraints & guardrails
- **Anthropic max 4 `cache_control` breakpoints/request** — this story uses 2 (system, moving message). Do not add more.
- **Provider-gated:** all changes live behind `modelProvider === 'anthropic'`. The OpenAI fallback path must be byte-for-byte unchanged (AC 3).
- **Cache mechanics reminder:** breakpoint caches the prefix *up to and including* its block; ephemeral TTL is 5 min (fine for active chats); switching models or changing the tool set invalidates the cache — neither changes within a turn here.

### Verification queries (AC 6)

> **⚠️ METRIC CORRECTION (Story 14.38 QA, 2026-06-15):** the original `cache_hit_pct`
> below used the denominator `sum(cache_read_input_tokens + input_tokens)`. That is
> **wrong** — the AI-SDK Anthropic `input_tokens` is the *total billed input,
> inclusive of `cache_read_input_tokens`* (the same fact `route.ts`'s quota carve-out
> relies on). Adding `cache_read` again double-counts it and **caps the metric at
> 50%**, so it can never reflect a real hit rate above 50% (14.38 measured 87% real
> hit while this formula read ~47%). The corrected formula divides by `input_tokens`
> alone. **Both queries below are corrected; re-baseline the 14.37 numbers when
> comparing.** No data migration — the stored columns are correct.

Baseline / post-deploy hit-rate + cost per context (corrected):
```sql
SELECT model, context_type, count(*) AS turns, round(avg(step_count),1) AS avg_steps,
  -- corrected: fraction of billed input served from cache (input_tokens is
  -- inclusive of cache_read_input_tokens, so it is the right denominator)
  round(100.0 * sum(cache_read_input_tokens)
        / nullif(sum(input_tokens),0),1) AS cache_hit_pct,
  round(avg(cost_usd_estimate)::numeric,4) AS avg_usd_per_turn
FROM chat_usage_events
WHERE created_at > now() - interval '30 days'
GROUP BY model, context_type ORDER BY avg_usd_per_turn DESC;
```
