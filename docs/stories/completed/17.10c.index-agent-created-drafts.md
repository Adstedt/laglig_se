# Story 17.10c: Index agent-created document drafts on creation — Brownfield Addition

## Status

Done

## Story

**As a** compliance consultant using the Laglig agent,
**I want** agent-authored document drafts to be indexed for search the moment they are created (not only after I manually open and re-save them in the editor),
**so that** the agent can reliably find existing drafts — to avoid creating duplicates, and to cite them as `[Utkast: …]` — instead of behaving as if they don't exist.

## Story Context

**Origin:** surfaced during Story 19.8 QA (gate `docs/qa/gates/19.8-draft-styrdokument-type-aware.yml`, finding QA-19.8-7). Probe C3 created a duplicate riskbedömning because the existing draft was invisible to `search_workspace_documents`. DB check: agent-created drafts (`Riskbedömning – Truckkörning på lagret`, `Handlingsplan – …`), >1 day old, had **0 chunks** in `content_chunks`. 19.8 mitigated its own duplicate check by switching to `list_workspace_documents` (direct DB), but the underlying indexing gap remains and affects every index-based consumer.

**Existing System Integration:**

- Integrates with: the workspace-document chunk-indexing pipeline (`lib/chunks/workspace-document-reindex.ts` → `indexWorkspaceDocument`) and the document creation path (`app/actions/documents.ts` → `createDocument`).
- Technology: Next.js server actions, `after()` post-commit callbacks, pgvector `content_chunks` (`source_type = 'WORKSPACE_DOCUMENT'`).
- Follows pattern: the existing post-commit `after()` indexing contract. **Two variants exist** — (a) *immediate* re-index `after(async () => { await indexWorkspaceDocument(...) })`, used by the save path (`documents.ts:866`) and `createDraftFromApprovedWithEdit` (`:1417`, rationale comment `:1299`); and (b) *mark-dirty + cron sweep* via `markWorkspaceDocumentDirty` (autosave, `:711`) for high-frequency edits. Status change (`:531`) uses `deindexWorkspaceDocument`. **For a one-time create, variant (a) — immediate index, copying `createDraftFromApprovedWithEdit` — is the right fit.**
- Touch points: `createDocument` (the gap); both agent paths route through it — `openDraftInEditor` and the `DRAFT_DOCUMENT` approve dispatch in `app/actions/pending-agent-actions.ts`.

**Root cause (confirmed):** `createDocument` is the **only** content-creating path that does NOT fire the `after(indexWorkspaceDocument(...))` trigger. A normal user creates a doc *in the editor*, whose autosave (`:711`) indexes it; the agent creates a fully-populated doc directly via `createDocument`, which never enqueues indexing — so an agent draft stays unindexed until a human happens to open and re-save it.

## Acceptance Criteria

**Functional Requirements:**

1. After `createDocument` commits with non-empty content, it enqueues `indexWorkspaceDocument(documentId, workspaceId)` via `after()` (post-commit, non-blocking), matching the existing pattern — so the new draft's chunks land in `content_chunks` without any manual editor save.
2. A document created from a `draft_styrdokument` approval **and** one created via `openDraftInEditor` are both indexed (both route through `createDocument`); verified by `content_chunks` count > 0 shortly after creation.
3. The indexed chunks carry the correct `metadata.tier = 'DRAFT'` (and `status`) so `search_workspace_documents` surfaces them with the right tier and `[Utkast: …]` citation grammar (per 17.10b semantics).

**Integration Requirements:**

4. Existing `createDocument` callers (template-based "Nytt dokument", import) continue to work unchanged; an empty/placeholder-only new doc must not generate spurious chunks (respect the existing `isEmptyTiptapDoc` guard in `indexWorkspaceDocument`).
5. Indexing runs in `after()` so the create response stays fast; failures are logged, never block document creation (same contract as the sibling triggers).
6. No double-indexing: a subsequent editor open + autosave is hash-gated inside `syncWorkspaceChunks` (sha256 of `content_html`), so re-indexing an unchanged doc is a no-op.

**Quality Requirements:**

7. Unit/integration test asserts `createDocument` with real content enqueues the index trigger (mirror the existing reindex-trigger tests).
8. Regression test (or the 19.8 C3 scenario re-run): after creating an agent draft, `search_workspace_documents` for its topic returns it — closing the duplicate-detection + `[Utkast:]` citation gap.
9. No regression in existing chunk-sync / search tests.

## Technical Notes

- **Integration Approach:** add the `after(async () => { await indexWorkspaceDocument(document.id, workspaceId) })` block at the end of `createDocument`'s `withWorkspace` callback, post-`$transaction`, gated on the created version having non-empty content. Copy the error-handling/logging shape verbatim from `createDraftFromApprovedWithEdit` (`documents.ts:~1417`).
- **Existing Pattern Reference:** `createDraftFromApprovedWithEdit` already does exactly this (`app/actions/documents.ts:1302`, trigger at `:~1417`, rationale comment `:~1299`) — the cleanest template. (Note: autosave at `:711` uses the *mark-dirty + cron* variant instead; do **not** copy that one — a one-time create should index immediately, not wait for the cron sweep.)
- **Key Constraints:** keep it in `after()` (don't block the create); respect `isEmptyTiptapDoc` so template stubs with only placeholders don't index noise; rely on `syncWorkspaceChunks` hash-gating to avoid duplicate embeds.
- **Out of scope:** the 14.16 incremental-chunk-sync work and any backfill of *already-created* unindexed drafts (a one-off backfill script could be a separate tiny task; note it but don't bundle).

### Findings

- **QA-17.10c-1 (LOW, advisory):** `isEmptyTiptapDoc` only treats null / no-content / single-empty-paragraph as empty. A template created via "Nytt dokument" carries real placeholder text (section headings + italic "Beskriv…" prompts), so it is **non-empty** → now indexed immediately on create, whereas pre-17.10c it indexed only after the user saved real content. **Impact: low** — headings are useful search signal; placeholder prompts are minor noise and are hash-replaced once filled; the 19.8 duplicate-guard already uses `list_workspace_documents` (DB), not search. **Not blocking.** The dev's "no spurious chunks" completion note is accurate for genuinely-empty docs but overstated for placeholder templates. *If template boilerplate proves noisy in search, tighten the empty-check to meaningful-text detection (e.g. skip docs whose only text is italic-placeholder marks) — gold-plating, defer until observed.*
