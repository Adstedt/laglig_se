## Story

**As a** Laglig.se user asking complex compliance questions (especially change assessments and task-scoped chat),
**I want** the AI to reason through multi-step legal analysis visibly before answering,
**so that** I can trust the conclusion, verify the logic, and catch any flawed assumption before acting on it.

## Context

The UI plumbing for reasoning already exists: `sendReasoning: true` is on in `toUIMessageStreamResponse()` (route.ts:189), and `ReasoningBlock` in `chat-message.tsx:363` renders reasoning parts with collapsible state (`streaming | done`). This story turns on the **model-side** feature via `providerOptions.anthropic.thinking` and tiers the token budget by context stakes. It also adds a complementary system-prompt guidance block so the model self-selects between short answers and deep reasoning within its budget.

**Two levers, not one:** The budget (`budgetTokens`) is a ceiling; the model decides per-response how much to use. Budget alone gives coarse control. The system-prompt "when to reason" guidance (AC 4) is the complementary steering lever — it tells the model which situations warrant deep reasoning vs. a direct answer. Ship both together; budget without prompt guidance leads the model to over- or under-use thinking inconsistently.

## Acceptance Criteria

### Functional Requirements

1. Extended thinking is enabled on the Anthropic model path (`AI_CHAT_MODEL === 'anthropic'`) with a per-context budget map:
   - `change` -> `budgetTokens: 8000`
   - `task` -> `budgetTokens: 3000`
   - `law` -> `budgetTokens: 2000`
   - `global` -> disabled (no `thinking` option passed)
2. When thinking is disabled for a context (i.e. `global`, or `budgetTokens === 0`), the `thinking` option is **omitted entirely** from `providerOptions.anthropic` — not set to `budgetTokens: 0`. This avoids sending an explicit off-config when the OpenAI path is also a possibility.
3. `ReasoningBlock` continues to render reasoning parts correctly (already works via `sendReasoning: true`). For `change` context, the block opens **expanded by default** to emphasize the model's working-out; other contexts remain collapsed by default.
4. **System prompt includes explicit "when to reason" guidance** so the model self-selects between short answers and deep reasoning within its budget. Added to `buildSystemPrompt` in `lib/agent/system-prompt.ts`, only appended when thinking is enabled for the resolved context. The guidance (in Swedish, matching the rest of the system prompt):

   ```text
   Du har utökat resonemang tillgängligt. Använd det när:
   - Du bedömer om en lag gäller för ett specifikt företags situation
     (storlek, bransch/SNI, antal anställda, jurisdiktion)
   - Du väger undantag, tröskelvärden eller villkor i lagen
   - Du sammanställer flera källor (lag + ändringsförfattning + myndighetsföreskrift)
   - Du gör en efterlevnadsbedömning eller riskvägd rekommendation
   - Användaren ska agera på ditt svar (spara, delegera, publicera)

   Hoppa över längre resonemang när:
   - Användaren ställer en ren faktafråga ("vad är SFS 2022:123?")
   - Svaret redan finns i ett dokument du hämtat via search_laws
   - Frågan gäller procedur ("hur skapar jag en uppgift?")
   - Ett direkt citat från primärkällan besvarar frågan helt

   När du resonerar, fokusera på beslutet. Upprepa inte fakta —
   arbeta genom de faktorer som påverkar slutsatsen.
   ```

### Integration Requirements

5. Existing chat flow — tool calls, citation sources, `smoothStream` pacing, follow-mode scroll, `CollapsedToolGroup` — continues to work unchanged.
6. New config follows the existing conditional spread pattern in `streamText()` — same approach used for `experimental_transform: smoothStream(...)`.
7. `stopWhen: stepCountIs(10)` unchanged. `maxDuration` increased from `60` to `90` to accommodate thinking tokens on tool-heavy change assessments.

### Quality Requirements

8. Manual QA covers all four contexts with both simple and complex prompts:
   - `global` — simple query, no reasoning block should appear (thinking disabled).
   - `law` — explain a law, small reasoning block.
   - `task` — scoped compliance task, visible reasoning before answer.
   - `change` — amendment assessment, reasoning block **expanded by default**, clear multi-step chain.
9. **Prompt-steering QA:** within an enabled context (e.g. `law`), compare two queries — a factual lookup ("vad ar SFS 2022:123?") vs. a multi-factor question ("galler arbetsmiljolagen for en visstidsanstalld konsult?"). The factual lookup should show minimal or no reasoning; the multi-factor question should show substantive reasoning. This verifies the prompt guidance is actually shaping behavior, not just the budget.
10. `MEMORY.md` updated with the new thinking config and prompt-steering pattern (in the "AI Chat Streaming & Reasoning" section).
11. No regression in existing chat behaviors (tool orchestration, citation rendering, stick-to-bottom scroll, collapsed tool groups).

### AI SDK `providerOptions` shape

```typescript
providerOptions: {
  anthropic: {
    thinking: { type: 'enabled', budgetTokens: thinkingBudget },
  },
}
```

Note: `budgetTokens` (camelCase) in AI SDK, NOT `budget_tokens` like the raw Anthropic SDK.

### `smoothStream` compatibility with thinking

`smoothStream` (route.ts:177–180) only operates on text stream parts — it buffers and paces the text output. Reasoning parts bypass this transform entirely; they are emitted through the `sendReasoning: true` path in `toUIMessageStreamResponse()` and arrive at the client as separate `reasoning` part types. No changes to `smoothStream` are needed and no interference is expected.

### Interleaved thinking (automatic)

Claude 4.6 Sonnet reasons *between* tool calls when thinking + tools are both active. No explicit configuration needed; it's a natural consequence of enabling thinking on paths that already use `createAgentTools`. Expect reasoning blocks interleaved with tool rows in the UI on `change`/`task` contexts.

### `contextType` threading — why it's needed

`ChatMessage` and `ChatMessageList` do **not** currently receive `contextType`. To make `ReasoningBlock` default-open for `change` context (AC 3), the value must be threaded: `ChatPanel` (already has it) -> `ChatMessageList` -> `ChatMessage` -> `ReasoningBlock`. `ChatSidebar` and `ChatModal` hard-code `contextType="global"` so they can simply omit the prop (collapsed by default). This is the only UI-layer change; it does not affect `CollapsedToolGroup` or any other existing component.

**Type to use:** `ChatContextType` from `@/lib/hooks/use-chat-interface` (line 23: `export type ChatContextType = 'global' | 'task' | 'law' | 'change'`). `ChatPanel` already imports and uses this type. Do NOT use `string` — the union type preserves type safety at the component boundary [Source: architecture/17-coding-standards.md#17.2, "Explicit over implicit"].

### Constraints

- Only applies when `modelProvider === 'anthropic'`; OpenAI path is entirely unaffected.
- Budget caps prevent runaway cost; thinking tokens are billed at the same rate as output tokens.

## Risk and Compatibility Check

**Primary Risk:** Thinking tokens push response past `maxDuration` for tool-heavy change assessments.
**Mitigation:** `maxDuration` raised to 90s. Budget capped per context — highest (`change`) at 8000 tokens ~ 4-6s thinking overhead. System prompt guidance (AC 4) reduces the chance the model uses full budget on simple questions, keeping average latency down.

**Secondary Risk:** Model reasons on *every* query within an enabled context, including simple lookups — wasting budget and adding latency.
**Mitigation:** AC 4 (system prompt "when to reason" guidance) is the direct countermeasure. AC 9 (prompt-steering QA) validates the guidance is actually taking effect.

**Rollback:** Remove the `THINKING_BUDGET` map, the conditional `providerOptions` spread, the `REASONING_GUIDANCE` block, and the `thinkingEnabled` option. Revert `maxDuration` to 60. Remove `defaultOpen` prop from `ReasoningBlock`. Single-PR revertable.
