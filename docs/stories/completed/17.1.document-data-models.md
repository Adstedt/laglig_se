## Story Context

**Existing System Integration:**

- Integrates with: `prisma/schema.prisma` (existing schema with WorkspaceFile, ContentChunk, Task, LawListItem models)
- Technology: Prisma ORM, Supabase PostgreSQL
- Follows pattern: Existing model conventions — uuid PKs, `workspace_id` scoping, `@@map()` for table names, `@@index()` for query patterns, `onDelete: Cascade` from workspace
- Touch points: Adds new models alongside existing ones — no modification of existing tables except additive `WORKSPACE_DOCUMENT` value to `SourceType` enum on `ContentChunk`

**IMPORTANT — Name Collision Avoidance:**

The existing schema already defines:
- `model DocumentVersion` (maps to `"document_versions"`) — used by `LegalDocument` for SFS law versioning
- `enum DocumentStatus` (values: `ACTIVE`, `REPEALED`, `DRAFT`, `ARCHIVED`) — used by `LegalDocument`

Therefore all new DMS models and enums MUST use the `WorkspaceDocument*` prefix to avoid Prisma and database collisions:
- `WorkspaceDocument` (not `Document`) → `@@map("workspace_documents")`
- `WorkspaceDocumentVersion` (not `DocumentVersion`) → `@@map("workspace_document_versions")`
- `WorkspaceDocumentTemplate` (not `DocumentTemplate`) → `@@map("workspace_document_templates")`
- `WorkspaceDocumentTaskLink` (not `DocumentTaskLink`) → `@@map("workspace_document_task_links")`
- `WorkspaceDocumentListItemLink` (not `DocumentListItemLink`) → `@@map("workspace_document_list_item_links")`
- `WorkspaceDocumentType` (not `DocumentType`) — new enum
- `WorkspaceDocumentStatus` (not `DocumentStatus`) — new enum (existing `DocumentStatus` belongs to `LegalDocument`)
- `WorkspaceDocumentVersionSource` (not `DocumentVersionSource`) — new enum

## Dev Notes

**CRITICAL: Existing Schema Conflicts to Avoid**

The schema already has these — do NOT reuse or modify them:
- `model DocumentVersion` → `@@map("document_versions")` (used by `LegalDocument` for SFS law versioning)
- `enum DocumentStatus` with values: `ACTIVE`, `REPEALED`, `DRAFT`, `ARCHIVED` (used by `LegalDocument`)
- `enum SourceType` with values: `LEGAL_DOCUMENT`, `USER_FILE`, `CONVERSATION`, `ASSESSMENT` (only add `WORKSPACE_DOCUMENT`)

All new DMS Prisma model names must use the `WorkspaceDocument*` prefix.

**Existing Model Conventions (follow exactly):**

```prisma
// Pattern from existing models (uuid PK, workspace scoped, @@map, @@index):
model ExampleModel {
  id           String   @id @default(uuid())
  workspace_id String
  created_at   DateTime @default(now())
  updated_at   DateTime @updatedAt

  workspace Workspace @relation(fields: [workspace_id], references: [id], onDelete: Cascade)

  @@index([workspace_id])
  @@map("example_models")
}
```

**Note on `cuid()` vs `uuid()`:** Some older models (`DocumentVersion`, `ContentChunk`, `ChangeEvent`) use `@default(cuid())`. New models should use `@default(uuid())` consistent with the majority convention (`User`, `Workspace`, `Task`, `LawListItem`).

**Self-Referential Version Relation (Prisma named relations):**

The `WorkspaceDocument.current_version_id → WorkspaceDocumentVersion` and `WorkspaceDocumentVersion.document_id → WorkspaceDocument` create a circular reference. Prisma handles this with named relations:

```prisma
model WorkspaceDocument {
  current_version_id String?               @unique
  current_version    WorkspaceDocumentVersion? @relation("WorkspaceDocumentCurrentVersion", fields: [current_version_id], references: [id])
  versions           WorkspaceDocumentVersion[] @relation("WorkspaceDocumentVersions")
}

model WorkspaceDocumentVersion {
  document_id String
  document    WorkspaceDocument  @relation("WorkspaceDocumentVersions", fields: [document_id], references: [id], onDelete: Cascade)
  current_for WorkspaceDocument? @relation("WorkspaceDocumentCurrentVersion")
}
```

**Agent-created versions and `created_by`:**

`WorkspaceDocumentVersion.created_by` stores a user ID string for human authors or the literal `"agent"` for AI-authored versions. **No FK relation to User** — a Prisma `@relation` (even with `User?`) creates a database-level FK constraint that would reject `"agent"` as a non-existent user ID. Resolve the display name at query time instead.

```prisma
model WorkspaceDocumentVersion {
  created_by  String   // User ID or "agent" — NO relation, resolve display at query time
  // No `creator User?` relation — FK would reject "agent"
  // Use source enum (TIPTAP | IMPORT | AGENT) to distinguish authorship type
  // ...
}
```

**Valid Status Transitions (for `updateDocumentStatus`):**

| From | Allowed To |
|------|-----------|
| DRAFT | IN_REVIEW, ARCHIVED |
| IN_REVIEW | APPROVED, DRAFT (send back) |
| APPROVED | SUPERSEDED, ARCHIVED |
| SUPERSEDED | ARCHIVED |
| ARCHIVED | _(terminal — no transitions out)_ |

- Transitioning to APPROVED → set `approved_by = userId` and `approved_at = now()`
- Transitioning away from APPROVED → clear `approved_by` and `approved_at`
- Any invalid transition → throw validation error
