# Story 21.3: Scope selection component (ScopeSelector)

## Status

Done

## Story

**As a** KMA-samordnare creating a compliance audit cycle,
**I want** a stand-alone React component that renders the laglista in the same grouped layout I already know from `/laglistor`, with tri-state checkboxes at the master, group-header and item-row levels,
**so that** I can declare a cycle's scope (all / specific groups / specific laws) without learning a new UI, and the component emits a `scope_definition` payload in the exact shape `createCycle` (Story 21.2) already accepts.

## Acceptance Criteria

### Functional

1. **Component location + exports.** A new client component `components/features/compliance-audit/scope-selector/ScopeSelector.tsx` is created with `'use client'` at file top. It exports:
   - a default React component `ScopeSelector`, and
   - a **type-only re-export** of `ScopeDefinition` from the existing owner module at `@/app/actions/compliance-audit-cycle`. The `ScopeDefinition` type already ships in Story 21.2's code at `app/actions/compliance-audit-cycle.ts:44-47` with the exact shape:
     ```ts
     // Owned by Story 21.2 — import, do NOT redeclare.
     export type ScopeDefinition =
       | { kind: 'all' }
       | { kind: 'groups'; groupIds: string[] }
       | { kind: 'items'; itemIds: string[] }
     ```
     The scope-selector barrel (`index.ts` — see Task 1.2) re-exports it so both import paths work:
     ```ts
     // components/features/compliance-audit/scope-selector/index.ts
     export { default as ScopeSelector } from './ScopeSelector'
     export type { ScopeDefinition } from '@/app/actions/compliance-audit-cycle'
     ```
     Callers may import from either `@/app/actions/compliance-audit-cycle` (primary owner) or `@/components/features/compliance-audit/scope-selector` (component-local convenience) — both resolve to the same type. **The dev MUST NOT declare a second `ScopeDefinition` type** (duplicate types will drift).
   Component props:
   ```ts
   export interface ScopeSelectorProps {
     listId: string                       // LawList id — drives the data fetch
     value?: ScopeDefinition              // optional controlled value (re-opening the wizard)
     defaultValue?: ScopeDefinition       // optional uncontrolled initial value
     onChange: (scope: ScopeDefinition) => void
     disabled?: boolean                   // full read-only (e.g. when re-entering after cycle creation)
     className?: string
   }
   ```
2. **Data source.** The component fetches the same data as the existing `compliance-detail-table` grouped view by calling the two existing server actions:
   - `getDocumentListItems({ listId, page: 1, limit: 500 })` — returns the `DocumentListItem[]` set (see `app/actions/document-list.ts:67-118` for the shape, `app/actions/document-list.ts:355-460` for the action). The 500-cap matches the NFR1 ceiling (Epic 21 section 2) and the 500-item benchmark in IV3 below; paginate in a follow-up story if lists exceed 500.
   - `getListGroups(listId)` — returns `ListGroupSummary[]` (see `app/actions/document-list.ts:1194-1200` + `1373-1421`). Used to render group headers even when a group contains zero items.
   Both calls happen in parallel via `useSWR` with the keys defined in AC 3. The component MUST NOT call any cycle server action or pre-resolve groups/items into cycle items — materialisation is Story 21.4's responsibility and happens server-side only.
3. **SWR keys — isolation from main laglista view.** The component uses **its own** SWR keys, distinct from any key already used by `/laglistor`. Use exactly:
   - `scope-selector:items:${listId}`
   - `scope-selector:groups:${listId}`
   This isolates mutations/revalidations: mounting/unmounting `ScopeSelector` MUST NOT invalidate the main `/laglistor` SWR caches (IV2). The main `/laglistor` surface today does not use these two keys — confirm via grep before writing the component (see Dev Notes → "SWR keys — collision check").
4. **Grouped rendering — visual reuse, code isolation.** The component renders (in DOM order): a master "Välj alla"-checkbox row → one collapsible group panel per `LawListGroup` (sorted by `position asc`) plus an "Ogrupperad" panel at the end if any items have `groupId === null` → inside each panel, a single checkbox header row (`{groupName} ({itemCount})`) and a list of item rows with checkbox + item title + `documentNumber`. Visual tokens (spacing, border, typography) MUST match the existing `components/features/document-list/compliance-detail-table.tsx` group rendering — the component **mimics** the layout, it MUST NOT import from that file, and MUST NOT modify it (AC 7 below). Group panels default to expanded. Collapse/expand toggle is purely cosmetic (does not change selection state). Ungrouped items are rendered under an "Ogrupperad" pseudo-group that is NEVER present in `{kind: 'groups'}` emissions (it has no id).
5. **Tri-state propagation — master ⇄ group ⇄ item.** Every checkbox supports exactly three visual states: `unchecked | indeterminate | checked`, driven by the Radix `data-state` attribute on the existing `components/ui/checkbox.tsx` primitive. Propagation rules are deterministic:
   - **Item checkbox:** `checked` iff the item id is in the selection set; clicking it toggles only that item.
   - **Group header checkbox:** `checked` iff all items in the group are in the selection set; `unchecked` iff none; `indeterminate` iff some-but-not-all. Clicking an `unchecked` or `indeterminate` group-header selects every item in the group; clicking a `checked` group-header deselects every item in the group. (Mirrors the canonical TanStack pattern used at `components/features/document-list/compliance-detail-table.tsx:712-724`.)
   - **Master "Välj alla" checkbox:** `checked` iff all items across all groups are in the selection set; `unchecked` iff none; `indeterminate` otherwise. Click behaviour: `unchecked | indeterminate → checked (select all)`, `checked → unchecked (deselect all)`.
6. **Emitted `ScopeDefinition` shape — exactly one of three shapes.** On every state change, the component calls `onChange(next)` with exactly one of:
   - `{ kind: 'all' }` — iff every item across every group is selected (AC 3 of the epic).
   - `{ kind: 'groups', groupIds: [...] }` — iff selection is a strict subset of all groups AND every selected group has ALL its items selected AND no partial groups exist AND no ungrouped items are selected. `groupIds` excludes the "Ogrupperad" pseudo-group. Order: `groupIds` sorted by group `position asc` (stable, serialisable, test-friendly). Min length: 1. (AC 4 of the epic.)
   - `{ kind: 'items', itemIds: [...] }` — any mixed/partial selection (including any selection that touches the ungrouped bucket). `itemIds` sorted by `(groupPosition asc, itemPosition asc, id asc)` — deterministic for golden-fixture tests. Min length: 1. (AC 5 of the epic.)
   The emission decision is pure selection-state → one-of-three — implement as a single function `deriveScope(selection: Set<string>, groups, items): ScopeDefinition | null` (Task 6), and unit-test it directly via the `__private` export (Task 10.3). If `deriveScope` returns `null` (empty selection), do NOT call `onChange` — `{kind:'items', itemIds: []}` and `{kind:'groups', groupIds: []}` are both invalid per the Zod schema in `app/actions/compliance-audit-cycle.ts:83-93` (`.min(1)` on both arrays).
7. **Live summary — Swedish copy, `aria-live`.** A `<div role="status" aria-live="polite">` rendered below the selector shows one of:
   - Empty selection: `'Inga lagar valda'`
   - All selected: `'Alla {N} lagar valda'` where N is total item count.
   - Groups-only shape: `'{itemCount} lagar valda i {groupCount} grupp(er)'` — use `'grupp'` for `groupCount === 1`, `'grupper'` otherwise.
   - Items shape: `'{itemCount} lagar valda'` (no group count — mixed selection has no clean group mapping).
   Singular/plural handling: `1 lag vald` (not `1 lagar valda`); `2+` → `N lagar valda`. Keep these exact strings literal — they will be asserted in tests.
8. **Existing compliance-detail-table is NOT modified.** The existing `components/features/document-list/compliance-detail-table.tsx` and its row components are not imported, wrapped, re-exported, or touched. Any shared presentation primitives extracted in this story (if the dev judges it worthwhile) must be created as NEW files under `components/features/compliance-audit/scope-selector/` (e.g. `scope-selector/row-primitives.tsx`) and may duplicate styles rather than refactor the original component. The table's behaviour on `/laglistor` is preserved (verified by IV1).

### Accessibility (AC 9 — carried forward from Story 21.1 PO Should-Fix #3)

9. **A11y compliance, explicit.** The component meets the following minimums, each verifiable in tests:
   - Every checkbox has a programmatic label (either a visible `<label htmlFor>` or an `aria-label`).
   - Indeterminate checkboxes set `aria-checked="mixed"` (Radix sets `data-state="indeterminate"` natively — confirm the `aria-checked` mirroring by `@testing-library/react` query `getByRole('checkbox', { checked: 'mixed' })`; if Radix does not set `aria-checked` itself, add it manually via a wrapper).
   - Keyboard navigation: `Tab` moves through the master → each group header → each item row in DOM order. `Space` toggles the focused checkbox. `Enter` on a group-header row toggles its expand/collapse state but does NOT change selection.
   - Focus ring visible on every interactive element (reuse `focus-visible:ring-2 focus-visible:ring-ring` from `components/ui/checkbox.tsx:16`).
   - The live summary uses `role="status"` + `aria-live="polite"` so screen readers announce selection count changes without interrupting keyboard flow.
   - No colour-only state signalling (the tri-state relies on Radix's icon + data-state, not colour alone).

### Testing (AC 10)

10. **Vitest + React Testing Library component tests** at `tests/unit/components/features/compliance-audit/scope-selector/scope-selector.test.tsx`. Minimum coverage:
    - **Tri-state propagation.** Given a fixture of 2 groups × 3 items each: asserting that (a) selecting all items in group A makes group-A-header `checked` and master `indeterminate`; (b) selecting one item in group A makes group-A-header `indeterminate` and master `indeterminate`; (c) clicking `indeterminate` group-A-header selects all its items; (d) clicking `checked` master de-selects everything.
    - **`ScopeDefinition` emission — all three shapes.**
      - Master-on → assert `onChange` last-called with `{ kind: 'all' }`.
      - All items of groups A+B selected (but not ungrouped bucket with items) → `{ kind: 'groups', groupIds: [A,B] }` (sorted by position).
      - 2 of 3 items in group A selected → `{ kind: 'items', itemIds: [...] }` with deterministic sort.
      - A partial selection that includes an ungrouped item → `{ kind: 'items', ... }` (never `{ kind: 'groups', ... }` even if whole groups are selected alongside).
    - **Empty selection guard.** Selecting then deselecting everything does NOT fire `onChange` with an empty `itemIds`/`groupIds`. `{ kind: 'all' }` flip-flop works correctly.
    - **Live summary text.** Three snapshot-style assertions on the exact Swedish strings listed in AC 7 (singular `grupp`, plural `grupper`, `Inga lagar valda`, `Alla N lagar valda`).
    - **SWR key isolation.** Asserted via `vi.mock('swr', ...)` inspecting the first arg to `useSWR` — expect both `scope-selector:items:${listId}` and `scope-selector:groups:${listId}` and nothing else.
    - **A11y.** `getByRole('checkbox', { checked: 'mixed' })` returns the indeterminate node; `userEvent.tab()` sequence matches AC 9; `getByRole('status')` returns the live summary container.
    - **`disabled` prop.** With `disabled=true`, every checkbox has `disabled` attribute; `userEvent.click` triggers no `onChange`.
    - **`value` (controlled) + `defaultValue` (uncontrolled).** Providing `value={{ kind: 'groups', groupIds: [A] }}` renders with group A fully checked + master indeterminate on first render, without calling `onChange` on mount (idempotent hydration).
    - **Truncation banner (AC 2 / Task 2.4).** With the `getDocumentListItems` mock returning `hasMore: true`, the component renders the exact Swedish banner string from Task 2.4 above the master row. With `hasMore: false`, no banner is rendered.
    - **Controlled-mode no-emit on external `value` change (AC 9 / Task 9.4).** Mount with `value={{ kind: 'groups', groupIds: [A] }}` and assert `onChange` was not called. Re-render with `value={{ kind: 'items', itemIds: [...] }}` and assert `onChange` was STILL not called. Only a user-driven click on a checkbox should ever cause an emission.

### Refactoring Performed

- **File**: `components/features/compliance-audit/scope-selector/ScopeSelector.tsx`
  - **Change**: Removed the native `disabled` attribute from the outer `<fieldset>`; replaced with an explanatory comment block. Per-checkbox `disabled={disabled}` threading (already in place) continues to enforce interactivity blocking where intended.
  - **Why**: `<fieldset disabled>` is inherited by every form control + button inside per the HTML spec (step 3 of the fieldset "disabled" attribute algorithm). This blocked the chevron expand/collapse buttons too, breaking Task 7.5's explicit requirement that chevrons remain interactive in disabled mode so users can still inspect the selection.
  - **How**: Styling-only `opacity-60` class retained for visual read-only affordance. Each `TriStateCheckbox` continues to receive `disabled={disabled}` as an explicit prop and short-circuits `onChange` internally. The chevron `<button>` is now untouched by the wrapper.

- **File**: `tests/unit/components/features/compliance-audit/scope-selector/scope-selector.test.tsx`
  - **Change 1**: Added a new dedicated test "keeps group-expand/collapse chevrons interactive even when disabled (Task 7.5)" that mounts the component with `disabled={true}`, asserts the chevron `<button>` is not `:disabled`, clicks it, and verifies the group's child item checkbox disappears (confirming the collapse actually happened).
  - **Why**: Locks in the Task 7.5 contract at the test level so future refactors can't re-introduce the fieldset-disabled regression.
  - **How**: RTL `getByRole('button', { name: /Komprimera grupp: .../i })` + `userEvent.click()` + `queryByRole('checkbox', { name: 'Välj: Lag A1' })` returning null after click.

- **File**: `tests/unit/components/features/compliance-audit/scope-selector/scope-selector.test.tsx`
  - **Change 2**: Split the combined "renders an Ogrupperad panel only when ungrouped items exist" test into two clean `it` blocks (one asserts absence, one asserts presence). Removed the redundant `render()` + `rerender()` sequence that double-mounted the component and relied on an incidental mock-state side-effect.
  - **Why**: The original test coincidentally worked but was fragile — it had two mounted component instances sharing the same SWR mock, so the "Ogrupperad" assertion in the second branch was satisfied by either instance.
  - **How**: Separate fixtures per test; `render()` once per `it`; single `expect` per case.

- **File**: `tests/unit/components/features/compliance-audit/scope-selector/scope-selector.test.tsx`
  - **Change 3**: Renamed the misleading test "returns { kind: 'groups' } with groupIds sorted by position asc" to "short-circuits to { kind: 'all' } when the full-group selection covers every item". Added a one-line inline comment explaining why `selection.size === allItemIds.size` wins before the groups-only branch is reached.
  - **Why**: The assertion verified `{ kind: 'all' }`, not sorting — the name misrepresented what the test actually proved. Sorting is covered elsewhere ("returns { kind: 'groups' } when strict subset of groups fully selected").
  - **How**: Test body unchanged; name + comment now match the assertion.

### Security Review

**PASS.** This is a purely presentational client component. All data flows through pre-existing `withWorkspace(..., 'read')`-gated server actions (`getDocumentListItems` at `app/actions/document-list.ts:372`, `getListGroups` at `:1385`) — no new auth surface. Item titles and group names are rendered via React JSX with automatic HTML escaping; there are no `dangerouslySetInnerHTML` calls, no URL construction from user input, no external network calls beyond the two auth-gated server actions. No PII handling, no secrets, no new environment variables.

### Performance Considerations

**PASS with context.** The 500-item mount benchmark measures at 560ms on jsdom in isolation — well below the story's real-browser target of <500ms (jsdom is measurably slower than a real browser; real-world performance should be better, not worse). Under full-suite concurrency (306 test files running in parallel on the same node process), mount times climb to ~1600ms because jsdom's synchronous DOM operations don't benefit from the multiple cores. The final test ceiling of 3000ms is generous by design — it's a soft guard against egregious N² regressions, not a hard perf wall, matching Story 21.2's Task 9.7 thresholding convention.

Implementation details I verified:
- `useMemo` keys the `grouped` data structure on `[itemsResult, groupsResult]` — no recomputation on selection changes.
- `deriveScope` is O(groups × items_per_group) = O(total items) per render, bounded at 500.
- `getGroupState` and `getItemState` are O(group items) and O(1) respectively.
- Immutable `Set<string>` mutations force React reconciliation only when needed (reference-equality check passes).
- No virtualisation — story-approved; lightweight rows (checkbox + title + documentNumber) render fast without it. If the 500-cap is raised later, virtualisation can be added as a follow-up without touching the state machine.
