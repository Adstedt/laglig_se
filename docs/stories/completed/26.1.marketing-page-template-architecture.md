# Story 26.1: Marketing-Page Template Architecture + MDX Content Surface + Metadata Bundle

## Status

Done

## Story

**As a** marketing/growth lead at Laglig.se,
**I want** a reusable marketing-page template architecture (layout primitives + section components + MDX content surface + metadata/OG/JSON-LD bundle + UTM- and event-instrumented CTAs, including the org-number test),
**so that** adding feature pages, industry pages, and topic pages becomes editorial work (an MDX file + a 30-minute review) instead of an engineering sprint — and every page ships both editorial depth AND conversion devices by construction.

## Acceptance Criteria

### Content surface

1. `@next/mdx` (and its required `@mdx-js/*` peer deps) installed and wired into `next.config`; `gray-matter` (already a dependency) used for frontmatter parsing. `pnpm build` succeeds with MDX enabled.
2. Content folder structure created: `content/marketing/{funktioner,branscher,omraden}/` (folders for `jamfor` and `kundcase` created empty — their schemas ship in 26.9/26.10).
3. Frontmatter Zod schemas in `lib/marketing/frontmatter-schemas.ts` — one schema per template kind (`featurePage`, `industryPage`, `topicPage`), each covering at minimum: `title`, `description`, `heroEyebrow`, `heroTitle`, `heroSubtitle`, `primaryCta` (label + href), `showOrgCheck` (boolean — renders `<OrgCheckCta>`), `relatedCatalogLaws` (array of `{ documentNumber?, slug?, title?, anchor? }`), `relatedPages` (array of internal routes), **required `faq`** (array of `{ question, answer }`, `.min(3)` — every marketing page ships page-specific Q&As; enforced at build time like every other required field), optional `ogVariant`. Schemas exported with inferred TS types.
4. Missing or invalid required frontmatter fails `next build` with an error naming the file and the failing field (build-time validation, not runtime).
5. A documented placeholder/template file `content/marketing/funktioner/_template.mdx` exists with every frontmatter field annotated for future authors; files prefixed `_` are excluded from routing and sitemap.

### Section component library

6. Ten section components in `components/marketing/sections/`, each a Server Component unless interactivity requires otherwise, each with TypeScript-strict props:
   - `<MarketingHero>` — eyebrow + Safiro title + subtitle + CTA slot + optional media slot
   - `<FeatureGrid>` — n-up grid of icon + title + description items (lucide-react icons)
   - `<SplitFeature>` — the Fieldly-style workhorse: alternating media + copy rows (image side alternates per row); media slot accepts `<ScreenshotFrame>` content or a plain photo
   - `<ProofBlock>` — quote/metric/logo proof section
   - `<CtaBlock>` — see AC 11–12
   - `<OrgCheckCta>` — see AC 13
   - `<CatalogLawList>` — **stub in this story**: renders provided entries as a styled list; live `LegalDocument` resolution is Story 26.3. Props signature must already match 26.3's contract (`Array<{ documentNumber?, slug?, title?, anchor? }>`)
   - `<ChangeFeedEmbed>` — typed props + empty/fallback state; live data source is resolved in Story 26.7 (tech-spike note in Dev Notes)
   - `<FaqAccordion>` — accordion of FAQ items (shadcn/ui Accordion); rendered on **every** page (faq frontmatter is required) and the SAME data feeds FAQPage JSON-LD via the metadata helper, guaranteeing markup/visible-content parity (Google requirement)
   - `<RelatedPagesGrid>` — card grid of internal links to sibling marketing pages
7. Three layout primitives in `components/marketing/templates/`: `<FeaturePageTemplate>`, `<IndustryPageTemplate>`, `<TopicPageTemplate>` — each composes the sections in a fixed order with consistent CTA placements (hero CTA, mid-page CTA/org-check, footer-strip CTA) and renders MDX body content in a typographically styled article container.
8. `<MarketingShell>` in `components/marketing/templates/marketing-shell.tsx` composes the **landing-v3 chrome** (`NavbarV3` + `FooterV3`) + breadcrumbs. All marketing pages render inside it. (Decision recorded in epic v0.2: marketing pages share the landing-v3 chrome, NOT `components/shared/navigation/navbar.tsx`.)

### Routes

9. Per-kind dynamic routes (epic Open Q1 resolved per PO recommendation): `app/(marketing)/funktioner/[slug]/page.tsx`, `app/(marketing)/branscher/[slug]/page.tsx`, `app/(marketing)/omraden/[slug]/page.tsx`. Each: reads the matching MDX file, validates frontmatter against its Zod schema, hydrates through its template, returns `notFound()` for a missing/underscore-prefixed slug. `generateStaticParams` walks the content folder so pages are statically generated.
10. The route group `(marketing)` has a `layout.tsx` that applies `<MarketingShell>` (and force-light theme if the landing chrome requires it — see `app/page.tsx` which mounts `ForceLightTheme`).

### Conversion system

11. `<CtaBlock>` accepts a `placement` prop (`'hero' | 'mid-page' | 'footer-strip'`) and emits CTA hrefs tagged `utm_source=marketing&utm_medium=organic&utm_campaign={kind}-{slug}&utm_content={placement}`. UTM construction lives in a pure helper `lib/marketing/utm.ts` (unit-testable).
12. CTA clicks fire `trackEvent('marketing_cta_click', { kind, slug, placement })` via `lib/track-event.ts` (Vercel + consent-gated GA4 — no raw `gtag` calls, no per-call-site consent check).
13. `<OrgCheckCta>` wraps the existing `OrgCheckForm` (`components/features/landing-v3/org-check-form.tsx`, `resultMode="inline"`) in a marketing-section frame (eyebrow + heading + supporting copy props). Rendered when frontmatter `showOrgCheck: true`. Fires `trackEvent('marketing_org_check_submit', { kind, slug })` on successful preview fetch — added via a callback prop on `OrgCheckForm` (new optional prop; default no-op so the homepage is untouched).

### Metadata, OG, JSON-LD

14. `lib/marketing/get-page-metadata.ts` exports `generateMarketingMetadata(kind, slug, frontmatter)` returning a complete Next `Metadata` object: title, description, canonical (`${NEXT_PUBLIC_BASE_URL}/{kind}/{slug}`), openGraph (incl. the generated OG image URL), twitter card. A sibling helper returns the JSON-LD payload(s): `BreadcrumbList` always; `Article` for feature/industry pages; `FAQPage` on **every** page, built from the required `faq` frontmatter (the same array `<FaqAccordion>` renders — single source of truth, so structured data never drifts from visible content). Pages render JSON-LD via the existing `<script type="application/ld+json" dangerouslySetInnerHTML>` pattern from `app/(public)/lagar/[id]/page.tsx`. Purpose: structured Q&A markup helps Google and AI answer engines parse page intent; rich-result chips are not expected (Google restricted FAQ rich results in 2023).
15. OG image generator at `app/(marketing)/og-image/[kind]/[slug]/route.tsx` returns a 1200×630 `ImageResponse`: page title in Safiro on the landing-v3 palette (warm ink on cream, single amber accent as kind marker — no sage/clay foreground), Laglig logo bottom-left. Safiro `.woff2` bundled as a build-time constant (epic Open Q2 resolved per PO recommendation). Response carries long-lived cache headers.

### Imagery support

16. `<ScreenshotFrame>` media primitive in `components/marketing/media/screenshot-frame.tsx`: wraps an image in minimal browser/device chrome (rounded card, subtle border + shadow consistent with landing-v3 surfaces) for in-app screenshots. Used inside `<SplitFeature>`/`<MarketingHero>` media slots and directly in MDX bodies.
17. Image conventions established and documented in `_template.mdx`: assets live under `public/images/marketing/{kind}/{slug}/` (people imagery shared across pages under `public/images/marketing/people/`), WebP preferred, rendered via `next/image` with required `alt`. Frontmatter supports `heroMedia?: { type: 'screenshot' | 'photo', src, alt }` validated by the Zod schemas (extends AC 3). **No image generation happens at build/runtime** — generated people photos are an editorial pre-step (Nano Banana 2 / Gemini image key held by Alexander; assets reviewed + committed; key never in the repo).

### Sitemap

18. `scripts/generate-sitemaps.ts` extended to walk `content/marketing/` and emit one `<url>` entry per published MDX file (lastmod from file mtime, priority 0.8). Underscore-prefixed files excluded. Marketing URLs join the existing chunk pool (no chunking changes; `SITEMAP_CHUNK_SIZE` stays 2 500).

### Quality gates

19. Visual language verified against `_prototypes/marketing-site/*.html` and the live landing-v3 page: Safiro `font-medium` for titles/section labels (never faux-bold), Google Sans Flex for body, near-monochrome warm ink/grays on cream with one sparing amber accent, existing Tailwind tokens only (no new fonts/colors).
20. Unit tests pass for: frontmatter schemas (valid + each missing-required-field case, incl. `heroMedia` and `faq.min(3)` rejection), UTM helper, metadata helper (canonical/OG/JSON-LD shape; FAQPage always present and exactly mirroring the `faq` input; Article for feature/industry), content-walker (underscore exclusion).
21. With a temporary smoke MDX file, the rendered page passes `pnpm lint`, `pnpm typecheck`, and a local Lighthouse SEO check ≥90; the smoke file is removed before story completion (first real content is 26.4).
22. No regression: homepage `/`, catalog routes (`/lagar/[id]`, `/eu/*`), legal pages, and the authenticated app build and render unchanged; `postbuild` sitemap generation still succeeds with the catalog pool.

### Org-number test (the epic's key conversion device)

`components/features/landing-v3/org-check-form.tsx` exports `OrgCheckForm({ className?, eyebrow?, resultMode? })`:
- Two-step client form: org number → optional website URL → POSTs `{ orgNumber, websiteUrl? }` to `/api/public/company-preview`; 404 ⇒ `not_found` error state, non-OK ⇒ `unavailable`.
- Response shape `PreviewResponse`: `{ company: { name, orgNumber, legalForm, address, municipality, sniCode, industry }, areas: string[], areaCount, inferredFlags, companySummary }`.
- Renders a "regelöversikt" payoff card with area badges and a `Testa gratis` CTA to `/signup?org={orgNumber}&url={websiteUrl}`.
- `resultMode: 'inline' | 'modal'` — use `'inline'` inside `<OrgCheckCta>` (mid-page section; layout shift is fine there).
- For AC 13 add an optional `onPreviewSuccess?: (preview) => void` prop (default no-op) — do NOT otherwise modify this component; the homepage hero depends on it.

### Analytics

`lib/track-event.ts` exports `trackEvent(name, props?)` — fans out to Vercel Analytics (unconditional, cookieless) + GA4 via the consent-aware `window.gtag` stub (Consent Mode v2 queues/redacts until `analytics_storage` granted). Client-only; never throws. Event names snake_case. **Note:** this file lands via the `feat/tracking-events` branch — if it hasn't merged when dev starts, coordinate (this story builds on the same branch or after merge). [Source: lib/track-event.ts]

### Sitemap pipeline (NOT app/sitemap.ts)

Sitemaps are **static XML written to `public/` at postbuild** by `scripts/generate-sitemaps.ts` (`package.json` `postbuild: tsx scripts/generate-sitemaps.ts`). Reasons documented in the script header (GSC silently rejected the dynamic metadata-route sitemaps; static files in `public/` are the proven path). Key facts:
- `SITEMAP_CHUNK_SIZE = 2_500` from `lib/constants/sitemap.ts`
- URL mapping via `getUrlPath(contentType, slug)` with an `_exhaustive: never` arm — keep this style for the marketing walker
- Catalog priority 0.7; marketing pages get 0.8; freshness via daily Vercel cron → deploy hook
- Base URL: `process.env.NEXT_PUBLIC_BASE_URL || 'https://laglig.se'`
[Source: scripts/generate-sitemaps.ts; lib/constants/sitemap.ts; package.json]

- **In-app screenshots**: real product surfaces, captured from the app, framed in `<ScreenshotFrame>` (browser/device chrome — see the Fieldly laptop-mockup pattern and `components/features/landing-v3/hero-product-shot.tsx` for the existing in-house treatment of product shots).

Style constraints (for the 26.4+ content briefs, documented in `_template.mdx`): Scandinavian workplace realism, natural light, warm grading compatible with the cream/ink landing-v3 palette, no glossy stock-photo look. Outputs reviewed, optimized to WebP, committed.
- `sharp` is auto-installed by Next.js for `next/image` optimization — no new image deps needed [Source: architecture/3-tech-stack.md#3.1]

- **MDX rendering approach risk (PO note):** `@next/mdx` is designed around imported/`page.mdx` modules; for dynamic routes reading a `content/` folder, the standard pattern is dynamic `import()` of the MDX file inside `generateStaticParams`-driven pages. Do **NOT** reach for `next-mdx-remote` — it is MPL-2.0, which is outside the approved license list [Source: architecture/3-tech-stack.md#3.6]

### Refactoring Performed

- **File**: `lib/marketing/get-page-metadata.ts` (+ 3 route pages, + test)
  - **Change**: Added `serializeJsonLd()` (escapes `<` → `<`) and switched all three marketing routes from raw `JSON.stringify` to it; added a regression test proving `</script>` cannot break out and the payload round-trips losslessly.

  - **How**: Single escaping point in the lib; pages can't forget it; documented "never raw JSON.stringify" in the function doc.

  - **Why**: Three copies of display labels drift; breadcrumbs, JSON-LD and OG images must agree.
  - **How**: Single source in the schemas module (React-free, importable everywhere), consumers updated.
