## Story

**As** the Laglig.se platform,
**I want** Anthropic prompt caching enabled on the chat API's system message on the Anthropic model path,
**so that** multi-turn chat sessions re-use the ~5k-token system prompt from cache at 10% of input cost instead of re-paying full input rate every turn.

## Context

Grounded cost analysis (conversation 2026-04-24 with Sarah/PO) measured the chat route's cold-request input payload at ~9,000 tokens — ~4,900 for the base system prompt loaded from `lib/agent/system-prompt.md`, ~4,000 for tool definitions passed via `tools:`, plus ~150 for injected company context and ~3,000 for `change_context` when present. **None of this is currently cached.** Every turn re-pays full input cost on the entire prefix, inflating per-query COGS by ~40% on multi-turn sessions relative to a properly cached configuration.

This story ships the **minimum viable caching** — a single ephemeral cache breakpoint on the system message, gated on the Anthropic path, with no refactor of `buildSystemPrompt`. Within-workspace caching is captured (a single user's multi-turn conversation, or multiple seats on the same Team workspace with identical company context). Cross-workspace cache sharing (the ~55% vs ~30% savings delta) is explicitly deferred to a future v2 story that would restructure `buildSystemPrompt` to return cacheable parts — not in scope here because (1) at current scale the incremental savings is small, and (2) v2 is cleanly additive and can ship on signal (cache hit rate, workspace count, or COGS threshold).

This story is a **prerequisite for Story 5.5 (usage limits per tier)** — tier prices should be calibrated against the post-caching COGS, not the current uncached COGS, and the separate telemetry story (14.27) will verify the actual cache hit rate before pricing is finalized.

## Acceptance Criteria

### Functional Requirements

1. Anthropic prompt caching is enabled in `app/api/chat/route.ts` on the system prompt via `providerOptions.anthropic.cacheControl = { type: 'ephemeral' }`. The cache breakpoint is placed so that the stable prefix (base system prompt + company_context + optional change_context + assessment_workflow + reasoning_guidance, all joined by `buildSystemPrompt`) becomes cacheable.

2. Cache control is **gated on the Anthropic model path** (`modelProvider === 'anthropic'`). The OpenAI path (`gpt-4-turbo`) receives no `providerOptions.anthropic` fields — byte-for-byte identical behavior to today.

3. `buildSystemPrompt` in `lib/agent/system-prompt.ts` is **not refactored**. It continues to return a single joined string via `sections.join('\n\n')`. Structured-parts refactor for cross-workspace cache sharing is explicitly out of scope (v2, deferred).

5. Tool definitions (the ~4k tokens passed via `tools:`) and `web_search` (Story 14.21) are explicitly **not** the subject of additional caching logic in this story. ### AI SDK v6 considerations

Installed versions (from `package.json` — authoritative):
- `"ai": "^6.0.50"`
- `"@ai-sdk/anthropic": "^3.0.23"`

In AI SDK v6 the `providerOptions` field on `streamText` applies to the overall request. For Anthropic, the `cacheControl` marker is applied to message parts. The simplest pattern that works for system-only caching is passing `providerOptions.anthropic.cacheControl` at the top level — the Anthropic provider then applies it to the system prompt. If runtime verification shows this isn't hitting cache, the fallback pattern is to convert `system: systemPrompt` (top-level string param) to a system `ModelMessage` in the `messages` array with `providerOptions` inline on the message part. Dev agent to pick whichever approach actually produces `cache_read_input_tokens > 0` on turn 2.

Reference: `@ai-sdk/anthropic` README + Anthropic's prompt caching docs (https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching).

### Minimum cacheable size

Claude Sonnet 4.x requires a cacheable prefix of **≥ 1,024 tokens**. The base system prompt loaded from `lib/agent/system-prompt.md` is 16,178 characters ≈ 4,900 tokens in Swedish (~3.3 chars/token). Adding injected company context and any `change_context` takes this comfortably higher. Threshold is not a concern.

### Cache TTL and cost

- Default TTL: 5 minutes, refreshed on each hit.
- Cache write: 1.25× input rate (first request in a new cache window pays this).
- Cache read: 0.1× input rate (subsequent requests within TTL).
- Break-even point: ~2 subsequent reads. Any multi-turn session benefits.

### What gets cached in v1

With `buildSystemPrompt` returning a single joined string, the cache key is effectively the concatenation of:

1. Base prompt (`lib/agent/system-prompt.md` — stable, ~4,900 tokens)
2. `<company_context>` block (per-workspace, ~150 tokens)
3. `<change_context>` block (per-amendment, ~3,000 tokens when present)
4. `<assessment_workflow>` block (stable, `change` context only, ~760 tokens)
5. `<task_context>` block (per-task/law, small)
6. `<reasoning_guidance>` block (stable, when thinking enabled, ~240 tokens)

The cache entry is therefore scoped to `(workspace, context, thinking-enabled)` tuple in practice. Users in the same workspace chatting about the same change/task share cache. Users in different workspaces do **not** share cache — that is the known v1 limitation and the explicit reason a v2 refactor exists on the roadmap.
