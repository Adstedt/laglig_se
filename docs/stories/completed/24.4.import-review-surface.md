# Story 24.4: Import — review surface (full-page, per-row decisions, commit)

## Status

Done

## Story

**As a** workspace member who just had my uploaded list matched against the catalog,
**I want** a focused full-page review surface where I can accept high-confidence rows in bulk, pick from candidates for ambiguous ones, reject false matches, and request catalog additions for unmatched rows — then commit the result as a real `LawList`,
**so that** my Notisum/Lex.nu/consultant Excel becomes a real Laglig laglista that I can use immediately, with a clear handoff for the rows that need human ingestion within the 24h SLA.

This is the **user-facing decision surface and commit flow** — the part where the import pipeline pays off. Generates a real `LawList` and `LawListItem` rows on commit. Not a modal; a focused full-page route.

## Acceptance Criteria

### Route + page shell

1. **New route `app/(workspace)/laglistor/skapa/[importId]/granska/page.tsx`** — server component that:
   - Loads the import with `getImport(importId)` (new server action, see AC 9)
   - Verifies workspace match + status ∈ `{AWAITING_REVIEW, COMMITTED, FAILED}`; redirects to `/laglistor` if not (and import is in earlier states like UPLOADED — those are mid-processing, not reviewable)
   - Renders `<ImportReviewPage>` client component with hydrated import + rows

2. **Page layout** uses Epic 22 primitives (`<PageHeader>`, `<TableToolbar>`, shadcn `<Table>`):
   - PageHeader: breadcrumb (`/laglistor` → `Importera` → filename), title `Granska matchningar`, meta line "{rowCount} rader · uppladdad {timeAgo}"
   - Confidence breakdown card (mirrors prototype frame ④b) below header: 3 stat tiles ("Hög", "Behöver bekräftelse", "Saknas i katalogen") with counts. Each tile is clickable → sets the filter chip below.
   - TableToolbar: filter chips ("Alla / Hög / Behöver bekräftelse / Saknas / Hanterade") with counts; primary action "Bekräfta och skapa lista" (disabled when 0 accepted rows OR import status != `AWAITING_REVIEW`)
   - shadcn `<Table>`: rows virtualised via `TanStack Virtual` for >50 rows

### Per-row UI

3. **Each row is a `<ImportRowCard>` component** — not a true table row but a card-style row to fit the side-by-side source-vs-matched layout. Layout:
   - **Left column (source row, 40% width):** titel, sfs_nummer (if any), område / lagansvarig / kommentar collapsed under "Visa originalrad" expandable
   - **Middle column (matched doc, 40% width):** title + sfs_nummer + document_type pill + "Visa i katalogen ↗" external link. If row is `UNMATCHED`, show "Inget matchande dokument hittades" with a sad-state icon.
   - **Right column (action, 20% width):** confidence pill (`<Badge tone>` from Epic 22: success/warning/danger ramp) + action buttons (see AC 4)

   **ASCII layout reference:**
   ```
   ┌────────────────────────┬────────────────────────┬──────────────┐
   │ SOURCE (40%)           │ MATCHED (40%)          │ ACTIONS (20%)│
   ├────────────────────────┼────────────────────────┼──────────────┤
   │ Miljöbalk (1998:808)   │ Miljöbalk (1998:808)   │ [● Hög]      │
   │ ▾ Visa originalrad     │ SFS · Visa katalog ↗   │ Acceptera    │
   │                        │                        │ Byt ▾ Avvisa │
   └────────────────────────┴────────────────────────┴──────────────┘
   ```

4. **Per-row actions** (visible based on `match_status`):
   - For `MATCHED_HIGH` and `MATCHED_MEDIUM`:
     - **Acceptera** primary CTA → `acceptRow(rowId)` server action → row's `match_status` flips to `ACCEPTED_BY_USER`, `user_decided_at` set
     - **Byt matchning ▾** dropdown → opens candidate picker showing `match_candidates[]` (top 5 from matcher); clicking a candidate calls `replaceRowMatch(rowId, candidateDocId)` → row's `matched_document_id` updated, `match_status` flips to `REPLACED_BY_USER`
     - **Avvisa** ghost CTA → `rejectRow(rowId)` → `match_status` flips to `REJECTED_BY_USER`
   - For `UNMATCHED`:
     - **Begär tillägg** primary CTA with optional note textarea → `requestCatalogAdd(rowId, note?)` → creates `CatalogIngestRequest` row; `match_status` flips to `CATALOG_REQUEST_PENDING`
     - **Avvisa** ghost CTA → `rejectRow(rowId)` → `match_status` flips to `REJECTED_BY_USER`
   - For all `*_BY_USER` and `CATALOG_REQUEST_*` states (already handled):
     - Show muted state with icon (check / X / clock) + "Ångra" link → resets row to its pre-decision state via `undoRowDecision(rowId)` (new server action; see AC 9)

5. **Candidate picker dropdown** — opens on "Byt matchning ▾" click. Renders `match_candidates` as a list:
   ```
   [▼] Byt matchning
       ─────────────────────────────────────────
       AFS 2014:43 Kemiska arbetsmiljörisker        87%
       AFS 2018:1 Hygieniska gränsvärden            72%
       SFS 2008:245 Förordning kemiska produkter    68%
       (+ "Sök efter annat dokument..." link → opens search modal — defer to follow-up; show as disabled "Kommer snart" for now)
   ```
   Click a candidate → calls `replaceRowMatch`, dropdown closes, row updates optimistically.

### Batch action

6. **Primary "Acceptera alla höga" CTA** in TableToolbar (visible only when `pendingHighCount > 0`):
   - Click → confirms via small inline confirmation ("Acceptera {N} höga matchningar utan granskning?") with Acceptera / Avbryt buttons
   - On confirm → calls new `acceptAllHigh(importId)` server action → bulk-flips all `MATCHED_HIGH` rows to `ACCEPTED_BY_USER` in a single transaction
   - Records activity-log: `law_list_import.bulk_accepted_high` with payload `{count}`

### Commit flow

7. **"Bekräfta och skapa lista" CTA** opens a small confirmation dialog (shadcn `<AlertDialog>` or `<Dialog>`):
   - Pre-fills list name from filename (sans extension): `import.filename.replace(/\.\w+$/, '')`. User can edit.
   - Shows summary: "{N} rader läggs till, {M} skickas för manuell registrering, {K} avvisas."
   - **Bekräfta** primary CTA → calls `commitImport({importId, listName})` server action

8. **`commitImport({importId, listName})`** — new server-action implementation (skeletoned in 24.1). Steps:
   - Verify import in workspace + status=`AWAITING_REVIEW`
   - **Verify `LawList` and `LawListItem` field names against `prisma/schema.prisma` before writing the create logic.** The story uses these target field names which the dev MUST cross-check against the actual schema before relying on them: `LawListItem.area_label`, `LawListItem.responsible_user_id`, `LawListItem.notes`. If the actual fields differ (e.g., `category` instead of `area_label`), use the schema's actual names. The mapping intent is "carry over user-supplied per-row metadata to the most-similar existing fields"; specific field names are best-effort guesses pending schema verification.
   - In a Prisma transaction:
     - Create `LawList` row with the given name + workspace_id + created_by
     - For each row in `[ACCEPTED_BY_USER, REPLACED_BY_USER]`:
       - Create `LawListItem` with `law_list_id` + `legal_document_id` (from row's matched_document_id)
       - Carry over `source_omrade` → area / category field; `source_kommentar` → notes / comment field; `source_lagansvarig` → `responsible_user_id` (best-effort lookup, see AC 8.bis below)
     - Update `LawListImport`: `status='COMMITTED'`, `committed_law_list_id` set, `committed_at` set
   - Send activity-log: `law_list_import.committed` with payload `{rowsAdded, rowsRequested, rowsRejected, lawListId}`
   - Trigger email notification: `import-review-ready.tsx` → "Din lista '{name}' är klar att använda" + CTA back to `/laglistor/{lawListId}`
   - Return `{success: true, data: {lawListId}}`
   - Client redirects to `/laglistor/{lawListId}` on success

8.bis. **`responsible_user_id` resolution algorithm** — best-effort match of `source_lagansvarig` string to a workspace member:
   1. Trim + lowercase the source value
   2. Query `prisma.user.findFirst({ where: { workspace_memberships: { some: { workspace_id: ctx.workspaceId } }, OR: [{ email: { equals: source, mode: 'insensitive' } }, { name: { equals: source, mode: 'insensitive' } }] } })`
   3. If exactly one match: assign that user's id
   4. If zero matches OR multiple matches: leave `responsible_user_id` null. The user can manually assign post-import.
   5. The source value is NOT discarded — it's kept on the `LawListImportRow` for audit. (We could optionally surface it as `LawListItem.notes` as fallback, but defer that nuance — for now, unmatched lagansvarig values just fall away from the imported item.)

9. **Two new server actions added to `app/actions/law-list-import.ts`:**
   ```ts
   export async function getImport(importId: string): Promise<...>            // hydrated import + rows + counts; used by page.tsx
   export async function acceptAllHigh(importId: string): Promise<...>        // bulk-accept
   export async function undoRowDecision(rowId: string): Promise<...>         // reset row to pre-decision state
   ```

### Failure / edge states

11. **Empty import (zero rows)** — defensive UI: render "Inga rader att granska" empty state with "Tillbaka till skapa-flödet" link.

12. **Import in `MATCHING` state** (user navigated mid-flight) — render polling loading state ("Vi matchar fortfarande..."). Use SWR with `refreshInterval: 3000` to poll `getImport`. When status flips to `AWAITING_REVIEW`, hard-refresh the page. **Important:** `refreshInterval` should be a function that returns 0 once status is no longer `MATCHING`, to stop polling immediately on transition (SWR pattern: `refreshInterval: (data) => data?.status === 'MATCHING' ? 3000 : 0`). Mirrors the existing pattern in `components/features/dashboard/law-list-generation-progress.tsx`.

13. **Import in `FAILED` state** — render error card with `error_message` from import + "Skapa supportärende" link (mailto for now; future: link to support form).

14. **Concurrent edit protection.** If two users review the same import: optimistic updates client-side, server actions are idempotent (re-accepting an already-accepted row returns success silently). `match_status` is the source of truth.

**Existing patterns to follow:**

- **shadcn `<Table>`** — `components/ui/table.tsx`. - **TanStack Virtual** — see `cycle-items-tab.tsx` for the virtualisation pattern when row count exceeds 50.
- **Optimistic UI** — see existing patterns in `kravpunkter-checklist.tsx` (Story 17.16/17.17) for SWR mutate + rollback.
- **Email send wrapper** — `lib/email/email-service.ts` is the canonical send module (uses Resend SDK; `SendEmailOptions` accepts `{ to, subject, react, ... }`). Existing email templates in `emails/` (e.g., `task-assigned.tsx`, `amendment-notification.tsx`) all use `<LagligEmailLayout>` from `emails/components/laglig-email-layout.tsx` as the wrapper. Mirror that pattern.
- **Activity-log entries** — see `lib/activity/action-constants.ts` for naming convention. Use `law_list_import.row_*` namespace.

**Key decisions baked in:**
- **Per-row actions, not per-row checkboxes.** Acceptance is explicit (Acceptera button) rather than implicit (checkbox + bulk submit). Reason: the cost of a wrong accept is high (a wrong law in your laglista). Explicit ratification per row + bulk only for high-confidence matches.
- **"Sök efter annat dokument..." in candidate picker is deferred** — would require a search modal with the full LegalDocument catalog. Future enhancement; for now, candidates from matching are the only options. Users with no good candidates rely on "Begär tillägg".
- **`responsible_user_id` parsing on commit is best-effort.** If the source `lagansvarig` value matches a workspace member's name or email, link it. Otherwise leave null. The user can assign manually after import.
- **Concurrency:** workspaces with multi-user review need optimistic updates + idempotent server actions. Last-write-wins on conflicts; no merge UI.
- **Commit is one-way.** Once committed, the import becomes a real LawList. Users can edit the LawList post-commit (existing functionality). Re-running the import is via fresh upload.

**Out of scope explicitly:**
- Catalog-requests admin queue UI (Story 24.5)
- `/laglistor/skapa` rewire (Story 24.6)
- Search-for-other-document candidate-picker option
- Real-time collaboration on review (no presence indicators, no cursors)
