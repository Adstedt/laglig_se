## Story

**As a** system component,
**I want** uploaded workspace files (PDF incl. scanned, DOCX, XLSX/XLS, TXT/MD, CSV) ingested into **faithful, structure-preserving markdown** — PDFs via the existing document→Claude pipeline, office/text formats via direct converters,
**so that** the agent can read every uploaded file and 17.9 can chunk + embed them for workspace-wide semantic search.

## Story Context

**Existing System Integration:**

- **Reuses the amendment ingestion pattern.** The real direct call is `parseAmendmentPdf()` at [`lib/external/llm-amendment-parser.ts:57`](../../lib/external/llm-amendment-parser.ts) — `new Anthropic()` + `client.messages.create` with a `type: "document"` / `media_type: 'application/pdf'` block and a retry/backoff loop. It sends the **whole PDF in one call** (see AC 7 — there is no page-splitter to reuse). The prompt `AMENDMENT_PDF_SYSTEM_PROMPT` lives in [`lib/sfs/amendment-llm-prompt.ts`](../../lib/sfs/amendment-llm-prompt.ts) and is **SFS-specific — must NOT be reused**; this story authors a generic transcription prompt.
- HTML→text via `htmlToMarkdown` / `htmlToPlainText` ([`lib/transforms/html-to-markdown.ts:43,503`](../../lib/transforms/html-to-markdown.ts)). ⚠️ This converter has legal-doc special-cases (e.g. `**N §**`→heading at `:487`); they're inert on generic HTML but the dev should eyeball table/heading output on a non-legal sample.
- **Do NOT reuse `validateLlmOutput`** ([`lib/sfs/llm-output-validator.ts:68`](../../lib/sfs/llm-output-validator.ts)): its signature is `(rawOutput, expectedSfsNumber)` and it hard-fails anything lacking `article.legal-document` / SFS scaffolding (`:186-271`) — generic transcription would always fail. This story writes a small generic sanity check instead.
- **Target model:** `WorkspaceFile` ([`prisma/schema.prisma:1290`](../../prisma/schema.prisma)) — has `id`(uuid), `workspace_id`, `uploaded_by`, `is_folder`, `mime_type`, `storage_path`, `category FileCategory`. Add extraction fields (additive). **Skip `is_folder = true` rows.**
- **Upload entry point:** `uploadFile()` in `app/actions/files.ts` — builds `buffer` (`files.ts:335`), enforces `MAX_FILE_SIZE = 25MB` (`:25`) + `ALLOWED_MIME_TYPES` (`:28-39`), uploads to Supabase Storage, calls `prisma.workspaceFile.create`. Wrappers `uploadFileAndLinkToTask`/`uploadFileAndLinkToListItem` delegate.
- **Async via cron** (not inline, not fire-and-forget — Vercel doesn't reliably run post-response work): mirror `app/api/cron/process-chunks/route.ts` (cron-secret auth, `maxDuration = 300`, batch + 30s timeout buffer) and the retry pattern of `retry-failed-pdfs`. `vercel.json` registers crons.
- Technology: `@ai-sdk/anthropic` + `@anthropic-ai/sdk` (installed), `mammoth` (installed), `xlsx`/SheetJS (installed — reads `.xlsx` **and** `.xls`), `papaparse` (installed — CSV). **`exceljs` is NOT installed and is not needed.**

**Out of scope:** styrdokument (`WorkspaceDocument`); chunking/embedding/retrieval (Story 17.9, which consumes this story's `extracted_text`).

## Acceptance Criteria

### Schema

1. `WorkspaceFile` gains (additive migration, hand off for manual apply): `extracted_text String? @db.Text` (canonical **markdown**), `extraction_status FileExtractionStatus @default(PENDING)`, `extracted_at DateTime?`, `content_hash String?` (sha256 of file bytes — dedupe + 17.9 incremental). New enum `FileExtractionStatus { PENDING, PROCESSING, DONE, EMPTY, UNSUPPORTED, ENCRYPTED, FAILED }`.
2. **Backfill guard:** the migration's data step sets **all pre-existing `WorkspaceFile` rows to `UNSUPPORTED`** (not `PENDING`) so the first cron run does NOT mass-LLM-extract the entire historical corpus. A deliberate one-off re-enqueue script (out of scope) can opt historical files back in. Only NEW uploads (AC 3) get `PENDING`.

### Supported types + routing

3. On upload, `uploadFile` sets `extraction_status` for **non-folder** rows by mime-type: supported → `PENDING`; any other *allowed* type → `UNSUPPORTED`. (We do not change `uploadFile`'s existing hard-reject of types outside `ALLOWED_MIME_TYPES`; the matrix governs allowed-but-maybe-not-extractable types.) Supported set:
   | Type | Front-end | → output |
   |---|---|---|
   | PDF `application/pdf` | Claude `document` block (text-layer **and** scanned/OCR) | semantic HTML → markdown |
   | DOCX `…wordprocessingml.document` | `mammoth.convertToHtml({ buffer })` | HTML → markdown |
   | XLS/XLSX `…spreadsheetml.sheet`, `application/vnd.ms-excel` | `xlsx`/SheetJS (reads both) | markdown tables |
   | CSV `text/csv` | `papaparse` | markdown table |
   | TXT `text/plain`, MD `text/markdown` | direct | markdown (md as-is; txt escaped) |
4. **MIME allowlist task (required):** `text/plain`, `text/markdown`, `text/csv` are NOT in `ALLOWED_MIME_TYPES` today (`files.ts:28`) — add them so txt/md/csv can be uploaded + ingested. Legacy `.doc` (`application/msword`), PowerPoint (`.ppt`/`.pptx`), and images (png/jpeg/gif) remain **allowed uploads but `UNSUPPORTED` for extraction** (no extractor v1). Standalone-image OCR + PPTX are explicit future follow-ups.

### PDF via the Claude document pipeline

5. PDF extraction reuses the `parseAmendmentPdf` `type:"document"` call shape (incl. its retry/backoff), with a **new generic prompt** (`lib/documents/file-extraction-prompt.ts` → `FILE_TRANSCRIPTION_PROMPT`): *transcribe the document faithfully and in full to clean semantic HTML — preserve all text, headings, lists, tables; never summarise, omit, paraphrase, or invent.* Output passes a **new generic sanity check** (`validateExtraction(html)` — non-empty, plausible length vs. page/byte size, not obviously truncated mid-tag); on failure → `FAILED`. **Do NOT call `validateLlmOutput`.**
6. Scanned/image PDFs are handled by the same path (Claude OCRs the document block — no separate OCR engine/rasterisation).
7. **Oversized PDFs:** Anthropic's PDF document block is limited (~100 pages / 32 MB per request). The 25 MB upload cap bounds size, but a ≤25 MB PDF can still exceed ~100 pages. There is **no existing page-splitter to reuse** (`parseAmendmentPdf` sends the whole buffer). v1 behaviour: attempt the call; if Anthropic rejects on size/pages, mark `FAILED` with a clear reason. Page-range splitting + per-batch concatenation is **explicit net-new future work**, not in this story.
8. Transcription runs on **`claude-haiku-4-5`** (cost — verified in use across the repo, e.g. `lib/import/grouping-llm.ts`).

### Storage representation

9. Per-format HTML → markdown via `htmlToMarkdown`, stored in `WorkspaceFile.extracted_text`. Raw HTML transient (not persisted). (Uploaded files are private + never rendered as pages, so the legal-doc html/json multi-representation is unnecessary.)
10. Truncate at 200 000 chars on a **clean block boundary**: cut at 200k, then back up to the last preceding blank-line / heading / completed-table-row (never mid-table/mid-heading — that leaves 17.9 a malformed final chunk); flag truncation in `metadata`.
11. Terminal status: success → `DONE` + `extracted_at`; no extractable text → `EMPTY`; encrypted/password PDF → `ENCRYPTED`; thrown error → `FAILED`. File always remains stored; errors logged, never surfaced as an upload failure.

### Async processing + lifecycle

12. Cron `app/api/cron/extract-files/route.ts`: **runs global-scope** (no session) using the **service-role storage client** + a direct `prisma` query for `PENDING` non-folder files (cron-secret auth header, `maxDuration` + 30s timeout buffer like `process-chunks`). **Idempotent claim:** `updateMany({ where: { id, status: PENDING }, data: { status: PROCESSING } })` and only proceed for rows it actually claimed (check the returned `count`). NB this atomic-claim is **net-new** — `process-chunks` is idempotent by re-query, not by a status claim, so don't expect to copy it. On `DONE`, hand off to 17.9's `syncWorkspaceChunks(..., 'USER_FILE', ...)` (see 17.9 Task 3) — guard/stub the call so 17.8 can land + pass tests before 17.9 exists.
13. **Schedule + latency:** register a schedule in `vercel.json` (e.g. every 10 min, tighter than `process-chunks` hourly). Searchability is eventually-consistent (extraction run → then 17.9 chunking) — acknowledge the two-hop lag.
14. **Retry / stuck rows:** a row left `PROCESSING` past a staleness window (cron timed out mid-file) is re-enqueued to `PENDING`; `FAILED` rows are retried up to N times (mirror `retry-failed-pdfs`). Track an attempt count in `metadata`.
15. `content_hash` computed at upload; a re-uploaded/replaced file with an unchanged hash skips re-extraction (stays `DONE`). Consumed by 17.9 dedupe.
16. `extractWorkspaceFile(fileId)` server action (`withWorkspace` — manual/admin re-trigger): re-download via `storage_path`, re-run, update. (The cron is the bulk path; this is the scoped single-file path.)

### Cost + faithfulness guardrails

17. Faithfulness is non-negotiable: verbatim prompt (AC 5) + `validateExtraction` gate. No summarisation/omission.
18. Per-workspace ingestion cost tracked via `ChatUsageEvent` (`prisma/schema.prisma:1496`): add a `FILE_EXTRACTION` value to `ChatContextType` (`:1480`, enum migration — fold into Task 1) and write the row with `user_id = WorkspaceFile.uploaded_by` (the model column is NOT NULL; the cron resolves the uploader). Surfaces in the AI-usage dashboard; quota-gating deferred to 5.5b.

### Integration

19. Existing upload UX/flow unchanged — extraction is a transparent background step; folders (`is_folder`) are never extracted.

## Dev Notes

- **Why the LLM pipeline, not `pdf-parse`:** reuses a battle-tested PDF→Claude→markdown path → scanned-PDF OCR for free + structure-preserving markdown (better 17.9 chunks). Per-doc cost acceptable at upload volume.
- **Don't LLM the office/text formats** — DOCX/XLS/XLSX/CSV/TXT/MD are already structured; convert directly (faithful + free). The LLM step is PDF-only.
- **Faithfulness:** an LLM can paraphrase/omit; verbatim prompt + `validateExtraction` gate; FAILED-on-validation beats a hallucinated clause.
- **Cron, not fire-and-forget:** Vercel won't reliably run post-response work; the `process-chunks` cron is the established queue. (17.9 follows the same rule — it triggers file chunking from this cron, not from a server action.)
- **`SourceType.USER_FILE`** (`schema:1801`) is what 17.9 stamps on file chunks — no enum work here.
- **17.8 stores markdown text only** — per-block `content_role` assignment belongs to 17.9 (don't add role logic here).
