## Story

**As a** compliance officer (or any user reading a styrdokument),
**I want** the styrdokument table and doc page to show me — at a glance — both the currently-effective approved version AND any draft in progress, with clear audit metadata throughout,
**so that** I never lose sight of "what's in force right now" during revision windows, and so the dual state is operationally obvious without me having to dig through version history.

## Acceptance Criteria

### Composite status badge (table)

1. **`document-status-badge.tsx` (or a new `dual-status-badge.tsx` wrapper) renders the composite for the dual state:**
   - When the doc has `current_approved_version_id` set AND `current_draft_version_id` set: render `Godkänd v{N} · Utkast v{N+1} pågår` (where `{N}` is the approved version_number and `{N+1}` is the draft version_number). Both pieces are styled `Badge` instances; the separator is a non-breaking middot. Status tone follows existing conventions: approved = `success` (or current shipped tone for APPROVED); draft = `neutral` (or current shipped tone for DRAFT).
   - When the doc has ONLY `current_approved_version_id`: render today's APPROVED badge (unchanged).
   - When the doc has ONLY `current_draft_version_id` AND `draft_status = 'DRAFT'`: render today's DRAFT badge (unchanged).
   - When the doc has ONLY `current_draft_version_id` AND `draft_status = 'IN_REVIEW'`: render today's IN_REVIEW badge (unchanged).
   - When the doc is `status = 'ARCHIVED'` or `status = 'SUPERSEDED'`: render today's badge (unchanged).
   - **v1.1 (smoke-driven redirect):** the composite badge's two halves are **clickable navigation regions** (not static labels):
     - Left half `Godkänd v{N}` → navigates to the doc's editor route with a `?view=approved` query param, opening the **read-only approved view** of v{N}. Useful when an auditor or non-revising user needs to see "what's gällande just nu" without entering the in-progress draft.
     - Right half `Utkast v{N+1} pågår` → navigates to the editor's default route, which under Story 17.16 AC 13 loads the draft version (`current_draft_version.content_json`) for editing.
     - Single-state badges (approved-only, draft-only, in-review-only, archived, superseded) remain clickable as a single link to the doc's default editor view (today's behavior, just now explicitly part of the spec).
     - Each clickable region is a proper anchor (`<a href>`) with its own `aria-label` — e.g., left = `"Öppna godkänd version 3 av Arbetsmiljöpolicy (läsläge)"`, right = `"Öppna utkast version 4 av Arbetsmiljöpolicy i editorn"`. Screen-reader users get distinct navigation affordances per half.
     - **Tab order:** approved half tabbed first, draft half second, then the row's other interactive elements. Keyboard click via Enter/Space follows the standard anchor pattern.
     - **Hover affordance:** subtle underline + cursor-pointer on hover to make the clickability discoverable; the parent row's existing hover treatment continues to apply for clicks landing outside the badge halves.
   - The composite badge container still exposes a `role="status"` `aria-label` reading the **full state** (`"Godkänd version 3, utkast version 4 pågår"`) so screen-reader users who don't tab through the navigation regions still get the complete state in their region listings.
   - **CP-001:** the badge NEVER renders raw identifiers (no `documentId`, no `versionId`) — only natural Swedish + version numbers.

2. **Table column update (`document-table.tsx`):** the existing status column uses the new dual-aware badge (no column header rename needed). Sort behavior on the status column unchanged. Row visual height stays consistent (composite badge stacks within the existing cell padding without forcing a row-height bump).

### Filter chips + list semantics

3. **`document-filters.tsx` (or equivalent) — "Visa godkända" filter:** matches rows where `current_approved_version_id IS NOT NULL`. Under the new model this includes APPROVED-with-draft-pending rows. The filter pill copy stays `"Godkända"` (no UI rename); behavior change is invisible to the user except that they no longer see policies vanish from the "Godkända" filter mid-revision.

4. **"Visa utkast" filter:** matches rows where `current_draft_version_id IS NOT NULL`. Under the new model this includes APPROVED-with-draft-pending rows (the same row appears under both "Godkända" AND "Utkast" filters — by design, because the doc IS both at once).

5. **"Senast ändrad" column:** sourced from a derived `last_meaningful_change_at` value: `MAX(current_draft_version.created_at, current_approved_version.approved_at, document.updated_at)`. So a stable APPROVED doc shows its last approval date; an in-revision doc shows its most recent draft save. Existing column header copy unchanged.

### Doc page header

6. **Doc page header (`components/features/documents/document-header.tsx` or per dev's locator):**
   - When the doc has `current_approved_version_id` set AND `current_draft_version_id` set: render the composite badge (per AC 1) PLUS a secondary metadata row: `Godkänd v{N} av {approver.name} den {approved_at, sv-locale-formatted}` from `current_approved_version`'s per-version timestamps (Story 17.16 AC 2). Under the composite badge, render a smaller indicator: `Utkast v{N+1} pågår — senast ändrat den {current_draft_version.created_at}`.
   - When only `current_approved_version_id` set: render today's header (approved metadata + standard badge — unchanged from current behavior, modulo 17.16 fixing the metadata-wipe bug).
   - When only `current_draft_version_id` set AND `draft_status = 'DRAFT'`: render today's DRAFT header.
   - When `draft_status = 'IN_REVIEW'`: render the IN_REVIEW header PLUS — if `current_approved_version_id` is also set — the approved metadata indicator below (so approvers know what they're replacing).
   - **CP-001 holds:** no raw IDs in the header copy.

7. **"Godkänn utkast" / "Skicka för granskning" actions in the header:** when the user has **`tasks:edit`** (SM v1.2 rigor pass: `documents:approve` does NOT exist in `lib/auth/permissions.ts` — only `documents:add`, `documents:remove`, `tasks:edit`; the v1.1 "fallback" phrasing was misleading, since there is no choice. Same SF-1 fix that landed in 17.16) AND `draft_status = 'IN_REVIEW'`, surface a primary "Godkänn utkast" button that calls `promoteDraftToApproved(documentId)` (17.16 AC 6). When `draft_status = 'DRAFT'`, surface a "Skicka för granskning" button that updates `draft_status` to `IN_REVIEW` (small new wrapper around `updateDocumentStatus`, or new server action — dev's call). When no draft is in progress AND the doc is APPROVED, today's header is unchanged.

### Editor banner

8. **Editor banner (`components/features/documents/editor/...`):** when the editor opens a doc with `current_draft_version_id` set AND `current_approved_version_id` set, render a banner above the editor area:
   - Copy: `Du redigerar Utkast v{N+1} som ersätter Godkänd v{N} efter godkännande.` ({N} = approved version number)
   - Icon: `lucide-react` `GitBranch`.
   - Visual: tinted-amber background, slate-text-on-amber, soft border. Non-dismissable (always visible while editing a draft).
   - Right side of banner: "Förkasta utkast" button (icon `Trash2`, ghost variant) — opens a confirmation `AlertDialog` (`"Vill ni förkasta utkast v{N+1}? Den godkända versionen v{N} förblir gällande."`) and on confirm calls `discardDraft(documentId)` (17.16 AC 7). On success, navigate back to the doc page or close the editor (dev's call based on the existing UX flow).
9. **No banner when the doc has only one pointer set** (never-approved DRAFT — banner-less, today's editor experience; or stable APPROVED — editor shouldn't even be reachable in the new model without first branching a draft).

### Approval card "Öppna dokument" link routing

10. **`update-document-renderer.tsx` + `add-document-section-renderer.tsx` (Stories 17.11 / 17.11b):** the "Öppna dokument" link in the APPROVED state of the cards uses `result_ref.documentId` to route to `/workspace/styrdokument/{documentId}/edit`. Under the new model, that URL must land the user on the draft version's editor view (since the agent's change was applied to the draft pointer). **SM v1.2 rigor pass correction — the "deprecated alias works transparently" assumption from v1.1 is INCORRECT post-17.16 v2.1.** The editor route (`app/(workspace)/workspace/styrdokument/[documentId]/edit/page.tsx:97-102`) loads the draft via 17.16 AC 13's **explicit fallback chain**: `current_draft_version.content_json ?? current_approved_version.content_json ?? current_version.content_json` (alias) `?? <empty doc>`. The alias is FROZEN on the approved version throughout the draft window (17.16 AC 4 + AC 5 + AC 11), so reading through it would silently load approved content — the draft pointer is read FIRST by explicit design. No code change required; this AC is a regression-test checkpoint against 17.16 AC 13's read order (NOT against any alias-transparency property).

17. **Versionshistorik "Återställ" button — Model-B-aware error handling (SM v1.2 rigor pass, new in v1.2):** the existing `VersionHistoryPanel`'s "Återställ" action calls `restoreDocumentVersion` (`app/actions/documents.ts:894`), which under Story 17.16 v2.1 is now Model-B-aware with Path A/B/C/D routing.     - **Path B — never-approved DRAFT**: restore succeeds, both pointers advance (today's behavior).
    - **Path C — APPROVED with no draft**: restore REFUSES with Swedish error `"Det godkända dokumentet kan inte ändras direkt.     - **Path D — ARCHIVED / SUPERSEDED**: restore REFUSES with Swedish error `"Dokumentet är arkiverat eller upphävt och kan inte återställas. Återaktivera dokumentet först."` UX: same pattern as Path C — disable the button with hint `"Återaktivera dokumentet först"`.
    - **Test coverage:** add unit tests for the panel's Återställ button state under each of the four document states (dual-state → enabled; never-approved → enabled; APPROVED-no-draft → disabled+hint; ARCHIVED/SUPERSEDED → disabled+hint). Existing `tests/unit/components/features/documents/version-history-panel.test.tsx` test file is the natural extension point.

## Risk and Compatibility Check

**Primary Risk:** The composite badge introduces a visual element that doesn't exist today. Edge cases (very long version numbers like `v123`, very narrow table columns at small viewports, RTL languages — not currently a concern but worth noting) could cause layout breakage in the table row.

**Mitigation:**
- Composite render uses `whitespace-nowrap` on each badge half and `truncate` on the surrounding cell — single-line guarantee at table density.
- Storybook (or test fixtures) cover extreme version numbers and narrow viewports.
- At very narrow viewports (mobile), the composite collapses to the approved badge only (with a small "+" indicator or icon to signal draft pending) — keeps the row legible. **Note for dev:** confirm this narrow-viewport treatment with PO before shipping; the default in this story spec is "always show composite" with the collapse as a stretch goal.

**Rollback:**
- All UI changes; rollback is straightforward (revert the components). No data migration tied to this story.
- If the composite badge causes visual regressions in production, feature-flag it (`dual_version_badge_v1`) and fall back to the single-status badge until the issue is resolved. The underlying schema + data layer remains correct because 17.16 owns those.

**Compatibility Verification:**
- [ ] No breaking changes to existing component APIs (`DocumentStatusBadge` keeps its existing prop shape; the dual variant is an internal render decision based on the doc's pointer state).
- [ ] Database changes — none in this story.
- [ ] UI changes follow existing design patterns (`Badge` primitive from `components/ui/`, Epic 22's tone matrix where applicable).
- [ ] Performance impact is negligible (no new DB queries; render-time composition only).
