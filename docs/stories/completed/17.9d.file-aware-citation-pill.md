# Story 17.9d: File-aware Citation Pill — Passage Preview + Open Uploaded-File Sources

## Status

Done

> 📌 **Why this exists (created 2026-05-22, rescoped 2026-06-12):** [[17.9c]] shipped `search_workspace_files` and made `[Källa: <filnamn>]` markers **resolve** to a citation pill — but clicking that pill still routes through the legal-document path: it opens the sidebar with **legal copy** ("Fullständig lagtext visas i lagläsaren") and is a **dead-end** (no link to the file), because file sources have no `path`/`slug`/`url` discriminator and fall through to the law fallback. Meanwhile [[17.10]] shipped the **styrdokument** pill pattern we like: a presence-of-id discriminator (`workspaceDocumentId`) that gives the pill an icon + an **"Öppna styrdokument"** link to the editor, no sidebar involved. **This story gives uploaded-file pills the equivalent affordance** — a working pill that opens the actual file — by mirroring the styrdokument pattern with a `fileId` discriminator. **The pill's hover card also previews the cited passage** (the snippet from the search hit) so the user can verify at a glance before opening — the way SFS citations point you at the section, not the top of the law. The preview is scoped to the file discriminator, so the shared legal snippet gate is never touched.

## Story

**As a** user reading an AI answer grounded in my workspace's uploaded files,
**I want** a `[Källa: <filnamn>]` pill that previews the cited passage and opens the actual document,
**so that** I can verify and trace the answer to its source — the way styrdokument and SFS citations let me check the original.

## Story Context

**Existing System Integration:**

- **Producer (Story 17.9c):** `search_workspace_files` (`lib/agent/tools/search-workspace-files.ts`) returns per result `fileId` (= chunk `sourceId`), `filename`, `snippet`, `citationKey` (= filename). The citations branch in `extractSourcesFromToolResult` (`lib/ai/citations.ts:197`) currently maps `filename → SourceInfo` with `documentNumber`/`title` = filename and `snippet` = the hit snippet, but **drops `fileId`**, and the pill **suppresses the snippet** because files have no `path`. This story carries `fileId` through (to open the file) and surfaces the already-mapped `snippet` in the hover card.
- **The pattern to mirror — styrdokument pills (Story 17.10, Done):** `SourceInfo` (`lib/ai/citations.ts:15`) already carries `workspaceDocumentId?` + `tier?`. In `citation-pill.tsx`, `isWorkspaceDocResolved = !!workspaceDocId` (`:101`) drives: a `FileText` icon on the pill trigger (`:209`) and an **"Öppna styrdokument" `<Link>`** in the hover card body (`:245–261`) that navigates to `/workspace/styrdokument/{id}/edit`. **It does not touch the sidebar at all.** This is the shape the file pill copies, with `fileId` as its discriminator.
- **Snippet gating today:** the hover card's preview text is `description = isChunkLevel ? source.snippet : null`, where `isChunkLevel = !!source.path` (`citation-pill.tsx:194`). Files have no `path`, so their snippet is suppressed. This story adds a **separate, file-scoped condition** (`isFileResolved`) rather than widening `isChunkLevel` — legal doc-level suppression stays exactly as-is.
- **The dead-end to fix:** file pills have no `url`/`slug`/`path`, so (a) the trigger gets no icon, (b) there's no open affordance, and (c) `handleClick` (`citation-pill.tsx:124`) falls through to `chatDetail.openDetail(...)` → the legal "DB source" branch in `citation-detail.tsx` (`:88`). Web sources already avoid this by `window.open`-ing and `return`ing early in `handleClick` (`:127`); files route away the same way.
- **File open (reuse target):** `QuickPreview` (`components/features/files/quick-preview.tsx:36`) — a self-contained modal overlay (props: `file`, `files`, `open`, `onClose`, `getFileUrl`). Signed-URL action `getFileDownloadUrl(fileId)` (`app/actions/files.ts:1303`, 1h URL).

**Out of scope:**
- **The sidebar (`citation-detail.tsx`)** — file pills route away from it (like web pills); the passage preview lives in the **hover card**, not a sidebar branch.
- **Widening the shared legal `path` snippet gate** — the file snippet is gated on `isFileResolved` (`!!fileId`), a *separate* condition from `isChunkLevel` (`!!source.path` `:194`). Legal/web/styrdokument snippet behavior is untouched.
- **`WORKSPACE_DOCUMENT`/styrdokument citations** — already shipped (17.10); we mirror, not modify, that pattern.
- Any change to the `search_workspace_files` tool's result shape.

## Acceptance Criteria

### Carry the file identity through

1. `SourceInfo` (`citations.ts:15`) gains an optional, additive field: `fileId?: string | null`. The 17.9c `search_workspace_files` branch in `extractSourcesFromToolResult` (`:197`) sets `fileId` (from `item.fileId`) **in addition to** the existing filename/snippet mapping. Legal, web, and styrdokument branches are untouched. (No `kind` union — match the established presence-of-id discriminator the styrdokument path uses.)

### File pill resolves, previews, and opens (mirror the styrdokument pattern)

2. In `citation-pill.tsx`, add `isFileResolved = !!source?.fileId` (mirroring `isWorkspaceDocResolved` `:101`). A file-resolved pill gets a **`Paperclip` icon** on the trigger (`:209` region). Use `Paperclip` specifically (not `FileText`) — styrdokument pills already use `FileText` (`:209`), so the uploaded-file pill needs a distinct glyph to stay visually distinguishable from both styrdokument and legal pills.
3. **The hover card previews the cited passage for a file-resolved source.** Widen the preview so `description` shows `source.snippet` when `isFileResolved` (in addition to today's `isChunkLevel`), e.g. `(isChunkLevel || isFileResolved) ? source.snippet : null`. The hover card shows **filename (title) + the snippet**. The legal doc-level suppression (`isChunkLevel` alone) must be unchanged for non-file sources.
4. The hover card body shows an **"Öppna filen"** affordance for a file-resolved source — visually mirroring the "Öppna styrdokument" `<Link>` (`:245–261`) — that opens the file for `fileId`:
   - **Preferred:** mount `QuickPreview` (`quick-preview.tsx:36`) — fetch the `WorkspaceFileWithLinks` by `fileId` and wire `getFileUrl` to `getFileDownloadUrl`. Avoids the popup-blocker problem below.
   - **Acceptable fallback (DEC for Dev):** open the `getFileDownloadUrl(fileId)` signed URL in a new tab. **Popup-blocker caveat:** the URL is fetched async, so a post-`await` `window.open` may be blocked — if taking this path, open a blank tab synchronously then set its `location`, or accept the limitation. Document the choice in the Dev Agent Record.
5. Clicking a file pill **does not open the legal sidebar.** In `handleClick` (`:124`), add a file branch that opens the file (per AC 4) and `return`s early — mirroring the web-source early return (`:127`) — so a file source never reaches `chatDetail.openDetail(...)` / the "lagläsaren" copy.

### Non-regression

6. **Legal (SFS), web, and styrdokument citations are behaviourally unchanged.** The `isChunkLevel` gate is **not** widened — the file preview is a separate `isFileResolved` condition. Both the snippet preview (AC 3) and the open/route logic (AC 4–5) must be strictly gated on `!!fileId` so they never intercept a non-file source. Verify against the existing `tests/unit/lib/ai/citations.test.ts` + a render check of the legal path (doc-level legal source still suppresses its snippet and still routes to the sidebar).

### Quality

## Dev Notes

- **Discriminator = presence of `fileId`, not a `kind` union.** Story 17.10 (Done) established the convention: styrdokument detected by `!!workspaceDocumentId`, web by `!!url && !slug`, legal by `slug`/`path`. Add files as `!!fileId` to stay consistent — do **not** introduce a `kind` string union (an earlier draft proposed this before 17.10 shipped; it's now redundant).
- **Snippet preview is file-scoped on purpose.** The single most important constraint: do **not** widen `isChunkLevel` (`!!source.path`). Add a parallel `isFileResolved` term to the `description` expression. Because legal sources never have a `fileId`, the legal doc-level suppression is provably unchanged — this is what keeps regression risk near zero while still giving files the glance-to-verify the snippet provides. (A long uploaded file opened cold lands the user at page 1 with no pointer to the cited passage; the hover preview is what makes verification actually work.)
- **Why files dead-end today (root cause):** every citation surface is keyed off legal/web/styrdokument discriminators a file doesn't have, so a file falls through to the legal "DB source" sidebar branch. The fix is symmetric with web sources: detect the file, preview + open it, and `return` before the sidebar.
- **`QuickPreview` reuse caveat:** it expects a full `WorkspaceFileWithLinks` + a `files[]` array (for prev/next nav) + a `getFileUrl` resolver. From a citation we only have `fileId`, so the preferred path fetches the file by id (pass a single-element `files[]`). The new-tab fallback avoids that wiring but carries the popup-blocker caveat in AC 4. Both meet AC 4.
- **No schema, no new deps, no server-action changes** beyond *calling* the existing `getFileDownloadUrl`. Frontend-only, ~2 files (`citations.ts`, `citation-pill.tsx`) + tests.

## Risk and Compatibility

- **Primary risk:** the file preview/route branch accidentally affects a non-file source (e.g. widening `isChunkLevel` would surface doc-level legal summaries as if they were the cited section). **Mitigation:** keep the file behavior on a *separate* `!!fileId` condition; AC 6 + the AC 7c test guard the legal path. **Rollback:** revert the 2-file frontend diff; file pills fall back to today's resolve-but-dead-end behavior (no data loss, no schema).
- **Compatibility:** no API/schema changes; one additive optional `SourceInfo.fileId`; UI mirrors the shipped styrdokument pill + existing `QuickPreview`; negligible perf impact (one file fetch only on click, only for the modal path). The `isChunkLevel` gate is untouched, so legal/web/styrdokument rendering is provably unchanged.

### Issue Found & Fixed During Review — DOM-001 (medium, resolved)

- **Symptom:** opening the modal logged 4 hydration/`validateDOMNesting` errors — *"In HTML, &lt;div&gt; cannot be a descendant of &lt;p&gt;"* and *"&lt;p&gt; cannot be a descendant of &lt;p&gt;"*.
- **Root cause:** the citation pill renders inside a markdown `<p>` (`MarkdownParagraph`). `QuickPreview` is a block-level modal (`<div role="dialog">` with nested `<div>`/`<p>`). Mounting it as a child of the pill nested block elements inside a `<p>` — invalid HTML. It *looked* fine only because the modal is `position: fixed` (out of visual flow). The sibling inline affordances (`<a>`/`<Link>`) never hit this; the file pill is the first to render a modal inline.
- **Fix:** wrapped the `QuickPreview` mount in `createPortal(<QuickPreview .../>, document.body)` so the modal escapes the `<p>` in the DOM. Zero visual change (fixed-position). Gated on `previewOpen` + `typeof document !== 'undefined'` so SSR/initial hydration is unaffected.
- **Regression test added:** renders the pill inside a `<p>`, opens the modal, asserts `modal.closest('p')` is `null` (proves the portal escape).
- **Verification:** `pnpm typecheck` clean, `eslint` clean, pill suite now **6/6** green.
