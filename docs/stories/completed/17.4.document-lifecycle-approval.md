# Story 17.4: Document Lifecycle — Status Workflow & Approval

## Status

Done

## Story

**As a** compliance manager,
**I want** documents to follow a controlled lifecycle from draft to approved with review dates,
**so that** I can maintain ISO-compliant document control with clear approval records.

## Story Context

**Existing System Integration:**

- Integrates with: `WorkspaceDocument` model (status, approved_by, approved_at, review_date, retention_until, document_number fields — from Story 17.1), `WorkspaceDocumentVersion` (immutable rows — from 17.1/17.3), editor component from Story 17.2 (`components/features/documents/editor/document-editor.tsx`), version history from Story 17.3, `ActivityLog`
- Technology: Prisma, React, shadcn/ui
- Follows pattern: `withWorkspace()` auth from `@/lib/auth/workspace-context`, `ActionResult<T>` return type, server action mutation pattern from `app/actions/documents.ts`, Zod validation schemas in `lib/validation/documents.ts`
- Touch points: Editor metadata bar, document server actions (`app/actions/documents.ts`), validation schemas (`lib/validation/documents.ts`), activity labels (`lib/constants/activity-labels.ts`)

**Model naming — `WorkspaceDocument*` prefix required:** The existing schema has `model DocumentVersion` and `enum DocumentStatus` for the SFS legal document system. All DMS models use the `WorkspaceDocument*` prefix:
- `WorkspaceDocument` (Prisma: `prisma.workspaceDocument`)
- `WorkspaceDocumentVersion` (Prisma: `prisma.workspaceDocumentVersion`)
- Enums: `WorkspaceDocumentType`, `WorkspaceDocumentStatus`, `WorkspaceDocumentVersionSource`

## Acceptance Criteria

**Status Display:**

1. Document status displayed as color-coded badge everywhere it appears (editor, modals):
   - Utkast / DRAFT (gray)
   - Under granskning / IN_REVIEW (blue)
   - Godkänd / APPROVED (green)
   - Ersatt / SUPERSEDED (orange)
   - Arkiverad / ARCHIVED (muted/gray, italic)

**Status Transitions:**

2. Status transitions enforced (invalid transitions rejected by server action — already implemented in 17.1, this story enhances with comment + ActivityLog):
   - DRAFT → IN_REVIEW
   - DRAFT → ARCHIVED
   - IN_REVIEW → APPROVED
   - IN_REVIEW → DRAFT (send back for revisions)
   - APPROVED → SUPERSEDED
   - APPROVED → ARCHIVED
   - SUPERSEDED → ARCHIVED
   - ARCHIVED → (terminal — no transitions out)
3. Status change UI: dropdown or action buttons on editor page showing only valid next states
4. Each status change accepts an optional comment (stored in `new_value` of the ActivityLog entry alongside old/new status)

**Approval:**

5. Transitioning to APPROVED records `approved_by` (current user) and `approved_at` (timestamp) on the `WorkspaceDocument` — already implemented in `updateDocumentStatus()` from 17.1; this story adds the UI and ActivityLog
6. When a document is APPROVED, the editor is read-only — content cannot be modified
7. To edit an approved document, user clicks "Skapa ny version" which: creates a new `WorkspaceDocumentVersion` from the current approved content (preserving immutability), sets document status to DRAFT, and enables editing

**Document Metadata Fields:**

8. `review_date` field — optional date picker, shown in editor sidebar/settings panel
9. `retention_until` field — optional date picker, shown in editor sidebar/settings panel
10. `document_number` field — optional text input for user-defined IDs (e.g., "POL-2026-001"), shown in editor settings panel
11. All metadata fields editable from a document settings panel (slide-out or section in editor)

**Review Date Indicators:**

12. In the workspace documents area: review dates overdue highlighted in red, upcoming within 30 days in amber
13. Overdue review count shown as a badge/indicator (optional enhancement)

**Audit:**

14. All status changes logged to ActivityLog with `action: 'document_status_changed'`, `old_value: { status }`, `new_value: { status, comment }`

**Integration Requirements:**

15. Editor from Story 17.2 respects read-only state for approved documents

**Quality Requirements:**

16. Status transition validation is server-side (cannot be bypassed from client) — already enforced by `VALID_STATUS_TRANSITIONS` in `lib/validation/documents.ts`

**"Skapa ny version" flow (AC 7):**

When user clicks "Skapa ny version" on an approved document:
1. Server action `createDraftFromApproved(documentId)` runs
2. Reads current version's `content_json` from `WorkspaceDocumentVersion`
3. In a `$transaction`: creates a new `WorkspaceDocumentVersion` row (preserving immutability), sets `WorkspaceDocument.status = DRAFT`, clears `approved_by`/`approved_at`, updates `current_version_id`/`current_version_number`
4. This is similar to `restoreDocumentVersion()` from 17.3 but with a status change instead of a version number in the summary
