## Story

**As a** user who asked the compliance partner to draft a styrdokument (policy, procedure, or similar),
**I want** the draft rendered as a reviewable card with "open editor" and "approve" affordances,
**so that** I can accept the draft directly or customize it in Tiptap before finalizing.

## Context

Today the agent cannot author full documents. Story 17.11 shipped `create_document` and `update_document` tools but these expect the agent to already know what to write — they're atomic writes, not a draft-and-approve flow. When a user asks "write me a policy for [law]", the agent currently answers in chat text that the user then has to copy into a new document manually.

This story adds a `draft_styrdokument` tool that:

1. Generates a full Tiptap-JSON document draft using the agent's knowledge of the company context, the law, and any existing policies.
2. Returns a `PendingAgentAction` of type `DRAFT_DOCUMENT` with a preview.
3. Lets the user approve directly (creates the `WorkspaceDocument` via Epic 17) or open the Tiptap editor to customize before finalizing.
4. Wires `WorkspaceDocumentTaskLink` / `WorkspaceDocumentListItemLink` rows from the `contextLinks[]` payload — so an approved draft is automatically connected to the law list item / task that motivated it.

## Acceptance Criteria

### Schema

1. `PendingAgentActionType` enum (already extended in 14.23) is further extended with `DRAFT_DOCUMENT`. Additive migration.
2. No new tables — `PendingAgentAction.params` JSON stores the full Tiptap draft payload: `{ title, docType, contentJson, contextLinks }`.
3. `contextLinks` schema: `Array<{ kind: 'TASK' | 'LIST_ITEM', id: string }>`. Resolved against `Task.id` or `LawListItem.id` at approval time.
3a. **Content-quality gate (v1):** the `draft_styrdokument` tool performs a minimal structural check on `contentJson` before persisting the pending row: **must have ≥3 top-level block nodes AND ≥1 heading node**. On failure, the tool returns a `ToolError` with guidance "Utkastet är för kortfattat. Inkludera minst en rubrik och tre stycken så det blir användbart för användaren." and does NOT create a pending row. This prevents low-effort drafts from reaching users. Stricter semantic validation (lint-level Swedish, required sections per `docType`) is deferred.

### Tool

4. New tool `lib/agent/tools/draft-styrdokument.ts`. Zod schema:
   ```
   {
     title: string,              // Swedish title of the draft document
     docType: WorkspaceDocumentType,  // reuse the Epic 17 enum (POLICY, PROCEDURE, RISK_ASSESSMENT, etc.)
     contentJson: unknown,       // Tiptap JSON root node
     contextLinks?: Array<{ kind: 'TASK' | 'LIST_ITEM', id: string }>,
     execute: boolean,           // inherited pattern (always false — see AC 5)
   }
   ```
5. Tool description in Swedish guides the agent: describe when to use (user asks for a draft), what makes a good draft (structured headings, reference the law, include rationale sections), required `contextLinks` population (include any list items / tasks that the chat is actively about). `execute: true` is not supported (removed as per 14.23's pattern).
6. On invocation: create a `PendingAgentAction` row with `action_type: DRAFT_DOCUMENT` and the full payload in `params`. Return `WriteToolResponse` envelope with `data: { pendingActionId }` and a preview string of the first ~200 characters of plaintext extracted from `contentJson`.

### Renderer

7. New component `components/features/ai-chat/agent-action-renderers/draft-document-renderer.tsx`.
8. `PENDING` state — compact preview:
   - Header: document-type badge (Swedish label — reuse `DOCUMENT_TYPE_LABELS` from `components/features/documents/document-type-labels.ts`) + title (editable `Input`).
   - Body: first ~200 characters of extracted plaintext from `contentJson`, with a "Visa mer" button that opens the full read-only draft **in the chat detail panel** (see AC 11a) — not an inline/modal viewer.
   - Context-links section: chips showing each linked entity ("Kopplad till: Uppgift '${taskTitle}', Kravpunkt '${listItemTitle}'"). Dismissable per-chip (removes from `params.contextLinks`).
   - Controls: "Öppna i editor" (secondary), "Godkänn" (primary), "Avvisa" (ghost).
9. `APPROVED` state: show title + type badge + link to the created document (`/workspace/styrdokument/${documentId}/edit`) + link chips for each wired link. Styling matches other APPROVED renderers.
10. `REJECTED` / `EXPIRED` states: standard muted read-only variants per 14.22's contract.
11. Compact variant (for batch card): collapsed summary line "Utkast: ${docType} '${title}'", expandable to full preview.
11a. **Full-draft view opens in the chat detail panel, NOT a hard-coded modal (canvas-ready seam).** The "Visa mer" affordance (AC 8) opens the full read-only draft via the existing `openDetail()` mechanism in `lib/ai/chat-detail-context.tsx` — the same detail-panel path read tools already use. Add a new `ChatDetailItem` variant `{ type: 'document-draft'; id: pendingActionId; data: { title, docType, contentJson } }` and render it in `chat-detail-sidebar.tsx` as a read-only Tiptap preview. **Why:** this panel is precisely the surface the future Agent Document Canvas (Story 14.33, Phase 1) upgrades in place — routing the preview through the panel now means this renderer needs zero rework when the canvas lands. The inline approval **controls** (Godkänn / Öppna i editor / Avvisa) stay on the card per 14.22/14.23 — only the full-document **preview** moves to the panel (consistent with 14.23 keeping read-tool detail items while removing write-preview cards; this variant is read-only, not a write-preview). Minimum-viable fallback if the panel route proves heavy: a dialog — but the panel route is strongly preferred. [Source: `lib/ai/chat-detail-context.tsx` `ChatDetailItem` union; `chat-detail-sidebar.tsx`; canvas roadmap → Story 14.33]

### Approval Dispatch — No-Edit Path

12. `approvePendingAction` dispatch for `DRAFT_DOCUMENT`. **Exact targets pinned via codebase lookup:**
    - Calls `createDocument({ title, documentType, contentJson })` at [`app/actions/documents.ts:54`](../../app/actions/documents.ts). The action wraps creation in a `prisma.$transaction`, creates `WorkspaceDocument` + `WorkspaceDocumentVersion v1` (with `source: 'TIPTAP'`), sets `current_version_id`, writes an `ActivityLog` row. Returns `ActionResult<{ id, title, versionNumber }>`. Status on create is implicit `DRAFT` (the Epic 17 lifecycle default).
    - **Note on `source` field:** Epic 17's version table uses `source: 'TIPTAP' | ...` (not `'AGENT'` as earlier draft copy implied). Agent authorship is tracked via `created_by` + `ActivityLog.entity_id` + optional metadata; don't attempt to add an `'AGENT'` source value unless Story 17.11 already defined it — **verify via `prisma/schema.prisma` `WorkspaceDocumentSource` enum before coding**.
    - Iterates `contextLinks[]` and dispatches. **Both link actions are POSITIONAL (not object-args) and return `ActionResult` with NO row id** — confirmed in source and learned during Story 14.23's build:
      - `kind: 'TASK'` → `linkDocumentToTask(documentId, taskId)` at [`app/actions/documents.ts:1089`](../../app/actions/documents.ts).
      - `kind: 'LIST_ITEM'` → `linkDocumentToListItem(documentId, listItemId)` at [`app/actions/documents.ts:1149`](../../app/actions/documents.ts).
    - Writes `result_ref = { documentId, links: Array<{ kind: 'TASK' | 'LIST_ITEM', id: string }>, partialLinkErrors?: Array<{ kind, id, error }> }`. The link actions return no row id, so `result_ref` stores the link **descriptors**, not link-row ids — mirrors Story 14.23's decision to store `{ taskId, documentId }` for its link types (AC 11 there).
    - `revalidatePath('/workspace/styrdokument')` and any task / law-list paths affected by the links.
13. If `createDocument` fails, the row stays `PENDING` with the dispatch error surfaced on the card. If the document is created but one or more links fail, the document is NOT rolled back (users keep the draft); instead the row is marked `APPROVED` with a partial-result warning on the card summary: "Dokument skapat, men ${n} koppling(ar) misslyckades" and the specific errors stored in `result_ref.partialLinkErrors`.

### Approval Dispatch — Open-Editor Path

14. "Öppna i editor" click on the renderer:
    - Calls a new server action `openDraftInEditor(pendingActionId)` which atomically (via `prisma.$transaction`):
      - Creates the `WorkspaceDocument` in `DRAFT` status with the pending row's `contentJson` — reuses `createDocument` logic internally OR replicates its transaction inline to guarantee atomicity (implementer choice; if reusing `createDocument`, wrap the call + pending-row update in a parent transaction with `Prisma.$transaction(async (tx) => {...})`).
      - Stores the new `documentId` in `PendingAgentAction.result_ref`.
      - Does NOT wire links yet (links attach on finalization — see AC 15).
      - Sets row status to `IN_EDITOR` (enum value added — see AC 18) so the card stops showing "Godkänn" and the row is excluded from "Approve all" in batch view.
    - Returns `{ documentId }`; renderer uses Next.js router to navigate to `/workspace/styrdokument/${documentId}/edit?agentApprovalId=${pendingActionId}`.
14a. **Navigation-failure edge case.** If the transaction succeeds but the navigation fails (network hiccup, user closes tab), the row is `IN_EDITOR` with a valid `documentId` but the user never enters the editor. To prevent orphan drafts: the `/workspace/styrdokument` list page (Epic 17) must surface any `IN_EDITOR` draft for the current user with a "Fortsätt godkännande" action (links to the editor with the same `?agentApprovalId=` param). Implementer adds a section or banner to the styrdokument list page reading pending actions of type `DRAFT_DOCUMENT` with status `IN_EDITOR` for the current user. If the list-page update is heavier than expected, minimum-viable alternative: document the `/workspace/styrdokument/{id}/edit?agentApprovalId=...` URL pattern so users (and the agent in next-turn context) can find their orphaned drafts.
15. The document editor page (`app/(workspace)/workspace/styrdokument/[documentId]/edit/page.tsx`) reads `agentApprovalId` from the URL and renders an inline banner:
    - Copy: "Väntar på godkännande från agent-utkast. Spara för att slutföra godkännandet, eller avvisa för att ta bort utkastet."
    - Two actions: "Slutför godkännande" (primary) → saves the current editor state, attaches the `contextLinks[]` from the pending row, marks the row `APPROVED`, and clears the banner. "Avvisa" (ghost) → deletes the draft document and marks the row `REJECTED`.
    - The banner component lives in `components/features/documents/pending-agent-approval-banner.tsx` and reads / mutates the pending row via a new SWR key `pending-action:${id}`.
    - **Banner unmount pattern:** the banner renders only when `data?.status === 'IN_EDITOR'`. When "Slutför godkännande" transitions status to `APPROVED`, SWR revalidation returns the new status, the render guard evaluates false, and the banner unmounts cleanly. Implementation: `const { data } = useSWR(...)` + `if (data?.status !== 'IN_EDITOR') return null` at the top of the component body. Do NOT use local optimistic state for the status — let SWR be the source of truth to avoid double-save / race conditions.
16. If the user navigates away from the editor without clicking either action, the row stays in `IN_EDITOR` state. Revisiting the document page re-shows the banner (idempotent). `expirePendingActions` also covers `IN_EDITOR` rows on its usual 7-day window.

### Rejection

17. Reject click (from either the card or the editor banner):
    - Deletes the `WorkspaceDocument` if one was created via the open-editor path (row status `IN_EDITOR` → delete doc + set `REJECTED`).
    - Sets row status to `REJECTED`.
    - `revalidatePath('/workspace/styrdokument')`.

### Status Enum Extension

18. `PendingAgentActionStatus` enum extended with `IN_EDITOR`. Migration additive.
19. Card fallback text for `IN_EDITOR` state: "Öppet i editor — slutför där eller avvisa".

### Non-Regression

20. All pending-action types from 14.22 + 14.23 continue to work. Batch card correctly includes `DRAFT_DOCUMENT` rows in per-row approve flows. **Note:** 14.23's "Godkänn alla" handler already iterates `PENDING`-only rows, so an `IN_EDITOR` row is non-dispatched for free — the only net-new work here is surfacing the "hoppad över — öppen i editor" note in the per-row summary (don't re-engineer the filter).
21. The Epic 17 document editor works unchanged for user-created documents (no `agentApprovalId` URL param → no banner).
22. Reasoning blocks (14.20), tool-call collapsing (14.18), citation pills (14.16), and chat streaming continue to render correctly.

### Security

23. The three new server actions — `openDraftInEditor`, `finalizeDraftFromEditor`, `rejectDraftFromEditor` — each independently enforce `hasPermission(role, 'tasks:edit')` before any write. These are finalize/write paths that **bypass `approvePendingAction`**, so they do NOT inherit the SEC-001 permission gate Story 14.23 added there — they must re-assert it themselves. A viewer / `AUDITOR`-role member must not be able to create, finalize, or delete a draft document. The `DRAFT_DOCUMENT` branch *inside* `approvePendingAction` (the no-edit path, AC 12) inherits the existing gate and needs no extra check.

### API Specifications

- **New server actions** (extensions to `app/actions/pending-agent-actions.ts`):
  - `openDraftInEditor(pendingActionId)` — atomic transaction: create `WorkspaceDocument` in `DRAFT`, set `result_ref.documentId`, transition row to `IN_EDITOR`. Returns `{ documentId }`.
  - `finalizeDraftFromEditor(pendingActionId, editorContent)` — update doc version, wire `contextLinks[]`, set row `APPROVED`.
  - `rejectDraftFromEditor(pendingActionId)` — delete the `WorkspaceDocument`, set row `REJECTED`.
- **Extension to `approvePendingAction`** — new `DRAFT_DOCUMENT` branch that handles the no-edit path. Dispatches to `createDocument` (object-args) + iterates `contextLinks[]` calling the POSITIONAL Epic 17 link actions (`linkDocumentToTask(documentId, taskId)` / `linkDocumentToListItem(documentId, listItemId)`). Inherits the `tasks:edit` gate already on `approvePendingAction` (SEC-001, 14.23).
- **All actions** return `{ success, data?, error? }`. [Source: architecture/17-coding-standards.md#17.5]
- **Partial failure semantics** — document created + link fails → row `APPROVED` with partial-warning. Document creation fails → row stays `PENDING`. [See AC 13 for exact contract.]

### Technical Constraints

- **Epic 17 model prefix** — use `WorkspaceDocument*` everywhere. Using bare `Document` collides with SFS legal-document models. [Source: docs/prd/epic-17-document-management-system.md — NAMING COLLISION NOTE ⚠️]
- **Stale context-link IDs** — `Task` or `LawListItem` referenced by `contextLinks[]` may be deleted between tool invocation and approval. Dispatch handles FK errors as partial-link failures, not full approval failures. [Source: architecture/18-error-handling-strategy.md#18.4 Service-Specific Error Handling]
- **No document rollback on partial link failure** — users value the draft content. Surface a warning, keep the document. [Source: design decision, product-level — see Dev Notes above]
- **Banner idempotency** — `IN_EDITOR` → `APPROVED` transition must unmount the banner cleanly without double-save. Implement via SWR's returned state rather than local optimistic state.
- **Content quality gating** — the tool description must steer the agent toward structured drafts (headings, sections). Consider a minimal structural validator (reject `contentJson` with < 3 block nodes or no headings). Accept as a warning only in v1; don't block.
- **Tiptap JSON size** — `PendingAgentAction.params` is `Json` typed. Large drafts (~20KB+) fit but watch for row-size edge cases. No pagination needed for v1.
- **Additive-enum migration, manual handoff** — `DRAFT_DOCUMENT` + `IN_EDITOR` are `ALTER TYPE ... ADD VALUE` only, no table changes. Generate with `prisma migrate dev --create-only` and hand the SQL to the user; never auto-apply or reset the dev DB. Run `prisma generate` for typegen. [Source: standing dev-workflow convention; Story 14.23 migration handoff]
- **Auth gate on the open-editor write paths (AC 23)** — `openDraftInEditor` / `finalizeDraftFromEditor` / `rejectDraftFromEditor` bypass `approvePendingAction`, so they do NOT inherit its SEC-001 `tasks:edit` gate (14.23). Each must call `hasPermission(role, 'tasks:edit')` itself before writing. The in-`approvePendingAction` `DRAFT_DOCUMENT` branch inherits the gate. [Source: docs/qa/gates/14.23-agent-action-extended-types-batch.yml — SEC-001]
- **Link actions are positional + return no id** — `linkDocumentToTask(documentId, taskId)` and `linkDocumentToListItem(documentId, listItemId)` take positional args and return `ActionResult` (no row id). Store link descriptors in `result_ref`, not link-row ids (see AC 12). [Source: `app/actions/documents.ts:1089`/`:1149`; Story 14.23 build discovery]
