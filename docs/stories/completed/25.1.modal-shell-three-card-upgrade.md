# Story 25.1: Modal shell + four-card path-choice upgrade

## Status

Done

## Story

**As a** new user who just finished the workspace-creation wizard,
**I want** the first-run modal to feel like a real Laglig surface — branded chrome with a Safiro accent, a calm dim/blur backdrop, four production-grade path cards (Mall / Generera / Tom lista / Importera) at full visual fidelity that match the in-app `<ManageListModal>` mental model, an inline template picker so I can pick a mall without leaving the modal, and a real import flow that mounts when I click Importera (instead of the "kommer snart" toast),
**so that** the modal stops looking like the B.0 wedge it was, the import path is finally unlocked (Epic 24.4 is now Done), "build it myself" is a first-class choice instead of a defeatist "Hoppa över" tail, and I can adopt a template, generate, build manually, or upload an existing list without bouncing through a second route or having to come back.

This is the **B.1 production upgrade** of Story 25.0's path-choice gate per the Epic 25 PRD. B.0 shipped a minimal Dialog with three plain cards (Mall / Generera / Importera-disabled) plus a tertiary "Hoppa över — bygg manuellt" link. With Epic 24.4 (`import-review-surface`) now Done, B.1 un-disables the Importera card and routes it to the existing `<ImportUploadStep>` (built by Story 24.2, already mounted by Story 24.6's `<ManageListModal>`), adds an inline template-pick sub-step that reuses the existing `<TemplateOptionCard>` + `getPublishedTemplates()` pair from Story 12.10b (no template schema duplication), **promotes the implicit "bygg manuellt" path from a tertiary text link to a first-class fourth card "Tom lista"** to match the in-app `<CreateListChooser>` four-options mental model, reframes the tertiary Hoppa över link as pure hard-opt-out (no longer the "I'll build manually" fallback — that's Tom lista's job now), and upgrades the Dialog chrome to brand spec: `max-w-[720px]`, `rounded-2xl`, a heavier modal shadow, `backdrop-blur-sm` on the overlay, and a Safiro brand bar at the top. The Generera card keeps its B.0 behaviour (fires `POST /api/workspace/generate-law-list` immediately on click) — generate-kickoff confirm is **out of scope for B.1**.

The recommended-chip on Mall, hover-lift, icons, meta lines, keyboard affordance, dismissal contract (no X / no Esc / no outside-click), and all five existing server actions are unchanged — B.1 is a *surface* rework with one minimal substrate addition (the Tom lista handler reuses the existing `minimiseFirstRunModal` action; no schema or new action required). The card grid restructures from `sm:grid-cols-3` (three cards) to `sm:grid-cols-2` (four cards in a 2×2 layout).

## Acceptance Criteria

### Dialog chrome upgrade

1. **Modal max-width widens from `max-w-2xl` to `max-w-[720px]`.** The B.0 modal uses `max-w-2xl` (= 672px) on `DialogContent` in `components/features/onboarding-modal/first-run-modal.tsx:62`. B.1 widens it to a fixed `max-w-[720px]` so the four cards (in a 2×2 grid) plus the inline template-pick / import-upload sub-steps have breathing room. (PRD line 129 originally specified `720px` for a three-card row; the same width works comfortably for the 2×2 four-card layout — each cell is ~344px, roughly matching `<CreateListChooser>`'s comfortable in-app card width.)
2. **Modal corner radius is `rounded-2xl` (16px).** The shadcn `DialogContent` defaults to `sm:rounded-lg` (8px) — see `components/ui/dialog.tsx:41`. Override via the `className` prop passed from `<FirstRunModal>` so the corners feel brand-level rounded, not utility-default. The override applies only to the first-run modal; do NOT modify `components/ui/dialog.tsx` (changes there affect every modal in the app).
3. **Modal shadow upgrades from `shadow-lg` to a heavier `shadow-modal`.** No `shadow-modal` utility exists in `app/globals.css` today (only the default Tailwind shadows). Add a custom utility under the existing `@layer utilities` block at `app/globals.css:122–125` (next to `.font-safiro`) — e.g. `.shadow-modal { box-shadow: 0 24px 60px -20px rgba(48, 39, 24, 0.18), 0 8px 24px -8px rgba(48, 39, 24, 0.08); }` (warm-tinted, matching the `--primary` palette `30 15% 12%` so it doesn't read as a cold grey halo). Apply via the `className` prop on `DialogContent`.
4. **Dialog overlay gets `backdrop-blur-sm`.** The shadcn overlay (`components/ui/dialog.tsx:24`) uses only `bg-black/80` today. B.1 needs a dim-AND-blur backdrop. Do NOT edit `components/ui/dialog.tsx` directly (every modal would change). Instead, replicate the overlay locally: render `<DialogOverlay className="bg-foreground/40 backdrop-blur-sm" />` inside the modal's `DialogPortal` and use shadcn's `DialogPortal` + `DialogPrimitive.Content` pattern so the overlay is no longer auto-mounted. **Alternative (simpler):** keep the auto-overlay and accept the default `bg-black/80`; only add `backdrop-blur-sm` if the local-overlay refactor is too invasive — the dev may choose either, but the chosen variant must be documented in the Change Log.
5. **Safiro brand bar at the top of `DialogContent`.** A slim accent strip (recommended: a 4px-tall band tinted with the warm-primary palette, e.g. `<div className="-mx-6 -mt-6 mb-2 h-1 rounded-t-2xl bg-primary/80" />` placed as the first child inside `DialogContent`, before `DialogHeader`). The exact pixel size and tint should match `_prototypes/onboarding-tutorial-modal.html` frame ② — see Dev Notes "Prototype reference" for caveats about the prototype still showing two cards in frame ①. (Per PRD line 129 + arch §6.3 referencing the prototype.)
6. **Dismissal contract from B.0 is preserved.** No X button rendered (`className="[&>button]:hidden"`), no Esc-key dismissal (`onEscapeKeyDown={(e) => e.preventDefault()}`), no outside-click dismissal (`onInteractOutside={(e) => e.preventDefault()}`). User can only close the modal via a path choice or the Hoppa över link. (B.0 contract per `first-run-modal.tsx:57–62` and Story 25.0 AC 23.)

### Sub-step state machine

12. **`<FirstRunModal>` gains an internal mode/step state.** Today the component is a thin wrapper that always renders `<PathChoiceStep>`. B.1 introduces a local `step: 'path-choice' | 'template-pick' | 'import-upload'` state (recommendation: `useState`) so the modal can swap its body between the three sub-steps without re-mounting the Dialog (which would lose focus, animation, and the open-event guard).
13. **State transitions:** `path-choice → template-pick` when the Mall card is clicked; `path-choice → import-upload` when the Importera card is clicked; `template-pick / import-upload → path-choice` when the user clicks "← Tillbaka". The Generera card, Tom lista card, and Hoppa över link **bypass** the sub-step state — they all close the modal immediately (Generera + Hoppa över preserve B.0 behaviour per AC 26 + 11; Tom lista is the new direct-action card per AC 27a–27d).
14. **Each sub-step renders a "← Tillbaka" affordance** at the top-left of the sub-step body (a `Button variant="ghost" size="sm"` with the `ArrowLeft` icon from `lucide-react` is the recommended shape). Clicking returns to `'path-choice'` without firing any server action. The Dialog header (`<DialogTitle>` + `<DialogDescription>`) **rewrites** per step:
    - `'path-choice'` — title "Hur vill du komma igång?", description "Välj hur du vill skapa din första laglista. Du kan ändra dig senare." (B.0 unchanged)
    - `'template-pick'` — title "Välj en mall", description "Välj en bransch-mall — listan skapas direkt."
    - `'import-upload'` — title "Importera befintlig laglista", description "Ladda upp en .xlsx / .csv eller klistra in raderna."

### Mall path — inline template picker

15. **Mall card click no longer routes to `/laglistor/mallar`.** B.0's `handleTemplate` in `path-choice-step.tsx:48–53` does `router.push('/laglistor/mallar')`. B.1 removes the `router.push` and instead transitions the modal to `step='template-pick'`. The `path_chosen={path:'template'}` event still fires immediately (it records *intent*, not completion). The `minimiseFirstRunModal()` write is **deferred** until the user actually selects a template (AC 18) or hits "← Tillbaka" (in which case it does NOT fire — user reverted).
16. **Template-pick sub-step renders a horizontal-card grid using existing `<TemplateOptionCard>`** from `components/features/document-list/template-option-card.tsx`. Data source: `getPublishedTemplates()` from `lib/db/queries/template-catalog.ts:36`. Fetch strategy: prefetch server-side in `app/(workspace)/dashboard/page.tsx` (same `Promise.all` pattern that already runs `getOnboardingState` — see Story 25.0 File List entry for `dashboard/page.tsx`) and pass the resulting `PublishedTemplate[]` as a new prop on `<FirstRunModal>` and forward to `<PathChoiceStep>` (or a new `<TemplatePickStep>` sibling). The prefetch is unconditional (cost ~1 DB read) so the picker opens with zero latency; if the user never opens the template-pick step the data is just unused.
17. **Loading and empty states:**
    - **Loading:** because templates are server-prefetched, no client-side loading state is needed. If the array is empty (templates table empty in dev/staging), render a fallback empty-state row: "Inga publicerade mallar tillgängliga. Återgå till valet ovan för att välja en annan väg." with a "← Tillbaka" CTA. (Neutral fallback copy — four cards now exist, so the user can pick Generera, Tom lista, or Importera as the alternative; do not name a specific fallback path.)
    - **No skeleton / spinner inside the sub-step.** First paint is synchronous from the server-prefetched array.
18. **Selecting a template** records `path_chosen={path:'template', template_slug: <slug>}` (extending the B.0 payload from `{path:'template'}` to include the chosen slug — `OnboardingEvent.payload` is `Json?` per `prisma/schema.prisma`, so no schema change), calls `minimiseFirstRunModal()`, calls `onClose()` to close the modal locally, then routes the user to `/laglistor/mallar/{slug}` (the existing template-detail page at `app/(workspace)/laglistor/mallar/[slug]/page.tsx` — Story 12.9 / 12.10). The detail page hosts the existing apply-template flow; B.1 does NOT duplicate apply-template UI inside the modal.
19. **"← Tillbaka" from the template-pick step** returns to `path-choice` and does NOT fire `minimiseFirstRunModal()` or record any event. The user has not committed to a path; reverting must be free.

### Import path — un-disabled, inline upload

20. **Importera card visual changes from disabled → active.** Remove `aria-disabled="true"`, `cursor-not-allowed`, and `opacity-60` from the Importera card in `path-choice-step.tsx:147–169`. Remove the `<Badge tone="neutral" variant="soft">Kommer snart</Badge>` chip. The card now has the same hover-lift, focus-visible, and cursor styling as the Generera card.
21. **Importera card click no longer toasts.** B.0's `handleImport` in `path-choice-step.tsx:84–89` shows `toast.info('Vi mejlar dig när importen är klar att använda.')`. B.1 removes that toast call and instead transitions the modal to `step='import-upload'`. The `path_chosen={path:'import'}` event still fires immediately (records *intent*); `minimiseFirstRunModal()` is deferred until a successful upload (AC 23) or "← Tillbaka" (in which case it does NOT fire).
22. **Import-upload sub-step renders the existing `<ImportUploadStep>` component** from `components/features/onboarding-modal/import-upload-step.tsx` (built by Story 24.2, already reused by Story 24.6's `<ManageListModal>` per the docblock in that file lines 7–18). Mount as `<ImportUploadStep onSuccess={handleImportSuccess} />`. **Do NOT** re-implement the upload form, the Fil/Klistra-in tabs, the dropzone, the template-download link, or the parse/match phase orchestration — that is 379 lines of working code already shipped.
23. **On successful import (`onSuccess` callback fires with `importId`):**
    - Record `path_chosen={path:'import', import_id: <importId>}` event (extends payload with the import id for downstream telemetry).
    - Call `minimiseFirstRunModal()` to persist `first_run_dismissed_at` so the modal doesn't re-open on reload.
    - Call `onClose()` to close the modal locally.
    - Route to `/laglistor/skapa/{importId}/granska` — the Story 24.4 review page. The `<ImportUploadStep>` default `onSuccess` already routes here via `useRouter().push(...)` (see `import-upload-step.tsx` line 58 docblock); B.1's `onSuccess` prop must do the routing itself (overriding the default) OR rely on the default by passing `onSuccess={undefined}` *and* still firing the telemetry + minimise actions in some other place. **Recommendation: pass a custom `onSuccess` that does all three (telemetry, minimise, route) so the substrate works regardless of `<ImportUploadStep>` internals.**
24. **Import upload failure (toast already surfaced by `<ImportUploadStep>` internally)** does NOT close the modal. Modal stays on `step='import-upload'`. User can retry or click "← Tillbaka" to revert.
25. **"← Tillbaka" from the import-upload step** returns to `path-choice` without firing any server action or routing. Because `<ImportUploadStep>` holds its own form state, hitting Tillbaka then re-entering import-upload will reset the form to empty — this is acceptable for B.1; do NOT add cross-mount state preservation.

### Generera path — unchanged

27. **No generate-kickoff confirm step in B.1.** The architecture's `generate-kickoff-step.tsx` (arch §6.1) is **out of scope for this story** — generation fires immediately on Generera click. If a confirm step becomes desirable, it's a separate story.

### Tom lista path — new fourth card (direct action)

27a. **Tom lista card click is a direct action — no sub-step.** Just like Generera, clicking Tom lista does not transition the modal to a sub-step. The handler `handleManual()` (new) records `path_chosen={path:'manual'}`, calls `minimiseFirstRunModal()`, calls `onClose()`, then routes to `/laglistor` (the user's list-of-lists page where they can click "Skapa lista" → Tom lista via the existing in-app `<ManageListModal>` to actually create the empty list). B.1 does NOT create the blank `LawList` row inline — that's a follow-up enhancement (see "Out of scope" Dev Notes); for B.1 the route is the cheapest viable handoff and reuses the proven in-app blank-list creation form.
27b. **Status remains `null`, dismissed is set.** `minimiseFirstRunModal()` writes `first_run_dismissed_at=NOW()` only; `law_list_generation_status` stays `null` (semantically correct — no generation is happening on the Tom lista path). `getOnboardingState`'s `firstRunOpen` guard now fails because `dismissed !== null`, so the modal will NOT auto-reopen on next dashboard visit. `fabVisible` succeeds because `status !== 'skipped'`, so the FAB (B.6 surface) will eventually render once that story ships — meaning the user can re-engage with onboarding even after picking Tom lista. **No new server action required**; reuse `minimiseFirstRunModal()` per Story 25.0 AC 7.
27c. **No change to `getOnboardingState`** (`lib/onboarding/get-onboarding-state.ts`). The function already handles the `status === null` + `dismissed !== null` state correctly per its existing guards (Story 25.0 AC 4–8). Verify in the unit tests that this state path is covered (Story 25.0's `get-onboarding-state.test.ts` already has it — no extension needed for B.1).
27d. **Path payload extends the enum.** `OnboardingEvent.payload.path_chosen.path` accepts `'template' | 'generate' | 'import' | 'skipped' | 'manual'` after B.1. Because `OnboardingEvent.payload` is `Json?` (Prisma schema), this is a payload-shape extension — no migration, no enum, no schema change. Update the inline TypeScript type annotation on `path_chosen`'s payload wherever it appears in `app/actions/onboarding-modal.ts` (if explicitly typed) or in test files (if used as a discriminator).

### Key decisions baked into this story

- **Generate-kickoff confirm step is OUT of scope.** Story 25.0 Dev Notes "Out of scope" listed it under "(B.1)" — but the Epic 25 PRD's B.1 story body (lines 123–148) does NOT include it. Resolving the ambiguity in favour of the PRD body: B.1 is strictly chrome + cards + template-pick + import-upload + Tom lista (per the four-card decision below). Generate fires immediately on click, same as B.0. A future story (or B.4 done-states bundle) can add the kickoff confirm if UX research demands it.
- **Four cards, not three: Tom lista is promoted from a tertiary "Hoppa över — bygg manuellt" tail to a first-class card.** Decision made 2026-05-16 (during B.1 SM drafting). Rationale: (1) the in-app `<CreateListChooser>` already surfaces "Tom lista" as a peer to Mall and Importera, so the first-run modal's three-card layout was inconsistent with the in-app mental model; (2) "Hoppa över — bygg manuellt" framed the manual path as a defeatist escape hatch instead of a constructive choice — users who genuinely want to build their own list from scratch should see it as a real option; (3) "Hoppa över guiden" now means exactly what it says (opt out of the entire onboarding flow, hide the FAB) instead of overloading the manual-build path. The grid changes from `sm:grid-cols-3` to `sm:grid-cols-2` (2×2 layout) — at `max-w-[720px]`, this gives each card ~344px (matches in-app card proportions); a 4-up `sm:grid-cols-4` would compress to ~165px and break the icon-meta-line layout.
- **Tom lista routes to `/laglistor` instead of creating the blank LawList inline.** B.1's Tom lista handler does NOT create the empty `LawList` row itself — it just records the event, sets `first_run_dismissed_at`, and routes to the list-of-lists page where the user clicks "Skapa lista" → Tom lista via the existing in-app `<ManageListModal>` to actually create the list. Rationale: reuses the proven in-app blank-list creation form (no duplication, no risk of divergent UX), keeps B.1's substrate-change footprint minimal (no new server action, no new schema value), and the extra click is acceptable for what is by design the "I want full manual control" path. A future enhancement could deep-link `/laglistor?create=blank` to auto-open the in-app modal in blank step, or expose a server action that creates the row + redirects to `/laglistor/{newId}` — both are flagged in "Out of scope" below.
- **Tom lista keeps `law_list_generation_status = null`.** The status field tracks the generation lifecycle, not the user's path commitment. Tom lista has no generation, so `null` is semantically correct (parallel to the existing in-app Tom lista flow which also doesn't set this column). The path commitment is recorded in `OnboardingEvent.path_chosen={path:'manual'}`. The FAB stays visible because the existing `fabVisible` guard is `dismissed !== null AND status !== 'skipped'` — `'manual'` is not in that exclusion list, and the absence of `status` doesn't matter.
- **Sub-step state lives in `<FirstRunModal>`, not in a router or URL param.** B.6's deep-link feature (`?onboarding=tutorial&tab=ai_agent`) is a separate concern and ships later. For B.1, the sub-step is purely client-side, ephemeral, lost on reload. Reloading mid-template-pick is fine — the modal re-opens to `'path-choice'` and the user picks again. This is consistent with the rest of B.0's "modal stays open until user picks" contract.
- **Template prefetch fetches the full `PublishedTemplate[]`, no pagination.** `getPublishedTemplates()` returns all published templates; the in-app surface caps display at the first 4 (`create-list-chooser.tsx:49`). For the first-run picker, render *all* of them in a scrollable list — this is the user's first encounter and limiting choice without explanation feels arbitrary. Performance: at ~10–20 published templates this is fine; revisit if catalog exceeds 50.
- **The Importera card's `path_chosen` event now fires on *click* (sub-step entry), not on successful upload.** This preserves the "intent" signal: how many users *tried* to import vs. how many *completed* an import. The `import_id` field is added on `onSuccess` (AC 23) so completion telemetry is still distinguishable (filter where `payload->>'import_id' IS NOT NULL`).
- **The Mall card's `path_chosen` event fires on *click* too (sub-step entry), not on template select.** Same reasoning. `template_slug` is added on actual selection. If the user enters template-pick and clicks "← Tillbaka", the event still records — this is acceptable because it signals "user considered template path" which is useful product data. If this proves too noisy in the funnel dashboard, post-launch can introduce a separate `template_picker_entered` event in a follow-up.
- **Import-upload sub-step focuses "← Tillbaka", not the first form field (AC 32).** Not sourced from architecture or prototype — this is an SM-level a11y decision. Rationale: when a screenreader user transitions into the import-upload sub-step, focusing the form's first field (Fil/Klistra-in tab or file input) skips past the step-change announcement entirely. Focusing the "← Tillbaka" button means the user hears something like "Tillbaka, button" first, which (combined with the per-step `<DialogTitle>` change, AC 14) signals "you are now on a new sub-step" before they scan the form. The other two focus targets (Mall card on path-choice, first `<TemplateOptionCard>` on template-pick) are obvious defaults — no rationale needed. Reversible if dev research finds this is wrong; not load-bearing.
- **No Visual Regression captures added in B.1.** PRD line 148 mentions "new visual regression captures match three-card prototype". The repo does not currently have a visual-regression tooling setup (Percy / Chromatic / Playwright screenshots). Adding one for one story is scope creep. The dev should manually verify chrome against `_prototypes/onboarding-tutorial-modal.html` frame ② and call out any divergences in Completion Notes.

#### Latent Story 16.4 bug uncovered during smoke testing (`generate-law-list` 409 on clean workspaces)

**TL;DR — out-of-scope fix that was necessary to unblock B.1's smoke test.** The `POST /api/workspace/generate-law-list` route returned 409 ("Generation already in progress") on every click for workspaces with `law_list_generation_status = NULL`, making the Generera path appear permanently broken. Root cause was SQL three-valued logic in the route's idempotency guard.

**Discovery sequence (kept here for the next person who hits this):**
1. Reset Almåsa workspace cleanly (status = NULL, all gen fields wiped). Verified via 15-second poll that status stayed NULL with zero activity.
2. Hard-refreshed `/dashboard`, restarted dev server. Single click on Generera → instant 409.
3. Added temporary `console.log` to the route handler. Confirmed: workspace was Almåsa, sessionUserId matched, current status = `null`, BUT `updateMany` still returned `count: 0`.
4. Diagnosed: Prisma's `NOT: { law_list_generation_status: 'in_progress' }` translates to SQL `field != 'in_progress'`. In SQL three-valued logic, `NULL != 'in_progress'` evaluates to **NULL** (not TRUE), so NULL rows are excluded from the match. A workspace with `status = NULL` could never match the WHERE clause → `count = 0` → 409.

**Why this was latent until now:** before Story 25.0 dropped the `createWorkspace` auto-fire, new workspaces always had `law_list_generation_status = 'pending'` immediately after creation — that non-NULL value masked the bug. Story 25.0 made NULL the canonical "user hasn't picked a path yet" state, which exposed it.

**Fix applied to `app/api/workspace/generate-law-list/route.ts`** (lines 38–55 of the post-fix file):

```ts
// before — broken for NULL-status workspaces
where: {
  id: workspaceId,
  NOT: { law_list_generation_status: 'in_progress' },
},

// after — explicit OR makes NULL-status rows match
where: {
  id: workspaceId,
  OR: [
    { law_list_generation_status: null },
    { law_list_generation_status: { not: 'in_progress' } },
  ],
},
```

**Scope caveat:** this change is in a Story 16.4 file, not in any Story 25.1 file. It was required to unblock smoke testing of the Generera path in 25.1. Suggest filing a follow-up Story 16.4 ticket to (a) add a regression integration test specifically against `status = NULL` initial state, (b) consider making the API truly async (return 202 + run generation in a background job) so the modal doesn't have to block for 5+ minutes on every Generera click — see the next bullet for why that matters.

**Adjacent UX gap (left for B.2/B.4):** the route handler currently blocks for the full LLM generation duration (`maxDuration = 300`). The modal's Generera click awaits the full response, so the modal sits with a "Genererar…" spinner for minutes with no progress feedback. Per the Epic 25 architecture doc this is what B.2 (progress strip inside the modal, sharing the banner's SWR data) + B.3 (tutorial tabs while waiting) + B.4 (done_generate celebration state) are designed to solve. B.1 keeps the simpler "click → close → banner" pattern as documented; the spinner-feels-stuck UX is a transient state.
