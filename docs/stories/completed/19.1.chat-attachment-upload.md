## Story

**As a** workspace member using the compliance chat,
**I want** to attach files (PDF, DOCX, XLSX, images) directly in the chat input,
**so that** I can ask the agent about my own documents in the moment ‚Äî without first uploading them to the file manager and then describing them ‚Äî and the agent reads the actual content I attached.

## Acceptance Criteria

### Attach UI (picker + drag-drop + chips)

1. The paperclip attach button in `ChatInputModern` (present at `chat-input-modern.tsx:264-280`, gated by `showAttach` + `onAttach`; currently inert because `chat-panel.tsx:417` passes `showAttach={false}` and no `onAttach`) becomes functional: `chat-panel.tsx` passes `showAttach={true}` + an `onAttach` handler opening a hidden `<input type="file" multiple>` accepting the same types as `ALLOWED_MIME_TYPES` (`app/actions/files.ts:30`).
2. Drag-drop of files onto the chat input area is accepted with the same MIME filter as AC 1.
3. Each picked/dropped file uploads via the existing `uploadFile` server action (`app/actions/files.ts:254`) with `category: 'CHAT_ATTACHMENT'` (new enum value, AC 11) **before** the message is sent. Upload is workspace-scoped via the action's existing `withWorkspace`.
4. Max **5 files per message**; a 6th shows the Swedish inline error "Max 5 filer per meddelande."
5. **Request-size cap (base64-aware, SF-3).** A **32 MB** cap is enforced client-side before send against the **estimated request size**, not raw bytes: files routed to base64 blocks (PDF ‚â§10 MB, images ‚â§5 MB ‚Äî see AC 9) count at **√ó1.37** (base64 inflation); files routed to the extracted-text path count at their (small) text length. Exceeding it shows the Swedish toast "Bifogade filer √∂verstiger gr√§nsen p√• 32 MB. Ta bort en eller flera filer och f√∂rs√∂k igen." (Rationale: the raw aggregate can pass a naive 32 MB check yet the actual Claude request ‚Äî carrying base64 ‚Äî exceeds 32 MB.)
6. Per-file size remains bounded by the server's existing `MAX_FILE_SIZE = 25 MB` (`app/actions/files.ts:27`); if `uploadFile` returns a size/quota error the send is blocked and the error is surfaced as a Swedish toast (reuse existing file-upload error copy).
7. Before send, attached files show as dismissable chips below the textarea (filename + MIME-aware icon). The X button removes the chip and cleans up the orphaned upload (`deleteFile(fileId)` or a lightweight cleanup action).

### Content-block conversion

8. New module `lib/agent/attachments-to-content.ts` exports:
   ```ts
   export async function attachmentsToContent(
     fileIds: string[],
     workspaceId: string
   ): Promise<AttachmentContentBlock[]>
   ```
   It maps each `WorkspaceFile` (looked up by id, workspace-scoped) to a Claude-native content block per AC 9. Output is typed against the installed `ai` package's user-content part types (`TextPart | ImagePart | FilePart`), **not** raw `@anthropic-ai/sdk` types (compatibility req).
9. Routing per file:
   - **PDF ‚â§ 10 MB** ‚Üí `document` block from base64 of the raw bytes (Supabase Storage download pattern at `app/actions/files.ts:474-477`). No extraction needed.
   - **PDF > 10 MB, DOCX, XLSX, PPTX, and any non-image text-path file** ‚Üí `text` block prefixed `[Fil: <filnamn>]\n` using `extracted_text`. **On-demand extraction (SF-1 latency):** if `extracted_text` is empty/null and `extraction_status` is not yet `DONE` (the every-10-min cron hasn't run), the converter **synchronously runs the same extraction routine the cron uses** (the function behind `extract-files`, per the retry precedent at `app/actions/files.ts:453`) so the file is readable **in the same turn**. Only if extraction then still yields nothing ‚Üí `[Fil: <filnamn> ‚Äî inneh√•ll ej tillg√§ngligt]`.
   - **Image (PNG/JPEG/GIF) ‚â§ 5 MB** ‚Üí `image` block from base64. Image > 5 MB ‚Üí text placeholder `[Bild: <filnamn> ‚Äî f√∂r stor f√∂r direkt analys]`.
10. **Workspace isolation:** `attachmentsToContent` verifies each file's `workspace_id === workspaceId` before reading bytes. A mismatched/missing file ‚Üí text placeholder (`[Fil: ok√§nd ‚Äî filen hittades inte]`), never a thrown 403 that aborts the whole message. A file in `FAILED`/`UNSUPPORTED`/`ENCRYPTED` with no usable bytes also degrades to a placeholder.

### Route wiring

11. New `FileCategory` enum value `CHAT_ATTACHMENT` (`prisma/schema.prisma:1354`). **Migrations are manual:** generate with `pnpm prisma migrate dev --create-only`, hand the SQL to the user; never auto-apply, `db push`, or `migrate reset`. The `UpdateFileSchema` Zod enum (`app/actions/files.ts:143`) must also accept `'CHAT_ATTACHMENT'`.
12. The chat client sends attachment ids as a **top-level request-body field** (`attachmentIds: string[]`), not as UIMessage file parts. `app/api/chat/route.ts` reads `attachmentIds` and, for the **last user message only**, calls `attachmentsToContent(attachmentIds, workspaceId)` and merges the blocks into that message's `content` (before its text). History messages keep today's text-only mapping (`route.ts:312-315`) to protect the token budget; `tool-*`/non-text parts stay excluded.

### Message-history rendering (persistence pinned, SF-2)

13. Attachment chips render in history (filename + MIME-aware icon: PDF‚Üí`FileText`, image‚Üí`Image`, spreadsheet‚Üí`Sheet`, other‚Üí`Paperclip`); a chip click downloads via `getFileDownloadUrl` (`app/actions/files.ts:1291`). **Persistence path:** attachments belong to the **user** message, which is saved client-side via `saveChatMessage` (`app/actions/ai-chat.ts:83`, called from `use-chat-interface.ts:238`). `saveChatMessageSchema.metadata` is already an optional Json field (`app/actions/ai-chat.ts:44`) ‚Üí store `metadata.attachments: { fileId, filename, mimeType }[]` there (**no migration**). `getChatHistory` already returns `metadata` (`ai-chat.ts:208,225` ‚Üí `ChatMessageData.metadata:69`), so `chat-message.tsx` renders chips from `message.metadata.attachments` on reload.
14. Chip rendering is **additive** in `chat-message.tsx` ‚Äî it does not change the existing `text` / `reasoning` / `tool-*` part rendering.

### RAG / Filer exclusion ‚Äî chat attachments are conversational, not curated (SF-1)

15. Chat attachments (`category = 'CHAT_ATTACHMENT'`) are **NOT embedded into the RAG index.** Guard the `extract-files` cron's embed seam (`app/api/cron/extract-files/route.ts:197-199`, the `DONE && markdown` ‚Üí `syncWorkspaceChunks` call, which passes `category` at `:206`): **skip `syncWorkspaceChunks` when `file.category === 'CHAT_ATTACHMENT'`.** The file is still *extracted* (so AC 9's text path + on-demand extraction work) ‚Äî only the embed is skipped. Result: a chat attachment never appears in `search_workspace_files` (17.9c) results.
16. Chat attachments are **NOT listed in Filer.** `getWorkspaceFiles` (`app/actions/files.ts:730`, `where` built at `:744`, `findMany` at `:778`) excludes `category = 'CHAT_ATTACHMENT'` by default (e.g. `where.category = { not: 'CHAT_ATTACHMENT' }` when no explicit category filter is supplied). Task/list-item file pickers (`getFilesForTask:2303`, `getFilesForListItem:2389`) likewise must not surface chat attachments.
17. **Promotion seam (for [[19.1b]]):** the exclusion is keyed purely on `category`, so promoting a chat attachment to a curated file is a category flip (`CHAT_ATTACHMENT` ‚Üí `OVRIGT`/etc.) + one `syncWorkspaceChunks` embed call (text already extracted). Do **not** hard-couple the exclusion anywhere that a category change wouldn't reverse. (The affordance + promote action itself is out of scope ‚Äî 19.1b.)
