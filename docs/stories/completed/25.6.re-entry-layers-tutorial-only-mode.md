# Story 25.6: Re-entry layers + tutorial-only mode (B.6)

## Status

Done

## Story

**As a** Laglig.se user who has already dismissed the first-run modal — either by clicking Generera + Minimera, by picking Mall, by completing the Importera handoff, or by hitting Hoppa över — but who later wants to re-read the tutorial, re-open the Feedback tab, or check the generation status mid-flight,
**I want** persistent, low-friction re-entry to the modal via (a) a corner FAB that always reflects the current generation state, and (b) a Hjälp-menu entry in the sidebar that always works regardless of where I am,
**so that** the tutorial is not a one-shot experience trapped behind a dismissal flag, and so the team can ship deep-linkable tutorial URLs (`?onboarding=tutorial&tab=ai_agent`) to support customers without forcing a fresh signup.

This is the **B.6 wedge** of Epic 25 — the final story in the epic. B.0–B.5 shipped the modal itself + its 6 tabs + done states + the Feedback form, but the modal is currently a **one-shot**: once `first_run_dismissed_at` is set (which happens on every successful path choice OR Minimera click) the `firstRunOpen` guard permanently returns `false`. The only way to re-reach the tutorial today is a DB reset (`pnpm tsx scripts/reset-nordviken-onboarding.ts`) — clearly not a user-facing path. B.6 closes this gap with the three-layer re-entry hierarchy from arch §6.5:

| Layer | Surface | Condition | Opens modal at |
|---|---|---|---|
| 1 | First-run auto-open (B.0, shipped) | `firstRunOpen === true` | `path-choice` |
| 2 | **Corner FAB (NEW in B.6)** | `fabVisible === true` (post-first-dismiss, FAB itself not dismissed, not skipped) | Last mode if work running; `tutorial-only` if done |
| 3 | **Hjälp sidebar entry (NEW in B.6)** | Always | `tutorial-only` (via URL `?onboarding=tutorial`) |

Concretely B.6 adds:

1. **New `'tutorial-only'` Step value** in `<FirstRunModal>`'s union — same body as `'tutorial'` but the progress strip is hidden (no in-flight work to track). Tabs include all 7 (laglista through Feedback). Reachable via FAB-when-done OR Hjälp menu OR URL deep-link.
2. **Real `fabVisible` + `fabState` derivation** in `lib/onboarding/get-onboarding-state.ts` (currently stubbed to `false` + `'idle'` per the B.0 deferral comment at lines 8-10).
3. **New `<OnboardingFab>` component** at `components/features/onboarding-modal/onboarding-fab.tsx` — bottom-right floating button on `/dashboard`, three visual states (`working` spinner + "Genererar laglista..." pill / `done` lightbulb + "Tutorial" tooltip / `idle` same as done), tiny X on hover to call `dismissOnboardingFab()`.
4. **New Hjälp sidebar entry** added to `components/layout/left-sidebar.tsx` bottom section (above or below "Inställningar"), linking to `/dashboard?onboarding=tutorial`. Lucide `LifeBuoy` or `HelpCircle` icon. Always rendered regardless of onboarding state.
5. **URL deep-link handling** in `<HemPage>`: read `?onboarding=tutorial[&tab=<tabId>]` via `useSearchParams`, mount `<FirstRunModal>` at `step='tutorial-only'` with `initialTab=<tabId>`, replace (not push) the URL on close so onboarding state doesn't pollute browser history.
6. **`<TutorialStep>` mode prop**: accepts `mode: 'tutorial' | 'tutorial-only'`; in `tutorial-only` the `<ProgressStrip>` is not rendered, the tab strip + tab body + Minimera footer stay as-is.
7. **Enable the three done-step "Fortsätt utforska" CTAs**: drop the `aria-disabled` + `<Tooltip>` wrapper + visual classes from `done-generate-step.tsx`, `done-import-step.tsx`, `done-template-step.tsx`; restore `onClick={onKeepExploring}`. The parent `<FirstRunModal>` wires `onKeepExploring` to transition the modal to `step='tutorial-only'`. Telemetry: `done_cta_clicked.cta='keep_exploring'` (taxonomy already reserved in arch §5.2 per 25.4 Task 11).
8. **Banner ↔ FAB visibility resolution**: hide `<LawListGenerationProgress>` on `/dashboard` when `onboardingState.fabVisible === true` (per arch §14 PM recommendation — the FAB carries the "Genererar laglista..." pill state already; rendering both is redundant).

The intentionally-deferred open question from 25.4 (whether FAB re-entry should restore the user to `done-*` if they dismissed mid-done-state) is **explicitly out of scope** here — see Out of Scope. B.6 ships the simpler "FAB always opens at last mode-or-tutorial-only" behaviour; smart done-state restoration is a future hardening pass if the data warrants it.

### Owner-ack'd decisions (v0.1 draft)

1. **FAB scope: `/dashboard` only via `<HemPage>` mount (NOT every workspace route).** Decision rationale: arch §14 Q4 left this open ("Render only on /dashboard, or on every workspace route?"); B.6 picks dashboard-only because (a) the existing `getOnboardingState` is called in the dashboard page server component, so the FAB visibility is already computed there; (b) mounting on every workspace route would require lifting the derivation into `app/(workspace)/layout.tsx` which expands scope; (c) the FAB's "Genererar laglista..." pill is most useful on the dashboard where the user is making decisions about the laglista anyway. Future enhancement: lift to workspace-layout if signals show users want FAB access from `/laglistor` or `/tasks`.

2. **Banner ↔ FAB visibility: hide banner when FAB visible.** Per arch §14 Q5 PM recommendation + AC 32. Trade-off: users on stale workspaces (FAB dismissed) who trigger generation will see the banner — which is correct behaviour (the banner is the inline fallback when neither FAB nor modal is showing the work state).

3. **Tutorial-only tab seeding: default to `'laglista'`, URL param overrides.** Per arch §14 Q7 PM recommendation. Predictable default + deep-link flexibility. Future enhancement: remember last-viewed tab via `first_run_tabs_viewed` schema column (already exists from B.0) — but B.6 keeps it simple.

4. **FAB position: `fixed bottom-6 right-6 z-40`.** Trade-offs: (a) `z-40` is one below Radix Dialog's z-50, so the FAB naturally hides behind the modal — no extra display:none needed; (b) `bottom-6 right-6` clears the chat input on the dashboard (which is centred max-w-2xl); (c) doesn't overlap with the existing dashboard banner since banner is hidden when FAB visible per AC 32.

5. **Hjälp sidebar entry: labelled "Guide" + LifeBuoy icon.** Story 25.4 referred to it as "Hjälp menu" / "Hjälp sidebar entry" but the in-product label needs to feel friendly. "Guide" is short, matches the visual rhythm of "Hem" / "Krav" / "Mallar" etc. "Tutorial" was considered but feels foreign in a Swedish UI. "Hjälp" was considered but implies support/contact-us. Final pick: **"Guide"** + LifeBuoy. Owner can swap in <1 min if a different label feels better.

6. **`onKeepExploring` enables the secondary CTA in done steps; primary CTA still routes away.** No CTA hierarchy change in B.6 — primary still wins. The "Fortsätt utforska" secondary just goes from disabled-with-tooltip to enabled-with-onClick. Architecturally, this is the simplest possible B.6 — no per-done-step state-machine extension.

7. **Smart done-state restoration NOT implemented in B.6.** Per the 25.4 Completion Notes "Flagged for B.6" item: persisting `last_modal_step` so the FAB could land users back on `done-generate` if they dismissed mid-done. Considered but rejected for B.6: (a) requires a new schema column or session storage; (b) the FAB's current "tutorial-only on done" target is consistent with arch §6.5; (c) the done states are by-design celebratory one-shots; if a user dismissed during a done state they probably saw it. Future hardening if data warrants.

### Key decisions baked into this story

- **`'tutorial-only'` is a sibling Step, not a `mode` on `'tutorial'`.** Reasoning: keeping it as a discrete Step value makes the JSX dispatch + the width conditional + `headerForStep` cleaner. Trade-off: one more case in three switch-like branches, but each addition is 1-5 lines.
- **The `<TutorialStep>` component DOES use a `mode` prop internally** (per AC 22) — because the only difference between tutorial and tutorial-only is the progress strip's presence. A separate `<TutorialOnlyStep>` component would duplicate ~280 lines of tab-strip + tab-body code for a single conditional.
- **FAB lives in `<HemPage>` not `app/(workspace)/layout.tsx`.** Per Owner-ack'd decision 1.
- **URL deep-link uses query params, not path segments.** `?onboarding=tutorial` not `/dashboard/tutorial`. Reasoning: keeps the route tree clean (no new page file), survives reload, integrates trivially with `useSearchParams`. Matches arch §6.5 line 374-377 spec.
- **`router.replace` (not `push`) on modal close** — per arch §6.5 line 377 + AC 14. Onboarding state should not pollute browser back-button history.
- **No `<FirstRunModal>` re-architecture for B.6.** The shell stays the same; only step-union + props extend. Same precedent as 25.4's step-union extension.
- **No new dependencies.** All new icons (`Lightbulb`, `LifeBuoy`, `X`) are in lucide-react already.

### Performance Considerations

**PASS with one note.**

- **SWR cache sharing**: when the modal is open, both `<TutorialStep>` (via `<ProgressStrip>`) AND `<HemPage>` subscribe to `/api/workspace/generation-status`. SWR dedupes the request — single poll covers both consumers.
- **Conditional polling**: `<HemPage>` passes `null` as the SWR key when `!fabVisible || !workInFlight` — no idle polling on the dashboard when the FAB isn't visible or work isn't in flight.
- **Auto-stop on completion**: `completedDetectedRef.current` ensures `router.refresh()` fires exactly once; after the refresh, `workInFlight` becomes false → polling stops.
- **Bundle delta**: ~150 LOC client component (OnboardingFab) + ~50 LOC HemPage extensions + ~40 LOC util. Negligible.
- **Note (non-blocking)**: 3-second SWR poll interval generates ~20 req/min for the in-flight window. Acceptable for the 5-min average generation duration; could relax to 5s if production volume warrants.
