# Story 17.16: Kravpunkter — Structured Compliance Checklist per Law List Item

## Status

Done

## Story

**As a** compliance manager or workspace member,
**I want** to define specific compliance requirements (kravpunkter) for each law list item and link evidence documents to each requirement,
**so that** I can clearly track which obligations apply to our organization, whether they are fulfilled, and what evidence supports each one — enabling structured audit readiness and traceable compliance.

## Acceptance Criteria

### Data Model

1. New `LawListItemRequirement` Prisma model with: `id` (UUID PK), `list_item_id` (FK → LawListItem, cascade delete), `text` (String), `is_fulfilled` (Boolean, default false), `position` (Float, default 0 — **matches `LawListItem.position` pattern** to allow fractional midpoints for drag-and-drop reorder without unique-constraint conflicts), `created_by` (FK → User), `created_at`, `updated_at`. Indexes: `@@index([list_item_id, position])`. Table map: `law_list_item_requirements`.
2. New `RequirementEvidenceLink` junction model with: `id` (UUID PK), `requirement_id` (FK → LawListItemRequirement, cascade delete), `file_id` (FK → WorkspaceFile, nullable), `workspace_document_id` (FK → WorkspaceDocument, nullable — prefixed per Epic 17 naming-collision rule to avoid ambiguity with `LawListItem.document_id` which FKs to `LegalDocument`), `linked_at`, `linked_by` (FK → User). Unique constraints: `[requirement_id, file_id]`, `[requirement_id, workspace_document_id]`. Table map: `requirement_evidence_links`. **Exactly one** of `file_id`/`workspace_document_id` must be set (XOR — validated at application level via Zod; both-null and both-set both rejected).
3. Relations added: `LawListItem` has many `LawListItemRequirement`; `WorkspaceFile` and `WorkspaceDocument` have many `RequirementEvidenceLink`.

### UI — Kravpunkter Section

4. The existing "Hur efterlever vi kraven?" accordion in the law list item modal is restructured into a single accordion item with two visual sub-sections: **top** is "Kravpunkter" (structured checklist, new), **bottom** is "Kommentar" (the existing free-text rich text editor, renamed, retaining all existing content and functionality).
5. Each kravpunkt row displays: checkbox (fulfilled/not), requirement text (inline editable on click), evidence count badge (e.g. "2 bevis") when evidence is linked, expand/collapse chevron to show linked evidence, and a delete button (X icon) with confirmation toast.
6. Expanded evidence view per kravpunkt shows: list of linked files and documents with name/title, type icon, and unlink button (X). A "+ Länka bevis" action opens a picker allowing the user to link files (via `FilePickerModal`) or workspace documents (via `DocumentPickerModal`). Supports multiple evidence items per kravpunkt.
7. "+ Lägg till kravpunkt" button below the checklist — inline text input appears on click, Enter to save, Escape to cancel. No modal. Focus auto-moves to the new input.
8. Progress indicator in the accordion trigger header: "Kravpunkter — N/M uppfyllda" showing fulfilled vs total count. Hidden when no kravpunkter exist (accordion trigger falls back to current label).
9. Empty state when no kravpunkter exist: "Inga kravpunkter definierade" with a muted prompt "Lägg till din första kravpunkt" and the add button.
10. When read-only (user lacks edit permission), checkboxes are disabled, no add/delete/edit/link/unlink actions shown. Evidence list is still viewable and expandable.
11. Error handling: if a server action fails during inline add, checkbox toggle, or evidence link, show a `toast.error()` with the error message. Revert optimistic state on failure.

### Compliance Status Suggestion

12. When kravpunkter exist for a law list item, show a small info icon (tooltip) next to the "Efterlevnad" dropdown in `DetailsBox` suggesting a status based on checklist completion: 0 fulfilled → "Ej påbörjad" (`EJ_PABORJAD`), some → "Pågående" (`PAGAENDE`), all → "Uppfylld" (`UPPFYLLD`). Values must match the existing `ComplianceStatus` enum at `prisma/schema.prisma:605` — no new enum values introduced. This is informational only — the user retains full manual control of the compliance status dropdown.

### Server Actions

13. `createRequirement(listItemId, text)` — creates kravpunkt at next position. Validates input with Zod (text: string, max 500 chars). Logs to ActivityLog. Returns created requirement.
14. `updateRequirement(requirementId, { text?, is_fulfilled? })` — updates text or fulfillment. Validates with Zod. Logs to ActivityLog with old/new values.
15. `deleteRequirement(requirementId)` — deletes requirement and its evidence links (cascade). Logs to ActivityLog.
16. `reorderRequirements(listItemId, orderedIds[])` — updates positions in batch. No activity log needed.
17. `linkEvidenceToRequirement(requirementId, { fileId?, workspaceDocumentId? })` — links evidence. Zod enforces XOR (exactly one of `fileId`/`workspaceDocumentId` set). Prevents duplicates (unique constraint). Logs to ActivityLog.
18. `unlinkEvidenceFromRequirement(requirementId, { fileId?, workspaceDocumentId? })` — removes evidence link. Zod enforces XOR. Logs to ActivityLog.
19. `getRequirementsForListItem(listItemId)` — returns all kravpunkter with their linked evidence (files + documents), ordered by position.
20. All server actions use `withWorkspace` middleware for authorization and workspace isolation, following the exact pattern in `app/actions/legal-document-modal.ts`.

### Audit & Activity

21. All create/update/delete/link/unlink operations logged to `ActivityLog` with entity_type `'requirement'`. `ActivityLog.entity_type` is free-form `String` (schema.prisma:1288) — no enum extension or migration required.
22. Activity tab in the law list item modal shows requirement events: "La till kravpunkt: {text}", "Markerade som uppfylld: {text}", "Tog bort kravpunkt: {text}", "Länkade bevis till: {text}".

### Integration Requirements

23. Existing `compliance_actions` field content preserved — it continues to power the "Kommentar" sub-section. No data migration needed.
24. Existing compliance status functionality unchanged — manual override always available via DetailsBox dropdown.
25. Existing evidence (Bevis) section in the right panel and evidence tab remain independent — general evidence at the law list item level is separate from requirement-specific evidence.
26. Existing `FilePickerModal` and `DocumentPickerModal` reused for evidence linking.
27. No regression in existing law list item modal functionality.

### Quality Requirements

28. All server actions covered by unit tests: CRUD, workspace isolation, cascade delete, reorder, link/unlink, duplicate prevention, Zod validation.
29. Component tests for: kravpunkter list rendering, inline add flow, checkbox toggle, evidence expand/collapse, empty state, read-only mode, progress indicator.
30. Performance: modal with 20+ kravpunkter (each with 3 evidence links) loads in under 1 second.

### Key Patterns to Follow

**Server Actions:** Follow `updateListItemComplianceActions` in `legal-document-modal.ts`:
- Zod schema validation at top
- `withWorkspace(callback, 'tasks:edit')` for auth
- Prisma query with workspace isolation join
- `logActivity()` call
- Redis cache invalidation: `deleteCache(\`list-item-details:${listItemId}\`)`
- Return `ActionResult<T>` type

**SWR Optimistic Updates:** Follow ComplianceActions pattern in `compliance-actions.tsx` (lines 97-132):
- `mutate(\`list-item-extra:${listItemId}\`, updater, { revalidate: false })`
- Also update `mutate(\`list-item:${listItemId}\`, ...)`
- For kravpunkter, add new cache key: `list-item-requirements:${listItemId}`

**Activity Logging:** Use `logActivity()` from `lib/services/activity-logger.ts`:
```typescript
await logActivity(ctx.workspaceId, ctx.userId, 'requirement', requirementId, 'requirement_created', null, { text })
```

**Naming Convention:** New models use `LawListItem` prefix scope — child records inherit workspace scope through `LawListItem → LawList → workspace_id` join path. No `Workspace` prefix needed. **However:** FK columns pointing at `WorkspaceDocument` **must** use the `workspace_document_id` column name per Epic 17's collision rule — bare `document_id` in this schema means FK→`LegalDocument`.

**Evidence XOR Zod Pattern:**
```typescript
const EvidenceInputSchema = z.object({
  fileId: z.string().uuid().optional(),
  workspaceDocumentId: z.string().uuid().optional(),
}).refine(
  (data) => (data.fileId ? 1 : 0) + (data.workspaceDocumentId ? 1 : 0) === 1,
  { message: 'Exactly one of fileId or workspaceDocumentId must be provided' }
)
```

**Position Float Pattern:** Mirrors `LawListItem.position` (schema.prisma:278). On reorder, assign evenly spaced values (1000, 2000, 3000...) so inserts between two items can use the midpoint. No unique constraint → no swap-dance under concurrent updates.

**Component Pattern:** KravpunkterChecklist should be a self-contained component that fetches its own data (like TasksAccordion), not rely on parent to pass requirement data. This keeps the ComplianceActions refactor minimal.
