## Story

**As a** compliance manager reading the agent's response,
**I want** every inline citation to correspond to a source the agent actually retrieved and verified,
**so that** I can trust the references and inspect the actual legal text behind each claim.

## Problem

The agent currently constructs citation labels freehand. When `search_laws` returns chunks for `kap3.§1`, `kap3.§2`, `kap3.§3`, but NOT `kap3.§3a`, the agent may still produce `[Källa: SFS 1977:1160, Kap 3, 3a §]` based on contextual inference from surrounding text. This:

1. **Violates the guardrail** — the system prompt says "Citera ENBART paragrafer som du har hämtat via search_laws" but the agent ignores this because it has no structured mechanism to enforce it.
2. **Breaks the sidebar** — the citation pill resolves to a document-level fallback (AI summary), not the actual section text.
3. **Erodes trust** — the user clicks a citation expecting to see the law text and gets nothing, or worse, a generic summary masquerading as the specific section.

## Root Cause

The `search_laws` tool returns `path: "kap3.§3"` and `documentNumber: "SFS 1977:1160"` as separate fields, leaving the agent to assemble the citation label itself. The agent sometimes:
- Cites a section it inferred from the content of an adjacent chunk
- Constructs a citation label that doesn't exactly match any returned path
- References sections from its training data rather than retrieved chunks

## Acceptance Criteria

### Pre-formatted Citation Keys in Tool Results

1. The `search_laws` tool includes a `citationKey` field in each result: a ready-to-use string like `SFS 1977:1160, Kap 3, 3 §` derived from the chunk's `documentNumber` + `path` using the existing `chunkCitationKey()` utility from `lib/ai/citations.ts`.
2. The `get_document_details` tool includes `citationKeys: string[]` — one per section extracted from the document content, so the agent knows exactly which sections are available to cite.
3. Citation keys use a consistent format matching the `[Källa: ...]` pattern the rehype plugin expects.

### System Prompt: Two-Tier Citation Model

4. The `<citation_rules>` section in `lib/agent/system-prompt.md` introduces a clear two-tier model:
   - **Tier 1 — Cited sources (`[Källa: ...]`):** Only use the exact `citationKey` strings returned by `search_laws`. These become clickable citation pills with verified source text behind them. The agent must never construct a `[Källa:]` marker for a section it didn't retrieve.
   - **Tier 2 — Cross-references (plain text):** When the retrieved text mentions another section the agent hasn't fetched (e.g., "se 3a §"), the agent may reference it as plain text — "(se 3a §)" or "enligt 3a §" — without a `[Källa:]` marker. This is informative context, not a substantiated citation.
5. The prompt includes an example showing both tiers in one response:
   - `"Arbetsgivaren ska systematiskt planera arbetsmiljöarbetet[Källa: SFS 1977:1160, Kap 3, 3 §]. Lagen anger också att arbetsgivare och arbetstagare ska samverka (se 3a §)."`
   - The first reference has a `[Källa:]` (retrieved). The second is a plain cross-reference (mentioned in the retrieved text but not itself retrieved).
6. The prompt explicitly states: "Om du vill ge en paragraf full källstatus med [Källa:]-markering, sök efter den med search_laws först. Korsreferenser som nämns i hämtad text kan anges utan [Källa:]-markering som vägledande hänvisningar."

### Validation

7. After the changes, every `[Källa: ...]` marker in the agent's responses should use an exact `citationKey` string from tool results. Cross-references to unretrieved sections appear as plain text without citation markers.
8. The citation source map (`extractSourcesFromToolResult`) should register entries keyed by the same `citationKey` strings, ensuring 1:1 resolution — every citation pill leads to real source text in the sidebar.

## Dev Notes

### Existing `chunkCitationKey` Utility

Already exists at `lib/ai/citations.ts:245`:

```typescript
export function chunkCitationKey(documentNumber: string, path: string): string | null {
  const match = path.match(/^kap(\w+)\.§(\w+)$/)
  if (!match) return null
  const [, chapter, section] = match
  if (chapter === '0') return `${documentNumber}, ${section} §`
  return `${documentNumber}, Kap ${chapter}, ${section} §`
}
```

This already produces the exact format the rehype plugin expects. Just needs to be wired into the tool responses.

### System Prompt Citation Rules (Current)

At `lib/agent/system-prompt.md` lines 51-68, the `<citation_rules>` section already says "Citera ENBART paragrafer och kapitel som du har hämtat via search_laws." The issue is that this is a natural-language instruction the LLM doesn't reliably follow. Adding the structured `citationKey` field makes it concrete — the agent has the exact string to use rather than having to construct it.

### Why Not Just a Prompt Fix?

A prompt-only fix ("be more careful about citations") is unreliable — LLMs don't perfectly follow negative instructions. The structural fix (providing `citationKey` in tool results) gives the agent the right answer rather than asking it to avoid the wrong one. This follows the Anthropic principle: "make the right thing the easy thing."

### Why Two Tiers Instead of Blocking Cross-References?

Swedish laws frequently cross-reference each other — §3 mentions §3a, §2 references Kap 5. If we block the agent from mentioning any section it hasn't retrieved, the responses become artificially narrow and miss relevant context a human compliance expert would naturally include.

The two-tier approach preserves this contextual richness while being honest about what's substantiated:
- **`[Källa:]` pill** = "I retrieved this, you can click to verify" → trust signal
- **Plain "(se 3a §)"** = "this is referenced in the law I read, you can explore further" → context signal

The user intuitively understands the difference: clickable = verified, plain text = informational pointer. This matches how legal professionals work — they cite primary sources but routinely mention related provisions in running text.

### Hallucination Surface After This Change

The remaining risk is: could the agent fabricate a plain-text cross-reference? Theoretically yes, but:
1. It's reading the reference from actual retrieved legal text, not inventing it
2. Swedish law has rigid numbering — section numbers aren't creative
3. A plain text reference carries no false authority (no citation pill, no source backing)
4. The critical path (cited sources with pills) is fully locked down via `citationKey`
