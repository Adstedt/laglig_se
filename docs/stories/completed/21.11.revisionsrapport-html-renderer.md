# Story 21.11: Revisionsrapport HTML renderer

## Status

Done

## Story

**As a** lead auditor preparing to hand a completed or sealed Lagefterlevnadskontroll cycle off to an external certifieringsorgan or archive it as internal documentation,
**I want** a Laglig-branded, deterministic HTML revisionsrapport rendered from the cycle's metadata, items, findings, and (when sealed) evidence snapshots,
**so that** I can preview the report inline in the Rapport tab before it is converted to PDF in Story 21.12, and the report is ready for certification-grade downstream use (reproducible bytes, seal-hash visible, ISO-19011-style sections, Swedish-first copy).

## Acceptance Criteria

### Functional

1. **Renderer module + pure signature.** A new file `lib/compliance-audit/revisionsrapport-renderer.ts` exports a pure function
   ```ts
   renderRevisionsrapport(input: RevisionsrapportInput): string
   ```
   that takes a fully-loaded cycle aggregate (cycle metadata + items + findings + evidence snapshots) and returns a complete, self-contained HTML document string (`<!DOCTYPE html>…</html>`). Pure = no DB access, no `fetch`, no `Date.now()`, no `Math.random()`, no external I/O — every rendered string is derived from `input`. **All timestamps rendered in the HTML come from `input` fields** (never `new Date()` inside the renderer) to satisfy determinism (AC 6). Source: architecture §5.1 (`renderRevisionsrapport(cycle: CycleWithRelations): string`), FR12.

2. **`RevisionsrapportInput` shape.** Exported alongside the renderer:
   ```ts
   export interface RevisionsrapportInput {
     cycle: CycleDetail                    // from app/actions/compliance-audit-cycle.ts
     items: CycleItemRow[]                 // from app/actions/compliance-audit-item.ts
     findings: FindingRow[]                // from app/actions/compliance-finding.ts
     snapshots: EvidenceSnapshotRow[]      // new — see below; empty [] for non-SEALED cycles
     workspace: { id: string; name: string }
     generatedAt: string                   // ISO 8601, supplied by caller (deterministic input)
   }
   ```
   - `CycleDetail`, `CycleItemRow`, `FindingRow` are already-exported shapes; re-use them directly (do not redeclare).
   - `EvidenceSnapshotRow` is **new, defined in this story** at the top of the renderer file:
     ```ts
     export interface EvidenceSnapshotRow {
       id: string
       lawListItemId: string | null
       requirementId: string | null
       evidenceKind: 'FILE' | 'DOCUMENT'
       evidenceSha256: string
       capturedAt: Date
       // Display data resolved at read time:
       displayName: string       // file name or document title
     }
     ```
     Story 21.9 (seal) will hydrate this shape from `ComplianceEvidenceSnapshot` + joined `WorkspaceFile` / `WorkspaceDocument` when snapshots exist. For AVSLUTAD cycles (pre-seal), pass `[]`.
   - `workspace` + `generatedAt` are caller-supplied so the renderer stays pure — the RSC consumer fills them from `ctx.workspaceId` / `new Date().toISOString()`.

3. **Data fetch server action — `getRevisionsrapportInput(cycleId)`.** New export in **`app/actions/compliance-audit-report.ts`** (NEW FILE — first action in the module; Story 21.12 will extend it with PDF generation + storage — see Dev Notes "Action-name deviation from architecture §5.1" for rationale). Contract:
   - **Zod input:** `{ cycleId: string().uuid() }`.
   - **Auth:** follow the established module convention (verified in `compliance-audit-cycle.ts:366-426` `listCyclesForWorkspace` and `:449-469` `getCycleById`):
     ```ts
     return await withWorkspace(async (ctx) => {
       if (!(ctx.hasPermission('activity:view') || ctx.hasPermission('tasks:edit'))) {
         return { success: false, error: 'Behörighet saknas' }
       }
       // …load + render…
     }, 'read')
     ```
     The outer `withWorkspace` scope is literal `'read'` (base gate), and the OR-check for `activity:view || tasks:edit` is inline. **Do not invent a `withWorkspace(cb, 'activity:view')` variant** — that's not how the second arg is used in this module. The report is read-only projection and must be visible to AUDITOR role (FR16 / architecture §12.2), which `activity:view` covers.
   - **Data loaded in parallel** via `Promise.all` (single round-trip clump):
     - `getCycleById(cycleId)` → `cycle`
     - `getCycleItemsForCycle(cycleId)` → `items`
     - `listFindingsForCycle({ cycleId })` → `findings`
     - `loadEvidenceSnapshotsForCycle(cycleId, ctx.workspaceId)` — **new module-private helper** in the same file. Joins `ComplianceEvidenceSnapshot` → `WorkspaceFile` / `WorkspaceDocument` with snake_case→camelCase mapping; returns `EvidenceSnapshotRow[]`. Filters `cycle_id = cycleId` and `cycle.workspace_id = ctx.workspaceId` (tenant isolation; mirror the `getCycleById` pattern at `compliance-audit-cycle.ts:437-474`).
   - **Edge cases:** if `cycle` is not found → `'Kontrollen hittades inte'`; if any of the parallel reads returns `success: false`, propagate the first error (Swedish).
   - **`generatedAt`** is the timestamp at action call time: `new Date().toISOString()`. Passed down to the renderer as input — the renderer itself does not call `new Date()`.
   - **Returns:** `{ success: true, data: { input: RevisionsrapportInput, html: string } }`. The action pre-renders the HTML server-side so the RSC consumer can send HTML → client without a second round-trip. On failure: `{ success: false, error }`.
   - **Caching:** no `revalidatePath` — this is a read action. No Redis caching layer added in this story; preview is freshly computed on every tab mount. Story 21.12 will add storage for PDF output, not HTML caching.

4. **Mandatory sections (FR12).** Rendered in this order. Each is a `<section>` with a unique `id` for anchoring from the TOC (AC 7). Section titles swap between the default (neutral Swedish) and an EXTERN variant based on `cycle.auditType` (FR18):

   | Section `id` | Default (`INTERN` + unset) | EXTERN — Revisionsrapport framing |
   |---|---|---|
   | `titelsida` | (cover, no heading — title is the cycle name) | same |
   | `omfattning` | Omfattning | Omfattning |
   | `revisionskriterier` | Kriterier | Revisionskriterier |
   | `metodik` | Metod | Metodik |
   | `sammanfattning` | Sammanfattning | Sammanfattning |
   | `avvikelser` | Avvikelser | Avvikelser |
   | `observationer` | Observationer | Observationer |
   | `forbattringsforslag` | Förbättringsförslag | Förbättringsförslag |
   | `styrkor` | Styrkor | Styrkor |
   | `konklusion` | Konklusion | Konklusion |
   | `signatarer` | Signering | Signatarer |

   Swap table is exported as a module-private `SECTION_LABELS: Record<'internal' | 'external', Record<SectionId, string>>`. **Do not invent additional section ids.** Epic AC 2 explicitly lists this set.

5. **Per-section content rules (Swedish copy, deterministic):**
   1. **Titelsida (cover page)** — renders:
      - Cycle `name` as `<h1>` (22pt).
      - Metadata block, label copy swaps by `cycle.auditType` per FR18 (neutral-Swedish default; ISO-flavoured for EXTERN):
        - **INTERN (default)** — `Laglista: {cycle.lawList.name}`, `Period: {scheduled_start}–{scheduled_end}` (date-fns Swedish locale `d MMM yyyy`), **`Kontrolledare: {cycle.leadAuditor.name ?? '—'}`**, **`Kontrolltyp: Intern`**, `Status: {cycleStatusLabel(cycle.status)}`, `Workspace: {workspace.name}`, `Rapport genererad: {date-fns format of generatedAt, 'd MMM yyyy HH:mm'}`.
        - **EXTERN** — same block with `Revisionsledare: …`, `Revisionstyp: Extern`. Other labels unchanged.
      - The Revisionsledare / Kontrolledare swap is deterministic (no conditional rendering of the row — the row is always present; only the label string flips).
      - **If `cycle.sealHash` is non-null**: a prominent seal block rendered with `class="seal-block"`:
        - Heading `Fastställd kontroll` + timestamp `Fastställd: {cycle.sealedAt}` (date-fns `d MMM yyyy HH:mm`) + `Fastställd av: {cycle.sealedBy.name ?? '—'}`.
        - Full 64-char hash on its own line in `<code class="seal-hash">{cycle.sealHash}</code>` — NOT truncated on the cover (FR13 explicitly requires visible seal hash on cover + footer).
      - **If `cycle.sealHash` is null**: no seal block rendered; the status badge carries the state. Do NOT render a placeholder.
   2. **Omfattning** — a prose paragraph describing the scope, composed from `cycle.scopeDefinition`:
      - `{kind: 'all'}` → `"Kontrollen omfattar samtliga dokument i laglistan "{lawList.name}" ({items.length} dokument)."`
      - `{kind: 'groups', groupIds}` → `"Kontrollen omfattar följande grupper i laglistan "{lawList.name}": {group names}. Totalt {items.length} dokument."` (group names resolved from the items' `groupName` set, deduped + alphabetised).
      - `{kind: 'items', itemIds}` → `"Kontrollen omfattar {items.length} specifikt utvalda dokument ur laglistan "{lawList.name}"."`
      - **Followed by** a compact summary table: columns `Dokument | Dok.nr | Grupp | Bedömning | Ansvarig | Signerad`. One row per item. `Bedömning` shows the Swedish label (see AC 6). `Signerad` shows `✓ {signedOffBy.name}` if signed, else `—`. Rows sorted by `groupName asc NULLS LAST, lawTitle asc` (stable input order from `getCycleItemsForCycle` is already this — reuse it).
   3. **Revisionskriterier / Kriterier** — prose paragraph, variant-dependent first sentence per FR18:
      - **INTERN:** `"Kontrollen genomförs mot kraven i laglistan "{lawList.name}" (intern kontroll)."`
      - **EXTERN:** `"Revisionen genomförs mot kraven i laglistan "{lawList.name}" (extern revision)."`
      - Second sentence (shared): `"Bedömningsgrund: gällande svensk lagstiftning samt organisationens egna kravpunkter per dokument i laglistan per {law_change_cutoff_date}."`
   4. **Metodik / Metod** — short auto-generated prose, variant-dependent verb per FR18:
      - **INTERN:** `"Genomgången har utförts som en strukturerad dokumentgranskning av varje ingående dokument. För varje dokument har efterlevnad bedömts på skalan Uppfylld / Delvis uppfylld / Ej uppfylld / Ej tillämplig, med motivering."`
      - **EXTERN:** replace "Genomgången" with "Revisionen" in the first sentence; otherwise identical.
      - Second sentence (shared): `"Eventuella avvikelser, observationer och förbättringsförslag dokumenteras separat. Varje bedömning har signerats av ansvarig. {auditType === 'EXTERN' ? 'Revisionen' : 'Kontrollen'} har letts av {leadAuditor.name}."`
      - For EXTERN only, append: `"Revisionens genomförande följer principerna i ISO 19011."` Do NOT append this for INTERN (FR18).
   5. **Sammanfattning** — stats table with these rows (in order):
      - Totalt antal dokument: `{items.length}`
      - Uppfyllda: `{count of efterlevnadsbedomning === 'UPPFYLLD'}`
      - Delvis uppfyllda: `{count of 'DELVIS'}`
      - Ej uppfyllda: `{count of 'EJ_UPPFYLLD'}`
      - Ej tillämpliga: `{count of 'EJ_TILLAMPLIG'}`
      - Obedömda: `{count of null}` (defensive; should be 0 for AVSLUTAD per Story 21.6 guard, but render the row anyway so the table is always the same shape)
      - Antal avvikelser: `{count of findings where type === 'AVVIKELSE'}`
      - — varav major: `{count where severity === 'MAJOR' AND type === 'AVVIKELSE'}`
      - — varav minor: `{count where severity === 'MINOR' AND type === 'AVVIKELSE'}`
      - Antal observationer: `{count of type === 'OBSERVATION'}`
      - Antal förbättringsförslag: `{count of type === 'FORBATTRING'}`
      - Öppna / stängda findings: `{count closedAt === null} / {count closedAt !== null}`
   6. **Avvikelser** — for each finding where `type === 'AVVIKELSE'`, grouped by severity (`MAJOR` first, then `MINOR`, then any `severity === null` under a "Allvarlighetsgrad saknas" subheading). **Rationale for the null-severity branch:** Story 21.7 AC 4 enforces severity-required-for-AVVIKELSE server-side via Zod `.refine`, so this branch should be unreachable in production data. It exists strictly as defence against future schema drift (enum changes, migration import errors). The branch is covered by an explicit test (AC 14 "Defensive null fields") that asserts no-throw; do **not** interpret its presence as suggesting server-side validation is weak. Within each group, sort by `createdAt asc`. Each finding renders as a `<article class="finding">` block:
      - Heading `<h3>{title}</h3>` (includes a small severity badge: "Major" red / "Minor" amber; CSS classes only, no external icons).
      - Metadata row (muted, small): `Dokument: {lawListItem.title ?? '—'}` · `Kravpunkt: {requirement.text (truncated to 120 chars) ?? '—'}` · `Förfallodatum: {dueDate ?? '—'}` · `Status: {closedAt ? 'Stängd ' + dateFormat(closedAt) : 'Öppen'}`.
      - `<p class="description">{description}</p>` (escaped; newlines → `<br/>`).
      - If `rootCause`: `<div class="root-cause"><strong>Grundorsak:</strong> {rootCause}</div>`.
      - If `correctiveActionTaskId`: `<div class="corrective-action"><strong>Korrigerande åtgärd:</strong> uppgift {correctiveActionTask.title}{correctiveActionTask.completedAt ? ' (klar)' : ' (pågår)'}</div>`.
      - **Empty-group handling:** if there are zero AVVIKELSE findings total, render the section with a single line `"Inga avvikelser identifierade."` (not an empty section).
   7. **Observationer** — same structure as Avvikelser, filtered to `type === 'OBSERVATION'`, NO severity subheadings. Empty → `"Inga observationer identifierade."`.
   8. **Förbättringsförslag** — same structure, filtered to `type === 'FORBATTRING'`. Empty → `"Inga förbättringsförslag identifierade."`.
   9. **Styrkor** — MVP empty section with explanatory copy: `"Styrkor dokumenteras i kommande versioner av revisionsrapporten."` Keep the `<section id="styrkor">` rendered so the TOC entry + section anchors stay stable across future enhancement.
   10. **Konklusion** — auto-generated Swedish paragraph based on the efterlevnadsbedömning distribution:
       - **All `UPPFYLLD` or `EJ_TILLAMPLIG`** AND zero open AVVIKELSE findings → `"Samtliga tillämpliga krav i laglistan "{lawList.name}" bedöms som uppfyllda. Inga öppna avvikelser kvarstår vid rapporttillfället."`.
       - **Any `EJ_UPPFYLLD` present** OR **any open AVVIKELSE finding** → `"Kontrollen identifierade {N} krav som inte är uppfyllda samt {M} öppen(a) avvikelse(r) som kräver åtgärd. Se avsnittet Avvikelser för detaljer och ansvariga åtgärdsuppgifter."`.
       - **Any `DELVIS` present but no `EJ_UPPFYLLD` and no open AVVIKELSE** → `"Kontrollen bedöms i huvudsak uppfylla kraven i laglistan "{lawList.name}". {N} krav är delvis uppfyllda och bör följas upp enligt angivna förbättringsförslag."`.
       - **Mixed / fallback** → `"Kontrollen har genomförts och samtliga dokument har bedömts. Se sammanfattning och avvikelselistan för detaljer."`.
       - Selection is deterministic — rules are evaluated in the listed order; first match wins.
   11. **Signering / Signatarer** — table with columns `Dokument | Bedömning | Motivering | Signerad av | Datum`. One row per `ComplianceAuditItem` with `signed_off_at !== null` (defensive filter; by Story 21.6's Complete gate all items are signed, but render only the signed subset so a pre-complete preview of a PAGAENDE cycle shows a partial table cleanly). Sort same as Omfattning table. Motivering truncated to 200 chars with `…` on overflow.

6. **Efterlevnadsbedömning label helper.** Module-private `bedomningLabel(value: EfterlevnadsBedomning | null): string` returns `'Uppfylld' | 'Delvis uppfylld' | 'Ej uppfylld' | 'Ej tillämplig' | '—'` (null → dash). Used throughout sections 5.1, 5.5, 5.11. Keep adjacent to `SECTION_LABELS` in the renderer file so label maps are one grep hit away from each other.

7. **Table of contents.** Rendered above the first content section (after the cover), as a `<nav class="toc">` with `<ol>` of section-anchor links. Entries match the `SECTION_LABELS` chosen variant. TOC entries include Sammanfattning's top-level counters as a second-line sub-caption (e.g., `"Sammanfattning ({items.length} dokument, {findings.length} findings)"`) — purely for at-a-glance readability; stays deterministic.

8. **Laglig brand + typography tokens.** Renderer produces a standalone HTML document with inline `<style>` (same pattern as `lib/documents/tiptap-to-pdf.ts:24-70`). **No external CSS file** — PDF renderer in 21.12 needs the HTML to be self-contained. Token choices:

   - Body: 11pt / line-height 1.5 / `#1a1a1a`.
   - h1 22pt, h2 16pt (section headings), h3 13pt (finding titles), h4 11pt bold.
   - Tables: `border-collapse: collapse; width: 100%;` — td/th 1px `#ccc` border, padding 6px 8px, th `background: #f5f5f5; font-weight: 600;`.
   - Seal block: `background: #ecfdf5` (emerald-50), `border: 1px solid #10b981` (emerald-500), padding 16px, rounded 4px. Seal hash: `font-family: 'Courier New', monospace; font-size: 10pt; word-break: break-all;`.
   - Finding severity badges: `.badge-major` red `#dc2626` bg `#fef2f2`; `.badge-minor` amber `#d97706` bg `#fffbeb`.
   - Footer running text (AC 9) muted grey `#666`.
   - Page-break hints: `section { page-break-inside: avoid; }` on sections shorter than a typical page; `.finding { page-break-inside: avoid; }` on finding articles. These are hints for the Puppeteer pipeline in 21.12 — harmless in the inline preview.

9. **Footer string.** Rendered as a final `<footer>` block inside the HTML body:
   - Always includes: `"Rapport genererad {generatedAt formatted} · Kontroll-ID: {cycle.id.slice(0, 8)}… · Laglig.se"`.
   - **If `cycle.sealHash` is non-null**: append ` · Seal: {cycle.sealHash}` (full 64 chars — FR13 "visible on the cover page and in the manifest footer of the last page").
   - Puppeteer's header/footer template is a 21.12 concern; this body footer is sufficient for the HTML preview and serves as the source for 21.12's footer extraction.

    - Render branches by `cycleStatus`:
      - `PLANERAD` / `PAGAENDE` → placeholder copy `"Rapport blir tillgänglig när kontrollen slutförs (AVSLUTAD). Du kan slutföra kontrollen via åtgärdsmenyn när samtliga poster är signerade."` Do NOT call the fetch for these states (conditional SWR key: pass `null` when pre-complete).

Rationale: the report HTML is a full `<!DOCTYPE>` document with its own `<style>`; rendering it in an iframe isolates its style scope from the surrounding app layout and prevents accidental CSS collision with shadcn tokens. **Note:** `srcdoc`/auto-height-iframe is a new pattern in this codebase — no existing precedent to mirror (verified by grep; no other component uses `contentDocument.documentElement.scrollHeight`). Implement fresh with standard DOM: `iframe` height set via `ref` + `onLoad` handler that reads `iframe.contentDocument?.documentElement.scrollHeight ?? 600` and applies it to `iframe.style.height`. Cap at `2000px` to prevent runaway; if content exceeds the cap, the iframe gets its own scrollbar (acceptable for a preview surface).
    - **Loading state:** skeleton of the same iframe height (600px default) with `role="status"` + screen-reader text `"Laddar revisionsrapport…"`.
    - **Error state:** inline error card `"Rapporten kunde inte genereras: {error}"` + retry button that calls `mutate()`.

12. **Integration Verification.**
    - **IV1:** Renderer handles edge cases without throwing: zero findings, 100+ findings, all UPPFYLLD, all EJ_TILLAMPLIG, mixed distributions, a cycle with zero items (defensive — returns a recognisable empty-cycle HTML rather than an empty string), a finding with no `lawListItem` back-reference, a finding with `severity === null` (defensive branch).
    - **IV2:** Rendering a 200-item cycle with 50 findings completes in under 2 seconds on test hardware (Vitest performance assertion — `expect(durationMs).toBeLessThan(2000)` with 200-item fixture).
    - **IV3:** Existing document rendering (legal documents, amendments) unaffected — `lib/documents/tiptap-to-pdf.ts`, `lib/documents/tiptap-to-docx.ts` remain untouched; `pnpm test` on `tests/unit/lib/documents/**` stays green.

13. **Determinism — byte-identical reproduction (AC 6 of the epic).**
    - Given the same `RevisionsrapportInput`, two calls to `renderRevisionsrapport` produce **byte-identical strings**. This means:
      - No `Date.now()`, no `Math.random()`, no `crypto.randomUUID()` inside the renderer.
      - No `Set` / `Map` iteration without pre-sorting (key order determinism).
      - No non-stable ES2015 Map traversal — use explicit `.sort(compareFn)` on every array before iteration for display.
      - date-fns locale is fixed to `sv` explicitly (import `import { sv } from 'date-fns/locale'` and pass `{ locale: sv }`).
    - **Golden-fixture test** (unit test, blocking CI):
      - Fixture input at `lib/compliance-audit/__fixtures__/revisionsrapport-input.json` — a cycle with 3 items (UPPFYLLD / EJ_UPPFYLLD / EJ_TILLAMPLIG), 3 findings (1 AVVIKELSE MAJOR with task link, 1 OBSERVATION, 1 FORBATTRING closed), 1 evidence snapshot, audit_type = EXTERN, sealHash present.
      - Fixture output at `lib/compliance-audit/__fixtures__/revisionsrapport-output.html`.
      - Test reads both, compares `expect(renderRevisionsrapport(input)).toBe(expectedHtml)`.
      - **Fixture update policy (matches canonicalize convention per architecture §10.2):** any renderer change that intentionally mutates output requires a deliberate fixture update in the same PR with a one-line comment in the PR description justifying the change. CI test fails loudly if fixtures drift.

### Action-name deviation from architecture §5.1
