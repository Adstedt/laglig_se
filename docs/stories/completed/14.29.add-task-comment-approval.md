## Story

**As a** user whose compliance partner wants to leave a note on a task,
**I want** the proposed comment rendered inline for review before it posts,
**so that** the agent never writes into a task's comment thread without my sign-off, and the note carries my approval.

## Context

Tasks have a comment thread (the `Comment` model, polymorphic over `task_id` / `law_list_item_id`, with up-to-3-level threading). A natural agent behaviour after assessing a change or reviewing a task is to summarise its reasoning as a task comment ("Bedömning: den här ändringen påverkar inte vår rutin eftersom …"). Today the agent can only say this in chat; the note never lands on the task where the responsible person will see it.

This story adds an `add_task_comment` write tool that creates an `ADD_TASK_COMMENT` `PendingAgentAction`. It is the simplest of the Phase 7 sibling types: **append-only**, so there is no diff and no open-in-editor path — just a previewed comment the user approves or rejects. On approval the dispatch calls the existing `createComment` server action.

For v1 the agent posts at thread root (`parentCommentId` omitted) unless explicitly replying. The agent **must not** emit `@`-mention syntax: `createComment` parses `@[name](id)` tokens out of the `content` and writes them to `Comment.mentions`, firing MENTION notifications ([`app/actions/task-modal.ts:984-991`](../../app/actions/task-modal.ts)). "Mentions stays empty" is therefore enforced **only** by steering the agent's content (AC 5), not by the dispatch — so the tool description must explicitly forbid mention syntax.

## Acceptance Criteria

### Schema

1. `PendingAgentActionType` enum is extended with `ADD_TASK_COMMENT`. Additive migration (enum value only).
2. No new tables. `params` JSON shape:
   ```
   {
     taskId: string,
     taskTitle: string,          // denormalised at propose time for the renderer header (AC 7)
     content: string,
     parentCommentId?: string,   // omitted = thread root
     entity_version: string      // parent task.updated_at ISO at propose time (forward-compat with 14.31)
   }
   ```

### Tool

3. New tool `lib/agent/tools/add-task-comment.ts`. Zod schema mirrors `CreateCommentSchema` ([`app/actions/task-modal.ts:219-223`](../../app/actions/task-modal.ts)): `{ taskId: z.string().uuid(), content: z.string().min(1).max(5000), parentCommentId?: z.string().uuid(), execute: boolean }`. Empty content → `ToolError` ("Kommentaren får inte vara tom."); the `.max(5000)` ceiling matches the server schema so over-long input fails at tool time with a Swedish message rather than as a raw `ZodError` thrown from `createComment` at dispatch.
4. On `execute: false`: verify the task is in the active workspace via the same lookup `createComment` performs — `prisma.task.findFirst({ where: { id: taskId, workspace_id: workspaceId }, select: { id, title, updated_at } })` ([`app/actions/task-modal.ts:959-961`](../../app/actions/task-modal.ts)). Capture `title` → `params.taskTitle` and `updated_at` → `params.entity_version`, create the `PendingAgentAction` row, return the envelope with `data: { pendingActionId }`.
5. Tool description (Swedish): use when the user asks the agent to leave a note/comment on a task, or to record an assessment summary on the relevant task. The description **must explicitly instruct the agent not to write `@`-mention syntax** (`@[name](id)`) — the comment must be self-contained — because the dispatch's `createComment` would otherwise extract those tokens into `Comment.mentions` and fire notifications (see Context).
6. `execute: true` is **not** supported (returns a `ToolError` — matches 14.23).

### Renderer

7. New component `components/features/ai-chat/agent-action-renderers/add-task-comment-renderer.tsx`, registered in the `AgentActionCard` dispatch map and the batch card's compact path.
8. `PENDING` state: header ("Kommentar till uppgift") + the resolved task title + an editable comment `Textarea` (debounced 500ms → `updatePendingActionParams`) + "Godkänn" / "Avvisa".
9. `APPROVED` state: read-only comment text + a link to the task. The task board is at `/tasks` and the detail deep-link is `/tasks?task=${taskId}` (confirmed via `searchParams.get('task')` in `components/features/tasks/task-workspace/index.tsx`). Navigate by `taskId`, not the denormalised title.
10. `REJECTED` / `EXPIRED` states: standard muted read-only variants per 14.22.
11. `compact?: boolean` variant: collapsed summary "Kommentar: \"${first ~40 chars}…\"" expandable to the full editable comment.
