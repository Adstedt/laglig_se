# Story 3.10: Chat History Enhancements — Pagination, Search, Export

## Status

Done

## Story

**As a** user,
**I want** enhanced chat history features like search, pagination, date grouping, and export,
**so that** I can efficiently navigate, find, and share past conversations.

## Pre-requisites / Already Delivered

The core chat persistence and conversation management was delivered in Stories 14.x:

- ✅ `ChatMessage` model in DB with workspace scoping, context types, conversation archiving
- ✅ Auto-persist every message to DB via `saveChatMessage` server action
- ✅ Load active history on mount via `getChatHistory` (limit 50, no pagination)
- ✅ Archive/restore conversations via `archiveConversation`, `loadConversation`, `getConversationHistory`
- ✅ Conversation history UI in `HemChat` with `ConversationHistory` component
- ✅ Clear active conversation via `clearChatHistory`
- ✅ Cascade delete when workspace is deleted (`onDelete: Cascade`)

This story adds **polish and power-user features** on top of that foundation.

## Acceptance Criteria

1. **Cursor-based pagination** — `getChatHistory` server action supports cursor-based pagination; the chat message list implements infinite scroll (scroll-to-top triggers loading older messages in batches of 30)
2. **Date grouping** — Messages in the chat list are visually grouped under date headers ("Idag", "Igår", "Förra veckan", date string for older) using Swedish locale
3. **Individual message deletion** — User can delete a single message via a context action (hover menu or long-press); works for both active and archived conversations
4. **Keyword search** — A search input in the conversation history view filters across all conversations (active and archived) by message content; results show matching conversation with a highlighted snippet (120 chars max) of the matching message
5. **Export conversation** — User can export an active or archived conversation as a `.txt` file download via a button in the conversation header

### Existing Schema (no migration needed)

The `ChatMessage` model already has all required fields and indexes:

```prisma
model ChatMessage {
  id              String          @id @default(uuid())
  workspace_id    String
  user_id         String
  role            ChatMessageRole
  content         String          @db.Text
  metadata        Json?
  context_type    ChatContextType @default(GLOBAL)
  context_id      String?
  conversation_id String?
  created_at      DateTime        @default(now())

  workspace Workspace @relation(fields: [workspace_id], references: [id], onDelete: Cascade)
  user      User      @relation(fields: [user_id], references: [id], onDelete: Cascade)

  @@index([workspace_id, context_type, context_id])
  @@index([workspace_id, created_at])
  @@index([user_id])
  @@index([conversation_id])
  @@map("chat_messages")
}
```

The `@@index([workspace_id, created_at])` index supports cursor pagination. The keyword search in AC 4 uses Prisma's `contains` with `mode: 'insensitive'` (maps to `ILIKE` in PostgreSQL, auto-parameterized — no SQL injection risk). At current scale this will use a sequential scan, which is acceptable. If performance degrades, add a GIN trigram index later (`CREATE EXTENSION pg_trgm; CREATE INDEX ... USING gin (content gin_trgm_ops)`).

### Cursor Pagination Pattern

```typescript
// Server action signature
async function getChatHistory(input: {
  contextType: ChatContextType
  contextId?: string
  limit?: number  // default 30
  cursor?: string // message ID to paginate from
}): Promise<{ messages: ChatMessageData[], nextCursor: string | null }>

// Prisma query
const messages = await prisma.chatMessage.findMany({
  where: { workspace_id: wsId, context_type: type, ...(cursor && { created_at: { lt: cursorDate } }) },
  orderBy: { created_at: 'desc' },
  take: limit + 1, // fetch one extra to detect hasMore
})
const hasMore = messages.length > limit
if (hasMore) messages.pop()
return { messages: messages.reverse(), nextCursor: hasMore ? messages[0].id : null }
```

### Scroll Position Preservation

When prepending older messages, capture `scrollHeight` before insert and restore `scrollTop` after:

```typescript
const container = scrollRef.current
const prevHeight = container.scrollHeight
// ...prepend messages...
requestAnimationFrame(() => {
  container.scrollTop = container.scrollHeight - prevHeight
})
```

### Server Action Patterns

All new server actions must follow existing patterns in `app/actions/ai-chat.ts`:
- Use `withWorkspace()` for auth + workspace scoping
- Validate input with Zod schemas
- Return `ActionResult<T>` type
- Scope all queries to `workspace_id`
- Error handling: throw `Error` or let Zod validation errors propagate — the `withWorkspace` wrapper catches and returns `{ success: false, error }`. There is no `AppError` class in the codebase; use standard `Error`.

### AI SDK Message State Reconciliation

`useChatInterface` wraps AI SDK's `useChat`, which maintains its own in-memory message array keyed by `chatId`. When prepending older paginated messages, use `setMessages([...olderMessages, ...currentMessages])` to update the SDK's internal state. Do **not** manage a separate message array outside the SDK — the SDK is the single source of truth for the message list.

### Refactoring Performed

- **File**: `lib/hooks/use-chat-interface.ts`
  - **Change**: `loadMore` callback — changed `setMessages([...olderUIMessages, ...messages])` to `setMessages(prev => [...olderUIMessages, ...prev])`, removed `messages` from dependency array
  - **Why**: `messages` in the dependency array caused: (1) stale closure risk if multiple rapid loadMore calls overlap, and (2) unnecessary callback recreation on every message change during streaming (every 50ms due to `experimental_throttle`), which cascades through IntersectionObserver recreation in `chat-message-list.tsx`
  - **How**: Functional updater form (`prev => ...`) always receives the latest state, eliminating stale closure risk and removing `messages` from deps

- **File**: `components/features/dashboard/conversation-history.tsx`
  - **Change**: `highlightSnippet` — replaced `regex.test(part)` with `i % 2 === 1` (index parity check)
  - **Why**: `RegExp.test()` with the `g` flag is stateful — it advances `lastIndex` after each successful match. When called in a `.map()` loop on different strings, this causes alternating matches to fail (every other highlighted term would be missed)
  - **How**: `String.split()` with a single capture group always places matches at odd indices (1, 3, 5...), making index parity a deterministic and correct way to identify matches

### Performance Considerations

- **Search**: Prisma `contains` with `mode: 'insensitive'` triggers sequential scan (no GIN index). Acceptable at current scale per story dev notes. Add `pg_trgm` GIN index if needed.
- **Search N+1**: `searchConversations` fires a `COUNT(*)` query per unique conversation in results. Bounded to ~50 conversations max (initial findMany limit). Could be optimized to a single query with `groupBy`.
- **Date grouping**: `groupMessagesByDate` called on every render without memoization. Function is O(n) with n ≤ ~100 messages. With `experimental_throttle: 50` batching, this is not a bottleneck.
- **loadMore dependency fix**: Removing `messages` from `loadMore` dependency array also eliminates unnecessary IntersectionObserver recreation during streaming.
