## Story

**As a** user iterating on a styrdokument with the agent,
**I want** repeated approved edits to update the *same open draft* instead of minting a new version each time,
**so that** the version number reflects meaningful published milestones — not 27 draft iterations — while I still keep a full audit trail and one clean draft to finalize.

## Context

Today every approved agent `update_document` edit creates a **new** `WorkspaceDocumentVersion` row, even when the document already has an open draft. A founder watching a doc reach **v27** was seeing 27 draft iterations wearing version numbers — version-number inflation that makes the history noisy and the numbers semantically meaningless.

The dual-version model (17.16) already represents "an approved policy with a revision in progress" via `current_draft_version_id`, and the editor's `autosaveDocument` already **updates the open draft version in place** (no new row). The agent path simply doesn't use that pattern: its dispatch calls `saveDocumentVersion`, which `.create()`s a fresh version every time.

**The fix is narrow and reuses an existing pattern:** when the document already has an open draft (`current_draft_version_id != null`), an approved agent edit should **update that draft version in place** (like autosave) and write a per-edit **ActivityLog** audit row — *not* mint a new version. The **first** edit on an APPROVED-no-draft document still branches one draft (`createDraftFromApprovedWithEdit`), and **finalize** (`promoteDraftToApproved`) still promotes that single draft to an approved version. The public/approved version number then advances only on finalize.

**Decision (founder, 2026-06-13):** Audit trail = ActivityLog (who / when / change-summary) per edit. We accept **losing content-level rollback to intermediate agent edits** (only the live draft + the approved baseline are recoverable) in exchange for a small, reversible, no-migration change. A future story can add an append-only draft-revision log if granular rollback proves necessary.

**Decision (founder, 2026-06-13):** **No migration.** Documents already inflated by the old behaviour are left as historical; this story changes go-forward behaviour only.

## Acceptance Criteria

1. **In-place update when a draft is open.** When approving an agent `UPDATE_DOCUMENT` proposal where the live document has `current_draft_version_id != null` (an approved doc with a revision in progress **OR** a never-approved DRAFT), the edit updates that **existing** draft `WorkspaceDocumentVersion` row in place (`content_json` / `content_html` / `extracted_text` / `change_summary`, plus `title` on rename). It MUST NOT create a new `WorkspaceDocumentVersion` and MUST NOT advance `current_version_number` or `version_number`.
2. **First-edit branch unchanged.** When the document is APPROVED with no open draft (`status === 'APPROVED' && current_draft_version_id == null`, i.e. `autoBranchEligible`) and the proposal carries `creates_draft === true`, the first edit still branches exactly one new draft via `createDraftFromApprovedWithEdit` (one version bump to *open* the draft). Subsequent edits then take the in-place path (AC 1).
3. **Per-edit audit trail.** Every in-place draft edit writes one `ActivityLog` row capturing `user_id`, timestamp, and `change_summary` — the audit record that replaces the previously-minted version rows. Agent authorship + the operation discriminator (17.11c AC 8) are preserved **without** relying on a unique `version_number` as the match key.
4. **Consolidated-batch path matches.** The consolidated approve path (`applyConsolidatedDocumentEdits`, [`pending-agent-actions.ts`](../../app/actions/pending-agent-actions.ts):1092–1310) **already** folds every edit in a batch onto one in-memory tree and performs **one** write per batch (:1214–1229) — it is NOT one write per edit. The fix: when a draft is already open, that single batch write updates the draft **in place** (one ActivityLog audit row carrying the combined batch summary, zero version bump) instead of minting a version; a batch landing on an APPROVED-no-draft doc still branches exactly once via `createDraftFromApprovedWithEdit`. (Inflation here is one-per-batch — the *single* path's one-per-edit is the primary source of the v27 the founder saw.)
5. **Grounding stays current.** An in-place draft edit triggers the draft reindex (so `[Källa:]` grounding reflects the edited draft), matching the current explicit-checkpoint behaviour of `saveDocumentVersion` (`after()` → `indexWorkspaceDocument`).
6. **Finalize unchanged.** `promoteDraftToApproved` still promotes the single open draft to an approved version; the approved/public version number advances **only** on finalize. No change to the finalize or discard flows.
7. **Non-regression — scoped change.** `saveDocumentVersion`'s behaviour for **non-agent** callers (e.g. editor explicit "save version", any version-minting flow) is unchanged — the fix routes only the agent dispatch's draft-exists arm to the in-place save. APPROVED-no-draft defensive guards (17.16 AC 5 Path C) remain. Existing inflated documents are left as historical (no migration).
8. **Activity-feed accuracy.** Because an in-place edit writes the new `document_draft_edited` action instead of the registered `document_version_saved`, the activity feed for agent draft edits changes. The new action MUST render with an accurate Swedish label (e.g. "justerade utkastet") — **not** via the generic fallback arm and **not** as the misleading "sparade version N" (no new version exists). No blank/raw entries.

## Dev Notes

### Root cause (traced live, 2026-06-13)

- [`app/actions/documents.ts`](../../app/actions/documents.ts):759 `saveDocumentVersion` — calls `tx.workspaceDocumentVersion.create({ version_number: nextVersionNumber })` on **every** call (:808–820). `isDraftInProgress` (:805) only decides whether the deprecated `current_version_id` alias advances (:829–834) — it does **not** gate version creation. So each agent edit mints a new row. **This is the inflation.**
- [`app/actions/documents.ts`](../../app/actions/documents.ts):639 `autosaveDocument` — the contrast: `workspaceDocumentVersion.update({ where: { id: targetVersionId } })` (:691) edits the open draft **in place**, no new row, marks dirty for debounced reindex. **This is the pattern to reuse.**
- [`app/actions/pending-agent-actions.ts`](../../app/actions/pending-agent-actions.ts):548 `autoBranchEligible = status === 'APPROVED' && current_draft_version_id == null`; :570 `shouldAutoBranch = creates_draft && autoBranchEligible`; :630–644 the fork (`createDraftFromApprovedWithEdit` vs `saveDocumentVersion`). The **else** arm is the one to change.

### Key constraints

- **Predicate for in-place vs branch.** "Update in place" iff `current_draft_version_id != null` (covers both an approved doc with a revision in progress and a never-approved DRAFT). "Branch" iff APPROVED-no-draft + `creates_draft`. The propose-time/approve-time race (proposal computed `creates_draft=true` but a draft now exists) already falls through to the non-branch arm today — under this story that arm becomes the in-place update, which is correct.
- **Don't globally change `saveDocumentVersion`.** Other callers rely on its mint-a-version semantics (editor explicit save, etc.). Scope the change to the agent dispatch's draft-exists arm only.
- **Audit-stamp match key (AC 3) — BOTH paths.** 17.11c AC 8 stamps agent authorship onto the `document_version_saved` row matched on `(entity_id, action, new_value.version_number)` — single path [`pending-agent-actions.ts`](../../app/actions/pending-agent-actions.ts):659–700, consolidated :1276–1308. In-place edits don't write that row and don't change `version_number`, so this key breaks. Cleanest fix: have `updateDraftVersionInPlace` return the audit row's id and patch by id (no version_number match). Apply to **both** Task 2.2 and Task 3.2 — the consolidated stamp has the identical assumption.
- **Return contract (AC 4, single result_ref).** `updateDraftVersionInPlace` MUST return `{ id, versionNumber }` to match `saveDocumentVersion` / `createDraftFromApprovedWithEdit` (both `Promise<ActionResult<{ id; versionNumber }>>`) so the consolidated `result_ref` (:1240–1244) and the single-path result handling keep working: `id` = the in-place-updated draft version id, `versionNumber` = the **unchanged** `current_version_number`.
- **`change_summary` on the draft row** is overwritten to the latest edit's summary; the per-edit summaries live in the ActivityLog rows (acceptable under the ActivityLog-is-the-audit decision).
- **No schema change, no migration.** Pure dispatch + a new server action reusing existing columns/pointers.
- **Activity-presentation layer (AC 8) — easy to miss.** The new `document_draft_edited` action string touches 4 small maps, none of which are schema: `KNOWN_ACTIONS` ([`action-constants.ts`](../../lib/activity/action-constants.ts)), label ([`activity-labels.ts`](../../lib/constants/activity-labels.ts)), category ([`categories.ts`](../../lib/activity/categories.ts)), and a `case` in [`format-activity.ts`](../../lib/activity/format-activity.ts). The sibling `document_draft_*` actions are audit-only/unregistered, so this is a *deliberate* choice to surface the edit accurately rather than inherit the generic fallback. (If the product instead wants agent draft edits hidden from the feed, that's a one-line decision — but then confirm the fallback arm + any feed filter hide it cleanly. Default for this story: **surface it** with an accurate label.)
