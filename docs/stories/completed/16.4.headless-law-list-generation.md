# Story 16.4: Headless Law List Generation Skill

## Status

Done

## Dependencies

- **Story 16.3** (contextual onboarding questions) — activity flags must be populated on CompanyProfile for the agent to use
- **Epic 14** (agent tools) — `createAgentTools()`, `search_laws`, `get_company_context` must exist (Done)
- **Epic 12** (templates) — curated template data available as agent reference via `get_template_laws` tool (Done)

## Story

**As a** newly onboarded user,
**I want** the system to automatically generate a personalized law list for my company based on my business profile,
**so that** I land on a populated dashboard with relevant laws, monitoring, and compliance tracking from day one.

## Acceptance Criteria

### Headless Agent Skill

1. New server-side skill at `lib/agent/skills/generate-law-list.ts` using `generateText()` from AI SDK (not `streamText()` — no chat UI). This establishes the `lib/agent/skills/` directory as a new architectural pattern for headless agent capabilities.
2. Uses existing `createAgentTools(workspaceId, userId)` tool registry — same `search_laws`, `get_company_context` tools as the chat agent. Skill function signature: `generateLawList(workspaceId: string, userId: string): Promise<GenerateLawListResult>`
3. Two new tools added to the tool set for this skill (both use Prisma directly — NOT server actions which require cookie-based `withWorkspace` context):
   - `get_template_laws` — returns curated law recommendations for a given regulatory area from `LawListTemplate` data. Agent uses these as expert-curated suggestions to evaluate against the company profile, not as a blind copy. Returns `{ area, laws: Array<{ documentId, title, description }>, templateName }`.
   - `add_laws_to_list` — bulk-adds laws to a LawList with grouped sections and `business_context`.
4. System prompt instructs the agent to work like an experienced compliance consultant: (1) understand the company via `get_company_context`, (2) for each relevant regulatory area, check `get_template_laws` for curated recommendations and evaluate each against the company profile (keep or skip), (3) search for additional laws the templates don't cover (industry-specific, activity-flag-specific), (4) add all applicable laws with personalized `business_context` via `add_laws_to_list`
5. Step limit: `stopWhen: stepCountIs(20)` (NOT `maxSteps` which does not exist in the AI SDK) — template lookups + targeted searches + add operations
6. Model: `anthropic('claude-sonnet-4-6')` (balance of quality and cost)
7. Skill returns: `{ listId, itemCount, groups, tokensUsed, durationMs }`

### Trigger & Execution

8. Triggered server-side after `createWorkspace()` completes successfully — call the generation API route from the server action itself (not fire-and-forget from client). If the API call fails to fire, set status to `'pending'` so the dashboard retry logic (AC 22) picks it up.
9. Runs asynchronously (non-blocking) — user is redirected to dashboard immediately
10. New fields on Workspace:
    - `law_list_generation_status`: `'pending' | 'in_progress' | 'completed' | 'failed'`
    - `law_list_generation_progress`: `Json?` — stores live progress steps, e.g. `{ steps: [{ label: "Analyserar ert företag", status: "done" }, { label: "Kontrollerar arbetsmiljöregler", status: "active", detail: "12 lagar" }] }`
11. Status updated at each phase: `pending` (workspace created) → `in_progress` (skill starts, progress steps updated per tool call) → `completed`/`failed`
12. If skill fails (LLM error, timeout), status set to `'failed'` with error logged — user can still use empty dashboard and add laws manually
13. Generation endpoint is idempotent: if `law_list_generation_status` is already `'in_progress'`, return early without starting a second run

### Law List Output

14. Creates a default `LawList` named "Er laglista" with `is_default: true`
15. Items grouped into `LawListGroup` sections: "Arbetsrätt", "Bolagsrätt", "Skatt", "Miljö", "Dataskydd", etc.
16. Each `LawListItem` includes:
    - `document_id` — FK to existing `LegalDocument` (agent must search and find real documents)
    - `business_context` — Personalized "Hur påverkar denna lag oss?" text written to the existing `LawListItem.business_context` field (NOT `commentary`). This field is already displayed in the efterlevnad view tooltip. Content should cover all three aspects: (1) why the law is relevant to this specific company, (2) which processes/departments/products are affected, and (3) context for audits/internal communication. Example: "Ni omfattas av Arbetsmiljölagen som arbetsgivare med 12 anställda inom restaurangbranschen. Lagen berör era köksprocesser, serveringspersonal och arbetsmiljöansvarig chef. Relevant vid Arbetsmiljöverkets inspektioner och vid ert systematiska arbetsmiljöarbete."
    - `source: 'ONBOARDING'` (exists in Prisma `LawListItemSource` enum; note: the Zod `LawListItemSourceEnum` in `lib/validation/document-list.ts` is missing `TEMPLATE` — pre-existing gap, not this story's concern)
    - `compliance_status: 'EJ_PABORJAD'`
    - `position` — ordered within group
17. Target: 40-80 laws depending on company complexity
18. No duplicate documents in the list (unique constraint on `[law_list_id, document_id]`)

### Dashboard Progress Indicator

19. Dashboard shows generation progress when `law_list_generation_status` is `'pending'` or `'in_progress'`
20. Progress component shows live step-by-step progress, not just a spinner. Each agent tool call maps to a user-visible step:
    - `get_company_context` → "Analyserar ert företag" ✓
    - `get_template_laws("arbetsmiljö")` → "Kontrollerar arbetsmiljöregler" ✓ (with count, e.g. "12 lagar")
    - `get_template_laws("miljö")` → "Kontrollerar miljöregler" ✓
    - `search_laws(...)` → "Söker branschspecifika regler..." ●
    - `add_laws_to_list(...)` → "Lägger till lagar i er lista..." ●
    - Final → "Skriver anpassade beskrivningar" ●
    Steps render as: ✓ done (muted), ● active (with subtle pulse animation), ○ pending. Card header: "Vi skapar er personliga laglista..."
21. Polls `GET /api/workspace/generation-status` every 3 seconds (use `useSWR` with `refreshInterval: 3000`). Response includes `{ status, progress?, itemCount?, error? }` where `progress` is the steps array from `law_list_generation_progress`.
22. On load, if status is `'pending'` (trigger never fired or failed), re-fire the generation POST as a retry
23. On completion: progress card transitions smoothly to law list summary card showing total law count + group breakdown + link to law list
24. On failure: show "Laglistan kunde inte skapas automatiskt. Du kan skapa din lista manuellt." with link to template catalog

### Cost & Performance

25. Target cost: <$0.25 per generation (Sonnet, ~10-15 tool calls). Raw estimate ~$0.24; prompt caching and reduced tool calls may bring this lower.
26. Target duration: <60 seconds
27. Log token usage and duration to Sentry as custom breadcrumbs/transactions for monitoring

### Manual Re-generation

28. Button in Settings or Law List page: "Generera om laglista" — re-runs the skill with current company profile
29. Shows confirmation dialog: "Detta ersätter din nuvarande laglista. Ändringar du gjort behålls inte."
30. Re-generation uses the same skill, creates a new list (old one archived as "Er laglista (arkiverad {date})", not deleted)

## Dev Notes

### Headless Agent Pattern

This is the first production use of `generateText()` in the app. The chat route uses `streamText()`. Both share the same tool registry. The simulation scripts (`scripts/tmp-simulate-full-agent.ts`) demonstrate the exact pattern:

```typescript
import { generateText, stepCountIs } from 'ai'
import { anthropic } from '@ai-sdk/anthropic'
import { createAgentTools } from '@/lib/agent/tools'

const result = await generateText({
  model: anthropic('claude-sonnet-4-6'),
  system: skillPrompt,
  messages: [{ role: 'user', content: 'Build a comprehensive law list.' }],
  tools: { ...createAgentTools(workspaceId, userId), add_laws_to_list: addLawsToListTool },
  stopWhen: stepCountIs(20),
})

// Access results:
// result.text — final response
// result.steps — array of tool invocations
// result.usage — { inputTokens, outputTokens }
```

### Template-as-Tool Pattern (Consultant Model)

The agent uses curated templates as **reference material**, not as a blind copy source. This mirrors how an experienced compliance consultant works: they reference their firm's internal checklists while making decisions specific to each client.

**Why template-as-tool, not template-copy-then-fill-gaps:**
- Agent evaluates each template suggestion against the company profile — skips irrelevant items (e.g., heavy machinery laws for a software company)
- Agent can reason about why something applies or doesn't (captured in `business_context`)
- For areas without templates, agent falls back to `search_laws` — same capability, just less guidance
- One system to invest in (the agent) rather than two (templates + agent)
- As templates grow, agent gets better input — less searching, more personalizing

**Available templates (seeded via Epic 12):**
- `scripts/seed-arbetsmiljo-template.ts` — arbetsmiljö laws (applies to all employers)
- `scripts/seed-miljo-template.ts` — miljö laws (applies when environmental flags set)
- Future: add templates for skatterätt, bolagsrätt, dataskydd as coverage expands

**When no template exists for an area:** Agent uses `search_laws` to find applicable laws directly. The system degrades gracefully — templates improve quality where they exist but aren't required.

### Skills Directory Pattern

`lib/agent/skills/` is a new directory establishing the pattern for headless agent capabilities. A skill is:
- **Input:** workspace context (workspaceId, userId)
- **Output:** typed result object
- **Internals:** system prompt + tools + `generateText()` + `stopWhen`
- **Reusable:** same brain, same tools, different prompt and trigger

Future skills follow the same contract: `lib/agent/skills/assess-law-change.ts`, `lib/agent/skills/generate-compliance-report.ts`, etc.

### Why Prisma Directly (Not Server Action)

The existing `addDocumentToList()` at `app/actions/document-list.ts` is a `'use server'` action that calls `withWorkspace()`. This reads cookies to resolve the workspace context, which is NOT available in a headless API route execution. The `add_laws_to_list` tool must use Prisma directly with the `workspaceId` passed explicitly.

### Tool Response Pattern

Read tools in the codebase use `wrapToolResponse()` from `lib/agent/tools/utils.ts`. Write tools have a two-step confirm pattern (`execute: false` → preview → `execute: true`). For the headless skill, `add_laws_to_list` should execute directly (no confirmation needed — it's a background job, not interactive).

### Zod/Prisma Enum Note

The Prisma `LawListItemSource` enum includes `ONBOARDING`, `MANUAL`, `IMPORT`, `TEMPLATE`. The Zod `LawListItemSourceEnum` in `lib/validation/document-list.ts` only has `['ONBOARDING', 'MANUAL', 'IMPORT']` — `TEMPLATE` is missing. This is a pre-existing gap. The `add_laws_to_list` tool uses Prisma directly (not the Zod schema), so it can use `ONBOARDING` without issue.

### Cost Estimation

Based on simulation scripts and Sonnet pricing ($3/1M input, $15/1M output):
- `get_company_context`: ~500 tokens
- `get_template_laws` (per call, ~2-3 calls): ~500 input + ~2K output each (curated data, cheaper than search)
- `search_laws` (per call, ~3-5 calls for gap-filling): ~2K input + ~1K output each
- `add_laws_to_list` (per call, ~3 calls): ~1K tokens each
- System prompt + orchestration: ~5K tokens
- **Total estimate: ~20K input ($0.06) + ~12K output ($0.18) = ~$0.24**
- With prompt caching (repeated system prompt): ~$0.15-0.20
- Template-first approach reduces search calls from ~10 to ~3-5, saving ~$0.05-0.10 per generation
- AC target: <$0.25 (comfortably within budget)

### Future: Agent Learning Path

The architecture naturally supports progressive improvement without model retraining. User behavior data is already captured via `LawListItem.source`, `compliance_status`, and edit history. Future stories can leverage this:

1. **Prompt enrichment** (near-term) — query aggregate patterns from existing workspaces ("restaurants typically add Alkohollagen, remove Maskindirektivet") and inject as guidance into the system prompt
2. **Template evolution** (mid-term) — automated monthly reports flag template items frequently removed (>50%) or laws frequently added manually (>30%), PO reviews and updates templates
3. **Company similarity matching** (long-term) — embed company profiles, find similar companies, use their validated law lists as additional reference for the agent

Each level is a separate story, not a prerequisite for 16.4.

## QA Results

### Review Date: 2026-03-30

### Reviewed By: Quinn (Test Architect)

### Code Quality Assessment

Solid implementation. Architecture follows established patterns (tool factory, `wrapToolResponse`, Prisma-direct for headless context). System prompt is well-crafted with clear Swedish instructions. Two bugs fixed during review — both would have caused visible UX issues in production.

### Refactoring Performed

- **File**: `components/features/dashboard/law-list-generation-progress.tsx`
  - **Change**: Fixed SWR `refreshInterval` to use callback form checking `latestData?.status` instead of static `initialStatus` prop
  - **Why**: Polling never stopped after generation completed — `isActive` was a const derived from the server-rendered prop, not the latest SWR data
  - **How**: SWR's `refreshInterval` callback receives latest data, enabling dynamic stop when status reaches `completed`/`failed`

- **File**: `lib/agent/skills/generate-law-list.ts`
  - **Change**: Moved progress updates from post-hoc `result.steps` iteration to `onStepFinish` callback
