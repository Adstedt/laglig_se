## Story

**As a** user reviewing a proposed section edit,
**I want** to tweak the proposed text directly in the approval card,
**so that** small corrections ("1 → 2 gånger per år", a typo, a date) don't require a chat round-trip that re-reads the whole document and risks a duplicate proposal.

## Context

When the user wants a minor change to a *pending* `update_document` proposal, the only path today is to ask the agent in chat. The agent re-runs `list_styrdokument` + `get_workspace_document` and proposes again — slow, and it produces a second card for the same section (confusing; the consolidation fold then applies both). For trivial edits this is overkill: the proposed content already lives in `PendingAgentAction.params.newSectionContentJson`, and editing it in place needs no model call.

The CREATE_TASK / obligation renderers already prove the pattern (editable fields → debounced `updatePendingActionParams`). This story brings the same affordance to the proposed **section body** (and the optional `newTitle`).

**Scope decision — plaintext, not rich.** The proposed body is Tiptap JSON (`TiptapNode[]` — an array of block nodes, not a doc wrapper). Full rich inline editing is out of scope. v1 offers a **plaintext** textarea seeded from the proposal's plain text; on save it re-wraps into paragraph nodes (one node per blank-line-separated block). This is **lossy for lists / tables / nested structure** — so the edit affordance is gated to "simple" proposals (`newSectionContentJson` is paragraphs-only) and otherwise falls back to "ask the agent / open in editor". `newTitle` is a plain string and always editable.

## Acceptance Criteria

1. In the `update_document` proposal card (PENDING only), the "Justera" disclosure exposes an **editable plaintext textarea** seeded from the proposal's proposed section body (`newSectionContentJson` → plain text, **paragraph-preserving**), plus an editable `new_title` input when the proposal carries a rename. The affordance appears in BOTH the single-action card and the compact consolidated-batch row (both already pass `onParamsChange` — see Dev Notes).
2. Editing debounces 500ms → `updatePendingActionParams(id, params)` (reuse `useDebouncedParamsChange`). Because `updatePendingActionParams` **replaces `params` wholesale** (no server-side merge), the debounced write MUST send the **complete** params object (all existing snapshot fields — `documentId` / `documentTitle` / `sectionHeading` / `oldSectionContentJson` / `entity_version` / `creates_draft` / `newVersionNumber` / `changeSummary` — spread through unchanged) with only `newSectionContentJson` (re-wrapped to paragraph `TiptapNode[]`) and/or `newTitle` overwritten. The colored diff re-renders against the edited proposed text.
3. **Lossy-edit guard:** the textarea is only offered when `newSectionContentJson` is **paragraphs-only** (`nodes.every(n => n.type === 'paragraph')`). When the body contains lists/tables/headings/other block types, show the read-only diff (current behaviour) with a hint ("Öppna i editor för att finjustera" / be agenten finjustera) instead of a textarea — never silently flatten rich content.
4. The edited proposal flows through the existing approve path unchanged: dispatch reads the live document and applies `updateSection` with the edited `newSectionContentJson` (single-approve [`app/actions/pending-agent-actions.ts`](../../app/actions/pending-agent-actions.ts):594–605 AND consolidated batch :1170–1175). No dispatch changes needed.
5. **No-op / empty guard (two parts):** when the textarea body is empty, (a) approve is disabled for that row with a clear message (`canApprove={false}`), AND (b) the debounced writer does **not** persist the empty body. Part (b) is non-negotiable: the approve dispatch guards only on `newSectionContentJson !== undefined` (:597 / :1171), so a persisted empty `[]` would otherwise survive into a later single-approve OR consolidated-batch approve and write an empty section.
6. **Non-regression:** rich proposals (read-only), rename-only proposals, and the consolidated batch approval all behave exactly as before for users who don't edit inline. `ADD_DOCUMENT_SECTION` is **out of scope by construction** — it routes through a separate `AddDocumentSectionRenderer` ([`agent-action-card.tsx`](../../components/features/ai-chat/agent-action-card.tsx):77), not `UpdateDocumentRenderer`, so this story cannot affect it.

### Key Constraints

- **Wholesale param replace (AC 2).** `updatePendingActionParams` overwrites the whole `params` JSON. Sending a partial object (e.g. just `{newSectionContentJson}`) would silently drop `documentId` / `entity_version` / `creates_draft` / `newVersionNumber` / `oldSectionContentJson`, breaking the approve dispatch. Always spread the existing `params` and overwrite only the edited fields — mirror `TaskApprovalRenderer`'s full-object rebuild ([`draft-document-renderer.tsx`](../../components/features/ai-chat/agent-action-renderers/draft-document-renderer.tsx):71 documents the same "replaces params wholesale" caveat).
- **Paragraph-preserving round-trip (AC 1, Task 1).** The existing flatteners space-join nodes and are unsuitable for the seed. The new `tiptapParagraphsToText` / `textToTiptapParagraphs` pair must round-trip paragraphs-only content without losing block boundaries.
- **Empty must not slip through batch approve (AC 5).** Because the dispatch guard is `!== undefined`, the empty guard has to prevent *persisting* an empty body, not merely disable the local approve button — otherwise the consolidated-batch path applies it.
- **Live diff off local state (AC 2, Task 2.2).** The persisted params lag the keystrokes by the debounce + round-trip; recompute the preview from local textarea state.
- **Scope is `update-document-renderer.tsx` + one util module.** No dispatch, no server-action, no schema changes. `ADD_DOCUMENT_SECTION` is a different renderer and is untouched.
- **No DB migration.** Inline edit writes to the existing `PendingAgentAction.params` JSON via an existing server action.
