# Story 21.9: Cycle Seal ‚Äî SHA-256 hash + evidence snapshot + immutability

## Status

Done

## Story

**As a** lead auditor finalising an audit-grade compliance cycle,
**I want** to fastst√§lla (seal) the cycle with a tamper-evident SHA-256 hash over a canonical manifest and frozen SHA-256 snapshots of every linked evidence artifact,
**so that** an external certifieringsorgan can verify months later that the cycle state and evidence presented at seal time have not been altered, and so that the revisionsrapport carries a reproducible integrity marker that survives any future edits, re-runs, or refactors.

     1. **Gather evidence per item.** For each `ComplianceAuditItem` in the cycle, resolve the item's linked artifacts via a new `lib/compliance-audit/gather-seal-evidence.ts` helper that replicates the JOIN logic of `getLinkedArtifactsForListItem` (`app/actions/linked-artifacts.ts:71-293`) but (a) accepts a `Prisma.TransactionClient`, (b) returns `{ lawListItemId, kind, fileId?, documentId?, requirementId? }[]` ‚Äî no filename/mime/type metadata needed, since the seal only cares about identity + bytes. The transaction accepts the pre-computed hashes as input. (Technically the evidence rows must exist at seal time; if they're deleted between gather and hash, the hash call fails ‚Äî treat as structured error `"Bevis har tagits bort under fastst√§llandet.         ```ts

          where: { id: cycleId, status: 'AVSLUTAD' },
          data: { status: 'SEALED', sealed_at: now, sealed_by_user_id: ctx.userId, seal_hash: hash },

        Caller catches the string-typed sentinel (or uses a custom Error class ‚Äî either works) and maps to structured result `"Kontrollens status √§ndrades under fastst√§llandet. F√∂rs√∂k igen."`.         - `evidence_file_id` XOR `evidence_document_id` (enforced by the DB CHECK constraint from Story 21.1 ‚Äî violating it throws, which aborts the transaction, which is correct)

        - `pdf_storage_path: null` (Story 21.12 will populate on regen; deferred)

        - `manifest: canonicalJsonString` (the exact JSON string hashed ‚Äî stored so the cycle's integrity can be re-verified offline, per architecture ¬ß4.1 ComplianceAuditReport manifest field)
        - **Safety rationale:** there is no DB-level `@@unique([cycle_id, report_kind])` constraint today. 4. **Evidence hashing ‚Äî `lib/compliance-audit/evidence-hash.ts` (NEW).**
   - **`hashFileEvidence(fileId: string, prismaClient?: PrismaLike): Promise<string>`.**
     - Loads `WorkspaceFile` by id; fails with `"Bevisfil ${fileId} hittades inte"` if missing.
     - If `file.storage_path === null` (folder, not a file): throws `"Bevisfil ${fileId} har ingen lagringsv√§g"` ‚Äî defensive; evidence snapshots should never reference folder rows.
     - Downloads bytes from Supabase Storage via `getStorageClient().storage.from('workspace-files').download(storage_path)` (matches the pattern at `app/actions/files.ts:834-842`). On 404 or error from Storage: throws a structured message `"Bevisfil ${fileId} kunde inte h√§mtas fr√•n lagring"`.
     - Computes SHA-256 via Node built-in `crypto.createHash('sha256').update(buffer).digest('hex')` ‚Üí returns 64-char hex.
     - **Streaming consideration:** for files <50MB (the vast majority ‚Äî workspace files average ~2MB per the existing upload ceiling), loading into memory is fine. A 200-item cycle with 10 files per item √ó 2MB = 4GB which would blow Node memory. **Pragmatic mitigation:** hash files serially (one at a time in memory, release before next). Document this as an NFR follow-up; the expected MVP workload is well below the problem threshold. See "Technical constraints" in Dev Notes.
   - **`hashDocumentEvidence(documentId: string, prismaClient?: PrismaLike): Promise<string>`.**
     - Loads `WorkspaceDocument` by id; fails with `"Styrdokument ${documentId} hittades inte"` if missing.
     - Builds a canonical-content object: `{ id, title, document_type, status, version_number: current_version_number, html_content, markdown_content, json_content }`. Null fields stay as `null`, NOT omitted ‚Äî stable schema. Use `JSON.stringify` with stable key order (deterministic via the `canonicalize` wrapper from AC 7).
     - Computes SHA-256 over the canonical string bytes. Returns 64-char hex.
     - **Why hash content not file:** documents are Tiptap-editable styrdokument stored in Postgres, not as Storage files. Hashing the rendered content (html + markdown + json) captures what a reviewer would actually read.
   - Both functions accept an optional `prismaClient: PrismaLike = prisma` ‚Äî typed `PrismaClient | Prisma.TransactionClient` per the established pattern in `lib/compliance-audit/authorization.ts:19`. Enables future transaction-participating reads if needed; for 21.9 both calls happen outside the seal transaction (per AC 3 step 2 rationale).

     ```ts

       cycleId: string
       workspaceId: string
       lawListId: string
       name: string
       auditType: 'INTERN' | 'EXTERN'
       scheduledStart: string   // ISO 8601 UTC
       scheduledEnd: string
       lawChangeCutoffDate: string
       leadAuditorUserId: string

       createdAt: string
       sealedAt: string         // ISO 8601 UTC (just-computed)
       sealedByUserId: string
       scopeDefinition: ScopeDefinition   // as stored
       overrideReason: string | null       // null when no override needed
       items: Array<{
         id: string
         lawListItemId: string
         efterlevnadsbedomning: EfterlevnadsBedomning | null
         motiveringSha256: string | null  // sha256(motivering) ‚Äî NOT the raw text, so the manifest is small + privacy-safe
         reviewedAt: string | null
         reviewedByUserId: string | null
         signedOffAt: string | null
         signedOffByUserId: string | null
       }>
       findings: Array<{
         id: string
         type: 'AVVIKELSE' | 'OBSERVATION' | 'FORBATTRING'
         severity: 'MAJOR' | 'MINOR' | null
         title: string
         descriptionSha256: string
         rootCauseSha256: string | null
         lawListItemId: string | null
         requirementId: string | null
         correctiveActionTaskId: string | null
         dueDate: string | null
         closedAt: string | null
         closedByUserId: string | null
       }>
       evidence: Array<{
         lawListItemId: string | null
         requirementId: string | null
         kind: 'FILE' | 'DOCUMENT'
         evidenceId: string
         sha256: string
       }>
     }
     ```
   - **Function:** `buildSealManifest(input: SealManifestInput): SealManifestInput` ‚Äî pass-through after deterministic sorting:
     - `items` sorted by `id` (UUID string sort, NOT `lawListItemId` because a cycle could theoretically have multiple snapshots of the same law ‚Äî it can't today, but the unique constraint is `(cycle_id, law_list_item_id)`; sorting by primary key is the stable choice).
     - `findings` sorted by `id`.
     - `evidence` sorted by `(kind, evidenceId)` lexicographic composite ‚Äî matches the uniqueness key for snapshot rows.
   - **Motivering + description hashing ‚Äî privacy + size rationale:** free-text fields (`motivering`, `description`, `root_cause`) are hashed (SHA-256 hex) rather than embedded. Two wins: (a) the sealed manifest never leaks sensitive text via storage-path or the activity-log `new_value` payload; (b) the manifest stays small (<50KB for a 200-item cycle) ‚Äî critical because it's stored as JSON on `ComplianceAuditReport.manifest` and every canonicalization round is an in-memory serialize. Reviewer reading a sealed PDF sees the raw motivering via the renderer (which reads from the LIVE cycle rows, not the manifest); the hash on the manifest verifies the LIVE row has not been edited post-seal.
   - **Empty-field contract:** null values stay null (not omitted). `JSON.stringify` with `canonicalize` wrapper (AC 7) normalises key ordering; omitted vs null-valued fields produce different hashes, so the contract is "always include every field, use null for absent." Document this in the file's JSDoc so a future field addition doesn't accidentally drop `undefined` into the hash input.

6. **Open-avvikelse gate ‚Äî policy decision frozen (Gate-with-manual-override).** Story 21.6's carry-forward ¬ß"Unresolved design question for 21.9" proposed three policies. PO has confirmed **Option 2 (Gate with manual override)** in this story's v0.1 draft (see Change Log below). Specifics:
   - **Blocking conditions:**
     - Any `ComplianceFinding` where `type === 'AVVIKELSE' && closed_at === null`.
   - **Not blocking:**
     - Open OBSERVATION / FORBATTRING ‚Äî informational, does not invalidate integrity claim.
     - Pending corrective-action tasks on CLOSED findings ‚Äî edge case (task got reopened post-closure); cycle's state is still "finding-closed" from the audit's perspective.

   - **Override mechanics:**
     - Server-side: `overrideReason` is required if `openAvvikelsesCount > 0`; validated as min 20 chars; persisted in ActivityLog `new_value.overrideReason` and the seal manifest's `overrideReason` field.
     - The `overrideReason` is part of the canonical manifest ‚Äî it's hashed into `seal_hash`. If a reviewer wonders what the seal claims to cover, the manifest answers including the override rationale.
     - UI-side: the dialog exposes the `<Textarea>` only when blocking conditions present.
   - **Rationale documented inline** in the dialog's JSDoc + the `sealCycle` server action's JSDoc: why only AVVIKELSE (severity-bearing structured nonconformity) gates; why OBSERVATION / FORBATTRING do not. Future story (post-MVP) could add a per-finding-type setting; not worth the knob in MVP.

### Seal design ‚Äî hash what, when, how

**Three core design decisions frozen in this story:**

1. **Hash covers: cycle metadata + items + findings + evidence manifest.** Per epic AC 2. Override reason when present is in the manifest + in the ActivityLog `new_value`.
2. **Motivering + description + root_cause are SHA-hashed in the manifest, NOT embedded.** Privacy + size. The manifest never contains raw free text ‚Äî only the cycle-row text contains it. The manifest's hash verifies those rows haven't been edited post-seal.
3. **Evidence content hashes are captured at seal time.** File bytes hashed via SHA-256 over the Supabase Storage bytes. Document content hashed via SHA-256 over a canonical JSON of `{ html, markdown, json, version, title, type, status }`. Captured into `ComplianceEvidenceSnapshot.evidence_sha256` ‚Äî per-snapshot, so a future re-verification can compare the stored hash against the current bytes.

**Three defensive choices:**

1. **`updateMany` with status-scoped WHERE (SF-2).** Makes the seal transaction race-safe without explicit `SELECT FOR UPDATE` locking. If two users click Seal concurrently, only one `updateMany` returns `{ count: 1 }` ‚Äî the other gets `{ count: 0 }` and fails clean via the thrown `SealRaceError`. Status guard is enforced at the DB, not just the application layer. Prisma's typed `update` would fail type-check on a composite `where: { id, status }` (non-unique); `updateMany` is the idiomatic escape hatch.
2. **Hash calls outside the transaction.** Storage downloads + hashing take seconds for a large cycle; holding a Postgres connection open for that long is a connection-pool risk. Read the evidence refs inside the tx (Task 6), release the tx, hash outside, then open a fresh tx for the INSERT + UPDATE + logActivity. See AC 3 step 2 note.
3. **Empty-evidence cycles are valid.** A 200-item cycle with zero linked artifacts can be sealed ‚Äî the `evidence` array in the manifest is `[]`. Hash is deterministic over the empty array. `complianceEvidenceSnapshot.createMany({ data: [] })` is a Prisma no-op.

### PO design question resolution ‚Äî Gate-with-manual-override (Option 2)

Story 21.6's carry-forward notes flagged an unresolved design question about whether Seal should be blocked on open AVVIKELSE / pending tasks. Three candidate policies were documented:

1. Permissive (symmetric with Complete).
2. Gate with manual override ‚Äî PO recommendation in 21.6's v0.3 revision.
3. Hard gate.

**This story implements Option 2.** Rationale:

- Option 1 undermines the audit-grade positioning ‚Äî an external certifieringsorgan could read a sealed PDF with an open AVVIKELSE in the findings section and reasonably wonder why it's called "fastst√§lld".
- Option 3 is too restrictive ‚Äî real-world √•tg√§rdsuppf√∂ljning is multi-week and the cycle's primary point of value (seal as a point-in-time snapshot) must be achievable without waiting for every remediation to close. Otherwise the sealed cycle date drifts months from the actual audit period.
- Option 2 balances both: default behaviour blocks, escape hatch allows override with a required text rationale. The rationale goes into the ActivityLog + manifest + eventually into the sealed PDF cover page (renderer extension deferred to 21.12 since the renderer is already shipped and not in scope for 21.9). Reviewer sees "why sealed with open avvikelser" in the same place they see the hash.

**Narrowing the gate to AVVIKELSE only** is the key refinement: OBSERVATION + FORBATTRING are informational + suggestions respectively; their openness doesn't compromise the integrity claim of a sealed cycle. ### Technical constraints

- **TypeScript strict** with `exactOptionalPropertyTypes`; use conditional spreads for `overrideReason` passthrough (`...(overrideReason ? { overrideReason } : {})`). Pattern established at `compliance-audit-cycle.ts:497-509`. [Source: architecture/17-coding-standards.md ¬ß17.2‚Äì17.3]
- **Next.js 16 App Router** ‚Äî server actions via 'use server' + RSC routes; client components with `'use client'` only where interaction is required (dialog, dropdown). [Source: architecture/3-tech-stack.md ‚Äî Frontend framework row]
- **Prisma 5.12+ (SF-2).** `$transaction` for the seal transition; `createMany({ data: [] })` is a no-op; **`updateMany({ where: { id, status: 'AVSLUTAD' }, data: {...} })` + `result.count === 1` check** is the race-safe idiom ‚Äî a plain `update` would fail type-check on the composite where (status is not a unique key). The `count === 0` branch throws a sentinel `SealRaceError` which maps to `"Kontrollens status √§ndrades under fastst√§llandet. F√∂rs√∂k igen."`. [Source: architecture/3-tech-stack.md ‚Äî ORM row; Prisma docs]
- **Zod 3.22+ (SF-3).** `SealCycleSchema` is `{ cycleId: uuid, overrideReason?: string().max(1000).refine(s => s === undefined || s.trim().length >= 20, ...) }`. The `.trim()` inside the refine is the load-bearing piece ‚Äî without it, whitespace-only strings (`" ".repeat(20)`) pass naive `.min(20)` but fail the UI's trimmed gate, creating a client/server contract desync. Server-side open-avvikelse-requires-override check is conditional on the DB-level count (cannot express as pure Zod `refine` ‚Äî done at the server-action layer after schema parse).
- **Canonicalization ‚Äî RFC 8785 JCS** via the pinned `canonicalize` 3.0.0 package. MIT licensed per license policy. Do NOT replace without a migration plan for existing sealed hashes. [Source: architecture/3-tech-stack.md ¬ß3.2; architecture ¬ß6.2]
- **SHA-256 via Node built-in `crypto`** ‚Äî no external dependency. Returns 64-char hex.
- **Supabase Storage** ‚Äî existing `workspace-files` bucket. `getStorageClient().storage.from(BUCKET_NAME).download(path)` returns a `Blob` (web) or `Buffer` (server context) depending on runtime; stream to a `Buffer` via `await blob.arrayBuffer()` ‚Üí `Buffer.from(...)` for the crypto `update()` call.
- **Workspace scoping** ‚Äî every mutation Prisma query filters through `workspace_id`. Tenant isolation maintained across all new paths. [Source: `compliance-audit-cycle.ts:188-195` precedent]
- **Activity logger signature** ‚Äî `logActivity(workspaceId, userId, 'compliance_audit_cycle', cycleId, 'cycle_sealed', oldValue, newValue)`. [Source: `lib/services/activity-logger.ts:8-28`]
- **Revalidation scope** ‚Äî `/laglistor/kontroller` (list) + `/laglistor/kontroller/${cycleId}` (detail). Matches 21.6 convention. [Source: architecture ¬ß2.3]
- **Error shape** ‚Äî `ActionResult<T>` with Swedish error strings. Pattern per `compliance-audit-cycle.ts:265-268`. [Source: architecture/18-error-handling-strategy.md ¬ß18.2]
- **Pre-push hook** ‚Äî `pnpm build` (see memory `## Build`). Seal transaction code is additive ‚Äî no build slowdown expected.
- **Memory ceiling for file hashing** ‚Äî sequential (not parallel) file hashing for >10-file cycles. Serial worst case for a 200-file √ó 10MB cycle: ~100MB peak, well under Vercel's 1GB Node heap limit. Parallel would blow up. Document this inline in `evidence-hash.ts`.
- **No new dependencies** ‚Äî `canonicalize` already installed. Everything else is Prisma + Node built-ins + Supabase client + existing shadcn primitives. [Source: architecture ¬ß3.2]

#### Why the policy changed

v0.4's INTEGRITY-001 hard gate ‚Äî rejecting seal whenever any linked styrdokument evidence was in `DRAFT` status ‚Äî was the correct integrity call at the time: `WorkspaceDocument.current_version` is mutated in-place by `autosaveDocument`, so a seal including a DRAFT doc silently invalidates re-verification after the user's next keystroke in that draft. In production at Alm√•sa Havshotell, however, it turned out to be operationally prohibitive. Linking DRAFT styrdokument to laglista items as *work-in-progress references* is routine ‚Äî not "evidence," per se, but ambient collateral. PO requested a softening mirroring Option 2 from the original Story 21.9 design discussion: **snapshot-and-accept-with-override**.
