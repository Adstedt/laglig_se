## Story

**As a** compliance-platform admin or AUDITOR-role user,
**I want** the agent to offer only tools my role is allowed to use, and every tool call durably logged with its inputs/outputs/outcome,
**so that** AUDITORs never trigger 403s from write-gated actions, and every agent action — reads, write proposals, and failures — is traceable for compliance audits.

## Context

`createAgentTools(workspaceId, userId, context?)` (`lib/agent/tools/index.ts:62`) always returns the full ~15-tool set regardless of role. An **AUDITOR** (read-only role) sees write tools (`create_task`, `draft_styrdokument`, …) the model will invoke and get 403-rejected by the underlying `withWorkspace(..., 'tasks:edit')` server actions → fruitless retry loops + degraded UX. And `PendingAgentAction` only tracks the *proposal lifecycle* of Tier-2 writes — it does not record pure reads, pre-proposal failures, or the full I/O envelope.

This is the **"do early" foundation** for Phase 3: the `role` param + the `wrapWithDecisionLog` seam are cheap to add now across ~15 tools and a refactor across 30+ later. Stories **19.2/19.3/19.4** (new tools) plug into the seam this story defines.

## Acceptance Criteria

### Role-based registry filter

1. `createAgentTools` gains an optional **4th positional param** `role?: WorkspaceRole` (`lib/agent/tools/index.ts:62`). When `role` is absent (legacy callers), the full tool set is returned unchanged — **no breaking change**.
2. A single exported `TOOL_REGISTRY_POLICY` maps every tool name → `'read' | 'write'`, defined so that **adding a tool without a policy entry is a TypeScript error** (exhaustive `Record` / `satisfies` check). **(NH-1 typing wrinkle:** `web_search` is injected at the route, **not** returned by `createAgentTools`, so `keyof ReturnType<typeof createAgentTools>` won't include it — and deriving from `ReturnType` is circular within the module. Define an explicit `ToolName` string-literal union (the 15 tool keys **+ `'web_search'`**) and use `TOOL_REGISTRY_POLICY = {…} satisfies Record<ToolName, 'read'|'write'>`.) Current tools:

   | read | write |
   |---|---|
   | `search_laws`, `search_workspace_files`, `get_document_details`, `get_change_details`, `get_company_context`, `suggest_followups`, `web_search` | `create_task`, `update_compliance_status`, `save_assessment`, `add_context_note`, `link_task_to_document`, `link_document_to_task`, `add_obligation`, `assign_task`, `draft_styrdokument` |

3. The role→tier rule derives from the existing **`hasPermission(role, 'tasks:edit')`** (`lib/auth/permissions.ts:114`) — single source of truth, no parallel matrix:
   - role has `tasks:edit` (OWNER/ADMIN/HR_MANAGER/MEMBER) → `read` + `write`
   - role lacks `tasks:edit` (**AUDITOR**, `permissions.ts:108`) → `read` only
   - `role === undefined` → full set (back-compat)
4. `app/api/chat/route.ts` resolves `role` and threads it. **Note:** the route already calls `getWorkspaceContext()` (which returns `role: WorkspaceRole`, `workspace-context.ts:29`) but currently keeps only `workspaceId` and discards `role` (the `try/catch` around `~:84`); capture `role` and pass it as the 4th arg to `createAgentTools` (`route.ts:261`). For AUDITOR, also exclude `web_search` at its injection point (`route.ts:259-263`). **Edge:** the `catch` fallback sets `workspaceId='default'` with no role — decide that path returns read-only (it's an unauthenticated fallback) rather than the full set.
5. AUDITOR read-only is test-verifiable: `createAgentTools(ws, user, ctx, 'AUDITOR')` returns keys that are a subset of the `read` tier and contain **none** of the `write` tier.

### `AgentDecisionLog` model

6. New Prisma model `AgentDecisionLog` (place near `ChatUsageEvent`, `schema.prisma:1518` / `PendingAgentAction`, `:1560` — same conventions: snake_case, `@@map`, `(workspace_id, created_at)` index):

   | field | type | notes |
   |---|---|---|
   | `id` | `String @id @default(uuid())` | |
   | `workspace_id` | `String` | FK → Workspace (`:101`), `onDelete: Cascade` |
   | `user_id` | `String` | FK → User (`:20`), `onDelete: Cascade` |
   | `chat_message_id` | `String?` | FK → ChatMessage (`:1469`), `onDelete: SetNull` — null outside a chat turn (future cron/CLI) |
   | `pending_action_id` | `String?` | FK → PendingAgentAction (`:1560`), `onDelete: SetNull` — set **only** for write proposals; the **exact** key approve/reject matches on (SF-1, see AC 13/14) |
   | `tool_name` | `String` | free-form (String, not enum, so new tools need no migration) |
   | `input_json` | `Json` | tool input at call time |
   | `output_json` | `Json?` | result (null if it threw before returning) |
   | `proposed_at` | `DateTime @default(now())` | wrapper-entry time |
   | `accepted_at` | `DateTime?` | set when the linked proposal is approved |
   | `accepted_by` | `String?` | approver user id (FK → User, named relation, `onDelete: SetNull`) |
   | `outcome` | `AgentDecisionOutcome` | enum below |
   | `model_version` | `String` | resolved model name (see AC 11) |
   | `created_at` | `DateTime @default(now())` | |

   Indexes: `(workspace_id, created_at)`, `(workspace_id, user_id, created_at)`, `(tool_name, created_at)`, `(chat_message_id)`, `(pending_action_id)`. `@@map("agent_decision_logs")`. The `accepted_by` FK needs a **named relation** on `User` (to disambiguate from `user_id`).
7. New enum `AgentDecisionOutcome { SUCCESS, WRITE_PROPOSED, WRITE_ACCEPTED, WRITE_REJECTED, ERROR }`.
8. **Manual migration** (project rule): generate with `npx prisma migrate dev --create-only --name add_agent_decision_log`, hand off the SQL, `prisma generate`. Never auto-apply/`db push`/reset. [Source: MEMORY — manual migrations]

### Log wrapper

9. New `wrapWithDecisionLog(toolName, execute, ctx: DecisionLogContext)` in `lib/agent/tools/decision-log.ts` wraps a tool's `execute` in `try/finally`, writing one `AgentDecisionLog` row after the inner call. Outcome mapping: result with `confirmation_required: true` → `WRITE_PROPOSED`; `ToolError` (`result.error === true`, `lib/agent/tools/types.ts`) → `ERROR` + the envelope as `output_json`; thrown → `ERROR` + null output; else → `SUCCESS`. **(SF-1)** When the result is a write proposal, extract its **`pendingActionId`** from the `WriteToolResponse` envelope (the write tools return it; cf. `extractPendingActionId` already used in `chat-message.tsx`) and store it as `pending_action_id` on the row — this is the precise key approve/reject update on (AC 13/14).
10. **Fail-safe:** the log write is wrapped in its own `try/catch`, logged as `console.error('[AGENT_DECISION_LOG_WRITE_FAIL]', err)` and **never** re-thrown / never alters the tool's return. Mirror the `ChatUsageEvent` fail-safe pattern (`app/api/chat/route.ts:543`).
11. Every tool from `createAgentTools` has its `execute` wrapped. `DecisionLogContext` (`{ workspaceId, userId, chatMessageId?, modelVersion }`) is assembled in the factory from existing `workspaceId`/`userId`/`context.assistantMessageId` + a new `modelVersion?` carried on `AgentToolContext` (`index.ts:55`). **`model_version` = the route's resolved model name (SF-2):** the route already computes `modelName = modelProvider === 'anthropic' ? 'claude-sonnet-4-6' : 'gpt-4-turbo'` (`route.ts:358-359`, in `onFinish`, for `ChatUsageEvent`). **Lift that computation to alongside `modelProvider` (`~:268`)** and reuse the one `modelName` var both at the `createAgentTools` call (`:261`, as `context.modelVersion`) and in `onFinish` — so `AgentDecisionLog.model_version` and `ChatUsageEvent.model` always agree. Do NOT log the `AI_CHAT_MODEL` provider string.
12. **Seam for 19.2-19.4:** new tools (a) add a `TOOL_REGISTRY_POLICY` entry (compile-enforced) and (b) wrap `execute` with `wrapWithDecisionLog` — one line each, no other plumbing. Document this in Dev Notes.

### Approve/reject propagation

13. `approvePendingAction` (`app/actions/pending-agent-actions.ts:155`), on successful dispatch, best-effort `agentDecisionLog.updateMany({ where: { pending_action_id: <id>, outcome: 'WRITE_PROPOSED' } })` → `accepted_at=now()`, `accepted_by=userId`, `outcome=WRITE_ACCEPTED`. **Match on `pending_action_id` (the exact proposal), NOT `chat_message_id + tool_name`** — a single turn can propose several writes of the same tool on the same message (14.23 `AgentActionBatchCard`), so the fuzzy key would wrongly mark sibling proposals accepted (SF-1). Failure must not block the dispatch (try/catch, no re-throw).
14. `rejectPendingAction` (`:412`) — equivalent best-effort update keyed on `pending_action_id` → `WRITE_REJECTED`.

### Non-regression

15. OWNER/ADMIN/HR_MANAGER/MEMBER get identical tool sets to today (no write tool removed). 16. AUDITOR sees no write `ToolCallRow` (none in the registry). 17. No new `TOOL_CONFIG` entries needed (server-side concern).

### Notes
- **`PendingAgentAction` vs `AgentDecisionLog`:** the former = proposal lifecycle; the latter = every tool *call* (reads, failed writes, direct writes). Related via `chat_message_id`. On approve, both update (PendingAgentAction.status=APPROVED + log.outcome=WRITE_ACCEPTED).
- **`save_assessment` is `write`** (excluded for AUDITOR) even though it uses `changes:acknowledge` not `tasks:edit` — AUDITOR lacks both, so the `tasks:edit` gate excludes it correctly. The read/write split is coarse but correct for the current role set (AUDITOR is the only read-only role).
- **No `TOOL_CONFIG` change** — server-side only.
- **Perf (NH-2):** the wrapper adds one `agentDecisionLog.create` per tool call → up to ~10 inserts on a `stepCountIs(10)` turn. Acceptable — the writes are indexed, fail-safe (never block the turn), and off the user's critical render path. If volume becomes a cost/latency issue, batch the rows in `onFinish` (one `createMany`) as a future optimization; v1 writes per-call for simplicity + correctness.
