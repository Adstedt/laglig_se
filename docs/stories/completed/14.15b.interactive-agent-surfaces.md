## Story

**As a** compliance manager chatting with the agent,
**I want** agent suggestions and tool results to appear as interactive cards in the work area sidebar — with forms, buttons, and real-time status — rather than flat text in chat bubbles,
**so that** I can review, modify, and confirm agent actions in a structured UI without copy-pasting or switching pages.

## Context

Story 14.15a established the sidebar shell and proved the pattern with citations and generic tool results. This story makes the sidebar **actionable** — the agent's write operations render as interactive forms, the assessment flow moves from a cramped chat footer to a full sidebar panel, and document/task references become rich detail views.

### The Key Shift: From Confirmation Text to Confirmation UI

Currently, when the agent calls `create_task` with `execute: false`, it returns a text preview in the chat:

> "Jag föreslår att skapa uppgiften: **Uppdatera riskbedömning för kemikaliehantering**. Ska jag gå vidare?"

The user types "ja" and the agent calls `create_task` with `execute: true`. This works but is clunky — the user can't modify the suggestion before confirming, and the preview is just text.

With the work area, the same flow becomes:

1. Agent calls `create_task(execute: false)` → sidebar opens with a **pre-filled task creation form**
2. User can edit title, description, priority, related document before confirming
3. User clicks "Skapa" → the form calls the server action directly (no round-trip through the agent)
4. Sidebar shows success state → chat receives a system message noting the task was created

The agent proposes, the sidebar renders, the user decides.

## Acceptance Criteria

### Write Action Preview Cards

1. When the agent returns a `WriteToolResponse` (from any write tool with `execute: false`), the sidebar automatically opens with a specialized preview card for that tool type.
2. Each write tool has a dedicated preview/form component:
   - `create_task` → Task creation form (editable: title, description, priority, related document)
   - `update_compliance_status` → Status change card (shows old → new status, editable reason, confirm/cancel)
   - `save_assessment` → Assessment form (impact level, analysis preview, editable notes, confirm)
   - `add_context_note` → Note preview card (editable note text, confirm)
3. Preview cards have "Bekräfta" (confirm) and "Avbryt" (cancel) buttons.
4. Confirming executes the action via the corresponding server action directly (NOT by sending "ja" back to the agent). This avoids an unnecessary LLM round-trip.
5. After successful execution, the card shows a success state with the result summary. A system-level message is appended to the chat noting the completed action (e.g., "Uppgift skapad: Uppdatera riskbedömning").
6. Cancelling closes the sidebar and appends a system message: "Användaren avbröt åtgärden."
7. The user can edit fields in the preview card before confirming — the sidebar form is not read-only.

### Assessment Resolution in Sidebar

8. When `contextType='change'` and the agent has replied, the `AssessmentResolution` component renders in the sidebar instead of as a `footer` in `ChatMessageList`. Currently the assessment footer renders in two places: `ChatPanel` (used by `change-assessment-modal.tsx`) and `change-assessment-view.tsx` (Hem dashboard). This story targets `change-assessment-view.tsx` — it already renders in the Hem layout where `ChatDetailProvider` is available.
9. The sidebar gives the assessment form more space: full-width status/impact selectors, expandable notes textarea, and the agent's AI analysis displayed above the form for reference.
10. The assessment card in the sidebar includes the change event summary (amendment SFS, change type, affected sections) as a header section — context the user previously had to remember from the chat.
11. The existing `AssessmentResolution` component is adapted (not rewritten) — extract the form logic into a shared hook or component that both the sidebar and the legacy footer can use during the transition.

### Task Detail View

12. When the agent references an existing task (via `create_task` result or inline mention), clicking opens a task detail card in the sidebar showing: title, description, status, priority, assignee, related document, creation date.
13. Quick actions available in the task detail: "Visa uppgift" (navigate to task modal/page), "Ändra status" (inline status selector).

### Document Detail View

14. When the agent references a document (via `get_document_details` result or inline mention), clicking opens a document detail card in the sidebar showing: title, document number, content type, summary/description, section list, "Visa dokument" link to the document reader.
15. If the document is in the user's law list, show compliance status badge and the assigned group.

### Law List Item Detail View

16. When the agent references a law list item (during assessment or compliance discussion), the sidebar shows: document title, compliance status with color badge, assigned group, business context notes, last change acknowledged date, and "Visa i laglistan" link.
17. Compliance status is editable inline — changing it creates a `ComplianceStatusLog` entry (reusing the existing server action from 14.10).

### Detail Type Registry

18. Extend the `ChatDetailItem` discriminated union from 14.15a with new variants:
    ```typescript
    | { type: 'write-preview'; id: string; toolName: string; data: WriteToolResponse }
    | { type: 'task'; id: string; data: TaskDetailData }
    | { type: 'assessment'; id: string; data: AssessmentDetailData }
    | { type: 'document'; id: string; data: DocumentDetailData }
    | { type: 'law-list-item'; id: string; data: LawListItemDetailData }
    ```
19. The `ChatDetailSidebar` switch statement in 14.15a is extended with cases for each new type.

### Agent-Initiated Sidebar Opening

20. Tool definitions in `lib/agent/tools/` gain an optional `sidebarHint` property in their response metadata: `'open' | 'suggest' | 'none'`.
    - `open`: Sidebar opens automatically when the tool result arrives (write previews)
    - `suggest`: A clickable "Visa detaljer" chip appears in the chat message (read results with rich data)
    - `none`: Tool result stays inline only (default, backwards-compatible)
21. The `ToolCallRow` in `chat-message.tsx` reads `sidebarHint` from the tool result metadata and renders accordingly.

### Chat System Messages

22. A new `SystemMessage` component renders lightweight, non-bubble messages in the chat for sidebar-initiated actions: "Uppgift skapad", "Bedömning sparad", "Status uppdaterad". Styled as centered, muted text (similar to date separators in `ChatMessageList`).
23. System messages are NOT persisted to chat history — they are ephemeral UI feedback.

### Non-Regression

24. All existing chat functionality continues to work: streaming, history, export, followup chips, citation hover cards.
25. The `AssessmentResolution` footer in `ChatPanel` continues to work in sidebar/panel mode (non-Hem pages) where the work area sidebar is not available.
26. Write tool confirmation via chat text ("ja"/"nej") continues to work as a fallback when the sidebar is not present (panel mode, mobile bottom sheet too small for forms).

### Write Tool Confirmation: Sidebar vs Chat Fallback

The two-step write confirmation pattern (`execute: false` → preview → `execute: true`) currently goes through the LLM:

```
Agent: create_task(execute=false) → "Ska jag skapa uppgiften X?"
User: "Ja"
Agent: create_task(execute=true) → "Uppgift skapad!"
```

With the sidebar, the confirmation bypasses the LLM entirely:

```
Agent: create_task(execute=false) → sidebar opens with form
User: clicks "Bekräfta" → server action called directly
Sidebar: shows success → system message in chat
```

This saves one LLM round-trip per write action (~$0.01-0.03 per confirmation). The agent doesn't need to know the user confirmed — the action is done.

**Fallback**: When sidebar is not available (panel mode, mobile), the existing chat-text confirmation continues to work. The `sidebarHint` is just a hint — if no sidebar provider is in the tree, the tool result renders inline as before.

### Sidebar Auto-Open Timing

When a tool result with `sidebarHint: 'open'` arrives during streaming, the sidebar should open after the tool call row renders (not immediately on the raw stream event). Use a `useEffect` in `ChatMessage` that watches for completed tool parts:

```typescript
useEffect(() => {
  const writePreviewPart = message.parts.find(
    p => isToolUIPart(p) && p.state === 'output-available' && p.output?._meta?.sidebarHint === 'open'
  )
  if (writePreviewPart) {
    openDetail({ type: 'write-preview', id: writePreviewPart.toolInvocationId, ... })
  }
}, [message.parts])
```

## Risk Assessment

### Implementation Risks

- **Primary Risk**: Write preview auto-open during streaming could cause layout flicker if the tool result arrives mid-scroll
- **Mitigation**: Debounce sidebar open by 100ms after tool result arrives; verify scroll position is preserved

- **Secondary Risk**: Server action direct calls bypass the agent's awareness — the agent might try to re-confirm an already-executed action
- **Mitigation**: After sidebar confirmation, inject a synthetic assistant message or tool result noting completion. The agent sees this in context and moves on.

- **Tertiary Risk**: Extracting `AssessmentResolution` logic into a shared hook could introduce regressions in the existing footer
- **Mitigation**: Keep existing component working — it consumes the hook but its JSX and behavior don't change. Test both sidebar and footer paths.

### Rollback Plan

- Each detail view component is self-contained — can be removed independently
- The `sidebarHint` metadata is additive to tool responses — removing sidebar rendering falls back to existing inline display
