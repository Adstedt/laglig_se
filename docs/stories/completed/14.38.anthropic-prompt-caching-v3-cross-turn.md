## Status

## Story

**As a** product owner responsible for chat unit economics,
**I want** a follow-up turn to read the prior turns' conversation from cache instead of re-billing the whole history at full input price,
**so that** multi-turn conversations stop restarting the cache at the system block every turn — lifting the cache hit rate toward ≥65% and cutting per-turn cost further, which is what makes the usage tiers viable.

## Background — what 14.37 found

Story 14.37 added within-turn caching (`prepareStep` moving breakpoint) and froze the system prefix. Its smoke test proved:
- **Within-turn caching works** — per-step `cache_read_input_tokens` climbs across steps; multi-step turns cost ~35–40% less.
- **Cross-turn caching does NOT work** — *every turn's first step reads only the ~27,040-token system block*; the accumulated conversation is re-billed full price. Aggregate hit% plateaus at **~48%**.

**Root cause (confirmed in code):** `lib/agent/build-model-messages.ts` (`buildModelMessages`) sends prior-turn history as a **text-only projection** — it drops `tool_use`/`tool_result` blocks and empty assistant turns, and the per-turn `pendingActionsBlock`/attachment injection on the live last user message is never replayed. So the message prefix sent on turn N+1 does **not** byte-match the *live* prefix that 14.37 cached during turn N. Anthropic caching is an exact prefix match, so the read stops at the (stable) system block and everything after is a cache miss.

There is also a **persistence gap**: assistant tool-only turns persist with empty content and tool parts are not stored (only text + metadata) — see 14.37/14.22 notes. So the exact live prefix (with tool blocks) cannot simply be reconstructed from the DB on the next turn. *(This is precisely why the spike rejected verbatim replay and chose **Option A** — cache the deterministic text history instead. See the Architect Decision.)*

## Acceptance Criteria

1. **Cross-turn read lands:** on a warm follow-up turn (within the cache TTL) in a multi-turn chat, the turn's **first** step reads materially more than the system block alone (i.e. it reads the prior turns' conversation from cache, not just ~27K). Verified via per-step `cache_read_input_tokens` and `chat_usage_events`.
2. **Hit rate climbs across a conversation:** in a warm back-and-forth, `cache_hit_pct` rises turn-over-turn as history accumulates, reaching **≥65%** on later turns of a multi-turn `GLOBAL`/`LAW`/`CHANGE` conversation (the target 14.37 deferred).
3. **Cost down further:** average `cost_usd_estimate` per turn for multi-turn conversations drops materially below the 14.37 (within-turn-only) level — especially 1-step follow-ups, which 14.37 left at the ~27K-read floor.
4. **Byte-stable history at the serialized-payload level (the linchpin):** the cached history prefix must be **byte-identical turn-to-turn at the rendered API-payload level — not merely semantically identical.** Anthropic's cache key is the serialized prefix bytes, and a drift fails **silently** (cache miss, no error, looks like "the feature didn't help"). No per-turn-volatile content may sit inside the cached history span. This is the property the whole story hinges on, so it is verified by an explicit **cross-turn payload-determinism diff** (see Tests). Guard explicitly against the silent breakers: whitespace/trailing-newline diffs, message/part ordering, **empty-message filtering** (`buildModelMessages`'s drop-empty-turn branch), metadata leakage into the model payload, attachment summaries, timestamps, `pendingActionsBlock`, and markdown normalization.
5. **No regression to 14.37 or markdown rendering:** within-turn caching still works; lists/tables/code/citations/approval cards render correctly; OpenAI path unchanged; ≤4 cache breakpoints.

### Anthropic caching principles (decision-neutral; verified 2026-06-15 vs Anthropic prompt-caching guidance + AI SDK docs via Context7)

> These are the canonical facts the **🏗️ Architect Spike Decision** below applies — cross-turn caching is the standard multi-turn pattern, not exotic tuning. (Earlier drafts of this section leaned toward "verbatim history / Option B"; that recommendation is **superseded** by the Architect Decision → Option A. The facts below are kept because they're approach-neutral.)

- **Prefix-match invariant:** "Any change anywhere in the prefix invalidates everything after it." Caching the conversation ≡ keeping the conversation prefix byte-identical across requests. The current per-turn mutation (re-injecting `pendingActionsBlock` into the user message) is exactly the "silent invalidator" the docs warn against — the Architect Decision removes it.
- **Documented multi-turn recipe:** "Put a breakpoint on the last content block of the most-recently-appended turn; each subsequent request reuses the entire prior conversation prefix; earlier breakpoints stay valid, so hits accrue incrementally." 14.37 already does the moving breakpoint; the missing piece is a breakpoint at the **stable-history boundary** + not mutating history.
- **Why we cache the projected text (not verbatim tool blocks):** the persisted text history is already deterministic **and** lean — so it is both byte-stable *and* cheap to keep in context. Re-billing it at 100% input price (today's behaviour) is far more expensive than reading it at ~10% cache-read price. Sending tool blocks verbatim would also make history byte-stable, but only by re-sending large tool results every turn — see the Architect Decision for why that trade is rejected here.
- **Cache TTL refinement:** evaluate `cacheControl: { type: 'ephemeral', ttl: '1h' }` for resumed/gappy chats — the 14.37 smoke test missed partly because the default 5-min TTL expired between turns (write cost 2× vs 1.25×; break-even ~3 reads).
- **20-block lookback:** a long agentic turn (>20 content blocks) can make the next turn's breakpoint miss the prior cache — may need an intermediate breakpoint. Stay within the **4-breakpoint** max (system + history + moving = 3; one to spare).
- **Compaction:** relevant only once the (lean text) history approaches the 1M window — **deferred for this story** (see Architect Decision); would have been mandatory under verbatim history.

### 🏗️ Architect Spike Decision (Winston, 2026-06-15) — supersedes the v0.2 Option-B lean

**Decision: Option A (stable projected-text history breakpoint). Do NOT do Option B (verbatim tool blocks). No schema/persistence change.**

**Why the v0.2 "verbatim is canonical" recommendation is wrong _for this codebase_:**
- The "send history verbatim" pattern is canonical only for apps that already send full history. **This app deliberately projects history to lean text** (`buildModelMessages` → `textOf`; tool blocks dropped). That projection exists to protect the token budget **and** the 1M context window.
- Option B would **re-send every prior tool result (search hits, law text — often 10s of KB) on every turn**, even cached: it bloats the real context, **forces compaction far sooner**, and requires **persisting tool blocks the app intentionally does not store** (`ChatMessage` keeps text + metadata only; `app/actions/ai-chat.ts` reloads text-only). High cost, schema change, faster window exhaustion.
- **Option A needs none of that.** The persisted text history is already deterministic and byte-stable turn-to-turn (verified: load path returns `content` text only; no timestamps/IDs/volatile fields reach the model). It just was never given a cache breakpoint.

**Root cause the spike pinned down (why 14.37 floors at 27,040):** there are only two breakpoints — system (14.26) and the *moving last-message* one (14.37). The moving breakpoint caches the **current turn's volatile prefix**, which never matches cross-turn. There is **no breakpoint at the stable-history boundary**, so cross-turn reads can only hit the system breakpoint.

**The subtle catch — Fix #2 (14.37) currently _poisons_ history caching.** `pendingActionsBlock` is prepended *into the current user message*. When that user message later becomes history, it's reloaded as **pure text (no pending-actions)** — so its bytes differ from what was cached when it was the live turn. Every user message is therefore "poisoned": cached-with-pending-actions, replayed-as-plain-text → mismatch. **Option A requires relocating `pendingActionsBlock` out of the user message** into a **separate ephemeral block positioned _after_ the stable-history breakpoint**, so history messages stay pure text and byte-stable. (This satisfies AC4.)

**Target design (Anthropic path):**
```
[ tools | system ]⟦bp1: 14.26⟧ [ ...stable history, PURE TEXT... ]⟦bp2: NEW, end-of-history⟧ [ pendingActions block ][ current user msg + live tool loop ]⟦bp3: 14.37 moving⟧
```
- **bp2 (new):** `cacheControl` on the **last history message** (the prior-turn message immediately before the current user turn) in `buildModelMessages`, Anthropic path, only when ≥1 history message exists. This is the cross-turn read-point; it chains turn-over-turn (turn N writes the text-history prefix; turn N+1 reads it and extends).
- Everything volatile (pending-actions, current user message, this turn's tool loop) sits **after bp2** → re-sent full price for the current turn only, never poisoning the cached history.
- **Breakpoint budget: 3 of 4** (system + history + moving). Mind the 20-block lookback on very long single turns.

**Compaction: NOT required for 14.38 — defer.** With lean text history, the window grows slowly (Q&A text only; e.g. 50 turns × ~500 tok ≈ 25K, trivially within 1M). Compaction/context-editing becomes relevant only much later; spin it into a separate story if/when long-chat context pressure is measured. (It would have been mandatory under Option B — another mark against B.)

**Optional refinement:** evaluate `cacheControl: { type: 'ephemeral', ttl: '1h' }` on bp1/bp2 for resumed/gappy chats (the 14.37 smoke test missed partly on the 5-min TTL).

**Net implementation for dev (small, no schema change):** (1) **write the cross-turn payload-determinism diff test first** (AC4 headline — it's the only thing that proves byte-stability, which fails silently); (2) relocate `pendingActionsBlock` to a separate ephemeral block after the history boundary; (3) add `cacheControl` to the last history message in `buildModelMessages`; (4) keep the 14.37 moving breakpoint; (5) verify cross-turn `cacheR` climbs past 27,040 on warm follow-ups.

**Security:** after relocation, `pendingActionsBlock` is still server-generated (`buildPendingActionsContext`) — same non-spoofable posture as 14.37. Repositioning it as a separate block changes only *where* it sits in the prompt, not its origin or trust level. Mid-conversation `system`-role messages are **not** an option here (Sonnet 4.6 doesn't support them — Opus 4.8 only), so use a `user`-role ephemeral block.
