## Dev Notes

### Required Stripe webhook events (reference)

Our handler at `app/api/webhooks/stripe/route.ts` currently switches on these five event types:

```text
checkout.session.completed        → mark workspace active on the chosen tier
customer.subscription.updated     → tier change + Pattern A counter reset on period advance
customer.subscription.deleted     → pause workspace
invoice.payment_failed            → start 3-day grace + email owner
invoice.payment_succeeded         → clear past-due grace
```

Other events Stripe may fire (`charge.succeeded`, `invoice.created`, `payment_intent.*`, V2 thin events like `invoice_payment.paid`) fall through to a no-op `return 200` response. This is intentional — the idempotency claim happens before the switch, so they're recorded but unhandled, and the route stays simple.

### Almåsa current_period_end repair (manual data fix, optional)

If Almåsa Havshotell AB is a real production account (it appears to be a test workspace based on the naming, but verify), its `current_period_end` is currently null even though it has a paid invoice in the table. This is a residual from prior 5.4 testing where the subscription was canceled and re-created.

To repair manually if needed:

```sql
UPDATE workspaces
SET current_period_end = '<actual period end from Stripe>',
    subscription_status = 'active'
WHERE id = 'e4cd55b0-8b2c-4209-bd19-0b40f50f04f1';
```

Find the actual `current_period_end` by querying Stripe: Customers → Almåsa's customer → Subscription → check the displayed period end.

### Trial tier selection gap (5.5e — separate story to draft)

UAT during 5.5 testing surfaced that `trial_picked_tier` is never written by the onboarding flow. Effect: every trial workspace falls through to Solo limits regardless of the tier the user intends to buy. A user evaluating Team during their 14-day trial gets blocked from inviting a 2nd or 3rd colleague.

**Not a blocker for this cutover** — Solo trial limits work correctly. Story 5.5e (~2-3 hours of work) should ship soon after launch to unblock Team trials.

### Webhook idempotency race fix (post-UAT, latest commit)

The webhook route had a TOCTOU race in its idempotency logic — two concurrent deliveries of the same `evt_id` (which Stripe does for V2 thin events alongside V1) would both pass `findUnique`, both attempt `create`, and the loser's catch block would `delete` the winner's row. This was fixed by using the unique constraint as the idempotency primitive: `create` directly, treat P2002 as duplicate, return 200 without running the switch handler.

If you see `prisma:error` warning logs in Vercel for `Unique constraint failed on stripe_event_id`, that's expected duplicate-delivery handling and the HTTP response is 200. Only treat 5xx responses as actual incidents.

### Stripe customer email mismatch (informational)

When a workspace has multiple owners, the Checkout flow uses the workspace's primary owner email for the Stripe customer — not necessarily the email of the user clicking through Checkout. The webhook handler keys off `subscription.metadata.workspaceId`, so billing routing is unaffected, but the human-readable Stripe Dashboard customer page may show an unexpected email if a non-primary owner triggered the Checkout. Worth knowing for support conversations.

### Cache invalidation after webhook updates (polish — recommended follow-up)

The settings tabs are wrapped in `unstable_cache` with a 5-minute TTL. After a Stripe webhook updates `workspaces.subscription_tier`, the user has to manually reload to see the new state (the success banner asks them to). A small follow-up is to call `revalidateTag(\`workspace-${workspaceId}\`)` in the webhook handler after each `workspace.update`. Estimated effort: ~3 lines of code, ~30 minutes including verification.

### What NOT to customize (out of scope for this story)

- **Stripe Elements migration** for in-app payment UI — explicitly deferred. Only revisit if a sales conversation specifically asks for "white-label" payment UX.
- **Per-locale Stripe overrides** — Stripe uses browser locale; you can't force Swedish for an English-locale browser. Live with this.
- **Custom fonts on Stripe surfaces** — not supported. Brand consistency is achieved via colors + logo only.
