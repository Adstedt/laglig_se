# Story 2.30: Unified Document Page Rendering with Design Polish

## Status

Done

## Story

**As a** user browsing any legal document (law, amendment, regulation, or EU directive),
**I want** all document pages to render with the same polished visual structure, consistent styling, and smooth navigation,
**so that** I get a high-quality, professional reading experience regardless of document type — and the team can ingest new content types by simply providing well-structured HTML.

## Acceptance Criteria

### Shared components

1. A shared `<DocumentContent>` component exists that accepts `htmlContent` (+ optional `fallbackText`) and renders it inside `<article className="legal-document">` with automatic table wrapping, link rewriting (via `isWorkspace` prop), and `StickyDocNav` — replacing `LegalDocumentCard`, `LawSectionWithBanner`, `HtmlContentRenderer`, and `ContentWithStyledHeadings`.
2. A shared `<DocumentPageLayout>` component wraps all document pages with consistent `has-hero-header`, `max-w-4xl`, spacing, and footer — accepting `isWorkspace` to toggle breadcrumbs/container style.
3. A shared `<DocumentHero>` component renders the themed header card (icon, title, badges, quick info bar) driven by `contentType`, replacing the 4 duplicated hero implementations.
4. All 8 page files (4 types × public/workspace) use the shared components, eliminating layout duplication. The 4 types are: lagar (SFS laws), föreskrifter (agency regulations), EU documents, and amendments.

### Document reading experience

5. `StickyDocNav` (TOC sidebar with search and scrollspy) works on all 4 document types that have pages.
6. The modal rendering path (`LagtextSection` in document list) is **not affected** — it continues using its own lightweight `legal-document-modal` wrapper.

### Design polish

7. **Hero card refinement:**
   - Tighter vertical spacing within the card (current implementations have inconsistent padding)
   - Badge row: consistent gap, badge size, and font weight across all content types
   - Icon box: consistent size (h-12 w-12), rounded-lg, theme-colored background — visually identical across types
   - Quick info bar: standardized separator line, consistent item spacing and font size
   - Action links (PDF, Riksdagen, source): consistent styling with subtle hover, icon size h-4 w-4
   - Responsive: icon box hidden on mobile (sm:flex), title wraps cleanly, badges stack on narrow viewports

8. **Content area polish:**
   - Tables: rounded-md corners, border, muted header row, row dividers, white cell backgrounds (matching the chat table styling from Story 14.21 in `globals.css`)
   - Line height: comfortable reading at 1.7 (matching `.legal-document` existing style)
   - Headings: clear hierarchy with consistent spacing above/below
   - Blockquotes: subtle left border + muted background for legal citations

9. **Status badges:** consistent color language — `Gällande` = green outline, `Upphävd` = muted/strikethrough, `Ändringsförfattning` = orange, document type badge = theme color.

10. **Visual acceptance:** gold-standard pages render with improved polish while preserving content structure: SFS 1977:1160 (public), AFS 2023:13 kap. 4 (workspace), GDPR 32016R0679 (workspace), and one amendment page.

11. All existing tests pass, `npx tsc --noEmit` clean.

### Completion Notes List

- **Architecture decision — `DocumentContent` children escape hatch.** The story's AC 1 lists the 4 surfaces being consolidated (`LegalDocumentCard`, `LawSectionWithBanner`, `HtmlContentRenderer`, `ContentWithStyledHeadings`), but SFS law pages also depend on `LawContentWithHighlights` (runs DOM post-processing for `/Träder i kraft.../` future-amendment highlights). Rather than baking that specific logic into the generic `DocumentContent`, I added an optional `children` prop that lets callers inject any renderer while still getting the `.legal-document` wrapper, table post-processing, and `StickyDocNav`. A new `LawDocumentContent` client component in `components/features/law/` composes `FutureAmendmentsBanner` + `DocumentContent` + `LawContentWrapper` and is what law pages use in place of the deleted `LawSectionWithBanner`. The card/border container and the "Lagtext" CardHeader were intentionally dropped — the article now flows directly beneath the hero, matching the amendment/EU/AFS pages visually.
- **Status badge language (AC 9).** Introduced a typed `DocumentStatusBadge` discriminated union (`active`/`repealed`/`draft`/`archived`/`not-in-force`) rendered by a private `HeroStatusBadge`. This unifies the previously-inconsistent inline badge logic across SFS/workspace/AFS variants (green emerald "Gällande", muted strikethrough "Upphävd", blue "Ikraft {date}"). Callers convert their native status (Prisma `DocumentStatus` + `isNotYetInForce` flag) via a small `resolveStatusBadge` helper in each page.
- **`exactOptionalPropertyTypes` compliance.** Tsconfig has `exactOptionalPropertyTypes: true`, so all optional props in `DocumentHero`, `DocumentContent`, etc. are typed `T | undefined` explicitly. Important when callers pass computed values that may be `undefined` (e.g. `resolveStatusBadge` returning `undefined` for unknown statuses).
- **`FloatingRelatedButton` preservation.** Task 9 directed deleting `components/features/content/` wholesale; this button shares the directory and is still used by `FloatingImplementationsButton`. Moved it to `components/features/floating-related-button.tsx` and updated the single consumer rather than leaving a stub directory.
- **Integration test failures are pre-existing.** Full test run shows 18 integration-test failures across Redis/Supabase/DB connection-dependent suites (caching, redis-integration, auth-user-creation, database/multi-content-schema, performance-audit-live). None touch the document-page code path. 3555 unit tests pass, including the full `paragraf-toc` suite (StickyDocNav untouched), AFS splitter/transformer, and the newly-fixed `lagtext-section` test.
