> 📌 **Origin (2026-06-24, Quinn / QA):** logged as **UX-001** during the Story 19.14 review. Story 19.14 enabled adaptive thinking, which **auto-enables interleaved thinking** (the model reasons between every tool step). The chat UI renders each reasoning part as its own "Tänkte klart" breadcrumb, so heavy multi-step turns now show a **ladder of 7+ repeated breadcrumbs** interleaved with tool chips (observed on a tool-heavy CHANGE assessment). This is **not a 19.14 defect** — 19.14 was config-only and never touched rendering; it merely exposed the existing per-part breadcrumb design to many more reasoning events. This story makes the multi-step "agent working" state calm, legible, and reassuring.

## Story

**As a** user watching the agent work through a multi-step assessment,
**I want** a single, stable "working" indicator (with elapsed time) that collapses into one tidy summary when finished,
**so that** I understand the agent is actively progressing (not stuck) without a noisy ladder of repeated "Tänkte klart" rows.

## Acceptance Criteria

### Layout — "beside = current state, beneath = the record"
1. **While a turn is live** (`isActive`), a single **"working" status row renders BESIDE the avatar** (same `flex items-center gap-2.5` pattern as today's `showThinking` row), in a **fixed position** — it updates in place and does NOT push a new row per reasoning/tool step (no ladder, no vertical jumping). The row persists from the first reasoning/tool step until the answer text begins streaming (or the turn ends). **The streaming reasoning breadcrumbs and running-tool rows that render as top-level rows today are suppressed during the live turn** — they are represented by this one row instead. **(Owner-confirmed 2026-06-24:** the "one calm row" model — *full* suppression of both reasoning rungs and running-tool chips — chosen over the lighter "row + live tool chips" alternative. Implement full suppression; do not show per-tool chips during the live turn.**)**
2. The live status shows the **current activity**, updating as the agent progresses (e.g. `Tänker…` → `Söker i lagdatabasen…` → `Läser…` → `Skriver utkast…`), derived from the **most recent active part**: a streaming reasoning part → a rotating phrase via `useRotatingThinkingPhrase`; a running tool part (`state !== 'output-available'`) → that tool's `TOOL_CONFIG[name].label` (`chat-message.tsx:83-305`). Fall back to the rotating phrase when no specific activity is identifiable.
3. **When the turn completes** (`turnSettled`), the live status is replaced by a **single collapsed summary rendered BENEATH the avatar** (part of scrollback): **"Tänkte och sökte i {N} steg · {duration} ›"**, expandable to the full step timeline. The individual reasoning + tool steps live **inside** this expandable group — they no longer render as top-level rows. `N` = count of agent steps in the turn (completed reasoning parts + tool calls). The summary verb adapts to what actually happened (reasoning+tools → "Tänkte och sökte…"; tools only → existing tool-group summary; see AC 9).
4. The collapsed summary is **consistent with `CollapsedToolGroup`'s visual language** (extend/compose it — same `bg-muted/40` row, `Check`, chevron, expanded per-step list — rather than introduce a divergent style).

### Elapsed timer
5. An **elapsed-time counter** ("not stuck" signal) renders in the live status (e.g. `⟳ {activity}… · 12s`). It is **threshold-in**: only appears after ~2–3 s so fast turns don't flash a timer. Timer start = when `isActive` first becomes true for the message.
6. On completion the timer **freezes into the summary** (`… · 34s`). **On history reload the duration is unavailable** (no live clock) — omit it from the summary gracefully (no "· undefined"). _(Sub-clause **DEFERRED, not required for Done / excluded from the M estimate**: reporting the frozen duration to telemetry — pairs with 19.14 AC9/OPS-001, which still wants a prod wall-clock signal. If trivial to add an `onFinish` duration log, do it; otherwise leave for the 19.14 OPS-001 follow-up.)_
7. **DEFERRED (optional, not required for Done / excluded from the M estimate):** on long turns (> ~30 s) show a subtle reassurance line (e.g. "Större bedömning pågår…") so long assessments don't read as broken. Capture as a fast-follow if user testing shows long turns still read as stuck.

### No-think / degenerate cases (preserve 19.14 AC8)
8. A turn where the model **skips thinking** (adaptive no-think) still degrades cleanly — no empty working row, no empty summary, no layout shift. A turn with **no tools and no reasoning** behaves exactly as today (plain streaming text under the avatar).
9. A turn with **only one** reasoning step (or none) does not show a heavyweight "N steg" summary where a minimal treatment is appropriate (avoid "Tänkte i 1 steg" noise — collapse a lone reasoning step to the existing single "Tänkte klart" breadcrumb or nothing; a single tool keeps today's standalone `ToolCallRow`).

### Accessibility
10. The **activity label** is announced via `aria-live="polite"` (announce on activity change — matches today's `ReasoningBlock` streaming label, `chat-message.tsx:1045`). The **ticking seconds are NOT in an aria-live region** (a screen reader must not announce every second — keep the counter outside the live region). The expand/collapse control is keyboard-operable and labelled (reuse `CollapsedToolGroup`'s `aria-expanded`/`aria-controls`/`aria-label` pattern, `:1272-1286`).

### Quality
11. `pnpm typecheck` + `pnpm eslint` clean. **Existing `tests/unit/components/features/ai-chat/collapsed-tool-group.test.tsx` and `sidebar-hint.test.tsx` still pass** (they render `ChatMessage` with synthetic `message.parts`). Add tests (new sibling file, e.g. `agent-progress-group.test.tsx`, reusing the existing `toolPart`/`assistantMessage` helpers + streamdown/chat-detail mocks) for: ladder-coalescing (N interleaved reasoning+tool parts → one live row while active → one summary on finish), no-think degrade (AC 8), single-step minimal treatment (AC 9), and timer threshold-in + freeze (AC 5–6, fake timers).

## Dev Notes

### Implementation approach (recommended)
- **Two independent mechanisms, one render path.** (A) *Grouping* (Task 1) changes what the **settled** turn looks like — fold reasoning+tools into one summary. (B) *Live row* (Task 2) changes what the **active** turn looks like — one pinned status instead of streaming rungs. They meet at the `showThinking`/`isActive` branch (`chat-message.tsx:892-903`). Build A first (it's testable against settled `message.parts` exactly like the existing `collapsed-tool-group.test.tsx`), then B.
- **Grouping detail (Task 1).** In `groupCompletedToolParts` (`:383`), a `step-start` is already skipped (`:410`) and tool runs already coalesce; the one change is: when `isReasoningUIPart(part) && part.state === 'done'`, push it into `currentGroup` rather than taking the `!isToolUIPart` flush branch (`:429`). A *streaming* reasoning part (`state !== 'done'`) during the live turn should NOT create a top-level rung (Task 2 suppresses it / Task 1 only absorbs completed ones). `CollapsedToolGroup` currently maps every item through `extractToolPartInfo` (`:1085`) and coalesces by `toolName` (`:1090`) — reasoning items have no `toolName`, so either branch them inside the component or wrap in `AgentProgressGroup`. **Reuse, don't reinvent** the chevron/expand/`+N till`/`aria-*` shell (`:1192-1308`).
  - **Interleaved-reasoning coalesce rule (important):** when a reasoning step lands *between two runs of the same tool* (e.g. `search_laws` → reasoning → `search_laws`), treat the reasoning step as its own step that **does not merge** the two tool runs into one `(N)` count — i.e. a reasoning item breaks the `toolName` run-coalesce just like a different tool would. The expanded timeline therefore reads `Sökte i lagdatabasen (3) · Tänkte · Sökte i lagdatabasen (2)` in stream order, not a single `(5)`. This keeps the per-step record faithful to what actually happened. The top-level **summary count `N`** still totals all steps (reasoning + tool calls) regardless of coalescing.
- **Live row detail (Task 2).** Today `hasRenderableContent` (`:886-889`) flips `showThinking` off as soon as a reasoning/tool/text part exists. New behavior: while `isActive`, keep a single status row beside the avatar until **answer text** starts streaming. Compute `currentActivity` from the parts: last part that is a streaming reasoning part → rotating phrase; last running tool part → `TOOL_CONFIG[toolName]?.label`. When the first non-empty text part appears, hand off to the normal text render (the row can disappear or convert to the settled summary on `turnSettled`).
- **Timer (Task 3).** Plain `Date.now()` is fine in a client component (this is app code, not a workflow script). Capture start in a `useRef`/state when `isActive` becomes true; tick via `setInterval` (1 s) gated on `isActive`; on `isActive=false` freeze the last value for the summary. Threshold-in: render the counter only once elapsed ≥ ~2–3 s.

### Layout & UX rationale (Quinn)
- **beside the avatar = current state, beneath = the record.** The live pulse stays pinned to the (non-moving) avatar so the eye has a stable anchor and the ladder disappears; the persisted detail becomes a scrollback summary like any other message part. This matches the existing `showThinking` row and the `CollapsedToolGroup` pattern — compose, don't reinvent.
- **Timer rationale:** a spinner alone can look hung; a ticking number proves liveness even during long silent stretches (a big reasoning block or document draft — exactly the 19.14 AC9 turn). Two independent cues: rotating **activity label** (what) + **timer** (still alive). Claude Code uses the same idiom.
- **a11y trap:** do NOT put the seconds counter in `aria-live` — it would be announced every second. Announce activity-label changes only.
- **Don't surface raw reasoning text** — English trace, intentionally masked (`ReasoningBlock` comment, `chat-message.tsx:1020-1031`). This story changes *grouping/affordance*, not visibility of the trace.
