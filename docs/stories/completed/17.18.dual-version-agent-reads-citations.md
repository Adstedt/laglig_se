## Story

**As an** AI compliance agent,
**I want** my workspace-document read tools to ground my responses in the **currently-effective approved version** of each styrdokument — and to know about any in-progress draft as a secondary signal — instead of silently degrading to the draft content during revision windows,
**so that** my `[Källa:]` citations stay accurate and trustworthy for the entire revision lifecycle of a policy, and so I can naturally hedge with `[Utkast:]` when a relevant pending revision is in flight.

## Acceptance Criteria

### Indexing contract refactor

1. **`indexWorkspaceDocument` content source changes:** under the new model, the indexed content for a doc keys off:
   - If `current_approved_version_id` is set: index the approved version's content as the canonical chunks (`metadata.tier: 'APPROVED'`).
   - If `current_draft_version_id` is set: index the draft version's content as secondary chunks (`metadata.tier: 'DRAFT'` — preserves 17.10b's DRAFT indexing, but with explicit tier metadata distinguishing it from the approved canonical).
   - For docs in the dual state (both pointers set): BOTH content snapshots are indexed as separate chunk sets, each with the appropriate `metadata.tier`. Chunks from each tier are independently searchable.
   - For docs in a single-pointer state (only approved OR only draft): behavior is unchanged from 17.10b — just one set of chunks indexed.
   - When `current_draft_version_id` becomes null (draft discarded or promoted): the DRAFT-tier chunks are **removed** (chunk rows physically deleted, not marked-stale) by `indexWorkspaceDocument` itself on the next invocation. **PO v1.1 ratification (SF-1):** v1.0 said "removed (or marked stale)" leaving an undefined choice; pinned to **removed** because (a) it keeps the search query simple — no `WHERE metadata.tier_stale = false` filter to maintain — and (b) it bounds index growth on docs that go through many promote/Förkasta cycles (the Almåsa toilets policy reached v9 over a few smoke iterations; mark-stale would have accumulated 6+ orphaned tiers worth of chunks for one doc). The chunk rows themselves are short-lived and cheap to recreate; the version row history that compliance cares about lives in `workspace_document_versions`, not chunks. Story 17.16's `discardDraft` and `promoteDraftToApproved` server actions trigger `after(indexWorkspaceDocument(...))`; the sweep cron `/api/cron/sweep-draft-reindex` also calls into the same function — both paths inherit the cleanup behavior.

2. **Chunk metadata extension:** the existing `metadata.status` field (DRAFT / IN_REVIEW / APPROVED) is augmented with `metadata.tier` (`'APPROVED'` | `'DRAFT'`). `tier` is the operational signal for citation routing (Story 17.18 AC 4); `status` is retained for backward compatibility and for `list_workspace_documents` filtering. For pure-DRAFT (never-approved) docs, `metadata.tier = 'DRAFT'` AND `metadata.status = 'DRAFT'`. For dual-state docs, the approved chunks carry `metadata.tier = 'APPROVED'` AND `metadata.status = 'APPROVED'`; the draft chunks carry `metadata.tier = 'DRAFT'` AND `metadata.status = 'DRAFT'` (or `'IN_REVIEW'` per draft_status).

### Read tool refactor — `search_workspace_documents`

3. **`search_workspace_documents` (`lib/agent/tools/search-workspace-documents.ts`) — return shape:**
   - Each hit row still represents a chunk. The hit row's `metadata.tier` distinguishes approved vs draft tier (per AC 2).
   - When the search query matches chunks from BOTH tiers of the same document, the tool returns up to ONE approved-tier hit AND up to ONE draft-tier hit per document (instead of two arbitrary chunks). This prevents the agent from being overloaded with redundant retrievals when the same conceptual content exists in both tiers.
   - The hit row includes a `dualState: boolean` flag indicating whether the source doc has both pointers set. The agent uses this to know whether to hedge with `[Utkast:]` even on approved-tier hits.
   - `citationKey` for approved-tier hits stays clean: `"<title>"` (or `"<title> (<short-id>)"` for collision-disambiguation per 17.10 CITE-002). **PO v1.1 ratification (SF-2):** For draft-tier hits, `citationKey` is **`"<title> (utkast v<draft.version_number>)"`** — uses the draft row's actual `version_number`, NOT `<N+1>` where N is the approved version. The v1.0 `<N+1>` framing assumed draft = approved + 1, which doesn't hold after multi-cycle promote/Förkasta (e.g., the Almåsa toilets policy ended its 17.17 smoke at draft v9 over approved v8 — N+1 = 9 here, but on a freshly-branched draft it would be N+1 = 9 ≠ approved.version_number + 1 = 9). Using `draft.version_number` directly is unambiguous and survives the multi-cycle case. **For never-approved DRAFT docs** (no approved version exists; draft.version_number could be v1 on a fresh doc), use `"<title> (utkast v<draft.version_number>)"` — same shape — so the agent always knows it's looking at a draft tier and which version, regardless of whether an approved baseline exists.

4. **Citation routing in the system prompt:** updated guidance teaches the agent:
   - For dual-state docs (`dualState: true` on the hit): cite the approved tier with `[Källa: <titel>]` AND mention the pending draft with `[Utkast: <titel> (utkast v<N+1>)]` if it materially changes the answer. Frame as: *"Ert godkända <titel> kräver X[Källa: …]. Ett pågående utkast föreslår Y[Utkast: …]."*
   - For approved-only docs: `[Källa: <titel>]` as today.
   - For draft-only docs (never-approved): `[Utkast: <titel>]` as today (per 17.10b's contract), with hedging language *"enligt utkast till <titel>…"* — unchanged from 17.10b.
   - System prompt token-budget guardrail (per 17.10's 6500 cap, 17.11b's interim 6497) must be respected. Aim for ≤ +60 tokens net addition for this story; if it pushes past 6500, the story spec requires PO sign-off to bump the cap (precedent: 17.10 bumped 5500 → 6500 for a real new tool surface).

### Read tool refactor — `get_workspace_document`

5. **`get_workspace_document` return shape:** when the target doc has both pointers set:
   - `approved`: `{ versionNumber, contentJson, contentHtml, approved_at, approved_by_name }` — populated from `current_approved_version`.
   - `draft`: `{ versionNumber, contentJson, contentHtml, draft_status, created_at }` — populated from `current_draft_version`.
   - `dualState: true`.
   - When the doc has only `current_approved_version_id`: `approved` populated, `draft: null`, `dualState: false`.
   - When the doc has only `current_draft_version_id` (never-approved): `approved: null`, `draft: populated`, `dualState: false`. (`citationKey` for the draft tier per AC 3.)
   - Backward-compatible field shape: the existing `content` / `contentHtml` top-level fields on the response stay populated with `approved` content when present, else `draft` content — so any caller not aware of the new dual-state shape gets the effective version's content by default (no behavioural surprise).
   - Tool description in the agent system prompt explains the dual response shape and when to read both snapshots.

### Read tool refactor — `list_workspace_documents`

   - NEW: `draftStatus` (`'DRAFT' | 'IN_REVIEW' | null`).

## Dev Notes

- **Why dedupe to one hit per tier per doc in search (AC 3)** — without dedup, a dual-state doc's same conceptual content (the parts unchanged between approved and draft) returns two near-duplicate chunks per query. That's noise the agent must reason away. Dedup at the search-tool layer is cheaper than asking the LLM to dedup at reasoning time.
- **Why `metadata.tier` instead of just `metadata.status`** — under the existing 17.10b model, `metadata.status` mapped 1:1 to the citation tier (`APPROVED → [Källa:]`, `DRAFT/IN_REVIEW → [Utkast:]`). Under Model B, that mapping is the same in the simple cases — but for dual-state docs, the AGENT needs to know which tier each chunk belongs to without ambiguity. `metadata.tier` is the explicit operational signal; `metadata.status` stays for filtering and list display.
- **Backward-compat `content` field on `get_workspace_document`** (AC 5) — this defends against any unknown caller that reads the top-level `content` field instead of the new `approved.contentJson`. Without it, a caller that doesn't know about Model B would unexpectedly read `null` for any doc in a state they didn't anticipate. The default-to-approved-content fallback preserves the principle of least surprise.
- **17.10b's DEC-2 contract is the hardest test of this story** — it's the *"agent never cites a DRAFT as canonical policy"* compliance contract. Under Model B, the dual-state case is the new edge case: the agent has both an approved AND a draft for the same doc, and the temptation to conflate them in summary text is real. The adversarial test (Task 7) is the load-bearing check. Mirror 17.10b's pattern: a prompt that asks *"Vad är vår officiella <X>-policy?"* must elicit a response that cites APPROVED tier only, and any mention of the draft is framed explicitly as *"ett pågående utkast"* with `[Utkast:]`. Same hold-the-line as 17.10b.
- **Token budget tightness** — **PO v1.1 ratification (REF-002):** v1.0 cited an outdated "6497 / 3-token headroom" figure. ## Risk and Compatibility Check

**Primary Risk:** The DEC-2 contract (agent never cites a DRAFT as canonical) holds under Model A's simpler model because the citation tier mapping was 1:1 with the doc's `status`. Under Model B's dual-state model, the agent has both content snapshots available simultaneously — and the LLM's natural tendency to summarize-and-blend could conflate them in response text even if the chunks are tagged correctly. A weak adversarial test would let this through.

**Mitigation:**
- The adversarial DEC-2 hold test (Task 7) is explicitly the highest-priority test in this story. Mirror 17.10b's adversarial test pattern (real agent invocation, real seeded dual-state doc, prompt designed to tempt conflation).
- System prompt update (AC 7) is explicit about framing — the agent must say *"Ert godkända X kräver A. Ett pågående utkast föreslår B."*, NOT *"X kräver A och B."*. Concrete example required in the prompt to make the pattern automatic.
- If the adversarial test fails in dev: do NOT ship 17.18. Iterate on prompt engineering + the citation grammar until the test passes. The compliance integrity of `[Källa:]` is the load-bearing trust signal of the whole agent product; we don't ship a regression on it.

**Rollback:**
- All read-tool and indexing changes; rollback is straightforward (revert the tools + the indexing function). Already-indexed dual-state chunks would stay in place harmlessly; under the rolled-back tools, dual-state docs would return one tier's chunks (whichever happens to win the deduplication tie-break) — which is what 17.10b's behavior is, so it's a known prior state.
- System prompt rollback: revert the diff. Token budget returns to its prior figure.
- Adversarial test failure post-deploy: feature-flag the dual-state routing (`dual_version_citations_v1`) and fall back to single-tier routing while iterating.

**Compatibility Verification:**
- [ ] No breaking changes to existing tool APIs (return shapes are additive; backward-compat top-level fields preserved on `get_workspace_document`).
- [ ] Database changes — none in this story (the chunk metadata `tier` field is added to the existing JSON `metadata` blob; no schema migration).
- [ ] UI changes — none in this story (Story 17.17 owns UX; rehype citation pill is the only render-adjacent change and it's a render-time DOM attribute, not a layout change).
- [ ] Performance impact: search adds a dedup pass at the tool layer (cheap); indexing duplicates content for dual-state docs (modest storage increase, bounded by the number of dual-state docs at any given time — typically very few).
