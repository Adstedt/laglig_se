## Story

**As a** product owner,
**I want** the employee model wired for a clean future Fortnox sync,
**so that** enabling the integration later is an adapter + OAuth job, not a redesign.

## Acceptance Criteria

1. **Mapping adapter** `lib/integrations/fortnox/employee-mapper.ts` with pure, unit-tested `toFortnox(employee): FortnoxEmployee` and `fromFortnox(payload): EmployeeInput`. Enum mapping is **identity** (our enum members ARE the Fortnox codes — by design since 7.1); the adapter handles only field-name casing (snake_case ↔ PascalCase), date formats, Decimal conversion, and the `country` default (`SE`).
2. Typed `FortnoxEmployee` interface transcribed from `docs/reference/fortnox-employee-schema-analysis.md`, with a doc comment per field linking our column ↔ Fortnox property (single source of truth for the mapping).
3. `fromFortnox` tolerates a **partial/minimal Fortnox record** (only Email/FirstName/LastName) without throwing — populates what it can, leaves the rest null (→ "Ej komplett" via 7.4's rule, no code needed here).
4. **Round-trip test**: `toFortnox(fromFortnox(sample)) ≈ sample` for retained fields; unretained fields' preservation via `fortnox_raw` documented (the mapper does not manage `fortnox_raw` itself — the future sync job snapshots it; state this in the doc comment).
5. `@@unique([workspace_id, fortnox_employee_id])` already enforced (7.1 schema — verify, don't re-add); `FortnoxSyncStatus` transitions documented: `NOT_LINKED → LINKED → SYNCING → CONFLICT/ERROR`.
6. **No OAuth, no network calls, no UI, no schema change, no migration.** `docs/reference/fortnox-integration-plan.md` authored (matching the reference doc's style: metadata block, `---` sections, ts/json fenced blocks, dated footer) recording the deferred pieces: OAuth flow, conflict-resolution policy, sync direction, rate limits, fortnox_raw snapshotting, and the two-id distinction.

### Completion Notes List

- **Documented assumptions (all recorded in `docs/reference/fortnox-integration-plan.md`):**
  - Date wire format `YYYY-MM-DD` toward Fortnox (reference doc types fields as `date` with no format); parsing isolated in two helpers for easy adjustment against the live API.
  - `fortnox_raw` ownership: mapper never touches it; the future sync job snapshots the full raw payload (conflict baseline + preservation of unretained fields). Stated in the module doc-block and on `EmployeeInput`.
  - Encryption boundary: `personnummer` ↔ `PersonalIdentityNumber` mapped as an opaque field; sync job owns encrypt/decrypt; no crypto import.
  - Mapper is not a validator: `JobTitle` ≤ 30, Email-required-on-create, `EmployeeId` 1–15 chars pass through unmodified — sync job enforces at the API boundary.
  - Null emission policy: `toFortnox` emits the complete mapped field set with nulls; the sync job strips keys per API-call semantics.
- **Reference-doc ambiguities resolved:**
  - Task 4's "empty object → no throw" conflicts with the contract "a record without names is rejected with a typed error — the ONLY throw case" (an empty object has no names). Resolved: `fromFortnox({})` **throws** `FortnoxMappingError` (contract wins); "empty object → no throw" is tested against `toFortnox` (pure projection, zero throw cases). Both tested.
  - AC2 "transcribed verbatim": the Fortnox schema has ~100+ fields. Transcribed all mapped fields (non-optional, `| null`) plus the type-split-critical unmapped compensation fields (`MonthlySalary`, `HourlyPay`, `AverageHourlyWage`) and `FullName` as optional; the remaining ~80 payroll/vacation-ledger/tax/flex fields and nested arrays are documented as preserved via `fortnox_raw`, not transcribed property-by-property.
  - `fromFortnox` decimal output typed as precision-safe decimal **strings** (Prisma accepts strings for `Decimal` columns without float round-tripping) — keeps `EmployeeInput` structurally re-acceptable by `toFortnox` for the round-trip.
  - Enum drift guard implemented as const code arrays in the mapper (keeps the module free of `@prisma/client` imports, pure) + a test asserting set-equality against `Object.values()` of the Prisma enums, then looping every member both directions.
- **QA scrutiny list:**
  1. The "empty object → no throw" interpretation above (fromFortnox throws, toFortnox doesn't) — confirm intent.
  2. Date assumption `YYYY-MM-DD` — unverifiable until the OAuth story hits the live API.
  3. `EmployeeInput.country` is non-nullable `string` (default `'SE'`) while the column is `String? @default("SE")` — chosen so the default is explicit in the shape; flag if the upsert should instead omit it.
  4. Whitespace-only strings are treated as absent (→ null) throughout `strOrNull`; original (untrimmed) values are otherwise passed through unchanged.
  5. `toFortnox` runtime-tolerates a structurally invalid input (`{}`) beyond its type — deliberate defensive purity, not an invitation to bypass the type.
