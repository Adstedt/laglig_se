## Story

**As** the Laglig.se platform operator,
**I want** per-turn chat usage (token dimensions + cost + workspace + user + context) persisted to a `ChatUsageEvent` event log, plus admin query surfaces that aggregate the data per-workspace and per-user,
**so that** I can (1) calibrate tier pricing on measured data instead of modeled estimates (Story 5.5b), (2) verify Story 14.26's cache effectiveness at scale, (3) diagnose cost outliers per workspace or per user, and (4) feed Stripe metered overage reporting when that ships.

## Context

Story 14.26 landed a temporary `onFinish: ({ totalUsage }) => console.log('[CACHE]', totalUsage)` as the minimal observability scaffold for cache verification. That log is ephemeral — it lives in Vercel function logs with 7-30 day retention, not in a queryable table. This story converts that scaffold into a **persistent event log** with four new surfaces:

1. **`ChatUsageEvent` Prisma model** — one row per chat turn, capturing all six token dimensions (input / output / cache_read / cache_write / reasoning / fresh), context type, model, workspace, user, and computed cost estimate.
2. **`onFinish` write path** — replaces the console.log with a Prisma insert; fail-safe on write errors (don't crash the stream response if telemetry fails).
3. **Cost estimation helper** — `lib/usage/cost-estimator.ts` pure function mapping `(model, tokens)` → `cost_usd_estimate`. Lookup table of Anthropic + OpenAI pricing for the models in use today.
4. **Admin aggregation queries + minimal UI** — `getUsageByWorkspace`, `getUsageByUser`, `getUsageTimeSeries` in `lib/admin/queries.ts`, plus a `/admin/(dashboard)/usage` page with two tabs (per-workspace, per-user) showing cost + token counts for configurable time ranges (7d / 30d / 90d).

**This is a pure additive telemetry story.** No enforcement, no UI warnings for end users, no Stripe integration — those are Story 5.5b's scope. 14.27 just writes events and exposes an internal read surface. Enforcement logic reads from this log as its source of truth when 5.5b ships.

**Per-user scoping is explicit and important.** Workspaces can have multiple seats (Team tier: 5 users). Two kinds of questions need to be answered from this data:

- **Per-workspace:** "How many tokens did Workspace X consume this month?" → basis for tier enforcement + overage billing.
- **Per-user:** "Within Workspace X, which user is driving the consumption?" → basis for internal debugging, fairness analysis, and identifying power users vs quiet seats.

The schema indexes support both query shapes (see Dev Notes).

### Target shape (reference — dev agent to verify AI SDK v6 callback fields)

```ts
onFinish: async ({ totalUsage, steps }) => {
  // Story 14.27: persistent usage telemetry. Replaces 14.26's temporary console.log.
  // Fail-safe: telemetry write errors must NOT propagate to the chat response.
  try {
    const inputTokens = totalUsage.inputTokens ?? 0
    const outputTokens = totalUsage.outputTokens ?? 0
    const cacheReadInputTokens = totalUsage.cachedInputTokens ?? 0
    // NB: cache write field name — confirm against actual Story 14.26 runtime logs.
    // Runtime evidence from 14.26 review showed `cacheWriteTokens` in inputTokenDetails.
    // AI SDK v6 may expose it as `cacheCreationInputTokens` at top level OR nested.
    const cacheWriteInputTokens =
      (totalUsage as unknown as { cacheCreationInputTokens?: number })
        .cacheCreationInputTokens ??
      (totalUsage as unknown as { inputTokenDetails?: { cacheWriteTokens?: number } })
        .inputTokenDetails?.cacheWriteTokens ??
      0
    const reasoningTokens = totalUsage.reasoningTokens ?? 0

    const modelName =
      modelProvider === 'anthropic' ? 'claude-sonnet-4-6' : 'gpt-4-turbo'
    const costUsdEstimate = estimateCostUsd({
      model: modelName,
      inputTokens,
      outputTokens,
      cacheReadInputTokens,
      cacheWriteInputTokens,
      reasoningTokens,
    })

    await prisma.chatUsageEvent.create({
      data: {
        workspace_id: workspaceId,
        user_id: userId,
        model: modelName,
        context_type: (contextType ?? 'global').toUpperCase() as 'GLOBAL' | 'TASK' | 'LAW' | 'CHANGE',
        input_tokens: inputTokens,
        output_tokens: outputTokens,
        cache_read_input_tokens: cacheReadInputTokens,
        cache_write_input_tokens: cacheWriteInputTokens,
        reasoning_tokens: reasoningTokens,
        step_count: steps?.length ?? 1,
        cost_usd_estimate: costUsdEstimate,
      },
    })
  } catch (err) {
    // eslint-disable-next-line no-console -- fail-safe telemetry: log but never re-throw
    console.error('[CHAT_USAGE_EVENT_WRITE_FAIL]', err)
  }
},
```

### Schema

```prisma
model ChatUsageEvent {
  id                         String          @id @default(uuid())
  workspace_id               String
  user_id                    String
  model                      String          // e.g., "claude-sonnet-4-6", "gpt-4-turbo"
  context_type               ChatContextType

  // Token dimensions — all non-negative; default 0 for model-path-absent fields
  input_tokens               Int             @default(0)
  output_tokens              Int             @default(0)
  cache_read_input_tokens    Int             @default(0) // Anthropic-only when >0
  cache_write_input_tokens   Int             @default(0) // Anthropic-only when >0
  reasoning_tokens           Int             @default(0) // Story 14.20 thinking tokens; Anthropic-only

  // Agentic loop depth — 1 for single-call turns, 2+ for tool-using turns
  step_count                 Int             @default(1)

  // Derived — USD cost estimate at write time (stored for fast aggregation)
  cost_usd_estimate          Decimal         @db.Decimal(10, 6)

  created_at                 DateTime        @default(now())

  // Relations
  workspace Workspace @relation(fields: [workspace_id], references: [id], onDelete: Cascade)
  user      User      @relation(fields: [user_id], references: [id], onDelete: Cascade)

  @@index([workspace_id, created_at])
  @@index([user_id, created_at])
  @@index([workspace_id, user_id, created_at])
  @@index([context_type, created_at])
  @@map("chat_usage_events")
}
```

### Cost estimator pricing table

Rates per 1M tokens (verify against Anthropic + OpenAI pricing pages before merge — prices change):

```ts
const PRICING: Record<string, ModelPricing> = {
  // Anthropic Claude 4.x family (verify against https://www.anthropic.com/pricing)
  'claude-sonnet-4-6': {
    input: 3.00,       // $3/MTok
    output: 15.00,     // $15/MTok
    cacheRead: 0.30,   // $0.30/MTok (10% of input)
    cacheWrite: 3.75,  // $3.75/MTok (1.25× input, 5-min TTL)
  },
  'claude-opus-4-5': {
    input: 15.00,
    output: 75.00,
    cacheRead: 1.50,
    cacheWrite: 18.75,
  },
  'claude-haiku-4-5': {
    input: 1.00,
    output: 5.00,
    cacheRead: 0.10,
    cacheWrite: 1.25,
  },
  // OpenAI (verify against https://openai.com/api/pricing)
  'gpt-4-turbo': {
    input: 10.00,
    output: 30.00,
    cacheRead: 0,       // No cache feature parity on this model
    cacheWrite: 0,
  },
}
```

Cost formula:

```ts
cost = (inputTokens - cacheReadInputTokens - cacheWriteInputTokens) / 1_000_000 * rates.input
     + cacheReadInputTokens / 1_000_000 * rates.cacheRead
     + cacheWriteInputTokens / 1_000_000 * rates.cacheWrite
     + outputTokens / 1_000_000 * rates.output
```

Note: reasoning tokens are billed as output tokens on Anthropic per their docs, so they're included in `outputTokens` and don't need separate pricing. The `reasoning_tokens` column is for analysis, not cost computation.

### Admin query shapes (reference)

**`getUsageByWorkspace`** — Prisma implementation sketch:

```ts
// SECURITY: validate rangeDays + limit + offset as integers at function entry
// before passing to $queryRaw. Even though Prisma's tagged template auto-parameterizes
// ${interpolation}, interpolating a string into `INTERVAL '${n} days'` would NOT be
// parameterized (interval literal inside a string literal). Use arithmetic instead:
// `NOW() - (INTERVAL '1 day' * ${rangeDays}::int)` — Prisma parameterizes rangeDays
// as a number bound to the ::int cast, which is safe.

const rangeDaysInt = Math.floor(Math.max(1, Math.min(365, rangeDays))) // clamp + coerce
const limitInt = Math.floor(Math.max(1, Math.min(100, limit)))
const offsetInt = Math.floor(Math.max(0, offset))

const rows = await prisma.$queryRaw<WorkspaceUsageRow[]>`
  SELECT
    e.workspace_id as "workspaceId",
    w.name as "workspaceName",
    w.subscription_tier as tier,
    SUM(e.cost_usd_estimate)::text as "totalCostUsd",
    SUM(e.input_tokens)::bigint as "totalInputTokens",
    SUM(e.output_tokens)::bigint as "totalOutputTokens",
    SUM(e.cache_read_input_tokens)::bigint as "totalCacheReadTokens",
    COUNT(*)::bigint as "turnCount"
  FROM chat_usage_events e
  INNER JOIN workspaces w ON w.id = e.workspace_id
  WHERE e.created_at >= NOW() - (INTERVAL '1 day' * ${rangeDaysInt}::int)
  GROUP BY e.workspace_id, w.name, w.subscription_tier
  ORDER BY SUM(e.cost_usd_estimate) DESC
  LIMIT ${limitInt} OFFSET ${offsetInt}
`
```

**Security note on parameterization:** The `INTERVAL '1 day' * ${n}::int` pattern is safe because Prisma's tagged-template interpolation parameterizes the interpolated value, and the explicit `::int` cast enforces integer type at the SQL boundary. The rejected alternative `INTERVAL '${n} days'` interpolates the value INSIDE a string literal, which Prisma does NOT parameterize — that would be vulnerable to injection if `n` ever originated from user input. Apply the same clamping + cast pattern to `getUsageByUser` and `getUsageTimeSeries`.

**Type coercion:** Decimal comes back as a string from PostgreSQL via raw SQL; parse or keep as string for UI display. BigInt comes back as native bigint; `.toString()` in the UI layer.

## Risk and Compatibility

### Primary Risk

**AI SDK v6 callback field names drift or are nested differently than the story assumes.** Specifically the cache-write field location (top-level `cacheCreationInputTokens` vs nested `inputTokenDetails.cacheWriteTokens`). Dev must verify against the actual `totalUsage` object shape at runtime and code the right accessor.

### Secondary Risk

**High-traffic DB write amplification.** Every chat turn now triggers a Prisma INSERT. At a few hundred chats/day this is noise. At tens of thousands/day it could meaningfully affect DB connection pool usage. Mitigation: the write is fire-and-forget (async, fail-safe), non-transactional, and the table has no foreign-key cascades that would add write latency. For current scale (beta → early production), single INSERT per chat turn is comfortably under threshold. Revisit if `chat_usage_events` grows past ~1M rows and writes start contending with `ChatMessage` writes.

### Mitigations

1. AC 4 + AC 15 require the fail-safe path test — a Prisma insert failure must NOT crash the stream response. Dev must verify this path exists in the code AND is exercised by the integration test.
2. Prisma migration (Task 1) runs `migrate dev` locally — review the generated SQL before applying to staging/prod. No data-destructive operations — purely additive.

### Rollback

- Code rollback: revert `app/api/chat/route.ts` to the Story 14.26 state (restore the console.log, remove the Prisma write). The `ChatUsageEvent` table can be left in place (orphaned but harmless).
- Schema rollback: `pnpm prisma migrate resolve --rolled-back add_chat_usage_events` + manually drop the table via `DROP TABLE chat_usage_events;`. No data is referenced by other tables, so drop is clean.
- Partial rollback: if only the admin UI is problematic, the schema + write path can stay while the `/admin/usage` page is removed.

### Compatibility

- No breaking changes to any API contract — the chat route's request/response is unchanged
- Database migration is purely additive (one new table + indexes)
- No UI changes in the main app (admin-only surface)
- Performance: one additional async Prisma INSERT per chat turn; negligible impact
- No dependency version changes
