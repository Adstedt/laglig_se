## Story

**As a** user chatting with the AI compliance agent,
**I want** the agent to semantically search my workspace's own uploaded files (policies, procedures, evidence PDFs, spreadsheets),
**so that** it can answer questions and give advice grounded in our actual documents ‚Äî not only the global legal corpus ‚Äî while never seeing another tenant's files.

## Acceptance Criteria

### `search_workspace_files` tool

1. New tool `search_workspace_files` in `lib/agent/tools/search-workspace-files.ts`, created via a `createSearchWorkspaceFilesTool(workspaceId)` factory (closure-captured `workspaceId`, mirroring `search-laws.ts`).
2. Input: `query` (string, required, Swedish), `limit` (number, 1‚Äì20, default 5). Optional `category` filter (FileCategory enum) is a **nice-to-have** ‚Äî include only if it's a clean pass-through; otherwise defer.
3. Calls `retrieveContext(query, workspaceId, { sourceTypes: ['USER_FILE'], topK: limit })`. **Never** passes `sourceType: 'LEGAL_DOCUMENT'` and never widens to other source types.
4. Returns per result: `fileId` (= `sourceId`), `filename` + `category` (from chunk `metadata`/`contextual_header`), `snippet` (= chunk content), `relevanceScore` (rounded), `citationKey`.
5. Empty-result path returns a `wrapToolError` with a Swedish "inga resultat" message + a rephrase hint (mirror `search-laws.ts:54-61`).
6. Registered in `createAgentTools()` (`index.ts:67`) alongside the existing tools; the existing tools are unchanged.

### Citations

7. `extractSourcesFromToolResult` (`citations.ts:54`) gains a `search_workspace_files` branch that maps each result ‚Üí a `SourceInfo` keyed by its `citationKey`, so `[K√§lla: <filename>]` markers resolve to a source pill. **Citation key = the filename** (consistent with the existing `USER_FILE` rendering in `transparency.ts:98-101`). **`SourceInfo.documentNumber` is non-optional `string` (`citations.ts:16`); files have no document number ‚Äî carry the filename in `documentNumber`** (recommended) rather than relaxing the field to optional. **Why:** relaxing `documentNumber` to optional is cross-cutting ‚Äî it cascades typecheck/null-handling across every `SourceInfo` consumer (`resolveSource`, the client citation-pill components); carrying the filename keeps the type non-optional and localizes the change to this branch. **No `resolveSource` change needed:** a bare filename label (e.g. `rapport.pdf`) fails the legal-pattern regexes in `parseCitationLabel`, falls through to `resolveSource` step 3 (`sourceMap.get(trimmed)`, `citations.ts:248`), and resolves against the filename-keyed entry this branch sets. Document the chosen approach in the Dev Agent Record.
8. System prompt (`system-prompt.md`) updated: tell the agent it can search the workspace's **uploaded files** with `search_workspace_files`, when to prefer it vs `search_laws`, and to cite results as `[K√§lla: <filename>]`.
9. `TOOL_CONFIG` (`chat-message.tsx:71`) gains a `search_workspace_files` entry: `label` "S√∂ker i uppladdade filer" + `doneLabel` "S√∂kte i uppladdade filer" + an icon. Use a file/search lucide icon (e.g. `FileSearch`) ‚Äî add it to the existing lucide-react import block alongside `Search`/`FileText`; or reuse the already-imported `FileText`.

### QA carry-overs from 17.9's gate (this is the path that activates them)

10. **REL-001 fix:** `syncWorkspaceChunks` dedupe (`sync-workspace-chunks.ts:86-97`) must **not** skip re-embedding when chunks exist but carry **null embeddings**. A transient embed failure currently leaves `embedding = NULL` + the file `DONE`; a same-`content_hash` re-sync then short-circuits, making the file **permanently unretrievable** despite the documented "retryable via re-sync" recovery. Fix: skip only when `content_hash` matches **AND** zero chunks have a null embedding (raw `SELECT count(*) ‚Ä¶ WHERE embedding IS NULL`, since `embedding` is `Unsupported()` in Prisma). This becomes user-visible the moment retrieval goes live, so it ships here.
11. **TEST-001:** add a **row-level** cross-tenant isolation **integration test** (real DB, not mocked `$queryRaw`): seed `USER_FILE` chunks in workspace A and workspace B; assert a `search_workspace_files` / `retrieveContext(['USER_FILE'])` call scoped to A returns **only** A's rows and **never** B's. This is the highest-risk behaviour in the whole file-RAG feature and is currently only asserted at the SQL-parameter level. **Harness already exists ‚Äî use it, do not stand up new infra:** `pnpm test:integration` (`vitest.integration.config.mts`, includes `tests/integration/**/*.test.ts`, loads `.env.local` + `tests/setup.ts`). These tests hit the **real dev DB via `new PrismaClient()`** (no ephemeral container), so seed with a unique marker and clean up in `afterAll`. See AC 11 detail in Task 5.

### Isolation / non-regression

12. Isolation holds end-to-end: the tool only ever returns the caller's workspace's `USER_FILE` chunks (relies on 17.9's write-side invariant + the read-side workspace clause). Covered by AC 11.
13. `search_laws` and the other existing agent tools are behaviourally unchanged (additive registration only).

### Quality

14. Unit tests for the tool: result mapping, empty-result error path, and that it calls `retrieveContext` with exactly `sourceTypes: ['USER_FILE']` (never `LEGAL_DOCUMENT`). Plus the AC 10 dedupe-recovery unit test and the AC 11 integration test. `pnpm typecheck` + `pnpm eslint` clean on all changed files.
