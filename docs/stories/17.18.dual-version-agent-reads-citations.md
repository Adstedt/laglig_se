# Story 17.18: Agent Reads + Citation Routing Under the Dual-Version Model

## Status

Draft

> 🆕 **Third of three in `docs/prd/epic-17-addendum-dual-version-visibility.md`.** Sits on top of Story 17.16's data-model foundation AND Story 17.17's UX surface. Refactors the agent's three workspace-document read tools (`search_workspace_documents`, `get_workspace_document`, `list_workspace_documents` — all shipped in Story 17.10) and the 17.10b auto-reindex contract so the agent grounds its citations in the **currently-effective approved version** at all times, with the in-progress draft surfaced as a secondary signal when relevant. Closes the epic; Story 17.11c (agent auto-branch on APPROVED) is a separate follow-up.

## Story

**As an** AI compliance agent,
**I want** my workspace-document read tools to ground my responses in the **currently-effective approved version** of each styrdokument — and to know about any in-progress draft as a secondary signal — instead of silently degrading to the draft content during revision windows,
**so that** my `[Källa:]` citations stay accurate and trustworthy for the entire revision lifecycle of a policy, and so I can naturally hedge with `[Utkast:]` when a relevant pending revision is in flight.

## Story Context

**Existing System Integration (load-bearing — story 17.18 is brownfield-agent-additive):**

- **17.16 ships the data foundation.** `WorkspaceDocument.current_approved_version_id` + `current_draft_version_id` populated by the backfill migration and maintained by refactored server actions.
- **17.17 ships the UX surface.** Composite badge + doc page header + editor banner; users can visually distinguish the dual state. Story 17.18 ensures the AGENT's view of the world matches the user's.
- **Story 17.10 (Done) ships the three read tools.** Today's behavior:
  - `search_workspace_documents` (`lib/agent/tools/search-workspace-documents.ts`): semantic search over the 17.9b chunk index; each chunk row carries `metadata.status` (DRAFT / IN_REVIEW / APPROVED); citation routing in chat-message rehype emits `[Källa: X]` for APPROVED-status chunks and `[Utkast: X]` for DRAFT/IN_REVIEW chunks (per Story 17.10b's contract).
  - `get_workspace_document` (`lib/agent/tools/get-workspace-document.ts`): full-content read by `document_id`, returns content from the doc's `current_version` (which is the same as today's `current_version_id` alias).
  - `list_workspace_documents` (`lib/agent/tools/list-workspace-documents.ts`): enumeration tool, returns lists by type/status/links.
- **Story 17.10b (Done) ships the indexing contract.** `saveDocumentVersion` wraps `indexWorkspaceDocument` in `after()`. The 17.10b extension widened the indexable set from `{APPROVED}` to `{DRAFT, IN_REVIEW, APPROVED}`, content-hash-gated; status transitions are cheap metadata-only updates when content is unchanged. The sweep cron `/api/cron/sweep-draft-reindex` handles draft autosave debounce.
- **What changes under Model B (17.16):** when a doc has BOTH `current_approved_version_id` AND `current_draft_version_id` set (the new "APPROVED-with-draft-pending" state), the document conceptually has TWO content snapshots in play — the effective approved one (which is what should ground `[Källa:]`) and the pending draft one (which should ground `[Utkast:]` if the agent surfaces it). Today's indexing treats `current_version_id` as the single canonical chunk source; if that points at the draft (which it does post-branch via 17.16's alias logic), the effective policy stops being findable by `search_workspace_documents` — undoing 17.10b's compliance-safe contract.
- **Story 17.11 + 17.11b approval cards:** the agent's `update_document` and `add_document_section` write tools target the draft pointer per 17.16 AC 8–10. This story does NOT change the write tools' content target (it's already correct under 17.16). It DOES change how the agent reads back the doc state — the read tools need to expose both pointers so the agent can decide which to ground in.

**Reuse from existing infrastructure (no re-implementation needed):**
- `lib/workspace-documents/index-workspace-document.ts` — the indexing function. Story 17.18 changes the content source it indexes against, not its structure.
- `lib/ai/rehype-citation-pills.ts` (17.10b extension) — `data-tier` attribute differentiation for `[Källa:]` vs `[Utkast:]`. Story 17.18 may extend this to support a third pattern (`[Källa: X (utkast pågår)]` or paired) per AC 4.
- Chunk metadata shape from 17.10b (`metadata.status`, `metadata.version_number`).
- 19.7c registry narrowing harness — the dual-version refactor must not break the harness's expectations.

**Out of scope (deferred or follow-up):**
- Agent auto-branch on APPROVED — Story 17.11c.
- Diff-aware tool that compares approved vs draft content for the agent — useful future story, but out of scope here.
- True supersession of one policy by another structurally different one — separate concept; not in this epic.
- Workspace-document write tools beyond the existing 17.11/17.11b — owned by those stories.

## Acceptance Criteria

### Indexing contract refactor

1. **`indexWorkspaceDocument` content source changes:** under the new model, the indexed content for a doc keys off:
   - If `current_approved_version_id` is set: index the approved version's content as the canonical chunks (`metadata.tier: 'APPROVED'`).
   - If `current_draft_version_id` is set: index the draft version's content as secondary chunks (`metadata.tier: 'DRAFT'` — preserves 17.10b's DRAFT indexing, but with explicit tier metadata distinguishing it from the approved canonical).
   - For docs in the dual state (both pointers set): BOTH content snapshots are indexed as separate chunk sets, each with the appropriate `metadata.tier`. Chunks from each tier are independently searchable.
   - For docs in a single-pointer state (only approved OR only draft): behavior is unchanged from 17.10b — just one set of chunks indexed.
   - When `current_draft_version_id` becomes null (draft discarded or promoted): the DRAFT-tier chunks are removed (or marked stale) by a follow-up index sweep. Story 17.16's `discardDraft` and `promoteDraftToApproved` server actions trigger `after(indexWorkspaceDocument(...))` — Story 17.18 ensures the indexing function handles tier cleanup on these transitions correctly.

2. **Chunk metadata extension:** the existing `metadata.status` field (DRAFT / IN_REVIEW / APPROVED) is augmented with `metadata.tier` (`'APPROVED'` | `'DRAFT'`). `tier` is the operational signal for citation routing (Story 17.18 AC 4); `status` is retained for backward compatibility and for `list_workspace_documents` filtering. For pure-DRAFT (never-approved) docs, `metadata.tier = 'DRAFT'` AND `metadata.status = 'DRAFT'`. For dual-state docs, the approved chunks carry `metadata.tier = 'APPROVED'` AND `metadata.status = 'APPROVED'`; the draft chunks carry `metadata.tier = 'DRAFT'` AND `metadata.status = 'DRAFT'` (or `'IN_REVIEW'` per draft_status).

### Read tool refactor — `search_workspace_documents`

3. **`search_workspace_documents` (`lib/agent/tools/search-workspace-documents.ts`) — return shape:**
   - Each hit row still represents a chunk. The hit row's `metadata.tier` distinguishes approved vs draft tier (per AC 2).
   - When the search query matches chunks from BOTH tiers of the same document, the tool returns up to ONE approved-tier hit AND up to ONE draft-tier hit per document (instead of two arbitrary chunks). This prevents the agent from being overloaded with redundant retrievals when the same conceptual content exists in both tiers.
   - The hit row includes a `dualState: boolean` flag indicating whether the source doc has both pointers set. The agent uses this to know whether to hedge with `[Utkast:]` even on approved-tier hits.
   - `citationKey` for approved-tier hits stays clean: `"<title>"` (or `"<title> (<short-id>)"` for collision-disambiguation per 17.10 CITE-002). For draft-tier hits, `citationKey` becomes `"<title> (utkast v<N+1>)"` so the agent can render the version explicitly in `[Utkast:]` pills.

4. **Citation routing in the system prompt:** updated guidance teaches the agent:
   - For dual-state docs (`dualState: true` on the hit): cite the approved tier with `[Källa: <titel>]` AND mention the pending draft with `[Utkast: <titel> (utkast v<N+1>)]` if it materially changes the answer. Frame as: *"Ert godkända <titel> kräver X[Källa: …]. Ett pågående utkast föreslår Y[Utkast: …]."*
   - For approved-only docs: `[Källa: <titel>]` as today.
   - For draft-only docs (never-approved): `[Utkast: <titel>]` as today (per 17.10b's contract), with hedging language *"enligt utkast till <titel>…"* — unchanged from 17.10b.
   - System prompt token-budget guardrail (per 17.10's 6500 cap, 17.11b's interim 6497) must be respected. Aim for ≤ +60 tokens net addition for this story; if it pushes past 6500, the story spec requires PO sign-off to bump the cap (precedent: 17.10 bumped 5500 → 6500 for a real new tool surface).

### Read tool refactor — `get_workspace_document`

5. **`get_workspace_document` return shape:** when the target doc has both pointers set:
   - `approved`: `{ versionNumber, contentJson, contentHtml, approved_at, approved_by_name }` — populated from `current_approved_version`.
   - `draft`: `{ versionNumber, contentJson, contentHtml, draft_status, created_at }` — populated from `current_draft_version`.
   - `dualState: true`.
   - When the doc has only `current_approved_version_id`: `approved` populated, `draft: null`, `dualState: false`.
   - When the doc has only `current_draft_version_id` (never-approved): `approved: null`, `draft: populated`, `dualState: false`. (`citationKey` for the draft tier per AC 3.)
   - Backward-compatible field shape: the existing `content` / `contentHtml` top-level fields on the response stay populated with `approved` content when present, else `draft` content — so any caller not aware of the new dual-state shape gets the effective version's content by default (no behavioural surprise).
   - Tool description in the agent system prompt explains the dual response shape and when to read both snapshots.

### Read tool refactor — `list_workspace_documents`

6. **`list_workspace_documents` return shape:** each row in the enumeration includes:
   - Existing fields (title, document_number, type, etc.).
   - NEW: `currentApprovedVersionNumber` (`number | null`).
   - NEW: `currentDraftVersionNumber` (`number | null`).
   - NEW: `draftStatus` (`'DRAFT' | 'IN_REVIEW' | null`).
   - NEW: `dualState` (`boolean`).
   - The existing `status` field stays as the top-level doc lifecycle indicator.
   - Filter parameters extended: caller can request `dualStateOnly: true` to enumerate only docs with revision in progress (powers the agent's "what's currently being drafted" awareness without a separate dedicated tool).

### System prompt + citation grammar

7. **System prompt (`lib/agent/system-prompt.md`) updated** to teach the dual-state grounding:
   - New short subsection under the existing styrdokument citation guidance (Story 17.10b extension). Explains:
     - APPROVED-only doc → `[Källa: X]` only.
     - DRAFT-only (never-approved) doc → `[Utkast: X]` only, hedge with "enligt utkast till X" (unchanged from 17.10b).
     - Dual-state doc (both pointers set) → `[Källa: X]` for the effective answer + optional `[Utkast: X]` if the draft materially changes the answer. Frame as *"Ert godkända X kräver A[Källa: …]. Ett pågående utkast föreslår B[Utkast: …]."*
   - Concrete example added (Swedish, like the existing 17.10b examples).
   - **Token-budget check:** the addition must keep the prompt below 6500 tokens (the established cap). If unable, propose a focused trim elsewhere in the prompt OR escalate to PO for a budget bump per 17.10's precedent.

8. **`rehype-citation-pills.ts` extension (if needed):** if AC 4's "paired" rendering (`[Källa: X][Utkast: X]` on the same fact) needs visual differentiation in the rendered citation pills (e.g., the `[Utkast:]` pill carries a small "v4" suffix from `citationKey`), extend the rehype plugin to handle the new `citationKey` shape (`"<title> (utkast v<N>)"`). If the existing `data-tier` attribute differentiation is sufficient and the version number renders naturally, no rehype change is required — verify in dev.

### Backwards compatibility

9. **Stories 17.10 + 17.10b regression suites must remain green.** Today's tests cover the single-state cases (APPROVED-only and DRAFT-only); those cases continue to work under the refactored tools (just with the new `dualState: false` flag and `approved`/`draft` field names mapped from existing single-snapshot data). Tests in this story extend the suites with dual-state assertions.

10. **No agent behavior regression for never-approved drafts.** The 17.10b DEC-2 contract (agent NEVER cites a DRAFT as canonical policy) must hold. The new dual-state routing only changes behavior for dual-state docs (which 17.10b's tests didn't cover, because Model A made that state unreachable). Adversarial test (per 17.10b precedent): a prompt asking *"Vad är vår officiella <X>-policy?"* against a dual-state doc must result in the agent citing the APPROVED tier and explicitly framing the draft as pending (`"Ett pågående utkast föreslår…"`), NOT conflating the two.

## Dev Notes

- **Why dedupe to one hit per tier per doc in search (AC 3)** — without dedup, a dual-state doc's same conceptual content (the parts unchanged between approved and draft) returns two near-duplicate chunks per query. That's noise the agent must reason away. Dedup at the search-tool layer is cheaper than asking the LLM to dedup at reasoning time.
- **Why `metadata.tier` instead of just `metadata.status`** — under the existing 17.10b model, `metadata.status` mapped 1:1 to the citation tier (`APPROVED → [Källa:]`, `DRAFT/IN_REVIEW → [Utkast:]`). Under Model B, that mapping is the same in the simple cases — but for dual-state docs, the AGENT needs to know which tier each chunk belongs to without ambiguity. `metadata.tier` is the explicit operational signal; `metadata.status` stays for filtering and list display.
- **Backward-compat `content` field on `get_workspace_document`** (AC 5) — this defends against any unknown caller that reads the top-level `content` field instead of the new `approved.contentJson`. Without it, a caller that doesn't know about Model B would unexpectedly read `null` for any doc in a state they didn't anticipate. The default-to-approved-content fallback preserves the principle of least surprise.
- **17.10b's DEC-2 contract is the hardest test of this story** — it's the *"agent never cites a DRAFT as canonical policy"* compliance contract. Under Model B, the dual-state case is the new edge case: the agent has both an approved AND a draft for the same doc, and the temptation to conflate them in summary text is real. The adversarial test (Task 7) is the load-bearing check. Mirror 17.10b's pattern: a prompt that asks *"Vad är vår officiella <X>-policy?"* must elicit a response that cites APPROVED tier only, and any mention of the draft is framed explicitly as *"ett pågående utkast"* with `[Utkast:]`. Same hold-the-line as 17.10b.
- **Token budget tightness** — 17.11b already trimmed the prompt to 6497 tokens (3 below the 6500 cap). This story adds the dual-state citation guidance + one example. If 3 tokens isn't enough headroom (almost certainly isn't), do ONE of: (a) trim 50–100 tokens elsewhere (look at the existing redundant "see also" cross-references); (b) escalate to PO for a budget bump to 7000 (precedent: 17.10 bumped 5500 → 6500 for a real new surface). Don't ship over budget silently.
- **`dualStateOnly: true` on `list_workspace_documents` (AC 6)** — useful capability for the agent: when the user asks "Vilka policyer pågår just nu revisions på?", the agent can answer with one tool call instead of enumerating all and filtering in reasoning. Cheap to add; no dedicated tool needed.
- **17.16 + 17.17 dev-sequence dependency (AC 11)** — please don't enter dev on 17.18 until both prereq stories have hit Done. The integration tests in this story rely on the dual-state being reachable end-to-end, which requires 17.16's `createDraftFromApproved` refactor + 17.17's UI (for manual verification). If you must pre-build, do Task 1 (indexing) and Task 2 (search) — they can be unit-tested with mocked Prisma without 17.16's full migration applied — but defer integration tests and the adversarial DEC-2 test until 17.16/17.17 are Done.
- **17.11c is a separate follow-up.** It introduces the agent's auto-branch behavior (when `add_document_section` / `update_document` targets an APPROVED doc with no draft pending, transparently create the draft). 17.18 deliberately does NOT do this — it only refactors READ tools. Keep the write-tool behavior unchanged (refuse-with-guidance on APPROVED-no-draft) in 17.18.

## Risk and Compatibility Check

**Primary Risk:** The DEC-2 contract (agent never cites a DRAFT as canonical) holds under Model A's simpler model because the citation tier mapping was 1:1 with the doc's `status`. Under Model B's dual-state model, the agent has both content snapshots available simultaneously — and the LLM's natural tendency to summarize-and-blend could conflate them in response text even if the chunks are tagged correctly. A weak adversarial test would let this through.

**Mitigation:**
- The adversarial DEC-2 hold test (Task 7) is explicitly the highest-priority test in this story. Mirror 17.10b's adversarial test pattern (real agent invocation, real seeded dual-state doc, prompt designed to tempt conflation).
- System prompt update (AC 7) is explicit about framing — the agent must say *"Ert godkända X kräver A. Ett pågående utkast föreslår B."*, NOT *"X kräver A och B."*. Concrete example required in the prompt to make the pattern automatic.
- If the adversarial test fails in dev: do NOT ship 17.18. Iterate on prompt engineering + the citation grammar until the test passes. The compliance integrity of `[Källa:]` is the load-bearing trust signal of the whole agent product; we don't ship a regression on it.

**Rollback:**
- All read-tool and indexing changes; rollback is straightforward (revert the tools + the indexing function). Already-indexed dual-state chunks would stay in place harmlessly; under the rolled-back tools, dual-state docs would return one tier's chunks (whichever happens to win the deduplication tie-break) — which is what 17.10b's behavior is, so it's a known prior state.
- System prompt rollback: revert the diff. Token budget returns to its prior figure.
- Adversarial test failure post-deploy: feature-flag the dual-state routing (`dual_version_citations_v1`) and fall back to single-tier routing while iterating.

**Compatibility Verification:**
- [ ] No breaking changes to existing tool APIs (return shapes are additive; backward-compat top-level fields preserved on `get_workspace_document`).
- [ ] Database changes — none in this story (the chunk metadata `tier` field is added to the existing JSON `metadata` blob; no schema migration).
- [ ] UI changes — none in this story (Story 17.17 owns UX; rehype citation pill is the only render-adjacent change and it's a render-time DOM attribute, not a layout change).
- [ ] Performance impact: search adds a dedup pass at the tool layer (cheap); indexing duplicates content for dual-state docs (modest storage increase, bounded by the number of dual-state docs at any given time — typically very few).
