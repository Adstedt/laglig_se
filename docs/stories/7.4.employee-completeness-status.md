# Story 7.4: Completeness / "Ej komplett" status

## Status

Done

## Story

**As an** HR manager,
**I want** incomplete records flagged,
**so that** I know which employees lack LAS-critical data.

## Acceptance Criteria

1. An employee is **"Ej komplett"** if any LAS-critical field is missing — `personnummer`, `employment_date` (anställningsdatum), `employment_form` (anställningsform), `personel_type` (personaltyp) — **or** no kollektivavtal is assigned while `CompanyProfile.has_collective_agreement = true`.
2. An **"Ej kompletta"** filter tab in the register surfaces exactly these employees (joins the existing Alla/Aktiva/Inaktiva `WorkspaceViewTabs` from 7.2).
3. Per-record reasons listed on the Personalkort compliance sidebar (replacing 7.3's placeholder slot), e.g. "Saknar anställningsform", "Inget kollektivavtal tilldelat".
4. Register header shows a summary count via `PageHeader`'s `stats` prop (e.g. **"8/11 kompletta"** — the slot 7.2 deliberately left empty for this story).
5. Group headers show a per-group completeness rollup (e.g. "Lager 4/6 kompletta") — the epic marks this "may"; include it, it's cheap once the rule exists.
6. The table's Status column shows an amber **"Ej komplett"** badge alongside Aktiv/Inaktiv when a record is incomplete (Fortnox mirror; incomplete ≠ inactive).

### The rule is COMPUTED, never persisted

No `is_complete` column, no migration. Completeness is a pure function over the already-loaded serialized rows (`EmployeeRow` — 7.2's `toEmployeeRow` output; Decimals already numbers, relations already `{id,name}|null`). This keeps the rule free to evolve (7.5+ adds agreement statuses; Fortnox sync adds more fields) without data migrations. All surfacing (tab/stats/badge/rollup/sidebar) derives from the same single function — **one source of truth, zero drift**.

Read-only use here — do not touch the write paths.

### Traps
