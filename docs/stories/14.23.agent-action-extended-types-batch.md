## Story

**As a** user receiving multiple related agent proposals in one turn,
**I want** them consolidated into a single card with per-action edit + "Approve all",
**so that** I can review and approve related actions as a coherent unit without clicking through three separate cards.

## Context

Story 14.22 shipped the inline approval card for a single action type (`CREATE_TASK`) behind the `FEATURE_INLINE_AGENT_APPROVALS` flag. This story:

1. Adds four additional approval types — link task to document, link document to task, add obligation (kravpunkt), assign task. Each is a new agent tool + a new renderer plugged into the existing `AgentActionCard` dispatch.
2. Introduces a **multi-action batch layout**: when the agent emits ≥2 pending actions from a single assistant message, they render as one consolidated card with per-row edit and an "Approve all" affordance.
3. Removes the legacy sidebar write-preview flow: `write-preview-task.tsx`, `write-preview-note.tsx`, `write-preview-status.tsx` are deleted, and the `FEATURE_INLINE_AGENT_APPROVALS` flag is retired. Inline becomes the only approval path.

> **Note:** the "rich CREATE_TASK proposal" (Status / Ansvarig / Förfallodatum / kravlänk fields mirroring the task modal) was briefly folded in here, then **split out to [Story 14.32](14.32.rich-task-proposal-fields.md)** on 2026-05-21 per the PO sizing recommendation — it's thematically orthogonal and depends only on 14.22.

Rollout is staged: 14.22 ran both paths under a flag to validate parity; 14.23 commits to inline once 14.22 has been in production for ≥ 1 release cycle.

> **Carryover from 14.22 QA (SEC-001) — PO decision for this story:** approve/reject (and the dispatched domain writes) are gated on workspace-membership + ownership only, with no *role* permission check. Before 14.23 widens the write-action surface (link / obligation / assign), decide whether these should require an explicit scope (e.g. `tasks:edit`) so a viewer-role member can't approve a write. See `docs/qa/gates/14.22-agent-action-card-foundation.yml` (SEC-001).

## Acceptance Criteria

### New Approval Types

1. `PendingAgentActionType` enum is extended with: `LINK_TASK_TO_DOCUMENT`, `LINK_DOCUMENT_TO_TASK`, `ADD_OBLIGATION`, `ASSIGN_TASK`. Migration is additive (enum value addition only).
2. Four new agent tools are added under `lib/agent/tools/`:
   - `link-task-to-document.ts` — params: `taskId`, `documentId`. On `execute: false` creates a `PendingAgentAction` of type `LINK_TASK_TO_DOCUMENT`.
   - `link-document-to-task.ts` — params: `documentId`, `taskId`. Same pattern.
   - `add-obligation.ts` — params: `lawListItemId`, `text`, `bevisRequired?: boolean`. Creates a `LawListItemRequirement`.
   - `assign-task.ts` — params: `taskId`, `userId`. Sets `Task.assignee_id`.
3. Each new tool follows the `wrapWriteToolResponse` + pending-row-creation pattern established in 14.22. The `params` JSON snapshot is the editable source of truth.
4. Each tool's `execute: true` branch is removed (or no-ops with an error) — the inline approval path is now the only way to finalize these actions. Rationale: legacy dual-path was only needed during 14.22 rollout.

### Per-Type Renderers

5. Four new renderer components under `components/features/ai-chat/agent-action-renderers/`:
   - `link-task-to-document-renderer.tsx`
   - `link-document-to-task-renderer.tsx`
   - `add-obligation-renderer.tsx`
   - `assign-task-renderer.tsx`
6. Each renderer reuses existing shared components where possible:
   - `link-task-to-document` renderer reuses `DocumentPickerModal` (`components/features/documents/document-picker-modal.tsx`) to pick/swap the document.
   - `link-document-to-task` renderer reuses `TaskPickerDialog` (`components/features/documents/task-picker-dialog.tsx`) to pick/swap the task.
   - Obligation renderer mirrors the kravpunkt inline editor from Story 17.16 (`kravpunkter-checklist.tsx`: text input + "bevis krävs" switch).
   - Assign renderer mirrors the inline assignee `<Select>` pattern in `create-task-modal.tsx` (~L387-408) over the workspace `members` list — no extractable picker component exists; replicate the inline pattern.
7. Edits on any renderer debounce 500ms and call `updatePendingActionParams` (shared server action from 14.22).
8. `APPROVED` state for each type renders a read-only summary with a link to the affected entity (task, document, law list item).
9. Swedish copy throughout; preview strings match the existing vocabulary for each domain.

### Server-Action Dispatch Updates

10. `approvePendingAction` dispatch table (from 14.22) is extended with branches for each new `action_type`. **Exact targets pinned via codebase lookup:**
    - `LINK_TASK_TO_DOCUMENT` → calls `linkDocumentToTask({ documentId, taskId })` at [`app/actions/documents.ts:1089`](../../app/actions/documents.ts).
    - `LINK_DOCUMENT_TO_TASK` → calls the same `linkDocumentToTask` (the relationship is symmetric at the DB level — the `WorkspaceDocumentTaskLink` row is identical; only the UI framing differs based on which entity the user navigated from).
    - `ADD_OBLIGATION` → calls `createRequirement({ lawListItemId, text, bevisRequired })` at [`app/actions/law-list-item-requirements.ts:161`](../../app/actions/law-list-item-requirements.ts) (Story 17.16).
    - `ASSIGN_TASK` → calls `updateTasksBulk([taskId], { assigneeId: userId })` at [`app/actions/tasks.ts:718`](../../app/actions/tasks.ts). No dedicated single-task update action exists; calling bulk with a 1-element array is the supported path today. A dedicated `assignTask` helper is explicitly out-of-scope here; create it in a follow-up if multi-field task updates become common.
10a. **Task↔document link dual-type rationale.** `LINK_TASK_TO_DOCUMENT` and `LINK_DOCUMENT_TO_TASK` dispatch to the same server action but remain distinct `action_type` values so the renderer can present the direction-appropriate preview ("Koppla uppgift → dokument" vs "Koppla dokument → uppgift"). Swapping directions post-approval is not supported.
11. `result_ref` shape per type:
    - Link: `{ linkId }` (or the relevant link row id)
    - Obligation: `{ requirementId }`
    - Assign: `{ taskId, assigneeId }`
12. On dispatch failure for any type, the row stays `PENDING` and the server action returns `{ success: false, error }` — same pattern as 14.22.
13. `revalidatePath` targets per type:
    - Task↔document links: `/tasks`, `/dokument`
    - Add obligation: `/laglistor`
    - Assign: `/tasks`, `/dashboard` (if the dashboard surfaces assignments)

### Multi-Action Batch Card

14. New component `components/features/ai-chat/agent-action-batch-card.tsx` — renders 2+ `PendingAgentAction` rows that share the same `chat_message_id` as a single consolidated card.
15. The batch card SWR key is `pending-actions:by-message:${chatMessageId}` — fetcher returns an ordered list of pending actions for that message. Mutations revalidate both this key and the per-action `pending-action:${id}` keys.
16. Layout: one header ("Föreslagna åtgärder — N stycken"), one `AgentActionRenderer` per row using each row's existing per-type renderer in a compact variant (see AC 17), and a sticky footer with "Godkänn alla" (primary) + "Avvisa alla" (secondary ghost).
17. Each per-type renderer gains a compact variant (prop `compact?: boolean`). In compact mode: collapsed by default, expandable to show editable fields; summary line shows the action-type badge + a 1-line preview. Per-row approve/reject controls remain.
18. "Godkänn alla" iterates over all `PENDING` rows and calls `approvePendingAction` **sequentially in array order** (not parallel — see AC 18a). Partial approval (e.g., 2/3 succeed) is allowed — the card shows per-row status and a summary ("2 av 3 godkända"). Failed rows stay `PENDING` with their error message.
18a. **Order invariance and per-row independence.** Each `approvePendingAction` call is independent — no row's approval semantically depends on another. However, sequencing matters for error isolation: if the agent proposes "create task + link task to doc" in the same message, the task must exist before the link can succeed. For v1, the agent's tool-call order IS the approval order (rows are ordered by `created_at` ascending), and the agent is responsible for proposing actions in dispatchable order. If a row's dispatch fails because a prerequisite row in the same batch was rejected/errored, the failure error message must be Swedish-readable (e.g., "Uppgiften behöver skapas först innan koppling kan göras"). Cross-row compensation (roll back row N on row N+1 failure) is explicitly out of scope.
19. "Avvisa alla" calls `rejectPendingAction` for all `PENDING` rows via `Promise.all`. No confirmation modal — reject is non-destructive (rows remain visible in `REJECTED` state).
20. Single-action case (only 1 pending action for the message): renders as the existing single `AgentActionCard` — no batch layout, no "Approve all" affordance.

### `chat-message.tsx` Routing

21. The render branch from 14.22 is updated to support the batch case. When a `tool-*` part's output carries `pendingActionId`, look up the row's `chat_message_id` and check how many siblings share it:
    - 1 row: render `<AgentActionCard pendingActionId={id} />` (unchanged from 14.22).
    - 2+ rows: render `<AgentActionBatchCard chatMessageId={messageId} />` **once** per message, after the last tool part in the batch. Subsequent tool parts that resolve to pending actions in the same batch contribute to the same card and do NOT render their own standalone `AgentActionCard`.
22. Implementation: the parts-to-render logic (see 14.18's `groupCompletedToolParts`) gets a new pass that groups pending-action-carrying tool parts by their shared `chat_message_id` (fetched once via SWR from the batch endpoint). Parts without pending actions flow through the existing grouping logic unchanged.
