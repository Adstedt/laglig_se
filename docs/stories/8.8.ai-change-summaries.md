# Story 8.8: AI Amendment Summaries for Notifications

## Status

Ready for Review

## Story

**As a** user tracking laws that have been amended,
**I want** each amendment notification to include a brief, accurate AI-generated summary of what changed and why it matters,
**so that** I can quickly understand the significance of the change from my email without opening the app.

## Context & Dependencies

**Supersedes:** v2 draft that proposed reusing the Story 12.3 summering/kommentar pipeline. That pipeline was designed for whole-law descriptions ("what does this law regulate?"), not amendment change summaries ("what changed and why does it matter?"). This version uses a purpose-built amendment prompt.

**Builds on:**

- `discover-sfs-amendments` cron (10:00-13:00 UTC) — discovers amendments, creates `AmendmentDocument` + `LegalDocument` (content_type=SFS_AMENDMENT)
- `AmendmentDocument.parse_status = COMPLETED` — indicates amendment is fully parsed with markdown content
- `ChangeEvent.ai_summary` — target field, already rendered in `emails/amendment-digest.tsx` (line 76) with fallback text
- `notify-amendment-changes` cron (07:00 UTC) — sends digest emails, already reads `ai_summary`
- Story 14.14 — base laws now have `json_content` with per-§ structure + `summary` (from Story 12.3)

**Relationship to Story 8.4:**

Story 8.4 (`notify-amendment-changes`) already renders `ChangeEvent.ai_summary` in the email template. Currently this field is null for most amendments, falling back to "En ändring har upptäckts i denna lag." This story ensures meaningful summaries are pre-generated before the email cron runs.

**Future: Tier 2 deep-dive (separate story)**

A future story could enrich summaries further by fetching the proposition (förarbete) referenced in the amendment footnotes via the Riksdagen API, extracting the relevant section, and producing a deeper "why" analysis. This story explicitly does NOT do that — it generates summaries purely from the amendment text.

## Acceptance Criteria

### Background Enrichment Cron

1. New cron endpoint at `/api/cron/enrich-amendment-summaries` runs daily at 06:30 UTC (after cleanup-workspaces at 06:00, before notify-amendment-changes at 07:00)
2. `maxDuration = 300` with 30s timeout buffer
3. Finds `ChangeEvent` records where `ai_summary IS NULL` and the linked `AmendmentDocument` has `parse_status = COMPLETED`
4. Batch limit: 30 amendments per run (timeout protection)
5. Instrumented with `startJobRun`/`completeJobRun`/`failJobRun`, added to `JOB_REGISTRY` and `vercel.json`

### Summary Generation

6. For each un-enriched amendment, sends the `AmendmentDocument.markdown_content` (the clean, unlinkified source — NOT the amendment LegalDocument's linkified version) to Claude Sonnet with an amendment-specific prompt (NOT the 12.3 whole-law prompt)
7. The prompt produces 2-3 neutral sentences (~40-60 words) covering:
   - What parts of the law changed (which §§, what kind of change)
   - The practical implication (tighter requirements, new obligations, deadlines)
   - Effective date (if available in the amendment text)
8. Tone: neutral, factual, informative — no "Vi ska" (that's for the on-site personalized assessment)
9. Includes base law title and existing summering (if available) as context for the LLM to reference

### Storage

10. Stores the generated summary on `ChangeEvent.ai_summary` and sets `ChangeEvent.ai_summary_generated_at` to current timestamp (prevents re-processing, enables ops tracking)
11. Also stores on the amendment's `LegalDocument.summary` field for agent context and the Changes Tab

### Reliability

12. If LLM generation fails or times out for a specific amendment, skip it and continue (will retry next cron run; 8.4 falls back to static text "En ändring har upptäckts i denna lag.")
13. Cron returns JSON stats: `{ enriched, skipped, failed, alreadyHadSummary, duration }`
14. Stats included in daily ops digest (new row in chunk health or ingestion section)

## Dev Notes

### Two-Tier UX: Email Summary → Agent Deep Dive

This story delivers **Tier 1** — a brief, neutral summary in notification emails. The summary is the hook that tells the user "something changed, here's the gist." It is NOT personalized to the user's industry or workspace.

**Tier 2** happens in-app when the user clicks through from the email into the agent-driven change assessment workflow (Story 14.10). The agent has:
- This summary as starting context
- The full amendment markdown + section changes
- The base law with `json_content` (per-§ structure from Story 14.14)
- The user's workspace context (tracked laws, industry, role)

The agent then provides the personalized "Vi ska..." compliance assessment tailored to the user's situation. Two tiers, two moments, one pipeline.

### Why NOT Reuse the 12.3 Prompt

The Story 12.3 `buildSystemPrompt()` generates:
- **Summering**: "What does this law regulate?" (whole-law scope, neutral)
- **Kommentar**: "What must we do?" ("Vi ska..." compliance voice)

For amendments, users need:
- **"What changed?"** — which §§, what kind of change, effective date
- **"Why might this matter?"** — practical implication, tighter/looser requirements

These are fundamentally different questions. A purpose-built prompt will produce more accurate, concise results than trying to adapt the whole-law prompt.

### Input Data Available Per Amendment

| Source | What It Contains | Use In Prompt |
|---|---|---|
| `AmendmentDocument.markdown_content` | Full amendment text including prop references, footnotes, new § wording | Primary input — the ground truth |
| Base law `LegalDocument.title` | e.g., "Arbetsmiljölag (1977:1160)" | Context — "this law regulates..." |
| Base law `LegalDocument.summary` | Existing 12.3-generated summering of the base law | Context — helps LLM frame the change |
| `SectionChange[]` | Which §§ changed, change types (AMENDED/NEW/REPEALED), new text | Structure — tells LLM scope of change |
| `AmendmentDocument.effective_date` | When the change takes effect | Include in summary |

### No Conflict with Story 12.3

Story 12.3's content generation pipeline targets `SFS_LAW` and `AGENCY_REGULATION` content types only. This story targets `SFS_AMENDMENT` content type. They write to the same `LegalDocument.summary` field but on different document records, so there is no overwrite risk.

### What We Do NOT Use (Intentionally)

- `SectionChange.old_text` — always null (never populated by the pipeline)
- `DocumentVersion` diffing — adds complexity for marginal gain at this tier
- Web search / proposition fetching — reserved for future Tier 2 story
- Hallucination checker — the amendment summary is short and grounded in the amendment text; the checker adds latency and the 12.3 checker was designed for longer content. If quality issues emerge, add it later.

### Prompt Design (Reference)

```
Du får en ändringsförfattning (SFS) och kontext om baslagen den ändrar.

Skriv 2-3 meningar (max 60 ord) som beskriver:
1. Vilka delar av lagen som ändras och vad ändringen innebär i sak
2. Eventuell praktisk konsekvens (skärpta krav, nya skyldigheter, ändrade gränsvärden)
3. Ikraftträdandedatum om det framgår

Ton: saklig, neutral, informativ. Skriv för en person som redan följer lagen
och snabbt vill förstå vad som ändrats.

Skriv INTE "Vi ska" eller annan skyldighetsformulering.
Svara ENBART med sammanfattningen, ingen JSON, ingen rubrik.
```

Use `max_tokens: 256` — sufficient for ~60 words and prevents runaway output.

### Timing Budget (5 min total)

- Query un-enriched amendments: <5s
- Summary generation: ~3-5s per amendment (Sonnet, short prompt) × max 30 = ~150s
- DB writes: <5s
- Buffer: 30s
- Total: well within 300s

### Cost Estimate

- ~500 input tokens per amendment (markdown excerpt + context)
- ~100 output tokens (2-3 sentences)
- At Sonnet rates: ~$0.002/amendment
- Daily cost for ~5-10 amendments: < $0.02
