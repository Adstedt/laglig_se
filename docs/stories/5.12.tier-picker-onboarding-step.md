# Story 5.12: Tier-Picker Step in Onboarding Wizard

## Status

Ready for Review

## Story

**As a** new workspace owner signing up for Laglig.se,
**I want** to choose Solo / Team / Enterprise during onboarding before completing workspace creation,
**so that** my 14-day trial gives me the limits and features of the tier I'm actually evaluating, and the platform's enforcement (seats, storage, AI tokens) matches my intent from day one.

2. **Step renders three tiles** (Solo / Team / Enterprise) using a new shared `<TierCard>` component extracted from `components/features/landing/pricing-section.tsx`. Single source of truth — no duplicated price/feature copy across landing + onboarding.
3. **Tile copy is calibrated to v1 limits** and derived from `lib/usage/limits.ts` + a new `lib/billing/tier-display.ts` display layer. Solo: **499 SEK/mån** / 1 användare / 3M tokens ("≈ 100 AI-frågor") / 1 GB. Team: **1 299 SEK/mån** / 3 användare (+300 SEK/extra plats) / 9M tokens ("≈ 300 AI-frågor") / 5 GB. Enterprise: anpassad / obegränsat. **Monthly pricing only** — Stripe yearly Prices aren't wired yet (see AC 6). Marketing copy + enforcement limits cannot drift because both consume the same module.

   - `employeeCount > 20` → Team with stronger Enterprise hint visible alongside Team tile
   - Reasoning sentence rendered under the badge, in Swedish (e.g. *"Du har 8 anställda och kollektivavtal — Team inkluderar HR-modulen"*).
5. **`?plan=solo|team|enterprise` from marketing CTAs persists** through signup → onboarding via the existing `OnboardingStore` localStorage bridge (mirrors how `?org=` / `?flags=` / `?summary=` are handled at `app/(auth)/signup/_signup-form.tsx:45-58`). On wizard mount, `pickedTier` is pre-selected if present. A small note appears: *"Ditt val från startsidan: {Tier}. Vill du ändra?"*. The recommendation badge is NOT suppressed when it disagrees with the picked tier — both signals coexist.
6. **No yearly/monthly toggle in v1** — Stripe only has monthly recurring Prices wired today (Story 5.4 created `STRIPE_SOLO_PRICE_ID` / `STRIPE_TEAM_PRICE_ID` as monthly EUR/SEK Prices; no annual Price IDs exist). Both `<TierCard>` and the marketing `pricing-section.tsx` render monthly pricing only. The current marketing surface's `isYearly` toggle + `yearlyPrice` field are **removed** — they were marketing-only artifacts that implied a yearly conversion path that doesn't exist on the Stripe side. When yearly Prices are added (separate future story), restore the toggle in `<TierCard>` and reintroduce `yearlyPriceSek` in `lib/billing/tier-display.ts`.

9. **`createWorkspace` server action** accepts `pickedTier` from FormData (validated against `'SOLO' | 'TEAM' | 'ENTERPRISE'` — `TRIAL` not accepted as picked tier). All writes happen inside the existing `prisma.$transaction` (`workspace.ts:171`):
   - `subscription_tier: 'TRIAL'` (unchanged)
   - **For `pickedTier ∈ {'SOLO', 'TEAM'}`:** `trial_picked_tier: pickedTier` — direct mapping. `enterprise_inquiry_at` not set.
   - **For `pickedTier === 'ENTERPRISE'`:** `trial_picked_tier: 'TEAM'` (caps trial limits at Team to bound COGS during the wait-for-sales window — see AC 10) + `enterprise_inquiry_at: new Date()` (sales follow-up signal queryable by sales tooling).
   - If `pickedTier` is missing (legacy/safety path), default to `'SOLO'` with `console.warn` — should never happen in the wizard, but tolerated.
10. **Enterprise picked — Team-trial + inquiry signal pattern:**
    - Workspace created with `subscription_tier='TRIAL'`, `trial_picked_tier='TEAM'`, `enterprise_inquiry_at = NOW()` (per AC 9). During the 14-day trial the workspace operates under **Team-tier limits** (3 seats, 5 GB, 9M tokens, 80% soft warn, 200% hard cap) — preserves onboarding momentum without exposing unbounded Enterprise-trial COGS.
    - **ConfirmStep copy when Enterprise picked:** *"Vi har bett om kontakt om Enterprise — vi hör av oss inom 24 timmar."* + *"Under tiden får du Team-funktionalitet i 14 dagar så du kan utvärdera plattformen medan vi pratar."*

    - Sales-rep follow-up workflow (out of band, MVP-manual): call → qualify → create Enterprise Stripe price → manually flip `subscription_tier='ENTERPRISE'` + clear `trial_picked_tier` + clear `enterprise_inquiry_at`. Operator-applied; not part of this story.
11. **Invite-bound signups skip the tier-picker step entirely.** The wizard's existing `showWizard` gate (`OnboardingWizard:44-46`) already handles the `invitations.length > 0` branch by rendering `<PendingInvitations>` instead of the wizard — no special-case logic needed for invite skip; the new step rides inside the wizard branch only.

Tile content updates: 399 → 499 SEK Solo · 899 → 1 299 SEK Team · feature copy "100 frågor/mån" → "≈ 100 AI-frågor (3M tokens)" · "Obegränsad AI-chat" → "≈ 300 AI-frågor (9M tokens)" · "Upp till 10 användare" → "3 användare ingår, +300 SEK/extra plats". **Yearly toggle UI removed** (per AC 6) — the `isYearly` state and the toggle button group are deleted; `yearlyPrice` field on the section's tier-data array goes away. Section subhead copy adjusts accordingly (drop any "Spara X% med årsabonnemang" anchors if present).

   - 25 employees → TEAM with `enterpriseHint=true`

16. **Enterprise inquiry sales notification:** when `enterprise_inquiry_at` is set during workspace creation, the action sends a Resend email via the existing `lib/email/email-service.ts` to the address in `env.SALES_NOTIFICATION_EMAIL` (new env var; default `sales@laglig.se`). `from: 'notifications'` to match Story 5.4's transactional pattern. ## Dev Notes

### Why this story exists

The 5.5 parent spec (`docs/stories/completed/5.5.usage-limits-per-tier.md:66`) anticipated this story: the trial-picked-tier mechanism was designed assuming a signup-flow tier-picker would write `trial_picked_tier`. 5.5a/b/c shipped enforcement code that reads the column, but no UI surface writes it. Today every new workspace falls back to Solo limits (the lookup default) regardless of intent. Story 5.12 closes that loop AND fixes the marketing-vs-enforcement copy drift in `pricing-section.tsx`.

### Step placement decision

After Verksamhet, before Bekräfta. Rationale: by step 3, we know `employeeCount`, `sniCode`, and `activityFlags` — enough to compute a data-informed recommendation badge, which is the differentiator vs static landing cards. Earlier placements weaken or eliminate the recommendation. The "tell us about you → tell us what you do → here's what fits → confirm" rhythm matches B2B SaaS consultative-onboarding best practice and respects the no-card-trial commitment level (low friction, high information).

### Pricing source of truth

Project memory `project_tier_pricing_v1.md` (decided 2026-04-27) defines SEK pricing 499 / 1 299 + 300 SEK add-on seat / Enterprise custom. The 5.5 parent spec `lib/usage/limits.ts` `TIER_LIMITS` table holds the *limits* derived from these prices. New `lib/billing/tier-display.ts` adds the *display copy* (price strings, feature bullets) layered on top. Limits and display drift is impossible because both surfaces consume the same module.

### Enterprise inquiry pattern (AC 9, 10, 16)

Enterprise picks during onboarding **decouple two signals**:

1. **`trial_picked_tier`** = the tier whose LIMITS apply during the trial. For Enterprise picks → set to `'TEAM'`. This bounds COGS during the wait-for-sales window (Team's 200% hard cap on AI tokens prevents runaway spend if the user is heavy AND sales doesn't close).
2. **`enterprise_inquiry_at`** = "this user told us they want Enterprise" timestamp. New nullable column on `Workspace`. Sales tooling polls this column to find leads.

**Why this design:** `TIER_LIMITS.ENTERPRISE` returns `null` for all numeric limits (unlimited). A 14-day trial under `trial_picked_tier='ENTERPRISE'` would expose unbounded LLM COGS to anyone who picks Enterprise — including aspirational/accidental picks. By trial-mapping Enterprise → Team, we keep the user's onboarding momentum intact (their company info isn't lost, they can use the product immediately) while giving sales a high-quality lead signal AND bounding cost. Sales rep follow-up converts the workspace to actual Enterprise pricing manually (out of band).

No index needed — sales queries it ad-hoc (low volume; <5% of signups expected to be Enterprise picks at MVP scale).

**Email template** mirrors the `emails/payment-failed.tsx` Story 5.4 pattern (React Email, `from: 'notifications'`, Swedish copy). Send fires AFTER `prisma.$transaction` commits — a failed email never rolls back a successful workspace. Failure path is `console.error('[ENTERPRISE_INQUIRY_EMAIL_FAIL]', err)` and continue (mirrors Story 14.27's `[CHAT_USAGE_EVENT_WRITE_FAIL]` fail-safe pattern).

**Trial-end behavior for Enterprise inquiries:** Story 4.10 (Draft) handles trial expiration uniformly across tier picks — at day 14, the workspace is paused regardless of whether `enterprise_inquiry_at` was set. If Enterprise inquiry is still un-resolved by sales at day 14, the user gets the same expiration UX as a Team-picked trial. Sales rep can clear `paused_at` + flip `subscription_tier='ENTERPRISE'` manually if they close the deal late. Not in scope for 5.12.

**Operator handoff checklist** (the dev agent puts this in the PR description / completion notes):

Until the migration is applied, `pnpm prisma generate` will produce a Prisma Client that knows about a column the live DB doesn't have — integration tests that hit the real DB will fail with `column "workspaces.enterprise_inquiry_at" does not exist`.
