# Story 6.20: Task Groups — Organize Tasks by Area

## Status

Draft

## Story

**As a** compliance manager with many tasks across different regulatory areas,
**I want** to organize tasks into named groups (e.g., "GDPR", "Arbetsmiljo", "Brandsäkerhet") with collapsible accordion sections in the list view,
**so that** I can visually organize, navigate, and manage tasks by compliance area, matching the grouping experience I already have in law lists.

## Acceptance Criteria

**Database & API:**

1. A new `TaskGroup` model exists in the database, scoped per workspace, with fields: id, workspace_id, name, position (float for insertion ordering), created_at
2. Unique constraint on `(workspace_id, name)` — no duplicate group names per workspace
3. `Task` model gains an optional `group_id` foreign key to `TaskGroup` with `onDelete: SetNull` — deleting a group moves tasks to ungrouped
4. Server actions exist for full group CRUD: `createTaskGroup`, `updateTaskGroup`, `deleteTaskGroup`, `getTaskGroups`
5. Server actions exist for task-group assignment: `moveTaskToGroup(taskId, groupId)`, `bulkMoveTasksToGroup(taskIds, groupId)`, `reorderTaskGroups(groups)`
6. All server actions validate workspace access, use Zod schemas, and return typed `ActionResult`

**Group Manager Dialog:**

7. "Grupper" button appears in the task workspace toolbar (Zone D secondary action, matching law list's Grupper button)
8. Clicking "Grupper" opens a TaskGroupManager dialog for CRUD operations
9. Create: Input field with validation (max 50 chars, unique per workspace)
10. Edit: Inline editing with check/cancel buttons, Enter/Escape keyboard support
11. Delete: Confirmation dialog showing task count, warning that tasks move to "Ogrupperade"
12. Reorder: Up/down buttons per group with optimistic UI updates
13. Dialog text uses "uppgifter" (not "dokument") — localized for task context

**Grouped Accordion Table (Lista tab):**

14. When groups exist and no filters/search are active, the Lista tab renders a grouped accordion table
15. Each group displays as a collapsible section with header: chevron toggle, folder icon, group name (clickable to filter), task count badge
16. Collapsed groups show only the header row
17. Expanded groups show the task list table (from Story 6.19) with that group's tasks
18. "Ogrupperade" section appears at bottom for tasks without a group
19. Empty groups (0 tasks) display with "0 uppgifter" badge and empty state message
20. Fallback to flat table when: no groups exist, filters are active, search is active, or filtering by specific group

**Expand/Collapse Controls:**

21. Clicking chevron toggles individual group expand/collapse
22. "Visa alla" button expands all groups
23. "Dolj alla" button collapses all groups
24. Expand/collapse state persists in the task-list-store (from Story 6.19)
25. Expand/collapse controls appear between toolbar rows and the table content

**Drag-and-Drop Between Groups:**

26. Tasks can be dragged from one group's table to another group's header
27. Group header highlights with `border-primary bg-primary/5` when valid drop target during drag
28. Dropping on group header moves task to that group via `moveTaskToGroup` server action
29. Reordering within same group works via table row drag
30. Single outer DndContext wraps all sections (prevents nested DndContext issues)
31. Custom collision detection prioritizes group headers over row targets
32. Drag overlay shows task title during drag

**Group Filter:**

33. Clicking a group name in the accordion header activates group filter — shows only that group's tasks in flat table
34. Active group filter displayed as `GroupFilterChip` (reuse existing component from law list) in toolbar Zone B `activeFilters`
35. Group filter reflected in URL as `?group={groupId}` parameter
36. Clearing group filter returns to grouped accordion view

**Bulk Actions:**

37. Bulk selection (checkboxes) works within each expanded group section
38. BulkActionBar appears when tasks selected, showing combined count across groups
39. "Flytta till grupp" (Move to group) option added to bulk action bar dropdown
40. Selecting tasks in multiple groups shows combined BulkActionBar

**Task Group Assignment (Individual):**

41. Task group can be assigned/changed via inline dropdown in a "Grupp" column (optional, hidden by default in column settings)
42. Task modal includes group assignment selector in the right panel properties section
43. Create task modal includes optional group assignment dropdown

**Toolbar Integration:**

44. "Grupper" button renders in toolbar Row 1, Zone D as secondary action (between export-area and primary CTA)
45. When groups exist and grouped view is active, "Visa alla" / "Dolj alla" buttons appear in toolbar betweenRows area alongside item count
46. When group filter is active, GroupFilterChip appears in toolbar Row 2 as `activeFilters`

**Mobile Responsive:**

47. Group headers have 44px minimum touch targets
48. Expand/collapse buttons remain accessible on mobile
49. Table horizontally scrolls within each accordion section

### Reuse Strategy

**Directly Reusable (import as-is):**

- `GroupFilterChip` from `document-list/group-filter-chip.tsx` — fully generic, takes `groupName` + `onClear`

**Pattern Reuse (create task-specific versions following same architecture):**

- `TaskGroupManager` ← follows `GroupManager` pattern but calls task server actions and uses "uppgifter" text
- `GroupedTaskListTable` ← follows `GroupedDocumentListTable` pattern but typed to `TaskWithRelations` instead of `DocumentListItem`
- `TaskGroupTableSection` ← follows `GroupTableSection` pattern but wraps task ListTab table instead of DocumentListTable

**Why not extract generic versions?**
The grouped table components are deeply typed to their domain entities (`DocumentListItem` vs `TaskWithRelations`) with domain-specific update callbacks (`complianceStatus` vs `columnId`). Extracting a generic wrapper would require complex generics that add more complexity than the code duplication they prevent. The law list and task list will likely diverge further over time (compliance indicators vs priority/status indicators in headers). Following the same pattern with domain-specific implementations is the pragmatic choice.

### Database Schema Reference

**LawListGroup (pattern to mirror):**

```prisma
model LawListGroup {
  id          String   @id @default(uuid())
  law_list_id String
  name        String
  position    Float    @default(0)
  created_at  DateTime @default(now())

  law_list LawList       @relation(fields: [law_list_id], references: [id], onDelete: Cascade)
  items    LawListItem[]

  @@unique([law_list_id, name])
  @@index([law_list_id, position])
  @@map("law_list_groups")
}
```

**TaskGroup (new — workspace-scoped, not list-scoped):**

```prisma
model TaskGroup {
  id           String   @id @default(uuid())
  workspace_id String
  name         String
  position     Float    @default(0)
  created_at   DateTime @default(now())

  workspace Workspace @relation(fields: [workspace_id], references: [id], onDelete: Cascade)
  tasks     Task[]

  @@unique([workspace_id, name])
  @@index([workspace_id, position])
  @@map("task_groups")
}
```

**Key difference:** LawListGroup is scoped per `law_list_id` (a workspace can have multiple law lists, each with their own groups). TaskGroup is scoped per `workspace_id` (tasks are workspace-wide, not list-scoped). This means task groups are shared across all task views (Lista, Kanban, Calendar, etc.).

### Server Action Pattern (from document-list.ts)

Each group action follows this pattern:

```typescript
export async function createTaskGroup(
  input: CreateTaskGroupInput
): Promise<ActionResult<{ id: string }>> {
  try {
    const parsed = CreateTaskGroupSchema.safeParse(input)
    if (!parsed.success) {
      return { success: false, error: 'Ogiltiga indata' }
    }

    const { workspaceId } = await withWorkspace()

    // Check duplicate name
    const existing = await prisma.taskGroup.findUnique({
      where: {
        workspace_id_name: {
          workspace_id: workspaceId,
          name: parsed.data.name,
        },
      },
    })
    if (existing) {
      return { success: false, error: 'En grupp med det namnet finns redan' }
    }

    // Get next position
    const maxPosition = await prisma.taskGroup.aggregate({
      where: { workspace_id: workspaceId },
      _max: { position: true },
    })
    const nextPosition = (maxPosition._max.position ?? 0) + 1

    const group = await prisma.taskGroup.create({
      data: {
        workspace_id: workspaceId,
        name: parsed.data.name,
        position: nextPosition,
      },
    })

    revalidatePath('/tasks')
    return { success: true, data: { id: group.id } }
  } catch (error) {
    console.error('Error creating task group:', error)
    return { success: false, error: 'Kunde inte skapa grupp' }
  }
}
```

### Kanban Interaction

Task groups are separate from Kanban columns:

- **Kanban columns** = workflow status (Att gora, Pagaende, Klar)
- **Task groups** = organizational category (GDPR, Arbetsmiljo, Brandsäkerhet)

A task has BOTH a `column_id` (status) and a `group_id` (category). They are orthogonal. In the Kanban view, tasks could optionally show a group badge, but that is out of scope for this story.
