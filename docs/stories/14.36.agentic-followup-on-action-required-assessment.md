## Story

**As a** user who has just assessed a law change as requiring action,
**I want** the agent to propose the right follow-up(s) — a task, a drafted document, a new kravpunkt, etc. — based on my assessment and notes,
**so that** the action I flagged is actually operationalized and tracked, instead of evaporating into a saved status.

## Context

`Åtgärd krävs` (Action Required) literally means work is needed — but today the assessment flow ends at the saved record. The user identified action and nothing operationalizes it. This story makes the confirmed assessment a **trigger for an agentic follow-up turn**: the agent reasons over the full context (the change, impact level, status, the user's `anteckningar`, company context, the law) and **proposes the appropriate compliance action(s)** from its existing toolset — *whatever fits*:

- needs a new/updated routine → `draft_styrdokument` (Story 14.24)
- someone must do something → `create_task`
- the change adds a concrete requirement → `add_obligation` (kravpunkt)
- it changes our compliance posture → `update_compliance_status`
- assign / link → the assign/link tools

When more than one fits, they consolidate into the **batch card** ("Föreslagna åtgärder"). The user approves / edits / rejects. The assessment saves regardless — the follow-up is a suggestion, never a gate.

## Acceptance Criteria

### Trigger

1. Confirming an assessment with status **`ACTION_REQUIRED`** triggers an agentic follow-up turn (a system message bridges the deterministic save → an agent turn carrying the assessment context: change, impact, status, `anteckningar`, law, company). `DEFERRED` MAY trigger a dated reminder/task; `REVIEWED` / `NOT_APPLICABLE` do **not** trigger anything.
2. The assessment save is **never blocked** by the follow-up; "Klar" still closes the workflow whether or not the user acts on a proposal.

### Agentic proposal (full toolset, judicious)

3. The agent selects from its existing toolset the **minimal sufficient** action(s) for the change — `create_task`, `draft_styrdokument`, `add_obligation`, `update_compliance_status`, link/assign — and proposes them via the standard approval card(s); 2+ proposals render as the `AgentActionBatchCard`.
4. Proposals are **grounded in the `anteckningar`** + the assessment — if the user's note implies a specific action ("måste uppdatera brandskyddspolicyn"), the agent proposes *that* (a draft of that document), not a generic task.
5. The agent proposes **nothing** when the assessment/notes imply no concrete action is warranted (no spam). Reuse the 14.23 anti-re-proposal rules — never re-propose an action already in `<pending_agent_actions>`.
6. Pre-fill quality: impact level maps to task priority (e.g. `HIGH` → `HIGH`/`CRITICAL`); the proposed action(s) auto-link to the law list item (via the validated `contextLinks`, Story 14.24) and reference the change.

### Approval + non-regression

7. Approval/dispatch is the existing inline-card flow (gated by `tasks:edit`); nothing new in the dispatch layer.
8. The change-assessment workflow, `save_assessment`, and all existing agent flows are unchanged outside this trigger + the prompt steering.

## Dev Notes

### Behaviour-risk note (read first)

The hard part here is **agent behaviour**, not plumbing — the tools, cards, and batch all exist. "Be judicious, propose the minimal sufficient set, ground in the notes, never re-propose" is exactly the class of steering that produced re-proposal bugs in 14.23 (fixed prompt-side, verified live). **Budget for a smoke-test + QA loop on the prompt behaviour**, and remember `system-prompt.md` is read once at server startup (restart to test).

### Relevant source

- **`lib/agent/system-prompt.md`** — `assessment_workflow` block + the "Åtgärder" steering (already tells the agent to call write tools directly + not re-propose decided actions). Extend it for the follow-up trigger.
- **`lib/hooks/use-assessment-form.ts`** / **`components/features/changes/assessment-resolution.tsx`** — the confirm path; the trigger hangs off a successful `ACTION_REQUIRED` save.
- **Approval-card machinery** — `PendingAgentAction`, `createPendingActionRow`, `RENDERERS` + `ACTION_TYPE_LABELS` (`agent-action-card.tsx`), `AgentActionBatchCard`, `approvePendingAction` dispatch. All reused as-is.
- **`lib/compliance-audit/task-spawner.ts`** (Epic 21) — precedent + dedup reference.

### Technical Constraints

- **Suggestion, never a gate** — assessment saves independently; the follow-up is opt-out.
- **`tasks:edit` gating** — inherited from the existing approval/dispatch layer; no new write surface.
- **Trigger scope** — `ACTION_REQUIRED` (+ optionally `DEFERRED`). Not the no-action statuses.
- **Dedup** — coordinate with Epic 21 so a cycle-managed change isn't double-tracked.

### Testing

- Vitest + RTL for the trigger (status-gated). The agent's selection/judiciousness is not unit-testable → manual playbook + (if harness) an integration test: `ACTION_REQUIRED` + note → proposal of the right type; `REVIEWED` → nothing; multi-action → batch card.
