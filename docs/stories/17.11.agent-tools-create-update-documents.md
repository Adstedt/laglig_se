## Story

**As an** AI compliance agent,
**I want** to propose a section-level edit to an existing workspace document (subject to user approval),
**so that** I can help users keep a policy/procedure current without re-drafting the whole document — while never silently mutating an approved document.

## Story Context

**Existing System Integration:**

- **Create is already shipped (14.24).** Out of scope here. This story is **update-only**.
- **Approval pattern (14.22/14.23/14.24):** new edits flow through a `PendingAgentAction` (`action_type: UPDATE_DOCUMENT`) → approval-card renderer → `approvePendingAction` dispatch. **No `execute:true` path** (removed per 14.23).
- **Models:** `WorkspaceDocument` / `WorkspaceDocumentVersion` (Epic 17, `WorkspaceDocument*` prefix). New content saved as a new `WorkspaceDocumentVersion` with `source: 'TIPTAP'`.
- **Reuse:** `ActionRendererFrame` + `useDebouncedParamsChange` (14.23 "spine & whisper" card chrome, 4 states); `createPendingActionRow` helper; the `approvePendingAction` dispatch switch + its `tasks:edit` gate (SEC-001); the read-only **chat detail panel** canvas seam (`document-draft`-style `ChatDetailItem`, 14.24 AC 11a) for the before/after preview.
- **Content format:** agent emits the replacement section as **Tiptap JSON** (matches `draft_styrdokument`).

**Out of scope:** creating new documents (14.24's `draft_styrdokument`); whole-document rewrites; markdown conversion; the read/list tools (Story 17.10); the legal `Document`/`DocumentVersion` (SFS) models.

## Acceptance Criteria

### Schema

1. `PendingAgentActionType` extended with **`UPDATE_DOCUMENT`** (additive `ALTER TYPE ... ADD VALUE`; generate `--create-only` and hand the SQL to the user — never auto-apply). No new tables.
2. `params` JSON shape for `UPDATE_DOCUMENT`: `{ documentId, sectionHeading, newSectionContentJson (Tiptap), changeSummary }`.

### Tool — `update_document`

3. New tool `lib/agent/tools/update-document.ts` (`update_document`). Zod schema: `document_id` (string), `section_heading` (string — heading text to locate), `updated_content` (**Tiptap JSON** for the replacement section), `change_summary` (string). **No `execute` param** — the tool only ever proposes a pending row.
4. **Guards (at tool time):** target document is in the caller's workspace; status is **DRAFT or IN_REVIEW** (reject APPROVED/SUPERSEDED/ARCHIVED via `wrapToolError` — guide the user to branch a new editable version first via the existing **`createDraftFromApproved`** action (`app/actions/documents.ts:755`)); `section_heading` exists in the document. On any failure → `ToolError`, no pending row.
5. On success: create a `PendingAgentAction(UPDATE_DOCUMENT)` and return a `WriteToolResponse` whose preview shows the **section heading + a short before/after excerpt**.

### Renderer

6. New renderer `components/features/ai-chat/agent-action-renderers/update-document-renderer.tsx`, **built on `ActionRendererFrame`** (do not hand-roll chrome/states):
   - **PENDING:** document title + target section heading + a compact before→after excerpt; "Visa mer" opens the full read-only before/after in the **chat detail panel** (canvas seam, per 14.24 AC 11a) — not a modal. Controls: "Godkänn" / "Avvisa".
   - **APPROVED:** title + link to the document at its new version.
   - **REJECTED / EXPIRED:** standard muted variants.
7. Register the renderer in **both** exported maps in `agent-action-card.tsx` — `RENDERERS[UPDATE_DOCUMENT]` and `ACTION_TYPE_LABELS[UPDATE_DOCUMENT]` (14.23 multi-type registry).

### Approval dispatch

8. `approvePendingAction` gains an `UPDATE_DOCUMENT` branch (inherits the existing `tasks:edit` gate). Flow: re-read the current `content_json` → **re-assert the DRAFT/IN_REVIEW status guard** (state may be stale since proposal) → `updateSection()` (AC 9) to produce the **full** updated document → call the existing **`saveDocumentVersion(documentId, contentJson, changeSummary?, title?, contentHtml?)`** (`app/actions/documents.ts:537`), which appends a new `WorkspaceDocumentVersion` (`source: 'TIPTAP'`), bumps `current_version_id`/`current_version_number`, and logs activity — all in one `$transaction`. ⚠️ **`saveDocumentVersion` takes the WHOLE document `contentJson`, not a section** — pass the full `updateSection` output. It also expects `contentHtml`; generate it from the Tiptap JSON (e.g. `generateHTML(json, extensions)`) so `extracted_text` derives correctly. **Note:** `saveDocumentVersion` does NOT itself enforce the DRAFT/IN_REVIEW guard — the dispatch branch must (AC 11).
9. Pure section-replacement utility `lib/documents/update-document-section.ts` → `updateSection(contentJson, sectionHeading, newContentNodes)`: locate the heading (case-insensitive), replace from that heading to the next heading of **same-or-higher level** (preserving the heading, replacing the body), return the full updated doc. Heading-not-found → informative error.
10. `ActivityLog` on update uses `entity_type: '**workspace_document**'` (the live convention — **not** `'document'`, which is the SFS legal entity). `saveDocumentVersion` (AC 8) **already writes** a `document_version_saved` row, so an agent update is auditable for free. To distinguish *agent* edits from human ones, stamp agent authorship in the row's `new_value` (e.g. `{ ..., by: 'agent', pendingActionId }`) rather than minting a redundant action label. Only add a distinct `document_updated_by_agent` action label (in `lib/activity/action-constants.ts`) if product wants it surfaced separately in the feed.

### Security / non-regression

11. Workspace scoping and the DRAFT/IN_REVIEW guard are enforced **both** at tool time (AC 4) **and** at dispatch (AC 8) — never trust the proposal-time snapshot. A role lacking `tasks:edit` cannot approve (inherited gate).
12. `draft_styrdokument` and all existing agent tools/renderers are behaviourally unchanged (additive).

### Prompt

13. `lib/agent/system-prompt.md` updated (restart dev server to load): when to **update** an existing document (`update_document`) vs **draft a new one** (`draft_styrdokument`); provide the replacement section as Tiptap JSON; cannot edit Approved documents.
14. `TOOL_CONFIG` (`chat-message.tsx`) gains an `update_document` entry (`label`/`doneLabel`/`icon`, e.g. "Uppdaterar dokument" / "Föreslog ändring" / `FileEdit`).

## Dev Notes

- **Naming:** use `WorkspaceDocument*` everywhere; bare `Document`/`DocumentVersion` are the SFS *legal* models (collision). Version `source: 'TIPTAP'`. [Confirmed in-code by 14.24.]
- **Reuse 14.24's proven seams** rather than re-inventing: `ActionRendererFrame`/`useDebouncedParamsChange`, the `document-draft`-style read-only detail-panel preview (`chat-detail-context.tsx` + `chat-detail-sidebar.tsx`), `createPendingActionRow`, and the `approvePendingAction` gate. See `docs/stories/completed/14.24.agent-draft-document-approval.md` for the exact patterns + line refs.
- **Version-append action (pinned by PO validation v2.1):** **`saveDocumentVersion(documentId, contentJson, changeSummary?, title?, contentHtml?)`** at `app/actions/documents.ts:537`. Appends a new `WorkspaceDocumentVersion` (`source: 'TIPTAP'`), bumps `current_version_id` + `current_version_number`, and writes a `document_version_saved` `ActivityLog` row — all in one `$transaction`, workspace-scoped via `withWorkspace`. Takes the **whole** document `contentJson` (not a section) and expects `contentHtml` for `extracted_text` derivation. It does **not** enforce the document-status guard — the dispatch branch must (AC 8/11). `createDocument` (`documents.ts:54`) is new-doc only and is **not** used here.
- **Version `source` enum:** `WorkspaceDocumentVersionSource` (`schema.prisma:1997`, default `TIPTAP`). Agent edits use `TIPTAP` — there is no `'AGENT'` value; agent authorship is tracked via `created_by` + the activity `new_value` stamp (AC 10), mirroring 14.24.
- **Why approval-gated (not atomic):** document mutations go through `PendingAgentAction` since 14.23; an agent editing a live document must be reviewable. The `execute=false/true` two-phase pattern from the original 17.11 draft is deprecated.
