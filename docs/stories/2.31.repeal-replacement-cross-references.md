# Story 2.31: Repeal & Replacement Cross-References

## Status

Draft

## Story

**As a** compliance officer reviewing a repealed law,
**I want** to see which document replaced it and navigate directly to the replacement,
**so that** I can quickly find the current, applicable regulation without manual searching.

## Context

When a law or agency regulation is repealed, `createRepealEvent()` sets `last_change_ref` to the repealing document's SFS number (a string like `"SFS 2025:732"`). This captures *what* caused the repeal but doesn't create a navigable link between the two documents.

The `CrossReference` model and `ReferenceType` enum exist but have no repeal-specific types. Repeal relationships are only discoverable by reading the `last_change_ref` string on the repealed document — there's no reverse lookup ("what did this document replace?") and no chain traversal.

This gap becomes critical for agency regulations (myndighetsföreskrifter), where full document replacement is the norm: "AFS 2023:3 upphäver och ersätter AFS 2001:1". Unlike SFS amendments that modify sections, agency regulations replace entire documents wholesale.

### Existing System Integration

- **Integrates with:** `CrossReference` model, `change-detection.ts`, `ReferenceType` enum
- **Follows pattern:** Existing cross-reference creation (e.g., `AMENDS`, `CITES`)
- **Touch points:** `createRepealEvent()`, Prisma schema, document detail pages (future — display not in scope)

## Acceptance Criteria

### Schema Changes

1. `ReferenceType` enum extended with `REPEALS` and `REPLACED_BY` values
2. Migration is additive-only — no existing data modified
3. `prisma generate` succeeds and `tsc --noEmit` passes with no new errors

### Cross-Reference Creation on Repeal

4. When `createRepealEvent()` is called **and** the repealing document exists in the database, two `CrossReference` rows are created within the same transaction:
   - `source=repealing_doc, target=repealed_doc, type=REPEALS`
   - `source=repealed_doc, target=repealing_doc, type=REPLACED_BY`
5. The `context` field on both cross-references contains the repealing document's reference string (e.g., `"Upphävd genom SFS 2025:732"`)
6. When the repealing document does **not** exist in the database (common for SFS amendments that aren't separate docs), no cross-reference is created — existing behavior unchanged
7. `createRepealEvent()` remains idempotent: calling it twice for the same repeal does not create duplicate cross-references (use upsert or check-before-create)

### Query Support

8. Given a repealed document ID, querying `CrossReference` with `source_document_id + type=REPLACED_BY` returns the replacing document
9. Given a replacing document ID, querying `CrossReference` with `source_document_id + type=REPEALS` returns all documents it replaced
10. Replacement chains are traversable: if Doc A was replaced by Doc B, which was replaced by Doc C, following `REPLACED_BY` links from A leads to B then C

### Backward Compatibility

11. Existing `ChangeEvent` creation logic is unchanged — `createRepealEvent()` still creates the `ChangeEvent` and sets `last_change_type`/`last_change_ref` on `LegalDocument` exactly as before
12. No existing cross-references are affected
13. All existing tests continue to pass

### Document Number Matching

The `repealedBySfs` parameter is a string like `"SFS 2025:732"`. The `LegalDocument.document_number` field stores the canonical identifier (e.g., `"SFS 2025:732"` for SFS laws, `"AFS 2023:3"` for agency regulations). The lookup is a direct string match:

```typescript
const repealingDoc = await tx.legalDocument.findFirst({
  where: { document_number: repealedBySfs },
  select: { id: true },
})
```

### Idempotency Strategy

Use a unique constraint check or `findFirst` before creating. The `CrossReference` model doesn't currently have a unique constraint on `(source_document_id, target_document_id, reference_type)`, so either:
- Add a composite unique index (preferred — prevents duplicates at DB level), or
- Use `findFirst` + conditional create (simpler, no migration needed)

Recommendation: add `@@unique([source_document_id, target_document_id, reference_type])` to `CrossReference` in the same migration. This also protects other reference types from duplicates.

### Agency Regulation Relevance

For SFS laws, the repealing document is often an amendment (SFS ändringsförfattning) that may not exist as a separate `LegalDocument`. Cross-references will be created opportunistically — only when both documents are in the database.

For agency regulations (the main driver for this story), the pattern is always full document replacement: the new föreskrift exists as its own `LegalDocument`, so cross-references will be created consistently.
