# Story 7.5b: Large-PDF extraction — kollektivavtal-scale documents

## Status

## Story

**As an** HR manager,
**I want** full-length kollektivavtal PDFs (60–150+ pages) to extract successfully,
**so that** the 7.5 upload → RAG flow works for real agreements, not just short documents.

## Context (user checkpoint, 2026-07-03)

First real upload — Vårdföretagarna avtal, 2.9 MB — FAILED: `lib/documents/extract-file.ts` caps single-call output at `PDF_MAX_TOKENS = 16_384`; the transcription hit `stop_reason: max_tokens`, was retried 3× (deterministic failure, 172 s, 3× cost), and marked FAILED. The code itself names the missing follow-up: "Large docs need the page-splitting follow-up." Full-length avtal are 7.5's flagship input — this is a blocker, not an edge case. The fix benefits ALL workspace-file extraction, not just kollektivavtal.

## Acceptance Criteria

1. **Raise the single-call output ceiling:** `PDF_MAX_TOKENS` raised to the maximum `claude-haiku-4-5` supports (verify the current limit from the `@anthropic-ai/sdk` types/docs — believed 64k); use the **streaming API** to collect output if non-streaming rejects long generations or risks network timeouts at that size. `MAX_CHARS` raised proportionally (a 64k-token transcription ≈ 250k+ chars; the current 200k cap would silently truncate) — verify the chunking pipeline handles the larger input (it chunks by section; size per chunk unchanged).
2. **Never retry a deterministic oversize:** `stop_reason === 'max_tokens'` exits the retry loop immediately (retries remain for transient API errors only). This is a straight cost bug today.
3. **Page-splitting for >100-page PDFs** (the Anthropic API's PDF input limit): detect page count (a pdf lib exists — `lib/external/pdf-parser.ts`; verify its capabilities, else use `pdf-lib`/equivalent already in the dependency tree — do NOT add a new dependency without checking what exists); split into page-range windows under the input limit, extract each window (sequentially is fine — cron `maxDuration 300` is the budget; note in completion if a very large doc risks it), concatenate markdown in order with a page-window comment marker between segments.
4. **Oversized-window fallback:** if a single window's transcription still hits max_tokens at the raised ceiling, halve the window and retry once at the smaller size before failing — FAILED remains possible but only for pathological documents.
5. Status semantics unchanged: success → `DONE` + chunk hand-off exactly as today (CA files → agreement READY per 7.5); genuine failure → FAILED honestly. Existing small-PDF behavior byte-equivalent where it already worked.
6. Telemetry: `FILE_EXTRACTION` usage rows still written per call (now possibly multiple per file — sum or one row per call, follow the existing pattern; note the choice).

## QA Results

### Review Date: 2026-07-03

### Reviewed By: Quinn (Test Architect)

### Code Quality Assessment

High-quality, tightly-scoped implementation of a cost/reliability-critical path. The refactor cleanly separates concerns: `transcribePdfBuffer` (one call + transient-retry policy), `runWindow` (halve-once + validation semantics), `pdf-windows.ts` (pure window math + pdf-lib mechanics). All six ACs are implemented and test-pinned; the four load-bearing external-API claims were independently verified against the current Anthropic reference: claude-haiku-4-5 output cap **is** 64K tokens, the PDF input limit **is** 100 pages for 200K-context models, streaming **is** required above ~16K output, and `.stream(...).finalMessage()` is the SDK-recommended collection pattern.

**Focus-area verdicts:**

1. **No-retry-on-oversize — CONFIRMED STRUCTURALLY.** `stop_reason === 'max_tokens'` returns `{kind:'oversize'}` from inside the `try` — there is no code path back into the `for` retry loop for a *completed* response; only thrown errors re-iterate. This holds identically in the window path and the halve-once path (halves call the same `transcribePdfBuffer`). Tests assert exactly 1 call for both unhalvable legs (1-page doc; unknown page count) and exactly 2 calls when a half re-oversizes (no second halving, second half never attempted). The transient leg is separately pinned (529 → backoff → success, 2 calls). The 3×-cost bug from the checkpoint upload is dead.
2. **Window math — CONFIRMED.** `computeWindows` is provably contiguous/complete (start accumulates; sizes sum to pageCount; ceil/floor+remainder gives 250 → 84+83+83, 101 → 51+50); the test's `assertCoverage` pins contiguity, ordering, full coverage and the ≤100 bound. Byte-equivalence on the small path is real, not aspirational: `useWholeBuffer=true` sends the ORIGINAL buffer (no pdf-lib round-trip) and the test asserts `source.data === pdf.toString('base64')`. The 150-page test decodes both sliced sub-PDFs and verifies 75+75 real pages.
3. **Streaming parity — CONFIRMED.** Errors from `finalMessage()` reject into the same `catch` as the old non-streaming call: encrypt/password → ENCRYPTED, everything else → backoff retry. Usage is captured from the final message *before* the oversize check, so oversize calls still bill honestly — same as pre-7.5b.
4. **Marker pollution downstream — ACCEPTABLE (verified in the chunker, read-only).** The marker joins as its own `\n\n`-separated paragraph. In `chunkWorkspaceMarkdown` it normally merges into an adjacent ~400-token block (harmless; arguably useful page provenance in RAG content). It can **never** pollute `contextual_header`/section detection: `extractLeadingHeading` requires `^#{1,6}\s` on a block's *first line*, and `mergeParagraphs` guarantees a heading always *starts* its own block — a marker can never precede a heading within a block. A marker-ONLY garbage chunk is possible but requires the marker (34 chars > MIN_CHUNK_CHARS 20) to land between two blocks that both can't absorb it — low probability, low impact (MARKER-001, note-for-later).
5. **MAX_CHARS 800k — CONFIRMED.** `truncate()` boundary-backup logic unchanged; `truncated` semantics unchanged; test pins >200k output survives. Grep: no functional consumer hardcodes 200k — the only stale artifact is the cron's log *string* "truncated at 200k chars" (LOG-001, cosmetic, out of zone, dev self-flagged).
6. **ENCRYPTED pre-detection — NO FALSE-POSITIVE RISK IN PRACTICE.** pdf-lib's encrypted error text ("…is encrypted…") matches `/encrypt|password/i`; its corrupt-input errors ("Failed to parse PDF document… No PDF header found", struct-parse errors) do not — corrupt buffers fall through to the legacy single-call path (test-pinned: unparseable buffer still extracts DONE via one call). The regex-on-message pattern itself is pre-existing (identical to the pre-7.5b API-error leg), not new exposure.
7. **Contract stability — CONFIRMED.** `ExtractFileResult` shape unchanged; consumer suites (cron route 5, chunks 66) pass unmodified. Git-verify: the working-tree cron-route diff is **Story 7.5's** chunk-routing work (`indexExtractedFile`/`markAgreementFailedForFile` — comments and the untracked `lib/chunks/ingest-extracted-file.ts` attribute it unambiguously), not a 7.5b zone violation. 7.5b's footprint is exactly its File List. Telemetry: one summed usage object → one FILE_EXTRACTION row per file, matching the existing cron pattern.

**Dev scrutiny-list verdicts:**

- **Blank-window semantics — correct design.** Per-window `no text content` → no segment; whole-doc EMPTY only when ALL windows are blank. This is the right generalization of the single-doc EMPTY leg (a 120-page doc with a blank scanned cover section should not be EMPTY). Both legs test-pinned. One cosmetic edge: if window 1 is blank, the first *emitted* segment carries no marker (`i === 0` rule), so its page provenance is implicit (PROV-001, note-for-later).
- **Stale cron log line — real but cosmetic, correctly left.** Out of zone per the story's own boundary. Fix the string (or derive from MAX_CHARS) in the next story that opens the cron route.
- **Telemetry-on-FAILED — not an issue; behavior-preserving.** Pre-7.5b code ALREADY returned `usage` on completed-call FAILED (oversize, no-text-block) — verified in the diff. 7.5b preserves that and matches the old no-usage behavior when zero calls completed. Recording cost for tokens actually burned is the honest choice.
