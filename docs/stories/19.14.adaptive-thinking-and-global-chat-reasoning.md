# Story 19.14: Adaptive Thinking + Reasoning for Global Chat

> üìå **Origin (2026-06-21, PO):** raised during a pre-distribution review of the agent's reasoning capabilities. Two findings drove this story: (1) the chat route configures **fixed per-context thinking budgets** via the now-deprecated `thinking: { type: 'enabled', budgetTokens }` path ‚Äî on Sonnet 4.6 the modern, non-deprecated equivalent is **adaptive thinking** (`thinking: { type: 'adaptive' }` + `effort`), where the model self-regulates how much to think per turn; (2) the **`global` (Hem dashboard) chat has reasoning fully disabled** ‚Äî `THINKING_BUDGET` has no `global` key, so it resolves to `0`, leaving the most open-ended surface (open applicability questions) with zero reasoning. This story migrates the thinking configuration to adaptive + per-context `effort` and, in doing so, enables reasoning for `global`. **Scope discipline:** this is a *configuration* change on the existing Anthropic thinking seam ‚Äî **not** a model-tier change. Upgrading the assessment path Sonnet‚ÜíOpus is a separate, deliberate decision and is explicitly out of scope here. Pairs naturally with **19.13** (grounding eval) so the quality delta can be measured rather than assumed.

## Story

**As a** product owner,
**I want** the agent's chat to use Anthropic adaptive thinking with a per-context effort level (including a sensible default for the previously-unreasoned global chat),
**so that** the agent reasons appropriately on hard applicability/threshold questions across *all* chat surfaces, on the non-deprecated API path, without hand-tuning fixed token budgets.

## Story Context

**Existing System Integration:**

- **Chat route:** `app/api/chat/route.ts` is the single seam. It currently holds:
  - `THINKING_BUDGET: Record<string, number>` ‚Äî `{ change: 8_000, task: 3_000, law: 2_000 }`. **No `global` key** ‚Üí `global` falls through to `0`.
  - `const thinkingBudget = contextType ? (THINKING_BUDGET[contextType] ?? 0) : 0` ‚Äî resolves the budget; `0` disables thinking.
  - `contextType` typed `'global' | 'task' | 'law' | 'change'`.
  - `providerOptions` built **only when `thinkingBudget > 0 && modelProvider === 'anthropic'`**, as `{ anthropic: { thinking: { type: 'enabled', budgetTokens: thinkingBudget } } }`.
  - Model is `claude-sonnet-4-6` on the Anthropic path (`modelProvider = process.env.AI_CHAT_MODEL ?? 'openai'`); the OpenAI path (`gpt-4-turbo`) has no thinking.
  - `thinkingEnabled: thinkingBudget > 0` is written to telemetry (the `onFinish` / decision-log path).
  - `reasoningTokens = totalUsage.reasoningTokens ?? 0` is already captured.
  - `export const maxDuration = 90` ‚Äî comment notes the 90 s budget exists to accommodate thinking on tool-heavy assessments.
- **System prompt:** `lib/agent/system-prompt.ts` injects `REASONING_GUIDANCE` ("Resonemangsv√§gledning") **when thinking is enabled**. That gating must track the new "is reasoning on for this context?" signal, not the removed `budgetTokens > 0`.
- **AI SDK:** the project uses the Vercel AI SDK (`streamText`, `anthropic()` provider, `providerOptions.anthropic`) ‚Äî **not** the raw Anthropic SDK. Adaptive thinking and effort are expressed through `providerOptions.anthropic` (see Dev Notes for the verified shape).

**Out of scope:**

- **Model-tier change** (Sonnet ‚Üí Opus) on any path ‚Äî separate decision.
- The OpenAI fallback path ‚Äî it stays thinking-less; adaptive applies to the Anthropic path only.
- Per-turn dynamic effort selection beyond the static per-context map.

## Acceptance Criteria

### Configuration
1. The fixed `THINKING_BUDGET` numeric map is replaced by a per-context **effort** map (e.g. `THINKING_EFFORT: Record<contextType, 'low'|'medium'|'high'|'max'>`), covering all four context types **including `global`**.
2. On the Anthropic path, thinking is configured as **adaptive** (`thinking: { type: 'adaptive' }`) with the context's `effort`, replacing the deprecated `{ type: 'enabled', budgetTokens }` shape. The exact `providerOptions.anthropic` nesting for `effort` is confirmed against the installed `@ai-sdk/anthropic` version before merge (AC 11).
3. Per-context effort is **locked** (SM sign-off 2026-06-23): **`change ‚Üí high`, `task ‚Üí medium`, `law ‚Üí medium`, `global ‚Üí medium`**. `global` is thereby adaptive-enabled (was disabled). `max` is **not** used on any context (90 s `maxDuration` guardrail ‚Äî see AC 9).
4. The OpenAI path (`modelProvider !== 'anthropic'`) is unchanged ‚Äî no thinking config emitted.

### Wiring & telemetry
5. The `thinkingEnabled` telemetry flag (and any other `thinkingBudget > 0` consumer) is re-derived from the new signal (adaptive-on for the context) ‚Äî no dangling reference to the removed `budgetTokens` variable.
6. `REASONING_GUIDANCE` injection in `lib/agent/system-prompt.ts` is gated on the new "reasoning on for this context" signal (now true for `global` too), not on `budgetTokens > 0`.
7. `reasoningTokens` continues to be captured in telemetry (no regression to the existing usage/decision-log write).
8. **Reasoning-block UI continuity** (Epic-19 DoD, `epic-19-‚Ä¶:191` ‚Äî "existing chat streaming, reasoning blocks, tool-call collapse, citation pills continue to work"): the streamed reasoning/thinking blocks still render in `chat-message.tsx` under adaptive thinking. Manually verify a `change` turn still shows the thinking display, and that a `global` turn (newly reasoning-enabled) renders reasoning blocks without breaking the message layout. Adaptive emits reasoning only when the model elects to think, so a no-think turn must degrade cleanly (no empty block / no layout shift).

### Safety & guardrails
9. The `maxDuration = 90` interaction is reviewed: chosen effort levels must not routinely exceed the 90 s streaming budget on tool-heavy `change` assessments. **Measure** via the existing per-turn duration telemetry (the `onFinish` usage/decision-log write that already records token + timing data) on a representative tool-heavy `change` turn. If `high`/`max` risks timeouts, dial back the default and note it (a comment + Change Log entry); do not raise `maxDuration` in this story.
10. No new env var is *required*; if effort is made env-overridable, it must default safely (missing/invalid env ‚Üí the static map value, never a crash).

### Quality
11. `pnpm typecheck` + `pnpm eslint` clean. `contextType` typing stays exhaustive over the four contexts (a missing context key is a compile error, not a silent `0`/undefined).
12. **Docs-verified:** a one-line note in Dev Notes/PR confirming the chosen reasoning-config shape was checked against (a) the installed `@ai-sdk/anthropic` types and (b) the Anthropic reference for Sonnet 4.6 adaptive-thinking support. The installed stack is **`ai@^6.0.50` + `@ai-sdk/anthropic@^3.0.23`**, so two valid forms exist on this version ‚Äî the provider-options form (`providerOptions.anthropic.thinking` + `effort`) **and** the AI SDK v6 portable top-level `reasoning: 'low'|'medium'|'high'` param. Task 4 evaluates both and records which was chosen and why; the public docs also show `effort` both as a sibling of `thinking` and nested within it across SDK versions, so the exact nesting must be confirmed if the provider-options form is kept.

## Dev Notes

- **Installed stack:** `ai@^6.0.50`, `@ai-sdk/anthropic@^3.0.23` (`package.json`). Both reasoning-config forms below are available on this version.
- **Verified docs (2026-06-21):**
  - **AI SDK (context7, `/vercel/ai`):** adaptive thinking is `providerOptions.anthropic.thinking: { type: 'adaptive' }`, returning `reasoningText` / `reasoning`. Effort is shown as `providerOptions.anthropic.effort: 'low'|'medium'|'high'|'max'`. ‚öÝÔ∏è **Shape caveat:** some doc/SDK versions show `effort` as a **sibling** of `thinking` (`{ thinking: { type:'adaptive' }, effort:'high' }`), others **nested** (`{ thinking: { type:'adaptive', effort:'high' } }`). Confirm against the installed `@ai-sdk/anthropic` if this form is chosen (AC 12). The deprecated budget form (`{ type:'enabled', budgetTokens }`) is what we're replacing.
  - **AI SDK v6 portable param:** `ai@6` also exposes a provider-agnostic top-level `reasoning: 'low'|'medium'|'high'` on `streamText`, which maps to the Anthropic adaptive/effort config under the hood. Cleaner and version-stable; the trade-off is it's one step removed from the raw provider knobs. Task 4 picks between this and the provider-options form.
  - **Anthropic reference + official docs (`platform.claude.com/docs/en/build-with-claude/adaptive-thinking`, verified 2026-06-23):** Sonnet 4.6 **supports adaptive thinking**; `thinking: {type: "adaptive"}` is the exact shape; `budget_tokens` (`type: "enabled"`) is **deprecated on Sonnet 4.6** (still functional ‚Üí why the current code works); omitting `thinking` = disabled (today's `global` behaviour). Confirmed point-by-point against the official page.
- **Effort ‚Äî raw API shape & Sonnet ceiling (official docs):** in the **raw Anthropic API**, effort lives at **`output_config: { effort }`** (a top-level sibling of `thinking`, *not* inside it). The AI SDK maps its `providerOptions.anthropic` (or the v6 portable `reasoning`) onto that ‚Äî which is exactly why AC 12 requires confirming the SDK's surface shape. **Valid effort on Sonnet 4.6 = `low | medium | high(default) | max`.** `xhigh` exists but is **Opus/Fable-only ‚Äî do not use it on Sonnet 4.6.** Default effort is `high` (Claude almost always thinks); lower levels let it skip thinking on trivial turns ‚Äî the mechanism that makes enabling `global` safe.
- **‚öÝÔ∏è `display` default is model-dependent ‚Äî latent trap for the future Opus upgrade (AC 8 + out-of-scope note):** on **Sonnet 4.6, `thinking.display` defaults to `"summarized"`**, so reasoning summaries keep rendering exactly as today ‚Äî **AC 8 is safe on the current model with no `display` change**. But on **Opus 4.8/4.7 the default flips to `"omitted"`** (empty `thinking` field) ‚Äî so if the separately-scoped Sonnet‚ÜíOpus model bump ever happens, reasoning blocks would silently go blank in `chat-message.tsx` unless `display: "summarized"` is set explicitly. Flag this in whatever story owns the model upgrade; it is **not** handled here.
- **Why adaptive over bumping budgets:** the original ask was "should we raise the per-context budgets?" Adaptive answers that structurally ‚Äî a trivial `global` lookup thinks little, a hard `change` applicability question thinks more, within the effort ceiling ‚Äî so we stop hand-tuning fixed token counts. This was the explicit recommendation that spawned the story.
- **Single seam.** All thinking config lives in `app/api/chat/route.ts`. Keep the change localized there + the one gating line in `lib/agent/system-prompt.ts`. Do not scatter effort logic.
- **Don't break the OpenAI fallback.** `providerOptions` must remain absent (or thinking-less) when `modelProvider !== 'anthropic'`. Today that's enforced by the `modelProvider === 'anthropic'` guard ‚Äî preserve it.
- **Telemetry continuity.** `thinkingEnabled` and `reasoningTokens` feed existing dashboards/decision logs (14.27 / 19.5 lineage). Re-derive `thinkingEnabled` from the new signal; don't drop the field.
- **Cost is not the driver.** Per the originating analysis, thinking tokens (billed as output tokens) are a rounding error at this product's B2B volume; `effort` is the cost/latency lever if ever needed. Do not add cost-gating machinery in this story.
- **Latency is the real constraint.** `maxDuration = 90`. Favor `medium` broadly and reserve `high` for `change`; only consider `max` if a measured assessment turn stays comfortably under 90 s.
