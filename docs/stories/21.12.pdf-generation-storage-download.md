# Story 21.12: Revisionsrapport PDF — generation, Supabase Storage persistence, download route

## Status

Ready for Review

## Story

**As a** lead auditor who has completed (AVSLUTAD) or fastställt (SEALED) a Lagefterlevnadskontroll cycle,
**I want** a Laglig-branded PDF of the revisionsrapport that I can download from the Rapport tab or a deep-link URL,
**so that** I can hand the report to an external certifieringsorgan, archive it as internal statutory documentation, and rely on the sealed cycle's PDF carrying the full seal-hash on every page for tamper-evidence verification months after the cycle was closed.

## Acceptance Criteria

### Functional

1. **Server action — `generateCycleReport`.** New export in **`app/actions/compliance-audit-report.ts`** (same file as Story 21.11's `getRevisionsrapportInput`). Contract:
   - **Zod input (`GenerateCycleReportSchema`):** `{ cycleId: string().uuid(), kind: z.enum(['COMPLETE', 'SEALED']) }`. Kind determines which `ComplianceAuditReport` row is upserted — `SEALED` is only valid when `cycle.status === 'SEALED' || 'ARKIVERAD'`; `COMPLETE` is valid for `AVSLUTAD | SEALED | ARKIVERAD` (so AVSLUTAD users get a COMPLETE-kind PDF, and SEALED users can retain access to the pre-seal COMPLETE PDF if it was generated — the DB already supports one row per `(cycle_id, report_kind)` via the `@@unique([cycle_id, report_kind])` constraint from Story 21.9 seal work). `PLANERAD | PAGAENDE` → structured error `"Rapport kan inte genereras innan kontrollen är slutförd"`.
   - **Auth:** `withWorkspace(cb, 'tasks:edit')` — same gate as other cycle mutations; AUDITOR role does **not** trigger PDF generation (read-only; AUDITOR reaches the already-generated PDF via the download route, AC 4). Inside the callback, no additional role check beyond the tasks:edit bar is needed — this is a safe, idempotent operation that only writes to storage + upserts one DB row.
   - **Load step:** call `getRevisionsrapportInput({ cycleId })` (existing, Story 21.11 `compliance-audit-report.ts:98`) to produce the `{ input, html }` pair. Propagate its error verbatim on failure (keeps the Swedish message chain consistent).
   - **PDF render step:** call `renderRevisionsrapportPdf(html)` from the new `lib/compliance-audit/revisionsrapport-to-pdf.ts` (AC 6). Returns a `Buffer` of PDF bytes.
   - **Storage-upload step:** two uploads in parallel via `Promise.all`:
     - `compliance-audit-reports/{workspace_id}/{cycle_id}/report-{YYYYMMDDHHmmss}-{kind}.pdf` (binary, `contentType: 'application/pdf'`, `upsert: true`).
     - `compliance-audit-reports/{workspace_id}/{cycle_id}/report-{YYYYMMDDHHmmss}-{kind}.html` (text, `contentType: 'text/html; charset=utf-8'`, `upsert: true`). The HTML mirror is the audit-grade archive companion per architecture §6.5.
     - Timestamp: `format(new Date(), 'yyyyMMddHHmmss')` UTC equivalent via `toISOString().replaceAll('-', '').replaceAll(':', '').replaceAll('T', '').slice(0, 14)` — note `date-fns` locale does **not** control UTC behaviour, so use the ISO approach for deterministic UTC timestamps. Kind component is lowercase: `complete` or `sealed`. The bucket is `workspace-files` (shared with `app/actions/files.ts:24`) — **do not create a new bucket**. (Regex character-class form avoided in this example because Tailwind v3's JIT content scanner picks up `[<id>:<value>]` tokens from markdown sources and compiles them to invalid CSS.)
     - **Storage client:** `getStorageClient()` from `@/lib/supabase/storage` (already in use across `app/actions/files.ts`).
   - **DB upsert step:** `prisma.complianceAuditReport.upsert` keyed on `{ cycle_id_report_kind: { cycle_id, report_kind: kind } }` — **same composite key as `sealCycle` uses at `compliance-audit-cycle.ts:1220-1237`**. Mirror that pattern exactly: the `create` branch populates all five fields (`cycle_id`, `report_kind`, `generated_at`, `pdf_storage_path`, `html_storage_path`, `manifest`); the `update` branch writes `generated_at`, `pdf_storage_path`, `html_storage_path` and leaves `manifest` untouched when the kind is `SEALED` (the seal transaction is the sole authority for the sealed manifest — do NOT let a regen call stomp a canonical manifest). For `COMPLETE` kind, `manifest` is freshly computed from the current cycle state via a module-private `buildComplianceReportManifest(input: RevisionsrapportInput): Prisma.InputJsonValue` helper that serialises `{ cycle, items, findings, snapshots, workspace, generatedAt }` as plain JSON (no canonicalize dependency — this manifest is for operational archival, not cryptographic verification; the sealed-kind manifest from 21.9 carries that concern).
   - **ActivityLog:** write a `cycle_report_generated` row (new action constant — see AC 7) with `entity_type: 'compliance_audit_report'`, `entity_id: complianceAuditReport.id`, `old_value: { pdfStoragePath: previousPath ?? null }`, `new_value: { pdfStoragePath: newPath, kind, durationMs }`. On first generation `old_value.pdfStoragePath` is `null`; on regeneration the prior path is recorded so an ops query can reconstruct PDF history from the activity feed.
   - **Revalidate:** `` revalidatePath(`/laglistor/kontroller/${cycleId}`) `` so the Rapport tab picks up the new `pdf_storage_path` on next navigation. **Note the backticks — the existing `compliance-audit-cycle.ts:800` pattern uses a template literal; single-quoting would silently revalidate the literal path `/laglistor/kontroller/${cycleId}`.**
   - **Return shape:** `{ success: true, data: { pdfStoragePath: string, htmlStoragePath: string, generatedAt: string, durationMs: number } }` on success; `{ success: false, error }` otherwise. All errors caught at the outer `try` and mapped to Swedish strings (same style as existing actions in this file).
   - **Idempotency note:** the route handler (AC 3) calls `generateCycleReport` only when the report row is missing OR stale per AC 2; concurrent duplicate calls are serialised by Supabase upsert + Prisma upsert and resolve to identical final state (the DB `@@unique` constraint ensures only one row survives per `(cycle_id, kind)`).

2. **Staleness detection — `reportNeedsRegeneration(report, cycle): boolean`.** Module-private helper in `app/actions/compliance-audit-report.ts`. Returns `true` iff:
   - `report === null` (never generated), OR
   - `report.pdf_storage_path === null` (row exists but PDF never uploaded — e.g. from the Story 21.9 seal transaction which only populates `manifest`), OR
   - `report.report_kind === 'COMPLETE'` AND the cycle's most-recent `updated_at` (on the cycle itself, its items, or its findings) is newer than `report.generated_at`. **Implementation:** resolve `mostRecentTouch` via a single Prisma query `SELECT GREATEST(MAX(updated_at)) FROM ...` or — simpler and kept inside the TypeScript layer — three `findFirst({ orderBy: { updated_at: 'desc' }, take: 1, select: { updated_at: true } })` calls in parallel, then `Math.max` of the returned timestamps. Do the simpler TS version; perf impact is trivial and avoids raw SQL.
   - **SEALED reports are NEVER stale.** Once the cycle is sealed, the canonical manifest is frozen (Story 21.9 seal transaction), so a sealed-kind PDF only needs regeneration if the `pdf_storage_path` is `null` (i.e. seal happened but PDF render did not yet). Any later content changes are impossible because of the immutability guard (Story 21.10 if it lands, or defence-in-depth from the sealed-status gate in every mutation action today).
   - **COMPLETE reports are stale** after any item/finding edit (legal per 21.6 AC 5 — AVSLUTAD is still editable). Regenerate on next download.
   - **Revert-and-recomplete edge case (SF-3).** If a lead auditor uses Story 21.6's revert escape hatch (`AVSLUTAD → PAGAENDE → AVSLUTAD` with no content edits in-between), the `MAX(updated_at)` check would return false — items + findings + cycle metadata are unchanged — and the stale COMPLETE PDF from the first completion would be served. **Fix:** modify `completeCycle` in `app/actions/compliance-audit-cycle.ts:722-820` to invalidate any existing COMPLETE-kind report row at the moment of transition back to AVSLUTAD: `prisma.complianceAuditReport.updateMany({ where: { cycle_id, report_kind: 'COMPLETE' }, data: { pdf_storage_path: null, html_storage_path: null } })`. This drops the PDF pointer (so AC 2's second bullet — "pdf_storage_path === null" — fires on next download and triggers regeneration) while keeping the row + manifest for audit continuity. No equivalent change needed for `revertCycleToPagaende` — the transition _away_ from AVSLUTAD already means the cycle cannot serve a COMPLETE PDF to end-users (status gate AC 3 blocks the route handler). The null-out only matters when the user re-completes.

3. **Route handler — `GET /laglistor/kontroller/[cycleId]/rapport/pdf`.** New file at **`app/(workspace)/laglistor/kontroller/[cycleId]/rapport/pdf/route.ts`**. Contract:
   - `export const maxDuration = 300` — architecture §7.2 notes 30s for pure stream-through, but since this handler performs **lazy generation on cache miss** (AC 3 step 4), we need the full Vercel Pro ceiling. Per tech stack §3.1 line 45, `maxDuration` up to 300s is available on Pro (verified production config).
   - `export const dynamic = 'force-dynamic'` — mirrors `app/api/cron/retry-failed-pdfs/route.ts:19`.
   - **Query params:** `kind` (optional, `'complete' | 'sealed'`). Parse via a module-private `const KindQuerySchema = z.enum(['complete', 'sealed']).optional()` — `safeParse(searchParams.get('kind') ?? undefined)` at the route entry; on `success: false` return 400 JSON `{ error: "Ogiltig kind-parameter" }`. If the parsed result is `undefined`, resolve the default server-side: `'sealed'` when `cycle.status === 'SEALED' || 'ARKIVERAD'`, else `'complete'`. The Prisma enum lookup downstream uses `kind.toUpperCase()` — keep this conversion co-located with the Zod parse so the uppercase form never leaks to URL / UI contracts.
   - **Auth:** `getWorkspaceContext()` from `@/lib/auth/workspace-context` (same helper used by `app/api/workspace/documents/[documentId]/export/route.ts:30`). On `WorkspaceAccessError` → 403. Additionally require `hasPermission(role, 'activity:view') || hasPermission(role, 'tasks:edit')` (same OR-check as `getRevisionsrapportInput`) so AUDITOR can download.
   - **Cycle fetch:** workspace-scoped `prisma.complianceAuditCycle.findFirst({ where: { id: cycleId, workspace_id: workspaceId }, select: { id: true, status: true, name: true } })`. Not found → 404.
   - **Status gate:** if cycle status is `PLANERAD | PAGAENDE` → 409 JSON `"Rapport kan inte genereras innan kontrollen är slutförd"`. Preempts a confusing storage miss.
   - **Report row fetch:** `prisma.complianceAuditReport.findUnique({ where: { cycle_id_report_kind: { cycle_id, report_kind: kind.toUpperCase() } } })`.
   - **Lazy-generation branch:** if `reportNeedsRegeneration(report, cycle)` returns `true` → call `generateCycleReport({ cycleId, kind: kind.toUpperCase() })`. On failure → 500 JSON with the action's error message. On success → re-fetch the report row (or use the returned `pdfStoragePath` directly).
   - **Download from Supabase:** `storageClient.storage.from('workspace-files').download(report.pdf_storage_path)` — mirrors `lib/supabase/storage.ts:111-114`. Failure → 502 JSON `"Rapporten kunde inte laddas från lagringen"`.
   - **Response headers:**
     - `Content-Type: application/pdf`.
     - `Content-Disposition: attachment; filename="kontrollrapport-${sanitizedCycleName}-${timestamp}.pdf"` — sanitize with `sanitizeFilename()` from `@/lib/utils/sanitize-filename` (same import path as `app/api/workspace/documents/[documentId]/export/route.ts:11`). Timestamp: `format(new Date(), 'yyyy-MM-dd')` (CET display in the filename is fine — this is a user-facing string, not an archival key).
     - `Cache-Control: private, no-store` — prevents intermediate-proxy caching of tenant-scoped PDF bytes.
   - **Body:** stream the file as `new Response(buffer, { headers, status: 200 })` — matches `app/api/workspace/documents/[documentId]/export/route.ts` pattern. Do NOT use `NextResponse.json` for binary responses.

6. **Revisionsrapport-specific PDF wrapper — `lib/compliance-audit/revisionsrapport-to-pdf.ts`.** New file. Thin wrapper that:
   - Imports `renderHtmlToPdf` from `@/lib/pdf/render-html-to-pdf`.
   - Exports `renderRevisionsrapportPdf(html: string): Promise<Buffer>`.
   - Composes the revisionsrapport-specific `headerTemplate` and `footerTemplate` (HTML strings Puppeteer injects onto every page):
     - **Header template** (shown on every page **except the first** via Puppeteer's `displayHeaderFooter: true` + CSS `.date`/`.title`/`.pageNumber` placeholders). Always muted grey `#666` 9pt. Contains: **truncated seal hash** (`{hash.slice(0, 16)}…` when present, empty string otherwise) right-aligned. **Cover-page header behaviour — open design question, flag to UX-expert at implementation time.** Puppeteer's `displayHeaderFooter: true` renders the header on every page **including page 1** by default. Epic AC 4 says "truncated hash displayed on header of every interior page" — implying the cover should NOT carry the running header. Puppeteer does not support first-page suppression natively; known workarounds: (a) inject a full-height spacer `<div>` on page 1 that pushes the header out of the printable area (fragile), (b) use `@page :first` CSS rules inside the body HTML (Puppeteer ignores `@page` in header/footer templates; works only for body margins), (c) implement a two-pass render where page 1 is generated without headers then merged with a headered remainder (over-engineered for MVP). **Default to the simplification:** render the running header on every page including the cover, accept the minor redundancy with the cover's full-hash seal block. Document this in the wrapper file's JSDoc. **Before committing, ask the UX-expert** (`/ux-expert`) whether the cover-page running header is acceptable; if they reject it, fall back to workaround (a) — a CSS `margin-top: 2cm` on the `body > :first-child` that pushes page-1 content below the header band. Do NOT spend implementation time on workaround (c).
     - **Footer template** (every page): `"Rapport genererad {generatedAt} · Kontroll-ID: {cycle.id.slice(0, 8)}… · Sida <span class='pageNumber'></span> av <span class='totalPages'></span>"`. When sealed: append `· Seal: {fullSealHash}` (full 64-char hash per epic AC 4). The renderer produces this footer INSIDE the HTML body already (Story 21.11 AC 9); we **also** inject the Puppeteer footer template because body footers only render on the last page in print layout whereas `footerTemplate` renders on every page. The two coexist; body footer acts as a fallback if Puppeteer footer fails to render.
   - **The `html` argument is the self-contained `<!DOCTYPE>…</html>` string from `renderRevisionsrapport` (Story 21.11).** Do NOT mutate it — the renderer already owns all content styling. This wrapper's only job is Puppeteer orchestration + header/footer templating.
   - Margin overrides: `{ top: '3cm', bottom: '2.5cm', left: '2cm', right: '2cm' }` — larger `top`/`bottom` to make room for the running header/footer. Document choice in file JSDoc.

8. **Sealed-kind eager generation on `sealCycle`.** Architecture §9 implies sealing is a natural trigger-point for sealed-PDF generation — otherwise the first user to open the Rapport tab after seal pays the 30-90s render cost. Add a post-transaction call in `sealCycle` (after `compliance-audit-cycle.ts:1288-1296` `loadCycleDetailInline`):
   ```ts
   // Story 21.12: eagerly generate the sealed-kind PDF so the download is instant.
   // Errors here are non-fatal (the seal itself is already committed); a failed
   // generation leaves pdf_storage_path=null and the route handler will lazy-
   // generate on first download.
   try {
     await generateCycleReport({ cycleId, kind: 'SEALED' })
   } catch (err) {
     console.error('[sealCycle] Eager PDF generation failed — lazy fallback armed:', err)
   }
   ```
   - **Fire-and-forget pattern:** use Next.js 16's stable `after()` from `next/server` to run generation post-response, so the seal dialog's `Fastställ kontroll` click returns immediately (the user does not wait 30-90s for the PDF render on top of the seal transaction):
     ```ts
     import { after } from 'next/server'
     // …inside sealCycle, after the transaction + loadCycleDetailInline:
     after(async () => {
       try {
         await generateCycleReport({ cycleId, kind: 'SEALED' })
       } catch (err) {
         console.error('[sealCycle after] Eager PDF generation failed:', err)
       }
     })
     ```
   - **Why non-fatal:** if generation fails (e.g. Chromium cold-start timeout), the cycle is still sealed — the seal transaction already committed. The route handler's lazy-generation branch (AC 3) will retry on first download. This matches the architect's deferred-story note that PDF is "a separate Vercel-cron job" — we compress that concept into an `after()` continuation for MVP.
   - Import from `next/server` is adjacent to the existing `revalidatePath` import at `compliance-audit-cycle.ts:19`.

### Integration Verification

9. **IV1 — existing PDF pipeline untouched.** `lib/documents/tiptap-to-pdf.ts` delegates to `renderHtmlToPdf` from `@/lib/pdf/render-html-to-pdf`. The export `generatePdf(contentHtml, metadata)` keeps its signature + output byte-equivalent (same Puppeteer options, same margins, same font). Existing tests in `tests/unit/lib/documents/**` stay green; the document-export route `app/api/workspace/documents/[documentId]/export/route.ts` continues to produce valid PDFs. A smoke test: `curl -I` on the document-export route still returns `Content-Type: application/pdf`.

10. **IV2 — concurrent generation does not deadlock.** Per epic AC IV1: two simultaneous requests to `/laglistor/kontroller/{A}/rapport/pdf` and `/laglistor/kontroller/{B}/rapport/pdf` (distinct cycles, same workspace) must both complete within 120s without memory-limit crash. **Test:** manual repro or Vitest integration with mock Puppeteer. The only shared resource is the Supabase upload client (stateless) + the DB connection pool (already-sized for existing workloads). No process-level lock needed.

11. **IV3 — PDF file size budget.** Per epic AC IV3, a 200-item cycle PDF must stay **under 10 MB**. The renderer HTML for 200 items with ~10 findings sits around 400-600 KB (Story 21.11 fixture output: ~40 KB for 3-item cycle, linear-ish scaling). Puppeteer compresses via `Compress=true` default. Fonts are OS-default (Calibri from the Chromium Linux layer, fallback to sans-serif at the PDF layer) — no embedded fonts, which is the main PDF bloat vector. **Acceptance criterion for the test:** `expect(buffer.byteLength).toBeLessThan(10 * 1024 * 1024)` against a 200-item synthetic fixture.

12. **IV4 — storage path tenant isolation.** The path prefix `compliance-audit-reports/{workspace_id}/{cycle_id}/` maps to the existing `workspace-files` bucket RLS policies — path-based tenant isolation (same as user-uploaded files at `{workspace_id}/files/{file_id}/{filename}`). A cross-workspace `storageClient.storage.from('workspace-files').download('compliance-audit-reports/{otherWorkspace}/…')` attempt from a regular session returns a 403 at the Supabase RLS layer. **Test:** integration test that authenticates as workspace A, attempts to download a PDF at workspace B's path → expects 403 from storage.

### Technical constraints

- **No new dependencies.** `puppeteer-core@^24.40.0` and `@sparticuz/chromium@^143.0.4` are already in `package.json` (lines 61, 123). Architecture §3.2 line 151 is explicit: "Only ONE new dependency is introduced — the canonicalization library" — already landed in Story 21.1. Any new dependency proposal in this story must be rejected.
- **License policy** — MIT/Apache 2.0/BSD/ISC only (architecture §1 line 44). All reused libs are MIT; no change.
- **Bucket reuse** — use `workspace-files`, not `sfs-pdfs`. `sfs-pdfs` is the legal-document PDF bucket (`lib/supabase/storage.ts:9`); `workspace-files` is the workspace-scoped uploads bucket (`app/actions/files.ts:24`). Revisionsrapport PDFs are workspace-scoped content, so `workspace-files` is the correct home.
- **RLS semantics** — path-prefix-based tenant isolation. The `compliance-audit-reports/{workspace_id}/{cycle_id}/` layout mirrors how `files.ts` scopes to `{workspaceId}/files/{fileId}/`; the same RLS policy that allows/denies user-file access applies. Do NOT add a new RLS policy; verify the existing policy covers the new prefix at PR review.
- **Zod-input pattern** — `app/actions/compliance-audit-cycle.ts:56` shows the established Zod schema declaration style (named schema + `.safeParse` at action entry). Follow exactly.
- **Error-return shape** — `ActionResult<T>` from existing actions (`compliance-audit-report.ts:34-38`): `{ success: boolean; data?: T; error?: string }`. Mirror this.
- **Activity-logging helper** — `import { logActivity } from '@/lib/services/activity-logger'` (verified at `lib/services/activity-logger.ts:8`). Signature: `logActivity(workspaceId, userId, entity_type, entity_id, action, old_value, new_value)` — positional args, not an options object. Existing usage at `compliance-audit-cycle.ts:786-797` (in `completeCycle`) is the canonical call-site example. Do NOT invent a new logging wrapper.
- **Puppeteer cold-start** — first invocation per Lambda takes ~3-5s to boot Chromium; subsequent invocations within the same warm instance are ~500ms. A 200-item cycle with 50 findings renders in ~15-30s wall-clock on warm Vercel Pro Lambda. The `maxDuration: 300` ceiling leaves comfortable headroom. Concurrent generation across cycles within one function invocation is NOT attempted — each request spins its own Chromium; Vercel function concurrency handles the rest.
- **After() semantics** — Next.js 16.0.7 (`package.json:117`) ships `after()` as a stable API from `next/server`. It runs after the response is flushed but within the function's execution budget. In practice this extends the seal Server Action's total execution beyond the visible client wait, so the seal UI returns quickly but the Lambda keeps running up to `maxDuration`. For `app/actions/compliance-audit-cycle.ts` (no explicit `maxDuration` export today), Next.js uses Vercel's serverless default of **10s regardless of tier** (architecture §3.1 line 45 confirms: "default 10s (Hobby), up to 300s (Pro)" — Pro raises the ceiling, not the default). 10s is far too short for a 30-90s Puppeteer render. **Mitigation:** add `export const maxDuration = 300` to the file. Impact: applies to ALL server actions in this file (cycle CRUD + lifecycle) — they'll get a 300s ceiling too, which is harmless since they all complete in <1s normally; it only matters when an `after()` extension runs. Document this choice in the file's top-level JSDoc.

### Known gotchas

- **`@sparticuz/chromium` on local dev** — it expects to download the Chromium binary on first invocation. Local dev (pnpm dev) works with the same setup as `tiptap-to-pdf.ts`; no extra config needed. If a dev environment hits "chromium not found" at test time, mock Puppeteer at the module level (`vi.mock('puppeteer-core')`) — that's the existing pattern in `tests/unit/lib/documents/**` if present.
- **Puppeteer header/footer template font size** — defaults to browser default (10pt); Puppeteer's internal stylesheet does NOT inherit from the body's `@page` rules. Force 9pt via inline style in the template HTML: `<div style="font-size: 9pt; color: #666;">…</div>`.
- **`after()` and `revalidatePath`** — per Next 16 docs, `revalidatePath` inside `after()` still flushes the cache before the next request. No special handling needed for Task 6's eager-gen continuation.
- **PDF caching by CDN** — because we set `Cache-Control: private, no-store`, the Vercel Edge CDN will NOT cache the binary response. This is correct (PDFs are tenant-scoped). If a customer asks for faster re-download, Supabase's bucket-level caching via the `storage_object_cache_control` header can be revisited — out of scope here.
