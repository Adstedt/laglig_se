# Story 25.0: First-run path-choice gate (token-saver MVP)

## Status

Approved

## Story

**As a** new user who just finished the workspace-creation wizard,
**I want** Laglig to ask me how I'd like to start my law list ŌĆö pick a template, generate one with AI, import an existing list, or skip and build manually ŌĆö instead of silently running an LLM generation in the background while I land on a chat-first dashboard with no orientation,
**so that** Laglig doesn't burn $0.50ŌĆō$1.50 of tokens generating a list I never asked for, I'm not bewildered by the empty dashboard, and I have a real choice about how to get my first list on the page.

This is the **pre-MVP wedge for Epic 25** (the "B.0" path-choice gate per the epic PRD). It ships independently of Epic 24 with a minimal modal ŌĆö three path cards + tertiary skip ŌĆö and the full schema/action/state-derivation substrate that the rest of the epic (B.1ŌĆōB.6) builds on. The import card is visibly disabled in B.0 until Epic 24.4's review surface lands; B.1 un-disables it. No tabs, no kickoff confirm, no done states, no FAB ŌĆö those are B.1+.

## Acceptance Criteria

### Schema additions

1. **Prisma migration `add_first_run_modal_columns`.** Adds three nullable columns to `Workspace` (no existing column dropped or renamed):
   - `first_run_dismissed_at` ŌĆö `DateTime?` ŌĆö timestamp set when user closes/minimises the first-run modal. `null` = never closed ŌåÆ modal eligible to auto-open on dashboard visit (subject to other guards). Not-null = modal does not auto-open again.
   - `tutorial_fab_dismissed_at` ŌĆö `DateTime?` ŌĆö placeholder for B.6's corner FAB dismissal. Added in B.0 so we don't churn the Workspace model twice. Stays `null` after B.0.
   - `first_run_tabs_viewed` ŌĆö `Json?` with `@default("[]")` ŌĆö placeholder for B.6's tab-viewed tracking. Added in B.0 for the same reason. Stays `[]` after B.0.

2. **Schema default fix on `law_list_generation_status`.** The existing schema declares `law_list_generation_status String? @default("pending")` (see `prisma/schema.prisma:172`). The Epic 25 architecture's `getOnboardingState()` function (arch ┬¦6.4) checks `=== null` to decide whether to open the modal. Both cannot be true. **The same migration drops the `@default("pending")`** so new workspaces created after the auto-fire is removed (AC 14) land at `null`. Existing workspaces with `pending`/`in_progress`/`completed`/`failed`/`skipped` values are unaffected (defaults only apply to new rows).

3. **New `OnboardingEvent` model.** Workspace-scoped event log, polymorphic JSON payload, no schema-level enum on `event_type` (free-form string per the architecture's ┬¦5.2 fail-safe / additive philosophy ŌĆö same approach as `ChatUsageEvent`):
   ```prisma
   model OnboardingEvent {
     id           String   @id @default(cuid())
     workspace_id String
     user_id      String
     event_type   String   // 'modal_opened' | 'path_chosen' | 'tab_viewed' | 'modal_dismissed' | 'fab_dismissed' | 'feedback_submitted' | ...
     payload      Json?    // event-specific data
     created_at   DateTime @default(now())

     workspace    Workspace @relation(fields: [workspace_id], references: [id], onDelete: Cascade)
     user         User      @relation(fields: [user_id], references: [id], onDelete: Cascade)

     @@index([workspace_id, created_at])
     @@index([event_type])
   }
   ```
   - PRIMARY KEY uses `@default(uuid())` ŌĆö matches the Epic 24 ratified convention for new tables and the existing `ChatUsageEvent.id` precedent at `prisma/schema.prisma:1470`.
   - Add the inverse relations on `Workspace` and `User` models: `onboarding_events OnboardingEvent[]`.
   - **No RLS policy block in the migration** (Epic 24 convention ŌĆö RLS is `ENABLE` only via the existing global migration; per-table policies are handled by the Workspace cascade and the `workspace_id` filter on every query).

### Onboarding-state derivation

4. **New module `lib/onboarding/get-onboarding-state.ts`** exporting:
   ```ts
   export type OnboardingState = {
     firstRunOpen: boolean      // true ŌåÆ auto-open modal at path_choice
     fabVisible: boolean        // false in B.0 (FAB ships in B.6)
     fabState: 'working' | 'done' | 'idle'  // 'idle' in B.0
   }
   export async function getOnboardingState(workspaceId: string): Promise<OnboardingState>
   ```
   Three guards combined with `&&`:
   - `FRESH` ŌĆö `Date.now() - workspace.created_at.getTime() <= 24 * 60 * 60 * 1000` (24h cap, matches arch ┬¦6.4 rationale)
   - `first_run_dismissed_at === null`
   - `law_list_generation_status === null` (works because AC 2 drops the `pending` default)

   Returns `{ firstRunOpen: false, fabVisible: false, fabState: 'idle' }` for any workspace not found, on DB error, or when any guard fails. **`fabVisible` is hardcoded `false` in B.0** ŌĆö the full derivation (arch ┬¦6.4) ships in B.6 alongside the FAB component.

5. **`getOnboardingState` is pure.** Only reads from Prisma; no writes, no telemetry events, no side effects. Safe to call on every dashboard render.

### Server actions

6. **New file `app/actions/onboarding-modal.ts`** exporting **five** server actions, each `'use server'`-marked and wrapped in `withWorkspace(cb)` from `lib/auth/workspace-context.ts` (no permission scope ŌĆö workspace-self-write only). Action signatures:
   ```ts
   export async function minimiseFirstRunModal(): Promise<{ ok: true } | { ok: false; error: string }>
   export async function skipLawListGeneration(): Promise<{ ok: true } | { ok: false; error: string }>
   export async function dismissOnboardingFab(): Promise<{ ok: true } | { ok: false; error: string }>
   export async function recordTabViewed(tabId: string): Promise<{ ok: true } | { ok: false; error: string }>
   export async function recordOnboardingEvent(
     eventType: string,
     payload?: Record<string, unknown>
   ): Promise<{ ok: true } | { ok: false; error: string }>
   ```

7. **`minimiseFirstRunModal`** ŌĆö sets `Workspace.first_run_dismissed_at = NOW()`. **Also writes an `OnboardingEvent` row** with `event_type='modal_dismissed'` and `payload={from_state: 'path_choice'}` (only state available in B.0). Returns `{ok:true}` on success.

8. **`skipLawListGeneration`** ŌĆö sets both `law_list_generation_status='skipped'` AND `first_run_dismissed_at=NOW()` in a single update. Writes an `OnboardingEvent` row with `event_type='path_chosen'` and `payload={path:'skipped'}`. Returns `{ok:true}` on success.

9. **`dismissOnboardingFab`** ŌĆö sets `tutorial_fab_dismissed_at=NOW()`. Writes an `OnboardingEvent` row with `event_type='fab_dismissed'` and `payload={from_state:'visible_idle'}`. **Stubbed but exposed in B.0** ŌĆö there's no FAB to dismiss yet, but the action ships now so the FAB component in B.6 calls a stable contract. Add a unit test exercising the action; do not wire any UI to it in B.0.

10. **`recordTabViewed`** ŌĆö appends `tabId` to `Workspace.first_run_tabs_viewed` if not already present (idempotent). Writes an `OnboardingEvent` row with `event_type='tab_viewed'` and `payload={tab_id: tabId}`. **Stubbed but exposed in B.0** ŌĆö same rationale as AC 9. The tutorial tabs ship in B.3.

11. **`recordOnboardingEvent`** ŌĆö generic fire-and-forget event writer. Used by the modal for `modal_opened` (fired on mount in `<FirstRunModal>`) and `path_chosen={path:'generate'|'import'|'template'}` events from the three card click handlers. The action wraps a single `prisma.onboardingEvent.create({...})` call in try/catch and **logs `[ONBOARDING_EVENT_WRITE_FAIL]` via `console.error` on failure** (mirrors Story 14.27's `[CHAT_USAGE_EVENT_WRITE_FAIL]` pattern; the same fail-safe is also applied in `app/actions/workspace.ts` per `[ENTERPRISE_INQUIRY_EMAIL_FAIL]`). **Never re-throws.** Returns `{ok:true}` even on internal failure ŌĆö telemetry writes are best-effort.

12. **Fail-safe pattern applies to the other four actions too** for the OnboardingEvent row write only (the primary `workspace.update` call is allowed to throw ŌĆö that one matters for correctness). Pattern: `prisma.workspace.update({...})` runs first and any failure returns `{ok:false, error}`; the `prisma.onboardingEvent.create({...})` runs after in a try/catch with `[ONBOARDING_EVENT_WRITE_FAIL]` console.error.

### Feature flag

30. **Wrap the modal mount in `hem-page.tsx` behind an env-driven feature flag `FIRST_RUN_MODAL_V0`** (read once at module load, per the existing project convention; see how `LAWLIST_IMPORT_GROUPINGS_ENABLED` is handled in `lib/import/grouping-llm.ts` from Story 24.7). **Default-on semantics:**
    - **Unset** (env var missing): treated as **on** ŌĆö modal renders. Rationale: unset ŌåÆ safe default ŌåÆ ship-and-rollback model.
    - `'false'` or `'0'`: treated as **off** ŌĆö modal does not render.
    - Any other value (including `'true'`, `'1'`, empty string, typos): treated as **on**.
    - Production / preview / dev all default to **on** unless `FIRST_RUN_MODAL_V0=false` is explicitly set. The flag is the **one-line emergency-disable** path per the epic PRD's Risk Mitigation section.

    When the flag resolves to **off**:
    - The modal does not render even if `onboardingState.firstRunOpen === true`.
    - The auto-fire removal (AC 14) is NOT reverted by the flag ŌĆö that's a permanent code change. If the flag is flipped off in an incident, dashboard reverts to its pre-Epic-25 state (banner if status non-null, plain chat-first if null), with no first-run modal AND no auto-fire of generation. Users in that state can still trigger generation manually via the in-app `<CreateListChooser>` / `<ManageListModal>` (Story 12.10b + 24.6). This is the documented incident posture per arch ┬¦13.

## Dev Notes

### ŌÜĀ Architecture-vs-code discrepancies flagged during drafting

The architecture doc `docs/architecture/first-run-onboarding-modal.md` contains three statements that conflict with the actual codebase. The dev should rely on this story's ACs (which are reconciled against the code) and not the architecture's claims for these three points:

1. **Auto-fire location.** Architecture ┬¦2.1 + ┬¦8 source-tree state the auto-fire of `POST /api/workspace/generate-law-list` lives in `app/onboarding/_components/confirm-step.tsx`. **It does not.** The actual location is `app/actions/workspace.ts:347ŌĆō364`, inside the `createWorkspace` server action, in a block labelled `// Story 16.4: Trigger law list generation (fire-and-forget)`. `confirm-step.tsx` only calls `onSubmit()` and has no knowledge of generation. See AC 13 + AC 14 for the actual remove-here-instead instruction.

2. **Schema state for new workspaces.** Architecture ┬¦6.4 + ┬¦5.1 imply `law_list_generation_status` is `null` for fresh workspaces. **It defaults to `"pending"`** per `prisma/schema.prisma:172` (`law_list_generation_status String? @default("pending")`). Without dropping that default in the same migration, `getOnboardingState`'s `=== null` guard would never fire for new workspaces and the modal would never open. AC 2 mandates the default drop in the migration.

3. **Wizard test location.** Architecture ┬¦11 + ┬¦16 (Developer Handoff) say "Update existing test (`tests/unit/components/onboarding/confirm-step.test.tsx`) to assert the call is NOT made." **That test file does not contain any auto-fire assertion** ŌĆö it covers Story 5.12 tier-aware copy. The auto-fire assertion (if it exists) is in `tests/unit/actions/workspace.test.ts`. See AC 15.

These are documentation bugs in the architecture doc ŌĆö not story-level scope changes. The intent of every claim is preserved; only the file paths are wrong. Story 25.0's Change Log records the reconciliation.

### Key decisions baked into this story

- **Mall path navigates to `/laglistor/mallar` instead of an inline template picker.** PRD Story B.0 bullet 4 mentions an inline template-pick sub-step using `getPublishedTemplates()` + `<TemplateOptionCard>`. That adds ~half a day of wiring and is not required to deliver the token-saver wedge. The catalog index already lets the user pick and apply a template. **The inline picker moves to B.1** as part of the brand-chrome upgrade. AC 19 preserves the token-saver outcome (no LLM call on the template path) which is what B.0 is for. Token-cost telemetry will validate this in production.
- **Importera card stays disabled in B.0 even if Epic 24 is partially live.** B.0 ships before Epic 24.4's review surface. Un-disabling the card requires that review surface to exist (so the import flow has somewhere to go on success). B.1 owns the un-disable.
- **No FAB, no tab framework, no done states in B.0.** The PRD explicitly defers all of these to B.2ŌĆōB.6. B.0 is the minimum substrate to stop the token bleed; richer UI ships in subsequent stories.
- **Stubbed actions `dismissOnboardingFab` and `recordTabViewed` ship in B.0** even though no UI calls them. Rationale: their contracts are well-defined now (arch ┬¦7.1) and shipping them in B.0 means B.3/B.6 don't have to ship action + UI in the same PR. Unit tests cover the actions; no integration cost.
- **Schema default drop is non-trivially in-scope.** It's the most subtle part of B.0 ŌĆö easy to miss, breaks the entire trigger logic if missed. AC 2 + Dev Note flag #2 above call this out twice deliberately.
- **`OnboardingEvent.id` uses `@default(uuid())`.** Matches Epic 24's ratified `@default(uuid())` convention for new tables and the existing `ChatUsageEvent.id` precedent (verified at `prisma/schema.prisma:1470`). An earlier draft of this story claimed cuid was the precedent ŌĆö that was incorrect; uuid is consistent across all recent models.
