# Story 24.5: Catalog requests ‚Äî admin queue + 24h SLA fulfilment

## Status

Ready for Review

## Story

**As a** Laglig.se ops team member responsible for keeping the legal catalog current,
**I want** an admin queue showing all pending "request to add" items from imports ‚Äî sorted by age with SLA breach indicators ‚Äî and a one-click fulfilment flow that re-matches the originating row and emails the user,
**so that** the 24h manual-ingestion SLA is operationally measurable and switchers see their unmatched rows resolved without follow-up support tickets.

This is the **ops-facing back-office surface** for `CatalogIngestRequest` rows created by Story 24.4. Closes the 24-hour loop: user requests ‚Üí ops sees ‚Üí ops fulfils ‚Üí original row auto-rematches ‚Üí user gets email.

### Fulfilment flow ‚Äî "Markera hanterad"

7. **"Markera hanterad" CTA opens a modal** (`<Dialog>`) with:
   - Required field: **LegalDocument ID** that has been ingested. Implementation: simple text input + "Validera" button that calls `prisma.legalDocument.findUnique({where: {id}, select: {id, title, sfs_number, document_type}})` server-side. **Validation contract:**
     - Empty input ‚Üí toast.error "Ange ett dokument-id"
     - Lookup returns null ‚Üí inline form-error "Inget dokument med det id:t hittades"
     - Lookup returns row ‚Üí render the matched title + sfs_number inline below the input as visual confirmation, enable Bekr√§fta button
   - Required confirm checkbox: "Jag bekr√§ftar att dokumentet √§r fullst√§ndigt ingested och kan matchas mot k√§llraden."
   - Optional admin note (carries forward from detail panel)
   - **Bekr√§fta** primary CTA ‚Üí calls `fulfillCatalogRequest({requestId, fulfilledWithDocumentId, adminNote?})` server action. Disabled until both validation passed AND checkbox ticked.

8. **`fulfillCatalogRequest({requestId, fulfilledWithDocumentId, adminNote?})`** ‚Äî implements the skeleton from 24.1. Steps (single Prisma transaction):
   - Verify request status = `PENDING`
   - Verify `fulfilledWithDocumentId` resolves to a `LegalDocument` row
   - Update `CatalogIngestRequest`: `status='FULFILLED'`, `handler_user_id=ctx.userId`, `fulfilled_with_document_id=...`, `admin_note=...`, `fulfilled_at=NOW()`
   - Update the corresponding `LawListImportRow`: `match_status='CATALOG_REQUEST_FULFILLED'`, `matched_document_id=fulfilledWithDocumentId`, `confidence_score=1.0`, `match_reasoning='Manuellt ingesterat av ops'`
   - Trigger email: `emails/catalog-request-fulfilled.tsx` to the requesting user (see AC 11)
   - Activity-log: `catalog_request.fulfilled` (new entity type + action)
   - Return `{success: true}`
   - **Idempotency:** if request already `FULFILLED`, return success with note that it was already fulfilled; do not re-email.

### Rejection flow ‚Äî "Avvisa som dubblett"

9. **"Avvisa som dubblett" CTA** opens a smaller confirmation:

10. **`rejectCatalogRequest({requestId, adminNote?})`** ‚Äî new server-action implementation:
    - Verify request status = `PENDING`
    - Update `CatalogIngestRequest`: `status='REJECTED'`, `handler_user_id=ctx.userId`, `admin_note=...`, `rejected_at=NOW()`

    - Activity-log: `catalog_request.rejected`
    - **No email sent** on rejection (intentional ‚Äî avoids cluttering user inboxes with negative outcomes; user can see rejection state if they revisit the import)

## Dev Notes

**‚öÝ Cross-cutting Epic 24 conventions (ratified 2026-05-07 per Story 24.1 QA gate `docs/qa/gates/24.1-import-schema-and-server-action-skeleton.yml`):**

1. **Activity-log action keys use dot notation as a deliberate Epic 24 namespace.** Examples already in this story's AC 113-116: `catalog_request.fulfilled`, `catalog_request.rejected`. The dot is a semantic separator; do NOT rewrite to snake_case.
2. **RLS is ENABLE-only ‚Äî no `CREATE POLICY` blocks.** Important for this story: the original AC 7 of Story 24.1 mentioned an "admin override" CREATE POLICY for cross-workspace access on `CatalogIngestRequest`. **That policy is NOT needed and should NOT be added.** Admin actions in this story already get cross-workspace access automatically because Prisma connects as the postgres role (BYPASSRLS) ‚Äî admin queries via `prisma.catalogIngestRequest.findMany({})` (no workspace filter) will return rows from all workspaces by default. The `getAdminSession()` check at the top of each server action is the only authorization gate. Workspace isolation for non-admin paths is enforced by `withWorkspace` + workspace_id filters in user-facing actions, not by RLS.
3. **Primary keys use `@default(uuid())` not `@default(cuid())`.** Not directly applicable (no new models in this story).

These three points were spec-text vs project-pattern divergences in the original Epic 24 stories; PO ratified the project-pattern interpretation. They are NOT to be re-litigated in this story's QA review.

**Key decisions baked in:**
- **Sort by age ascending** ‚Äî surface SLA-breach risks first. Within same age bucket, stable sort by import_id avoids visual jumping when new rows arrive.
- **No real-time push** ‚Äî ops checks the queue periodically. SWR pip refresh = 60s. Consider Pusher / Realtime if the volume warrants; for now polling is fine.
- **Fulfilment requires legal_document_id** ‚Äî ops must have actually ingested the document first. The catalog-requests queue is downstream of ingestion, not coupled to it. Document IDs are public-ish in admin views.
- **Rejection doesn't auto-flip the import row** ‚Äî keeps user agency intact. They might still want to manually accept an ambiguous match instead of letting the rejection stand.
- **Idempotent fulfilment** ‚Äî re-fulfilling the same request is a no-op + no double-email. Critical for ops who might double-click.
- **Activity log scope** ‚Äî workspace activity log surfaces the fulfilment/rejection on the originating workspace's timeline, not just admin-side. So users see "Katalogtill√§gg hanterat" alongside their own actions.

**Out of scope explicitly:**
- Real-time push (queue uses SWR polling)
- Bulk-fulfilment (one at a time for now)
- Auto-fulfilment heuristics (no "match by SFS-number to existing doc" automation)
- Cross-workspace duplicate detection (if two workspaces request the same source row, show two requests; ops can reject the duplicate)
- Re-ingestion workflow (if a fulfilled doc later gets revoked, no auto-cascading; manual cleanup)
