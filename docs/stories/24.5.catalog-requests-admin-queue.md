# Story 24.5: Catalog requests — admin queue + 24h SLA fulfilment

## Status

Draft

## Story

**As a** Laglig.se ops team member responsible for keeping the legal catalog current,
**I want** an admin queue showing all pending "request to add" items from imports — sorted by age with SLA breach indicators — and a one-click fulfilment flow that re-matches the originating row and emails the user,
**so that** the 24h manual-ingestion SLA is operationally measurable and switchers see their unmatched rows resolved without follow-up support tickets.

This is the **ops-facing back-office surface** for `CatalogIngestRequest` rows created by Story 24.4. Closes the 24-hour loop: user requests → ops sees → ops fulfils → original row auto-rematches → user gets email.

## Acceptance Criteria

### Admin route + page

1. **New route `app/admin/(dashboard)/catalog-requests/page.tsx`** — server component:
   - Reads `?status=pending|fulfilled|rejected|all` (default `pending`) and `?range=7|30|90` (default `30`) URL params
   - Calls `listPendingCatalogRequests({status, rangeDays})` server action (skeletoned in 24.1, implemented here)
   - Renders `<CatalogRequestsList>` client component

2. **Admin auth.** Page is gated by the existing admin layout at `app/admin/(dashboard)/layout.tsx` which calls `getAdminSession()` from `lib/admin/auth.ts` and redirects to `/admin/login` if absent. The new `catalog-requests/page.tsx` lands inside that layout group automatically — no per-page auth wiring needed. Server actions in `app/actions/catalog-ingest-request.ts` use the same admin-session check (mirror existing `app/actions/admin/*.ts` pattern; if no canonical wrapper exists, inline the `getAdminSession()` check at the top of each action with an early return on null).

3. **Page layout** uses Epic 22 primitives:
   - `<PageHeader>`: title `Katalogtillägg`, meta `{pendingCount} väntande · SLA bryts om {breachCount}` (red text if breachCount > 0)
   - `<TableToolbar>`: filter chips (Väntande / Hanterade / Avvisade / Alla) with counts
   - shadcn `<Table>` with columns: SLA-status (red/amber/green dot) · Källrad-titel · Workspace · Begärd av · Ålder · Hanterare · [Hantera ▶ button]

### Listing + sort logic

4. **Sort order:** by `created_at` ascending — oldest first. Within same minute, secondary sort by import_id for stability.

5. **SLA-tier visual cue per row:**
   - Red dot — age > 24h (SLA breached)
   - Amber dot — age 12–24h (approaching breach)
   - Green dot — age < 12h (within budget)
   - Use `<Badge tone>` from Epic 22's primitive: `danger` / `warning` / `success` variants

   **Worked example:** for `created_at = '2026-05-04T08:00:00Z'`, current time `'2026-05-06T12:00:00Z'`:
   - Age = 52h → red dot → tone `danger` → "SLA-brytt"
   - For `created_at = '2026-05-06T00:00:00Z'`: age 12h → amber → tone `warning` → "Närmar sig deadline"
   - For `created_at = '2026-05-06T10:00:00Z'`: age 2h → green → tone `success` → "Inom budget"

6. **Row click expands an inline detail panel** (or opens a side drawer — pick whichever matches the existing admin pattern):
   - Source row context: full original row (titel, sfs_nummer, område, lagansvarig, kommentar) — read from `LawListImportRow` joined data
   - Origin: workspace name + workspace id (clickable to admin workspace detail) + import filename + import created_at
   - Requester: user name + email
   - Admin note field (`<Textarea>`) — pre-filled with current `admin_note` if any
   - Two action buttons: **Markera hanterad** (primary) + **Avvisa som dubblett** (ghost destructive)

### Fulfilment flow — "Markera hanterad"

7. **"Markera hanterad" CTA opens a modal** (`<Dialog>`) with:
   - Required field: **LegalDocument ID** that has been ingested. Implementation: simple text input + "Validera" button that calls `prisma.legalDocument.findUnique({where: {id}, select: {id, title, sfs_number, document_type}})` server-side. **Validation contract:**
     - Empty input → toast.error "Ange ett dokument-id"
     - Lookup returns null → inline form-error "Inget dokument med det id:t hittades"
     - Lookup returns row → render the matched title + sfs_number inline below the input as visual confirmation, enable Bekräfta button
   - Required confirm checkbox: "Jag bekräftar att dokumentet är fullständigt ingested och kan matchas mot källraden."
   - Optional admin note (carries forward from detail panel)
   - **Bekräfta** primary CTA → calls `fulfillCatalogRequest({requestId, fulfilledWithDocumentId, adminNote?})` server action. Disabled until both validation passed AND checkbox ticked.

8. **`fulfillCatalogRequest({requestId, fulfilledWithDocumentId, adminNote?})`** — implements the skeleton from 24.1. Steps (single Prisma transaction):
   - Verify request status = `PENDING`
   - Verify `fulfilledWithDocumentId` resolves to a `LegalDocument` row
   - Update `CatalogIngestRequest`: `status='FULFILLED'`, `handler_user_id=ctx.userId`, `fulfilled_with_document_id=...`, `admin_note=...`, `fulfilled_at=NOW()`
   - Update the corresponding `LawListImportRow`: `match_status='CATALOG_REQUEST_FULFILLED'`, `matched_document_id=fulfilledWithDocumentId`, `confidence_score=1.0`, `match_reasoning='Manuellt ingesterat av ops'`
   - Trigger email: `emails/catalog-request-fulfilled.tsx` to the requesting user (see AC 11)
   - Activity-log: `catalog_request.fulfilled` (new entity type + action)
   - Return `{success: true}`
   - **Idempotency:** if request already `FULFILLED`, return success with note that it was already fulfilled; do not re-email.

### Rejection flow — "Avvisa som dubblett"

9. **"Avvisa som dubblett" CTA** opens a smaller confirmation:
   - Optional reason text (saved to admin_note)
   - **Avvisa** confirm → calls `rejectCatalogRequest({requestId, adminNote?})` (new server action)

10. **`rejectCatalogRequest({requestId, adminNote?})`** — new server-action implementation:
    - Verify request status = `PENDING`
    - Update `CatalogIngestRequest`: `status='REJECTED'`, `handler_user_id=ctx.userId`, `admin_note=...`, `rejected_at=NOW()`
    - Leave the `LawListImportRow` unchanged (still `CATALOG_REQUEST_PENDING` from the user's perspective — they can manually accept/reject in the review surface separately, or we could auto-flip; see AC below)
    - Activity-log: `catalog_request.rejected`
    - **No email sent** on rejection (intentional — avoids cluttering user inboxes with negative outcomes; user can see rejection state if they revisit the import)

### Email — fulfilment notification

11. **New email template `emails/catalog-request-fulfilled.tsx`** — React Email component:
    - Subject: "Vi har lagt till {sfs_or_title} i katalogen"
    - Body:
      ```
      Hej {firstName},

      Du begärde att vi skulle lägga till "{source_titel}" i katalogen för din import {filename} ({import_created_at}). Det är nu klart.

      Raden har automatiskt matchats mot {legal_document_title} och är nu en del av din lista {law_list_name} (om listan redan var skapad). Annars laddar du om granskningsvyn för att se den.
      ```
    - CTA: "Öppna listan →" or "Återgå till granskning →" depending on whether import is `COMMITTED` or `AWAITING_REVIEW`
    - Wraps content in `emails/components/laglig-email-layout.tsx` (existing layout primitive)
    - Send via `lib/email/email-service.ts` (canonical send module — uses Resend SDK; `SendEmailOptions` accepts `{ to, subject, react }`). The `lib/email/send.ts` path mentioned in some other stories does NOT exist — always use `email-service.ts`.

### Compatibility & out-of-scope guards

19. **No changes to Story 24.4's review surface.** Once a row is fulfilled here, the user sees the updated state next time they visit the review page (or via the email link). No real-time push.

20. **Existing admin pages unchanged** — additive sidebar entry + dashboard card only.

21. **Rejection of a request does NOT auto-flip the originating LawListImportRow.** The row stays in `CATALOG_REQUEST_PENDING` state from the user's perspective. They can re-decide via the review surface (accept a different match, or `rejectRow`). This is intentional — ops rejection is a "we're not adding this" signal, not an "this row is now invalid" signal.

## Dev Notes

**Source artefacts:**
- Epic PRD: `docs/prd/epic-24-import-existing-law-list.md` — see "Story 24.5" for SLA semantics, fulfilment auto-rematch, rejection-does-not-affect-row decision
- Brief: `docs/import-law-list-brief.md` — strategic context: "the 24h SLA is an ops commitment, so the 'request to add' queue needs to be visible to the ingestion team"
- Story 24.1 — schema for `CatalogIngestRequest` and `LawListImportRow`
- Story 24.4 — `requestCatalogAdd` action that creates rows in this queue

**Existing patterns to follow:**
- **Admin route protection** — handled at the layout level (`app/admin/(dashboard)/layout.tsx` calls `getAdminSession()` from `lib/admin/auth.ts`). New page lands inside the layout group automatically. No per-page wiring needed.
- **Admin-action wrapping** — verify if `app/actions/admin/*.ts` has a canonical `withAdminAuth(cb)` wrapper. If yes, use it. If no, inline `getAdminSession()` early-return at the top of each server action and add a TODO to create the wrapper as a follow-up.
- **`<PageHeader>` + `<TableToolbar>` + shadcn `<Table>` + `<FilterChip>` + `<Badge tone>`** — Epic 22 primitives, all confirmed at `components/ui/`. Use exactly as on `/admin/dashboard/users` and `/admin/dashboard/workspaces`.
- **SWR pip count** — see existing sidebar pip count patterns (e.g., the unread-changes count on the workspace sidebar). 60s refresh interval is the convention.
- **Email send** — `lib/email/email-service.ts` (canonical send module — uses Resend SDK; takes `{ to, subject, react }`). React Email components in `emails/`, wrap in `emails/components/laglig-email-layout.tsx`.
- **Admin sidebar location** — `components/admin/admin-sidebar.tsx` (NOT `components/layout/...`).
- **Activity-log entity type** — `lib/activity/action-constants.ts`, `categories.ts`, `format-activity.ts`. Add new entity type `catalog_request` to all three.

**Key decisions baked in:**
- **Sort by age ascending** — surface SLA-breach risks first. Within same age bucket, stable sort by import_id avoids visual jumping when new rows arrive.
- **No real-time push** — ops checks the queue periodically. SWR pip refresh = 60s. Consider Pusher / Realtime if the volume warrants; for now polling is fine.
- **Fulfilment requires legal_document_id** — ops must have actually ingested the document first. The catalog-requests queue is downstream of ingestion, not coupled to it. Document IDs are public-ish in admin views.
- **Rejection doesn't auto-flip the import row** — keeps user agency intact. They might still want to manually accept an ambiguous match instead of letting the rejection stand.
- **Idempotent fulfilment** — re-fulfilling the same request is a no-op + no double-email. Critical for ops who might double-click.
- **Activity log scope** — workspace activity log surfaces the fulfilment/rejection on the originating workspace's timeline, not just admin-side. So users see "Katalogtillägg hanterat" alongside their own actions.

**Out of scope explicitly:**
- Real-time push (queue uses SWR polling)
- Bulk-fulfilment (one at a time for now)
- Auto-fulfilment heuristics (no "match by SFS-number to existing doc" automation)
- Cross-workspace duplicate detection (if two workspaces request the same source row, show two requests; ops can reject the duplicate)
- Re-ingestion workflow (if a fulfilled doc later gets revoked, no auto-cascading; manual cleanup)
