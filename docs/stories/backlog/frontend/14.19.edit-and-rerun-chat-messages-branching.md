## Story

**As a** user chatting with the compliance agent,
**I want** to edit a previously sent message and rerun the assistant from that point — while still being able to toggle back to my earlier version of the question and its response,
**so that** I can iterate on prompts without losing prior answers (which I may want to compare or return to).

## Context

Today, `components/features/ai-chat/chat-message.tsx` lets users delete their own messages and the assistant turns that followed. To rephrase a question, the user must delete and retype from scratch. ChatGPT, Claude.ai, and Cursor all solve this with **branching**: editing a user message creates a sibling under the same parent, and the UI shows a `< 2 / 3 >` toggle so the user can switch between versions of the question (and the divergent assistant subtrees that follow each one).

The current `ChatMessage` model is a flat ordered list scoped by `(workspace_id, context_type, context_id, conversation_id)` and ordered by `created_at`. To support branching, we add two nullable self-referencing pointers:

- `parent_message_id` — points to the message this one replies to (assistant→user, user→assistant, or user→null for the conversation root).
- `active_child_message_id` — points to which child is currently shown in the UI. Walking from the root and following `active_child_message_id` produces the flat `UIMessage[]` array that the AI SDK consumes.

This model is **additive** (no destructive migration), preserves all existing data via a one-time linear backfill, and requires no new tables. The AI SDK doesn't know or care that the DB is a tree — at the boundary, we project the active path into a flat array and feed it to `setMessages`.

The backend chat route (`app/api/chat/route.ts`) is **stateless** and remains unchanged — it consumes whatever `messages` array the client sends.

## Acceptance Criteria

**Schema & Data:**

1. `ChatMessage` gains two nullable self-FK columns: `parent_message_id` (the message this one is in reply to) and `active_child_message_id` (which child is currently surfaced in the UI). Both have `onDelete: SetNull`.
2. `ChatMessage` gains an optional `edited_at: DateTime?` column for surfacing a "(redigerad)" hint in the UI and for analytics.
3. A migration backfills existing rows: within each `(workspace_id, context_type, context_id, conversation_id)` scope ordered by `created_at`, each message's `parent_message_id` is set to the previous message's id, and each message's `active_child_message_id` is set to the next message's id. The scope's first message has `parent_message_id = NULL`; the last has `active_child_message_id = NULL`.
4. The migration is idempotent (safe to re-run) and is verified against staging data before production.

**Server Actions:**

5. New action `editMessageAsBranch(messageId, newContent)` in `app/actions/ai-chat.ts`:
   - Verifies the target message is owned by the caller's user and workspace, and `role === 'USER'`.
   - Creates a new `ChatMessage` row with the same `parent_message_id`, `workspace_id`, `user_id`, `context_type`, `context_id`, `conversation_id` as the target, `role = USER`, `content = newContent`, `edited_at = now()`.
   - Updates the parent's `active_child_message_id` to point to the new sibling. (If `parent_message_id` is null — i.e., the edit is on the root user message — the new sibling becomes the new root candidate; the active-root pointer is resolved per AC 7.)
   - Wraps both writes in a single Prisma transaction.
   - Returns the new sibling's id so the client can immediately request a regenerate.
6. New action `setActiveBranch(messageId)` updates the parent's `active_child_message_id` to point to `messageId`. If the message is a root (no parent), it updates the conversation's active-root pointer (see AC 7).
7. **Active-root resolution** — because root user messages can have siblings (an edit on the very first turn), the "root" of an active path is determined by: (a) the row in the scope with `parent_message_id IS NULL` and `is_active_root = true`, OR (b) if no such flag exists yet, the most recently created root. Add a boolean `is_active_root` column on `ChatMessage` (default `false`) and set it to `true` on the active root for each scope; flip it during `setActiveBranch` calls on root edits. This avoids needing a separate `Conversation` table.
8. New action `getActivePath(scope)` replaces the read path of `getChatHistory` for active-conversation reads:
   - Fetches all rows in scope in one query.
   - Builds an in-memory tree keyed by `id`.
   - Walks from the active root following `active_child_message_id` pointers, returning a flat array.
   - Returns sibling metadata for each user message that has siblings: `{ messageId, siblings: [{ id, position, total }] }` so the UI can render the `< n/m >` toggle without a second query.
9. Existing `archiveConversation` and `loadConversation` actions continue to work — they operate on `conversation_id`, which is preserved by branching. Loading an archived conversation walks the active path within that `conversation_id` scope.
10. Existing `deleteChatMessage` cascades down `active_child_message_id` only — sibling subtrees are preserved (deleting one branch doesn't nuke another). If the user deletes the active root, the next-most-recent sibling is promoted (if any), otherwise the conversation is empty.

**Hook & UI Integration:**

11. `useChatInterface` is updated to call `getActivePath` instead of the current linear `getChatHistory` for active conversations. Returned `messages` reflect the active path; sibling metadata is exposed as a separate map `siblings: Map<messageId, { position, total, siblingIds }>`.
12. `useChatInterface` exposes new methods:
    - `editMessage(messageId, newContent)`: calls `stop()`, calls `editMessageAsBranch`, replaces local `messages` with the new active path (which now includes the new sibling and excludes the old subtree), then calls `regenerate()`.
    - `switchBranch(messageId)`: calls `stop()`, calls `setActiveBranch(messageId)`, refetches the active path, replaces local `messages`. Does **not** trigger regenerate (the assistant subtree under the switched-to branch is already persisted).
    - `canEdit: boolean` — true when `status === 'ready' || status === 'error'`.

**Frontend UI:**

13. Each user message displays an **edit (pencil) icon** in its hover-revealed action row, alongside the existing delete control. Uses `Pencil` from `lucide-react`.
14. Clicking edit converts the message into an inline `Textarea` pre-filled with the current text, with two buttons: **Spara & kör igen** (primary) and **Avbryt** (secondary). `Enter` (no Shift) saves; `Esc` cancels; `Shift+Enter` adds a newline.
15. User messages that have siblings (i.e., have been edited at least once) display a `< 2 / 3 >` toggle below the message bubble. Clicking arrows calls `switchBranch` with the previous/next sibling id. The toggle shows the current position and total (e.g., `< 2 / 3 >`).
16. User messages that have been edited (have siblings or `edited_at !== null`) show a small subtle "(redigerad)" label next to the timestamp.
17. Edit icon is disabled (hidden or grayed with tooltip "Vänta tills svaret är klart") when `canEdit === false`. Switch-branch arrows are also disabled during streaming.
18. Edit affordance is **only** on user messages, never on assistant messages. Assistant messages keep their existing actions (copy / thumbs / regenerate).
19. Cancel restores the message to its pre-edit text without any DB writes or local state changes to messages.

**Quality & Compatibility:**

20. Backfill migration leaves zero orphan rows (every row either has `parent_message_id IS NULL` and `is_active_root = true`, or has a valid `parent_message_id` referencing an existing row in the same scope).
21. Existing chat behavior (send new message, receive response, delete message, archive conversation, load archived conversation) continues to work unchanged.
22. Tests:
    - Server-action unit tests in `tests/unit/app/actions/ai-chat.test.ts` for `editMessageAsBranch`, `setActiveBranch`, `getActivePath`, and the modified `deleteChatMessage` cascade behavior.
    - Backfill migration test (script + assertion) verifying tree integrity on a fixture dataset of 50+ messages across 5+ scopes.
    - Component tests in `tests/unit/components/features/ai-chat/chat-message.test.tsx` for: edit icon visibility, inline editor keyboard handlers, sibling toggle rendering, switch-branch click behavior.
    - Hook test in `tests/unit/lib/hooks/use-chat-interface.test.ts` (or extend existing) for `editMessage` and `switchBranch` flows including the abort-in-flight-stream guard.
23. Manual E2E pass: edit a mid-conversation message → verify new assistant turn streams in, old branch is preserved in DB, `< 2/2 >` toggle appears, switching back shows the old assistant response, switching forward returns to the new one. Repeat after archiving and reopening the conversation.
24. Vercel Analytics events fire: `ai_chat_edit` (on save) and `ai_chat_branch_switch` (on toggle), each with `{ messageId, contextType }`.

## Dev Notes

### Tree Model — Worked Example

Conversation timeline:

```
[root user] "Vad gäller för semester?"
    └─ [assistant] "Semesterlagen säger ..."
        └─ [user] "Och vid sjukdom?"
            └─ [assistant] "Då gäller karensregler ..."
```

User edits "Och vid sjukdom?" → "Vad händer vid arbetsskada?". The tree becomes:

```
[root user] "Vad gäller för semester?"
    └─ [assistant] "Semesterlagen säger ..."     ← assistant.active_child_message_id flips
        ├─ [user A] "Och vid sjukdom?"           ← original branch (still in DB)
        │   └─ [assistant] "Då gäller karensregler ..."
        └─ [user B] "Vad händer vid arbetsskada?"   ← new active branch
            └─ [assistant] (newly streamed)
```

UI on user message B shows `< 2 / 2 >`. Clicking `<` calls `setActiveBranch(userA.id)`, which sets `assistant.active_child_message_id = userA.id`. Re-walking the active path from the root yields the original branch. The new branch is preserved in the DB but not currently surfaced.

### Why Per-Message `active_child_message_id` (not a Conversation table)

Alternatives considered:

| Approach | Pros | Cons |
|---|---|---|
| `Conversation` table with `active_leaf_message_id` | Single pointer per conversation; clean | Requires new table + relation; more invasive migration; conversation_id is currently a tag, not an entity |
| Per-message `active_child_message_id` | Additive; no new table; tree walks naturally; fine-grained | Requires `is_active_root` to disambiguate root edits |
| Materialized `active_path` array on Conversation | Fast reads | Denormalized; complex writes; sync hazards |

Per-message pointer wins for the brownfield context — minimal schema disruption, leverages the existing `conversation_id` scoping unchanged.

### Backfill SQL Sketch

```sql
-- Within the migration after column adds:
WITH ordered AS (
  SELECT
    id,
    workspace_id, context_type, context_id, conversation_id,
    LAG(id)  OVER (PARTITION BY workspace_id, context_type, context_id, conversation_id ORDER BY created_at) AS prev_id,
    LEAD(id) OVER (PARTITION BY workspace_id, context_type, context_id, conversation_id ORDER BY created_at) AS next_id,
    ROW_NUMBER() OVER (PARTITION BY workspace_id, context_type, context_id, conversation_id ORDER BY created_at) AS rn
  FROM chat_messages
)
UPDATE chat_messages cm
SET
  parent_message_id = o.prev_id,
  active_child_message_id = o.next_id,
  is_active_root = (o.rn = 1)
FROM ordered o
WHERE cm.id = o.id;
```

Note: Postgres `LAG`/`LEAD` partition exactly on the scope tuple, ensuring no cross-scope pointers. NULL `context_id` and NULL `conversation_id` are handled correctly by `PARTITION BY` (NULLs group together within the same workspace/context_type, which matches the active-conversation scope semantics).

### Active-Path Walk Performance

For typical conversation sizes (<200 messages per scope), in-memory tree walk after a single `findMany` is fine — no recursive CTE needed. If we ever see scopes with thousands of messages, we can switch to a recursive CTE that walks `active_child_message_id` directly in SQL. Not needed at MVP.

### AI SDK Integration Surface

`useChat` from `@ai-sdk/react` (AI SDK 5) treats `messages` as opaque state. Our integration:

- DB stores a tree.
- `getActivePath` projects the tree to a flat `UIMessage[]`.
- `setMessages(activePath)` feeds the SDK.
- `regenerate()` re-runs streaming from the current tail.
- `stop()` aborts in-flight before any tree mutation, so the in-flight stream's `onFinish` doesn't write a stale assistant turn to the wrong parent.

The SDK does not need to know about the tree. This is the intended escape-hatch design.

Reference: [AI SDK useChat docs](https://ai-sdk.dev/docs/reference/ai-sdk-ui/use-chat).

### Race Conditions to Handle

1. **Edit during stream** — guarded by AC 17; edit icon hidden when `canEdit === false`.
2. **Switch branch during stream** — same guard via AC 17 on toggle arrows.
3. **Double-click on save** — disable save button while the `editMessage` promise is pending.
4. **Concurrent edits from two browser tabs** — last-write-wins on `active_child_message_id`. Acceptable for MVP; tabs will desync and a manual refresh reconciles.
5. **In-flight stream after `stop()`** — `useChat`'s `stop()` aborts the fetch but the route may still process for a few hundred ms. Since the route is stateless and our persistence happens client-side via `saveChatMessage`, this is harmless: an aborted stream simply doesn't trigger a save.

## Risk and Compatibility Check

**Primary Risk:** A bug in the backfill or in `deleteChatMessage`'s cascade scope could orphan or destroy messages. The tree structure is harder to reason about than a flat list, so a single off-by-one in the active-pointer walk could surface as "messages disappeared".
**Mitigation:** Verification script (Task 1.4) run against staging before prod. Comprehensive unit tests for cascade scope (AC 22). Conservative deletion behavior (only cascade down active pointer, never across siblings).
**Rollback:** Schema changes are additive — drop the new columns. App code rollback restores linear behavior. No data loss.

**Compatibility Verification:**

- [ ] No breaking changes to `app/api/chat/route.ts` (still stateless, still consumes `messages: UIMessage[]`).
- [ ] Database changes are additive only (four new nullable columns + new indexes).
- [ ] Existing `archiveConversation` and `loadConversation` flows preserved.
- [ ] UI changes are net-additive (one new icon + inline editor + sibling toggle); existing actions unchanged.
- [ ] Performance: one extra `findMany` per active-path read, in-memory tree walk is O(n) where n is messages-in-scope (typically <200). Negligible.
