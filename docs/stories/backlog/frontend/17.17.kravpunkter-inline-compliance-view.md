# Story 17.17: Inline Kravpunkter Editor in Efterlevnads Table Row Accordion

## Status

Draft

## Story

**As a** compliance manager or workspace member working through my law list in the Efterlevnads (compliance) view,
**I want** to view and fully edit kravpunkter (add, remove, toggle fulfilment, attach evidence) directly inline from the table row accordion,
**so that** I can triage and update compliance on 20+ laws in one pass without context-switching into the legal document modal.

## Acceptance Criteria

1. In the Efterlevnads table row-expansion accordion (`compliance-detail-table.tsx`), the right card (`Hur efterlever vi kraven?`, backed by `LawListItem.compliance_actions` free text) is **replaced** by an inline instance of the existing `KravpunkterChecklist` component bound to the row's `listItemId`.
2. The left card (`Hur påverkar denna lag oss?`, backed by `LawListItem.business_context`) is **preserved unchanged** — impact context is still valuable and independent of the checklist.
3. Inline kravpunkter supports all capabilities of the existing modal implementation (no behavioral divergence): view the list with fulfilled/unfulfilled state, toggle `is_fulfilled` (auto-saves), inline add a new kravpunkt, inline edit `text`, delete with confirmation dialog, reorder, and link/unlink evidence files and workspace documents via the existing `FilePickerModal` and `DocumentPickerModal`.
4. Empty state when a law list item has zero kravpunkter: reuses the existing empty-state UI from `KravpunkterChecklist` ("Inga kravpunkter definierade" with the "Lägg till din första kravpunkt" prompt and inline add input).
5. When a `LawListItem` has non-empty `compliance_actions` free-text (legacy content authored before the inline kravpunkter migration), surface it as a **one-time dismissable banner** at the top of the kravpunkter area inside the row accordion: *"Tidigare beskrivning: [text] — Dölj."* Dismissal is persisted in `localStorage` keyed by `listItemId`. The underlying `compliance_actions` DB value is **not** deleted or mutated — it remains accessible via the legal document modal.
6. The `businessContext` card continues to render and edit exactly as before (no changes to the left column).
7. The `KravpunkterChecklist` inside the legal document modal (unchanged entry point from Story 17.16) continues to work. This story adds a *second* mount of the same component — it does not replace the modal instance.
8. Fetching: mount `KravpunkterChecklist` only when the accordion row is expanded, so the SWR fetch stays lazy. Both surfaces (modal + table row) share the SWR cache key `list-item-requirements:${listItemId}` — mutations in one surface are reflected in the other on next revalidate without extra code.
9. Progress (`fulfilled / total`) reported by the inline component via its existing `onProgressChange` callback is surfaced in the row's status indicator region where applicable (use the same pattern already applied in `DetailsBox` in the modal; if no clean integration point exists in the compliance table row, the callback can be a no-op — note the decision in the implementation).
10. Read-only mode: when the user lacks `tasks:edit` permission, the inline component renders in read-only mode (existing `readOnly` prop on `KravpunkterChecklist`). Permission check follows the same pattern as the modal — see `complianceReadOnly` threading in `app/(workspace)/laglistor/page.tsx:62`.
11. A unit test asserts the inline mount path: rendering `compliance-detail-table.tsx` with an expanded row containing a mocked list item shows `<KravpunkterChecklist>` with the correct `listItemId`.
12. Existing tests continue to pass: `tests/unit/components/features/document-list/legal-document-modal/kravpunkter-checklist.test.tsx` (8 tests) and `tests/unit/components/features/document-list/compliance-detail-table.test.tsx` (verify no regressions from the card swap).

### Previous Story Insights (Story 17.16 — Done)

- `KravpunkterChecklist` is a **self-contained** client component that handles its own data fetching (SWR), mutations (server actions with optimistic updates + rollback via `toast.error()`), and UI state (collapsibles, inline edit, confirmation dialogs). It exposes a minimal prop surface: `listItemId: string`, `readOnly?: boolean`, `onProgressChange?: (p: { fulfilled: number; total: number }) => void` [Source: docs/stories/completed/17.16.kravpunkter-compliance-checklist.md#Task-5].
- SWR cache key is `list-item-requirements:${listItemId}`. Two concurrent mounts of the component with the same `listItemId` share the same in-flight request and cache entry via SWR's built-in dedup — zero extra plumbing needed for modal ↔ inline sync [Source: docs/stories/completed/17.16.kravpunkter-compliance-checklist.md#Task-5, docs/architecture/21-caching-and-data-fetching-strategy.md#swr].
- Story 17.16 already renamed the modal's accordion label from "Hur efterlever vi kraven?" → "Kravpunkter" and restructured the accordion body into `[KravpunkterChecklist] + separator + [Kommentar (RichTextEditor over compliance_actions)]`. Only the **legal document modal** instance was covered. The compliance detail table row accordion was **not** touched by 17.16 — it still renders the old two-card layout [Source: docs/stories/completed/17.16.kravpunkter-compliance-checklist.md#Task-7, AC 4].
- Permission gating convention: AUDITOR role lacks `tasks:edit`; page computes `complianceReadOnly = !hasPermission(ctx.role, 'tasks:edit')` and threads it through to compliance UI [Source: app/(workspace)/laglistor/page.tsx:62].
- Optimistic update with rollback is the established pattern — see how 17.16 wires it in `KravpunkterChecklist` [Source: docs/stories/completed/17.16.kravpunkter-compliance-checklist.md#Task-5 (checkbox toggle, delete, link)]. No changes needed for this story — behavior inherited.

### Data Models (no changes in this story)

- `LawListItemRequirement` — kravpunkt: `id` (UUID), `list_item_id` (FK → LawListItem, cascade), `text` (String), `is_fulfilled` (Boolean, default false), `position` (Float, default 0), `created_by` (FK → User), timestamps. Index `[list_item_id, position]` [Source: prisma/schema.prisma#LawListItemRequirement lines 353-370].
- `RequirementEvidenceLink` — evidence junction: `id` (UUID), `requirement_id` (FK → LawListItemRequirement, cascade), `file_id` (FK → WorkspaceFile, nullable — XOR), `workspace_document_id` (FK → WorkspaceDocument, nullable — XOR), `linked_at`, `linked_by`. Unique constraints `[requirement_id, file_id]` and `[requirement_id, workspace_document_id]`. XOR enforced at application layer via Zod `.refine()` [Source: prisma/schema.prisma#RequirementEvidenceLink lines 372-392].
- `LawListItem.compliance_actions` — legacy free-text field that backed the old "Hur efterlever vi kraven?" card. Preserved in DB; surfaced as a dismissable banner per AC 5. Not dropped in this story; a follow-up story can migrate + remove [Source: prisma/schema.prisma#LawListItem; Story 17.16 AC 23 explicitly preserved this field].

### API / Server Actions (already in place from Story 17.16 — no changes)

All server actions are at `app/actions/law-list-item-requirements.ts`:

- `getRequirementsForListItem(listItemId)` — returns requirements with linked evidence (files + workspace documents), ordered by position. Used as SWR fetcher by `KravpunkterChecklist`.
- `createRequirement(listItemId, text)`
- `updateRequirement(requirementId, { text?, is_fulfilled? })`
- `deleteRequirement(requirementId)`
- `reorderRequirements(listItemId, orderedIds[])`
- `linkEvidenceToRequirement(requirementId, { fileId?, workspaceDocumentId? })`
- `unlinkEvidenceFromRequirement(requirementId, { fileId?, workspaceDocumentId? })`

All use `withWorkspace('tasks:edit')` for auth, log to `ActivityLog`, and invalidate Redis cache `list-item-requirements:${listItemId}` [Source: docs/stories/completed/17.16.kravpunkter-compliance-checklist.md#Task-2, Task-3].

### Technical Constraints

- **No schema changes.** Per-kravpunkt notes/kommentar is deliberately deferred — if desired later, draft a follow-up Story 17.18.
- **Lazy fetch.** Mount `KravpunkterChecklist` only when the row is expanded. Pairs well with the recent default-collapsed change in commit `c7cea40` (laglistor groups now default collapsed).
- **`localStorage` key naming** for legacy banner dismissal: `laglig:kravpunkter-legacy-banner-dismissed:${listItemId}`. Namespaced with `laglig:` prefix to avoid collision with other features.
- **SSR safety:** guard all `localStorage` access with `typeof window !== 'undefined'` to avoid hydration errors — this is the standard Next.js App Router idiom for client-only browser APIs. The component using localStorage must be a Client Component (`'use client'` directive) or the access must happen inside a `useEffect`.
- **No new packages.** Everything needed is already installed (zustand, swr, shadcn/ui primitives, zod, sonner toasts).
