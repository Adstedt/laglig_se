# Story 2.6: EU Consolidated-Text Ingestion & Amendment Monitoring

## Status

InProgress

## Story

**As a** Swedish SME compliance owner,
**I want** the EU regulations and directives that are actually in force to be ingested in Swedish as clean, current consolidated text — and monitored so that when one of them is amended, corrected, or repealed I get an AI-bedömd notification,
**so that** the EU law on my laglista is trustworthy and up to date, the same way SFS and SKOLFS already are, without drowning me in the tens of thousands of expired or irrelevant EU acts.

## Context

This story **supersedes the corpus strategy of Story 2.5** and adds the forward-monitoring half that 2.4/2.5 never covered. Three decisions from the investigation drive it:

1. **In-force, not all-time.** Story 2.5 targeted "~63,045 regulations + ~4,752 directives" — but that counts every act ever published, including expired and repealed ones. The **in-force** universe available in Swedish is far smaller. A live SPARQL count during investigation returned **7,323** in-force `REG`+`DIR` works with a Swedish expression (may undercount — the `in-force` triple is optional in the CDM; exact count is reconciled in Task A1). This is the corpus. The ~40k "never-amended tail" that would have blown scope back up does **not** exist once you filter to in-force.

2. **Consolidated text is the canonical body.** We store the **latest consolidated version** (CELEX sector `0`, e.g. `02014R0139-20180404`) where one exists — the current text with all amendments already merged — and fall back to the original act for never-amended acts. Consolidated works available in Swedish number ~27,689 across all act types/versions (live count); dedup to one-per-base-act in Task A1. This is a far better reading experience than original + a chain of amendment acts.

3. **Amendment alerts ride the relationship graph, not consolidated diffs.** Consolidation *lags* (weeks–months after the OJ) and is only produced once a change takes *effect* — so it structurally misses future-dated amendments (the `UPCOMING_AMENDMENT` case). The amending act itself appears in CELLAR on OJ-publication day via `cdm:resource_legal_amends_resource_legal`, which the client **already fetches** (`fetchDocumentRelationships`, `AMENDED_BY`). Consolidated-version changes are used only as the *text-refresh* trigger, never as the alert trigger.

**Noise is filtered at the surfacing layer, not at ingestion.** We ingest the whole in-force corpus so nothing crucial can be "missed." Which EU docs appear on a given customer's laglista is decided by `lib/shared/regulatory-area-mapper.ts` + law-list assembly (out of scope here), and monitoring only notifies a workspace when the changed doc is on *its* laglista (`document_id → LawListItem → workspace`). So a user never sees an irrelevant EU change even though the corpus is complete.

**Reuse:** the downstream half of the change pipeline (`ChangeEvent` → `notify-amendment-changes` → `ChangeAssessment`) is **source-agnostic** (confirmed by Stories 9.5/9.6). The EU-specific work is (Phase A) a corpus-scoped ingester and (Phase B) an EU discoverer that emits `ChangeEvent`s. The HTML transformer (`lib/eu/eu-html-transformer.ts`) and linkification pipeline (`lib/linkify/`) already exist and are tested. NIM-via-SPARQL was proven in Story 2.5.

### Relationship to prior stories

- **2.4** (Done, partial): built `lib/external/eurlex.ts`, `EuDocument` model, and ingested original-act documents year-by-year. **Current DB baseline (audited 2026-07-01) — see Dev Notes:** 6,190 EU docs (6,146 reg + 44 dir); 5,588 have HTML but only **9 are normalized** (GDPR + 3 more regs on 2026-04-02, + 5 test directives on 2026-02-13 — a law-list prototype set), **~0 linkified**, **0 chunked/embedded**, and monitoring metadata (`in_force`/`amended_by`/dir-codes/NIM) is **empty even for GDPR**. So today's rows are a raw seed with a 9-doc showcase — not production content.
- **Migration = upsert-in-place, not wipe.** These rows are keyed by base CELEX (same identity 2.6 uses), so the ingester upserts onto them — preserving `document_id`s and any law-list/cross-ref/assessment references. Do NOT delete-and-reingest. Reconcile in three buckets: in-corpus → update; new (all directives, missing regs) → insert; in-DB-but-not-in-corpus (expired) → mark `REPEALED`/`ARCHIVED`, never silently delete. **Data-quality:** some `document_number`s are malformed (e.g. CBAM stored as `"Regulation (EU) 2023/956"`, not `32023R0956`) — the reconciliation must normalize these to base CELEX or it will create duplicates.
- **2.4b / 2.5** (Draft): the "full ~67k original-act ingestion + HTML transform + linkify + NIM" plan. Phase A here **replaces** the corpus scope (in-force consolidated, not all-time original) and **reuses** its transformer/linkify/NIM work. 2.5 can be closed as superseded once this is approved.

## Acceptance Criteria

### Phase A — Corpus definition & staged ingestion (AC 1–12, incl. 3a dry run)

1. A SPARQL corpus query in `lib/external/eurlex.ts` returns the target set: **`REG` + `DIR` works with a Swedish expression that are `resource_legal_in-force = true` OR have a future `resource_legal_date_entry-into-force`** (~7,323 in-force + ~8 not-yet-in-force adopted acts the flag misses — A1-spike-confirmed; the ~68 future-EIF acts already flagged in-force are inherently included). Each resolved to its **latest consolidated version** via `?cons cdm:act_consolidated_based_on_resource_legal ?base` + max `cdm:act_consolidated_date` where one exists (~2,211), else the original act (~5,112). Consolidated versions dedup to **one row per base act**.
2. **Use strict `in-force = true` — do NOT relax to "no flag + no end-of-validity".** A1 spike proved that relaxing sweeps in ~25k stale acts (32,655 vs 7,323), while only **9** consolidated (i.e. maintained) acts lack the flag — so flagless ≈ un-maintained noise, correctly excluded. Final corpus count + the consolidated/original split are logged and recorded in the Dev Agent Record.
3. **The pipeline runs as three decoupled, independently re-runnable stages — normalization happens POST ingestion, not inline:**
   - **Stage 1 — Ingestion:** fetch consolidated (or original) Swedish HTML from CELLAR REST (`fetchDocumentContentViaCellar`), store the **raw** HTML + `full_text` + `EuDocument` metadata baseline, upserting on base CELEX. No transform yet.
   - **Stage 2 — Normalization:** a separate pass reads stored raw HTML → `transformEuHtml()` → `linkifyHtmlContent()` → writes normalized+linkified HTML back to `html_content`. Runs without re-fetching from CELLAR, so it can be re-run cheaply when the transformer improves.
   - **Stage 3 — Chunk + embed:** a separate pass chunks + embeds the normalized text for RAG.
   Decoupling means a transformer fix re-runs only Stage 2+3, and an embedding change re-runs only Stage 3 — no CELLAR re-fetch. Each stage is idempotent and independently resumable.
3a. **Dry run first (gate before full run):** run all three stages end-to-end on **10 representative docs** (mix: chaptered/flat/minimal, ≥1 directive, ≥1 with a consolidated version, ≥1 never-amended, plus GDPR as a known-good baseline) and verify: renders with correct TOC/preamble in the reader, references linkified, chunks+embeddings created, and `EuDocument` monitoring baseline populated. Only proceed to the full ~7.3k run once the 10-doc dry run passes. Provide a `--limit N` / `--celex <list>` flag to scope the run.
4. `LegalDocument` rows use `content_type = EU_REGULATION | EU_DIRECTIVE`; `document_number` = the **base-act CELEX** (e.g. `32014R0139`, not the dated consolidated CELEX), so the identity is stable across future consolidations.
5. `EuDocument` metadata is populated as the **monitoring baseline** — critically `in_force`, `end_of_validity`, `amended_by_celex`, `corrected_by_celex`, `transposition_deadline` (directives), plus `eli_identifier`, `directory_codes`, `subject_matters`, `eurovoc_concepts`, `authors`, `legal_basis_celex`, `cites_celex`, `eea_relevant`, `signature_date`.
6. The **current consolidated CELEX** (dated, e.g. `02014R0139-20180404`) and its consolidation date are stored (on `EuDocument.metadata`) so Phase B can detect when a *newer* consolidated version appears. Null for never-amended acts.
7. NIM cross-references built per Story 2.5: `fetchSwedishNIM()` populates `EuDocument.national_implementation_measures`, and `CrossReference` rows with `reference_type = IMPLEMENTS` link matched SFS laws → EU directives.
8. Linkified references to SFS/agency/court docs saved as `CrossReference` with `reference_type = REFERENCES` via `saveCrossReferences()`; idempotent re-linkify (existing `<a class="legal-ref">` stripped first).
9. Ingestion is **checkpointed/resumable** and does **sequential** DB writes (respect `connection_limit=1`; no parallel workers — the 2.4 pool-exhaustion root cause). Per-doc chunk-write concurrency bounded (9.6 pool-hardening).
10. CELLAR rate limiting: 2 req/sec, HTTP 429 `Retry-After` honored, exponential backoff (2→4→8→16s, max 5). Correction-suffix CELEX (`R(01)` etc.) that 404 are skipped.
11. Stage 3 chunks + embeds the normalized text for RAG (existing `process-chunks` path / same hook as SFS & agency regs). Baseline today: **0** EU docs are chunked — this stage brings all ingested EU docs into search/RAG for the first time.
12. Each stage's re-runs are idempotent (`--stage ingest|normalize|embed`, plus `--force`): existing docs are updated **in place** (upsert on base CELEX) without duplicating rows or clobbering issuer attribution; the normalized-vs-raw check uses the `class="legal-document"` signal (see Dev Notes), NOT `class="sfs"`.
12a. **Reconciliation of the existing 6,190 rows:** malformed `document_number`s (e.g. CBAM `"Regulation (EU) 2023/956"`) are normalized to base CELEX before matching; docs present in the DB but absent from the in-force corpus are marked `REPEALED`/`ARCHIVED` (never hard-deleted, to preserve law-list/cross-ref/assessment references); the run logs the three buckets (updated / inserted / retired).

### Phase B — Amendment monitoring (AC 13–22)

13. A **`discover-eu-changes` cron** (`app/api/cron/discover-eu-changes/route.ts`) follows the standard cron scaffold (CRON_SECRET auth, `startJobRun`/`completeJobRun`/`failJobRun`, `maxDuration = 300`, batched/limited per run to fit the Vercel window) and models its detector on the SKOLFS pattern (`lib/agency/skolfs-detector.ts` + a pure `classifyEuDiff()` in `lib/eu/eu-change-detection.ts`).
14. The detector re-runs `fetchDocumentRelationships()` over the corpus (batched, 50/CELEX) and diffs the fresh `amended_by` / `corrected_by` sets against the stored `EuDocument.amended_by_celex` / `corrected_by_celex` baseline.
15. A **new** CELEX in `amended_by` → fetch the amending act's `date_entry-into-force`; emit `ChangeEvent` with `change_type = AMENDMENT` if the effective date is today/past, or **`UPCOMING_AMENDMENT`** if future-dated (with `effectiveDate` in the payload). The amending CELEX is the dedup key.
16. A **new** CELEX in `corrected_by` → `ChangeEvent` with `change_type = METADATA_UPDATE` (Rättelse/corrigendum).
17. `resource_legal_in-force` flipping to false, or an `end-of-validity` date reached → `change_type = REPEAL`.
18. For **directives**, a `transposition_deadline` in the future (newly seen) → `UPCOMING_AMENDMENT` framed as an implementation deadline; NIM additions (a new Swedish measure appears) may update the `IMPLEMENTS` graph.
19. New in-force acts within the corpus definition that are **not yet in the DB** are detected (forward feed, analogous to `sync-sfs`), ingested via the Phase A pipeline, and emit `NEW_LAW` (PO decision: `NEW_LAW` events are created but only notify workspaces once the doc is on a laglista — same as every other source).
20. Each `ChangeEvent` is created with `ai_summary` **set in-detector** (bypassing the SFS-coupled `enrich-amendment-summaries` join, per the 9.6 pattern), `content_type` set, `notification_sent: false`; deduped on `(document_id, amendment_sfs = amending/correcting CELEX, change_type)`. `LegalDocument.last_change_{type,ref,at}` updated.
21. On detection of a **newer consolidated version** (a dated sector-`0` CELEX later than the stored one), the doc's Swedish text is **re-fetched/re-transformed/re-linkified/re-embedded** and the stored consolidated-CELEX baseline advanced; any matching `UPCOMING_AMENDMENT` for that act is reconciled to in-force. This is the *text-refresh* path, separate from the alert path.
21a. **Upcoming effective-date surfacing.** At ingest (Phase A), the **full set** of future `resource_legal_date_entry-into-force` and `resource_legal_date_deadline` values per doc is stored on `EuDocument.metadata` (both are **multi-valued** — the AI Act `32024R1689` has 5 EIF + 7 deadline dates). The doc's next-upcoming milestone is exposed for display ("tillämpas fr.o.m. YYYY-MM-DD"), reusing Story 2.15's effective-date-pending status where applicable.
21b. **Upcoming milestone monitoring.** The detector emits an `UPCOMING_AMENDMENT` `ChangeEvent` per newly-seen future milestone (dedup key includes the milestone date), and as each date passes it flips the doc to reflect the now-effective provision (and advances the "next milestone"). This shares the date-scan pass with the REPEAL check (AC 17). **Data-quality:** `end-of-validity = 9999-12-31` is a sentinel for "indefinite" — treat as no end date (do NOT emit REPEAL); ignore past EIF/deadline dates as already-effective.
22. **Notify + assess reuse verified unchanged:** seed a workspace with an EU reg/dir on its laglista, fire a synthetic EU `ChangeEvent`, confirm `notify-amendment-changes` picks it up (no `content_type` filter) and `ChangeAssessment` works. No regression: lint/typecheck/build green, full unit suite green.

### Source-agnosticism map (confirmed 9.5/9.6)

| Stage | Verdict |
|---|---|
| `ChangeEvent` model | Source-agnostic — `EU_REGULATION`/`EU_DIRECTIVE` exist; global, keyed on `document_id` |

| `ChangeAssessment` | Source-agnostic — hangs off any ChangeEvent |
| Discovery | Source-coupled — build EU detector (this story) |

### Corpus definition (the core decision)

**`REG` + `DIR` works with a Swedish expression that are `in-force = true` OR have a future `date_entry-into-force` = ~7,331** (7,323 in-force + ~8 adopted-but-not-yet-in-force the flag misses; A1-spike-confirmed), stored as the latest consolidated version (~2,211) where one exists, else the original act (~5,112). Deterministic, complete-within-in-force-and-upcoming. **Upcoming ≠ a separate corpus:** most crucial upcoming law (GDPR-style, AI Act) is already in via `in-force=true`; the value-add is surfacing the future *milestone dates* it carries (AC 21a/21b), not ingesting more docs. **Not** the all-time ~67k (counts expired/repealed) and **not** the relaxed 32,655 (the ~25k flagless acts are un-maintained noise — only 9 consolidated acts lack the flag, proving the flag reliably marks maintained legislation). Decisions/other act types out of scope (possible later extension). Relevance filtering happens at the surfacing/notification layer, so corpus completeness ≠ user-facing noise.

**Normalization signal correction:** the transformer emits `<article class="legal-document" id="eu-{celex}">` (with `preamble-accordion`, `section.kapitel`, `paragraf` anchors) — **NOT** `class="sfs"` as the Story 2.5 example shows. Any "is it normalized?" check must use `class="legal-document"`. GDPR renders perfectly precisely because it has this schema; the other ~5,579 HTML docs are raw CELLAR markup and would not.

### Why relationship-diff, not consolidated-diff, for alerts

Consolidation lags the OJ by weeks–months and is only produced once a change takes effect → misses future-dated amendments entirely. `cdm:resource_legal_amends_resource_legal` (reverse → `AMENDED_BY`, already in `buildRelationshipsQuery`, `lib/external/eurlex.ts:815`) surfaces the amending act on publication day, carrying its own `date_entry-into-force` → drives AMENDMENT vs UPCOMING_AMENDMENT. Consolidated-version change = text-refresh trigger only (AC 21).

### Data models

- **LegalDocument**: `content_type EU_REGULATION|EU_DIRECTIVE`, `document_number` = base CELEX, `html_content` (transformed+linkified), `full_text`, `metadata` (structureType/stats/consolidated-CELEX+date), `last_change_{type,ref,at}`.
- **EuDocument** (1:1): `celex_number`, `in_force`, `end_of_validity`, `amended_by_celex`, `corrected_by_celex`, `transposition_deadline`, `national_implementation_measures`, `directory_codes`, `subject_matters`, `eurovoc_concepts`, `legal_basis_celex`, `cites_celex`, `eea_relevant`, `signature_date`, `eli_identifier`.
- **ChangeEvent**: `{document_id, content_type, change_type, amendment_sfs (=amending/correcting CELEX), ai_summary, changed_sections, notification_sent:false}`.
- **CrossReference**: `REFERENCES` (linkify) and `IMPLEMENTS` (NIM: SFS → EU directive).
- `ChangeType` already includes `AMENDMENT`, `REPEAL`, `METADATA_UPDATE`, `NEW_LAW`, `UPCOMING_AMENDMENT` — **no migration needed**.

### APIs (grounded)

- SPARQL: `https://publications.europa.eu/webapi/rdf/sparql` (public). CELLAR REST: `https://publications.europa.eu/resource/celex/{CELEX}` (`Accept: text/html`, `Accept-Language: sv`, handles HTTP 300).
- Live counts (A1 spike, 2026-07-01): in-force REG+DIR w/ SV = **7,323** (of which ~2,211 have a consolidated version, ~5,112 never-amended); relaxed filter (rejected) = 32,655; consolidated all-types (sector-0) w/ SV = 27,689 / EUR-Lex UI 27,680.
- Consolidation predicates (A1 spike): `act_consolidated_based_on_resource_legal` (cons→base, single), `act_consolidated_date` (version date), `act_consolidated_consolidates_resource_legal` (folded-in amending/correcting acts). CELEX is a typed literal → always `STR()`-compare.
- Effective-date predicates (spike, from `32024R1689` AI Act): `resource_legal_date_entry-into-force` and `resource_legal_date_deadline` are **multi-valued** (phased application / staggered compliance). `resource_legal_date_end-of-validity = 9999-12-31` = indefinite sentinel (NOT a real end date). Only ~8 REG+DIR (SV) with a future EIF are missed by the in-force flag; ~68 more are already flagged in-force.
- NIM SPARQL (from 2.5): `measure_national_implementing_implements_resource_legal` + `…_implemented_by_country <…/country/SWE>` → 10,470 entries / 2,483 directives.
