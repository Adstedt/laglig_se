# Story 5.13: Upgrade Stripe Node SDK to v22 (API 2026-03-25.dahlia)

## Status

Draft

## Story

**As a** maintainer of the Stripe billing integration,
**I want** to upgrade the `stripe` Node SDK from `^14.21` to `^22.x` and refresh the version pin in our architecture documentation,
**so that** we benefit from active Stripe maintenance, post-2025-03-31.basil API surfaces, and remove a dead-code fallback that exists only to bridge the v14 era.

## Context

Story 5.4 (Stripe billing integration) shipped on `stripe@^14.21` (resolved to `14.25.0`). The QA gate ([`docs/qa/gates/5.4-stripe-subscription-billing.yml`](../../../qa/gates/5.4-stripe-subscription-billing.yml), 3rd-pass entry) flagged that v14 is **8 majors behind** the current Stripe Node SDK (`v22`, mapped to API version `2026-03-25.dahlia`).

The implementation is currently correct against current docs ([Stripe Node releases](https://github.com/stripe/stripe-node/releases), [SDK versioning](https://docs.stripe.com/sdks/versioning)) — the dev defensively wrote a `current_period_end` fallback that handles the basil-era schema change where the field moved from `subscription.current_period_end` onto `subscription.items.data[0].current_period_end` ([changelog](https://docs.stripe.com/changelog/basil/2025-03-31/deprecate-subscription-current-period-start-and-end)). So this is **not a bug fix** — it's debt remediation:

1. Stripe's [versioning support policy](https://docs.stripe.com/sdks/versioning) actively maintains only the latest major; older SDKs receive security fixes only.
2. Newly-provisioned Stripe accounts default to a much newer API version than v14 knows about. When the operator creates the production account today, it pins to `2026-03-25.dahlia` or later.
3. The `current_period_end` fallback is dead code once we're on a v18+ SDK (which always reads from the item).

### Lineage of the stale pin

The version came from [`docs/architecture/3-tech-stack.md`](../../architecture/3-tech-stack.md) row "Payments", which says `Latest API (stripe npm package 14.21+)`. That doc was last updated when v14 was current (late 2023). Story 5.4 (drafted Nov 2025) copied the `^14.21` constraint verbatim. The PO validation pass on Story 5.4 (Sarah, v3.0, 2026-05-04) didn't catch the drift. The dev (James) followed the spec — correctly per the agent-activation contract that prohibits unilateral dependency decisions.

This story closes the loop AND adds a process annotation to the architecture doc so a future tech-stack reader knows when to re-check.

### Existing System Integration

- **Touched packages:** `package.json` (`stripe` dep), `pnpm-lock.yaml`
- **Touched code:** all current Stripe SDK call sites listed in Tasks below
- **Touched docs:** `docs/architecture/3-tech-stack.md` row "Payments"
- **Reuses:** existing test mocking pattern at [`tests/unit/billing/stripe-integration.test.ts`](../../../../tests/unit/billing/stripe-integration.test.ts) — mocks live at our Stripe-SDK boundary, so most tests should pass unchanged after the bump
- **No DB migration. No new external dependencies. No user-visible behavior change.**

### Dead-Code Removal — current_period_end Fallback

6. The defensive fallback in [`app/api/webhooks/stripe/route.ts`](../../../../app/api/webhooks/stripe/route.ts) (the block currently shaped as `item?.current_period_end ?? (subscription as unknown as {...}).current_period_end`) is **simplified to read only from the item**:
   ```ts
   const periodEnd = subscription.items.data[0]?.current_period_end
   ```
   The `as unknown as { current_period_end?: number }` cast is removed. This is safe **only** after the SDK is bumped to v18+ (basil API), which v22 satisfies.
7. Inline comment that previously justified the fallback (referencing v14 / 2024-04-10) is removed in the same edit so the code does not lie about its own history.

## Dev Notes

### Why this is low-risk

- **Mocking surface unchanged.** Tests in [`tests/unit/billing/stripe-integration.test.ts`](../../../../tests/unit/billing/stripe-integration.test.ts) mock at the Stripe SDK boundary (the `stripeMock` object replaces methods on the imported `stripe` instance). Bumping the SDK doesn't move our test boundary. Webhook signature verification uses the **real** `stripe.webhooks.constructEvent` against payloads signed with `realStripe.webhooks.generateTestHeaderString` in-test — this surface has been stable across all major Stripe SDK versions.
- **Defensive code already in place.** The `current_period_end` fallback was written knowing the basil-era schema move would happen. Now we just retire the fallback.
- **No DB migration.** Schema unchanged.
- **No env changes.** Same `STRIPE_SECRET_KEY` etc.
- **Webhook payloads stay backward-compatible** for the events we handle. Stripe maintains long backward-compat windows for webhook payloads independent of SDK majors.

### Migration-style risks to watch for during Task 3

- **`Stripe.Invoice.amount_due`** and other `?: number | null` fields may be tightened to non-optional in v22. Our code already does `?? 0` and `?? null` guards in most places, but verify.
- **`Stripe.Checkout.Session.subscription`** type union may shift; we already handle `string | { id } | null` defensively in [`app/api/webhooks/stripe/route.ts`](../../../../app/api/webhooks/stripe/route.ts) (`typeof session.subscription === 'string'` guard).
- **`Stripe.SubscriptionItem.current_period_end`** is now top-level on the item type in v18+; the cast `as Stripe.SubscriptionItem & { current_period_end?: number }` becomes unnecessary after the bump.

### Out of scope

- Stripe Tax, Customer Portal flow customization, Klarna/Swish enablement — all configured in Stripe Dashboard, not SDK-version-dependent.
- The operator items still pending from Story 5.4 (Stripe products/EUR prices, Vercel env vars, Dashboard webhook URL, AC 12 lifecycle in Stripe Test Mode) are pre-requisites for end-to-end production validation but are NOT this story's scope. They block the *deployment* of Story 5.4, not this *upgrade* story.
- OBS-001 (cache invalidation lag) and OBS-002 (BillingTab placeholder) — both deferred from Story 5.4 and remain out of scope here.
