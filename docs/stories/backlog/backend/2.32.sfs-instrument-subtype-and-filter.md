# Story 2.32: SFS Instrument Subtype & Filter (Lag vs Förordning)

## Status

Ready for Review (Tasks 1–8 implementation complete; final manual UI smoke test pending dev-server restart to clear stale Prisma client cache)

## Story

**As a** user browsing legal documents,
**I want** to distinguish and filter Swedish lagar (laws) from förordningar (regulations) — and from kungörelser — within the SFS catalogue,
**so that** I can correctly understand each document's legal status and narrow listings to the instrument I actually need.

## Context

Today the `ContentType` enum lumps both lagar (laws passed by Riksdagen) and förordningar (regulations issued by the regering under delegated authority) under a single value, `SFS_LAW`. The same conflation exists for `SFS_AMENDMENT` — an amendment can amend either a lag or a förordning. These are legally distinct instruments at different levels of the Swedish legal hierarchy (lag > förordning), but the UI labels both as "Lag" and the catalogue filter cannot separate them.

Splitting `ContentType` itself would touch ~30+ code paths (cross-references, linkifier resolution, slug routing, sitemap, theming, prefetch URLs) for what is fundamentally a display + filter concern. Instead, this story adds a sibling field `sfs_instrument` on `LegalDocument` that captures the document's actual form (`LAG | FORORDNING | KUNGORELSE | OTHER`), keeps `ContentType` stable, backfills from titles, and surfaces it as a new filter dimension on the catalogue plus correct badge labels.

The detection already exists: `lib/sfs/classify.ts` reliably classifies SFS titles via prefix and "balk" patterns with confidence 0.95. We reuse the public `classifyLawType()` export (line 224) for both the one-shot backfill of existing rows and the live sync pipeline so backfill and ingest cannot drift apart.

## Dev Notes

### Why a Sibling Field, Not an Enum Split

A previous design discussion considered splitting `ContentType.SFS_LAW` into two values (`SFS_LAW` + `SFS_ORDINANCE`). That approach was rejected for MVP because every `content_type === 'SFS_LAW'` check across the codebase (themes, badges, slug routing, linkifier resolution, sitemap, filters, cross-references, prefetch URLs) would need review — ~30+ touchpoints. The sibling-field approach scopes the change to display + filter, which is where the user-visible problem actually lives. If a future requirement (e.g. URL structure differing for laws vs regulations) emerges, the enum split can graduate from this foundation.

### Existing Detection Logic to Reuse

The public export to use is **`classifyLawType(title): SfsClassification`** at `lib/sfs/classify.ts:224`. It returns `{type, category, targetType, targetSfs, confidence}` — for our purposes only `.type` and `.confidence` are used. Internally it delegates to a private `detectDocumentType` helper at line 72 (do **not** import that one — it's not exported).

`classifyLawType` already handles:
- Bare `Lag (YYYY:N)` and `Lag YYYY:N` titles → `type: 'lag'`, confidence 0.95
- `Förordning(en) ...` → `type: 'förordning'`, confidence 0.95
- `Kungörelse ...` → `type: 'kungörelse'`, confidence 0.95
- `*balk(en)` patterns (e.g. `Brottsbalk (1962:700)`) → `type: 'lag'`, confidence 0.95 (regex at `classify.ts:109`)
- Amendments: splits on `om ändring`/`om upphävande` (logic at `classify.ts:81-90`) and classifies on the prefix (so `Lag (2024:407) om ändring i förordningen (1998:899)…` correctly classifies as `'lag'`)
- Fallback patterns (`stadga`, `instruktion`, `reglemente`) → `type: 'förordning'`, confidence 0.7

The new `inferSfsInstrument` helper applies the **0.9 confidence threshold**: anything below stays `OTHER` rather than guessing. This keeps the data conservative — we'd rather have OTHER for an ambiguous title than mislabel a förordning as a lag.

**Reference implementation:**

```ts
// lib/sfs/instrument.ts
import { classifyLawType } from './classify'
import type { SfsInstrument } from '@prisma/client'

export function inferSfsInstrument(title: string): SfsInstrument {
  const { type, confidence } = classifyLawType(title)
  if (confidence < 0.9) return 'OTHER'
  if (type === 'lag') return 'LAG'
  if (type === 'förordning') return 'FORORDNING'
  if (type === 'kungörelse') return 'KUNGORELSE'
  return 'OTHER'
}
```

### Filter Conditional-Visibility Rule

The Författningstyp filter is meaningful only when SFS docs are in scope. Concrete rule:

```ts
const sfsInScope =
  selectedTypes.length === 0 ||
  selectedTypes.includes('SFS_LAW') ||
  selectedTypes.includes('SFS_AMENDMENT')
```

When `sfsInScope === false`, hide the section *and* drop any `instrument` value from the prefetch input so it doesn't generate misleading cached results.

UI gracefully falls back to the existing `getDocumentTheme(contentType).label` (i.e. "Lag") when `sfs_instrument` is `OTHER`.

**Rollback Plan:** Drop the column via reverse migration. UI commits revert independently. The `inferSfsInstrument` helper and ingest wiring can stay (write a default value harmlessly) or be removed without affecting any other system.
