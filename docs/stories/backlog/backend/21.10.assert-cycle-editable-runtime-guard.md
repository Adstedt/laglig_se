# Story 21.10 — `assertCycleEditable` runtime guard (defence-in-depth hardening)

## Status

Backlog (deferred from Epic 21 MVP — UAT-ready as-is, see rationale below).

## Story

As an **Epic 21 maintainer**,
I want a **shared `lib/compliance-audit/cycle-guards.ts#assertCycleEditable(tx, cycleId)` helper** wired into every mutation server action that targets a `ComplianceAuditCycle` or `ComplianceAuditItem`,
so that **transaction-participating writes are rejected the moment the cycle becomes SEALED or ARKIVERAD, even if a UI race or a stale optimistic update lets a write request slip past the inline `assertCycleEditableUi` check**.

## Why this is deferred (NOT a UAT blocker)

The inline `assertCycleEditableUi(status)` helpers in `app/actions/compliance-audit-cycle.ts` and `app/actions/compliance-audit-item.ts` already enforce the read-only invariant for SEALED + ARKIVERAD cycles correctly today. Each mutation action loads the cycle (or item-with-cycle), calls `assertCycleEditableUi(existing.cycle.status)`, and returns the Swedish error string `"Kontrollen är fastställd eller arkiverad — ändringar är inte tillåtna."` on failure. This is a structurally identical guard to what 21.10 will deliver — the difference is purely *transactional participation*: the inline helper runs BEFORE the transaction opens (load → check → mutate), whereas 21.10's helper would run INSIDE the transaction so a concurrent seal between the load and the write is also caught.

The race window is narrow (sub-second), and the cycle lifecycle is sequential per-cycle (one user typically owns the audit). 21.10 is therefore tech-debt hardening, not a correctness gap. UAT can proceed without it.

## Acceptance Criteria (high-level — defer detailed breakdown to drafting)

1. New module `lib/compliance-audit/cycle-guards.ts` exports `assertCycleEditable(tx: Prisma.TransactionClient, cycleId: string): Promise<void>` — throws a typed sentinel error when the cycle is SEALED or ARKIVERAD.
2. All `TODO(21.10)` markers in the codebase replaced with calls to the new guard. Grep `TODO(21.10)` should return zero hits after this story ships.
3. Each mutation action wraps its existing logic in `prisma.$transaction(async (tx) => { await assertCycleEditable(tx, cycleId); ... })` so the guard sees the same transactional snapshot as the write.
4. Existing inline `assertCycleEditableUi` helpers either deleted (preferred) or retained only as a pre-tx fast-path bailout that returns the Swedish error without opening a transaction.
5. New unit tests prove the tx-internal guard catches a concurrent-seal race that the inline helper would miss (mock the cycle status changing between `load` and `update`).
6. Existing test matrix (the SEALED/ARKIVERAD `describe.each` block in `compliance-audit-item.test.ts` and the equivalent in `compliance-audit-cycle.test.ts`) stays green after the refactor.
