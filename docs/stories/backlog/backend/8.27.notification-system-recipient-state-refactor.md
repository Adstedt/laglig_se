# Story 8.27: Notification System — Per-Recipient State & Engagement-Based Dedup

## Status

Draft

**Dependencies / sequencing**: Coordinate with Story 8.20 (`docs/stories/backlog/backend/8.20.refactor-notification-cron-content-generation.md`) — both stories modify `app/api/cron/notify-amendment-changes/route.ts`. Recommend landing 8.20 first to shrink the cron's responsibility surface before this rewrite. If 8.27 lands first, 8.20 needs a small rebase.

## Story

**As a** workspace member who has already engaged with an amendment in-app,
**I want** the daily email digest to skip notifications I've already read or assessed,
**so that** I don't get duplicate emails for changes I've already handled.

## Acceptance Criteria

### A. Schema (Phase 1)

1. New `Notification.email_sent_at` (`DateTime?`) and `Notification.dismissed_at` (`DateTime?`) columns are added via additive Prisma migration. A partial index `notifications_email_pending_idx` over `(email_sent_at, read_at, dismissed_at, created_at) WHERE email_sent_at IS NULL AND read_at IS NULL AND dismissed_at IS NULL` accelerates the new digest query.

### B. Engagement signals → mark notification read (Phase 2)

2. New helper `lib/notifications/mark-notification-read.ts` exporting `markChangeEventNotificationsRead(userId: string, changeEventId: string): Promise<void>` updates all unread Notifications matching `(user_id, entity_type='change_event', entity_id=changeEventId, read_at IS NULL)`. Wrapped in try/catch (best-effort, non-blocking — assessment write must NOT fail if the helper throws).
3. Helper invoked from `app/actions/change-assessment.ts` `createOrUpdateAssessment()` immediately after the upsert succeeds (line ~92 in current implementation).
4. Helper also invoked from `lib/agent/tools/save-assessment.ts` `createSaveAssessmentTool()` in the `execute=true` branch after the upsert (line ~95 in current implementation).

### C. Real-time in-app Notification creation (Phase 3)

5. `app/api/cron/discover-sfs-amendments/route.ts` (the verified ChangeEvent creation site per `grep prisma.changeEvent.create` — also re-verify in `lib/sfs/` for any auxiliary creation paths) calls `processChangeEventNotifications([newEventId])` immediately after each `ChangeEvent` insert. In-app notifications appear within seconds of detection rather than 7+ hours later when the digest cron fires.
6. AC 5 is gated behind `process.env.NOTIFICATIONS_REALTIME_INAPP === 'true'`. Default off until staging soak completes; flip on in prod after ≥3-day soak.
7. Idempotency holds: `createChangeNotifications` (`lib/notifications/amendment-notifications.ts:144–153`) already has dedup check; calling it from both sync cron (new) and digest cron (during transition) does not duplicate Notification rows.

### D. Email digest reads from Notification queue (Phase 4 — the actual fix)

8. `app/api/cron/notify-amendment-changes/route.ts` query rewritten from `prisma.changeEvent.findMany({ where: { notification_sent: false } })` to:
   ```ts
   prisma.notification.findMany({
     where: {
       email_sent_at: null,
       read_at: null,
       dismissed_at: null,
       type: { in: AMENDMENT_NOTIFICATION_TYPES },
       created_at: { lt: subMinutes(new Date(), QUIESCENCE_MINUTES) },
     },
     include: { user: true, workspace: true },
   })
   ```
   where `QUIESCENCE_MINUTES = 15` and `AMENDMENT_NOTIFICATION_TYPES = ['AMENDMENT_DETECTED', 'LAW_REPEALED', 'RULING_CITED']`.
9. After successful send, marks delivery on the Notification rows: `prisma.notification.updateMany({ where: { id: { in: notificationIds } }, data: { email_sent_at: new Date() } })`.
10. The old block at lines 444–449 (`changeEvent.updateMany({ notification_sent: true })`) is skipped when the new flag is on; left intact when off (backwards compat).
11. AC 8–10 gated behind `process.env.NOTIFICATIONS_QUEUE_DIGEST === 'true'`.

### E. Cleanup (Phase 5, after 1-2 sprints stable)

12. `ChangeEvent.notification_sent` column dropped via additive migration (reversible). Dead block in `notify-amendment-changes/route.ts` removed. Feature flags removed (new behavior unconditional). Story 8.4 docs updated to reference new architecture.

### Technical Constraints

- **Migrations strictly additive in Phases 1–4.** Phase 5 drops one column; the rollback is restoring it from a migration. No data destruction in any phase.
- **Mark-as-read helper must be best-effort.** Wrapped in try/catch; never throws upward; never blocks the assessment write. A failure here logs and continues — engagement signal is a nice-to-have for the email skip, not a hard dependency.
- **Idempotency required.** `processChangeEventNotifications` is called from BOTH the sync cron (new, Phase 3) AND the digest cron (existing, until Phase 5 cleanup). The dedup check at `lib/notifications/amendment-notifications.ts:144–153` already handles this — verify in tests, don't refactor.
- **Cron schedule unchanged.** `notify-amendment-changes` continues to run daily at 07:00 UTC (`vercel.json`). The 15-min quiescence window is a defensive freshness threshold, not a tight real-time constraint. If/when cron frequency changes (e.g., hourly), the window may need re-tuning.
- **Two independent feature flags.** Both off = current behavior (zero-risk default). Both on = full new flow. Mixed states are safe — see Risk section in Tasks above. Don't combine into a single flag; the staged rollout depends on independent toggling.
- **No new npm packages, no new env vars beyond the two feature flags, no breaking API changes.** This is a pure refactor + bugfix.
- **Manual migration application.** Per project convention (Story 17.17 v1.6, Story 17.18 v2.0), Supabase migrations are applied manually via SQL Editor, not via `prisma migrate deploy`. Document the SQL in the migration file as usual.
- **Coordinate with Story 8.20.** Both touch `notify-amendment-changes/route.ts`. Recommend landing 8.20 first (it shrinks the cron's responsibility surface, making 8.27's Phase 4 rewrite cleaner). If 8.27 lands first, 8.20 will need a small rebase.
- **In-app vs email cadence asymmetry (post Phase 3).** Once Phase 3 is ON, in-app notifications appear in real-time (within seconds of detection) while emails still wait until the daily 07:00 UTC digest. A user might assess in-app immediately, get no email next day (correct, the engagement-suppression fix), but a less-engaged user could wait 18+ hours for the email about a change that's already visible elsewhere. Acceptable for v1 — daily cadence is intentional to avoid email fatigue. **Out of scope for this story**; if real-time emails become a need, scope a follow-up to add hourly digest cron OR per-event email at detection time (with sensible per-user rate-limiting).
