# Story 5.5d: Stripe Metered Overage

## Status

Backlog — DEFERRED

> **Do not pick up this story yet.** Pre-conditions for unblocking are listed below. The whole point of v1 (Story 5.5c) is to ship the hard cap and *gather telemetry* so this story can be built with calibrated numbers, not estimates.

## Story

**As a** product owner,
**I want** the AI token quota to flip from a hard cap to per-token overage billing once we have real usage data,
**so that** heavy users keep working (paying for what they use) instead of hitting a wall, while light users keep their predictable monthly bill.

## Pre-conditions for unblocking

This story does NOT enter active development until ALL of the following are true:

1. **≥30 days** of `ChatUsageEvent` rows in production
2. **≥20 active workspaces** (paying or trialing) generating chat traffic
3. **Calibrated per-token price** — derived from the production data, with documented analysis showing:
   - p50, p90, p99 cost-per-turn distribution
   - Average tokens/month per active user (real mix of context types)
   - Mean monthly $ COGS per workspace
   - Per-token overage price chosen at ≥75% margin against measured average cost-per-token
4. **Decision to ship** — confirmed by product owner that overage billing is wanted (vs. e.g., raising included quotas instead)

When all four are true: lift this story from `backlog/backend/` to `docs/stories/`, set Status to Ready for Development, and proceed.

## Acceptance Criteria

1. **Stripe Meter created** in Test mode + Live mode:
   - Display name: `AI tokens (Laglig)`
   - Event name: `ai_tokens_consumed`
   - Aggregation: `sum` of `payload.tokens`
   - Customer mapping: `payload.stripe_customer_id`
2. **Metered Price added to each tier Product** (Solo + Team):
   - Pricing model: per-unit, billed against the meter
   - Amount: per the calibrated per-token price (decided in pre-condition step 3)
   - Tax behavior: exclusive (matches base Price)
   - Tax code: `txcd_10103000` (matches base Price)
3. **Checkout flow updated** — `app/api/billing/checkout/route.ts` `line_items` becomes an array of two items: the base recurring Price + the metered Price (no `quantity` on the metered item — Stripe pulls it from meter events).
4. **Metered Subscription Item backfill** — for existing subscriptions created before 5.5d ships, run a one-shot script to add the metered Item to each subscription via `stripe.subscriptionItems.create`. Idempotent — re-runnable.
5. **Per-turn meter event emission** — in `onFinish` callback in `app/api/chat/route.ts`:
   - When `WorkspaceUsage.tokens_used_this_period < includedQuota`: do NOT emit a meter event (still within included, no overage)
   - When projected post-turn usage > `includedQuota`: emit `stripe.billing.meterEvents.create({ event_name: 'ai_tokens_consumed', payload: { stripe_customer_id, tokens: <overageTokensThisTurn> } , identifier: <chatUsageEventId> })`
   - The `identifier` is the `ChatUsageEvent.id` for idempotency — Stripe rejects duplicate `(event_name, identifier)` pairs, protecting against retry-driven double-billing
6. **Hard cap relaxed** — Story 5.5c's HTTP 402 at 2× included is replaced with a runaway-protection cap at ~10× included (or an absolute SEK ceiling — to be decided when this story unblocks). Above the runaway cap, request is rejected with `code: 'AI_TOKEN_RUNAWAY_CAP'`. The 80% soft warn behavior stays the same.
7. **Billing dashboard surfaces overage** — usage widget on `/settings?tab=billing` shows:
   - Included quota usage (existing — green progress bar)
   - Overage tokens consumed this period (new — separate counter)
   - Projected month-end overage cost: `overageTokens × per-token price` formatted in SEK
   - Visible warning banner when overage starts: "Du är över din inkluderade kvot — extra användning kostar X SEK / 1 000 tokens"
8. **Customer Portal** already shows the metered Item line on the upcoming invoice — verify in test mode that the per-token charges appear correctly without code changes (Stripe handles the rendering once the metered Item is on the subscription).
9. **Test coverage:**
   - Meter event emission: only fires on overage tokens, not included tokens
   - `identifier` is the `ChatUsageEvent.id` — rejected on retry
   - Backfill script is idempotent (running twice does not create duplicate Items)
   - Runaway cap rejects above 10× included
10. **No retroactive billing** — the calibration period (pre-5.5d) is not retroactively billed. Overage activation is a one-time cutover keyed to a calendar month boundary (e.g., "overage active for all workspaces starting 1 June" to give pre-launch notice). Once active, overage billing follows each workspace's individual Stripe billing cycle per Pattern A — i.e., the per-customer reset model from Story 5.5c remains the source of truth for "what month am I in."

### Why we wait

Setting an overage rate without real data risks two failure modes:

- **Too low** — heavy users above included pay less than their cost. Overage becomes a discount, not a margin protector. We lose money on the most engaged users.
- **Too high** — overage feels predatory. Users hit the meter, see a steep per-token rate, churn rather than upgrade. Worse than a hard cap.

The 30-60 day calibration window in `project_tier_pricing_v1.md` is intentional — set the rate against real cost-per-turn p90, not synthetic estimates.

### Backfill strategy

Existing Stripe subscriptions (post-5.4, pre-5.5d) only have the base recurring Item. To start metering them, we need to add the metered Item. Options:

1. **One-shot backfill script** that iterates over all `Workspace.stripe_subscription_id IS NOT NULL` and calls `stripe.subscriptionItems.create({ subscription, price: METERED_PRICE_ID })`. Run once at deploy time. Idempotent if we check `subscription.items` before creating.
2. **Webhook-driven add** on `customer.subscription.updated` — adds the metered Item if missing on each subscription event. Slower (depends on user activity to trigger), more graceful.

Recommendation: Option 1 for predictability. Option 2 as a defensive fallback for any subscriptions missed by the script.

### Overage event identifier

`ChatUsageEvent.id` is a UUID — use as the `identifier` field on `meterEvents.create`. Stripe enforces uniqueness on `(event_name, identifier)` per account, so retries cannot double-bill. If `ChatUsageEvent` write succeeded but Stripe meter event failed, the next call with the same `identifier` is a no-op (Stripe ignores duplicates by spec).

### Per-token price formatting

The tier pricing memory tentatively suggests `~$0.084/turn` translates to ~$2.80/M tokens at typical ~30K tokens/turn. Calibration MUST be re-derived from production p90 data, not from the v1 estimate. Tax behavior on the metered Price stays Exclusive (ex moms).
