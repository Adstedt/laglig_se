# Story 4.15: Replace Password Signup with Magic Link Authentication

## Status

Draft

## Story

**As a** new user signing up from the landing page,
**I want** to create my account with just my name and email and receive a magic link that logs me in directly,
**so that** I can reach the onboarding wizard faster without creating and re-entering a password.

## Story Context

**Problem:** The current signup flow requires 4 form fields (name, email, password, confirm password), email verification, then re-entering credentials on the login page. This breaks momentum between the high-engagement hero preview and the onboarding wizard. During E2E testing, the email verification + re-login step was identified as the biggest friction point in the onboarding funnel.

**Solution:** Replace password-based signup/login with Supabase magic link (`signInWithOtp`). The signup form shrinks to 2 fields (name, email). The magic link email both verifies the email and logs the user in — one click lands them in onboarding.

**Existing System Integration:**

- **Auth stack:** Supabase Auth (identity) + NextAuth.js (JWT sessions) + CredentialsProvider
- **Key constraint:** NextAuth CredentialsProvider calls `supabase.auth.signInWithPassword()` — this must be replaced with OTP token verification
- **Session strategy:** NextAuth JWT (30-day sessions) — unchanged
- **Callback route:** `/auth/callback` already exchanges Supabase codes for sessions — reusable for magic links
- **localStorage bridge:** Onboarding data (org number, flags, summary) persists in localStorage across the signup→verify→onboarding flow — unaffected by this change

**Dependency:** None — this replaces existing auth, no new infrastructure required. Supabase supports `signInWithOtp` natively.

## Acceptance Criteria

### Signup Flow

1. Signup form at `/signup` has exactly 2 fields: "Fullständigt namn" and "E-postadress"
2. No password, confirm password, or password strength indicator fields exist on the signup form
3. Submitting the form calls `supabase.auth.signInWithOtp({ email })` with `emailRedirectTo` pointing to `/auth/callback`
4. User metadata (`name`) is passed via `options.data` so it persists to the Supabase user record
5. On success, the page shows "Kontrollera din e-post" with messaging: "Vi har skickat en inloggningslänk till din e-postadress. Klicka på länken för att komma igång."
6. The green localStorage banner ("Vi har din företagsinformation redo — den fylls i automatiskt efter registrering") still appears when onboarding data exists
7. Query parameters from the hero preview (`org`, `url`, `flags`, `summary`) are still saved to localStorage on page load

### Magic Link Callback

8. Clicking the magic link in the email hits `/auth/callback` which exchanges the code for a Supabase session via `exchangeCodeForSession()`
9. The callback upserts the user in the local database (Prisma `User` table) with name and email from the Supabase session — same as the current NextAuth `signIn` callback does
10. The callback creates a NextAuth-compatible session (JWT) so the user is fully authenticated without visiting `/login`
11. After session creation, the callback redirects to `/onboarding` (if no workspace exists) or `/dashboard` (if workspace exists)

### Login Flow

12. Login page at `/login` replaces the password field with a "Skicka inloggningslänk" button
13. User enters email → clicks button → receives magic link email → clicks link → logged in
14. The "Kom ihåg min e-post" localStorage feature continues to work (pre-fills email field)
15. Login page shows clear feedback: "Vi har skickat en inloggningslänk till [email]. Kontrollera din inkorg."
16. If the email is not registered, the same message is shown (no user enumeration)

### Password Reset Flow

17. The "Glömt ditt lösenord?" link and `/reset-password` flow are removed — they are no longer needed
18. The `/reset-password/confirm` page is removed

### Admin Auth

19. Admin login at `/admin/login` is **unchanged** — it continues to use password authentication (separate concern, separate user base)

### Validation & Error Handling

20. Email validation uses the existing `z.string().email()` Zod schema
21. Rate limiting: Supabase's built-in OTP rate limiting applies (no custom implementation needed)
22. If `signInWithOtp` fails, a user-friendly Swedish error message is shown
23. If the callback receives an invalid/expired code, redirect to `/login?error=invalid_link` with message "Länken är ogiltig eller har gått ut. Begär en ny."

### Cleanup

24. `PasswordSchema` and `ConfirmPasswordSchema` removed from `lib/validation/auth.ts` (or retained only for admin use)
25. `SignupSchema` updated to require only `name` and `email`
26. `LoginSchema` updated to require only `email`
27. NextAuth `CredentialsProvider` updated: instead of `signInWithPassword`, it verifies the Supabase session token from the callback

### Supabase `signInWithOtp` Reference

```typescript
// Signup: creates user if not exists, sends magic link
const { error } = await supabase.auth.signInWithOtp({
  email,
  options: {
    data: { name }, // stored in user_metadata
    emailRedirectTo: `${origin}/auth/callback`,
  },
})

// This single call handles both signup AND login:
// - New email → creates unconfirmed user, sends magic link
// - Existing email → sends magic link (login)
```

### NextAuth Session from Callback — Approach

The callback needs to create a NextAuth session cookie after Supabase auth. Two approaches:

**Option A (Recommended): Direct JWT cookie**
After `exchangeCodeForSession`, create a NextAuth-compatible JWT and set it as the `next-auth.session-token` cookie. This bypasses the CredentialsProvider entirely for magic link users.

**Option B: Internal redirect through NextAuth**
After `exchangeCodeForSession`, redirect to an internal NextAuth endpoint that accepts a one-time token. More complex, more NextAuth-idiomatic.

Recommend Option A for simplicity. The JWT structure is defined in the existing NextAuth `jwt` callback.
