# Story 7.2: Personalregister ‚Äî employee list view

## Status

Done

## Story

**As an** HR manager,
**I want** a personnel register list mirroring Fortnox,
**so that** I can see and manage all employees at a glance.

## Acceptance Criteria

1. Route `/personalregister` (workspace-nav equivalent of the epic's `/hr/employees`), gated by `employees:view`.
2. **Page header reuses the canonical `PageHeader`** (`components/ui/page-header.tsx`) exactly as the Laglistor page does ‚Äî `title="Personalregister"`, `subtitle`. (The "8/11 kompletta" stats rollup is **deferred to Story 7.4**, which owns the completeness rule; the `stats` prop is optional in the epic.) **Do not build a new header.**
3. **`primaryAction` = a black "+ L√§gg till anst√§lld" button**, styled and positioned identically to Laglistor's "L√§gg till dokument" (mirror `LawListPrimaryAction`). Clicking it writes the `?anstalld=ny` URL param (the 7.3 create-mode modal target; modal itself is Story 7.3 ‚Äî see AC7).
4. **Table follows the canonical Laglistor table pattern** ‚Äî same TanStack scaffolding, shared primitives (`GroupEditor` pattern, `SortableHeader`, `EmptyState`, shadcn `Table`), same UX (row-click, inline group editor, empty state, load-more if needed). **Deviation from epic letter, approved:** `DocumentListTable` is hardcoded to `ColumnDef<DocumentListItem>` with no external column API, so a parallel `EmployeeListTable` typed over `EmployeeView` rows is built on the same scaffolding. The Laglistor `DocumentListTable` itself is **not modified**.
5. Columns: Anst√§llnings-ID (`employee_id_ref`), Anst√§lld (namn), Personnummer (masked unless `employees:manage` ‚Äî display what `EmployeeView.personnummer` already contains), Personaltyp, Anst√§llningsform, L√∂neform, Kollektivavtal (agreement name or "Ej ifylld"), Status (Aktiv/Inaktiv).
6. Status filter tabs via `WorkspaceViewTabs`: **Alla / Aktiva / Inaktiva** (driven by `Employee.inactive`). The **"Ej kompletta" tab is deferred to Story 7.4** with the completeness rule.
7. Search filters by name / personnummer / anst√§llnings-ID ‚Äî **client-side** over the loaded list (personnummer is encrypted at rest; ciphertext is not DB-searchable; decrypted values exist in memory only for `employees:manage`).
8. Row click writes `?anstalld=<id>` via the History API (exactly the `?document=` pattern from `DocumentListPageContent`); the Personalkort modal mounts on that param in Story 7.3 ‚Äî this story wires param + state sync only.
9. **Grouping by org unit** (`EmployeeGroup`): rows render under collapsible group headers (Lager / Huvudkontor / HR‚Ä¶), employees can be **dragged between groups** and moved via the inline `GroupEditor`-style select; ungrouped employees render under "Ogrupperad".
10. Create / rename / reorder / delete groups; deleting a group leaves its employees ungrouped (FK is `onDelete: SetNull` ‚Äî verify UI reflects it).
11. Nav: enable the existing stubbed **HR** item in the left sidebar (`href: '/personalregister'`, drop `disabled`/`"snart"` badge), mirror in the mobile sidebar. Access enforcement lives at the route (sidebar has no permission-gating mechanism today ‚Äî do not build one in this story).
